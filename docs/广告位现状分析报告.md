## 四、发现的问题

### 问题 1：抽奖页存在两个功能重复的弹窗位

- `lottery_popup`（ID:3）日价 80 钻石，无任何广告计划
- `lottery_popup_ad`（ID:7）日价 200 钻石，有 1 条广告计划

两者 description 完全相同（"抽奖页面弹窗广告位"），slot_type 和 position 也一样，疑似历史遗留导致的重复数据。ID:3 从未被使用过。

### 问题 2：slot_category 数据与代码模型定义不一致

代码 `models/AdSlot.js` 中 slot_category 的默认值为 `display`，注释说明有效值为 `display`（展示广告）和 `feed`（信息流广告）。但数据库中 ID:14 和 ID:15 两条信息流广告位的 slot_category 存储的是 `standard`，而非 `feed`。这是一个数据不一致问题。

### 问题 3：个人中心轮播位长期停用

`profile_carousel`（ID:8）描述为"暂未开放"，is_active=0，且从未关联过任何广告计划。需要确认是计划中的功能还是已废弃。

### 问题 4：信息流广告位已就绪但无投放

`market_feed`（ID:14）和 `exchange_feed`（ID:15）均已启用，配置完整（含 CPM 价格），但没有任何广告主投放。需要确认是刚上线还是推广不足。

### 问题 5：5 个广告位处于空置状态

9 个广告位中有 5 个没有关联任何广告计划（lottery_popup、profile_popup、profile_carousel、market_feed、exchange_feed），空置率 55.6%。

### 问题 6：广告图片上传功能整条链路缺失

> 发现时间：2026-03-11
> 验证方式：逐层代码审查（前端 HTML → 前端 JS → 后端路由 → 后端 Service → 数据模型）

**现象**：管理后台创建广告计划时，无法上传广告图片。`content_type` 默认为 `image`，但实际没有任何上传入口。

**代码链路分析**：

| 层级 | 文件 | 现状 | 问题 |
|---|---|---|---|
| 数据模型 | `models/AdCreative.js` | `image_url` 字段存在 | 正常，数据层已预留 |
| 后端 Service | `services/AdCreativeService.js` | `createCreative()` 接受 `image_url` 参数 | 正常，但依赖调用方传入 URL |
| 后端路由 | `routes/v4/console/ad-campaigns.js` | 创建接口只接收 JSON（`content_type`、`text_content`） | 没有 multer 中间件，不支持文件上传 |
| 前端 JS | `admin/src/modules/content/pages/ad-management.js` | `saveCampaign()` 提交纯 JSON 对象 | 没有 FormData，没有文件处理逻辑 |
| 前端 HTML | `admin/ad-management.html` | 创建表单中无 `<input type="file">` | 没有图片选择器、拖拽区域、预览组件 |
| 详情展示 | `admin/ad-management.html:975` | `<img :src="cr.image_url">` 能展示素材 | 只读展示正常，但创建时无法写入 |

**影响范围**：

- 所有需要图片素材的广告类型（popup 弹窗、carousel 轮播图、feed 信息流）均无法上传图片
- 目前只有纯文本类型的系统通知（announcement）能正常创建
- 数据库中已有的广告计划，其 `ad_creatives.image_url` 均为 NULL

**补全方案（先上传后提交，复用现有图片上传体系）**：

> 核心思路：前端先调独立的图片上传接口 `POST /api/v4/console/images/upload` 拿到 `public_url`，再在创建广告计划时把 URL 一起提交给后端，后端创建 campaign 后自动创建 AdCreative 记录。不需要给 ad-campaigns 路由加 multer，不需要改数据库。

**Step 1 — 前端 JS（`admin/src/modules/content/pages/ad-management.js`）**

- 引入已有的 `imageUploadMixin`（`admin/src/alpine/mixins/image-upload.js`）
- Alpine 组件混入 `...imageUploadMixin()`
- 新增状态：
  - `campaignImagePreview`：本地 blob 预览 URL
  - `campaignImageUrl`：上传成功后的远程 URL（用于提交）
- 新增方法：
  - `handleCampaignImageUpload(event)`：调用 mixin 的 `uploadImage(file, { business_type: 'uploads', category: 'ad_creative' })`，成功后设置 `campaignImageUrl`
  - `clearCampaignImage()`：清除预览和 URL
- `openCreateCampaignModal()` 中重置 `campaignImagePreview` 和 `campaignImageUrl`
- `saveCampaign()` 提交数据时，如果 `campaignImageUrl` 有值则附加 `data.image_url`

**Step 2 — 前端 HTML（`admin/ad-management.html`）**

- 在创建广告活动弹窗中，"弹窗显示模式"字段之前，添加图片上传区域（仅 `campaign_category !== 'system'` 时显示）：
  - 未选图片时：虚线框 + 点击触发隐藏的 `<input type="file">`
  - 已选图片时：`<img>` 预览 + 右上角删除按钮 + 上传中 loading 遮罩
  - `<input type="file" accept="image/jpeg,image/png,image/gif,image/webp">` 隐藏，通过 `@change` 触发 `handleCampaignImageUpload`

**Step 3 — 后端路由（`routes/v4/console/ad-campaigns.js`）**

- `POST /operational` 路由：透传 `image_url`、`content_type`、`text_content`、`display_mode` 给 service
- `POST /system` 路由：同上透传这四个字段
- `POST /`（商业广告）路由：将 `image_url`、`content_type`、`text_content`、`display_mode` 加入白名单字段

**Step 4 — 后端服务层（`services/AdCampaignService.js`）**

- `createOperationalCampaign()`：创建 AdCampaign 后，如果 `data.image_url` 或 `data.text_content` 有值，自动创建一条 AdCreative 记录（`review_status: 'approved'`，运营类型跳过审核）
- `createSystemCampaign()`：同上逻辑
- `createCampaign()`（商业广告）：同上，但 `review_status: 'pending'`（商业广告需审核）

**涉及文件清单**：

| 文件 | 改动类型 | 说明 |
|---|---|---|
| `admin/src/modules/content/pages/ad-management.js` | 修改 | 引入 imageUploadMixin，添加图片状态和方法 |
| `admin/ad-management.html` | 修改 | 创建弹窗中添加图片上传 UI |
| `routes/v4/console/ad-campaigns.js` | 修改 | 三个 POST 路由透传图片相关字段 |
| `services/AdCampaignService.js` | 修改 | 三个创建方法中自动创建 AdCreative |

**不需要改动的部分**：

- 图片上传接口（`routes/v4/console/images.js`）— 已有，直接复用
- 图片上传 Mixin（`admin/src/alpine/mixins/image-upload.js`）— 已有，直接复用
- ImageService / Sealos 对象存储 — 已有，直接复用
- 数据库 / 数据模型 — 不需要迁移，`ad_creatives.image_url` 字段已存在

### 前端渲染规则

1. 前端在固定位置预埋广告位组件，通过 `slot_type` + `position` 向后端请求投放数据
2. 后端返回空数据（无投放或广告位关闭）时，前端对应区域完全隐藏，不占用页面空间
3. 后端返回有效数据时，前端根据 `slot_type` 渲染对应模板（弹窗/轮播/公告/信息流卡片）
4. 前端不关心广告位是否启用、有没有人投放、如何计费，这些逻辑全部由后端控制

### 现有隐藏机制说明

| 广告位类型 | 隐藏方式 | 控制变量 | 所在文件 |
|---|---|---|---|
| popup（弹窗） | 后端返回空列表 → 不触发弹窗组件 | 弹窗数据为空时组件不渲染 | `packageLottery/popup-banner/popup-banner.ts` |
| carousel（轮播） | `showCarousel = false` → swiper 区域隐藏 | `showCarousel` | `pages/lottery/lottery.ts` |
| announcement（公告） | `showAnnouncements = false` → 公告条隐藏 | `showAnnouncements` | `pages/lottery/lottery.ts` |
| feed（信息流） | 尚未实现，待开发时需遵循同样的隐藏规则 | — | — |
