## 四、发现的问题（按归属分类，基于代码审查实际验证）

### 4.1 后端数据库项目问题

#### 问题 B1：slot_category 数据与模型注释不一致

- **现象**：`models/AdSlot.js` 注释说明 slot_category 有效值为 `display` 和 `feed`，但 ID:14（market_feed）和 ID:15（exchange_feed）在数据库中存储的是 `standard`
- **影响**：如果代码层做 `slot_category === 'feed'` 判断会漏掉这两个广告位
- **修复方案**：直接更新数据库：`UPDATE ad_slots SET slot_category='feed' WHERE ad_slot_id IN (14,15)`

#### 问题 B2：抽奖页存在两个功能重复的弹窗位

- **现象**：`lottery_popup`（ID:3）和 `lottery_popup_ad`（ID:7）的 slot_type、position 完全相同，仅 slot_key 和定价不同
- **ID:3 从未被使用**：0 条关联计划
- **修复方案**：直接 DELETE（已决策，见第七节）

#### ~~问题 B3：ad_creatives.image_url 绕过了 image_resources 体系~~ — 已过时

- **原始描述**：ad_creatives 的 image_url 绕过 image_resources 体系
- **实际状态（2026-03-17）**：`ad_creatives.image_url` 列已在 2026-03-16 媒体体系迁移中移除，`image_resources` 表已被 DROP。替代方案为 `ad_creatives.primary_media_id`（FK → `media_files.media_id`），通过 MediaService 统一管理
- **当前状态**：ad_creatives 表已清空（决策 5），此问题不再存在

#### 问题 B4：ad-delivery 接口返回的 image_url 是对象 key 而非完整 URL — ✅ 已修复

- **原始问题**：ad-delivery 直接透传对象 key，前端需要自行拼接 URL
- **修复结果**：ad-delivery 路由现在通过 `AdBiddingService` 获取 `creative.media_object_key`（由 `primary_media_id` → `media_files.object_key` 关联查询得到），再拼接为完整代理 URL：`${PUBLIC_BASE_URL}/api/v4/images/${media_object_key}`
- **前端使用**：直接 `<image src="{{item.image_url}}">` 即可，无需拼接逻辑

#### 问题 B5：createSystemCampaign() 不自动创建 AdCreative

- **原始问题**：`createSystemCampaign()` 不自动创建 AdCreative。路由层已传入 `primary_media_id`、`content_type`、`text_content`、`display_mode`，但 Service 层未使用
- **修复结果**：当 `primary_media_id` 或 `text_content` 有值时，在同一事务内自动创建 AdCreative（`review_status: 'approved'`，系统类型免审核）

#### 问题 B6：管理员创建商业广告路由白名单缺少图片字段 — ✅ 已修复

- **原始问题**：`ALLOWED_ADMIN_CREATE_FIELDS` 白名单缺少素材相关字段
- **修复结果**：白名单已包含 `primary_media_id`、`content_type`、`text_content`、`display_mode`（2026-03-16 完成）。`image_url` 未纳入白名单——因为 `ad_creatives` 表已无此列，素材图片通过 `primary_media_id`（FK → `media_files.media_id`）关联
- **POST `/operational` 和 POST `/system`**：已正确透传 `primary_media_id`、`content_type`、`text_content`、`display_mode`（不传 `image_url`，因为该列不存在）

#### 问题 B7：createCampaign()（商业广告）不自动创建 AdCreative — ✅ 已修复

- **原始问题**：`createCampaign()` 不自动创建 AdCreative
- **修复结果**：当 `primary_media_id` 或 `text_content` 有值时，在同一事务内自动创建 AdCreative（`review_status: 'pending'`，商业广告需审核）

#### 问题 B8：用户端创建广告路由白名单缺少图片字段 — ✅ 已修复

- **原始问题**：`ALLOWED_CAMPAIGN_CREATE_FIELDS` 白名单缺少素材相关字段
- **修复结果**：白名单已包含 `primary_media_id`、`content_type`、`text_content`、`display_mode`。`image_url` 未纳入白名单——因为 `ad_creatives` 表已无此列，素材图片通过 `primary_media_id`（FK → `media_files.media_id`）关联

#### 问题 B9：ad-delivery 路由 slot_type 校验不包含 feed

- **代码验证**：`routes/v4/system/ad-delivery.js` 第 59 行 `validSlotTypes = ['popup', 'carousel', 'announcement']`，不包含 `'feed'`，但 `models/AdSlot.js` 第 27 行 `VALID_SLOT_TYPES = ['popup', 'carousel', 'announcement', 'feed']`
- **影响**：feed 类型广告位（market_feed、exchange_feed）无法通过 ad-delivery 接口下发
- **需要拍板**：见决策 6

#### 问题 B10：ad-delivery slotKey 构造逻辑与 feed 广告位 slot_key 不匹配（二次校验新发现，严重）

- **代码验证**：`routes/v4/system/ad-delivery.js` 第 64-65 行 `const slotKey = position + '_' + slot_type`，即用 `{position}_{slot_type}` 拼接 slot_key 去查询。但 feed 广告位的 slot_key 不遵循此规则：
  - `market_feed`（ID:14）：position=`market_list`，slot_type=`feed` → 拼接出 `market_list_feed` ≠ `market_feed`
  - `exchange_feed`（ID:15）：position=`exchange_list`，slot_type=`feed` → 拼接出 `exchange_list_feed` ≠ `exchange_feed`
- **影响**：即使修复 B9（validSlotTypes 加 feed），ad-delivery 仍然无法查到 feed 广告位，因为 `AdBiddingService.selectWinners('market_list_feed', userId)` 查不到 slot_key=`market_feed` 的广告位
- **同时影响**：删除 ID:3（lottery_popup）后，ID:7 的 slot_key 是 `lottery_popup_ad`，而 `position=lottery + slot_type=popup` 拼接出的是 `lottery_popup`，也查不到
- **修复方案**：决策 8 已定，方案 A：重命名 slot_key（Step B11）

#### 问题 B11：用户端广告位查询 VALID_SLOT_TYPES 范围过窄（二次校验新发现）

- **代码验证**：`routes/v4/user/ad-slots.js` 第 36 行 `const VALID_SLOT_TYPES = ['popup', 'carousel']`，仅 2 种类型
- **对比**：模型层 `models/AdSlot.js` 第 27 行支持 4 种：`['popup', 'carousel', 'announcement', 'feed']`
- **影响**：用户端（小程序广告主）无法通过 `slot_type=announcement` 或 `slot_type=feed` 筛选广告位。虽然不传 slot_type 参数时能返回全部活跃广告位（含 feed），但传 `announcement` 或 `feed` 会被路由校验拒绝
- **修复方案**：`VALID_SLOT_TYPES` 扩展为 `['popup', 'carousel', 'announcement', 'feed']`，与模型层保持一致

### 决策 6：feed 类型广告位的下发机制

#### 问题说明

数据库里有 2 个信息流广告位（`market_feed` 和 `exchange_feed`，slot_type=feed），但 ad-delivery 接口（`routes/v4/system/ad-delivery.js` 第 59 行）的校验写死了 `['popup', 'carousel', 'announcement']`，不允许 `feed`。而 `models/AdSlot.js` 第 27 行 `VALID_SLOT_TYPES` 已包含 `'feed'`。模型层已支持，路由层卡住了。

feed 广告与 popup/carousel/announcement 的区别：feed 嵌入到列表页商品流中（如交易市场商品列表里每隔几条穿插一条广告），不是独立的 UI 覆盖物。

#### 行业做法对比

| 公司/平台 | 广告下发方式 | feed 广告如何处理 |
|---|---|---|
| **Google Ads** | 统一 Ad Manager API，slot_type 参数区分格式 | Native/Feed 和 Banner/Interstitial 用同一个 API，format 参数不同 |
| **Meta Ads** | 统一 Marketing API | Feed ad、Stories ad、Reels ad 全部通过同一 placement 参数区分 |
| **阿里妈妈/淘宝联盟** | 统一广告服务 API | 信息流广告、横幅广告、开屏广告都是同一接口的 ad_type 参数 |
| **腾讯广告（广点通）** | 统一 SDK/API | Banner、Native（信息流）、Splash 均通过 slot_id 区分，SDK 内统一请求 |
| **美团广告** | 统一投放引擎 | 不同 position 对应不同广告位，信息流和弹窗只是 slot 配置不同，下发接口相同 |
| **穿山甲（字节跳动）** | 统一 SDK | 信息流原生广告和开屏/插屏广告用同一个 loadAd() 方法，code_id 不同 |
| **Steam / Epic** | 统一 Storefront API | 推荐位、轮播位、列表穿插位都走同一内容下发，position 参数不同 |
| **闲鱼/转转** | 统一推荐引擎 | 搜索列表穿插广告和首页 Banner 走同一广告服务 |
| **小型虚拟物品交易平台** | 通常一个接口搞定 | 不会因为 feed 类型单独建接口，slot 配置区分即可 |
| **游戏内广告平台（Unity Ads / AdMob）** | 统一 SDK | Rewarded、Interstitial、Banner、Native 四种格式一个 SDK 入口 |

**行业共识**：没有公司为不同的广告格式（feed/popup/carousel/banner）建独立接口。统一入口 + 参数区分 是绝对的行业标准。穿插逻辑由前端（或前端 SDK）负责，不是广告下发接口的职责。

#### 两种方案对比

| 维度 | A. 扩展 ad-delivery | B. 新建独立接口 |
|---|---|---|
| 后端改动量 | **1 处**：第 59 行数组加 `'feed'` | 新建 1 个路由文件，约 50 行代码 |
| 前端对接成本 | **零额外成本**：调 `getAdContent('feed', 'market_list')` | 需要新增 1 个 API 封装函数 |
| 是否符合行业标准 | **是**：与 Google/Meta/阿里/腾讯做法一致 | 不符合，违背统一下发原则 |
| 后端已有架构兼容性 | **完全兼容**：模型层已支持 feed，竞价引擎已支持，响应结构通用 | 需要重复实现竞价选中 + 扁平化逻辑 |
| 长期维护成本 | **低**：1 个入口维护 | 高：2 个入口，逻辑同步成本 |
| 穿插逻辑归属 | 前端负责（调 API 拿广告列表，自己决定插入位置） | 同左，与接口设计无关 |
| 未来扩展 | 再加新广告类型只需在数组加 1 个值 | 每种新类型都要考虑是否建新接口 |

#### 本项目实际情况

- `models/AdSlot.js` VALID_SLOT_TYPES 已包含 `'feed'`
- `AdBiddingService.selectWinners()` 竞价引擎对 slot_type 无限制，feed 类型直接可用
- ad-delivery 的响应结构（`image_url`、`title`、`link_url` 等）feed 广告完全通用
- 穿插逻辑（每 N 条商品插 1 条广告）应由小程序列表页前端控制，与广告下发接口无关

**已决策：方案 A — 扩展 ad-delivery 路由支持 feed**

理由：模型层已支持、竞价引擎已支持、响应结构已通用，路由校验是唯一卡点。符合 Google/Meta/阿里/腾讯/美团/穿山甲/Unity Ads 所有大厂的统一下发标准。新建独立接口是过度设计，制造不必要的维护成本。

### 决策 8：feed 广告位 slot_key 命名不一致如何修复？（二次校验新增，需要拍板）

#### 问题说明

ad-delivery 路由用 `position + '_' + slot_type` 拼接 slot_key 去匹配广告位。但以下 3 个广告位的 slot_key 不遵循此规则：

|---|---|---|---|---|---|
| 7 | lottery_popup_ad | lottery | popup | lottery_popup | **否**（删除 ID:3 后，`lottery_popup` 查不到 `lottery_popup_ad`） |
| 14 | market_feed | market_list | feed | market_list_feed | **否** |
| 15 | exchange_feed | exchange_list | feed | exchange_list_feed | **否** |

#### 两种方案

| 维度 | A. 重命名 slot_key（改数据库） | B. 改 ad-delivery 查询逻辑（改代码） |
|---|---|---|
| 改动量 | 3 条 UPDATE SQL | 改 ad-delivery 路由约 5 行：先 `AdSlot.findOne({ where: { slot_type, position, is_active: true } })` 拿到 slot_key，再调 `selectWinners` |
| 数据一致性 | **全部 slot_key 遵循统一命名规则**，未来新增广告位不需要特殊处理 | slot_key 可以是任意值，但 ad-delivery 路由每次请求多一次数据库查询 |
| 对现有代码的影响 | 仅影响 slot_key 值，ad-delivery 路由逻辑不变 | 需要在路由层引入 AdSlot 模型依赖 |
| 对 ad_campaigns 的影响 | 无影响（ad_campaigns 用 ad_slot_id 关联，不依赖 slot_key） | 无影响 |
| 性能 | **无额外查询** | 每次 ad-delivery 请求多 1 次 SELECT |
| 长期维护 | **低**：命名规则统一，新增广告位只需遵循 `{position}_{slot_type}` | 中：任何 slot_key 都可以工作，但命名不一致增加认知负担 |

**已决策：方案 A — 重命名 slot_key**

理由：slot_key 是内部标识符，前端从不直接使用（ad-delivery 接口参数是 slot_type 和 position，不是 slot_key），重命名零风险。统一命名规则后长期维护成本更低。
