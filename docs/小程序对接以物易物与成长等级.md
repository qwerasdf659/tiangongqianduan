## 一、需后端程序员确认 / 提供的信息（1 项优化建议 + 2 项契约确认）

### B-1【优化建议】换物配方接口建议补充产出商品的展示字段

- **接口**：`GET /api/v4/exchange/barter/recipes`
- **现状**：`data.recipes[]` 仅下发 `output_exchange_item_id`（产出商品主键），不含产出商品名称与履约类型。
- **前端影响**：
  1. 判断"该配方产出是否为实物（是否必选收货地址）"，前端目前需要在用户点击"去合成"时额外调用一次 `GET /api/v4/exchange/items/:exchange_item_id` 读取 `fulfillment_type`——多一次请求往返（已实现兜底：查询失败时由后端 `BARTER_ADDRESS_REQUIRED` 错误码兜底拦截，功能不受阻）；
  2. 配方卡片无法直接展示"产出：毛巾礼盒"这类商品名，用户只能看到配方名。
- **希望后端提供**：在 recipes[] 每项追加两个只读展示字段（均为现有 exchange_items 数据，不涉及敏感信息）：

| 字段 | 类型 | 含义 |
|---|---|---|
| `output_item_name` | string | 产出商品名称（exchange_items.item_name 快照） |
| `output_fulfillment_type` | string | 产出履约类型：`physical`（实物快递，前端据此要求选地址）/ `virtual` / `voucher` |

- **优先级**：低（当前实现可用，此项仅为减少一次请求 + 提升配方卡片信息量）。

> **✅ 后端答复（2026-07-11 已实现并上线验证）**：两个字段已按建议原名追加到 `recipes[]` 每项，可直接消费、删除"二次查商品详情"的兜底请求：
>
> - `output_item_name`：产出商品名称（如"毛巾礼盒（换物）"），null 表示产出商品已不存在（异常态，正常不会出现）；
> - `output_fulfillment_type`：取值 `physical` / `voucher` / `virtual`。**判定源与后端履约分流完全同源**（产出商品关联模板的 item_type：product→physical、voucher→voucher、其它→virtual），即 `output_fulfillment_type==='physical'` ⇔ 换物必传 `address_id`（否则后端返回 `BARTER_ADDRESS_REQUIRED`），前端可放心据此提前拉起地址选择。
>
> 当前真实库线上返回示例（试点配方，可直接联调）：
>
> ```json
> {
>   "recipes": [{
>     "recipe_code": "towel_set_x2_to_new_towel_set",
>     "name": "毛巾礼盒×2 换 新毛巾礼盒",
>     "required_item_template_id": 16,
>     "required_quantity": 2,
>     "output_exchange_item_id": 697,
>     "is_enabled": true,
>     "per_user_limit": 1,
>     "total_limit": 100,
>     "output_item_name": "毛巾礼盒（换物）",
>     "output_fulfillment_type": "physical"
>   }]
> }
> ```

### B-2【契约确认】换物成功响应的 `order_status` 字段

- **接口**：`POST /api/v4/exchange/barter`
- 对接文档 §十一-M1 写明成功响应含 `{ order_no, order_status, consumed_item_ids, minted_item }`。前端已按 `order_status='pending'`（实物走快递）/ `'completed'`（券/道具即时到账）分流展示。
- **请后端确认**：字段名确为 `order_status`（而非 `status`）。如实际为 `status`，请告知前端，改一处判断即可。

> **✅ 后端答复（已核对源码 `BarterService.executeBarter` 返回体）**：确认字段名为 **`order_status`**，取值只有两种：`'pending'`（实物，走快递发货链）/ `'completed'`（券/道具，即时到账且 `minted_item` 非 null）。前端现有分流判断无需改动。
> 另注意两点：
> 1. **幂等重放**时响应 data 为首次成功的完整结果 + 追加 `is_duplicate: true`，`order_no` 与首次一致，前端按成功处理即可；
> 2. "我的订单"列表读取订单实体时，订单实体上的状态字段是 `status`（ExchangeRecord 模型字段，与普通兑换单一致）——`order_status` 只出现在换物提交接口的响应体里，两处语境不同勿混用。

### B-3【契约确认】升级通知 type 值

- **通道**：`user_notifications` 拉取 + WebSocket `new_notification`
- 前端已按对接文档 §十一-M5 注册 `type='growth_level_upgrade'`（标签"等级"、点击跳转成长等级页）。
- **请后端确认**：升级通知写入 user_notifications 时 type 值确为 `growth_level_upgrade`。

> **✅ 后端答复（已核对源码 `consumption/CoreService` 升级通知 + `NotificationService` 写入链）**：确认 type 值为 **`growth_level_upgrade`**，原样写入 user_notifications.type 并随 WebSocket `new_notification` 事件推送，前端注册无需改动。

### B-4【后端主动勘误】升级通知的载荷字段名是 `metadata`（不是 `data`）

> **⚠️ 对接文档 §十一-M5 曾写"`data` 含 to_level_name / to_earn_multiplier"，经 2026-07-11 核对实际链路更正**：
>
> - `GET /api/v4/user/notifications` 列表项与 WebSocket `new_notification` 事件里，升级通知的业务载荷字段名均为 **`metadata`**（`data` 是后端发送方法的入参名，落库与下发时统一为 metadata）；
> - `metadata` 内容：`{ from_level_key, to_level_key, to_level_name, to_earn_multiplier, consumption_record_id }`；
> - 如前端已按 `data` 取值，请改读 `metadata`（一处）；`type`/`title`/`content` 字段名不变，文案后端已生成无需拼装。
