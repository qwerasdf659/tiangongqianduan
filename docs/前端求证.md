# SESSION_INVALIDATED 认证异常 — 后端排查文档

## 问题概述

| 项目 | 内容 |
|------|------|
| **发现时间** | 2026-02-18 01:02:13 |
| **影响范围** | 所有需要认证的REST API请求（新/旧会话均无法发送消息） |
| **问题现象** | 后端返回 `401 SESSION_INVALIDATED`，但用户JWT Token有效且无其他设备登录 |
| **前端版本** | v5.2.0（TypeScript + MobX + Socket.IO） |
| **触发场景** | 点击客服会话 → 加载聊天历史 → 401被拒 |

---

## 复现步骤

1. 用户正常登录（user_id=31, role_level=100）
2. JWT Token解码成功（exp=1771440424，未过期）
3. 点击客服会话列表中的某个会话（session_id=1873）
4. 前端同时发起：
   - REST API请求：`GET /api/v4/system/chat/sessions/1873/messages?page=1&limit=50`
   - Socket.IO连接：`wss://omqktqrtntnn.sealosbja.site` with `auth: { token }`
5. **结果矛盾**：
   - REST API → `401 SESSION_INVALIDATED`（认证失败）
   - Socket.IO → `connection_established`（连接成功）
6. 用户确认：**没有其他设备登录**

## 问题根因分析

```
同一个JWT Token，同一时刻（01:02:13~01:02:14）:

REST API认证中间件:
  jwt.verify(token) → 通过
  checkSessionStatus(user_id) → ❌ SESSION_INVALIDATED
  → 返回 401

Socket.IO握手认证:
  jwt.verify(token) → 通过
  → ✅ 连接成功（未检查session状态？）

矛盾点: REST API有额外的session状态检查，Socket.IO没有
```

### 怀疑的后端问题方向

1. **user_sessions 表的记录被错误标记为 invalidated**
   - 可能原因：定时任务误清理、并发登录检测逻辑误判、session过期策略过于激进
   - 排查：查询 `user_sessions` 表中 user_id=31 的所有记录及其状态变更时间

2. **REST API 和 Socket.IO 认证中间件不一致**
   - REST API 的 `authenticateToken` 中间件包含 session 有效性检查
   - Socket.IO 的握手认证仅做 `jwt.verify()`，缺少 session 检查
   - 需要统一两端的认证逻辑

3. **单设备登录策略误触发**
   - 错误消息提示"可能是其他设备登录导致"，但用户确认无其他设备
   - 可能是小程序冷启动/热启动时的 `checkAuthStatus()` 触发了新的session，导致旧session被invalidated
   - 排查：检查 session 创建和 invalidation 的时间线

## 前端当前处理逻辑（无需修改）

前端 `client.ts` 对 401 的处理是正确的：

```
收到 401 响应
  │
  ├── code === 'TOKEN_EXPIRED'
  │     └──→ handleTokenExpired() → 自动刷新Token → 重试请求
  │
  └── 其他code（包括 SESSION_INVALIDATED）
        └──→ handleTokenInvalid()
              ├──→ 清除认证数据（Store + Storage）
              ├──→ 弹窗提示用户
              └──→ 跳转登录页
```

前端在收到后端的 `SESSION_INVALIDATED` 后，正确地执行了清除认证并引导重新登录的逻辑。问题在于**后端不应该在用户无其他设备登录的情况下返回此错误码**。

## 关于日期格式的说明

当前后端返回的 `created_at` 字段格式为 `"2026-02-15 02:15:02"`（空格分隔）。

**iOS JS引擎不支持此格式**，仅支持：

- `"2026-02-15T02:15:02"` （ISO 8601，推荐）
- `"2026/02/15 02:15:02"` （斜杠分隔）
- `"2026-02-15"` （仅日期）

前端已通过 `safeParseDateString()` 工具函数做了兼容处理，后端无需修改日期格式。但**建议**后端在未来版本统一返回 ISO 8601 格式（带T分隔符和时区偏移），例如：

```
"2026-02-15T02:15:02+08:00"
```

# 客服聊天 — 后端 DataSanitizer 启用需求

> **文档日期**: 2026-02-19  
> **提出方**: 前端（微信小程序）  
> **优先级**: 🔴 高 — 聊天功能核心阻断  
> **关联文档**: `docs/DataSanitizer字段对齐与奖品图片方案.md`（2026-02-18 已拍板）

---

## 一、问题现象

用户在微信小程序中无法发送客服消息，新会话和旧会话均受影响。

控制台日志：

```
[chat-msg] 📨 sendMessage 调用
  { inputContent: "你哈", contentLength: 2, sessionId: undefined, wsConnected: true }
[chat-msg] ⚠️ 会话ID为空，无法发送消息
```

## 二、根本原因分析（纯文本数据流图）

```
用户点击"+"创建新会话
        │
        ▼
前端调用 POST /api/v4/system/chat/sessions
        │
        ▼
后端返回会话数据（当前未经 DataSanitizer）
        │
        │  ┌──────────────────────────────────────────────┐
        │  │ 后端实际返回:                                 │
        │  │ {                                             │
        │  │   "success": true,                            │
        │  │   "data": {                                   │
        │  │     "customer_service_session_id": 19473,     │
        │  │     "status": "waiting",                      │
        │  │     ...                                       │
        │  │   }                                           │
        │  │ }                                             │
        │  └──────────────────────────────────────────────┘
        │
        ▼
前端读取 session.id （DataSanitizer 预期输出字段）
        │
        ▼
session.id === undefined  ← 根本原因：后端未调用 sanitizeChatSessions()
        │
        ▼

```

## 三、问题定位

参考 `docs/DataSanitizer字段对齐与奖品图片方案.md` 第 2.2 节：

**结论**: `sanitizeChatSessions()` 方法本身逻辑正确，但没有被任何客服聊天 API 路由调用。

## 四、需要后端处理的事项

### 4.1 在客服聊天 API 路由中启用 DataSanitizer

以下 API 路由的响应数据需要经过 `sanitizeChatSessions()` / 相应 sanitize 方法处理后再返回：

| 序号 | API 路由 | 说明 | 需要 sanitize 的数据 |
|------|---------|------|---------------------|
| 1 | `POST /api/v4/system/chat/sessions` | 创建客服会话 | 返回的 session 对象 |
| 2 | `GET /api/v4/system/chat/sessions` | 获取会话列表 | sessions 数组中每个 session |
| 3 | `GET /api/v4/system/chat/sessions/:id/messages` | 获取消息历史 | messages 数组中每条 message |
| 4 | `GET /api/v4/system/chat/sessions/search` | 搜索消息 | messages 数组中每条 message |

### 4.2 字段映射预期

DataSanitizer 处理后，前端期望收到的字段名：

**会话对象 (session)**:

```
数据库原始字段                      → DataSanitizer 输出字段
customer_service_session_id (BIGINT) → id
```

**消息对象 (message)**:

```
数据库原始字段                → DataSanitizer 输出字段
chat_message_id (BIGINT)     → id
customer_service_session_id  → session_id（外键引用）
```

### 4.3 前端临时适配说明

在后端启用 DataSanitizer 之前，前端已添加字段兼容提取逻辑：

```typescript
// 前端当前提取方式（兼容 DataSanitizer 启用前/后）
const sessionId = session.id ?? session.customer_service_session_id
const messageId = msg.id ?? msg.chat_message_id
```

**后端启用 DataSanitizer 后**，前端将移除 `?? customer_service_session_id` 部分，
统一使用 `id` 字段（符合 DataSanitizer 商业安全设计）。
