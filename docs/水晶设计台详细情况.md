## 你现在设计台用的是什么

**Canvas 2D + 数学投影模拟的"伪 3D"**,不是真 3D。具体技术栈:

- **渲染**:微信原生 `<canvas type="2d">`,用 `drawImage` 把珠子实拍图贴上去。
- **立体感来源**:不是三维几何,而是**二维绘图技巧**——圆形裁剪 + 径向渐变暗角 + 高光点(`createRadialGradient`)模拟球面,离屏 canvas 缓存这些模板。
- **"3D 翻转"的本质**:我用**纯数学**把圆环上每颗珠子的坐标做了 yaw(绕Y轴)+ pitch(绕X轴)旋转,再做透视投影(`focal/(focal-z)` 近大远小),按深度 z 排序决定谁先画(前后遮挡)。**珠子本身还是那张 2D 平面图片**,只是位置/大小/前后顺序在变。
- **光照**:固定的,高光永远在珠子左上角,不会随翻转改变。

一句话:**它是"会算 3D 坐标的 2D 贴图",珠子是纸片,不是真正的球体。**

## 和 Three.js 移植方案的区别

Three.js 是**真 3D 引擎**(WebGL)。核心区别:

| 维度 | 你现在的(Canvas 2D 伪3D) | Three.js 真3D |
| --- | --- | --- |
| **珠子本体** | 2D 平面图片,翻转时其实是纸片 | 真球体网格(Sphere Mesh),任意角度都是立体 |
| **光照** | 固定高光,不随旋转变 | 真实光源,高光/阴影随转动实时变化 |
| **材质** | 渐变模拟,假通透 | PBR 材质,真折射/反射/次表面散射(水晶通透感真实) |
| **翻转** | 数学投影 + 深度排序模拟 | GPU 真三维变换,天然正确 |
| **性能** | CPU 逐颗画,珠子多会掉帧 | GPU 批量渲染,几百颗也流畅 |
| **微信支持** | 原生支持,已在跑 | 需 **XR-Frame** 或 threejs-miniprogram 适配层 |
| **包体积** | 无额外依赖 | 引擎本身几百 KB~MB |

- 几百种 × 每种建模+调贴图 → 这是**持续的美术外包成本**,不是一次性的。

现在的 2D 方案,加一种珠子只要**一张实拍图**(几乎零成本),这是它最大的优势。

## 三、性能与包体积

- Three.js 引擎本身几百 KB~1MB+,微信**单分包 2MB 上限**已经很紧(你现在 1.5MB)。
- 3D 模型 + 高清贴图体积大,几百种珠子的贴图**必须上 CDN 按需加载**,又是一套加载/缓存/降级逻辑。
- 低端安卓机跑 WebGL + 几十颗珠子实时光照,可能卡顿甚至崩,要做大量性能降级。

## 四、维护成本

- 3D 渲染管线比 2D 复杂一个量级,以后谁维护、谁调 bug,门槛高。
- 微信 XR-Frame 还在演进,踩官方 bug 的风险比成熟的 Canvas 2D 高。

## 我的务实结论

**真 3D 移植的总成本 ≈ 你目前整个 diy-lite 从零到现在的好几倍**,而且大部分是**持续的美术素材成本**,不是一锤子买卖。

关键判断依据:**diy2、diy3 这两个成品都没上真 3D**,都用 Canvas 2D 伪 3D。同行都这么选,说明对手串这个品类,伪 3D 的投入产出比明显更优——手串是平面圆环,珠子实拍图本身质感就好,不像游戏角色非真 3D 不可。

**建议**:除非你的核心卖点就是"极致的珠子通透质感 / 任意角度把玩"(比如高客单价的定制珠宝),否则不值得上真 3D。现在的伪 3D 继续打磨(比如给高光加一点随角度变化)性价比高得多。
