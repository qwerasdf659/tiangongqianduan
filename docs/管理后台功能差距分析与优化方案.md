## 一、项目概况（基于实际代码与数据库）

### 1.1 商业模式

**餐厅积分抽奖系统 V4.0** — 一个围绕餐饮消费的积分忠诚度与游戏化平台。

核心业务闭环：

```
用户到店消费 → 扫码提交消费记录 → 审核通过获得 POINTS
    → 消耗 POINTS 参与抽奖 → 获得物品（优惠券/收藏品）+ 材料资产（red_shard 等）
        → 材料资产可在「兑换市场」B2C 兑换实物/虚拟商品
        → 物品/材料可在「交易市场」P2P 挂牌出售（以 DIAMOND 结算）
        → 材料资产可通过「转换规则」分解为 DIAMOND
```

## 二、兑换市场管理现状与差距

### 2.1 当前实现（基于真实代码与数据）

**数据库现状（2026-03-17 直连数据库验证）：**
- `exchange_items` 表：仅 2 个商品（"衣服" cost:10 red_shard, stock:49, sold:1 / "宝石1" cost:100 red_shard, stock:299, sold:1）
- `exchange_records` 表：3 笔订单 —— **1 笔 completed（#417）、1 笔 refunded（#418）、1 笔 pending（#554）**
- `exchange_order_events` 表：8 条事件，完整记录了 pending→approved→shipped→completed 和 pending→approved→refunded 两条完整生命周期
- 商品全部在 `lucky` 空间，rarity 全部为 `common`

**exchange_records 状态流转已验证的真实数据：**
```
订单 #417（user_id:31, 宝石1）：pending → approved(03-17 04:41:30) → shipped(04:41:40) → completed(04:41:40) ✅ 完整履约
订单 #418（user_id:31, 宝石1）：pending → approved(03-17 04:41:49) → refunded(04:41:49, 原因:用户申请退款，商品缺货) ✅ 退款成功
订单 #554（user_id:32, 衣服）：pending(03-15 06:42:21) ← 等待审核
```

**已有字段能力（exchange_records 表，26 个字段）：**
```

shipped_at / received_at / approved_at / rejected_at / refunded_at  ← approved_at/shipped_at/refunded_at 已有真实数据
rating / rated_at  ← 评价字段（CoreService.rateOrder() 已实现，1-5 分）
```

**已有管理后台页面：**

**已有 API 能力（admin/src/api/market/exchange.js + routes/v4/console/marketplace.js）：**
- 商品 CRUD（列表/详情/创建/编辑/删除/状态切换）
- 订单列表/详情 + **审核通过/发货/确认完成/驳回/退款**（全部已实现）
- 统计趋势、缺图检测、批量绑定图片
- 用户挂牌统计、用户挂牌列表、用户挂牌上限配置
- 市场概览统计、价格历史

### 2.2 关键差距分析

#### ~~差距 1：兑换订单履约流程不完整~~ → ✅ 已解决（2026-03-17 代码验证）

**现状（v3.0 更新）：** 后端 `ExchangeCoreService` 已实现完整状态机（`ORDER_STATUS_TRANSITIONS`），支持 approve/ship/receive/complete/reject/refund/cancel 全部 7 种状态转换。管理后台路由 `POST /exchange_market/orders/:order_no/{approve|ship|refund|complete|reject}` 已全部注册。前端 `exchange-orders.js` composable 已实现全部操作方法。

**已实现的管理操作（真实数据库已验证）：**
- ✅ 审核通过（pending → approved）— 订单 #417、#418 已走通
- ✅ 发货操作（approved → shipped）— 订单 #417 已走通
- ✅ 完成订单（shipped → completed）— 订单 #417 已走通
- ✅ 退款（approved/shipped → refunded）— 订单 #418 已走通，材料资产已回退
- ✅ 驳回（pending → rejected）
- ✅ 用户取消（pending → cancelled）— 自动回退材料
- ✅ 用户确认收货（shipped → received）
- ✅ 用户评价（received/completed/shipped → rated，1-5分）
- ✅ 超时自动确认收货（`autoConfirmShippedOrders()`，发货 7 天后自动确认）

#### 差距 2：商品管理深度不足

**现状：** exchange_items 表结构是扁平的单品模式，每个商品只有一个 cost_amount，没有规格/SKU 概念。

**缺失能力：**
- 商品只有 2 个，无分类筛选（category_defs 有 9 个分类但商品未充分利用）
- 无 SKU/规格管理（如同一商品不同颜色/尺寸）
- 无库存预警（stock 字段存在但无告警机制）
- 无商品详情富文本（description 字段为空字符串）
- 无批量上下架/改价
- 标签字段（is_new/is_hot/is_lucky/is_limited/tags）全部为默认值，未被运营使用

#### 差距 3：缺少售后退款机制

**现状：** exchange_records 表有 `refunded_at` 字段但无对应的退款流程。用户兑换后如果商品有问题，没有退款/退货通道。

## 三、交易市场管理现状与差距

### 3.1 当前实现（基于真实代码与数据）

**业务模式：P2P 用户间交易** — 卖家挂牌物品或可替代资产，买家用 DIAMOND 购买，平台收取 5% 手续费。

**数据库现状：**

| 维度 | 数据 |
|------|------|
| 市场挂牌 | 313 条（on_sale:23 / sold:21 / withdrawn:269） |
| 交易订单 | 148 笔（completed:22 / cancelled:126） |
| 完成率 | 14.9%（22/148），85.1% 的订单被取消 |
| 取消原因 | 主要为「订单超时自动取消（锁定超过15分钟）」 |
| 挂牌类型 | listing_kind 字段：item（物品）+ fungible_asset（可替代资产） |
| 结算货币 | DIAMOND，手续费率 5% |
| 竞价系统 | bid_products:0 / bid_records:0（已建表但无数据） |

**交易流程（已实现）：**
```
卖家挂牌(on_sale) → 买家下单锁定(locked) → 冻结买家 DIAMOND(frozen)
    → 结算：买家扣款 / 卖家收款(net) / 平台收手续费(fee) → completed
    → 或超时15分钟自动取消 → cancelled → 解冻
```

**已有管理后台页面：**
- `trade-management.html`：交易订单 + 兑换订单 + 上架统计 + 市场概览（4 Tab）
- `bid-management.html`：竞价管理（7 态状态机，但无真实数据）
- `asset-management.html`：材料资产类型/账户/交易/物品/虚拟账户/统计（6 Tab）
- `item-lifecycle.html`：物品全生命周期追踪
- `orphan-frozen.html`：孤儿冻结检测与清理

**已有 API 能力（admin/src/api/market/trade.js）：**
- 交易订单列表/详情/统计
- 挂牌列表/CRUD/强制下架
- 孤儿冻结检测与清理

### 3.2 关键差距分析

#### 差距 1：交易订单取消率过高（85%），缺少运营干预工具

**现状：** 126 笔取消 vs 22 笔完成。大量订单因 15 分钟超时被自动取消，说明买家下单后未及时完成支付/确认。

**缺失能力：**
- 超时时间可配置化（当前硬编码 15 分钟）
- 取消原因分类统计与分析看板
- 高取消率用户预警
- 买家信用评分/交易限制

#### 差距 2：挂牌撤回率过高（269/313 = 86%）

**现状：** 269 条挂牌被撤回，尤其是 fungible_asset 类型（262 条撤回），说明卖家频繁挂牌又撤回。

**缺失能力：**
- 撤回频率限制（防刷单）
- 挂牌质量管控（最低价格/最高价格限制）
- 市场价格异常检测

#### 差距 3：竞价系统空转

**现状：** bid_products 和 bid_records 表均为空，管理后台已实现完整的竞价管理页面（7 态状态机），但无真实业务数据。

#### 差距 4：缺少交易纠纷处理

**现状：** P2P 交易完成后，如果买家对物品不满意，没有申诉/仲裁通道。系统有 SYSTEM_ESCROW 账户但未被使用。

## 四、统一账本与财务管理现状与差距

**缺失能力与方案：**

#### 5.2.1 批量操作（淘宝「批量管理」）
- 批量上下架：勾选多个商品一键切换 active/inactive

- 库存预警阈值配置：stock 低于 N 时自动告警
- 库存变动日志：谁在什么时候调整了库存，调整了多少

- 设置商品定时上架时间（如活动开始时自动上架）
- 设置商品定时下架时间（如活动结束时自动下架）
- 需要新增字段：`publish_at` / `unpublish_at`
- 需要新增定时任务（复用现有 jobs 框架）

#### 5.2.4 商品排序与推荐控制（淘宝「商品排序」）
- 当前只有 sort_order 单一排序字段
- 增加：置顶功能（一键置顶到最前）
- 增加：按销量/按上架时间/按价格 多维排序
- 增加：推荐位管理（首页推荐、分类页推荐）

#### 5.2.5 商品数据看板（京东「商品分析」）
- 单品数据：浏览量、兑换量、兑换转化率、库存周转
- 商品排行：兑换量 Top N、浏览量 Top N
- 趋势图：按日/周的兑换趋势
- 当前 stats Tab 有统计分析，但需要增加单品维度的数据下钻

#### 5.2.6 商品详情增强
- description 支持富文本编辑（当前是纯文本，2 个商品都为空）
- 商品视频上传支持
- 商品参数表（如尺寸、材质、产地等结构化属性）
- 需要新增字段：`video_url` / `attributes` (JSON)

#### 5.2.7 多 SKU / 规格管理

**现状：** 当前 `exchange_items` 是单品模式（1 商品 = 1 SKU），没有 SKU 相关表。兑换时 `exchange_records.item_snapshot` 只记录了商品名和价格，没有规格信息。

**方案：引入 SPU + SKU 两层模型**

现有 `exchange_items` 表升级为 SPU（商品主体），新增 `exchange_item_skus` 表作为 SKU（规格变体）。

**数据模型设计：**

**全量 SKU 模式（审查优化 #2）：**

> 原方案使用 `has_skus` 字段区分单品/多SKU 两种模式，导致库存扣减逻辑分裂。
> 优化后：所有商品统一走 SKU 表，单品商品自动创建一个默认 SKU（spec_values 为空）。
> 项目未上线，现有 2 个商品迁移成本为零。

- **取消 `has_skus` 字段**，所有商品都有至少一个 SKU
- 单品商品：自动创建默认 SKU（`spec_values: {}`，价格和库存从 SPU 复制）

- 所有库存查询和扣减逻辑只有一套，统一走 `exchange_item_skus` 表

## 七、与传统电商后台的本质差异

**重要提醒：** 本项目不是淘宝/京东式的传统电商平台，不应盲目照搬。

| 维度 | 淘宝/京东 | 本项目 |
|------|-----------|--------|
| 商业模式 | 多商家入驻电商 | 单商户餐饮积分忠诚度平台 |
| 商品来源 | 多商家上架 | 平台官方配置（B2C 兑换） |
| 支付方式 | 人民币 | 虚拟资产（red_shard / DIAMOND） |
| 交易模式 | B2C 为主 | B2C 兑换 + P2P 交易并存 |
| 物流 | 实物物流为核心 | 虚拟物品为主，少量实物 |
| 商户数量 | 数万+ | 当前 1 个 |
| SKU 复杂度 | 极高（规格/颜色/尺寸） | 低（单品模式） |
| 结算 | 复杂多方分账 | 简单平台手续费 |

**因此，以下传统电商功能对本项目优先级较低或不适用：**
- ~~SPU/SKU 多规格管理~~ → **已纳入 P0 实施（见 5.2.7 + 6.1 更新）**
- ~~多级类目树~~ → **改为两级分类 + 标签混合方案（见 8.5）**
- ~~物流追踪对接~~ → **改为轻量快递查询方案（见 8.6）**
- 发票管理 → 虚拟资产交易不涉及
- 多商家结算/分账 → 当前单商户
- 传统优惠券/满减 → 不适用于材料资产兑换模式

### 8.1 交易市场挂牌排序控制

#### 8.1.1 现状分析

**数据库：** `market_listings` 表没有 `sort_order` 字段，无运营排序能力。

**API：** `services/market-listing/QueryService.js` 仅支持 3 种固定排序：
```javascript
switch (sort) {
  case 'price_asc':  order = [['price_amount', 'ASC']]; break
  case 'price_desc': order = [['price_amount', 'DESC']]; break
  case 'newest':
  default:           order = [['created_at', 'DESC']]
}
```

**管理后台：** `trade-management.html` 只有订单查看和强制下架，无排序干预。

**用户端：** 排序完全由算法决定，运营无法置顶优质挂牌或推荐商品。

#### 8.1.2 方案

**1. 数据库改动 — `market_listings` 表新增字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| sort_order | INT DEFAULT 0 | 运营排序权重，数值越小越靠前（与兑换商品、活动排序方向统一） |
| is_pinned | TINYINT(1) DEFAULT 0 | 是否置顶（置顶的始终排在最前） |
| is_recommended | TINYINT(1) DEFAULT 0 | 是否推荐（可用于前端"推荐"标签展示） |
| pinned_at | DATETIME NULL | 置顶时间（多个置顶时按此排序） |

> **设计规范（审查优化 #1）：** 全项目排序字段统一为 `sort_order`（INT，越小越前）+ `is_pinned` + `pinned_at`，三个模块（兑换商品、交易市场、活动）使用同一套排序模式，避免运营混乱。

**2. 排序逻辑改造 — `QueryService.js`：**

```javascript
// 改造后的排序逻辑
let order = []

// 第一优先级：置顶（is_pinned DESC, pinned_at DESC）
order.push(['is_pinned', 'DESC'], ['pinned_at', 'DESC'])

// 第二优先级：运营排序
order.push(['sort_order', 'ASC'])

// 第三优先级：用户选择的排序方式
switch (sort) {
  case 'price_asc':  order.push(['price_amount', 'ASC']); break
  case 'price_desc': order.push(['price_amount', 'DESC']); break
  case 'newest':
  default:           order.push(['created_at', 'DESC'])
}
```

**3. 管理后台改造 — `trade-management.html` 新增「挂牌管理」操作：**

- 「置顶」按钮：一键置顶/取消置顶某条挂牌
- 「推荐」按钮：标记/取消推荐
- 「调整排序」：输入框修改 sort_order
- 批量操作：勾选多条挂牌批量置顶/取消/调整排序

**4. 后端 API 新增（`routes/v4/console/`）：**

| 端点 | 方法 | 功能 |
|------|------|------|
| `/console/market-listings/:id/pin` | PUT | 置顶/取消置顶 |
| `/console/market-listings/:id/recommend` | PUT | 推荐/取消推荐 |
| `/console/market-listings/:id/sort-order` | PUT | 修改排序值 |
| `/console/market-listings/batch-sort` | PUT | 批量调整排序 |

**5. 用户端排序增强 — 新增排序选项：**

| 排序选项 | 说明 |
|----------|------|
| `recommended`（新增） | 综合推荐排序（置顶 > 权重 > 时间），作为新的默认排序 |
| `newest` | 最新上架（保留） |
| `price_asc` | 价格从低到高（保留） |
| `price_desc` | 价格从高到低（保留） |
| `hot`（新增） | 按浏览量/成交量排序（需新增 view_count 字段或从 trade_orders 统计） |

### 8.5 商品分类体系升级：两级分类 + 标签混合方案

#### 8.5.1 行业方案调研与对比

| 方案 | 代表平台 | 核心思路 | 复杂度 | 查询性能 |
|------|---------|---------|--------|---------|
| A. 扁平分类 | 招行积分商城、航司里程商城 | 单层列表，商品直接挂分类 | 低 | 高 |
| B. 两级固定分类 | 美团、腾讯虚拟物品、闲鱼 | parent_id 自引用，限制 2 层 | 低 | 高 |
| C. 多级类目树 | 淘宝/京东 | parent_id 无限递归，3~5 层 | 高 | 中 |
| D. 分类+标签混合 | 美团商户、Steam | 固定分类做主维度，标签做辅助筛选 | 中 | 高 |
| E. 游戏模板+实例 | 暴雪/米哈游/网易 | item_template 定义原型，item_instance 记录持有 | 高 | 低 |

#### 8.5.2 各方案适配性分析

**A. 扁平分类（当前方案）：**
- 现状：`category_defs` 表 5~9 个分类，零改动
- 问题：商品量增长后筛选粒度不够（"数码电子"下 iPhone/耳机/手环 混在一起）
- 适合：商品 < 50 个

**B. 两级固定分类：**
- 改动：`category_defs` 加 `parent_code` 字段
- 优点：筛选粒度够用，查询只多一次 JOIN，Sequelize 原生支持
- 适合：商品 50~500 个

**C. 多级类目树（淘宝/京东）：**
- 不选理由：Sequelize 不原生支持递归 CTE，需要手写 SQL 或引入 closure table（额外 2 张表）。项目 1 个商户、几十~几百个商品，完全过度工程
- 适合：商品 > 10000，品类极其丰富

**D. 分类+标签混合：**
- 现状：项目已经在用！`category` 是固定分类，`tags` JSON / `rarity_defs` / `space` 都是标签维度
- 只需规范化现有标签体系

**E. 游戏模板+实例：**
- 现状：项目已经是这个模式（`item_templates` + `items` 表）
- 它解决的是物品生命周期问题，不是分类问题，分类仍需 A~D 中选一个

#### 8.5.3 推荐方案：B + D 混合（两级分类 + 标签增强）

**选型理由：**

| 考量 | 分析 |
|------|------|
| 业务匹配 | 积分兑换 + P2P 交易平台，最接近「积分商城 + 游戏交易」，两级分类是这类平台的标配 |
| 技术适配 | `category_defs` 只需加一个 `parent_code` 字段，Sequelize 关联改动极小 |
| 长期维护 | 两级分类结构清晰，不需要递归查询和 closure table |
| 标签已有 | `tags` JSON、`rarity_defs`、`space` 已提供多维度筛选，不需要额外建表 |
| 不过度设计 | 多级类目树对几十~几百个商品的体量是过度工程 |

#### 8.5.4 数据库改动

**`category_defs` 表新增字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| parent_code | VARCHAR(50) NULL | 父分类代码，NULL 表示一级分类。FK → category_defs.category_code |
| level | TINYINT NOT NULL DEFAULT 1 | 层级（1=一级，2=二级），应用层限制最多 2 级（不设数据库 CHECK 约束，保留未来扩展到 3 级的灵活性） |

> **设计规范（审查优化 #3）：** 层级限制通过应用层业务规则校验，不在数据库加 CHECK 约束。这样未来如果需要 3 级分类，只需改业务规则，不用改数据库结构。

#### 8.5.5 Sequelize 模型改造

**`models/CategoryDef.js` 新增：**

```javascript
// 新增字段
parent_code: {
  type: DataTypes.STRING(50),
  allowNull: true,
  comment: '父分类代码，NULL=一级分类',
  references: { model: 'category_defs', key: 'category_code' },
  onUpdate: 'CASCADE',
  onDelete: 'SET NULL'
},
level: {
  type: DataTypes.TINYINT,
  allowNull: false,
  defaultValue: 1,
  comment: '层级（1=一级，2=二级）',
  validate: { isIn: [[1, 2]] }
}

// 新增关联（自引用）
static associate(models) {
  // 父分类
  CategoryDef.belongsTo(models.CategoryDef, {
    foreignKey: 'parent_code',
    targetKey: 'category_code',
    as: 'parent'
  })
  // 子分类
  CategoryDef.hasMany(models.CategoryDef, {
    foreignKey: 'parent_code',
    sourceKey: 'category_code',
    as: 'children'
  })
}

// 新增静态方法
static async getTree() {
  const all = await this.findAll({
    where: { is_enabled: true },
    order: [['level', 'ASC'], ['sort_order', 'ASC']]
  })
  const top = all.filter(c => c.level === 1)
  return top.map(t => ({
    ...t.toJSON(),
    children: all.filter(c => c.parent_code === t.category_code)
  }))
}
```

#### 8.5.6 查询方式（零递归）

```javascript
// 1. 获取分类树（前端下拉/筛选用）
const tree = await CategoryDef.getTree()
// 返回：[{ category_code: 'electronics', children: [{...phone}, {...headphone}] }, ...]

// 2. 按一级分类查商品（包含所有子分类）
const subCodes = await CategoryDef.findAll({
  where: { parent_code: 'electronics' },
  attributes: ['category_code'],
  raw: true
})
const allCodes = ['electronics', ...subCodes.map(c => c.category_code)]
const items = await ExchangeItem.findAll({
  where: { category: { [Op.in]: allCodes } }
})

// 3. 按二级分类精确查商品
const items = await ExchangeItem.findAll({
  where: { category: 'phone' }
})
```

不需要递归 CTE，不需要 closure table，Sequelize 原生支持，查询性能和现在一样。

### 8.6 轻量快递查询方案（快递100 + 快递鸟双通道）

#### 8.6.1 现状分析

**`exchange_records` 表现有字段：**
- `shipped_at`：发货时间（已有）
- `admin_remark`：管理员备注（已有，当前用于手动记录快递信息）
- 无快递公司字段、无快递单号字段

**问题：** 管理员发货后只能在 `admin_remark` 里手写快递信息，用户无法查看物流轨迹。

#### 8.6.2 方案概述

同时对接快递100和快递鸟两个平台，通过抽象层统一调用，互为备用：

```
管理后台发货弹窗
  → 选择快递公司 + 填入快递单号
  → 保存到 exchange_records

管理后台/用户端 点击「查询物流」
  → 后端 ShippingTrackService（抽象层）
      → 优先调用快递100 API
      → 快递100 失败/超时 → 自动降级到快递鸟 API
  → 返回统一格式的物流轨迹
  → 前端展示时间线
```

#### 8.6.3 快递100 vs 快递鸟 对比

| 维度 | 快递100 | 快递鸟 |
|------|--------|--------|
| 支持快递公司 | 1000+ | 400+ |
| 免费额度 | 100次/天 | 500次/天 |
| 单号自动识别 | 支持 | 支持 |
| 推送回调 | 收费 | 免费版支持 |
| Node.js SDK | 官方 npm 包 | 需自行封装 HTTP |
| 文档质量 | 好 | 一般 |
| 定位 | 主通道（SDK 成熟） | 备用通道（免费额度大） |

**双通道策略：**
- 快递100 作为主通道：有官方 Node.js SDK，对接最简单
- 快递鸟作为备用通道：免费额度更大（500次/天），快递100 故障时自动切换
- 未来如果订单量增长，可启用快递鸟的免费推送回调，减少主动查询次数

#### 8.6.4 数据库改动

**`exchange_records` 表新增字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| shipping_company | VARCHAR(50) NULL | 快递公司代码（如 sf/yt/zt/yto） |
| shipping_company_name | VARCHAR(100) NULL | 快递公司名称（如 顺丰速运） |
| shipping_no | VARCHAR(100) NULL | 快递单号 |

#### 8.6.5 后端架构 — ShippingTrackService（抽象层）

```javascript

  constructor() {
    this.providers = [
      new Kuaidi100Provider(config.kuaidi100),  // 主通道
      new KdniaoProvider(config.kdniao)          // 备用通道

  }

  /**
   * 查询物流轨迹（自动降级）
   * @param {string} shippingNo - 快递单号
   * @param {string} companyCode - 快递公司代码
   * @returns {Object} 统一格式的物流轨迹
   */
  async queryTrack(shippingNo, companyCode) {
    for (const provider of this.providers) {
      try {
        const result = await provider.query(shippingNo, companyCode)
        if (result.success) return result
      } catch (err) {
        logger.warn(`${provider.name} 查询失败，尝试下一个`, { err })
      }
    }
    return { success: false, message: '所有快递查询通道均不可用' }
  }
}
```

### 9.5 优化 #7：API 端点命名统一

**问题**：现有 API 端点命名风格不完全一致，新增端点应遵循统一规范。

**命名规范**：

```
管理后台端点：/api/v4/console/{resource}
用户端端点：  /api/v4/shop/{module}/{resource}

排序相关端点统一格式：
  PATCH  /{resource}/:id/pin      → 置顶/取消置顶
  PUT    /{resource}/reorder       → 批量排序
  PATCH  /{resource}/:id/sort     → 单个排序值修改
```

**要点**：
- 资源名使用复数形式（`items` 不是 `item`）
- 动作用 HTTP 方法表达，URL 中不出现动词（`POST /ship` 例外，因为是子资源操作）
- 查询参数统一：`page` / `page_size` / `sort_by` / `sort_order`

**优先级**：贯穿所有开发阶段，作为编码规范执行

### 10.9 决策项：行业方案对比与推荐（基于真实数据库验证）

> 以下 6 项决策，每项均从大厂、中小公司、游戏公司、虚拟物品交易平台、二手平台等多个行业角度对比分析，结合本项目 **"餐饮积分忠诚度 + 游戏化 + P2P 交易"** 的独特定位给出推荐。
>
> 决策原则：项目未上线、不兼容旧接口、一次性投入换长期低维护成本、降低技术债务。

---

#### 决策 1：SKU 模式选择

**行业方案对比：**

| 类型 | 代表平台 | SKU 设计 | 理由 |
|------|---------|---------|------|
| **综合电商** | 淘宝/京东/拼多多 | 全量 SPU+SKU，单品也有默认 SKU | 百万级商品，规格组合无穷，必须统一模型 |
| **积分商城** | 招行积分/航司里程/京东金豆 | **无 SKU**，单品扁平模式 | 商品量 50~300，以虚拟卡券为主，不需要尺码颜色 |
| **游戏道具商城** | 米哈游/暴雪/网易 | **无 SKU**，每个道具是独立 item_template | 虚拟物品天然无规格概念 |
| **餐饮 SaaS** | 美团商户版/大众点评商户 | **简化规格**：菜品+口味/份量，不走 SKU 表 | 餐饮场景规格有限（大杯/中杯/小杯） |
| **虚拟物品交易** | Steam/5173/dd373 | **无 SKU** | 虚拟物品唯一标识就是 item_id |

**本项目实际情况（基于真实数据库）：**

**推荐：按 5.2.7 + 9.1 优化 #2 方案落地 — 全量 SKU 模式**

| 考量 | 分析 |
|------|------|
| 原方案已确认 | 5.2.7 完整设计 + 9.1 优化 #2 升级为全量模式 + 6.1 列为 P0 |

#### 决策 7：排序字段方案 — 是否三个模块同批加字段？

**行业对比（按公司类型分类）：**

| 类型 | 代表 | 排序字段设计 | 何时加？ | 是否统一模式？ |
|------|------|------------|---------|--------------|
| **大型电商** | 淘宝/天猫 | 每个内容模块统一 `rank_score` + `is_sticky`（置顶）+ `quality_score`，schema 设计时一步到位 | 建表时就带 | ✅ 全站统一排序字段名和语义 |
| **大型电商** | 京东 | 商品 `sort_order`，活动 `display_order`，促销 `priority` — 命名不统一导致长期维护痛点 | 各模块独立加，不同时期 | ❌ 不统一，京东内部公认的技术债 |
| **本地生活** | 美团 | 商家/菜品/活动/Banner 统一 `rank` + `is_top`（置顶），建表时标配 | 建表时就带 | ✅ 统一 |
| **本地生活** | 大众点评 | 同美团体系 | 同上 | ✅ 统一 |
| **社交/内容** | 腾讯（微信/QQ） | 内容模块共用 `display_config` JSON blob（包含 order/pinned/featured） | 首版就设计 | ✅ 统一但用 JSON 而非独立字段 |
| **游戏商城** | 米哈游（原神/星铁） | 游戏内商店/活动 Banner/公告统一 `display_priority` INT + `is_featured` BOOL + 时间窗口 | 建表时标配 | ✅ 统一 |
| **游戏商城** | 暴雪/网易 | 同上，商城物品 `sort_order` + 活动 `priority` + 推荐位 `featured_slot` | 建表时标配 | ✅ 统一 |
| **游戏交易** | Steam 市场 | **无管理员排序字段**，纯算法排序（最新/价格/热门） | 不加 | N/A（P2P 不需要运营排序） |
| **游戏交易** | 5173/dd373 | 同 Steam，纯算法排序 + 付费置顶（`boost_expire_time`） | 部分加 | 🟡 部分统一 |
| **积分商城** | 招行积分/航司里程 | 商品 `sort_order` + `is_recommended`，活动 `display_order`，字段名不一定统一但语义一致 | 建表时就带 | 🟡 语义统一但字段名不同 |
| **二手平台** | 闲鱼 | 算法排序为主，`boost`（付费曝光）为辅，无管理员排序 | 不加 | N/A |
| **二手平台** | 转转 | 同闲鱼 | 不加 | N/A |
| **小公司/创业** | 常见模式 | 模块 A 加了 `sort_order`，模块 B 用 `display_order`，模块 C 用 `priority` — 每次新需求临时加，字段名/语义全不统一 | 需要时临时加 | ❌ 不统一，是最常见的技术债来源 |

**核心规律：**
- **大厂（阿里/美团/米哈游）**：建表时就带排序字段，全站统一字段名和语义
- **京东/部分中厂**：不同时期加，字段名不统一，事后承认是技术债
- **小公司**：临时加，最终变成每个表排序字段名不一样的混乱局面
- **P2P 交易平台（Steam/5173/闲鱼）**：不需要管理员排序，用户生成内容靠算法排序
- **B2C 平台（本项目的兑换市场）**：必须有管理员排序控制

**本项目定位分析：**
- 兑换市场是 **B2C 官方控制**（平台配置商品）→ 必须有管理员排序
- 交易市场是 **P2P 用户生成**（用户挂牌）→ 排序需求弱于兑换，但运营仍需置顶/推荐能力
- 活动是 **平台运营控制** → 必须有展示排序
- 三个模块的排序需求层次不同，但字段模式应统一

**拍板结论：一次性全量 migration + 分批开发 UI。**

| 决定 | 理由 |
|------|------|
| **数据库字段：一个 migration 文件，三表同批加完** | 项目未上线，零数据迁移风险。ALTER TABLE ADD COLUMN 是最轻量的数据库操作。大厂（阿里/美团/米哈游）的标准做法就是建表时统一带上排序字段。小公司临时加的做法是公认的技术债来源 |
| **字段名全项目统一：`sort_order` INT ASC + `is_pinned` TINYINT + `pinned_at` DATETIME + `is_recommended` TINYINT** | 京东的教训：`sort_order`/`display_order`/`priority` 混用导致运营混乱。统一字段名 = 统一 SQL 查询模式 = 统一前端 composable |
| **服务层和 UI：分批开发** | 字段加了不代表功能就上线了，字段默认 0/NULL 不影响现有逻辑。先做兑换商品排序（已有 `sort_order` 基础最小增量），再做活动排序，最后做挂牌排序 |
| **查询逻辑统一模式：`ORDER BY is_pinned DESC, pinned_at DESC, sort_order ASC, created_at DESC`** | 三个模块的 QueryService 用同一套排序 SQL 模板，减少维护成本 |

**Migration 内容（一个文件）：**
```sql
-- exchange_items（已有 sort_order，加 3 个字段）
ALTER TABLE exchange_items
  ADD COLUMN is_pinned TINYINT(1) NOT NULL DEFAULT 0,
  ADD COLUMN pinned_at DATETIME NULL,
  ADD COLUMN is_recommended TINYINT(1) NOT NULL DEFAULT 0;

-- market_listings（加 4 个字段）
ALTER TABLE market_listings
  ADD COLUMN sort_order INT NOT NULL DEFAULT 0,
  ADD COLUMN is_pinned TINYINT(1) NOT NULL DEFAULT 0,
  ADD COLUMN pinned_at DATETIME NULL,
  ADD COLUMN is_recommended TINYINT(1) NOT NULL DEFAULT 0;

-- lottery_campaigns（加 6 个字段）
ALTER TABLE lottery_campaigns
  ADD COLUMN sort_order INT NOT NULL DEFAULT 0,
  ADD COLUMN is_featured TINYINT(1) NOT NULL DEFAULT 0,
  ADD COLUMN is_hidden TINYINT(1) NOT NULL DEFAULT 0,
  ADD COLUMN display_tags JSON NULL,
  ADD COLUMN display_start_time DATETIME NULL,
  ADD COLUMN display_end_time DATETIME NULL;
```

> 注：`lottery_campaigns` 用 `is_featured` 而非 `is_recommended`，因为活动场景下 "精选" 比 "推荐" 更符合语义。字段语义统一但命名可以根据业务语境微调。

| **游戏内购** | 米哈游/暴雪/网易 | **严格不退**（除法律要求） | 苹果/谷歌渠道退款会封号 | 终身限 1-2 次误购退款 |
| **游戏道具** | Steam | **2 小时/14 天退款窗口** | 月限 3-5 次，超过被标记 | 退款率 > 20% 触发账号审查 |
| **游戏交易** | 5173/dd373 | **确认收货后不退**（P2P 最终性） | 争议走仲裁，不走退款 | N/A |
| **二手交易** | 闲鱼 | **确认后不退**，争议走客服仲裁 | N/A | N/A |
| **虚拟卡券** | 美团/饿了么 | 未使用可退，已使用不退 | 退款后券立即失效 | 无月限，但已使用的绝不退 |
| **潮玩交易** | 得物/StockX | 确认收货后不退（除假货） | 假货退款：鉴定 → 退款 → 卖家封号 | 仅假货场景可退 |

**核心规律：**
1. **虚拟物品行业（游戏/积分商城/卡券）**：退款极严格，通常 **不允许** 或 **仅管理员可操作**
2. **实物电商（淘宝/京东）**：法律要求 7 天无理由，但有风控防刷
3. **P2P 交易（闲鱼/5173/Steam）**：确认收货后 **不可退**，争议走仲裁
4. **本质区别**：虚拟物品退款 = 资产回退（无物流成本），实物退款 = 物流+退货+质检（高成本）

**本项目实际情况：**
- 兑换市场是 **B2C 平台官方兑换**（用户消耗 red_shard 换商品）
- 退款 = 管理员操作（`CoreService.refundOrder()`，需 `requireRoleLevel(100)`）
- 用户 **不能** 自助发起退款（没有用户端退款接口）
- 退款实质是 **资产回退**（red_shard 回到用户账户）+ **库存恢复**
- 最接近的行业模型是 **招行积分商城**（管理员审核退款）+ **米哈游游戏内购**（极严格退款）

**拍板结论：配置驱动三层防护，初始全部关闭，上线后按需开启。**

| 层级 | 规则 | 初始值（上线时） | 建议调整时机 | `system_configs` key |
|------|------|-----------------|------------|---------------------|
| **第一层：冷却期** | 同一用户同一商品，退款后 N 小时内不可再次兑换 | **0**（关闭，因为退款是管理员操作，不存在用户刷退） | 若开放用户自助退款，设为 24h | `refund_cooldown_hours` |
| **第二层：月限** | 单用户每月退款上限 N 次 | **0**（关闭，不限，同上理由） | 若开放用户自助退款，设为 5 次/月 | `refund_monthly_limit` |
| **第三层：大额审批** | 退款金额 > N 材料时需要二次审批 | **0**（关闭，管理员操作本身就是审批行为） | 若多管理员协作，设为 500 材料 | `refund_approval_threshold` |

**为什么初始全部关闭？**

| 考量 | 分析 |
|------|------|
| 当前退款是管理员操作 | `requireRoleLevel(100)` 已是权限保护，管理员本身就是"审批人"，加规则 = 管理员审批自己 |
| 积分商城行业惯例 | 招行积分/航司里程的退款本就是管理员人工审核，不需要额外防刷规则 |
| 游戏行业惯例 | 游戏内购退款几乎不存在，防刷规则更无意义 |
| 配置驱动 = 零代码调整 | 所有阈值存 `system_configs`，未来开放用户自助退款时改配置即可，不改代码 |
| 审计已有 | `exchange_order_events` + `asset_transactions` 完整记录退款操作，可事后追溯 |

**技术实现（复用已有 `system_configs` + `SystemSettingsService`）：**
```javascript
// 退款前检查（CoreService.refundOrder 中调用）
async function checkRefundRules(userId, itemId) {
  const cooldown = await SystemSettingsService.get('refund_cooldown_hours', 0)
  const monthlyLimit = await SystemSettingsService.get('refund_monthly_limit', 0)
  const approvalThreshold = await SystemSettingsService.get('refund_approval_threshold', 0)
  
  if (cooldown > 0) { /* Redis key 检查冷却期 */ }
  if (monthlyLimit > 0) { /* 查 exchange_order_events 本月退款次数 */ }
  if (approvalThreshold > 0) { /* 金额 > 阈值时走审批链 */ }
}
```

> **与京东对比**：京东 ¥200 以下自动审批、¥2000+ 人工审核。但京东是实物电商有退货物流成本。本项目是虚拟资产回退，无物流成本，管理员直接操作更高效。

#### 决策 9：实施优先级 — Phase 顺序和 migration 策略

**行业对比（新产品上线实施顺序）：**

| 类型 | 代表 | 上线前实施顺序 | 关键原则 |
|------|------|--------------|---------|
| **大型电商** | 淘宝（新品类上线） | 商品目录(SPU/SKU) → 分类/搜索 → 排序/推荐 → 物流 → 售后 | 先有货再有展示再有履约 |
| **大型电商** | 京东（新频道上线） | 同上 | 同上 |
| **本地生活** | 美团（新城市上线） | 商家入驻 → 菜单/定价 → 排序/推荐 → 配送 | 同上 |
| **游戏商城** | 米哈游（新活动） | 物品定义 → 展示排序 → 交易逻辑 → 结算 | 先定义再展示再交易 |
| **积分商城** | 招行（新品上架） | 商品配置 → 分类/标签 → 排序/推荐 → 兑换/物流 | 同上 |
| **SaaS 创业** | Y Combinator 模式 | MVP 扁平目录 → 排序优化（提升转化率）→ SKU（如需要）→ 物流 | 先跑通再优化再扩展 |
| **游戏交易** | 5173 | 物品挂牌 → 搜索/排序 → 交易 → 结算 | 先有供给再有发现 |

**核心规律：所有行业的上线顺序都是 "目录（有什么卖）→ 发现（怎么找到）→ 履约（怎么拿到）"。**

**但有一个关键区别：数据库 schema 和业务逻辑的开发节奏可以不同。**

| 策略 | 适用场景 | 代表 |
|------|---------|------|
| **Schema 一步到位，代码分批开发** | 未上线项目，零数据迁移风险 | 大厂新项目立项时的标准做法 |
| **Schema 分批加，代码跟着加** | 已上线项目，每次 migration 有风险 | 已上线产品迭代 |
| **Schema 和代码全一起做** | 小团队、紧急上线 | 创业 MVP |

**本项目情况：**
- 未上线，数据库里是测试数据
- 所有需要的字段都是 NULL/DEFAULT 值，加了不影响现有逻辑
- Sequelize migration 支持一个文件多表操作
- `sequelize-cli db:migrate` 已在项目 `package.json` 配置好

**拍板结论：一次性全量 migration → 分批开发业务逻辑。**

**Phase 0（数据库一步到位，约 0.5 天）：**

一个 migration 文件，加完 **所有** 待建表和待加字段：

| 操作 | 目标 | 字段/表 |
|------|------|--------|
| CREATE TABLE | `exchange_item_skus` | 完整建表（见 5.2.7 表结构） |
| ALTER TABLE | `exchange_items` | +`spec_names` JSON, +`min_cost_amount` BIGINT, +`max_cost_amount` BIGINT, +`is_pinned` TINYINT, +`pinned_at` DATETIME, +`is_recommended` TINYINT |
| ALTER TABLE | `market_listings` | +`sort_order` INT, +`is_pinned` TINYINT, +`pinned_at` DATETIME, +`is_recommended` TINYINT |
| ALTER TABLE | `lottery_campaigns` | +`sort_order` INT, +`is_featured` TINYINT, +`is_hidden` TINYINT, +`display_tags` JSON, +`display_start_time` DATETIME, +`display_end_time` DATETIME |
| ALTER TABLE | `category_defs` | +`parent_code` VARCHAR(50) NULL FK自引用, +`level` TINYINT DEFAULT 1 |
| ALTER TABLE | `exchange_records` | +`shipping_company` VARCHAR(50), +`shipping_company_name` VARCHAR(100), +`shipping_no` VARCHAR(100), +INDEX(shipping_no) |
| INSERT | `exchange_item_skus` | 为现有 2 个商品各创建 1 条默认 SKU（`spec_values: '{}'`） |
| INSERT | `system_configs` | 退款防刷 3 个配置项（初始值全 0） |

> **为什么一个 migration？** 项目未上线，不存在"回滚某个字段但保留另一个字段"的场景。一个文件 = 一次 review = 原子操作。如果 migration 失败，`sequelize-cli db:migrate:undo` 整体回滚。

**Phase 1（已完成）：** 兑换订单履约闭环 ✅

**Phase 2（SKU 系统，约 3-4 天）：**
- 后端：`ExchangeItemSku` Model + `ExchangeAdminService` SKU CRUD + `CoreService` 兑换流程改造
- Web 前端：商品编辑增加规格管理区域 + SKU 矩阵表格

- 小程序：分类改两级下拉 + 排序选项新增

**Phase 4（快递+数据清理，约 2-3 天）：**
- 后端：`ShippingTrackService` + 快递100/快递鸟双 Provider + ship 路由改造接收结构化快递信息
- Web 前端：发货弹窗改造（快递公司下拉+单号输入）+ 物流时间线
- 小程序：物流轨迹页
- 数据清理：测试数据清理 + 正式门店创建

**Phase 2/3/4 可以并行吗？**

| 并行方案 | 可行性 | 说明 |
|---------|--------|------|
| Phase 2 和 Phase 3 并行 | ✅ 可以 | SKU 改造和排序改造无代码依赖，不同文件不冲突 |

| 全部并行 | ⚠️ 理论可以但 review 压力大 | 如果有多人协作可以全部并行 |

> 如果只有 1 个开发者：串行 Phase 2 → 3 → 4（约 10 天）
> 如果有 2 个开发者：Phase 2 ∥ Phase 3 → Phase 4（约 6 天）
> 如果有 3 个开发者：Phase 2 ∥ Phase 3 ∥ Phase 4（约 4 天）

#### 三个决策的行业共识总结

| 维度 | 大厂（阿里/美团/米哈游） | 小公司/创业 | 本项目最佳选择 |
|------|------------------------|-----------|--------------|
| 排序字段 | 建表时统一带上 | 需要时临时加（技术债） | **一步到位**（大厂做法） |

| 上线前 migration | 一次性做完所有 schema | 边做边加 | **一次性全量 migration**（大厂新项目做法） |
| 开发节奏 | schema 先行，业务分批 | 全部一起 | **schema 一步到位，业务分批开发**（大厂做法） |

**对本项目后端技术栈的适配性：**

| 技术栈 | 是否支持 | 说明 |
|--------|---------|------|
| Sequelize migration | ✅ | 一个 migration 文件支持多表操作，`sequelize-cli db:migrate` / `db:migrate:undo` 原子执行 |
| `system_configs` + `SystemSettingsService` | ✅ | 退款防刷阈值直接写入 system_configs，零代码调整 |
| ServiceManager 依赖注入 | ✅ | 新增 SKU/排序/物流服务通过 `services.registerService()` 注册 |
| TransactionManager 事务 | ✅ | SKU 库存扣减、退款资产回退均走事务 |

| Redis 缓存 | ✅ | 分类树缓存、物流轨迹缓存、退款冷却期均用 Redis |
| IdempotencyService | ✅ | `CANONICAL_OPERATION_MAP` 已覆盖兑换操作路径，新增 SKU/排序路径映射即可 |

> **结论：项目现有技术栈完全覆盖三个决策的所有需求，不需要引入任何新框架或新依赖（快递100 npm SDK 除外）。**
