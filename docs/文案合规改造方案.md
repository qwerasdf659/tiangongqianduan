## 一、路线B 定性依据：为什么不需要公示固定概率

本方案的法律前提是：本系统并非"固定概率抽奖"，而是"预算内随机发放"，因此不适用"必须公示中奖概率"的监管要求。

- **系统本质**：出奖经两步动态过滤（先按用户预算账户/星石配额过滤奖品池，再在剩余奖品中按档位权重随机），每个用户每次可中奖品池不同，不存在"全局固定概率表"。
- **与付费抽奖的区别**：付费抽奖（如游戏抽卡）须公示概率，因其"花真钱 + 固定概率 + 可博高价值"三者俱全；本系统是"免费积分 + 动态概率 + 赢不超预算"，三者俱不满足。
- **破解方向**：将定性从"抽奖中奖"调整为"消费回馈/权益发放"，射幸性被预算门控锁死，绕开"必须公示固定概率"的死结。
- **如实告知替代公示**：不公示不存在的固定概率，但在规则中说明"回馈在您的消费回馈额度内随机发放，分高/中/低三档（比例 X%/Y%/Z%），实际可得档位受您的消费累计额度影响"。

> ⚠️ 红线：前提是"积分必须真正免费、不可用钱购买"。一旦开放"人民币买积分/买回馈机会"，即被拉回付费抽奖监管，必须公示概率。详细推演见 `抽奖概率公示破解方案.md` 第二至四节。

## 二、核心原则：内外分离，只改"用户可见层"

合规看的是**对用户下发了什么**，不是数据库字段叫什么。监管、法院、用户都接触不到内部字段名（`lottery_draws`、`win_probability`），只能看到界面文案、API 返回内容、用户协议措辞。

因此本次改造遵循**内外分离**原则：

```
数据库/代码层(内部，保持不变)        展示层(用户可见，本次改造)
─────────────────────────        ──────────────────────────
lottery_draws            ──→      "消费回馈记录"
win_probability          ──→      (不下发，已由 DataSanitizer 脱敏)
prize_name               ──→      "回馈好礼"
"恭喜中奖"(通知模板)       ──→      "已为您发放回馈"
```

翻译只发生在四个出口：① API 响应文案、② 通知模板（`NotificationService`）、③ 数据库内的展示文案、④ 小程序前端界面。**数据库 schema、字段名、API 路径、类名变量名一律不动。**

> 行业惯例：星巴克、京东、航空里程等成熟平台的后端字段照样叫 `lottery`/`draw`/`probability`，他们只改对外展示叙事，不改 schema。改字段是"高风险、零合规收益"操作。

### 档位分级对照（"几等奖" → "回馈分级"）

等级可以有，但不能用"一二三等奖/安慰奖"这类有奖销售术语。底层沿用引擎已有的 `reward_tier`（high/mid/low）+ `rarity_code`，**字段不改，只改对外叫法**：

| 内部档位（已有，不改） | ❌ 不要叫 | ✅ 路线B 叫法 |
|---|---|---|
| high（高档） | 一等奖 / 特等奖 | 尊享回馈 / 头号好礼 |
| mid（中档） | 二等奖 / 三等奖 | 优享回馈 / 精选好礼 |
| low（低档） | 安慰奖 / 参与奖 | 常享回馈 / 暖心好礼 |
| fallback（兜底） | 谢谢参与 | 基础礼遇（保证不空手） |

要点：
- 对用户强调"**获得了什么好礼**"（具体星石/碎片/权益），而非"**中了第几等**"。
- 路线B 下每次都是"获得回馈"，**没有"没中/未中奖"概念**（系统本就 100% 必发，天然契合）。
- 稀有度光效（传说/史诗/稀有的颜色视觉）表达"好礼珍稀度"，非"中奖排名"，**可保留**。
- **两层分级要分开**：① 随机回馈档位（尊享/优享/常享/基础礼遇）描述每次虚拟好礼大小；② 会员权益层级（青铜/白银/黄金/钻石）描述可解锁的高价值实物（见第七节）。高价值实物**不进随机档位**。

## 七、高价值实物复合门槛（路线B 的功能前提，非文案）

这是路线B 成立的**实质前提**：只要"碎片固定换高价值实物"还在，改任何文案都无效——碎片会被反向锚定成"提货券/类货币"，射幸性与"以小博大"随之回归。**此项属功能开发，不是文案替换。**

**原则**：高价值实物不以"固定碎片数"直接兑换，而用"VIP等级 + 奖章 + 任务 + 限量"复合门槛发放，打散单一价格锚点，使其定性为"会员尊享权益"而非"虚拟币购物"。

| 兑换方式 | 是否形成币价锚点 | 风险 |
|---|---|---|
| ❌ 固定：100 碎片 = 1 部手机 | 是，碎片变提货券 | 高，代币化 + 射幸回归 |
| ✅ 复合门槛：钻石VIP + 集齐奖章 + 完成任务 + 消耗碎片 | 否，价值无法单一锚定到碎片 | 低，实物变"会员尊享权益" |

**商品分级（结合真实库商品，实测 + 决策 17.4）**：
- 实测 `item_templates` 仅两类：`product` 10 条（参考价 0~5000）、`voucher` 20 条（参考价 0~500），**无 `prop` 类记录**。`item_type` 列实测为 `varchar(50)`（**非枚举**），意味着加 `prop` 无需改表结构、直接写值即可（利于扩展）。
- **判定方式（已定案）**：不用价格阈值反推价值，改为新增显式字段 `item_templates.value_tier enum('low','mid','high')`，由运营配置：
  - `low`（日常物，参考价≤约100）：碎片直兑，风险可接受。
  - `mid`（中档，约100~1000）：轻门槛（VIP 等级）。
  - `high`（高档，>1000 或运营标记，含实测参考价为 0 的精品类）：必须套复合门槛，做成"会员成就权益"。

**待核查/落地要点（实测更正）**：
- 用户等级体系**实测确认存在**：`user_growth_levels`、`user_premium_status` 两表均在库中，可承载"VIP门槛"（可复用）。
- "奖章/成就""任务"相关表**实测不存在**（库中无 `%badge%`/`%achievement%`/`%task%`/`%medal%` 表），需新增（可扩展）。
- `item_templates.reference_price_points`（实测 `decimal(10,2)`）不得作为"市价"直接展示给用户，也不作为价值判定唯一依据（改用 `value_tier`），以免帮其建立现金锚点。

> 本节为方向与落地要点；具体表/字段缺口与门槛配置方案可另行展开为施工图。

### 7.5 具体用例：复合门槛兑换"水晶 + 星石 + 特定奖品 + 特定等级"（实测可行性 + 落地设计）

> 需求：某奖品要求用户"持有特定等级 + 同时消耗特定水晶 + 星石 + 某个指定奖品/道具"才能兑换。兑换目标**实物与虚拟道具都要支持**。

| 消耗指定奖品/道具 | 当前无"兑换前置消耗某 item"逻辑 | ⚠️ 需扩展 |

```
exchange_redeem_requirement（新增，门槛在 channel_prices 单资产计价之上叠加）
├─ exchange_item_id / sku_id      关联商品/SKU
├─ min_growth_level_key           特定等级（关联 user_growth_levels.level_key，如 'diamond'）
├─ extra_cost_assets (JSON)       多资产组合，如 [{asset:'red_core_shard',amount:100},{asset:'star_stone',amount:50}]
├─ required_consume_items (JSON)  需消耗的特定奖品/道具 [{item_template_id, quantity}]
├─ required_badges / required_tasks (JSON)  预留（本用例可空）
└─ is_enabled / publish_at / unpublish_at   上下架窗口（对齐 channel_prices 风格）
```

**🔴 合规红线（实测 `AssetProductGuard`，目标是实物 vs 虚拟道具的分叉，必须遵守）**：
- `AssetProductGuard.assertPriceAssetAllowed` 红线①：**星石（`star_stone`）永不得为"有价值商品"（`item_type='product'/'voucher'` 或有参考价）计价**；红线②：有价值实物/券**仅限水晶系（`*_core_shard/_core_gem`）计价**。
- 因此本用例的资产组合按**兑换目标类型分叉**：
  - **目标=实物/券（valuable）**：组合里**只能用水晶系**（如 `red_core_shard`），**禁止含 `star_stone`**——加了会被守卫拦 `STONE_BUY_VALUABLE_FORBIDDEN`。"消耗特定奖品 + 等级门槛"可正常叠加。
  - **目标=虚拟道具（`item_type='prop'`，零价值）**：水晶系与星石**均可**，"水晶+星石+消耗道具+等级"组合完全放行（星石买零价值道具是向下销毁）。
- 落地校验须把 `extra_cost_assets` 里每个资产也过一遍 `assertPriceAssetAllowed`，而非只校验主 `channel_prices` 那一条。

**兑换校验接入点（实测）**：在 `services/exchange/CoreService.js` 的 `exchangeItem()` 扣费前（约 165-272 行单资产计价处之前）插入门槛校验：① 查 `user_growth_levels` 当前等级 ≥ `min_growth_level_key`；② 校验 `extra_cost_assets` 余额足够且逐项过守卫；③ 校验 `required_consume_items` 持有量足够；全满足后在同一事务内逐项扣减 + 销毁指定道具，否则抛 `BusinessError` 提示缺哪项。

**待你拍板**：① 等级阈值（`min_history_points` 现为占位值，需定真实数字）；② 每个高门槛奖品的具体配方（哪些水晶/多少星石/消耗什么道具/最低等级），由运营在 admin 配置。

## 八、竞价标的限制（双层防护：硬约束 + 运行时守卫）

### 8.1 现状（真实代码 + 真实库核查）

竞价已有部分防护，并非完全未卡死（实测 `services/shared/AssetProductGuard.js`）：

- 实测 `bid_products` 表当前 **0 条**记录。

**残留两个洞（实测代码确认）**：
- `isValuable(template)`（57-63 行）判定为 `['product','voucher'].includes(item_type) || reference_price_points > 0`。问题：真实库有参考价为 0 的实物（`item_templates` product/voucher 的 `min_ref` 实测均为 0.00），若某实物 `item_type` 被误配或将来引入 `prop` 混淆，**参考价 0 不会触发 valuable 判定**——但因当前 `item_type='product'/'voucher'` 仍能命中前半条件，眼下尚安全。隐患在于"判定同时依赖 item_type 与参考价"的二元逻辑，将来加 `prop` 后易出歧义。
- 缺少"实物/券类型禁止进入竞价"的**正面显式规则**：当前靠 `assertPriceAssetAllowed`（星石不能买有价值物）+ `is_biddable` 间接挡，没有"竞价标的必须是 prop"的正向断言。

### 8.2 方案：硬约束 + 运行时守卫双层

**第一层 · 硬约束（铁律，让违规组合无法存在）**
- 规范 `item_templates.item_type` 枚举：`product`/`voucher` = 有价值实物侧；`prop` = 零价值虚拟道具侧。
- 建立"物品类型 × 可计价货币"**白名单常量表**（如 `config/` 下集中定义），写入/上架/发起竞价时强制校验。
- **竞价标的硬规则：仅允许 `item_type='prop'`（纯虚拟、不可发货）进入竞价；`product`/`voucher` 一律禁止。** 此规则写入数据结构与启动期校验，从源头杜绝"实物进竞价"。

**第二层 · 运行时守卫（强化现有 `AssetProductGuard`）**
- 把 `isValuable()` 判定基准从"参考价 > 0"改为"**以 `item_type` 枚举为准**"（`product`/`voucher` 恒为有价值），不再依赖可能为 0 的参考价。
- 在竞价发起入口增加显式断言：标的 `item_type` 必须属于"可竞价虚拟道具白名单"，否则拒绝。
- 保留原有两条红线（星石不买有价值物、有价值物仅水晶系计价）作为兜底。

### 8.3 分工原则

| 层 | 管什么 | 实现位置 |
|---|---|---|
| 硬约束 | 铁律红线（实物永不进竞价、星石永不买实物） | 类型枚举 + 白名单常量 + 启动期/写入期校验 |
| 运行时守卫 | 会变的业务规则 + 兜底拦截 | `AssetProductGuard`（强化判定基准） |

> 设计理由：铁律靠类型从源头杜绝（后人想破坏都难），业务规则靠守卫集中可测试。符合"长期低维护、降低技术债"目标。属功能开发，工作量小（约 1-2 天）。

- 模型层实测 `models/User.js` 第 41 行定义 `mobile` 字段、第 136 行 `fields:['mobile']` 唯一索引、第 272 行 `User.findByMobile = (mobile)=>({where:{mobile,status:'active'}})`；`services/UserService.js` 的 `findByMobile`（150/170 行）与注册判重（505 行）均 `where:{mobile}` 明文查询——**是盲索引改造的关键收口点**。
- ⚠️ 旧 `隐私政策.md` 若声称"手机号采用哈希脱敏处理存储"，与实测明文存储不符，构成虚假告知（注：本仓库 `docs/隐私政策.md` 实测已在工作区删除/移动，落地时以最终上线版本为准，务必表述与实际一致）。
- `.env` 实测已有 `ENCRYPTION_KEY`（32位）、`PII_HASH_SECRET`（决策25 注释），加解密基础设施现成。

### 9.2 方案：应用层 AES 加密 + HMAC 盲索引双列（中型公司主流）

选型理由：`.env` 已有 `ENCRYPTION_KEY`、`PII_HASH_SECRET`，基础设施现成，无需上 KMS；Sequelize 可用 getter/setter 或 hooks 做透明加解密，业务代码几乎无感；盲索引解决"加密后仍能按手机号查/判重/登录"。

**双列设计（以手机号为例）**：

| 列 | 用途 | 算法 | 可逆 |
|---|---|---|---|
| `mobile_encrypted` | 存储密文，发短信/展示时解密 | AES-256（密钥 `ENCRYPTION_KEY`） | 可逆 |
| `mobile_hash` | 唯一性约束、登录/判重查询 | HMAC-SHA256（密钥 `PII_HASH_SECRET`） | 不可逆 |

- 原 `mobile` 明文列**移除**（你未上线、数据可重建，无需迁移）。
- `mobile_hash` 加唯一索引，替代原 `mobile` 唯一约束。
- 收货信息（`receiver_name`/`receiver_phone`/`detail_address`）加密存储，展示脱敏（`王**`、`138****1234`）。

### 9.3 落地改造点

- **模型层**：`models/User.js` 用 Sequelize getter/setter（或 `beforeSave` hook）对手机号透明加解密；新增虚拟字段在读取时返回明文给业务、写入时自动生成 `mobile_encrypted` + `mobile_hash`。
- **查询收口**：`UserService.findByMobile` 内部把 `where: { mobile }` 改为 `where: { mobile_hash: hmac(mobile) }`；缓存 key 同步改用 `mobile_hash`，杜绝明文落 Redis。
- **流程合规**：补"首次登录授权同意"（用户协议 + 隐私政策勾选），记录同意时间戳。
- **文档对齐**：改造完成后 `隐私政策.md` 的"哈希脱敏存储"表述方为真；上线前若未完成，先改为与现状一致的客观表述，消除虚假告知。

> **落地方式（2026-06-08 修订）**：原建议"清库重建"基于"未上线、无真实数据"假设；实测库内有 139 个真实用户 + 测试账号，假设不成立，改用**非破坏式回填**（加双列→回填→切查询验证→删明文列），合规终态与清库相同（库内无明文）但数据零丢失。详见决策 17.2 修订与第二十二节。

## 十、星石消耗出口：虚拟装饰体系（游戏方案·去内核版）

### 10.1 现状（真实库核查）

- 星石（`star_stone`）**零正当消耗出口**：实测 `exchange_channel_prices` 全部 59 条计价**仅用 `red_core_shard`（红源晶碎片）**，无任何 `star_stone` 计价商品；`bid_products` 实测 0 条。星石只在冻结/转换间流动，无"花掉即销毁"的终点。
- 实测 `item_templates` 仅 `product`(10)/`voucher`(20) 两类，**无 `prop`（纯虚拟道具）数据**；`item_type` 列为 `varchar(50)`（非枚举，加 `prop` 无需改表）。`AssetProductGuard` 已预留 prop 逻辑（`assertVirtualPropOnly`，星石可买零价值道具向下销毁放行）。
- **无任何装饰/装备相关表**（实测库中无 `%decoration%` 表），需新建。

### 10.2 方案定调：取游戏运营玩法的壳，去游戏的内核

选择"游戏方案"以获得最大运营威力，但**必须剥离游戏内核**，守住"营销平台 + 纯展示装饰 + 明码标价"的合规底线。

**✅ 保留（纯运营手段，不碰红线）**：
- 赛季 / 限定 / 联名款（限时供应，造稀缺，促当季星石消耗）
- 品质分级（普通/稀有/史诗/传说，**仅视觉差异，零数值**）
- 套装收集（集齐额外展示效果）、排行榜 / 炫耀位
- 限时装饰（有效期制，促星石持续消耗、不枯竭）

**❌ 禁止（游戏内核，会致定位漂移/射幸回归/技术债飙升）**：
- **抽装饰 / 开箱**得装饰 —— 装饰必须**明码标价直接用星石购买**，严禁随机获取
- 装饰带**任何数值 / 功能加成**（如"佩戴提升中奖率"）—— 必须纯展示、零功能
- 升级 / 合成里再嵌随机性

> 本质：用游戏化的运营外壳，装消费回馈的合规内核，与全局路线B 思路一致。

### 10.3 数据模型（重度 SKU，但纯展示）

| 表 | 用途 | 说明 |
|---|---|---|
| `item_templates`（复用） | 装饰模板 | 新增 `item_type='prop'` 记录，`reference_price_points=0` |
| `decoration_sku`（新增） | 装饰 SKU | 品质、赛季、套装归属、是否限定；**无任何数值属性字段** |
| `user_owned_decorations`（新增） | 用户拥有 | `user_id, sku_id, equipped(是否佩戴), acquired_at, expires_at(NULL=永久)` |
| `decoration_season`（新增） | 赛季/限定 | 赛季周期、上下架时间，支撑绝版机制 |
| `exchange_channel_prices`（复用） | 星石计价上架 | `cost_asset_code='star_stone'`，明码标价（实测该表现仅有 `red_core_shard` 计价，新增 `star_stone` 计价是首次启用星石出口，符合 `is_biddable`/守卫预留） |

### 10.4 落地要点

- 装饰**明码标价**用星石购买，`AssetProductGuard` 自动放行（零价值道具星石可买，向下销毁）。
- 限时装饰 `expires_at` 到期由定时任务清理（复用现有 job 框架），保证星石持续消耗。
- 装饰**纯展示**：佩戴效果只影响 UI（头像框/气泡/主题/称号），不进入任何业务计算（尤其不碰抽奖/回馈逻辑）。
- 运营后台：装饰 SKU 配置、赛季管理、上下架——纳入 admin 后台。

### 10.5 运营价值（为何选此方案）

- 星石消耗**深度大**：收集、续费、换季持续吃掉星石，根治"星石攒着没处花"。
- 用户**粘性 / 活跃高**：收集欲、攀比、赛季追逐，提升留存与 DAU。
- 运营**抓手多**：可频繁出新品、做活动、造话题。
- **反哺营销科技定位**：装饰偏好成为用户画像维度，给商家更细的用户标签。

> 代价：重度 SKU 体系工程量与维护成本明显高于轻量方案（约 8-12 天）。已确认愿承担以换取最大运营威力。

## 十七、决策定案与行业依据（已拍板）

> 本节为已拍板结论。每条含【定案】+【行业对照】+【落地动作】。未列出的开放项（如 `code` 机器码、竞价 prop 口径、前端删映射不兼容）按文档既定推荐执行。

### 17.1 测试活动 id=25/26/27 —— 【定案：物理删除】

- **行业对照**：大厂（美团/阿里/腾讯）生产数据从不物理删（审计/对账/可追溯），但靠严格的 dev/test/prod 环境隔离让测试垃圾不进 prod；小公司 dev 库直接物理删；游戏/活动策划公司用 `is_online`/`is_deleted` 软下架保历史。
- **为何选物理删**：这三条是 dev 库测试垃圾（"走走走""12312"），非真实用户数据，未上线无审计包袱。表虽自带 `is_hidden` 列（软隐藏能力现成），但软隐藏价值在"保留真实历史"，对垃圾数据无意义反而留脏。
- **落地动作**：删前先查子表外键引用（`lottery_draws`、`lottery_campaign_prizes`、`lottery_campaign_quota_grants`、`lottery_campaign_user_quota` 等）；有关联行则连带清理。`id=1` 主活动不动。
- **⚠️ 运营原则（仅此次例外，今后不再物理删）**：物理删**只适用于 25/26/27 这类开发期测试垃圾**（无真实用户参与、无对账价值）。**今后所有正式运营活动一律软处理，禁止物理删**——`lottery_campaigns` 表实测无 `deleted_at` 列，软处理通过现有字段实现：活动自然结束置 `status='ended'`、下线留存置 `status='cancelled'` 或 `is_hidden=1`、临时停置 `status='paused'`。原因：正式活动关联真实 `lottery_draws`/积分流水/奖品发放，物理删会产生孤儿数据、破坏审计与客诉追溯。

### 17.2 PII 落地 —— 【定案修订：非破坏式回填双列，保留真实数据】

> ⚠️ **2026-06-08 决策修订**：原定"清库重建"，但落地前直连真实库核查发现库内已有 **139 个真实用户 + 测试账号 13612227930（user_id=31）**，清库会销毁真实数据与测试账号，违反"不删真实数据/保留测试账号/不用 mock"硬规则。故改为**非破坏式回填**。**两方案合规终态完全一致**（库内最终都是 `mobile_encrypted` + `mobile_hash` 双列、无明文 `mobile` 列），差异仅在数据：回填保留全部 139 条，清库会清零。已实测删明文列后测试账号登录正常。

- **行业对照**：大厂用 KMS/HSM + 信封加密 + HMAC 盲索引，密钥不落 `.env`；中小公司主流是应用层 AES-256 + HMAC 盲索引双列（即本文档第九节方案），密钥放环境变量/配置中心；二手/虚拟交易平台手机号+收货信息必加密、展示脱敏、盲索引解决登录判重。
- **为何改非破坏式回填**：实测库非"空库/无真实用户"，原"清库免迁移"前提不成立。回填步骤"加双列 → 回填现有明文 → 切查询验证 → 删明文列"每步可验证、可回滚，风险低于不可逆的清库；合规收益与清库相同（终态无明文）。
- **落地动作（已执行）**：迁移 `20260608005115`（加双列 + 回填 139 用户 + 盲索引唯一约束）→ 模型虚拟字段透明加解密 + `beforeFind` 盲索引改写 → 验证登录/注册/判重 → 迁移 `20260608005116`（删明文 `mobile` 列）。**密钥方案（2026-06-08 定案）**：维持 `.env` 明文方案（`ENCRYPTION_KEY`/`PII_HASH_SECRET`），用户确认不迁 Sealos secret/KMS，不再列为待办。
- **已知固有限制**：手机号加密后无法做"中间任意片段模糊搜"（密文随机、盲索引仅支持等值）。**已定案方案二完整版**：在全量 `mobile_hash` 外加 `mobile_prefix_hash`（前3位号段）+ `mobile_suffix_hash`（后4位尾号）双盲索引，支持「完整号 + 号段 + 尾号」搜，中间片段落到昵称/ID 搜。详见第二十二节。

### 17.4 高价值门槛判定线 —— 【定案：加显式 `value_tier` 字段 + 三层门槛】

- **行业对照**：大厂积分商城（阿里/美团）高价值实物（手机/家电）走"积分+现金"或预约/抽签/限量，从不纯积分直兑（防积分锚定成现金）；游戏把高稀有度锁在保底/限量后；二手平台按 RMB 价值分层风控。
- **为何不用纯价格阈值**：实测 `reference_price_points` 跨度 0~5000 且**有一批实物参考价为 0**，靠"价格>阈值"会漏判——这是数据质量问题，不该用价格反推价值。
- **落地动作（治本、低债）**：给 `item_templates` 新增显式枚举字段 `value_tier`（`low`/`mid`/`high`），由运营在 admin 配置，代码按字段判定而非猜价格。三层门槛：
  - `low`（日常物，参考价≤约100）：碎片直兑。
  - `mid`（中档，约100~1000）：轻门槛（VIP 等级）。
  - `high`（高档，>1000 或运营标记）：全复合门槛（VIP + 奖章 + 任务 + 限量 + 碎片）。
  - 参考价为 0 的实物由运营显式归入 mid/high，杜绝漏判。

### 17.5 admin 界面文案去"抽奖/中奖"化 —— 【定案最终版：方案 C+ 集中业务叙事字典】

> ⚠️ **2026-06-08 决策演进（三轮）**：
> 1. 原定"全不改"（admin 内部、合规零收益）；
> 2. 修订为"分层散改前端界面词"（运营对外沟通需对齐 C 端口径）；
> 3. **重新排查发现**：admin 显示的"抽奖/中奖"文字**一半来自后端 API 返回**（告警/通知/待办的 `title`，在 `NotificationService`/`LotteryHealthService`/`PendingCenterService` 等生成），且前端 `handleAlert` 用 `title.includes('抽奖'/'中奖')` 做路由判断——**纯前端散改既改不干净（后端文字改不到），又有"文案与路由判断耦合"的真实功能风险**。故最终升级为**方案 C+：集中业务叙事字典（单一数据源）**。

> **实测数据更正**：原文档"141 处"不准确。实测 admin 源码（排除 dist）"抽奖"共 **483 处 / 70 文件**；后端另有告警/通知 `title` 硬编码若干处。

**方案选型（C+ 为何最优，已排除 B/C++/D）**：

| 方案 | 文字住哪 | 改名工作量 | 新建成本 | 是否采用 |
|---|---|---|---|---|
| B 散改硬编码 | 70+ 文件散落 | 翻 70 文件、易漏、后端改不到 | 无 | ❌ 治标不治本 + 功能风险 |
| **C+ 集中字典（单一源）** | **1 个常量文件** | **改 1 处全局生效** | 低（复用现有 displayNameHelper 字典模式）| ✅ **采用** |
| C++ 字典进数据库 | DB 表 + 管理页 | 改表即生效 | 中（建表+UI+缓存）| ❌ 过度（你非高频改文案）|
| D i18n 文案中心 | 文案平台 | 运营自助改 | 高（框架+多语言）| ❌ 杀鸡用牛刀（中文单语言）|

**C+ 三层对齐原则（不变）**：

| 层 | 内容 | 对齐对象 | 处理 |
|---|---|---|---|
| 展示文字（前端界面 + 后端返回的 title/message）| 菜单/标题/图表 legend/导出列名/告警/通知文字 | 对齐 C 端 → 取自叙事字典 | ✅ 收口到字典 |
| 代码注释 / JSDoc | 模块说明、字段解释 | 对齐后端字段 | ❌ 不改 |
| 技术标识符 | `lottery_campaign_id`、`draw_id`、API 路径 `/lottery/*`、类型 `LotteryDrawRecord`、菜单 `id:'lottery'`、快捷键 `G+L`、对象 key（如 `draw_quality`）、`type:'lottery'` 枚举 | 必须对齐后端真实字段 | ❌ 绝不改 |

**C+ 落地形态（基于现有技术栈，详见第二十三节）**：
1. **复用现有数据库字典表 `system_dictionaries`**（不新建常量文件，避免与现有字典体系并列两套）：新增 `dict_type='business_narrative'` 一组词条作为业务叙事的**唯一数据源**。

3. 前端展示文字处理（执行时决策修正，见 23.5 第6条）：**静态显示文字（图表 legend/菜单/标题/导出列名）直接改为"回馈/发放"口径**（与字典值一致；行业对稳定标签的通行做法，避免 ECharts legend 与 series name 异步绑定的时序风险）；ECharts legend + series name + tooltip 判断同步改以保持绑定。仅后端动态 title 走字典。
4. `handleAlert` 等逻辑判断 **改用英文 `alert.type` 枚举**（本就有 `type==='lottery'`），不再 `includes` 中文 → 解除"文案-路由"耦合，更健壮。
5. 技术标识符（字段名/API/类型名/对象 key/`type` 枚举值）**全程不碰**。
6. 改写映射（叙事词）：抽奖→回馈、抽奖次数→回馈次数、**中奖次数→发放次数、中奖率→发放率**、抽奖活动→回馈活动、抽奖告警→回馈告警、抽奖记录→回馈记录、抽奖消耗积分→回馈消耗积分。

**为何存数据库字典而非常量文件**：① `system_dictionaries` 已存在（含 80+ dict_type）→ 零新文件零新机制、真正复用；② 文案属配置类数据，符合本项目"配置类可直接改库"规则，改名 UPDATE 一行即生效、不发版；③ 白送 `system_dictionary_history` 版本审计（谁改了文案有留痕）；④ 现有 `DisplayNameService.batchGet` 已带 Redis 缓存，性能无忧。

**关键技巧**：`{ key:'draw_id', label: 取自字典 }` 只改 `label`、不动 `key`。

### 22.2 User 模型 `beforeFind` 透明查询钩子（设计说明）

删除明文 `mobile` 列后，全项目 96+ 处 `where: { mobile }` 等值查询会因列不存在而失效。模型层加 `beforeFind` 钩子，把 `where.mobile`（字符串/数组）透明改写为 `where.mobile_hash = blindHash(mobile)`。

- **定性**：这是**模型对自身存储映射的封装**（明文查询入参 → 密文存储），不是"前后端字段映射层"，符合"以后端字段为准、不做兼容映射"原则——调用方写的仍是后端真实语义 `mobile`，由模型自洽地落到盲索引。
- **覆盖范围**：`findOne/findAll/findByPk/count/findAndCountAll` 等所有走 `beforeFind` 的等值查询，全部无需改代码即正确工作。写操作（update/destroy）实测无一处按 mobile 查，安全。

### 22.3 固有限制：加密手机号无法做「任意片段」模糊搜（已定案：方案二完整版）

密文随机、盲索引仅支持等值，**任何加密手机号方案都无法做"手机号任意中间片段模糊搜"**（行业共识）。实测管理端有 **7 处** `mobile LIKE '%片段%'` 模糊搜（`UserManagementService`、`UserRoleService`、`UserDataQueryService`、`CustomerServiceAgentManagementService`、`CustomerServiceSessionService`、`consumption/QueryService`、`BusinessRecordQueryService`），删明文列后这些会报 `Unknown column 'User.mobile'`。C 端用户登录（精确匹配）不受影响。

**【定案：方案二完整版 — 前3位号段 + 后4位尾号双盲索引】**（2026-06-08 用户拍板）

在全量 `mobile_hash`（完整号精确匹配）之外，额外建两列盲索引，支持运营按号段/尾号搜：

| 列 | 内容 | 支持的搜索 |
|---|---|---|
| `mobile_hash`（已有）| 完整 11 位的 HMAC | 输入完整手机号 → 精确命中 |
| `mobile_prefix_hash`（新增）| 前 3 位号段的 HMAC | 输入「136」→ 命中该号段全部用户 |
| `mobile_suffix_hash`（新增）| 后 4 位尾号的 HMAC | 输入「7930」→ 命中该尾号用户 |

- **能力**：完整号精确搜 ✅ + 号段前3位搜 ✅ + 尾号后4位搜 ✅；**中间任意片段（如「1361222」）仍搜不了**（加密物理上限，无解）。
- **路由判定逻辑**：运营输入归一化后按长度/位置分派——11位→`mobile_hash`；纯3位数字→`mobile_prefix_hash`；纯4位数字→`mobile_suffix_hash`；其余（含中间片段、非数字）→ 当昵称/用户ID搜，不再对 mobile 做 `LIKE`。
- **合规性**：✅ 仍无明文落库/落查询条件（三列均为不可逆 HMAC）。
- **已知取舍**：前3位号段哈希区分度低（号段有限），可被反推「号段分布」，但不暴露完整号，风险可接受（用户已知悉并接受）。
