## 方案A：仅调配置参数（零代码改动）

**预期效果：支撑 3000-5000 并发请求/秒，1-2万同时在线**

### 修改 ecosystem.config.js

```javascript
// 修改前
exec_mode: 'fork',
instances: 1,
node_args: '--max-old-space-size=512',

// 修改后
exec_mode: 'cluster',    // cluster模式，PM2自动负载均衡到多核
instances: 8,            // 8个worker进程（32核用8个，留余量给系统和DB）
node_args: '--max-old-space-size=2048',  // 每进程2GB堆内存
```

### 修改 config/database.js 连接池

```javascript
// 修改前
pool: {
  max: 40,
  min: 5,
  acquire: 10000,
  idle: 60000,
  evict: 30000,
}

// 修改后（每进程20个 × 8进程 = 160总连接，MySQL上限2400，安全）
pool: {
  max: 20,
  min: 3,
  acquire: 10000,
  idle: 30000,
  evict: 15000,
}
```

### 注意事项

- cluster模式下每个worker是独立进程，各自维护自己的连接池
- Socket.IO 在 cluster 模式下需要 Redis Adapter（见方案B1）
- PM2 cluster 模式自带零停机重启（`pm2 reload`）
- 如果不需要 WebSocket 跨进程通信，方案A可直接使用

| 指标 | 优化后 | 提升倍数 |
|------|--------|----------|
| HTTP 吞吐量 | ~5000-8000 req/s | 5-8x |
| 同时在线连接 | ~15000-20000 | 4-5x |
| 数据库并行查询 | 160 | 4x |
| 内存可用 | 16GB (8×2GB) | 32x |

## 方案B：单机深度优化（少量代码改动）

**预期效果：支撑 8000-15000 req/s，3-5万同时在线连接**

在方案A基础上增加以下优化：

### B1. Socket.IO Redis Adapter

解决 cluster 模式下 WebSocket 跨进程通信问题。当用户连接到 worker-1，但消息需要发送给连接在 worker-3 的用户时，需要 Redis 作为消息总线。

**安装依赖：**

```bash
npm install @socket.io/redis-adapter
```

**在 app.js 中 Socket.IO 初始化处添加：**

```javascript
const { createAdapter } = require('@socket.io/redis-adapter');
const Redis = require('ioredis');

const pubClient = new Redis(process.env.REDIS_URL);
const subClient = pubClient.duplicate();

io.adapter(createAdapter(pubClient, subClient));
```

### B2. 连接池 + 查询缓存优化

```javascript
// config/database.js
pool: {
  max: 25,       // 每进程25个 × 8进程 = 200总连接
  min: 5,
  acquire: 8000, // 缩短获取超时，快速失败
  idle: 20000,   // 更积极回收空闲连接
  evict: 10000,
}
```

热点数据 Redis 缓存策略：

| 数据类型 | 缓存时间 | 说明 |
|----------|----------|------|
| 抽奖配置 | 5分钟 | 读多写少，管理员修改时主动失效 |
| 用户积分余额 | 30秒 | 写入时同步更新缓存 |
| 系统设置 | 10分钟 | 极少变更 |
| 商品列表 | 2分钟 | 中等频率更新 |

### B3. 速率限制调整

```javascript
// middleware/RateLimiterMiddleware.js
// 通用API限制
windowMs: 60 * 1000,
max: 300,        // 从100提升到300（高并发场景）

// 抽奖限制（保持严格，防刷）
windowMs: 60 * 1000,
max: 30,         // 从20提升到30

// 登录限制（保持严格，防暴力破解）
windowMs: 60 * 1000,
max: 10,         // 保持不变
```

### B4. Node.js 运行时调优

```javascript
// ecosystem.config.js
node_args: '--max-old-space-size=2048 --max-semi-space-size=128',
// --max-semi-space-size=128: 增大新生代空间到128MB（默认16MB）
// 减少Minor GC频率，降低请求延迟抖动
```

### B5. Express 中间件优化

```javascript
// 启用 HTTP Keep-Alive（减少TCP握手开销）
const server = app.listen(PORT);
server.keepAliveTimeout = 65000;    // 65秒（大于Nginx默认60秒）
server.headersTimeout = 66000;      // 略大于keepAliveTimeout

// 启用 ETag 缓存（静态资源304响应）
app.set('etag', 'strong');
```

### 方案B效果预估

| 指标 | 优化后 | 提升倍数（vs当前） |
|------|--------|----------|
| HTTP 吞吐量 | ~10000-15000 req/s | 10-15x |
| 同时在线连接 | ~30000-50000 | 10-15x |
| WebSocket 长连接 | ~30000+ | 10x |
| P99 延迟 | <100ms | — |

### C1. Nginx 配置示例

```nginx
upstream lottery_backend {
    least_conn;                          # 最少连接数策略
    server 127.0.0.1:3000 weight=1;     # 本机
    server 10.0.0.2:3000 weight=1;      # 第二台机器
    server 10.0.0.3:3000 weight=1;      # 第三台机器
    
    keepalive 64;                        # 上游保持连接池
}

# WebSocket需要ip_hash保证同一用户连同一后端
upstream lottery_ws {
    ip_hash;
    server 127.0.0.1:3000;
    server 10.0.0.2:3000;
    server 10.0.0.3:3000;
}

server {
    listen 80;
    server_name your-domain.com;

    # API请求
    location /api/ {
        proxy_pass http://lottery_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        proxy_connect_timeout 5s;
        proxy_send_timeout 30s;
        proxy_read_timeout 30s;
    }

    # WebSocket
    location /socket.io/ {
        proxy_pass http://lottery_ws;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400s;       # WebSocket长连接保持24小时
    }

    # 静态资源（管理后台）
    location /admin/ {
        proxy_pass http://lottery_backend;
        proxy_cache_valid 200 1h;        # 静态资源缓存1小时
        expires 1h;
    }

    # 健康检查
    location /health {
        proxy_pass http://lottery_backend;
        access_log off;
    }
}
```

### C3. 数据库层扩展

```
MySQL 读写分离：
- 主库：处理 INSERT/UPDATE/DELETE
- 从库1-2：处理 SELECT 查询
- 连接分配：2400 / 3台机器 = 每台800连接上限

Sequelize 读写分离配置：
const sequelize = new Sequelize(database, null, null, {
  replication: {
    read: [
      { host: 'read-replica-1', username: 'user', password: 'pass' },
      { host: 'read-replica-2', username: 'user', password: 'pass' },
    ],
    write: { host: 'master', username: 'user', password: 'pass' }
  },
  pool: { max: 25, min: 5 }
});
```

```lua
local user_key = KEYS[1]           -- user:points:{userId}

if current_count >= daily_limit then
    return {-1, 'DAILY_LIMIT_EXCEEDED'}
end

-- 2. 检查积分余额
local balance = tonumber(redis.call('GET', user_key) or '0')
if balance < cost then

end

redis.call('DECRBY', user_key, cost)
redis.call('INCR', limit_key)
redis.call('EXPIRE', limit_key, 86400)  -- 24小时过期

return {1, 'SUCCESS', balance - cost}
```

## 7.3 ⚠️ 方案A「零代码」是假的——cluster 前必须先堵这 5 个真实漏洞

文档前文说「方案A 零代码改动」。**实测代码后这是错的**。把 `instances` 从 1 改成 8，下面 5 处会立刻出 bug。这是本次最重要的发现。

| 编号 | 真实代码位置 | cluster 下会怎样 | 必须做的修复 |
|------|-------------|-----------------|-------------|
| **R1** | `scripts/maintenance/scheduled_tasks.js`：40 个 `cron.schedule`，**只有约 4 个**用了 `distributedLock.withLock`/`tryLock` | 8 进程 = 每个 cron 同时跑 8 次（重复扣费、重复发奖、重复对账） | 给**所有**写类 cron 包 `withLock`；或只让 1 个 worker 跑 cron（判断 `process.env.NODE_APP_INSTANCE==='0'`） |
| **R2** | `routes/v4/lottery/middleware.js:23` `const requestCache = new Map()`（抽奖请求去重） | 内存 Map 不跨进程，去重失效，连点可能落到不同 worker | 改用现成的 `UnifiedDistributedLock`/Redis `SET NX`（你已有，几行） |
| **R3** | `app.js` Socket.IO 初始化无 Redis Adapter（`node_modules/@socket.io/redis-adapter` 未安装，实测只有 `component-emitter`） | 连 worker-1 的用户收不到 worker-3 推送的消息（客服/聊天断） | `npm i @socket.io/redis-adapter`，用**已有的** `getPubClient()/getSubClient()` 接上 |
| **R4** | `services/UnifiedLotteryEngine/**` 多处 `this.cache = new Map()`（ManagementStrategy/CacheManager/BudgetProviderFactory） | 每进程一份内存缓存，管理员改抽奖配置后，部分 worker 仍读旧配置（最多 5 分钟不一致） | 配置写入时用 Redis pub/sub 广播失效（`SystemConfigService` 已有 Redis 失效，照搬模式即可） |
| **R5** | `ecosystem.config.js` `max_memory_restart:512M` + `--max-old-space-size=512` | 8 进程仍各自 512M，没用上 127G 内存 | 调到每进程 2048M（见 7.4） |

**R1 是最危险的**——重复跑「扣费/发奖/对账」类定时任务会直接造成资损。**没堵 R1 之前，禁止开 cluster。**


## 7.4 推荐方案：选「方案B 的修正版」，不选「方案A」

**结论先行**：选**方案B（单机深度优化）的修正版**，但执行顺序必须是「先补齐 cluster 安全性（R1-R4），再开多进程（R5）」。

理由：
1. 方案A 的「零代码」前提不成立（见 7.3），直接开 cluster 会出资损级 bug。
2. 你的服务器 32 核 / 127G 完全是单机方案B 的甜点区，单机即可支撑数万并发，**无需进入方案C（多机/Nginx/读写分离）**——当前数据量（百级用户）连方案A 的余量都用不到，方案C 是纯浪费。
3. 方案B 的每一项增量都落在你**已有的** Redis/锁/限流设施上（见 7.2），维护成本最低、技术债最少，符合「未上线、一次做对、不兼容旧接口」的诉求。

## 8.1 D1：cluster 进程数 —— 先 4 还是直接 8？

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| 大厂（美团/阿里/腾讯） | 容器化，每 Pod 跑 1 个 Node 进程，用 K8s HPA 按 CPU 横向扩 Pod，单 Pod 不开多 worker | 弹性伸缩交给编排层，进程内简单化 |
| 小公司 | PM2 cluster，`instances` 设为 `核数-1` 或 `max` | 一台机器吃满，最省钱 |
| 游戏公司 | 网关层多进程，游戏逻辑常单线程分服/分区 | 状态强一致，靠分区而非多进程 |
| 活动策划公司 | PM2 cluster 2-4 进程，活动前临时加 | 平时流量低，活动期才需要 |
| 虚拟物品/二手平台 | 多进程 + 交易核心可能独立 | 交易要稳，查询要快 |

**关键认知**：进程数不是越多越好。worker 越多 → DB 连接池总数越大（`worker × pool.max`）→ 越接近 MySQL `max_connections=2400` 上限，且历史已有 `Aborted_connects=2503`（连接风暴前科）。

**给你的结论：D1 = 先 4 个 worker。**
- 理由1（数据）：你真实数据量百级用户，4 进程 × 现有性能足够覆盖上线初期数万并发的预留。
- 理由2（连接安全）：4 worker × pool 20 = 80 连接，对 2400 上限极安全，杜绝重演连接风暴。
- 理由3（可观测）：先 4 跑一两周，看 `pm2 monit` 的 CPU/内存/事件循环延迟，再决定加到 8，是大厂「灰度扩容」的小型版。
- 配置写法：`instances: 4`（不要用 `'max'`，`'max'=32` 会瞬间把连接池放大到危险区）。

> 这条**完全落在你现有 PM2 栈**，0 新依赖。


## 8.2 D2：40 个定时任务防重 —— 单 worker 守卫 还是 全包分布式锁？

实测：`scheduled_tasks.js` 有 40 个 `cron.schedule`，只有约 4 个包了 `distributedLock`。cluster 后其余 36 个会被 N 个 worker 同时触发。

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| 大厂 | 独立调度中心（XXL-JOB/ElasticJob/SchedulerX），任务与业务进程分离，天然单点触发 | 任务多、要可视化、要重试 |
| 小公司 | 要么单进程跑 cron，要么 Redis 锁 | 没精力上调度中心 |
| 游戏公司 | 活动结算用独立 worker + 分布式锁 | 结算不能重复（资损） |
| 活动策划公司 | 单独一个"定时任务实例"专门跑 cron | 简单、隔离 |
| 虚拟物品/二手平台 | 对账/结算任务必加分布式锁 + 幂等 | 钱相关，零容忍重复 |

**两种可选方案的本质区别：**

- **方案 A「单 worker 守卫」**：`if (process.env.NODE_APP_INSTANCE === '0') ScheduledTasks.initialize()`。只让 0 号 worker 注册全部 cron。改 1 处，维护成本最低。**缺点**：0 号 worker 崩溃→重启的几秒空窗内任务可能漏跑（但 cron 是周期性的，下个周期补上，对账类任务本身幂等）。
- **方案 B「全包分布式锁」**：每个 cron 体里包 `distributedLock.withLock`。最稳，但要改 36 处，且每处都要处理锁失败分支，**新增 36 处维护点**=技术债。

**给你的结论：D2 = 单 worker 守卫为主 + 资损敏感任务额外包锁（双保险）。**

这其实就是**大厂「独立调度实例」思路的单机低成本版**：
1. 用 `NODE_APP_INSTANCE==='0'` 让定时任务只在一个 worker 注册（等价于"调度实例"）——改 1 处，几乎零债务。
2. 对**直接动钱/动资产的少数任务**（广告扣费、资产对账、孤儿冻结清理、市场价格快照等），**额外**包 `distributedLock.withLock`。即使将来上多机（多台机器各有 0 号 worker），这层锁仍能防重复。

> 为什么不全包锁：你已有 `UnifiedDistributedLock`，但给 36 个任务逐个包锁是「为了 1% 的边界情况，新增 99% 的维护面」，违背你「降低技术债」的目标。单 worker 守卫 + 关键任务兜底，是成本/安全的最优点。
> 这条**完全复用你现有的 `UnifiedDistributedLock` + PM2 环境变量**，0 新依赖。


## 8.3 D3：Redis 单点 —— 上线就上高可用，还是先单点？（最需要你拍板的架构题）

实测：`REDIS_URL=redis://localhost:6379`，本机单实例。cluster 后所有 worker 共享它，它一挂：限流、抽奖去重、分布式锁、Socket.IO Adapter 全受影响。

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| 大厂 | Redis Cluster（3 主 3 从起）或云托管（Tair/ElastiCache），多可用区 | 故障域隔离，自动故障转移 |
| 小公司 | 单 Redis 起步，挂了人工拉起；稍成熟上"哨兵 Sentinel 1 主 2 从" | 成本与可用性折中 |
| 游戏公司 | 抽奖/库存的 Redis 必高可用，且常和 DB 做最终一致对账 | Redis 是热数据真相源，不能丢 |
| 活动策划公司 | 活动期间临时上托管 Redis，活动后降配 | 按需付费 |
| 虚拟物品/二手平台 | Redis 高可用 + 关键状态以 MySQL 为准 | Redis 丢了能从 DB 重建 |

**你这里有个关键的好消息（实测代码）**：
- 限流中间件 `RateLimiterMiddleware.js` 已实现 **Redis 不可用时降级到内存限流**。
- 业务**正确性**（抽奖扣分、交易、兑换）的真相源是 **MySQL + 行锁事务**，不是 Redis。Redis 只承担"加速/去重/广播"。
- 也就是说：**Redis 短暂挂掉 → 系统降级但不产生错账**（去重短时失效、限流变宽松、WebSocket 推送可能丢，但钱是对的）。

**给你的结论：D3 = 上线初期接受单 Redis（但要做 3 件低成本加固），中期再上哨兵。**

因为你说"愿意一次性投入"，给你**分档**让你拍：
- **省心档（推荐先做）**：保持单 Redis，但①开启 Redis 持久化（AOF，项目里已有 `scripts/monitoring/redis_persistence_setup.sh`）；②PM2/systemd 守护 Redis 自动拉起；③给 Redis 操作加超时与降级日志。成本几乎为 0，覆盖 95% 的故障场景。
- **稳健档（愿投入就做）**：上 **Redis Sentinel（1 主 2 从 + 哨兵）**。`ioredis` **原生支持 Sentinel**，你的 `UnifiedRedisClient` 改的是连接参数（从 `REDIS_URL` 改成 sentinels 数组），**业务代码一行不用动**。这是"用现有技术栈直接长出来"的高可用，不引入新中间件。
- **过度档（不推荐现在做）**：Redis Cluster 分片。你数据量极小，分片纯属浪费运维，制造债务。

> 我的倾向：**先做省心档随方案B 一起上线；把"稳健档 Sentinel"列为上线后第一个迭代**。因为你的正确性不依赖 Redis，单点风险是"可用性"而非"资损"，没必要为了上线 day-1 就背 Sentinel 的运维复杂度。
> 这条**复用 ioredis 原生 Sentinel 能力**，0 新中间件。需要你拍：**先省心档，还是上线即 Sentinel？**


## 8.4 D4：MySQL `innodb_buffer_pool_size` 只有 1GB（机器 127G）

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| 大厂 | buffer pool 设为物理内存 50-75%，独立 DB 服务器 | 让热数据全进内存，命中率 99%+ |
| 小公司 | 经常用默认值（128M-1G），上线后才发现慢 | 不懂调参 / 用托管默认 |
| 游戏/交易类 | buffer pool 调大 + 慢查询监控 | 读多、要低延迟 |

**事实**：1GB buffer pool 是 MySQL 偏保守的默认级别。你当前数据量小（最大表 `asset_transactions` 约 84MB），1GB 暂时够用，**所以现在不痛**。但这是给上线后数据增长埋的伏笔——一旦数据量超过 buffer pool，磁盘 IO 飙升，查询变慢。

**给你的结论：D4 = 上线前申请把 buffer pool 调到物理内存的合理比例。**
- 你的库是 Sealos 托管 MySQL（`dbconn.sealosbja.site`），`buffer_pool` 调整通常要通过 Sealos 控制台改实例规格/参数，**不是后端代码能改的**——这是**运维/DBA 动作，不属于后端代码改造**。
- 建议值：若 DB 实例独享 16G 内存，buffer pool 设 8-12G。具体看 Sealos 给这个 MySQL 实例分了多少内存（这点需要你在 Sealos 控制台确认实例规格）。
- **这条不在本次后端代码改造范围内**，是一次性运维配置，但建议上线前做掉。

> 需要你拍：**是否同意上线前由你/运维在 Sealos 控制台调 buffer pool？**（后端代码不涉及）


## 8.5 D5：历史连接风暴（`Aborted_connects=2503`、`Max_used_connections=587`）

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| 大厂 | 连接池监控 + 告警 + 链路追踪定位泄漏 | 连接泄漏=雪崩前兆 |
| 小公司 | 出事了才查，平时不管 | 没监控 |
| 交易类 | 严格连接池上限 + 慢查询 + 泄漏检测 | 连接耗尽=交易全挂 |

**事实**：`Max_used_connections=587` 说明历史上某时刻同时有 587 个连接（远超你单实例 pool 的 40）。可能来源：开过多个脚本/多次重启叠加、连接未释放、或别的客户端（实测当前 processlist 里有多个 `172.16.0.x` 来源）。`Aborted_connects=2503` 多为连接超时/认证失败/未优雅关闭累积（`Uptime` 长达约 28 天，是长期累计值，不一定是单次风暴）。

**给你的结论：D5 = 不阻塞方案B，但建议作为独立小任务排查。**
- 你的代码已具备连接池监控（`app.js` 里 `setInterval` 每 60s 打点 `pool.size/using/waiting`，且有 `monitor:pool` 脚本）——这是**已有资产，直接复用**。
- cluster 化后我会把连接池告警阈值按 `worker × pool.max` 重新校准，顺带观察是否还有异常增长。
- 是否现在深挖 587 的来源：**建议另开任务**，用 `monitor:pool` + processlist 抓一段时间，不必卡住本次上线。

> 需要你拍：**D5 是"随方案B 一起加强监控即可"，还是"现在就专门排查连接泄漏"？**

| 抽奖/扣减 | Redis Lua+异步 | DB事务 | Redis Lua原子 | 队列削峰 | DB行锁+对账 | **DB事务+行锁(已有)** |
| 交易一致性 | 分布式事务/Saga | 本地事务 | 最终一致 | 本地事务 | 状态机+行锁 | **本地事务+行锁(已有)** |
| 数据库 | 分库分表+TiDB | 单库 | 分区+缓存 | 单库 | 单库+读写分离 | **单库+索引+缓存** |
| 运维门槛 | 专职DevOps团队 | 1人 | 专职 | 1-2人 | 2-3人 | **1人可维护** |
| 适合QPS | 无上限 | <1.5万 | 高(分区) | 3000-1万 | 500-5000 | **单机数万够** |

**为什么不是其他方案：**
- **不学大厂**：你没有亿级数据、没有专职运维团队。微服务/K8s/分库分表/Service Mesh 对你是**纯负债**——美团自己都说"不宜过早微服务"。引入它们=技术债务暴涨，与你的目标完全相反。
- **不学纯游戏公司的 Redis Lua 抽奖**：他们用 Lua 是因为单活动瞬时 QPS 几十万把 DB 打爆。你的正确性已由 **DB 行锁事务**保证（实测数十处 `LOCK.UPDATE`），现在叠 Lua 是重复造轮子 + 引入 Redis/DB 双写一致性新债务。**等真有单活动高并发瓶颈时再上，且那时也是局部优化**。
- **要学游戏/交易平台的"思想"**：原子性（你用 DB 事务实现）、幂等（你已有 `IdempotencyService`）、状态机（你交易链路已有）、对账（你已有对账任务）。**这些你都已经做到了**，这正是你项目的底子好的地方。

**一句话定调**：你的项目 = **「模块化单体 + PM2 cluster」**，这是 Shopify/Notion/闲鱼早期都走过、2025 年行业重新认可的"模块化单体"路线。它给你大厂 90% 的能力、只要 10% 的成本，且**每一块都长在你现有技术栈上，不引入任何你团队不会维护的新东西**——这才是"长期维护成本低、技术债最少"的真正含义。

### 任务 0-1（R1）定时任务防重 = 单 worker 守卫 + 关键任务加锁（对应 D2）

涉及文件：`app.js`（定时任务初始化处）、`scripts/maintenance/scheduled_tasks.js`

改法：
1. **关键**：实测 cron 注册分散在 `app.js` 的**两个不同位置**，worker 守卫必须同时包住两处，漏一处就会重复跑：
   - 位置①`initializeApp()` 内（约 962 行）：`ApprovalChainTimeoutService.start()`（审核链超时升级，每30分钟）。
   - 位置②`server.listen` 回调内（约 1145-1182 行）：`ScheduledTasks.initialize()` + 三个 jobs require。
   - 另有 `services/ApprovalChainTimeoutService.js` 自己用 `cron.schedule` 注册，且 `ScheduledTasks` 内部还调用了 `daily-data-cleanup` 等——这些都在上述两个入口之下，守住入口即可。

```javascript
// app.js —— 统一用一个守卫判断（fork 模式下 NODE_APP_INSTANCE 为 undefined 也放行）
const isCronWorker = () => {
  const id = process.env.NODE_APP_INSTANCE
  return id === undefined || id === '0'
}

// 位置①initializeApp() 内：
if (isCronWorker()) {
  ApprovalChainTimeoutService.start()   // ← 之前清单遗漏，必须也包进守卫
}

// 位置②server.listen 回调内：
if (isCronWorker()) {
  ScheduledTasks.initialize()
  require('./jobs/ad-cron-jobs')
  require('./jobs/ad-campaign-expiry-jobs')
  require('./jobs/daily-notification-cleanup')
}
```

2. 对**直接动钱/动资产**的任务，在任务体内额外包 `distributedLock.withLock`（双保险，防将来多机）。优先级：广告扣费、资产对账、孤儿冻结清理、市场价格快照。其余只读/统计类任务靠守卫即可，不必加锁。

> 复用：`UnifiedDistributedLock`（已存在）。新增依赖：无。
> ⚠️ 修正记录（2026-05-30 三次排查）：原 R1 方案只包了位置②，**遗漏了位置①的 `ApprovalChainTimeoutService.start()`**。本次补正。审核链超时任务会改审批状态，重复跑可能导致重复升级，必须纳入守卫。



### 任务 0-2（R2）抽奖请求去重：内存 Map → Redis

涉及文件：`routes/v4/lottery/middleware.js`

现状：`const requestCache = new Map()` 不跨进程，cluster 后去重失效。

改法：把去重键改用 Redis 原子写（复用现有 Redis 客户端），TTL 5 秒：

```javascript

// 关键：用 SET key 1 NX PX 5000 抢占；返回 null 表示重复请求 → 直接 429
```

要点：

# 十、完整版补漏：R1-R5 之外还必须做的事（2026-05-30 二次排查）

> 第九章覆盖了 R1-R5。二次逐文件排查后，发现「完整的方案B 修正版」还有下面几项**必做但之前没单独列**的工作。不做这些，cluster 上线后仍会有功能性 bug 或运维坑。

## 10.1 ⚠️【新增 R6 · 高优先级】WebSocket 在线用户表是进程内 Map

实测 `services/ChatWebSocketService.js`：
- `this.connectedUsers = new Map()`、`this.connectedAdmins = new Map()` 存「userId → socketId」。
- 所有推送方法（`sendMessageToUser`/`broadcastToAdmins`/`pushNotification`/`pushExchangeStockChanged` 等约 20 处）都靠这两个 Map 查 socketId 再 `io.to(socketId).emit()`。

**cluster 下会怎样**：用户连在 worker-2，他的 socketId 只存在 worker-2 的 Map 里。当 worker-1 想给他推送（例如客服在 admin 端发消息，落在 worker-1），worker-1 的 Map 里查不到这个用户 → **推送丢失**。

**这不是 R3 适配器能完全解决的**：
- R3 的 Redis Adapter 解决的是「用 `io.to(room).emit` 按房间广播」时跨进程转发。
- 但你的代码是「先在本进程 Map 里查 socketId，查到才 emit」——查不到就直接 return，根本不会走到 adapter 转发。

**两个修法（D6 已拍板 ✅ 选 R6-A）**：

| 方案 | 做法 | 成本 | 适合 |
|------|------|------|------|
| **R6-A（推荐）改用 Socket.IO room 机制** | 连接时 `socket.join('user:'+userId)`；推送时 `io.to('user:'+userId).emit(...)`，删掉 connectedUsers/connectedAdmins 查表逻辑 | 改约 20 处推送点，但每处都变简单 | 与 R3 适配器天然配合，跨进程自动转发，最干净、零债务 |
| **R6-B 在线表搬到 Redis** | userId→socketId 存 Redis Hash，推送前查 Redis | 改动小一点，但仍要处理 socket 在哪个进程 | 不如 A 干净，且 socketId 跨进程不能直接 emit |

> **强烈建议 R6-A**：room 是 Socket.IO 官方为多进程设计的机制，配合你阶段1 要装的 Redis Adapter，`io.to('user:123').emit()` 会自动跨进程找到用户。这是「用框架原生能力消除自研内存表」，最符合低技术债目标。


## 10.2【新增 R7】连接数限制 5000 是「单进程」语义，cluster 后会错乱

实测 `ChatWebSocketService` 的 `MAX_TOTAL_CONNECTIONS=5000` 是基于单进程 Map.size 判断的。cluster 后每个 worker 各自数自己的 Map，4 个 worker 各放 5000 = 实际能上 2 万，但每个 worker 仍以为自己只到 5000——限制语义失真。

修法：
- 若采用 R6-A（room 机制），本地 Map 取消，连接数上限改用 Socket.IO 的 `io.engine.clientsCount`（单进程）或在 Redis 里全局计数。
- 简单做法：把单进程上限按 `5000 / worker数` 分摊（如 4 进程各 1250），或直接用 Redis 全局计数器。**建议上线初期连接量远没到 5000，先按「单进程上限 = 总目标/进程数」分摊即可，不必上 Redis 全局计数。**

## 10.3【新增 R8】优雅停机要适配 cluster reload

实测 `app.js` 已有 `gracefulShutdown`（监听 SIGTERM/SIGINT，`server.close`）——这是好底子。但 cluster 下要确认：
- `pm2 reload`（零停机重启）会逐个重启 worker，每个 worker 收到 SIGINT。需确认 graceful shutdown 里**先停止接收新连接、通知在线 WebSocket 客户端重连、再关 DB/Redis**，避免 reload 瞬间大量连接被硬断。
- WebSocket 客户端（小程序/admin）要有重连逻辑（这归前端确认，见 7.5）。

修法：复用现有 `gracefulShutdown`，补充「关闭前 `io.close()` 让客户端触发重连」。改动小。

## 10.4【新增 R9】`trust proxy` 与限流取 IP（上线挂 Nginx/网关时）

实测 `app.js` 已 `app.set('trust proxy', true)`，限流登录按 IP（`keyGenerator:'ip'`）。上线如果前面挂了 Sealos 网关/Nginx，要确认拿到的是真实客户端 IP（`X-Forwarded-For`）而不是网关 IP，否则**所有用户会被当成同一个 IP 限流**。

修法：确认网关透传 `X-Forwarded-For`，`trust proxy` 配置与网关层数匹配。属上线前验证项，非代码大改。

# 十一、D6（WebSocket 跨进程推送）× 各类公司怎么做 × 你该怎么选

> 和 D1-D5 一样，把新增的 D6 也按「大厂 / 小公司 / 游戏公司 / 活动策划公司 / 虚拟物品·二手平台」五类拆开对比，再给落在你现有技术栈（Socket.IO 4 + ioredis）上的结论。

## 11.1 D6 到底在解决什么问题

你的系统多进程（cluster）后，用户 A 连在 worker-2，客服在 admin 端发消息时请求落在 worker-1。worker-1 要把消息推给用户 A，但 A 的连接不在 worker-1 上——**怎么找到 A、把消息送达**，这就是 D6。

本质是「**实时消息的跨进程/跨机器路由**」问题，所有做 IM、推送、实时通知的系统都要解决。

## 11.2 五类公司怎么做

| 公司类型 | 他们怎么做 | 技术选型 | 为什么 |
|----------|-----------|----------|--------|
| **大厂（腾讯/阿里/美团）** | 独立长连接接入层（如腾讯 TGW/接入网关），连接状态存分布式路由表，消息走自研 MQ 投递 | 自研接入层 + 路由中心 + MQ（如 RocketMQ） | 亿级长连接、多机房，必须把"连接管理"和"业务"彻底解耦 |
| **小公司** | Socket.IO + Redis Adapter，用 room 按用户/房间广播 | Socket.IO room + Redis pub/sub | 框架原生能力够用，零自研 |
| **游戏公司** | 网关层维护连接，按区/服路由；同区玩家在同一进程 | 长连接网关 + 分区（玩家不跨进程） | 用"分区"规避跨进程，状态强一致 |
| **活动策划公司** | 活动期临时推送，常用三方推送服务或 Socket.IO Adapter | 三方推送 / Socket.IO | 活动短期，不自建 |
| **虚拟物品/二手平台** | 聊天/出价通知用 Socket.IO room + Redis，关键消息落库可补偿 | Socket.IO room + Redis + DB 兜底 | 消息可丢但要能补偿（离线消息查库） |

**规律**：
- 只有**大厂**才自建"接入层 + 路由中心 + MQ"那一套，因为他们有亿级连接和多机房，且有专门团队维护。
- **中小团队 / 交易类 / 活动类**几乎清一色用 **Socket.IO 的 room 机制 + Redis Adapter**——这正是框架为多进程设计的标准答案。
- **游戏公司**的"分区"思路不适合你（你不是按区分服的游戏）。


## 11.3 两个候选方案的本质区别（R6-A vs R6-B）

| 维度 | R6-A：Socket.IO room 机制 | R6-B：在线表搬 Redis |
|------|---------------------------|----------------------|
| 思路 | 连接时 `socket.join('user:'+id)`；推送 `io.to('user:'+id).emit()` | userId→socketId 存 Redis Hash，推送前查 Redis |
| 跨进程 | **Redis Adapter 自动转发**，你不用管用户在哪个进程 | 查到 socketId 后，**仍要解决"它在别的进程"无法直接 emit** |
| 与 R3 关系 | 天然配合，R3 装好就生效 | R3 装了也用不上，等于绕开框架另搞一套 |
| 自研代码 | 删掉 connectedUsers/connectedAdmins，代码变少 | 保留并改造在线表逻辑，代码变多 |
| 技术债 | 低（用框架原生能力） | 高（自研路由，未来要自己维护一致性） |
| 离线消息 | 不影响（你已有 `ChatMessage` 落库，离线查库补偿） | 同样不影响 |

**R6-B 的致命问题**：就算你在 Redis 里查到「用户 A 的 socketId 是 xxx」，这个 socket 对象只存在于它所在的那个 worker 进程里，worker-1 拿到 socketId 也 **emit 不出去**——你最终还是得靠 Redis Adapter 转发。所以 R6-B 是"绕了一圈又回到 Adapter"，纯属多余自研。

## 11.4 给你的结论：D6 = R6-A（Socket.IO room 机制）

理由完全贴合你的诉求：
1. **落在现有技术栈**：你已经是 Socket.IO 4 + ioredis，room 是 Socket.IO 原生功能，配合阶段1 要装的官方 Redis Adapter，零额外自研。
2. **长期维护成本最低 / 技术债最少**：删掉自研的 connectedUsers/connectedAdmins 两个内存表和约 20 处查表逻辑，换成框架标准用法，代码更少、更不容易错。
3. **未上线、可一次性投入**：现在一次改对（约 20 处推送点改成 `io.to('user:'+id)`），以后上多机也天然支持，不用再返工。
4. **不动前端契约**：前端收到的事件名（`new_message`/`new_notification` 等）和数据格式完全不变，改的纯是后端"怎么找到这个用户"的内部实现。
5. **离线消息已有兜底**：你有 `ChatMessage`/`CustomerServiceSession` 落库 + `offline_messages` 重连补偿，即使瞬时推送丢失也能补回，正是"虚拟物品/二手平台"那一类的稳健做法——你已经做到了。

**这就是中小团队 + 交易/IM 场景的行业标准答案**，不是大厂那套重型接入层（你不需要），也不是游戏分区（不适用），而是「Socket.IO room + Redis Adapter」这个被验证过的甜点方案。

> 一句话：D6 选 **R6-A**（已拍板 ✅）。和 D1-D5 同一个哲学——**用你现有框架的原生能力解决问题，不自研、不引重型中间件、不动前端契约**。这是五类公司里"中小团队 + 交易/IM"的标准做法，长期维护成本最低、技术债最少。

# 十三、A/B/C 之外还有哪些方案（全景对比）

> A/B/C 只是「单机调参→单机深度优化→多机+Nginx」这**一条扩展线**上的三档。跳出这条线，业界还有至少 6 类不同思路。下面逐一说清，并标注**是否适合你**。
> 重要前提（实测）：你的环境是 **Sealos（K8s 底座）**——`.env` 里 `INTERNAL_BASE_URL=http://devbox2.ns-br0za7uc.svc.cluster.local:3000` 是标准 K8s 服务 DNS。所以「K8s 方案」对你不是从零搭建，而是平台已具备。

## 方案 D：Kubernetes HPA 水平自动扩缩容（替代「自己开多进程」）

| 维度 | 说明 |
|------|------|
| 思路 | 不在一个进程里开 cluster，而是把后端打成镜像，K8s 跑多个 Pod（每 Pod 1 个 Node 进程），按 CPU/QPS 自动增减 Pod |
| 谁这么做 | **大厂标准做法**；Sealos 本身就是 K8s，你已经在这个底座上 |
| 优点 | 弹性伸缩、故障自愈、滚动发布、与云平台一体 |
| 缺点 | 需要写 Deployment/Service/HPA 配置、镜像化、健康探针；运维心智比 PM2 高 |
| **适合你吗** | **中期很适合，现在不必**。你单机 32 核用 PM2 cluster 就够数万并发；等真要跨机器弹性时，K8s HPA 比方案C 的「手动多机+Nginx」更先进。**Sealos 环境下这是比方案C 更自然的演进方向**——建议把方案C 替换成「方案D：K8s HPA」作为远期选项 |

## 方案 E：Node.js 原生 cluster 模块（不用 PM2）

| 维度 | 说明 |
|------|------|
| 思路 | 代码里用 `require('cluster')` 自己 fork worker，而非 PM2 托管 |
| 优点 | 无需 PM2，进程模型完全自控 |
| 缺点 | 要自己实现重启、日志、监控、零停机——等于重造 PM2 的轮子 |
| **适合你吗** | **不适合**。你已经在用 PM2，PM2 的 cluster 模式就是封装了 cluster 模块还附带进程管理。自己写 = 凭空增加技术债。**排除** |

## 方案 F：Worker Threads / 线程池（榨单进程多核）

| 维度 | 说明 |
|------|------|
| 思路 | 用 `worker_threads` 把 CPU 密集任务丢到线程，主线程只跑 IO |
| 谁这么做 | 有重 CPU 计算的服务（图像处理、加密、大量计算） |
| 优点 | 单进程内利用多核，共享内存 |
| 缺点 | 只解决 CPU 密集瓶颈，不解决「多进程高可用」；编程复杂 |
| **适合你吗** | **基本不需要**。你的瓶颈是 IO（DB/Redis/网络），不是 CPU。唯一例外：`sharp` 图片处理是 CPU 密集，但它本身已用 libvips 多线程。**不作为主方案，个别重计算接口可局部用** |


## 方案 G：消息队列削峰（异步化）

| 维度 | 说明 |
|------|------|
| 思路 | 高并发写请求先入队（Redis Stream / BullMQ / RabbitMQ / Kafka），后台 worker 慢慢消费，结果用 WebSocket 回推 |
| 谁这么做 | 游戏抽奖、秒杀、电商下单的瞬时洪峰场景 |
| 优点 | 削平流量尖峰，保护 DB；天然解耦 |
| 缺点 | 用户体验从「同步出结果」变「排队异步」；引入队列中间件要维护；一致性更复杂 |
| **适合你吗** | **现在不需要，未来局部可用**。你的抽奖正确性已由 DB 行锁事务保证，当前并发同步处理足够。**只有当某个活动出现"开抢瞬间"的极端洪峰、把 DB 打到瓶颈时**，才值得对那一条路径做异步化。届时优先用 **Redis Stream + BullMQ**（基于你已有的 ioredis，不引 Kafka），而不是重型 MQ |

## 方案 H：读写分离 / 数据库层扩展（方案C 的 C3 单独拎出）

| 维度 | 说明 |
|------|------|
| 思路 | MySQL 主从，写走主库、读走从库；Sequelize 原生支持 `replication` 配置 |
| 优点 | 读吞吐翻倍，不改业务代码 |
| 缺点 | 主从延迟导致「写后立即读」可能读到旧数据；要维护从库 |
| **适合你吗** | **远期、数据涨上来才需要**。实测你最大表 `asset_transactions` 约 84MB、`users` 133 行，离 DB 读瓶颈极远。现在做纯属浪费。**列为"数据量级跃升后"的储备方案** |

## 方案 I：Serverless / FaaS（函数计算）

| 维度 | 说明 |
|------|------|
| 思路 | 把接口拆成函数，按调用次数付费，平台自动弹性到 0 或到万级 |
| 谁这么做 | 闲鱼后期部分链路、突发流量活动 |
| 优点 | 极致弹性、不用管服务器 |
| 缺点 | 冷启动延迟、长连接（WebSocket）不友好、有状态服务难落地、与现有 Express 单体架构不兼容 |
| **适合你吗** | **不适合**。你有 WebSocket 长连接、有定时任务、是有状态的单体，FaaS 会逼你大改架构=巨大技术债。**排除** |

## 方案 J：边缘缓存 / CDN / 反向代理缓存（读侧优化，与 A/B/C 正交）

| 维度 | 说明 |
|------|------|
| 思路 | 静态资源走 CDN，热点只读接口走 Nginx/Varnish 缓存，请求根本不到 Node |
| 优点 | 成本极低、对读多写少的接口效果拔群 |
| 缺点 | 只对可缓存的读接口有效，动态/个性化接口用不上 |
| **适合你吗** | **可顺手做，与方案B 不冲突**。你已有 `compression` + `ETag` 基础。图片已走 Sealos 对象存储（可挂 CDN）。`app_theme`、商品列表等高频只读接口可加边缘/Nginx 缓存。**作为方案B 的补充小优化，不是替代** |

## 14.1 ⚠️【新增 R11 · 高优先级】Socket.IO 用了 polling 传输，cluster 下需要 sticky session

实测 `services/ChatWebSocketService.js:101`：

```javascript
transports: ['websocket', 'polling']  // 同时支持 WebSocket 和 HTTP 长轮询
```

**问题**：Socket.IO 握手时，如果走 `polling`（HTTP 长轮询），**一次连接由多个 HTTP 请求组成**。PM2 cluster 默认用操作系统的负载均衡把请求轮流分给不同 worker。于是：握手第 1 个请求落 worker-1，第 2 个落 worker-3 → worker-3 不认识这个会话 → **报错 `Session ID unknown`，连接建立失败、反复重连**。

这是 Socket.IO 多进程**最经典的坑**，R3 的 Redis Adapter **解决不了它**（Adapter 管的是消息广播，不是握手粘连）。

**三个解法（需你拍 D7）**：

| 方案 | 做法 | 成本 | 评价 |
|------|------|------|------|
| **R11-A（推荐）强制 websocket-only** | 把 `transports` 改成 `['websocket']`，禁用 polling | 改 1 行 | WebSocket 本身是单条 TCP 长连接，天然不需要 sticky。现代浏览器/微信小程序都支持 WS。**最简单、零额外组件** |
| R11-B sticky session | PM2 cluster 前加一层 sticky 负载均衡（如 `@socket.io/sticky` 或 Nginx `ip_hash`） | 中 | 要引入 sticky 层，多一个维护点 |
| R11-C 交给 K8s/网关 | 上线在 Sealos 网关/Ingress 配 sticky（基于 cookie/ip） | 中 | 依赖部署层，本地 cluster 仍有问题 |

> **推荐 R11-A**：你的客户端是微信小程序（`wx.connectSocket` 原生 WebSocket）和管理后台浏览器，**都支持纯 WebSocket**，没必要保留 polling 兜底。改成 `['websocket']` 后，握手是单条 TCP 连接，PM2 cluster 下不需要 sticky，最干净。
> ⚠️ 唯一要确认：小程序端连接配置没有强制 polling（默认就是 WS，基本无忧）。这归前端确认项。


## 14.2 ⚠️【新增 R12 · 上线前必改】实测限流当前是关闭的

实测 `.env:99`：

**影响**：这在开发期没问题，但**上线开 cluster 前必须改成 `false`**，否则：
- 多进程放大了吞吐，没有限流 = 防刷形同虚设，恶意脚本可无限刷抽奖/登录。

### 纠正 1：限流**没有**"降级到内存限流"——它是 fail-open（出错直接放行）

- 前文位置：7.2「优雅降级（Redis 不可用时降级到内存限流）」、8.3「限流已实现 Redis 不可用时降级到内存限流」、14.2「限流是 Redis 实现，多进程共享」。

  - `catch(error)` 分支里是 `logger.error(...)` + `next()`，即 **Redis 出错时直接放行**（fail-open），**没有任何内存限流兜底**（全文件无 `Map`/`memoryStore`/`内存限流`）。
  - 全代码库 `grep` 也确认：**没有内存限流实现**。
- **影响**：
  - 这不改变"业务正确性不依赖 Redis"（钱仍由 DB 行锁兜底，见下），但**"Redis 挂掉后限流仍生效"是错的**——Redis 挂掉后**限流会完全失效（放行所有请求）**。

## 16.1 D8：限流"内存降级" —— 五类公司怎么做

> 本质问题：限流依赖 Redis，Redis 挂了之后怎么办？你现状（实测）是 **fail-open 全放行**（`RateLimiterMiddleware.js` 的 `catch` 里只 `next()`）。

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| **大厂（美团/腾讯/阿里）** | 多级限流：网关层（Nginx/Sentinel/自研）先扛一道，应用层 Redis 再扛一道，**两层独立**。Redis 挂了网关层还在。常用阿里 **Sentinel（限流降级框架）**，内置"本地+集群"两套统计 | 限流是稳定性生命线，绝不允许单点失效导致裸奔 |
| **小公司** | 多数直接用 `express-rate-limit` 默认**内存存储**，或 Redis 存储但**没想过降级**（跟你现在一样 fail-open） | 没遇到 Redis 挂的场景，先跑起来 |
| **游戏公司** | 抽奖/秒杀入口限流极严，常是 **Redis + Lua 原子计数 + 本地令牌桶兜底**双保险 | 一旦限流失效，机器人瞬间刷爆，直接资损 |
| **活动策划公司** | 活动期临时加限流，平时弱；常用网关或 CDN 层限流（请求不到应用） | 活动短期，靠边缘层削峰最省事 |
| **虚拟物品/二手平台** | 交易/出价接口限流 + 风控联动；Redis 挂了**宁可降级到本地限流也不放行**（fail-closed 倾向） | 防刷单/防薅羊毛，放行=被薅 |

**规律**：除了"小公司图省事"，**所有认真做防刷的团队都不接受 fail-open**——要么网关层兜底，要么应用层加本地降级。你的场景（抽奖 + 虚拟物品交易）天然是"被刷重灾区"，更接近游戏公司/二手平台那一档。

### 给你的结论：D8 = **建议做内存降级（R13）**，但做"轻量版"

理由（贴合你的约束）：
1. **你的业务最怕裸奔**：抽奖防刷、登录防爆破、聊天防刷屏——这三条限流一旦 Redis 挂掉就全开，恶意脚本可无限刷。这正是游戏/二手平台绝不允许的。
2. **落在现有技术栈、零新依赖**：在 `RateLimiterMiddleware.js` 的 `catch` 分支里，加一个**进程内 `Map` 滑动窗口**作兜底（约几十行），不引任何库。Redis 正常时走 Redis（跨进程精确），Redis 挂时退到本地（每进程独立、粗粒度，但有总比没有强）。
3. **cluster 下的诚实说明**：本地降级是"每 worker 各算各的"，4 进程实际放大 4 倍阈值——**这是粗粒度兜底，不是精确限流**。但它的目标只是"Redis 短暂抖动时不裸奔"，足够了。
4. **为什么不学大厂的"网关层限流"**：那需要在 Sealos 网关/Nginx 配限流规则，属运维层、且与"应用层防刷"是两件事。**可作为上线后补充**，但现在先把应用层 fail-open 这个洞补上，成本最低、收益最直接。

> 你也可以选"不做"——前提是接受"Redis 挂 = 限流全失效"，并用 D3′ 把 Redis 可用性拉满（Sentinel）来对冲。但**加几十行内存兜底**是性价比更高的做法，符合"一次做对、降低技术债"。
> 这条**完全落在现有 `RateLimiterMiddleware.js`**，0 新依赖。

## 16.2 D9：抽奖配置缓存失效 —— 五类公司怎么做

> 本质问题：cluster 后每个 worker 各持一份进程内 `Map` 抽奖配置缓存（实测 `CacheManager.js`/`ManagementStrategy.js`/`BudgetProviderFactory.js`），管理员改配置后，部分 worker 还读旧值。怎么让所有 worker 及时拿到新配置？

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| **大厂（美团/腾讯/阿里）** | 专门的**配置中心**（Apollo/Nacos/自研），配置变更通过长轮询/推送实时下发到所有实例；本地缓存 + 监听变更事件 | 成千上万实例，必须有统一配置治理与秒级下发 |
| **小公司** | 要么**短 TTL**（缓存几十秒自动过期），要么改配置后**删 Redis key**让下次回源 | 没精力上配置中心，够用就好 |
| **游戏公司** | 活动配置进 Redis（共享真相源），各节点读 Redis；或配置中心 + 本地缓存 + pub/sub 失效 | 活动配置频繁热更，要求秒级一致、不能重启 |
| **活动策划公司** | 配置化是核心，常用 Redis 存活动规则 + 后台改完即时生效 | "配置化而非代码化"是这类公司的命脉 |
| **虚拟物品/二手平台** | 交易规则/费率配置进 Redis 或 DB + 缓存，改完主动失效 | 规则错=钱算错，要求改完立即一致 |

**规律**：除了大厂上"配置中心"，中小团队清一色两条路——**①配置进 Redis 当共享真相源（读 Redis）；②短 TTL 自动过期**。pub/sub 失效是更高级的玩法，但**前提是你本来就有 pub/sub 消费链路**——而你**没有**（15.2 纠正 2 实测全库 0 处订阅）。

### 给你的结论：D9 = **方案①（配置缓存搬 Redis，复用 `BusinessCacheHelper`）**

理由（贴合你的约束）：
1. **这是你项目里真实已存在的成熟模式**：`BusinessCacheHelper` 全是 Redis `get/setex/del`，`SystemConfigService` 改配置后就是调 `invalidateSysConfig() → del(key)`。抽奖配置照这个模式搬过去，**和系统其它配置的缓存方式统一**，不是另起炉灶。
2. **天然解决 cluster 一致性**：缓存在 Redis（所有 worker 共享），管理员改配置 → `del(key)` → 下次任意 worker 回源读到新值。**没有"部分 worker 读旧值"问题**。
3. **比自建 pub/sub 债务低**：pub/sub 要每个 worker 起订阅、处理重连/丢消息/消费幂等——你现在 0 条订阅链路，新建一套＝凭空加维护面。**搬 Redis 复用现成 helper，几乎零新概念**。
4. **比短 TTL 更准**：短 TTL（方案②）改完后仍有最长 30-60 秒不一致窗口。抽奖配置（中奖率/库存/消耗积分）算错就是资损方向，**能做到"改完即一致"就别留窗口**。

> **方案②（短 TTL）作为退路**：如果某些缓存搬 Redis 的改造量超预期，可对**非资金敏感**的缓存先用短 TTL 过渡。但抽奖核心配置建议直接搬 Redis。
> 这条**复用现有 `BusinessCacheHelper`（Redis）**，0 新依赖、0 新中间件。

> 顺带澄清：前文 R4 写的"照搬 SystemConfigService 的 pub/sub 模式"是**错的**（实测无 pub/sub），D9 选方案①等于把 R4 修正到"项目真实拥有的能力"上——这本身就是降低技术债（不引入一套你没用过、要新维护的 pub/sub 机制）。

## 16.3 D3′：Redis 高可用（Sentinel）排上线前还是上线后 —— 五类公司怎么做

> 本质问题：实测 `REDIS_URL=redis://localhost:6379` 是**本机单点**。cluster 后所有 worker 依赖它，它挂 = 限流失效（见纠正1）+ 抽奖去重失效 + WebSocket 跨进程广播断 + 配置缓存（若按 D9 搬 Redis）读不到。要不要上线前就上高可用？

| 公司类型 | 他们怎么做 | 为什么 |
|----------|-----------|--------|
| **大厂（美团/腾讯/阿里）** | Redis Cluster（多主多从）或云托管（Tair/云数据库 Redis），多可用区、自动故障转移，**从不单点** | 亿级流量，Redis 是热数据真相源，宕机=大面积故障 |
| **小公司** | **单 Redis 起步**，挂了人工拉起；成长到一定规模上 **Sentinel（1主2从+哨兵）** | 成本与可用性折中，Sentinel 是性价比之选 |
| **游戏公司** | 抽奖/库存的 Redis **必上高可用**（Sentinel 或 Cluster），且和 DB 做最终一致对账 | Redis 存的是热数据，丢了影响发奖/库存 |
| **活动策划公司** | 活动期临时上托管 Redis 高可用，活动后降配 | 按需付费，活动才需要 |
| **虚拟物品/二手平台** | Redis 高可用 + **关键状态以 MySQL 为准**，Redis 丢了能从 DB 重建 | Redis 当加速层，DB 是真相源 |

**规律**：大厂/游戏公司"必上高可用"，小公司"先单点再 Sentinel"。**决定上线前还是上线后的关键，是"Redis 挂掉会不会造成资损"**——会，就上线前上；只影响可用性/体验，就可以上线后补。

### 你的关键优势（实测）：Redis 挂掉**不会算错钱**

- 业务正确性（抽奖扣分、交易、兑换）的真相源是 **MySQL + `LOCK.UPDATE` 行锁事务**，**不是 Redis**。
- Redis 只承担"加速 / 去重 / 限流 / WebSocket 广播 / 配置缓存"——**全是可用性/体验层，不是账本**。
- 所以 Redis 短暂挂掉：限流失效（若做了 D8 内存兜底则降级不裸奔）、去重短时失效（DB 行锁仍防重复扣分）、WebSocket 推送可能丢（有 `ChatMessage` 落库可补偿）、配置读不到（回源 DB）——**系统降级但不产生错账**。

### 给你的结论：D3′ = **上线前做"省心档加固"，Sentinel 列为"上线前评估、视投入决定"**

给你**分档**拍（你说愿意一次性投入，所以把"稳健档"也列清楚）：

| 档位 | 做什么 | 成本 | 建议 |
|------|--------|------|------|
| **省心档（上线前必做）** | ①Redis 开 AOF 持久化（已有 `scripts/monitoring/redis_persistence_setup.sh`）；②PM2/systemd 守护 redis-server 自动拉起；③确认 ioredis 超时降级（实测已有 `connectTimeout:10s/commandTimeout:5s`）；④配合 D8 内存限流兜底 | 几乎为 0 | ✅ **必做**，覆盖 95% 故障 |
| **稳健档（愿投入就上线前做）** | 上 **Redis Sentinel（1主2从+哨兵）**。`ioredis` **原生支持**——实测 `UnifiedRedisClient` 用 `new Redis(redisUrl, options)`，改成 `new Redis({ sentinels:[...], name:'mymaster' })` 即可，**业务代码一行不动** | 1 套 Sentinel 部署 + 改连接参数 | ⭐ 你愿投入则**上线前一起上**，一次做对 |
| **过度档（不要做）** | Redis Cluster 分片 | 高运维 | ❌ 数据量极小，分片纯浪费、制造债务 |

**关键判断**：因为你的**正确性不依赖 Redis**，Redis 单点是"可用性风险"而非"资损风险"。所以：
- 若你想**最省心上线**：省心档 + D8 内存兜底就够，Sentinel 放上线后。
- 若你想**一次做到位**（你提到愿一次性投入）：**省心档 + Sentinel 一起上线前做**。Sentinel 完全是"用 ioredis 原生能力长出来"的，不引新中间件、业务零改动，符合你"长期维护成本低"的目标——这是**值得的一次性投入**。

> 我的倾向：既然你明确"愿一次性投入、要一次做对、降低长期债务"，**D3′ 选"省心档 + 上线前上 Sentinel"**。理由：Sentinel 对 ioredis 是零业务改动，上线前一次配好，省去日后"单点挂了半夜爬起来拉 Redis"的运维债。
> 这条**复用 ioredis 原生 Sentinel 能力**，0 新中间件。

## 19.3 ⚠️【压测揪出的资损级 Bug，已修复】每日抽奖配额完全未生效

> 这是本次最重要的发现。前 18 节的「功能冒烟 + 单测」从未暴露它，因为单测多在服务层用事务回滚、且不校验 `affectedRows` 真实语义。**只有带 token 打真实 cluster 才暴露**。

### 现象（真实库实测）
用户 31 当日配额行 `used_draw_count=2 / limit_value=3`（只剩 1 次），但发一个 `draw_count=3` 的请求**竟返回 `200 抽奖成功`**，`lottery_draws` 当日记录从 2 直接涨到 5，而配额行 `used_draw_count` **仍停在 2 没动**——即每日上限形同虚设，可无限超额抽奖。

### 根因（逐层定位）
`models/LotteryUserDailyDrawQuota.js` 的 `tryDeductQuota` 原子扣减写法：

```javascript
const [affectedRows] = await sequelize.query(`UPDATE ... WHERE used+:dc <= limit ...`,
  { type: sequelize.QueryTypes.UPDATE })
if (affectedRows === 0) { /* 判定配额不足 */ }
```

实测 `sequelize.query(..., { type: QueryTypes.UPDATE })` 的返回是 **`[resultSetHeader, affectedCount]`**，受影响行数在下标 `[1]`；而 `const [affectedRows]` 取的是下标 `[0]`（恒为 `undefined`）。于是 `affectedRows === 0` **永远不成立** → 即使 `WHERE used+draw_count<=limit` 不满足、实际更新 0 行，代码仍打印「配额扣减成功」并放行抽奖。日志实锤：`配额扣减成功 {draw_count:3, used:2}`（2+3>3 本应失败）。

### 同模式连带排查（同一类 Bug 一次清干净）
全库搜 `QueryTypes.UPDATE` 后逐处核对，发现同样「取错下标」共 3 处，均已修正为取 `[1]`：

| 文件 | 方法 | 危害 | 修复 |
|------|------|------|------|
| `models/LotteryUserDailyDrawQuota.js` | `tryDeductQuota` | **每日抽奖配额不生效**（资损/超额） | ✅ 取 `updateResult[1]` |
| `services/UnifiedLotteryEngine/pipeline/stages/SettleStage.js` | `_deductStock`(约541行) | **奖品库存超卖**（`INSUFFICIENT_STOCK` 永不触发） | ✅ 取 `stockUpdateResult[1]` |
| `services/UnifiedLotteryEngine/pipeline/stages/SettleStage.js` | `_updateUserQuota`(约1188行) | 活动级配额「已更新」debug 日志失真（非资损） | ✅ 取 `quotaUpdateResult[1]` |

> 另两处 `LotteryUserExperienceState.js` 的 `QueryTypes.UPDATE` 是 fire-and-reload（不读返回值），不受影响，无需改。

### 连带修复2：引擎错误透传（让 403 真正到达前端）
修配额后复测，超限请求被拦了但返回 `500 ENGINE_FAILED`，而非应有的 `403 DAILY_DRAW_LIMIT_EXCEEDED`。根因：`UnifiedLotteryEngine.execute_draw` 的 catch 无条件 `throw new BusinessError(..., 'ENGINE_FAILED', 500)`，把带 `statusCode=403/errorCode` 的业务错误压平成 500，导致 `routes/v4/lottery/draw.js` 的 403/409 分支无法命中。已改为：`BusinessError` 与带 `statusCode/errorCode` 的业务错误**原样透传**，仅未知系统错误才包 `ENGINE_FAILED`。
