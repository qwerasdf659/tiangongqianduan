## 九、设计决策

### ✅ 决策 1：approved_at + 兑换订单事件表 — 方案 C（混合）

**已定，无需拍板。**

项目已有 8+ 个事件/变更记录表（`user_status_change_records`, `user_role_change_records`, `admin_operation_logs`, `merchant_operation_logs`, `content_review_records`, `item_instance_events_legacy`, `lottery_draw_decisions`, `batch_operation_logs`），兑换订单不用这个模式是项目内部不一致。

**执行**：

**1) 新增 `approved_at` 列**（migration）：

- `ALTER TABLE exchange_records ADD COLUMN approved_at DATETIME DEFAULT NULL COMMENT '审核通过时间' AFTER rejected_at`
- `ExchangeRecord` Model 添加 `approved_at` 字段
- `CoreService.updateOrderStatus()` 的 `timestampMap` 补充 `approved: { approved_at: BeijingTimeHelper.createDatabaseTime() }`
- `marketOrderView` 和 `adminMarketOrderView` 追加 `'approved_at'`

**2) 新建 `exchange_order_events` 表**（migration + Model），复用 `user_status_change_records` 模式：

```
exchange_order_events
├── event_id          BIGINT PK AUTO_INCREMENT
├── order_no          VARCHAR(50) FK → exchange_records.order_no
├── old_status        ENUM(9态)
├── new_status        ENUM(9态)
├── operator_id       INT FK → users
├── operator_type     ENUM('user','admin','system')
├── reason            VARCHAR(500)
├── metadata          JSON
├── idempotency_key   VARCHAR(100) UNIQUE
├── created_at        DATETIME
索引：order_no, operator_id, (new_status, created_at)
```

**3) 写入时机**（事务内同步）：

| Service 方法 | 事件 | operator_type |
|-------------|------|:------------:|
| `exchangeItem()` | null → pending | user |
| `updateOrderStatus()` | old → new | admin |
| `cancelOrder()` | pending → cancelled | user |
| `rejectOrder()` | pending → rejected | admin |
| `confirmReceipt()` | shipped → received | user |
| `rateOrder()` | received → rated | user |
| `autoConfirmShippedOrders()` | shipped → received | system（operator_id 取 `SYSTEM_DAILY_JOB_USER_ID=11021`） |

### ✅ 决策 2：虚拟资产退还 — 方案 B（独立方法）

**已定，无需拍板。**

本项目定位 = 积分/材料兑换商城（类招行积分/原神商店），虚拟资产退还是即时同步的：`BalanceService.changeBalance(+amount)` 一行代码，在事务内完成。

**执行**：

- `CoreService.cancelOrder(user_id, order_no, { transaction })` — 用户取消（仅 pending），退还 pay_amount
- `CoreService.rejectOrder(order_no, operator_id, remark, { transaction })` — 管理员拒绝（仅 pending），退还 pay_amount，记录 admin_remark + rejected_at
- `CoreService.updateOrderStatus()` 保持纯状态变更，不触碰资产

退还调用模式（与 `exchangeItem()` 扣减完全对称）：

```javascript
const BalanceService = require('../asset/BalanceService')
await BalanceService.changeBalance({
  user_id: order.user_id,
  asset_code: order.pay_asset_code,
  delta_amount: +order.pay_amount,
  idempotency_key: `exchange_refund_${order.order_no}`,
  business_type: 'exchange_cancel_refund',
  meta: { order_no: order.order_no, reason: 'user_cancel' 或 'admin_reject' }
}, { transaction })
```

BalanceService 的幂等键机制（`idempotency_key: exchange_refund_{order_no}`）防止重复退还。

### 原始问题现象

```
GET https://omqktqrtntnn.sealosbja.site/api/v4/backpack/exchange/orders?page=1&page_size=10

HTTP 404
{
  "success": false,
  "code": "NOT_FOUND",
  "message": "接口不存在: GET /api/v4/backpack/exchange/orders?page=1&page_size=10"
}
```

### 原始根因判断

前端判断"后端未实现兑换订单相关路由"。

**审查修正**：判断正确。后端 Service 层已实现全部查询和大部分操作逻辑，但 backpack 域路由文件（622 行）未挂载订单相关端点。Service 代码可直接复用，路由层补全即可。shop 域有完整路由实现（303 行）但已标记迁移、不再挂载。

### 原始状态枚举（8 态）

前端列出 8 种状态，后端实际有 9 种（多 `completed`）。小程序需在 `ORDER_STATUS_MAP` 中补充 `completed: { label: '已完成', ... }`。

### 原始 item_snapshot 假设

前端假设 `{ name, image_url }`。

**审查修正**：后端实际写入 `{ exchange_item_id, item_name, description, cost_asset_code, cost_amount }`。无 `name` 字段（是 `item_name`），无 `image_url` 字段。决策 3 方案 A+C 已定后，快照将加入 `image_url`，DataSanitizer 脱敏后保留 `{ item_name, description, image_url }`。
