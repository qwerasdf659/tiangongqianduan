# 管理员/超管点击「审核详情」误弹"需要店员及以上权限" — 微信小程序前端对接文档

> 提出方：后端数据库项目
> 接收方：微信小程序前端项目
> 基准：以后端为唯一权威，前端零映射直读后端字段（snake_case）；前端本地权限判断必须与后端 role_level 规则对齐
> 核查方式：Node.js 直连真实库 `restaurant_points_dev`（`dbconn.sealosbja.site`，读 `.env`，非备份）+ 直读后端实际代码（`middleware/auth.js`、`routes/v4/shop/audit/index.js`、`routes/v4/shop/consumption/query.js`、`routes/v4/shop/staff/index.js`）+ 全仓库文案检索
> 日期：2026-06-24（北京时间）
> 结论先行：**这是微信小程序前端的问题。** 后端对 `role_level >= 100` 的管理员/超管在商家域接口是**无条件放行**的；"需要店员及以上权限"这句文案**后端代码中根本不存在**，是小程序前端自己的本地前置拦截误伤了管理员账号。

---

## 一、问题现象

超管账号 **13612227910** 登录小程序首页，点击「审核详情」按钮时，弹出 toast：**"需要店员及以上权限"**，无法进入审核页。

（见用户反馈截图：欢迎 13612227910，可用积分 89,610，点击「审核详情 →」即弹出该提示。）

---

## 二、根因核查（连真实库 + 直读后端代码，逐条验真）

### 2.1 该账号实测就是最高权限（连真实库）

实测真实库 `restaurant_points_dev`：手机号 **13612227910 = `user_id = 32`**，挂两个角色：

| role_id | role_name | role_level | permissions |
| --- | --- | --- | --- |
| 2 | admin | 100 | `{"*":["*"]}` |
| 254 | super_admin | 110 | `{"*":["*"]}` |

后端取角色最大值 → **`role_level = 110`**，权限通配 `*:*`。这是系统最高权限账号。

### 2.2 后端对管理员是「无条件放行」（直读代码）

商家域权限中间件 `requireMerchantPermission`（`middleware/auth.js` L1208-1220）第一条规则就是放行管理员：

```javascript
// 2. 超级管理员跳过所有检查
if (req.user.role_level >= 100) {
  return next()
}
```

即只要请求真正打到后端，`role_level=110` 的这个账号**必然被放行**，后端**不可能**对它返回"权限不足"。

> 补充：基于角色等级的另一个中间件 `requireRoleLevel`（L751-796）权限不足时返回的 `code` 是 `INSUFFICIENT_LEVEL`、`message` 是"需要权限等级 N 以上"；`requireMerchantPermission` 权限不足返回 `INSUFFICIENT_MERCHANT_PERMISSION`、"权限不足，需要 xxx 权限"。**两者都不会输出"需要店员及以上权限"这种文案。**

### 2.3 "需要店员及以上权限"这句文案，后端根本没有（全仓库检索）

对后端全仓库（排除 `node_modules`、`admin`）检索"店员及以上 / 店员以上 / 需要店员"等关键词：

```
后端无此文案
```

结论：这句提示**只可能来自小程序前端自己的代码**。后端没有产出它的任何路径。

### 2.4 "审核详情"对应的后端接口本身也不会拦管理员

无论小程序「审核详情」实际调用的是哪个后端接口，相关接口对管理员都是放行/可见的：

| 可能对应的后端接口 | 鉴权 | 对 13612227910(role_level=110) 的结果 |
| --- | --- | --- |
| `GET /api/v4/shop/consumption/detail/:id`（消费记录详情） | `authenticateToken`，服务层按 `role_level>=100` 判 `admin_privilege` 放行（`query.js` L96-104） | ✅ 放行 |
| `GET /api/v4/shop/audit/logs`（商家操作日志/审计） | `requireMerchantPermission('staff:read')` | ✅ role_level≥100 直接 next() 放行 |

→ 即后端任一候选接口都不会对该账号返回"权限不足"。

## 三、定位结论：问题在微信小程序前端的「本地前置拦截」

小程序在点击「审核详情」时，**先在前端本地做了一次权限判断**，且这套判断只认"门店店员/店长身份"（对应后端 `merchant_staff` role_level=20 这一档），没有给"管理员/超管（role_level≥100）"开放行通道：

- 该超管账号**不是任何门店的在职店员**（不在 `store_staff` 绑定里，它是平台级管理员，不挂具体门店）；
- 于是前端本地校验判定"你不是店员"→ 直接弹"需要店员及以上权限"，**请求根本没发给后端**；
- 而后端本会因 `role_level=110` 放行。前端的本地判断与后端规则**不一致**，造成误伤。

一句话：**前端用"是不是门店店员"来卡，漏掉了"管理员/超管应无条件放行"这条后端既定规则。**

---

## 四、修复方案（纯小程序前端改动，后端零改动）

> ✅ **最终决策（2026-06-24 已拍板）：采用方案 B —— 去掉前端本地硬拦截，权限判定全部交给后端，前端按后端返回处理。**
> 理由：权限判定只保留后端一处真相源，前端不再维护一套易与后端漂移的本地规则，彻底根治"前端本地判断与后端 role_level 规则不一致"导致的误伤，最贴合本项目"以后端为唯一权威、前端零映射"的既定原则。

### 4.1 方案 B（已采用）：去掉本地硬拦截，让后端判定

**核心改动**：删除「审核详情」点击时那段"判断是不是门店店员、不是就弹『需要店员及以上权限』"的本地前置拦截；改为点击即发请求，按后端统一响应体的 `success` / `code` 处理。

```javascript
// 伪代码：点击「审核详情」→ 不做本地权限预判，直接请求后端
async function onTapAuditDetail() {
  const res = await requestAuditDetail() // 调对应后端接口，返回后端统一响应体 {success, code, message, data}

  if (res.success) {
    return goAuditDetail(res.data) // 正常进入审核详情
  }

  // 权限不足：用后端 message，不要前端写死"需要店员及以上权限"
  if (res.code === 'INSUFFICIENT_MERCHANT_PERMISSION' || res.code === 'INSUFFICIENT_LEVEL') {
    return wx.showToast({ title: res.message || '权限不足', icon: 'none' })
  }

  // 未登录/登录过期：引导重新登录
  if (res.code === 'UNAUTHENTICATED') {
    return redirectToLogin()
  }

  // 其它错误：通用提示
  wx.showToast({ title: res.message || '操作失败，请重试', icon: 'none' })
}
```

**按后端返回处理对照表**：

| 后端返回 | 前端处理 |
| --- | --- |
| `success === true` | 正常进入审核详情、渲染 `data` |
| `success === false` 且 `code === 'INSUFFICIENT_MERCHANT_PERMISSION'` 或 `'INSUFFICIENT_LEVEL'` | 提示权限不足，**文案用后端 `message`**，不要前端写死"需要店员及以上权限" |
| `success === false` 且 `code === 'UNAUTHENTICATED'`（HTTP 401） | 引导重新登录 |
| 其它 `success === false` | 用后端 `message` 做通用错误提示 |

**落地要点**：
- **删除**「审核详情」处基于"门店店员身份"的本地权限预判分支（这正是误伤超管 13612227910 的根源）；
- 权限是否通过**完全以后端响应为准**：`role_level=110` 的管理员/超管点击后，后端 `requireMerchantPermission` 会 `return next()` 放行 → `success=true` → 前端正常进入，不再误弹；
- 前端**不要**写死"需要店员及以上权限"文案，权限不足时一律展示后端下发的 `message`，避免前后端文案/规则漂移；
- 全程零映射直读后端字段（`success`/`code`/`message`/`data`）。
