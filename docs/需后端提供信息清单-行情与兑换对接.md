## 一、阻塞级（影响功能正确性，必须后端确认）

### 1. `GET /api/v4/marketplace/analytics/history` 的真实响应结构（最高优先级）

**问题本质（纯文本数据流图）**：

```
                后端 analytics/history
                        │
        ┌───────────────┴────────────────┐
        ▼                                 ▼
 前端消费者 A                         前端消费者 B
 listing-detail.ts                   price-chart.ts（90d 周期分支）
 _loadPriceHistory()                 _fetchData()
        │                                 │
 期望: data 是数组 或 data.history       期望: data.data_points[] 且每项含 time_bucket
 每项字段: date / avg_price /            字段: time_bucket / avg_price / ...
          min_price / max_price /
          trade_count
        │                                 │
        ▼                                 ▼
   按 {{item.date}} 等渲染            按折线图 time_bucket 取时间轴
```

**冲突说明**：同一个 `analytics/history` 接口，前端两处消费者**假设了两种不同的返回结构**。
《求证文档》详列了 `trend/volume/summary/recent-trades` 四个接口的真实字段，但**没有**给出 `analytics/history` 的真实返回结构。前端无法自行推测，必须由后端给出权威结构。

**请后端提供**：
1. `GET /api/v4/marketplace/analytics/history?asset_code=red_core_shard&days=90` 的**真实响应 JSON 样例**（用真实库 `restaurant_points_dev` 实测，非预设）。
2. 顶层 `data` 到底是**数组**，还是**对象**（如 `{ history: [...] }` 或 `{ data_points: [...] }`）？
3. 数组每项的**真实字段名**：是 `date` 还是 `time_bucket`？数值字段（avg_price/min_price/max_price/trade_count）是数字还是字符串（原生 SQL 聚合常返回字符串）？
4. 该接口与 `price/trend` 的返回结构**是否一致**？若一致，前端将统一按 `data_points[].time_bucket` 处理；若不同，请明确两者差异。

> 拿到后，前端会把 `listing-detail.ts` 与 `price-chart.ts` 两处统一到后端真实结构，并同步修正 `typings/api.d.ts` 的 `PriceHistoryPoint`。**当前前端未擅自改这两处的解析，避免基于猜测引入错误。**

## 二、确认级（前端已按文档落地，需后端最终确认无误）

### 2. 5 个行情接口确已改为 `optionalAuth`（匿名可调）

前端已把以下方法 `needAuth: true → false`：
`getPriceTrend / getVolumeTrend / getPriceSummary / getRecentTrades / getPriceHistory`

**请后端确认**：以下 5 个接口在生产/测试环境**确实已是 `optionalAuth`**（不带 Authorization 返回 200，而非 401）：

```
GET /api/v4/marketplace/price/trend
GET /api/v4/marketplace/price/volume
GET /api/v4/marketplace/price/summary
GET /api/v4/marketplace/price/recent-trades
GET /api/v4/marketplace/analytics/history
```

> 若后端某个接口尚未改 `optionalAuth`，前端匿名请求会拿到 401。APIClient 已对 `needAuth:false` 的 401 做静默处理（不清 Token、不弹窗、只抛错由调用方兜底），不会影响已登录体验，但匿名用户会看到空态。

### 3. `recent-trades` 返回结构与字段

前端已按文档把入参 `limit → page_size`，返回按 **`data` 即数组**解析，单条字段按：
`trade_order_id / gross_amount / fee_amount / net_amount / settlement_asset / completed_at / listing_kind / offer_asset_code / offer_amount / offer_item_template_id`

**请后端确认**：
1. 上述字段名与 `data` 直接为数组的结构是否与真实返回一致。
2. 数值字段（gross_amount 等）确为字符串（前端已做 `Number()` 转换）。
3. 该接口**确实不返回**买卖双方昵称 / user_id / 订单号明文（前端据此判断匿名开放无脱敏顾虑）。若未来要展示对手方信息，属于**可能泄密点**，需先由业务方决策，前端不擅自展示。

### 4. `price/volume` 成交量字段名

前端成交量柱图已改为取 `total_star_stone_volume` / `total_item_volume`（不再用 `total_volume`）。

**请后端确认**：`GET /api/v4/marketplace/price/volume` 的 `data_points[]` 真实字段名确为 `total_star_stone_volume` 与 `total_item_volume`，且时间轴字段为 `time_bucket`。

### 5. 兑换订单 `pay_asset_name` 注入

前端兑换订单列表 / 详情已改为展示 `pay_asset_name || pay_asset_code`（中文优先，缺失兜底 code），列表按 `data.orders` 解析。

**请后端确认**：
1. `GET /api/v4/exchange/orders` 列表每条订单已注入 `pay_asset_name`（服务层 JOIN `material_asset_types`）。
2. 订单详情 `GET /api/v4/exchange/orders/:order_no` **是否同样注入** `pay_asset_name`？（《求证文档》4.1 称列表与详情两处都调用了 `_attachPayAssetNames`，前端已据此在详情页也用 `pay_asset_name`，请确认详情接口确实返回该字段，否则前端会自动 fallback 到 `pay_asset_code`。）

## 三、待确认的「分类计数」缺口（非阻塞，前端已留兜底）

### 6. 兑换订单各状态分类计数

前端兑换订单页顶部需要展示「待收货 / 待评价 / 处理中」各状态的数量徽标。
当前 `GET /api/v4/exchange/orders` 仅返回 `{ orders, pagination }`，`pagination.total` 只是**当前筛选条件下的总数**，**无法**一次性拿到各状态分类计数。

**前端现状**：只能在切到对应 Tab 时用该 Tab 的 `pagination.total` 更新单个数字，无法在「全部」Tab 下同时显示各状态数量。

**请后端评估**：是否可提供一个轻量的**订单状态计数接口**（如 `GET /api/v4/exchange/orders/stats` 返回 `{ pending, shipped, received, ... }`）？
若业务上不需要徽标，请明确告知，前端将移除该统计位，避免显示不准确的数字。

> 这是**业务需求层面的决策**，请业务方/后端确认是否需要此能力，前端不擅自在前端用多次请求拼凑（会造成额外请求且数据可能不一致）。
