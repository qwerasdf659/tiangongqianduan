# 数据库重置方案（方案 1：业务数据重置）

> 生成时间：2026-05-31
> 项目：restaurant-lottery-system-v4-unified（后端数据库项目）
> 目标库：`restaurant_points_dev`（`.env` 指定，Sealos 托管 MySQL 8.0.30）
> 适用环境：Sealos Devbox 单环境（开发/测试/生产统一连此真实库）
> 本方案性质：**只清"业务流水/交易/物品实例"数据，保留表结构 + 用户 + 角色 + 字典 + 活动配置**。不动表结构、不重建库。

---

## 0. 一句话定义

把测试期积累的脏业务数据（脏 item、test 充值流水、对账垃圾、抽奖/交易/客服记录等）按**外键拓扑顺序**在**单事务**内清空，并把账户余额归零；**保留**所有表结构、用户账号体系、角色权限、系统字典、抽奖/兑换/商品等配置数据。重置后系统可直接用 `13612227930 / 123456` 登录，对账天然 PASS。

---

## 1. 为什么选方案 1（而不是全库重建 / DROP DATABASE）

| 维度 | 方案1 业务数据重置（本方案） | 方案2 全库重建 | 方案3 DROP DATABASE |
|------|------------------------------|----------------|---------------------|
| 破坏范围 | 仅业务流水数据 | 结构+数据全删重建 | 整库删除重建 |
| 用户/角色 | ✅ 保留，可直接登录 | ❌ 需先补"用户种子"否则空库无法登录 | ❌ 同左 |
| 表结构 | ✅ 不动 | 回到 baseline | 回到 baseline |
| 可逆性 | 高（删前整表备份） | 低（undo:all 不可逆） | 极低 |
| Sealos 风险 | 低（纯 DML） | 高（FOREIGN_KEY_CHECKS=0 权限不确定） | 高 |
| 复用现有资产 | ✅ 复用 `DataManagementService.DELETE_TOPOLOGY` | 复用迁移 | 无 |
| 推荐度 | ⭐ 推荐 | 结构损坏时才用 | 不推荐 |

**关键实测事实（决定不能轻易全库重建）**：
- baseline 迁移（115 张表）**0 条 INSERT**；seeder 对 `users` 只用 `rawSelect('users', {mobile:'13612227930'})` **查找、不创建**。
- 即 `users`/`roles` **没有任何种子来源**，当前 133 users / 11 roles / 测试管理员（user_id=31）是历史产生的。
- 因此**裸跑 `npm run db:reset` 会得到无用户、无法登录的空库**。方案 1 规避了这个坑。

## 2. 数据分级（依据项目真实代码 `services/DataManagementService.js`）

### 2.1 要清空的数据（业务流水/交易/物品实例）

按 `DELETE_TOPOLOGY`（项目已基于 174 条外键排好的拓扑序）从叶子表到父表清空。核心分组：

| 互锁组 | 涉及表 | 清空要求 |
|--------|--------|----------|
| **材料资产互锁** | `account_asset_balances` ↔ `asset_transactions` | **整组一起清**；余额表清空（等价归零） |
| **物品三表互锁** | `items` ↔ `item_ledger` ↔ `item_holds` | **整组一起清** |
| 抽奖记录 | `lottery_draws` / `lottery_draw_decisions` / `lottery_*_quota` / `lottery_*_state` / `lottery_*_metrics` / `lottery_alerts` / `lottery_simulation_records` / `lottery_presets` | 清空 |
| 交易/兑换 | `trade_orders` / `redemption_orders` / `market_listings` / `market_price_snapshots` / `exchange_records` / `bid_records` / `bid_products` / `consumption_records` | 清空 |
| 客服 IM | `chat_messages` / `customer_service_sessions` / `customer_service_*` | 清空 |
| 广告日志 | `ad_*_logs` / `ad_report_daily_snapshots` / `ad_dau_daily_stats` / `ad_campaigns` / `ad_creatives` | 清空 |
| 通知/反馈/审计 | `user_notifications` / `admin_notifications` / `feedbacks` / `*_operation_logs` / `reminder_history` / `risk_alerts` | 清空 |
| 幂等/会话 | `api_idempotency_requests` / `authentication_sessions` / `websocket_startup_logs` | 清空 |

### 2.2 要保留的数据（结构 + 账号 + 配置）

| 类别 | 表 | 说明 |
|------|----|------|
| 账号体系 | `users` / `roles` / `user_roles` / `accounts` | **保留**（accounts 余额由互锁组清空后归零） |
| 系统字典 | `system_dictionaries`（412 条）/ `system_settings` | 保留（中文展示、枚举值来源） |
| 抽奖配置 | `lottery_campaigns`（4 个）/ `lottery_campaign_prizes` / `prize_definitions` / 各 `lottery_*_config` | 保留 |
| 资产/物品定义 | `material_asset_types` / `rarity_defs` / `item_templates` / `asset_group_defs` / `material_conversion_rules` | 保留 |
| 商城/类目 | `exchange_items` / `categories` | 保留 |
| 媒体 | `media_files` | 保留（按需，图片走 Sealos 对象存储） |
| 全部表结构 | 121 张表 | 不动 |

> 判定原则（符合项目规则）：**配置类表（奖品定义、活动配置、模板、字典）可直接保留/改**；**余额/物品持有/锁定状态的互锁表必须整组清空**，不能只清一半导致对账对不上。

## 3. 执行步骤（每步可独立验证，强制先备份）

### 阶段 0：前置安全（强制）

1. **确认连的是目标库**：`node` 读 `.env`，确认 `DB_NAME=restaurant_points_dev`、`DB_HOST=dbconn.sealosbja.site`。
2. **全量备份不可再生数据**（无 mysql 客户端，用 Node.js 导出到 `backups/`，已 gitignore）：
   - 必备份：`users` / `roles` / `user_roles` / `accounts` / `system_dictionaries` / `lottery_campaigns` / `lottery_campaign_prizes` / `material_asset_types` / `item_templates`。
   - 建议全表备份：把所有要清空的业务表也 dump 一份 JSON，便于回溯。
3. **停止写入**：建议重置期间 `npm run pm:stop` 停后端 cluster（避免清空过程中有新写入），重置后再 `npm run pm:start:pm2`。

### 阶段 1：执行清空（单事务 + FK 拓扑顺序）

新建脚本 `scripts/maintenance/reset_business_data.js`（复用 `DataManagementService.DELETE_TOPOLOGY`）：

- 单 `sequelize.transaction()` 包裹全过程，任一步失败整体回滚。
- 按 `DELETE_TOPOLOGY` 第 1→5 层顺序 `DELETE FROM`（先子表后父表）。
- **第 5 层中的 `lottery_campaigns` / `item_templates` / `ad_slots` / `ad_target_zones` 属配置表，重置时跳过**（保留）；只清 `asset_transactions` / `account_asset_balances`。
- 提供 `--dry-run`：先打印每表将删行数，不实际删除。
- 提供 `--confirm`：实际执行（无此参数默认 dry-run）。
- 清空 `account_asset_balances` 即等价余额归零（无需额外 UPDATE）。
- 全程结构化日志，记录每表删除行数。

```bash
# 先预演（只看将删多少行，不动数据）
node scripts/maintenance/reset_business_data.js --dry-run
# 确认无误后执行
node scripts/maintenance/reset_business_data.js --confirm
```

### 阶段 2：重置后验收（红线，全过才算成功）

1. **对账全 PASS**：`node scripts/reconcile-items.js`
   - 物品守恒 / 持有者一致 / 铸造一致 / 余额一致 / 全局守恒 全部 PASS（业务流水清空后天然平衡）。
2. **健康检查**：`curl /health` 返回 `SYSTEM_HEALTHY`、database+redis connected。
3. **登录可用**：用 `13612227930 / 123456` 登录成功（验证用户/角色/字典保留完好）。
4. **配置完好**：抽奖活动、奖品、资产类型、字典仍在（管理后台可见）。
5. **Redis 缓存清理**：重置后清掉业务相关 Redis 缓存键（抽奖配置/余额/限流计数），避免读到旧缓存——`redis-cli` 按前缀 SCAN 删除，或重启后端 cluster 让缓存自然失效。

## 4. 风险与回滚

| 风险点 | 处置 |
|--------|------|
| 误删配置数据 | 脚本明确区分"清空表"与"保留表"白名单；dry-run 先验证 |
| 清空过程中有并发写入 | 重置前 `pm:stop` 停服 |
| 互锁组只清一半导致对账不平 | 脚本保证三表/两表整组清空，单事务原子性 |
| 清错库 | 阶段0 强制确认 `DB_NAME` |
| 想恢复 | 阶段0 的 JSON 备份可逐表 `bulkInsert` 回填 |

**回滚方式**：若清空后发现问题，用阶段 0 备份的 JSON，按 FK 拓扑**反向**（父表→子表）`bulkInsert` 回填。因此阶段 0 的全表备份不可省略。
