## 一、后端数据库项目（权威源）

**技术栈：** Node.js 20+ / Express 4.21 / Sequelize 6.37 / MySQL (mysql2) / Redis (ioredis 5.4) / JWT + RBAC

**核心架构特征：**

- **事务边界治理：** 路由层 `TransactionManager.execute()` 统一管理，服务层 `assertAndGetTransaction` 强制外部事务，服务层禁止自建事务
- **统一账本：** `BalanceService.changeBalance()` 是所有资产变动的唯一入口，双录记账（每笔变动都有 counterpart 对手方流水）
- **幂等性：** `idempotency_key` 字段唯一约束 + HTTP Header `Idempotency-Key`
- **服务获取：** `req.app.locals.services.getService('snake_case_key')`，禁止直接 require models
- **响应格式：** `res.apiSuccess(data, message, code)` / `res.apiError(message, code, details, httpStatus)`，标准体 `{ success, code, message, data, timestamp, version, request_id }`
- **项目规模：** 197 路由文件、120 模型、202 服务、259 测试、76 个迁移脚本
- **数据库连接：** Sealos 托管 MySQL，`restaurant_points_dev`，timezone `+08:00`

## 二、两套规则系统的完整对比（代码级）

### 2.1 ExchangeRateService（实际在跑的）

- **服务文件：** `services/exchange/ExchangeRateService.js`（569 行）
- **服务键名：** `exchange_rate`
- **数据表：** `exchange_rates`（7 条数据，6 active / 1 paused）
- **真实交易量：** 169 笔
- **汇率表达：** `rate_numerator / rate_denominator`，计算 `FLOOR(from × num ÷ den)`
- **状态：** 三态 status（active/paused/disabled）
- **版本化：** 无（直接 UPDATE），但有 `effective_from` + `effective_until` + `priority` 字段（目前全 NULL 没用过）

- **手续费：** 只有 `fee_rate`，固定扣目标资产，无最低手续费

- **风控：** 无
- **business_type：** `exchange_rate_debit` / `exchange_rate_credit` / `exchange_rate_fee`
- **系统账户：** 扣减和入账都走 `SYSTEM_EXCHANGE`，手续费走 `SYSTEM_PLATFORM_FEE`

### 2.2 AssetConversionService（全禁用的）

- **服务文件：** `services/AssetConversionService.js`（610 行）
- **管理服务：** `services/MaterialManagementService.js`（615 行）
- **服务键名：** `asset_conversion`（执行）、`material_management`（管理）
- **数据表：** `material_conversion_rules`（2 条数据，全部 `is_enabled=0`）
- **真实交易量：** 578 笔（但规则全禁用了，是历史数据）
- **汇率表达：** `from_amount : to_amount` 整数比，计算 `floor(from ÷ rule.from × rule.to)`

### 6.1 统一表 `asset_conversion_rules`（新建）

```sql
CREATE TABLE asset_conversion_rules (
  conversion_rule_id        BIGINT AUTO_INCREMENT PRIMARY KEY,
  from_asset_code           VARCHAR(50) NOT NULL,
  to_asset_code             VARCHAR(50) NOT NULL,
  rate_numerator            BIGINT NOT NULL,
  rate_denominator          BIGINT NOT NULL,
  rounding_mode             ENUM('floor','ceil','round') NOT NULL DEFAULT 'floor',
  min_from_amount           BIGINT NOT NULL DEFAULT 1,
  max_from_amount           BIGINT NULL,
  daily_user_limit          BIGINT NULL,
  daily_global_limit        BIGINT NULL,
  fee_rate                  DECIMAL(5,4) NOT NULL DEFAULT 0.0000,
  fee_min_amount            BIGINT NOT NULL DEFAULT 0,
  fee_asset_code            VARCHAR(50) NULL,
  effective_from            DATETIME NULL,
  effective_until           DATETIME NULL,
  priority                  INT NOT NULL DEFAULT 0,
  status                    ENUM('active','paused','disabled') NOT NULL DEFAULT 'active',
  title                     VARCHAR(100) NULL,
  description               TEXT NULL,
  display_icon              VARCHAR(200) NULL,
  is_visible                TINYINT(1) NOT NULL DEFAULT 1,
  risk_level                ENUM('low','medium','high') NOT NULL DEFAULT 'low',
  created_by                INT NULL,
  updated_by                INT NULL,
  created_at                DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at                DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_from_to_status (from_asset_code, to_asset_code, status),
  INDEX idx_status_priority (status, priority DESC),
  INDEX idx_effective_time (effective_from, effective_until)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

计算公式统一为：`to_amount = rounding_mode(from_amount × rate_numerator ÷ rate_denominator)`

### 6.4 `executeConvert` 核心流程（统一后）

```
1. assertAndGetTransaction(transaction) — 强制外部事务
2. getEffectiveRule(from, to) — 查询生效规则（时间窗+优先级+status=active）
3. validateAmount(from_amount, rule) — 数量范围校验（min/max_from_amount）
4. validateDailyLimit(user_id, rule, from_amount) — 每日限额校验
5. rule.calculateConversion(from_amount) — 计算 gross/fee/net（支持 rounding_mode）
6. 幂等检查 — QueryService.getTransactionByIdempotencyKey(debit_key)
7. 三方记账：
   - debit:  BalanceService.changeBalance({ user_id, delta: -from_amount, business_type: 'asset_convert_debit', counterpart: SYSTEM_BURN })
   - credit: BalanceService.changeBalance({ user_id, delta: +net_to_amount, business_type: 'asset_convert_credit', counterpart: SYSTEM_MINT })
   - fee:    BalanceService.changeBalance({ system_code: 'SYSTEM_PLATFORM_FEE', delta: +fee_amount, business_type: 'asset_convert_fee', counterpart: userAccount })
8. 失效缓存 — BusinessCacheHelper.invalidate()
9. 返回 { from_amount, to_amount: net_to_amount, fee_amount, rule_id }
```

### 6.5 `adminCreateRule` 风控流程（继承 MaterialConversionValidator）

```
1. 验证 from_asset_code / to_asset_code 在 material_asset_types 中存在且 is_enabled=true
2. 终点货币检查：star_stone 禁止作为 from_asset_code（防止星石被转换走）
3. 查询所有 status='active' 的现有规则
4. 将新规则加入规则集合 → 构建有向图
5. DFS 循环检测 — 拦截 A→B→C→A 闭环
6. Bellman-Ford 负环检测 — 拦截套利闭环（rate 乘积 > 1）
7. 校验通过 → 写入 asset_conversion_rules 表
```

## 十、需要你拍板的决策

### 决策点 1：系统账户选择

**行业做法**

| 公司/平台 | 方案 | 具体做法 |
|---|---|---|
| 支付宝（蚂蚁集团） | 分开账户 | 每种资金流向有独立的虚拟账户：收款户、付款户、手续费户、备付金户，对账时按账户维度汇总 |
| 微信支付 | 分开账户 | 商户号、平台号、手续费号分离，每个方向独立记账 |
| 腾讯游戏（王者荣耀） | 单一系统户 | 材料合成就是"系统回收+系统发放"，用同一个系统池，因为游戏不需要金融级对账 |
| 米哈游（原神） | 单一系统户 | 材料合成/分解都走一个"系统仓库"，简单直接 |
| 网易游戏（梦幻西游） | 分开账户 | 有"销毁池"和"产出池"的概念，因为梦幻有真实的经济系统需要监控通胀 |
| 币安/OKX | 分开账户 | 热钱包、冷钱包、手续费钱包严格分离，审计要求 |
| Steam（V社） | 单一系统户 | 社区市场交易走统一的 Valve 账户 |
| 小公司/创业项目 | 单一系统户 | 简单够用，不需要复杂对账 |
| 银行核心系统 | 分开账户 | 监管要求，每种资金流向必须有独立科目 |

**核心区别**

| 方案 | 优点 | 缺点 | 适合 |
|---|---|---|---|
| 单一 SYSTEM_EXCHANGE | 简单，代码少，查询方便 | 对账时无法区分"系统回收了多少"和"系统发放了多少"，只能看净值 | 游戏材料合成、简单积分兑换 |
| 分开 BURN + MINT | 对账清晰（BURN 余额=系统回收总量，MINT 余额=系统发放总量），能监控通胀/通缩 | 多一个账户，代码稍复杂 | 有经济系统需要监控的项目、需要审计的项目 |

**你的项目实际情况**

- 你有完整的对账报告系统（`/api/v4/console/reconciliation/*`）
- 你有资产守恒校验逻辑（`SUM(delta_amount) = 0`）
- 你的 `BalanceService.changeBalance()` 已经强制双录记账，分开账户不增加代码复杂度
- 你有 `SYSTEM_BURN`、`SYSTEM_MINT`、`SYSTEM_EXCHANGE`、`SYSTEM_PLATFORM_FEE`、`SYSTEM_ESCROW` 五个系统账户已经在用
- 你的项目是餐厅积分抽奖系统，有源晶碎片→星石的经济循环，未来可能需要监控"系统总共回收了多少碎片、发放了多少星石"来调整经济平衡

**✅ 推荐：B — SYSTEM_BURN + SYSTEM_MINT**

理由：

1. 你已经有这两个账户在用，不是新建
2. `changeBalance` 双录记账机制下，分开账户零额外代码成本
3. 对账报告能直接看到"系统回收总量"和"系统发放总量"，对经济系统调控有价值
4. 从长期维护角度，信息量更大比信息量更少好 — 合并容易，拆分难
5. 169 笔历史 `SYSTEM_EXCHANGE` 流水需要迁移 `counterpart_account_id`，但数据量小，一次性成本

### 决策点 3：747 笔历史交易流水 business_type 迁移

**行业做法**

| 公司/场景 | 方案 | 原因 |
|---|---|---|
| 银行核心系统 | 绝不改历史流水 | 监管要求，历史交易记录不可修改 |
| 支付宝/微信支付 | 绝不改历史流水 | 审计合规，流水是法律证据 |
| 腾讯游戏 | 会改 | 游戏内部数据，没有合规约束，版本迭代经常批量迁移 |

| DeFi 项目 | 链上不可改，链下会改 | 链上数据不可变，但后台数据库可以迁移 |

**核心区别**

| 方案 | 长期维护成本 | 一次性成本 | 风险 |
|---|---|---|---|
| A 迁移 | 低（一个前缀搞定所有查询） | 几条 UPDATE SQL + 验证 | 改错了流水对不上（但有 idempotency_key 可校验） |
| B 不迁移 | 高（每次查询/报表/对账都要 OR 三套前缀，永远背着） | 零 | 零 |

**你的项目实际情况**

- 747 笔，数据量极小
- 没上线，没有外部系统依赖这些 business_type

**✅ 推荐：A — 迁移**

理由：

1. 747 笔，3 条 UPDATE SQL，30 秒搞定
2. 没上线，零外部依赖风险
3. "每次查询都要 OR 三套前缀"这个技术债务会跟你一辈子，现在不还以后更贵

| 大厂（阿里/腾讯） | 重命名 + TTL 自动清理 | 有回退余地，但不手动管理 |
| 游戏公司 | 直接 DROP | 版本迭代快，旧表没价值 |
| 小公司 | 直接 DROP 或重命名保留一段时间 | 看心情 |

**✅ 推荐：A — 重命名保留 30 天**

理由：

1. 成本为零（就是个 RENAME TABLE）
2. 万一迁移有问题可以回退
3. 30 天后确认没问题再 DROP，心里踏实
4. 不是银行不需要永久归档

### 决策点 5：风控逻辑

**行业做法**

| 公司/场景 | 方案 | 原因 |
|---|---|---|
| 腾讯游戏（王者荣耀） | 有循环检测 | 防止玩家通过 A→B→C→A 循环刷材料 |
| 米哈游（原神） | 有，但简单 | 材料合成只有单向（低→高），不存在循环可能 |
| 网易游戏（梦幻西游） | 有完整的经济系统风控 | 梦幻有真实的玩家交易经济，必须防套利 |
| 币安/OKX | 有，极其复杂 | 三角套利检测是交易所核心功能 |
| 电商（淘宝/京东） | 有 | 防止优惠券/积分循环套利 |
| 小公司/创业项目 | 通常没有 | 规则少，人工管理就够了 |
| DeFi（Uniswap） | 不防，反而鼓励 | 套利者帮助市场价格回归均衡 |

**核心区别**

| 方案 | 首期成本 | 长期价值 | 风险 |
|---|---|---|---|
| A 继承风控 | 中（代码已有，需要适配新表结构） | 高（未来加反向规则时自动保护） | 无 |
| B 暂不继承 | 零 | 无（未来加反向规则时要补） | 如果忘了补，可能被套利 |

**你的项目实际情况**

- `MaterialConversionValidator` 已经写好了，有循环检测 + Bellman-Ford 套利检测
- 目前所有规则都是单向的（碎片/源晶 → 星石），风控不会触发
- 但你的系统有 6 色碎片 + 6 色源晶 + 星石，未来如果加了"星石→碎片"或"碎片→源晶"的反向规则，没有风控就可能被套利
- 代码已经存在，不是从零写，只是适配新服务的调用方式

**✅ 推荐：A — 继承风控逻辑**

理由：

1. 代码已有，适配成本低（改 import 路径 + 表名）
2. 你说了"一次性投入、长期维护成本低"，风控是典型的"现在不加以后补更贵"
3. 目前单向规则不触发，零运行时开销
4. 万一未来运营同事手滑配了个循环规则，自动拦截比事后追查好

### 决策点 6：用户端路由

**行业做法**

| 公司/平台 | 路由设计 | 原因 |
|---|---|---|
| 支付宝 | `/api/trade/convert` | 按动作命名，不按实现方式 |
| 腾讯游戏 | `/api/craft/execute` | 按业务域命名（craft=合成） |
| 米哈游 | `/api/material/composite` | 按业务域命名 |
| Stripe | `/v1/prices`、`/v1/exchange_rates` | 按资源类型命名 |

| DeFi（Uniswap） | `/swap` | 统一 swap，不区分方向 |
| 小公司 | 各种都有 | 看开发者习惯 |

**核心区别**

| 方案 | 优点 | 缺点 |
|---|---|---|
| A `/assets/conversion/*` | 语义准确（"资产转换"是业务动作）、跟管理后台 `/console/assets/conversion-rules` 对齐、新路径无历史包袱 | 小程序前端要改 API 路径 |
| B `/assets/rates/*` | 沿用在跑的路径，ExchangeRateService 的 169 笔交易不受影响 | "rates"是"汇率"的意思，但统一后不只是汇率了，语义不准确；小程序前端也要改（因为还有 `/shop/assets/convert` 要合并过来） |

**你的项目实际情况**

- 不管选 A 还是 B，小程序前端都要改（因为 `/shop/assets/convert` 和 `/assets/conversion-rules` 这两个旧路径都要废弃）
- 管理后台路由已经决定统一到 `/console/assets/conversion-rules`
- 用户端路径跟管理后台路径保持一致的命名风格（都叫 conversion）更好维护

**✅ 推荐：A — /api/v4/assets/conversion/***

理由：

1. 反正小程序都要改，改到哪个路径成本一样
2. "conversion"（转换）比"rates"（汇率）语义更准确 — 统一后的服务不只是汇率兑换，还包括材料合成/分解
3. 跟管理后台 `/console/assets/conversion-rules` 命名风格一致
4. 新路径无历史包袱，不会跟旧的 `/assets/rates` 混淆

# 资产转换类型展示方案 — 前端如何区分分解/合成/兑换

> **解决的问题：** 两套规则系统（ExchangeRate + MaterialConversion）合并为统一的 `asset_conversion_rules` 表后，前端失去了区分"分解""合成""兑换"三种功能的依据。新表只有 `from_asset_code → to_asset_code` 的转换规则，没有任何字段告诉前端这条规则属于哪种业务功能，导致前端无法做 tab 分组、图标切换、文案渲染。
>
> **决策日期：** 2026-04-05
>
> **关联文档：** `docs/资产转换规则统一方案.md`
>
> **决策结论：** 采用"元数据自动推导 + 运营可选覆盖"方案（方案D+）

---

## 一、问题背景

合并后的 `asset_conversion_rules` 表结构：

```
conversion_rule_id | from_asset_code | to_asset_code | rate_numerator | rate_denominator | ...
```

前端需要把规则分成不同的功能 tab（分解/合成/兑换），但表里没有 `conversion_type` 字段。

现有 9 条规则全是 `shard/gem → star_stone`（材料→货币），未来会有：
- gem → shard（高阶拆低阶）
- shard → gem（低阶合高阶）
- red_shard → blue_shard（同阶互换）
- shard → star_stone（材料变现）

---

## 二、行业方案调研与对比

### 方案A：硬编码 ENUM（小公司/外包项目）

```sql
conversion_type ENUM('exchange','decompose','compose') NOT NULL
```

**代表：** 外包项目、小型独立游戏、毕设项目

| 优点 | 缺陷 |
|---|---|
| 简单直接，前端一个字段搞定 | 语义是主观的 — 碎片→星石算"分解"还是"兑换"？ |
| 查询无额外 JOIN | 运营改文案要改 ENUM 值 |
| | 新增"升级""熔炼""提纯"等语义要改 ENUM + 跑迁移 |

**长期维护成本：高。** 每次业务语义变化都是 schema 变更。

### 方案B：Recipe 配方表（游戏公司主流）

```
recipes: recipe_id, recipe_type
recipe_ingredients: recipe_id, asset_code, amount  （多输入）
recipe_outputs: recipe_id, asset_code, amount      （多输出）
```

**代表：** 米哈游（原神锻造/合成台）、暴雪（WoW 专业技能）、网易（逆水寒制造）

| 优点 | 缺陷 |
|---|---|
| 支持 N:M 转换（3碎片+1催化剂→1宝石） | 本项目是 1:1 资产对转换，不存在多输入多输出 |
| 极其灵活 | 三表结构查询复杂度翻倍 |
| | 事务边界更难管理 |

**长期维护成本：中等，但对本项目是不必要的复杂度。**

### 方案C：Tag 标签体系（美团/阿里/腾讯中台）

```sql
rule_tags: rule_id, tag_key, tag_value
-- 例如: (1, 'ui_category', 'decompose'), (1, 'display_tab', '分解')
```

**代表：** 美团优选（商品标签）、阿里（SPU/SKU 属性标签）、腾讯游戏中台

| 优点 | 缺陷 |
|---|---|
| 极度灵活，运营随时加标签不改 schema | 本项目只有 9 条规则，tag 中台是杀鸡用牛刀 |
| 支持多维度分类 | 查询要 JOIN tag 表，缓存要额外处理 |

**长期维护成本：低（规则量大时），但前期投入和复杂度不匹配本项目规模。**

### 方案D：元数据自动推导（Stripe 思路）

不在规则表加 type 字段。根据 `from_asset_code` 和 `to_asset_code` 关联 `material_asset_types` 的 `form` + `tier`，自动推导转换类型。

**代表思路：** Stripe（从 amount 正负推导 debit/credit）、以太坊 ERC-20（token swap 不区分类型）

推导规则：

```
from.tier > to.tier   → decompose（分解：高阶→低阶）
from.tier < to.tier   → compose（合成：低阶→高阶）
from.tier == to.tier  → exchange（兑换：同阶互换）
```

| 优点 | 缺陷 |
|---|---|
| 零 schema 变更 | **语义是主观的** — 运营可能不认同推导结果 |
| 新增规则自动覆盖 | 1个红源晶→10个红源晶碎片，推导是"分解"，用户可能觉得是"兑换" |
| 查询简单（JOIN asset_types） | shard→star_stone 推导是"合成"，用户可能觉得是"变现" |

**致命问题：** "分解/合成/兑换"本质上是运营语义，不是技术语义。同一条规则，运营今天叫"分解"，明天改版叫"精炼"，后天叫"回收"，都是合理的。纯推导无法覆盖这种主观性。

---

## 三、最终方案：D+ 元数据自动推导 + 运营可选覆盖

### 核心思路

1. 服务端根据 `tier` 自动推导出默认的 `conversion_type`
2. 规则表加一个 `display_category` 字段（**nullable VARCHAR**），运营可在管理后台手动覆盖
3. 前端取值逻辑：`display_category ?? 自动推导结果`

### 为什么不用 ENUM

`display_category` 用 `VARCHAR(50)` 而非 ENUM：
- 运营想叫"回收""精炼""熔炼""提纯"随便填
- 前端维护一个 `category → 图标/文案` 的映射表
- 新增展示分类不需要跑数据库迁移

### 数据库变更

```sql
ALTER TABLE asset_conversion_rules
  ADD COLUMN display_category VARCHAR(50) NULL
  COMMENT '运营自定义展示分类（NULL则由系统根据资产tier自动推导）'
  AFTER description;
```

### 推导逻辑

```javascript
// utils/conversionTypeResolver.js

/**
 * 从资产元数据自动推导转换类型
 *
 * @param {Object} fromAssetType - 源资产（含 form, tier, group_code）
 * @param {Object} toAssetType   - 目标资产（含 form, tier, group_code）
 * @returns {{ type: string, label: string }}
 */
function resolveConversionType(fromAssetType, toAssetType) {
  const fromTier = fromAssetType.tier
  const toTier = toAssetType.tier

  if (fromTier > toTier) {
    return { type: 'decompose', label: '分解' }
  }
  if (fromTier < toTier) {
    return { type: 'compose', label: '合成' }
  }
  return { type: 'exchange', label: '兑换' }
}
```

### 前端取值优先级

```
最终展示分类 = rule.display_category ?? resolveConversionType(from, to).type
```

### API 返回示例

```json

  "rules": [

      "from_asset_code": "red_core_shard",
      "to_asset_code": "star_stone",

      "to_display_name": "星石",
      "rate": "1:10",
```

| 3~6 | 其他色碎片 → star_stone | 1→10 | compose（合成） | 兑换/变现 |
| 7 | red_core_gem → star_stone | 2→10 | compose（合成） | 兑换/变现 |
| 8~9 | red_core_shard → star_stone（旧/测试） | 1→10 | compose（合成） | — |

**结论：** 现有规则全是低 tier→高 tier，推导结果全是"合成"。但运营大概率想叫"兑换"或"变现"，所以这些规则需要运营手动设置 `display_category = 'exchange'`。

这恰好验证了为什么纯推导不够 — 需要运营覆盖能力。

---

## 五、方案对比总结

| 维度 | A.ENUM | B.Recipe | C.Tag中台 | **D+.推导+覆盖** |
|---|---|---|---|---|
| schema 变更量 | 加 ENUM 列 | 新建 3 张表 | 新建 tag 表 | **加 1 个 VARCHAR 列** |
| 适合规则量 | <20 | 不限 | >100 | **不限** |
| 多输入多输出 | ✗ | ✓ | ✗ | **✗（本项目不需要）** |
| 运营改语义 | 改ENUM+迁移 | 改recipe_type | 改tag值 | **改 display_category 值** |
| 新增转换类型 | 改ENUM+迁移 | 加recipe_type | 加tag | **自动推导或填值** |
| 查询复杂度 | 低 | 高（3表JOIN） | 中（JOIN tag） | **低（JOIN asset_types）** |
| 长期技术债务 | 中 | 高 | 中 | **最低** |

| `POST /api/v4/assets/rates/preview` | `POST /api/v4/assets/conversion/preview` | 转换预览 |

### 7.2 字段名变更

| 旧字段名 | 新字段名 | 说明 |
|---|---|---|
| `exchange_rate_id` | `conversion_rule_id` | 规则主键 |
| `exchange_rate` | `rate_numerator` / `rate_denominator` | 汇率拆分为分子/分母（整数精度） |
| `source_asset_code` | `from_asset_code` | 源资产编码 |
| `target_asset_code` | `to_asset_code` | 目标资产编码 |
| `min_amount` | `min_from_amount` | 最小转换数量 |
| `max_amount` | `max_from_amount` | 最大转换数量 |

### 7.3 新增字段（前端需要展示）

| 字段 | 类型 | 说明 |
|---|---|---|
| `conversion_type` | string | 虚拟字段，自动推导：`compose`（合成）/ `decompose`（分解）/ `exchange`（兑换） |
| `conversion_label` | string | 虚拟字段，中文标签：合成 / 分解 / 兑换 |
| `type_source` | string | 虚拟字段，推导来源：`display_category`（运营指定）/ `tier_derived`（自动推导） |
| `display_category` | string/null | 运营手动指定的展示分类，覆盖自动推导 |
| `fee_rate` | number | 手续费率（0~1） |
| `fee_asset_code` | string/null | 手续费扣除的资产编码（null 表示从源资产扣） |
| `daily_user_limit` | number/null | 每日用户限额（null 表示无限制） |
| `risk_level` | string | 风险等级：`low` / `medium` / `high` |
