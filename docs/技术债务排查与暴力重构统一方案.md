## 一、总体结论

项目整体较干净：无 v1/v2/v3 路由；无孤儿路由文件；135 个模型与数据库 136 张表（含 sequelizemeta）一一对应；109 条迁移文件与 sequelizemeta 执行记录完全一致（2026-07-11 深夜复核）；旧 AssetService / PointsService / ExchangeRateService / C2C 市场服务与表均已物理删除。

剩余债务集中四类：
1. "拆分了一半"的服务：新服务已注册但路由仍走旧服务，真双轨并存（最高风险）
2. 数据迁移完成后未拆除的兼容分支（PII 明文兜底、DIY 旧格式、旧枚举值）
3. 死代码 / 死注册 / 一次性脚本 / 与项目无关文件
4. 超大文件（服务层、路由层、模型层各一批 1000+ 行）

## 二、边界划清：合法双轨（禁动）vs 兼容残留（必清）

### 2.1 合法业务双轨——保留，不是统一对象

| 双轨 | 内容 | 判定依据 |
|---|---|---|
| 背包双轨 | assets（可叠加：account_asset_balances + AssetTransaction 账本）↔ items（不可叠加：items + item_ledger 双录 + item_holds 锁） | 两种资产形态各有完整账本闭环，BackpackService 文件头明确声明 |
| 兑换频道双轨 | item_type：prop（道具商城/星石轨）↔ product/voucher（商品兑换/源晶轨） | 产品频道设计，exchange/QueryService 有白名单校验 |
| 积分货币双轨 | points 永续积分 ↔ event_points 活动积分（独立过期 job） | 双货币经济设计 |
| 商品编码双轨 | ProductSeries 可读系列号（SLNB-001）↔ ProductBatch 批次码 | 两个正交编码维度 |
| 用户等级双轨 | users.user_level（身份角色）↔ user_growth_levels（消费成长等级） | 语义不同 |
| 兑换订单双链路 | exchange_records（B2C 商城履约）↔ redemption_orders（核销码履约） | 不同履约链 |
| 基础设施降级 | Redis 限流退化内存限流、Sealos 内网转公网、分群规则回退 default | 容错降级，非新旧并存 |
| 统计分级查询 | 今日查 lottery_draws / 7-90 天查 hourly / 90 天以上查 daily | 性能分层 |

注意误判项：`old_status`（状态机审计字段）、`old_item_ids`（以物易物"投入的旧物"参数）、抽奖 `fallback`（保底档位）均为业务词，不是旧代码。

#### 13.4.1 行业对照：五个拍板项分别对应外界怎么做、差异在哪、为什么建议这么选

**A. 订单终态设计（completed 去留）**

| 阵营 | 做法 |
|---|---|
| 阿里/淘宝系电商 | 交易主单有限状态机，终态只有 TRADE_FINISHED（交易成功）/ TRADE_CLOSED（关闭）两类；"确认收货"是触发结算的中间事件不是终态；评价是独立域（评价表），从不进订单状态机 |
| 美团/京东 | 订单中心与履约中心分离：主单终态统一"已完成"，配送/自提/虚拟券各走自己的履约单状态机 |
| 游戏公司（腾讯游戏道具商城、Steam 市场） | 虚拟物品即时清算：下单≈发货≈完成，几乎没有中间态，终态就是 completed；履约失败靠幂等重试与补偿，不靠状态机加态 |
| 二手/小众交易平台（闲鱼等） | 确认收货即"交易成功"终态，评价独立；实物链路比电商短但终态同样唯一 |
| 典型小公司 | 单表单枚举，常把"已收货"当终态用，后补评价/售后时被迫往枚举里塞新值——状态机膨胀正是这类项目的头号债务源 |

差异与结论：原拍板 5"把 completed 映射成 received 当终态"实质是小公司式做法（把中间业务动作当终态）；本项目现行设计——一个状态机、两条履约路径（实物 pending→approved→shipped→received→rated→completed；虚拟/券/道具直达 completed）、单一终态——恰好就是大厂"主单单一终态 + 履约分流"与游戏行业"虚拟即时终态"的合体，结构上已是行业标准形态，零改造成本。已拍板采纳修正案（保留 completed，拍板 17）。若未来售后复杂化，行业惯例是加独立售后单（refund/aftersale 表）而不是往 status 枚举加值——这条作为长期演进原则一并记录。

**B. 会话设备标识（device_id 约束强度）**

| 阵营 | 做法 |
|---|---|
| 腾讯/阿里 | 设备标识必采（设备指纹 SDK / Web 持久化指纹），会话按 (user, device) 隔离——微信"登录设备管理"、淘宝设备管理同款形态；服务端对缺失设备号的请求走风控降级而不是放行等价会话 |
| 游戏公司 | 设备号硬必填（防多开/防作弊/封禁按设备维度），无设备号直接拒绝登录 |
| 典型小公司 | 单 token 无设备维度，被顶号/多端互踢问题靠客服解决 |

差异与结论：本项目的会话架构（设备级多会话 + 软上限 + 同设备轮换）已经是腾讯系形态，唯一不自洽的是字段允许 NULL 且代码里留着"legacy 客户端"兜底分支——NULL 会话享受不到 Redis 热路径（每请求回源 MySQL）、不参与设备列表/软上限统计、不做同设备轮换，等于架构对这 200 条会话失效。项目未上线 + sessions 必被擦除 + admin 已带 X-Device-Id，收紧到 NOT NULL 的成本只剩"小程序确认一下"，收益是删掉一整条兼容分支并让架构名副其实。已拍板选收紧（游戏公司/大厂强度，拍板 18）。

**C. PII 明文列残留**

大厂在个保法/等保合规下的标准动作是四步：双写→回填→切读→删明文列，明文列保留会直接过不了安全审计（美团、滴滴的数据安全改造均为此模式）；小公司常见双列并存烂尾。本项目已走完前三步（加密列全覆盖、明文列 0 条有值、代码读加密列），只差最后一刀。无行业分歧，已拍板 drop（拍板 19，删40）。

**D. 前端死端点**

大厂靠 API 网关的调用量监控 + 治理流程定期下线零流量接口；小公司靠 grep。做法不同但方向一致：指向不存在路由的客户端封装没有任何保留价值。无分歧，已拍板删除（拍板 20，删41）。

**E. 质量门禁位置**

大厂的硬卡口在 CI/CD 流水线（阿里云效、腾讯蓝盾的 merge 阻断），本地 pre-commit 只当提效手段——因为 `git commit --no-verify` 可绕过；小团队无 CI 项目的现实最优解是 pre-commit 硬门禁（一行改动、零基建成本）。本项目已核实无任何 CI 基建，属于后者。已拍板硬化，且用户明确拍板不做 CI/CD：pre-commit 即本项目唯一且最终的门禁位置，三道检查脚本（字段黑名单、API 字段合约、路由层规范）不做流水线迁移规划。

**顺带的行业印证（不需拍板）**：本项目的背包双轨（可叠加资产账本 ↔ 不可叠加物品实例 + 双录账本 + 锁）正是游戏行业"货币/堆叠道具 vs 装备实例"的标准分法；exchange_records（商城履约）↔ redemption_orders（核销码履约）双链路对应零售行业"电商单 ↔ 券码核销单"的标准拆分。第二章"合法双轨禁动"的判定与行业实践一致，进一步支持"以后端现有设计为权威、前端适配后端"的路线。
