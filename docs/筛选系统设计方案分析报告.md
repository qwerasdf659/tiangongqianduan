### 1.3 后端核心架构模式

- **路由层**: 参数解析 + 校验，不直连 models，通过 ServiceManager 获取 Service
- **服务层**: 业务逻辑 + 事务管理 + Redis缓存读写

- **缓存层**: BusinessCacheHelper 统一管理 Redis 缓存
- **数据脱敏层**: DataSanitizer 统一处理 PII 数据
- **显示名称层**: displayNameHelper 统一附加中文字段名

### 1.4 业务模型

天工小程序是一个餐厅积分系统，核心能力包括：

- **积分消费**：用户在餐厅消费获得积分，商家扫码核销
- **抽奖**：积分抽奖，支持多种奖品、保底、连抽
- **兑换商城**：积分兑换商品（幸运空间 lucky、臻选空间 premium）
- **交易市场**：用户间C2C交易物品（item）和可叠加资产（fungible_asset，如钻石、碎片）
- **竞价**：竞价购买商品
- **广告投放**：广告主自助投放（包天、竞价、CPM）

| 参数 | 类型 | 说明 | 后端实现方式 |

| sort_order | string | 排序方向 ASC / DESC | ORDER BY ? ASC/DESC |
| page | number | 页码，默认 1 | OFFSET (page-1)*page_size |

## 八、后端聚合计数实现方案（基于现有技术框架）

### 8.1 商城 QueryService 扩展设计

在 `services/exchange/QueryService.js` 的 `getMarketItems()` 方法中：

1. 接收 `with_counts` 参数（boolean）
2. 将现有 WHERE 条件构建拆分为可组合的 filter map：
   - `baseFilter`: `{ status: 'active', space: { [Op.in]: [space, 'both'] } }` + keyword
   - `categoryFilter`: `{ category: value }`
   - `priceFilter`: `{ cost_amount: { [Op.gte]: min, [Op.lte]: max } }`
   - `stockFilter`: `{ stock: { [Op.gt]: 5 } }` 或 `{ stock: { [Op.between]: [1, 5] } }`
3. 当 `with_counts=true` 时，用 `Promise.all()` 并行执行 N+1 个查询
4. 每个维度的 COUNT 查询 = baseFilter + 所有其他维度 filter - 自身维度 filter
5. 结果写入响应的 `filters_count` 字段

### 8.2 交易市场 QueryService 扩展设计

在 `services/market-listing/QueryService.js` 的 `getMarketListings()` 方法中：

1. 同样接收 `with_counts` 参数
2. 拆分 WHERE 条件为可组合的 filter map
3. 并行执行 COUNT 查询
4. 利用现有 22 个索引（含联合索引 idx_status_kind_category、idx_status_kind_asset_group）加速

### 8.3 价格区间计数的区间定义

价格区间定义从 `system_configs.product_filter.cost_ranges` 读取（统一后的数据源）。SQL 中使用 CASE WHEN 或多个 COUNT(IF(...)) 实现。

### 8.4 缓存策略

聚合计数结果随商品列表一起缓存在同一个 Redis key 中（BusinessCacheHelper 机制不变，只是缓存对象多了 `filters_count` 字段），无需额外缓存逻辑。

### 8.5 性能预估

| 场景 | 查询次数 | 预估耗时 |
|------|---------|---------|
| 首次加载（4 维度 + 1 列表） | 5 并行 SQL | 200-500ms |
| 切换筛选（缓存命中） | 0 | 10-50ms |
| 切换筛选（缓存未命中） | 5 并行 SQL | 200-500ms |

exchange_items 表当前 1 条，预估 50-300 条，所有 COUNT 走索引，单次 < 10ms。
market_listings 表当前 308 条，21 条在售，联合索引支撑，单次 < 20ms。

## 九、四项决策与推荐

### 决策 1: 商城 category 筛选值使用哪套体系？

**推荐: 直接使用 `category_defs.category_code`**

理由：
- 单一数据源，避免 exchange_page 配置与 category_defs 表不一致
- 运营新增/禁用分类 → 在 category_defs 表操作 → 所有筛选自动生效
- 行业标准做法（淘宝/京东/美团/有赞/5173 全部是数据库表驱动）
- `product_filter.categories` 是空数组，`exchange_page.shop_filters.categories` 使用了错误的中文名值 — 两者都应废弃，统一从 category_defs 读取

### 决策 2: C++ 和 C+++ 是否合并？

**推荐: 直接实现 C+++，一步到位**

理由：
- C++ 到 C+++ 增量代码 < 50 行（区别仅在 WHERE 条件组合方式）
- 分阶段实现反而需要改两次前端
- 数据量小（<500），5 个 COUNT + 索引支撑，总耗时 < 50ms
- 项目未上线无旧数据兼容负担

### 决策 3: 商城和交易市场的 facets 机制是否统一？

**推荐: 保持独立，共享底层字典表**

理由：
- 两者筛选维度完全不同（商城没有稀有度/资产分组，交易市场没有库存状态/基础标签）
- 商城从 `category_defs` + `system_configs.product_filter` 获取筛选选项
- 交易市场从 `GET /api/v4/market/listings/facets` 动态获取
- 行业标准：异质业务各自独立筛选维度（美团/5173/王者荣耀）

### 决策 4: 价格区间以哪套配置为准？ — **需你拍板**

| 配置来源 | 区间值 |
|---------|--------|
| `product_filter.cost_ranges` | 全部 / 100以内 / 100-500 / 500-1000 / 1000以上 |
| `exchange_page.shop_filters.cost_ranges` | 全部 / 50以内 / 50-200 / 200-500 / 500以上 |

**决策结论: 方案 A — 以 `product_filter` 为准（100/500/1000）**

执行动作：
1. 在管理后台 `exchange-page-config.html` 中将 `shop_filters.cost_ranges` 修改为与 `product_filter.cost_ranges` 一致：全部 / 100以内 / 100-500 / 500-1000 / 1000以上
2. 微信小程序从 `/api/v4/system/config/product-filter` 读取价格区间配置
3. 上线后根据实际商品价格分布，运营可在管理后台随时调整

### 14.2 商城筛选对接

**筛选选项数据源 API**:
```
GET /api/v4/system/config/product-filter
```
无需认证。返回 categories（分类）、cost_ranges（价格区间）、sort_options（排序）、stock_statuses（库存状态）。

**商品列表 API**:
```
GET /api/v4/backpack/exchange/items
Authorization: Bearer <token>
```

| 参数 | 类型 | 说明 | 取值来源 |
|------|------|------|---------|
| space | string | 空间 | lucky / premium |
| category | string | 分类代码 | product-filter API 返回的 categories[].value |
| min_cost | number | 最低价格 | product-filter API 返回的 cost_ranges[].min |
| max_cost | number | 最高价格 | product-filter API 返回的 cost_ranges[].max |
| stock_status | string | 库存状态 | in_stock / low_stock |
| keyword | string | 搜索关键词 | 用户输入 |
| sort_by | string | 排序字段 | sort_order / cost_amount_asc / cost_amount_desc / created_at_desc / sold_count_desc |
| sort_order | string | 排序方向 | ASC / DESC |
| page | number | 页码 | 默认 1 |
| page_size | number | 每页数量 | 默认 20 |
| with_counts | boolean | 是否返回聚合计数 | true 时返回 filters_count |

**with_counts=true 时响应新增字段 `filters_count`**:
```
filters_count: {
  categories: { "home_life": 3, "lifestyle": 1, "__null__": 2 },
  cost_ranges: { "0-100": 1, "100-500": 3, "500-1000": 1, "1000+": 0, "total": 5 },
  stock_statuses: { "in_stock": 3, "low_stock": 1, "out_of_stock": 1, "total": 5 }
}
```
- `__null__` 表示 category 字段为空的商品数量
- 交叉排除规则：每个维度的计数排除自身条件但保留其他维度条件

### 14.3 交易市场筛选对接

**筛选维度数据源 API**:
```
GET /api/v4/market/listings/facets
Authorization: Bearer <token>
```
返回 categories（类目）、rarities（稀有度）、asset_groups（资产分组）、listing_kinds（挂单类型）。每项包含 code/display_name 等字段。

**市场列表 API**:
```
GET /api/v4/market/listings
Authorization: Bearer <token>
```

| 参数 | 类型 | 说明 | 取值来源 |
|------|------|------|---------|
| listing_kind | string | 挂单类型 | item / fungible_asset（来自 facets.listing_kinds） |
| item_category_code | string | 物品类目代码 | facets.categories[].category_code |
| asset_group_code | string | 资产分组代码 | facets.asset_groups[].group_code |
| rarity_code | string | 稀有度代码 | facets.rarities[].rarity_code |
| asset_code | string | 资产代码 | 如 red_shard, DIAMOND |
| min_price | number | 最低价格 | 用户输入 |
| max_price | number | 最高价格 | 用户输入 |
| sort | string | 排序 | newest / price_asc / price_desc |
| page | number | 页码 | 默认 1 |
| page_size | number | 每页数量 | 默认 20 |
| with_counts | boolean | 是否返回聚合计数 | true 时返回 filters_count |

**with_counts=true 时响应新增字段 `filters_count`**:
```
filters_count: {
  listing_kinds: { "item": 21, "fungible_asset": 0 },
  item_categories: { "voucher": 21, "home_life": 0 },
  rarities: { "__null__": 21, "common": 0 },
  asset_groups: { "red": 0, "blue": 0 }
}
```
- item_categories 和 rarities 仅统计 listing_kind='item' 的挂单
- asset_groups 仅统计 listing_kind='fungible_asset' 的挂单

### 14.4 前端改造核心要点

1. **从客户端内存过滤改为服务端请求**: 筛选变更时重新调用 API，传递筛选参数
2. **分类代码使用 category_code**: 如 `home_life`/`lifestyle`/`food`，不使用中文名
3. **筛选选项从 API 动态获取**: 商城从 product-filter API，市场从 facets API
4. **计数标签**: `with_counts=true` 获取各维度数量，在筛选标签旁显示
5. **条件联动**: 切换一个维度后其他维度计数自动更新（后端交叉排除已保证）
6. **计数为0灰显**: filters_count 中值为0的选项可用灰色样式标记
7. **防抖处理**: 建议 300ms 防抖，避免频繁请求
8. **缓存机制**: 后端已有 Redis 缓存（商城和交易市场均有），不传 with_counts 时响应更快
