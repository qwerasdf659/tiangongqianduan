## 二、失败原因1 — 登录页/弹窗存在腾讯官方元素

### 2.1 审核原文

> 你好，你的小程序登录页面或弹窗（调用手机号快速验证组件的前置页面），存在混淆腾讯官方的元素，包括但不限于"微信"字样、微信官方 logo 等，请去除相关元素，如：将手机号授权登录提示修改为"手机号快捷登录"，再重新提交代码审核。

### 2.2 根本原因（纯文本数据流）

```
微信审核规则：登录页/弹窗不得含"微信"字样、微信 logo 等官方元素
        │
        ▼
登录弹窗按钮文案 = "微信一键登录"  →  含"微信"二字  →  判定混淆腾讯官方元素  →  驳回
        │
        ▼
官方修改指引示例：把"微信一键登录"改为"手机号快捷登录"
```

- 该按钮实际使用 `open-type="getPhoneNumber"`（拉起手机号快捷验证），改为"手机号快捷登录"既过审，业务语义也准确。
- 经核查，按钮为纯文字、无微信 logo 图标；其余"微信"字样均在代码注释、SCSS 类名（`wechat-btn`）、状态值（`mode==='wechat'`）中，用户界面不可见、审核不查，故不改动。

### 2.3 实际改动

文件 `components/login-popup/login-popup.wxml`（微信登录视图按钮文案）：

```65:66:components/login-popup/login-popup.wxml
        <text wx:if="{{loggingType !== 'wechat'}}">手机号快捷登录</text>
        <text wx:else>登录中...</text>
```

文件 `components/login-popup/login-popup.ts`（两处会弹给用户看的失败 toast 去除"微信"字样）：

- `'微信登录凭证获取失败'` → `'登录凭证获取失败'`
- `'微信登录失败'` → `'登录失败'`

## 三、失败原因2 — 首页未完整体验即要求授权登录

### 3.1 审核原文

> 小程序【首页】页面未完整浏览、体验页面详情时，即要求授权登录，属于登录规范不合规，请在用户体验页面详情后再自主选择授权登录。

### 3.2 根本原因（纯文本数据流）

```
游客进入首页（pages/lottery 抽奖页）
   │
   ├─ 页面本身：onLoad / onShow / initializePage 均不会自动弹登录
   │   游客可正常浏览：「请登录」按钮 + 脱敏积分(***) + 奖品预览  ✓
   │
   └─ 游客点【砸金蛋 / 连抽】按钮
          │
          ▼
     lottery-activity 组件 onChildDraw / onMultiDraw
          │  checkAuth() == false（未登录）
          ▼
     旧逻辑：triggerEvent('needlogin') → 页面 onShowLoginPopup() → 立即强弹登录窗  ✗
```

- 经全量核查，首页进入时没有自动弹登录（本就合规）；唯一不合规点是"游客点抽奖核心按钮立即强弹登录"。
- 整改原则：把"是否登录"的选择权交还用户，游客点抽奖只做轻提示引导，由用户自主点击页面顶部「请登录」按钮再登录。

### 3.3 实际改动

整改后链路：

```
游客进首页 → 可完整浏览（请登录按钮 + 奖品预览 + 脱敏积分），不弹窗  ✓
游客点抽奖 → 轻提示「登录后即可参与抽奖赚积分」，不强弹登录       ✓
游客自主点「请登录」按钮 → 才弹登录窗（用户主动选择）            ✓
```

文件 `pages/lottery/lottery.ts`：新增游客点抽奖的引导方法 `onGuestDraw`，`onShowLoginPopup` 保留给用户主动点击「请登录」按钮使用。

```javascript
/**
 * 游客点抽奖时的引导（lottery-activity 的 needlogin 事件）
 *
 * 合规要求：首页未完整体验前不得强制弹授权登录（微信审核规范）。
 * 故游客点抽奖不再自动弹登录弹窗，仅轻提示引导，由用户自主点击页面顶部
 * 「请登录」按钮再登录，把"是否登录"的选择权交还用户。
 */
onGuestDraw() {
  showToast('登录后即可参与抽奖赚积分', 'none', 2000)
}
```

文件 `pages/lottery/lottery.wxml`：两处 `lottery-activity` 的 `needlogin` 事件由 `onShowLoginPopup` 改绑为 `onGuestDraw`（第 313、353 行）。

> 仅改动首页抽奖页对游客点抽奖的响应；`pages/exchange` 等其他页面共用的 `needlogin`→弹窗逻辑未改动（那些是用户主动点兑换/抽道具时触发，属正常授权时机）。

## 四、失败原因3 — 登录环节缺少显著有效的取消按钮

### 4.1 审核原文

> 小程序【账号】登录环节，需为用户提供显著有效的可取消/拒绝或返回按钮，不得反复弹窗或强制用户进行登录才能体验，请整改后再提交审核。

### 4.2 根本原因（纯文本数据流）

```
旧状态：登录弹窗白色卡片内无任何可见的关闭按钮
        游客只能点弹窗外半透明遮罩关闭（不显著、用户不一定知道可点）
        → 审核判定"无显著可取消按钮，疑似强制登录"  ✗

整改后：弹窗右上角新增显著的 ✕ 关闭按钮
        游客可随时点 ✕ 关闭弹窗、继续浏览
        → 显著有效的取消入口，把"是否登录"选择权交还用户  ✓
```

### 4.3 实际改动

文件 `components/login-popup/login-popup.wxml`：在 `popup-header` 右侧新增关闭按钮（微信登录视图与验证码登录视图共用，两种模式均生效）：

```23:26:components/login-popup/login-popup.wxml
      <!-- 关闭按钮：显著有效的「取消/不登录」入口，用户可随时关闭弹窗继续浏览（微信审核合规要求） -->
      <view class="popup-close" bindtap="onCloseTap" aria-role="button" aria-label="关闭登录">
        <t-icon name="close" size="44rpx" color="#999999" />
      </view>
```

文件 `components/login-popup/login-popup.ts`：新增 `onCloseTap` 方法（复用已有 `onClose()`，会重置表单态并 `triggerEvent('close')`；登录请求进行中时不关闭，避免中断登录态写入）：

```javascript
/**
 * 点击右上角关闭按钮 — 显著有效的「取消登录」入口（微信审核合规：登录环节须提供
 * 可取消/拒绝/返回按钮，不得强制用户必须登录）。登录请求进行中时不可关闭，避免中断登录态写入。
 */
onCloseTap() {
  if (this.data.loggingType) {
    return
  }
  this.onClose()
}
```

文件 `components/login-popup/login-popup.scss`：新增 `.popup-close` 样式（绝对定位右上角，64rpx 加大可点区域）：

.popup-close {
  position: absolute;
  right: 0;
  top: 0;
  width: 64rpx;
  height: 64rpx;
  display: flex;

  justify-content: center;

> `t-icon name="close"` 为 TDesign 小程序版合法图标，项目其它弹窗（`result-modal`、`barter`、`addresses` 等）已在使用。

## 九、微信官方《手机号授权登录案例解析》对照（来源：微信知识库，2025-05-19）

> 官方背景：为更好保护用户隐私信息、优化用户体验，平台对小程序内的账号登录功能进行规范。部分小程序在处理用户手机号登录环节，存在不规范授权收集行为。
> 官方核心原则：**进入未浏览体验功能服务，即要求授权手机号登录 = 违规**。

### 9.1 官方不合规案例

| 编号 | 不合规行为 | 官方案例 |
| --- | --- | --- |
| 不合规案例① | 进入小程序，用户尚未体验页面功能，即要求授权手机号登录 | 一进首页/全部商品页就弹"手机号快捷登录" |
| 不合规案例② | 进入小程序，用户尚未体验页面功能，即要求授权手机号登录 | 进福利商城即弹"授权登录"（仅"暂不登录"为弱入口） |
| 不合规案例③ | 一进入小程序即要求用户提供手机号注册登录才能体验功能服务 | 打开即"登录"页拦截，未注册直接注册 |
| 不合规案例④ | 打开即弹窗要求授权手机号登录 | 进入即弹"登录/注册"遮挡全部功能 |
| 不合规案例①（详情拦截） | 用户浏览商品详情页面内容，提示需进行注册授权登录才可以浏览 | 浏览商品列表/详情时弹"您还未授权…去授权拿收益" |
| 不合规案例②（详情拦截） | 点击查看商品内容详情，要求授权手机号才能查看 | 点商品即跳"手机号一键登录"页 |
| 不合规案例（无取消按钮） | 点击查看内容后选择登录，没有取消/拒绝返回的选项按钮，强制进行授权手机号码登录 | 短信登录/密码登录页，无任何"取消/返回"入口，用户登录后无法退回继续浏览 |
| 不合规案例（内部人群无说明） | 体验范围给内部人群使用，一打开即要求授权登录，首页没有增加说明、未进行登录账号鉴权 | 特定人群小程序打开即"手机号快捷登录"，无任何用途说明 |

### 9.2 官方明确的三条登录规范（务必逐条满足）

> 来源：微信审核反馈与官方案例解析，归纳为三条强制规范。

1. **进入即不得强制登录**：用户进入小程序，应提供可浏览/体验小程序内容功能的能力，**不得在未体验任何功能页面时，即弹窗要求用户授权手机号码登录**。
2. **登录环节须去除腾讯官方元素**：登录页/弹窗不得含"微信"字样、微信 logo 等混淆腾讯官方的元素，避免"一键"用语（建议用"手机号快速验证/手机号快捷登录"）。
3. **登录须可取消/返回**：用户浏览体验完功能服务后，自主选择授权手机号码登录时，**应当为用户提供取消/拒绝的返回选项，不得强制要求用户登录无法返回**。

### 9.3 官方合规案例

> 合规标准：**用户浏览体验功能服务后，再自主选择授权手机号登录**；浏览商品详情后、需购买时再自主选择授权手机号登录；登录页须可自行选择"取消登录"返回。
> 官方正面例：电商/餐饮小程序游客可浏览首页/分类/商品详情，到"我的/下单/购买"等真正需要身份的环节才出现"立即登录"，且登录页提供显著的"取消登录"返回入口，由用户自主点击。

### 9.4 本项目对照自查结论

| 官方合规要求 | 本项目现状（整改后） | 结论 |
| --- | --- | --- |
| 规范1：进入小程序不得立即/未体验即要求授权登录 | 首页（抽奖页）游客可见请登录按钮 + 脱敏积分 + 奖品预览，`onLoad/onShow/initializePage` 不自动弹登录 | ✅ 符合 |
