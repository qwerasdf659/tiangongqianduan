## 六、🔴 座席端实时性（决定是否补 WebSocket）

文档只说"座席回复后通过 WebSocket 推给用户"，但**没说座席端如何实时收到用户发来的新消息**。
前端目前是 REST + 进页/手动刷新（**无实时推送**）。

**请确认**：
- [ ] 座席端是否需要实时接收用户的新消息？（即用户发消息时，座席端会话/聊天界面要不要自动更新）
- [ ] 若需要：座席端连 WebSocket 后的协议是什么？请提供：
  - 连接后要不要注册/加入房间？emit 什么事件、带什么参数？
  - 监听哪个事件名收新消息（如 `new_message`）？该事件 payload 字段结构？
  - 座席发消息走 WebSocket 还是只走 REST `send` 接口？两者会不会重复回推导致消息重复？

> 说明：若后端确认"座席端不需要实时、只需手动刷新"，前端保持现状即可，无需改动。

> ### 🟦 后端答复（六）：WebSocket 实时能力**已具备**，座席端可直接接入；协议如下
>
> 后端 `ChatWebSocketService`（Socket.IO，已随服务启动、cluster 多进程下用 Redis Adapter 跨进程推送）**已经支持**座席端实时收用户新消息。是否接入由前端按体验需要决定：
>
> **连接与鉴权（握手强制 JWT）**：
> - 连接时在 `socket.handshake.auth.token` 放用户的 `access_token`（与 REST 同一个 token）。
>   缺 token / token 失效 / 账号被禁用 → 握手被拒。
> - 连接成功后**后端自动按身份分房**，**前端无需手动 join/register**：
>   - 该账号若以"管理端身份"登录 → 自动加入 `admin:{user_id}` 与 `admins` 房间。
>   - 普通用户身份 → 自动加入 `user:{user_id}` 与 `users` 房间。
>   - （`register_user` 事件已被降级为"能力声明"，**不能也不需要**用它决定身份，身份由服务端 JWT 判定。）
>
> **座席端如何收用户新消息（关键）**：
> - 监听事件名：**`new_message`**。
> - 触发时机：用户给客服发消息时，后端 `broadcastToAllAdmins()` 向 `admins` 房间广播。
> - payload 字段结构（与 REST 消息字段同义，前端零映射）：
>   ```json
>   {
>     "chat_message_id": 22931,
>     "customer_service_session_id": 2156,
>     "sender_id": 32,
>     "sender_type": "user",
>     "content": "用户发来的内容 或 图片URL",
>     "message_type": "text",   // text / image
>     "created_at": "..."
>   }
>   ```
>   前端收到后：若当前正打开该 `customer_service_session_id` 会话 → 追加到消息流；否则给对应会话标记"有新消息"。
>
> **座席发消息走哪条路（避免重复）**：
> - ✅ **座席发消息只走 REST** `POST /api/v4/system/cs-agent/sessions/:id/send`（本文件第四节）。
> - 后端在该 REST 成功后会**主动把这条消息 WebSocket 推给"用户端"**（`pushMessageToUser` → 用户的 `new_message`），
>   **不会回推给座席自己**。所以：座席端对自己刚发的消息用 REST 返回做乐观更新即可，
>   **不会**因为 WebSocket 再收到一条自己的消息而重复。（座席端的 `new_message` 只会收到"用户发来的"。）
> - ⚠️ 座席端**不要**用 WebSocket 的 `send_message` 事件发消息（那条链路是给用户端/管理后台用的）；座席发送统一走 REST，口径唯一。
>
> 其它可选监听（按需，不接也不影响核心收发）：
> - `session_list_update`：会话列表有变动（新会话/状态变化）时广播给 `admins`，前端可据此刷新列表。
> - `session_closed`：会话被关闭。
>
> **结论**：实时能力后端现成，**无需后端改动**。前端若要"用户发消息座席端自动更新"，按上面协议连 WS + 监听 `new_message` 即可；
> 若暂时只想手动刷新，保持纯 REST 现状也完全可用。

> ### 🟦 实测佐证（2026-06-14）：座席端"必须点刷新才看到用户新消息"已定位 = 前端未接 WebSocket
>
> 现象：客服回复台打开某会话（如会话 2185 / 用户7911）后，用户又发了新消息，座席端不自动显示，
> **必须点右上角刷新按钮**才能拉到最新消息。
>
> 逐段核对真实后端代码后定位（纯文本数据流图）：
>
> ```
> 用户端发消息 POST /api/v4/system/chat/sessions/:id/messages
>       │  sendUserMessage() 落库（sender_type='user'）
>       ▼
> 后端主动推送（routes/v4/system/chat.js 第330行起，带重试 pushMessageWithRetry）
>       ├─ 会话已分配座席 → pushMessageToAdmin(admin_id) → io.to(`admin:${admin_id}`).emit('new_message', msg)
>       └─ 会话未分配     → broadcastToAllAdmins()       → io.to('admins').emit('new_message', msg)
>       ▼
> ✅ 后端已把 new_message 实时发出（链路完整、可工作）
>       ▼
> 客服回复台（微信小程序）
>       ❌ 未建立 WebSocket 连接、未监听 'new_message'，纯 REST + 手动刷新
>       → 所以必须点刷新才能看到用户最新消息
> ```
>
> 结论：
> - **不是后端问题、不是 Web 管理前端问题**，是**微信小程序座席端未接入 WebSocket 实时推送**（即本节开头所述的待接入项）。
> - 后端推送链路已验证可用，**无需后端改动**。
>
> ⚠️ 前端接入时的一个**必须满足的前提**（否则连了也收不到）：
> - 后端"精准推送给已分配座席"用的是 **`admin:{admin_id}` 房间**。
> - 因此座席端连 WebSocket 时，**该账号必须以"管理端身份"建立连接**（用座席的管理端会话 token 放 `auth.token`，
>   后端按 user_type=admin 自动加入 `admin:{user_id}` 与 `admins` 房间），才能收到发往 `admin:{id}` 的 `new_message`。
> - 若座席只以"用户身份"连入（进了 `user:{id}` 房间），将收不到这条精准推送。
> - 前端到位后：监听 `new_message` → 命中当前打开会话则追加到消息流、否则刷新会话列表，即可免手动刷新。

## 八、实施说明：`chat_messages` 已读语义探查结论（决定 `unread_count` 怎么算）

> 出具方：后端。来源：直读 `models/ChatMessage.js` + `services/CustomerServiceSessionService.js` + 现有路由真实代码。
> 目的：第二节确认要做"座席未读用户消息"红点，本节说明数据库现状、由此带来的限制与最终落地口径。

### 8.1 表结构现状（关键：只有一个共享 `status`，没有分端已读标记）

`chat_messages` 表与已读相关的字段只有一个 **`status`**：

| 字段 | 类型 | 取值 | 说明 |
| --- | --- | --- | --- |
| `sender_type` | ENUM | `user` / `admin` | 谁发的（座席发的算 `admin`） |
| `status` | ENUM | `sending` / `sent` / `delivered` / `read` | **全表共享的一个消息状态，非"分端已读"** |

- ❗ **没有** `user_read_at` / `admin_read_at` 这种"分别记录用户侧/座席侧是否已读"的字段。
- 即"已读"只有一个全局状态位，**无法同时表达"用户读没读座席的消息"和"座席读没读用户的消息"两套已读**。

### 8.2 现有已读是怎么流转的（实测自代码）

| 动作 | 代码位置 | 效果 |
| --- | --- | --- |
| 用户查看聊天历史 | `getSessionMessages(mark_as_read=true, user_id)` | 把该会话 **`sender_type='admin'`** 的 `sent/delivered` 消息置为 `read`（= 用户读了座席消息） |
| Web 后台座席打开会话 | `POST /api/v4/console/customer-service/sessions/:id/mark-read` → `markSessionAsRead()` | 把该会话 **`sender_type='user'`** 的 `sent/delivered` 消息置为 `read`（= 座席读了用户消息） |
| 小程序 cs-agent 端 | **当前无任何"标记已读"端点** | 座席在小程序里看了消息，不会改 `status` |

- 模型已有现成方法：`ChatMessage.getUnreadCount(sessionId, senderType)` 和 `markSessionMessagesAsRead(sessionId, receiverType)`，口径就是上表。

### 8.3 由此带来的两个硬约束（前端需知晓）

1. **"座席未读用户消息"口径 = `sender_type='user' AND status IN ('sent','delivered')`**。
   这正是第二节拍板要的红点语义，可实现。

2. ⚠️ **已读状态在"Web 后台客服工作台"与"小程序简版座席端"之间是共享的**（同一个 `status` 字段、同一张表）。
   带来的实际表现：
   - 若同一条用户消息已被 Web 后台座席打开会话标记为已读，则小程序座席端对该会话的 `unread_count` 也会变成 0（反之亦然）。
   - 这对"同一个座席跨端"是合理的（哪端看了都算看了）；但**无法做到每个座席各自独立的未读计数**（因为没有按座席维度的已读表）。
   - 对简版客服端（单一座席视角看待回复）这是可接受的，符合"会话被某座席处理过即视为已读"的现有体系。

### 8.4 最终落地方案（后端将按此实施，不新增表、不改表结构）

- **`unread_count`**：座席列表对每个会话统计 `sender_type='user' AND status IN ('sent','delivered')` 的条数（复用 `ChatMessage.getUnreadCount(sessionId, 'user')`）。
- **何时清零（置已读）**：座席在小程序端打开某会话看消息时，把该会话用户消息标记为已读。
  - ✅ 已拍板方案（2026-06-14）：cs-agent 端**新增独立端点** `POST /api/v4/system/cs-agent/sessions/:id/read`，
    复用现有 `markSessionAsRead` 逻辑（把该会话 `sender_type='user'` 的 `sent/delivered` 消息置 `read`），口径一致。
  - 前端时机：**打开会话时主动调一次**该端点，红点即清零（时机由前端控制，不会误清）。
  - 返回：`{ updated_count: <本次置已读条数> }`。
- **`last_message`**：列表对每个会话取最新一条消息（`chat_message_id` / `content` / `sender_type` / `created_at`），随列表下发。

### 8.5 本次后端待实施清单（汇总，零新表、零改表结构）

1. cs-agent 会话列表端点：补 `last_message`（最新一条预览）+ `unread_count`（座席未读用户消息数，口径 `sender_type='user' AND status IN ('sent','delivered')`）。
2. **新增** `GET /api/v4/system/cs-agent/me`：返回 `{ is_agent, status }`，仅登录鉴权、不被 `requireCsAgent` 拦截（第五节）。
3. **新增** `POST /api/v4/system/cs-agent/sessions/:id/read`：座席打开会话时标记该会话用户消息已读，使红点清零（复用 `markSessionAsRead`）。
