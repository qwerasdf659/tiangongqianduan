## 三、前端 APIClient / 认证头 / Token / 错误处理 自检结论（纯文本数据流）

> 这部分是前端**已自查通过**的内容，供后端了解前端的对接行为，无需后端动作。

**(1) API 路径**：所有售后接口路径严格按方案A 第四节实现——
`POST /system/disputes`、`GET /system/disputes/my`、`GET /system/disputes/:id`（baseURL 由 `config/env.js` 统一拼接 `/api/v4` 前缀）。

**(2) 认证头处理机制**（`utils/api/client.ts`）：
```
请求发起
  └─ needAuth=true ──► 从 userStore 取 access_token（Store 优先，Storage 降级）
                          ├─ 有 token ──► validateJWTTokenIntegrity 校验
                          │                 ├─ 通过 ──► header.Authorization = "Bearer <token>"
                          │                 └─ 不通过 ─► handleTokenInvalid（清登录态 + 引导重登）
                          └─ 无 token ──► handleTokenMissing（静默抛 MISSING_TOKEN，由页面引导登录）
```
三个售后接口均 `needAuth:true`，认证头由 APIClient 统一注入，页面无需手动处理。

**(3) Token 有效性检查机制**：
```
响应 401
  ├─ code=TOKEN_EXPIRED/SESSION_EXPIRED ─► handleTokenExpired（POST /auth/refresh 刷新 → 用新 token 重试原请求，_tokenRefreshed 防死循环）
  ├─ code=SESSION_REPLACED ──────────────► handleSessionReplaced（同平台他端登录，提示重登）
  └─ 其它（MISSING/INVALID/NOT_FOUND）───► handleTokenInvalid（清态 + 弹窗引导重登）
```
JWT 双 Token：access_token 过期自动用 refresh_token 刷新，刷新成功后透明重放原请求，对页面无感。

**(4) 错误处理逻辑**（按 HTTP 状态码分流，统一抛带 `code`/`statusCode` 的 ApiError）：
```
400 BAD_REQUEST / 403 FORBIDDEN / 404 NOT_FOUND / 409 CONFLICT(自动重试一次) /
429 RATE_LIMIT_EXCEEDED / 500 INTERNAL_ERROR / 503 SYSTEM_MAINTENANCE(全屏维护遮罩+健康轮询)
```
- 售后**列表/详情**页：`showError:true` 由 APIClient 统一 toast；详情页额外对 403/404 做差异化文案。
- 售后**详情**页对 `statusCode` 做了 403/404 判定（依赖 Q2 后端返回正确错误码）。

> 自检结论：APIClient 的认证头注入、Token 双令牌刷新、401 细分、错误码分流均**正常实现**，无兼容性/旧版分支。
> 唯一外部依赖：上述错误处理的"友好文案分流"正确性，取决于 Q2/Q3 后端返回的 `statusCode`/`code` 是否如约定。
