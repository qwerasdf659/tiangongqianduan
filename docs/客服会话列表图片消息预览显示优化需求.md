# 客服会话列表图片消息预览显示优化需求

> 提出方：微信小程序前端项目
> 性质：需求文档（基于真实代码核查，定位为后端字段缺失）
> 核查方式：通读微信小程序前端实际代码 + 真机截图复现，未引用历史报告
> 日期：2026-06-18（北京时间）

---

## 一、需求背景（客户体验问题）

在「在线客服」会话列表页，当一条会话的最后一条消息是**图片**时，列表预览区当前直接显示了一长串原始图片链接：

```
[我] https://objectstorageapi.bja.sealos.run/br0z...
```

**期望效果**（对齐主流 App / 其他公司客服列表的体验）：会话列表预览只显示"**[图片]**"两个字，不把对象存储链接暴露给客户。

业务诉求：优化客户体验，会话列表不应展示对客户无意义、且观感差的内部存储 URL。

---

## 二、问题定位（纯文本数据流图）

```
用户在客服会话里发了一张图片
   │  消息落库 chat_messages 表：content=图片URL，message_type='image'
   ▼
后端 GET /api/v4/system/chat/sessions  返回会话列表
   └─ 每个会话 last_message 对象 = { chat_message_id, content, sender_type, created_at }
                                                          ⚠️ 缺少 message_type 字段
   ▼
微信小程序前端 processSessionList()（packageUser/chat/chat.ts）
   const msgType = session.last_message.message_type   ← 取到 undefined
   contentText = (msgType === 'image') ? '[图片]' : session.last_message.content
                 └─ msgType 是 undefined → 条件不成立 → 落到 else → 原样显示 content（图片URL）
   ▼
列表显示：[我] https://objectstorageapi.bja.sealos.run/br0z...
```

**一句话定位**：前端"显示 [图片]"的逻辑早已写好，但它依赖 `last_message.message_type` 字段判断消息类型；后端会话列表接口的 `last_message` 对象未下发该字段，导致前端无法识别图片，只能把链接（content）原样显示。

---

## 三、前端现状核查（证明前端逻辑已正确，无需改前端）

### 3.1 前端预览处理逻辑（已实现"图片→[图片]"）

文件：`packageUser/chat/chat.ts` `processSessionList()`

```js
const msgType = session.last_message.message_type
const contentText =
  msgType === 'image'
    ? '[图片]'
    : msgType === 'location'
      ? '[位置]'
      : session.last_message.content
preview = senderPrefix + contentText
```

说明：只要 `message_type === 'image'` 就显示 `[图片]`，与期望体验完全一致。问题在于 `message_type` 取不到值。

### 3.2 前端类型声明（印证字段缺失）

文件：`typings/api.d.ts` `interface ChatSession`

```ts
last_message: {
  chat_message_id: number
  content: string
  sender_type: string
  created_at: string
} | null
```

`last_message` 对象仅 4 个字段，**无 `message_type`**，与后端实际返回一致。

### 3.3 对照：聊天详情接口是有 message_type 的

会话详情消息接口 `GET /api/v4/system/chat/sessions/:id/messages` 返回的单条消息含 `message_type=text/image/file`，详情页能正确区分。**唯独「会话列表」接口的 `last_message` 子对象漏了该字段**，问题范围精确。

---

## 四、为什么不在前端用"链接猜测"兜底（拒绝兼容性方案的理由）

不采用"content 以 http 开头就当图片"这类前端推测，原因：

1. **不准确**：图片消息、文件消息、用户分享的普通网址，`content` 都是 `https://...` 链接，外观无法区分。靠 URL 猜测会把文件/网址误判成"[图片]"，客户点开发现不符，体验更差。
2. **违反项目规范**：`message_type` 是消息的业务语义字段，权威来源是后端。前端按 URL 猜测属于"前端自行计算 / 兼容性兜底"，本项目规则明确禁止——数据一律以后端返回为准。
3. **可维护性**：未来新增消息类型（语音、视频等）时，前端猜测逻辑会持续失准；由后端统一下发 `message_type` 才能长期一致。

---

## 五、需要后端配合的内容（本需求的真正落点）

> 本需求微信小程序前端**零代码改动**即可生效，关键在后端补字段。

### 5.1 接口

`GET /api/v4/system/chat/sessions`（获取用户会话列表）

### 5.2 修改点

在返回的每个会话 `last_message` 对象中**补充 `message_type` 字段**，取值与 `chat_messages.message_type` 保持一致：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| message_type | string | 消息类型：`text` / `image` / `file` / `location` 等（与单条消息接口同源同名） |

补字段后 `last_message` 期望结构：

```
last_message: {
  chat_message_id: number,
  content: string,
  sender_type: string,
  message_type: string,   // ← 本次新增
  created_at: string
}
```

### 5.3 验收口径

- 最后一条是图片的会话：列表预览显示「[图片]」，不显示链接。
- 最后一条是文件的会话：可显示「[文件]」（前端会按 message_type 扩展，见下方第六节）。
- 最后一条是文字的会话：正常显示文字内容。

---

## 六、后端补字段后的前端跟进（待 message_type 到位后，前端这边的小优化）

后端补齐 `message_type` 后，前端 `processSessionList()` 当前已支持 `image` / `location`，建议同步补充 `file` 类型显示为「[文件]」，使三类非文字消息都不暴露链接。此项为前端职责，待后端字段就绪后由前端实施，不阻塞本需求主线。

---

## 七、结论

| 项目 | 是否本次问题责任方 | 说明 |
| --- | --- | --- |
| 微信小程序前端项目 | 否 | "图片→[图片]"逻辑已实现，依赖后端字段；后端补齐后零改动生效（仅后续可选补「[文件]」） |
| Web 端管理前端项目 | 否 | 与本问题无关 |
| 后端数据库项目 | **是** | 会话列表接口 `GET /api/v4/system/chat/sessions` 的 `last_message` 对象缺少 `message_type` 字段，需补齐 |

**给后端的一句话**：请在 `GET /api/v4/system/chat/sessions` 返回的 `last_message` 里增加 `message_type` 字段（值同 `chat_messages.message_type`），微信小程序前端即可把会话列表里的图片消息显示为「[图片]」而非原始链接。
