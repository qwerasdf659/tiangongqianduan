## 五、互锁表注意（⚠️ 高风险，必须整组处理）

按项目规则，以下两组是"互锁表"，**禁止单表手删**，否则多层数据对不上账：

- **材料资产互锁**：`account_asset_balances` ↔ `asset_transactions`（余额 = 流水累计结果）。当前余额总可用 1.42 亿、冻结 11.5 万，均为测试数据。
- **物品三表互锁**：`items` ↔ `item_ledger` ↔ `item_holds`（持有者 = 账本推导，状态 = 锁定记录）。

处理原则：这两组必须**整组一起清空**（随测试用户一并清零），不能只清其中一张表。保留账号（31/32/11021）若需要初始余额，清档后由 seeder 统一重建。

### 6.1 已有系统的能力（实测代码确认）

| 能力 | 说明 |
| --- | --- |
| 三种清理模式 | `manual` 手动（按类目+时间+用户/活动筛选）、`auto` 自动（按保留天数定时）、`pre_launch` **上线前清档** |
| 四级数据分级 | L0 不可删（系统基石：roles/categories/material_asset_types 等）；L1 金融级（asset_transactions/items 等，清前强制对账）；L2 业务运营；L3 日志可自动清 |

| 多重安全门 | 预览令牌（Redis 5 分钟）+ 确认文字"确认删除" + 管理员短信验证码二次确认 + 操作限流（每小时 1 次）+ 审计日志 + 干跑 dry_run |
| 生产保护 | `NODE_ENV=production` 直接禁止清档 |

### 6.2 推荐执行步骤（上线前清档 pre_launch）

1. **先做 Sealos 数据库快照**（清档不可逆，唯一兜底）。
2. **启用清档开关**：当前实测 `feature_flags.data_pre_launch_wipe` 的 `is_enabled=0`（未启用），需先置为启用（清档功能的硬性前置门）。
3. **Web 后台「数据管理」页 → 上线清档模式 → 预览**：调 `POST /api/v4/console/data-management/preview {mode:'pre_launch'}`，确认将删除的表与行数。
4. **建议先干跑**：执行时带 `dry_run:true` 验证范围无误。
5. **正式执行清档**：`POST /cleanup`，提交预览令牌 + reason + 确认文字"确认删除" + 管理员验证码。系统自动：进维护模式 → L1 对账 → 按拓扑顺序分批删 → 重建超管 → 退出维护模式 → 写审计日志。

### 6.3 清档「pre_launch」的删除范围 vs 本次保留诉求的差异（关键，需处理）

现有 `pre_launch` 清档是**全量清空**逻辑（`DELETE_TOPOLOGY` 全表 + users 全删 + accounts 仅留 system），它**不会保留**你确认要留的「老良记」「餐厅消费回馈」，也会清掉 `lottery_campaigns / lottery_campaign_prizes / item_templates / exchange_items / merchants` 等配置表。

因此有两条路线（见第八节拍板）：
- **路线A（推荐，最省事）**：先 `pre_launch` 全量清档（连同老良记/餐厅消费回馈一起清），上线后由运营在后台**重新录入**这 2 条真实配置。理由：它们目前关联的是测试用户产生的流水，全清更干净、无残留脏关联。
- **路线B（保留 2 条真实配置）**：不用 `pre_launch`，改用 `manual` 手动模式按类目+时间/用户筛选清流水，再人工清配置表里的测试项，**白名单保住** merchant_id=6、lottery_campaign_id=1。代价：手动模式当前未覆盖 `merchants/stores/exchange_items/item_templates` 等配置表（它们不在 L2_CLEANUP_CATEGORIES 里），这部分测试配置需另行处理。

### 6.4 现有系统的覆盖盲区（需补充处理的配置表）

`manual` 手动清理的类目（L2_CLEANUP_CATEGORIES）只覆盖**流水类**（抽奖/消费/客服/通知/反馈/兑换/竞拍/垫付）。下列**配置表里的测试垃圾不在任何清理类目中**，无法通过手动模式删除：

- `exchange_items`(64)、`stores`(5)、`store_staff`(5)、`item_templates`(34)、`merchants`、`media_files/media_attachments`

→ 若走路线B，这些需要单独处理（后台对应模块逐条删，或补充清理类目）。若走路线A（pre_launch），这些表已在 `DELETE_TOPOLOGY` 中会被一并清空。

### 6.5 可复用 / 可扩展

- **可复用（核心）**：整套 `DataManagementService`、`/console/data-management` 路由、Web 数据管理页、`ItemLifecycleService` 对账、`SUPER_ADMIN_MOBILE` 重建机制、维护模式、审计日志——全部现成，零新增。
- **可扩展**：若选路线B需保留指定配置，可在 `_previewPreLaunchWipe` 增加白名单参数（保留 merchant_id=6 / campaign_id=1），或给 L2_CLEANUP_CATEGORIES 增配置表类目。这是唯一可能的少量代码改动，需你确认后再做。
- **不新增技术债**：优先用现成系统，不再新建迁移/脚本。

## 九、扩展方案：新增「配置数据」清理类目（你的诉求，可行）

> 诉求：把配置表（exchange_items/stores/item_templates/merchants/media_files 等）也做成「清理类目」的形式，在数据管理页勾选清理。
> 结论：**可行，且完全复用现有 `L2_CLEANUP_CATEGORIES` 机制**，但因配置表有大量外键子表，必须连带处理依赖，不能简单加一个类目了事。

### 9.1 实测外键依赖（决定能否删、按什么顺序删）

| 配置表 | 被哪些子表外键引用（删父表前必须先清子表） |
| --- | --- |
| `exchange_items` | bid_products、exchange_records、exchange_item_skus、exchange_item_attribute_values、exchange_redeem_requirement |
| `item_templates` | items、exchange_items |
| `merchants` | items、material_asset_types、stores、prize_definitions |
| `stores` | store_staff、user_hierarchy、risk_alerts、redemption_orders、merchant_operation_logs、consumption_records |
| `media_files` | item_templates、prize_definitions、ad_creatives、exchange_item_skus、diy_works、diy_templates、categories、exchange_items、media_attachments |

→ 含义：直接 DELETE 这些配置表会被子表外键 RESTRICT 挡住。新增类目必须把"先清子表"纳入，且删除顺序要进 `DELETE_TOPOLOGY`。

### 9.2 推荐做法（最小改动、复用现有架构）

在 `services/DataManagementService.js` 的 `L2_CLEANUP_CATEGORIES` 增加一个配置类目，并依赖现有 `DELETE_TOPOLOGY` 已含的顺序（exchange_items/item_templates 已在第4/5层，stores/store_staff 已在拓扑中）。示例（仅方案，待拍板后落地）：

```js
// L2_CLEANUP_CATEGORIES 新增（配置数据，慎用：清空后需重新录入）
exchange_catalog: {
  label: '兑换商品配置（含SKU/属性/兑换记录）',
  tables: [
    'exchange_records', 'exchange_item_skus',
    'exchange_item_attribute_values', 'exchange_redeem_requirement',
    'exchange_items'   // 父表放最后
  ],
  time_field: 'created_at'
},
store_catalog: {
  label: '门店配置（含员工/层级）',
  tables: ['store_staff', 'user_hierarchy', 'stores'],
  time_field: 'created_at'
},
item_template_catalog: {
  label: '物品模板配置',
  tables: ['items', 'item_templates'],
  time_field: 'created_at'
}
```

要点（与现有机制一致）：
- 类目内 `tables` 顺序 = 先子表后父表；执行时 `executeCleanup` 已会再按 `DELETE_TOPOLOGY` 全局拓扑二次排序，双保险。
- 现有手动模式已支持 `time_range` 和 `filters.user_id` 等条件，**可按时间只清测试期数据**，不必全删。
- 若要按 `[测试]`/状态过滤（如只删 `status='inactive'` 或 name 含"测试"），需给类目加 `status_filter`（机制已有，`_previewManualCleanup` 已支持 `category.status_filter`），或扩展一个 name 模糊过滤条件。

### 9.3 这与「路线B保留白名单」如何配合

- 新增类目 + 现有 `filters` 不足以"排除某条 ID 保留"。若要**保留 merchant_id=6 / campaign_id=1 同时清其余配置**，需在 `_previewManualCleanup` 增加"排除白名单 ID"的条件（如 `WHERE merchant_id <> 6`）。这是约 10~20 行的可控改动。
- 改动范围：仅 `DataManagementService.js`（后端），Web 数据管理页无需改（类目是后端 `getStats().categories` 动态下发，前端自动渲染勾选项）。需确认前端是否动态渲染类目（见 9.4）。

### 9.4 待我进一步确认的点（落地前）

- Web 数据管理页的「清理类目」勾选项是写死的还是读后端 `categories` 动态渲染——决定加类目后前端要不要同步改。
- `merchants`/`media_files` 是否纳入类目：它们被极多表引用，全清风险高，建议**不做成类目**，由对应业务模块（商户管理/媒体库）单独处理，避免误删。

## 十、行业对标：各类公司怎么处理"上线前测试数据"，哪种最适合本项目

> 目的：帮你拍板。先讲清行业分水岭，再落到你项目实际技术栈给结论。

### 10.1 核心分水岭：有没有「环境隔离」

行业里这件事只有一个本质区别——**测试数据是否从一开始就和生产物理隔离**。

| 类型 | 环境策略 | 上线前怎么"清测试数据" | 数据销毁观念 |
| --- | --- | --- | --- |
| **大厂（美团/腾讯/阿里）** | dev / test / staging / prod **物理多库隔离**，配独立造数平台（mock/数据工厂） | **根本不需要清**——生产库一开始就是空的，测试数据只在 dev/test 库。上线=切到全新 prod 库 + 灰度放量 | 生产数据**只归档不物理删**（分区表 TTL、冷热分离、对账后归档），删除走合规审批 |
| **中小公司** | 常单环境或 dev/prod 两套，预算有限 | 上线前**清库重建**：truncate/重置 + seed 基线数据；或直接建一个干净的新库 | 多为直接删，靠备份兜底 |
| **游戏公司** | 删档测试 / 不删档测试 两种 | **"删档"概念**：测试期结束停服维护，整库清档→重置→正式开服；保留账号体系/配置，清玩家进度/资产 | 停服 + 快照,清档前必快照,可回滚 |
| **活动策划/运营公司** | 活动库按期建 | 活动结束**归档**整批数据;配置与流水分离,流水可清、配置沉淀复用 | 归档为主 |

## 十七、行业实情：生产库怎么来、运营怎么接手，哪种最适合本项目（2026-06-22）

> 关键认知纠正：本项目此前一直在纠结"清现库当生产库用"。但行业里生产库**几乎从不靠'清测试库'得到**，而是「全新空库起步 + 运营录入业务数据」。把测试库当生产库清，正是你"感觉风险大、成本高"的根因。

### 17.1 各类公司怎么得到生产库

| 类型 | 生产库怎么来 | 测试数据怎么办 | 运营接手什么 |
| --- | --- | --- | --- |
| 大厂（美团/腾讯/阿里） | prod 独立空库，CI/CD 跑 migration 建表，**绝不碰 dev/test 库** | 永远留在 dev/test 库，物理隔离 | 运营在 prod 后台/运营平台录入：商品、活动、券、门店、价格等**配置类**数据 |
| 中小公司 | 新建 prod 库 + migration + seed 基础字典 | 测试库照旧 | 运营后台录商品/活动 |
| 游戏公司 | 正式服=全新空库；"删档"是清**玩家进度**，不是把测试库当正式服 | 测试服独立 | 策划录：活动、奖池、道具、概率、商城 |
| 活动策划公司 | 每活动一套配置，活动库新建 | 归档 | 运营录活动配置 |
| 虚拟物品交易/二手平台 | prod 空库起步，强审计 | 隔离 | 运营录类目、商品模板、费率；用户数据靠真实用户产生 |

**共同点：生产库的"业务数据"分两类——**
- **配置数据**（商品、活动、奖品、券、门店、价格、字典）→ **由运营在后台录入**，这就是你说的"交给运营"。
- **用户行为数据**（注册、抽奖、交易、流水）→ **由真实用户上线后自然产生**，不需要任何人"准备"，上线前就该是空的。

### 17.2 落到你项目：你真正要的不是"清档"，是"换一个干净的生产库"

实测结论（见前序章节）：当前库 **100% 测试数据，无真实用户/无真实入金/无真实订单**，唯一要留的「老良记」「餐厅消费回馈」还是**配置**、不是用户数据。

因此对你最优的不是"在测试库上费劲清档"，而是：

| 方案 | 做法 | 风险 | 成本 | 技术债 |
| --- | --- | --- | --- | --- |
| **A. 新建空生产库（推荐）** | 新 MySQL 库 → 跑 migration 建表 → seed 字典/全局配置 → 运营在后台录商品/活动 → 切 `.env` 指向新库上线 | **0**（不在任何库删数据，现库原样留作测试） | 低（只录配置，本就要录） | **最低**（生产从第一天纯净，无需维护清档白名单/孤儿/分类保留等复杂逻辑） |
| B. 现库 pre_launch 清档 | 现成清档功能全量清 | 中（不可逆，需快照） | 中（要维护清档拓扑/分类保留/孤儿补全等复杂度） | 中（清档逻辑要长期维护正确） |

**结论：方案 A 最适合你。** 它正好命中你的三个目标——长期维护成本低（不用维护清档复杂逻辑）、降低技术债（生产库零污染）、一次性投入（建库+录配置，本来就要做）。而且 dev/test 与 prod 物理隔离，是大厂的标准做法，根治"测试数据混进生产"。

### 17.3 方案 A 的"运营接手"清单（你/运营在新库后台录入）

上线后由运营在 Web 后台录入（与第十五节"可自助补全"一致，全部有现成页面）：
1. 媒体图片（商品图/奖品图/材料图标/分类图标）
2. 门店（挂老良记商户）
3. 物品模板 → 兑换商品 + SKU + 渠道价
4. 奖品目录
5. 抽奖活动「餐厅消费回馈」+ 奖品 + 档位/矩阵/定价
6. 广告位（如启用）、客服坐席

系统自动/seed 准备好、运营无需管：超管账号、字典（roles/categories/材料类型/稀有度/资产分组）、全局系统配置（抽奖策略/分群/报表/提醒/成长等级/配额/垫付额度）。

### 17.4 方案 A 的技术落地（基于你现有技术栈，零新机制）

### 22.1 拍板项1 — 金融互锁组（asset_transactions/item_ledger/items/item_holds/account_asset_balances）怎么清

**问题本质**：近 10 万条资产流水 + 物品账本是"金融级互锁数据"（余额=流水累计、持有=账本推导），清它们既要清干净又要保证账实一致，且不能给运营开"裸删金融表"的口子。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂（美团/腾讯/阿里） | 资金/资产数据**永不物理裸删**；上线前生产库本就是空库，测试数据留在隔离的 test 库。真要清走**专用数据治理任务** + 对账门禁 + 审批 + 留痕，绝不开放给运营按钮 | 专用任务 + 强对账 + 物理隔离 |
| 小公司/SaaS | 直接 `TRUNCATE` 资产表 + 重置，靠备份兜底 | 粗暴、无对账 |
| 游戏公司 | "删档"清玩家资产/进度，走**停服维护 + 专用清档工具 + 快照**，清前清后跑一致性校验，配置/数值表保留 | 专用清档工具 + 快照 + 校验 |
| 活动策划公司 | 活动结束归档整批流水，配置沉淀；流水按活动周期清，但金额类先对账后归档 | 归档优先 |
| 游戏虚拟物品交易/小众二手 | **金融级铁律**：流水 append-only 不可变，错了用冲正分录；批量清必须"先对账→整组清→再对账"，绝不单表删 | 整组清 + 双向对账 |

**方案区别**：① 新增受 L1 对账保护的"上线前一次性清行为"专用后端方法（清前后各跑一次 `reconcileAssets/reconcileItems`，整组事务清空）；② 把金融表临时塞进手动清理类目让运营勾选删。

**对你项目的建议**：✅ **方案① — 新增受 L1 对账保护的专用清理方法**。
- 理由：这正是大厂"专用数据治理任务 + 对账门禁"与交易平台"整组清 + 双向对账"的合身落地。你已有 `ItemLifecycleService.reconcileAssets/reconcileItems` 现成对账能力，复用它做清前/清后守恒校验，零造轮子。
- 绝不选方案②：把金融表开放给手动类目 = 给运营一个"一键裸删近 10 万条资产流水且不对账"的危险按钮，违背金融级铁律、留长期误删隐患，是纯技术债。
- 落地（贴合现有架构）：在 `DataManagementService` 加一个 `_cleanupBehaviorData()`（或并入清档的一个子步骤），按 `DELETE_TOPOLOGY` 顺序在**单一事务**内清这 5 张表 + 其它行为表，清前断言对账通过、清后断言余额/持有归零且守恒。失败整体回滚。

### 22.2 拍板项2 — `admin_operation_logs` 是否保留清档当次审计

**问题本质**：清理历史记在 `admin_operation_logs`，清它会把"本次清档操作"的审计自己抹掉（自引用悖论）。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂 | 审计日志**异地独立存储**（独立审计库/日志中心/对象存储 WORM），与业务库分离，业务清库根本不碰审计 | 审计与业务物理分离、不可变 |
| 小公司/SaaS | 审计和业务同库，清库时连审计一起清，或简单留最近 N 条 | 同库、弱留存 |
| 游戏公司 | 清档操作写入**独立 GM 操作日志**，玩家数据清掉、GM 审计永久留 | GM 审计独立保留 |
| 活动策划公司 | 操作日志随活动归档，关键审计另存 | 归档 |
| 游戏虚拟物品交易/小众二手 | 合规要求**审计不可删**（金融留痕义务），清业务数据时审计强制保留 | 审计强制留存 |

**方案区别**：保留当次（清理逻辑加"排除本次操作/保留最近 N 条"条件）vs 全清（简单，连审计一起空）。

**对你项目的建议**：✅ **保留清档当次及关键审计**（清 `admin_operation_logs` 时排除本次清档操作记录；可选保留最近少量管理操作）。
- 理由：你是含虚拟资产交易的平台，"谁在何时清了什么"是必须留痕的合规底线，与交易平台/游戏 GM 做法一致。代价仅是清理条件加一个 `WHERE operation_type <> '本次清档' OR created_at < :本次开始`，代码量很小。
- 长期更优（列为路线图，本次不强制）：把审计日志迁到**独立留存**（独立表/对象存储），从根上消除"清业务连带清审计"的悖论——这是大厂标准，但属上线后优化，不阻塞本次。
- 本次最低成本落法：清行为时对 `admin_operation_logs` 用"保留本次操作 + 保留清档审计条目"的条件清，其余测试期日志清掉。

### 22.4 拍板项4 — `user_growth_levels`(4) 归属冲突（14.4 保留 vs 19.4 清除）

**问题本质**：成长等级（青铜/白银/黄金…）到底是"全局配置"（保留）还是"用户行为数据"（清）。文档自相矛盾。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂 | 等级**定义/阈值表**是配置（保留）；用户**当前等级/成长值**是行为（随用户清） | 定义与实例分离 |
| 小公司 | 常混在一张表，难区分 | 易混淆 |
| 游戏公司 | 等级体系（经验曲线/段位定义）是策划配置永久保留；玩家等级进度删档清 | 配置留、进度清 |
| 活动策划/二手平台 | 等级规则配置保留，用户达成记录随用户清 | 同上 |

**判定（直读真实库语义）**：`user_growth_levels`(4 行) 实测内容是**等级定义**（青铜/白银/黄金/…的阈值），属"全局配置/定义表"，**不是**每个用户的成长进度（用户进度在 users 或 user_premium_status 等）。

**对你项目的建议**：✅ **按 14.4 保留**（它是等级定义配置，非用户行为）。
- 理由：与大厂/游戏"等级定义=配置保留，用户进度=行为清"完全一致。19.4 把它列入清除是早期归类错误，以 14.4 保留为准。
- 落地：确保它**不在**任何清理类目、且从 `DELETE_TOPOLOGY` 的清除范围排除（与第十四节决定保留的 7 张全局配置表同口径处理）。

### 22.5 拍板项5 — 执行触发方式（一支迁移脚本一键跑 vs Web 后台手动 + 多脚本分步）

**问题本质**：清行为 + 删测试用户 + 归零统计 + DROP 垃圾表这一组操作，怎么触发最稳、最可审计、技术债最低。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂 | **可重放的数据治理任务/迁移**，带 dry_run + 审批 + 幂等 + 回滚说明，CI 留档 | 脚本化、原子、可审计 |
| 小公司 | 手工敲 SQL 或点后台按钮，步骤散、易漏 | 易出错、难复现 |
| 游戏公司 | 停服维护窗口跑**一个清档批处理**，一键、原子、有日志 | 单批原子操作 |
| 活动策划公司 | 半自动脚本 + 人工核对 | 混合 |
| 游戏虚拟物品交易/小众二手 | 金融操作走**单事务脚本 + 双向对账 + 留痕**，绝不分散手点 | 单事务 + 对账 |

**方案区别**：一支带 dry_run 的迁移/脚本（原子、可审计、可回滚说明、一次跑完）vs 拆成 Web 后台手点类目 + 几支独立脚本（步骤多、易漏勾错勾、难复现）。

**对你项目的建议**：✅ **一支带 dry_run 的一次性脚本/迁移，一键原子执行**（清行为→删测试用户→归零→DROP 在受控顺序内完成，关键金融步骤包在事务 + 清前后对账里）。
- 理由：贴合大厂"可重放数据治理任务"与游戏"单批清档"。一次性、可 dry_run 预览、全程审计、顺序固定、可复现，远优于"后台手点多个类目 + 散落脚本"（后者易漏勾/错勾、无法复现、是长期技术债）。
- 与现有架构的结合：复用 `DataManagementService` 的预览/dry_run/审计/对账能力，把这套行为清理逻辑收敛进一个方法（或一支调用该 Service 的脚本），而不是让运营在 Web 上手点——运营手点保留给"上线后日常清理"（即事项1新增的配置类目那种低风险场景）。
- 注意：`deadlock_test_*` 的 DROP 走 sequelize-cli 迁移（DDL 归迁移管），与上面的 DML 清理脚本分开但同批执行。

### 28.1 ⚠️ 媒体清理高危 bug 及修复（与 §16 媒体清理同源，必读）

**现象**：上线联调中发现首页 banner 图、部分兑换商品图"配好后约 24 小时消失"。

**根因（直读代码 + 真实库验真）**：每小时定时任务 `jobs/hourly-cleanup-unbound-media.js` 调 `MediaService.getOrphanedMedia()` 判定"孤儿图"时，**只查 `media_attachments` 关联，漏查 `primary_media_id` 直引**。而图片有**两套引用方式**：

| 引用方式 | 用途 | 旧清理逻辑是否识别 |
| --- | --- | --- |
| `media_attachments`（多态关联，role=icon/gallery 等） | 资产图标/分类图标（红源晶/积分/星石等）、商品多图 | ✅ 识别 → 不误删（所以奖品图一直安全） |
| `primary_media_id`（外键直引） | `ad_creative`(banner/广告) / `exchange_item`(商品) / `prize_definition` / `item_template` 主图 | ❌ **漏查 → 误判为孤儿 → 物理删除** |

**连锁机理**：图被 `MediaFile.destroy` 物理删 → 数据库外键 `ON DELETE SET NULL`（`ad_creatives`/`item_templates`/`prize_definitions` 均为 SET NULL）自动把引用方 `primary_media_id` 置空 → 接口下发 `image_url=null` → 前端显示占位框。`exchange_items` 外键为 `NO ACTION`，故商品图删不掉（被外键挡住、清理任务报 failed），图侥幸还在但每小时被反复尝试删。

**修复（已落地并验真）**：`services/MediaService.js` 的 `getOrphanedMedia()` 把 `PRIMARY_MEDIA_TABLES`（exchange_item/prize_definition/item_template/ad_creative）四表的 `primary_media_id` 引用一并纳入"在用 media_id"集合，孤儿判定改为"既不在 media_attachments、也不被任何业务表 primary_media_id 引用"。复用现有 `PRIMARY_MEDIA_TABLES` 常量（无硬编码）。

**验真**：真实库实测被 `primary_media_id` 引用的 media（如商品图 121/122）修复后不再被判为孤儿（孤儿集对这些图为空）；ESLint 通过、`tests/jobs/hourly-cleanup-unbound-media.test.js` 4/4 通过、`/health`=healthy。

**责任归属**：纯后端 bug（清理逻辑漏一套引用），与 Web 管理端/小程序前端无关；管理端上传→绑定 `primary_media_id`→建 creative 链路、小程序取图链路均无 bug。已被误删的 banner 图为物理删除、不可恢复，需运营在管理端重新上传（修复后不再被删）。

> 关联提示：本 bug 与 §16「media 分类保留」同属"媒体清理正确性"。§16 保住的是 materials/categories 图标（走 media_attachments）；本章补上的是 primary_media_id 直引图（banner/商品/奖品/模板主图）。两套引用都纳入保护后，媒体清理才真正完整。
