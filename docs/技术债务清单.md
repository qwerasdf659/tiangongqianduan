## 一、立即修复（运行时风险）

### TD-001 camera.ts searchTimer 未在 onUnload 清理

**文件**: `pages/camera/camera.ts`
**严重度**: P0 — 运行时异常风险
**验证状态**: ✅ 已确认

`onSearchInput` 在第 216 行创建了 300ms 防抖定时器：

```typescript
this.searchTimer = setTimeout(() => {
  this.filterActivities()
}, 300)
```

但 `onUnload`（第 80-84 行）只清理了 MobX bindings，遗漏了 `searchTimer`：

```typescript
onUnload() {
  if (this.storeBindings) {
    this.storeBindings.destroyStoreBindings()
  }
  // ❌ 缺少: if (this.searchTimer) clearTimeout(this.searchTimer)
}
```

**风险**: 用户在搜索输入后立即离开页面，300ms 后 `filterActivities()` 仍会执行，`this.setData` 操作已卸载的页面实例。

**修复方案**: 在 `onUnload` 中添加 `if (this.searchTimer) clearTimeout(this.searchTimer)`。

### TD-002 `any` 类型全项目泛滥（300+ 处）

**文件**: 全项目 45+ 个 .ts 文件
**严重度**: P0 — 类型安全系统性失效
**验证状态**: ✅ 已确认（`as any` 出现 100+ 次，`: any` 出现 300+ 次）

典型问题分布：

| 类别 | 示例 | 涉及文件数 |
|------|------|-----------|
| 组件 data 声明 | `[] as any[]`, `null as any` | 20+ |
| 事件处理参数 | `onProductTap(e: any)` | 30+ |
| Store action 的 this | `action(function(this: any, ...)` | 全部 Store |
| 工具函数参数 | `getListingDisplayImage(listing: any)` | 5+ |
| API 函数缺少返回类型 | `async function userLogin(...)` 无 `: Promise<>` | 60+ |
| Socket.IO 事件数据 | `socket.on('new_message', (data: any) => ...)` | app.ts 全部 13 个事件 |

**影响**: TypeScript 类型检查被系统性绕过，项目已有完善的 `typings/api.d.ts`（定义了 `API.ExchangeProduct`、`API.MarketListing` 等 30+ 接口），但大量页面和组件未引用这些类型。

**修复方案**: 按层次逐步替换，优先级：Store action `this` → API 函数返回类型 → 页面事件参数 → 组件 data 声明。

## 二、短期处理（1-2 周）

### TD-003 积分余额刷新逻辑重复 3 份

**文件**: `pages/user/user.ts`(第 435-454 行) | `pages/lottery/lottery.ts`(第 450-468 行) | `pages/exchange/exchange.ts`(第 189-205 行)
**严重度**: P1 — 维护成本
**验证状态**: ✅ 已确认

三个页面各自实现了完全相同的模式：

```
调用 API.getPointsBalance()
→ 校验 balanceResult.success && balanceResult.data
→ 提取 available_amount / frozen_amount
→ 更新 pointsStore.setBalance(...)
→ 更新页面 this.setData({ totalPoints: ... })
→ catch 中降级读取 pointsStore 缓存值
```

**修复方案**: 在 `store/points.ts` 中新增 action 方法 `refreshFromAPI()`，将 API 调用 + 校验 + 更新逻辑收敛为一处。页面只需 `await pointsStore.refreshFromAPI()`。

### TD-004 登录状态检查 4 种模式不统一

**文件**: `pages/user/user.ts`(第 388 行) | `pages/camera/camera.ts`(第 90 行) | `pages/lottery/lottery.ts`(第 155 行) | `pages/exchange/exchange.ts`(第 92 行)
**严重度**: P1 — 一致性风险
**验证状态**: ✅ 已确认

| 页面 | 检查方式 | 重定向行为 |
|------|---------|-----------|
| user.ts | 手动: `userStore.isLoggedIn && !!userStore.accessToken` | 手动跳转 `/packageUser/auth/auth` |
| camera.ts | 手动: `userStore.isLoggedIn && !!userStore.accessToken` | 不重定向（允许未登录浏览） |
| lottery.ts | 工具函数: `checkAuth()` | 由 checkAuth 内部控制 |
| exchange.ts | 工具函数: `checkAuth()` | 由 checkAuth 内部控制 |

**修复方案**: 统一使用 `checkAuth(options?)` 工具函数，通过 `{ redirect: false }` 参数控制是否重定向。消除手动拼接的登录检查逻辑。

### TD-005 用户信息恢复逻辑重复 2 份

**文件**: `pages/user/user.ts`(第 380-386 行) | `pages/exchange/exchange.ts`(第 110-117 行)

**验证状态**: ✅ 已确认

两个页面各自从 Storage 恢复 userInfo 的逻辑完全相同：

```typescript
let userInfo = userStore.userInfo
if (!userInfo || !userInfo.user_id) {
  userInfo = wx.getStorageSync('user_info')
  if (userInfo && userInfo.user_id) {
    userStore.updateUserInfo(userInfo)
  }
}
```

**修复方案**: 在 `store/user.ts` 中新增 `ensureUserInfo()` 方法封装此逻辑。

### TD-006 缓存管理器代码重复

**文件**: `utils/config-cache.ts`(391 行) | `utils/exchange-config-cache.ts`(391 行)
**严重度**: P2 — 可维护性
**验证状态**: ✅ 已确认

两个文件实现了相同的 4 层降级缓存架构（本地缓存 → 远程 API → 过期缓存 → 内置默认），核心方法名和逻辑结构高度一致：

| 方法 | config-cache | exchange-config-cache |
|------|-------------|----------------------|
| 获取配置 | `getConfig()` | `getConfig()` |
| 读缓存 | `_getCachedConfig()` | `_getLocalCache()` |
| 写缓存 | `_saveToCache()` | `_saveToLocal()` |
| 静默更新 | `_updateConfigInBackground()` | `_silentUpdate()` |
| 强制刷新 | `forceRefresh()` | `forceRefresh()` |
| 清除缓存 | `clearCache()` | `clearCache()` |

**修复方案**: 提取泛型基类 `CacheManager<T>`，通过参数化缓存键名、API 端点、默认配置来消除重复。

### TD-008 错误处理 3 套系统共存

**文件**: `utils/simple-error.ts` | `utils/api-wrapper.ts` | `utils/api/client.ts`
**严重度**: P1 — 用户体验不一致
**验证状态**: ✅ 已确认

| 系统 | 调用方式 | UI 表现 | 适用场景 |
|------|---------|---------|---------|
| `simple-error.ts` `handleError()` | `handleError(error, '操作')` | `wx.showModal` 阻塞弹窗 | JWT过期/网络/业务错误 |
| `api-wrapper.ts` `safeApiCall()` | `safeApiCall(() => API.xxx(), {context: '操作'})` | `wx.showToast` 非阻塞提示 | 通用API调用包装 |
| `api/client.ts` `handleResponse()` | 自动触发（API层内部） | 抛出 `ApiError` 对象 | 底层请求错误 |

各页面在三种模式间随意选择，同一文件内混用多种。`safeApiCall` 内部还会调用 `handleError`，形成嵌套。

**修复方案**: 明确分层约定 — API 层统一抛 `ApiError`，页面层统一使用 `safeApiCall` 包装，弹窗级错误仅限 JWT 过期场景。

### TD-015 lottery.ts QR 刷新 setTimeout 未追踪

**文件**: `pages/lottery/lottery.ts` 第 655-657 行
**严重度**: P3（原标记 P0，下调）
**验证状态**: ✅ 已确认但下调严重度

```typescript
setTimeout(() => {
  this.generateUserQRCode()
}, 2000)
```

此定时器未赋值给变量，`onUnload` 中无法取消。但 lottery 是 TabBar 页面，几乎不会触发 `onUnload`（切换标签仅触发 `onHide`），且页面实例在隐藏状态下仍然有效。2 秒后在隐藏页面上执行 QR 码生成仅浪费资源，不会导致崩溃。

**修复方案**: 将 setTimeout 返回值赋给 `this._qrRefreshTimer`，在 `onHide` 和 `onUnload` 中清理。
