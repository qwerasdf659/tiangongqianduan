# 后端需求：交易市场行情接口对未登录用户开放（匿名可访问）

> **状态：⏳ 待后端实现**
> **提出日期：2026-06-02**
> **提出方：微信小程序前端**
> **优先级：中（不阻断已上线功能，影响未登录用户的行情查看体验）**

---

## 一、需求背景（业务语义）

交易市场（积分商城 → 交易市场 Tab）的「查看行情」面板，展示的是**市场公开行情数据**（价格走势、成交量、价格摘要）。

产品决策：**这些只读行情数据对未登录游客开放**，让未登录用户也能查看市场趋势，降低注册转化门槛。购买、挂单等涉及资产变动的操作仍要求登录，不在本次范围内。

前端已配合放开「查看行情」按钮的点击拦截（未登录也能展开面板）。但面板展开后会调用后端 4 个行情接口拉数据，而**这 4 个接口目前后端要求 JWT 登录鉴权**，导致未登录用户展开面板后接口返回 401、面板为空。

因此需要后端将这 4 个行情接口改为**匿名可访问（无需登录即可调用）**。

---

## 二、问题数据流（纯文本）

```
未登录游客点击「查看行情」
        │
        ▼
前端展开 price-chart 行情面板（前端已放开拦截 ✅）
        │
        ▼
price-chart 组件并行请求 4 个后端行情接口
        │
        ▼
接口当前要求登录鉴权 → 游客无 JWT Token → 后端返回 401 ❌
        │
        ▼
面板能打开，但数据为空（前端 catch 后展示空态）
```

根本原因：**鉴权是后端在路由层强制的**。前端的「是否携带 Token」开关无法绕过后端鉴权——只要后端路由仍要求登录，匿名请求一律 401。**该接口是否对匿名开放，决定权在后端。**

1. **是否同意将上述只读行情接口对匿名用户开放？**

# 后端需求 - 小程序版本闸门接口（强制更新）

> 用途：让前端在小程序冷启动时判断"当前用户版本是否过低"，过低则全屏强制更新、连首页都进不去，从业务层兜住"微信缓存旧代码"的用户。
>
> 前端已完成对接，等待后端提供下方接口即可生效。
>
> 文档版本：v1.0 ｜ 关联前端版本：5.2.0 ｜ 日期：2026-06-02

---

## 一、背景与目标

微信小程序自带的 `wx.getUpdateManager` 有两个硬限制：

1. 只能在**冷启动**时检查更新；
2. 无法拦住"还没更新就在用旧版"的用户（已打开旧版的用户要等下次冷启动）。

因此纯前端做不到"所有旧版用户立刻被挡住"。解决方案是加一个**后端版本闸门**：小程序启动时请求一个接口，拿到"最低可用版本号"，前端把本地版本和它对比，低于就触发全屏强制更新遮罩。

这样即使微信端还缓存着旧代码，业务层也会把低版本用户拦在更新页，无法进入正常功能。

---

## 二、需要后端提供的接口

### 接口：获取小程序版本闸门配置

| 项目 | 内容 |
| --- | --- |
| **Method** | `GET` |
| **Path** | `/api/v4/system/app-version` |
| **认证** | 无需认证（`optionalAuth` 或公开，**不能要求登录**） |
| **调用时机** | 小程序冷启动 `onLaunch` 阶段，每次启动 1 次 |
| **幂等** | 是（纯读取） |

#### 请求参数

无必填参数。可选支持 `platform` 查询参数（前端暂不传，后端可忽略）：

```
GET /api/v4/system/app-version
```

#### 响应格式（成功）

沿用项目统一响应结构 `{ success, code, data }`：

```json
{
  "success": true,
  "code": "APP_VERSION_SUCCESS",
  "data": {
    "latest_version": "5.3.0",
    "min_version": "5.2.0",
    "force_update": true,
    "update_message": "检测到新版本，请更新后继续使用",
    "platform": "miniprogram"
  }
}
```

#### 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `latest_version` | string | 否 | 当前线上最新版本号（语义化版本，如 `5.3.0`），仅展示/统计用 |
| `min_version` | string | **是** | **最低可用版本号**。前端本地版本 `< min_version` 时强制更新。这是闸门核心字段 |
| `force_update` | boolean | **是** | 强制更新总开关。`false` 时前端直接放行，不做任何拦截（用于随时一键关闭闸门） |
| `update_message` | string | 否 | 强制更新遮罩上显示的提示文案，不传则用前端默认文案 |
| `platform` | string | 否 | 平台标识，固定 `miniprogram` |

> **版本号格式**：统一用 `主.次.修订`（如 `5.2.0`）。前端比较逻辑按 `.` 分段转数字逐段比较，缺失段补 0（`5.2` 等价 `5.2.0`）。请勿带 `v` 前缀或后缀（不要 `v5.2.0`、`5.2.0-beta`）。

## 三、前端处理逻辑（已实现，供后端理解联调）

前端在 `app.ts → onLaunch` 调用 `checkVersionGate()`，逻辑如下：

1. 请求 `GET /system/app-version`；
2. 若 `success !== true` 或无 `data` → **放行**（不拦截）；
3. 若 `force_update !== true` 或缺 `min_version` → **放行**；
4. 比较本地版本（`globalData.version`，当前 `5.2.0`）与 `min_version`：
   - 本地 `< min_version` → 进入强制更新模式，全屏遮罩拦截，按钮"立即更新"；
   - 否则 → 放行。
5. **接口超时/报错 → 一律放行**（容错优先，避免后端抖动导致全员打不开小程序）。

> 关键约束：这个接口**绝不能阻塞启动**，也不能因为它失败而让用户进不去。拦截只在"明确返回了更高的 min_version"时发生。

## 四、版本号来源说明（前端侧）

- 前端本地版本取自 `app.ts` 的 `globalData.version`，当前为 `5.2.0`，每次发版前端会同步更新该值与 `package.json` 的 `version`。
- 后端的 `min_version` 由运营/研发在发新版后手动配置（建议存到 `system_configs` 表，`config_key = 'app_min_version'` 之类），无需读取微信的版本号。
- 微信平台层面的"线上版本号"和这里的 `min_version` 是两套，互不依赖。`min_version` 是业务自定义的强制基线。

## 五、建议的后端实现要点

1. **数据来源**：建议存 `system_configs` / `system_settings` 表，key 形如：
   - `app_min_version` → `min_version`
   - `app_latest_version` → `latest_version`
   - `app_force_update` → `force_update`（布尔）
   - `app_update_message` → `update_message`
2. **可热改**：运营改表即可调整闸门，不需重新部署后端。
3. **默认安全值**：配置缺失时建议返回 `force_update: false`（即默认不拦截），避免误伤。
4. **缓存**：该接口高频调用（每次冷启动），建议加内存/Redis 缓存，TTL 30~60s 即可。
5. **复用 status 风格**：可参考现有 `GET /api/v4/system/status` 的实现风格（同为公开、轻量、不依赖登录）。

### 实际实现方案（2026-05-28）

在 `services/exchange/QueryService.js` 中新增 `_attachPayAssetNames()` 私有方法：

- **数据来源**：`material_asset_types` 表（非 `asset_group_defs`，后者是分组级别）
- **关联字段**：`exchange_records.pay_asset_code` = `material_asset_types.asset_code`
- **输出字段**：`pay_asset_name`（取自 `material_asset_types.display_name`）
- **影响接口**：
  - `GET /api/v4/exchange/orders`（用户订单列表）
  - `GET /api/v4/exchange/orders/:order_no`（用户订单详情）

```javascript
// services/exchange/QueryService.js — _attachPayAssetNames()
async _attachPayAssetNames(orders) {
  const list = Array.isArray(orders) ? orders : [orders]
  const assetCodes = [...new Set(list.map(o => o.pay_asset_code).filter(Boolean))]
  if (assetCodes.length === 0) return

  const MaterialAssetType = this.models.MaterialAssetType
  const assets = await MaterialAssetType.findAll({
    where: { asset_code: assetCodes },
    attributes: ['asset_code', 'display_name'],
    raw: true
  })

  const nameMap = {}
  assets.forEach(a => { nameMap[a.asset_code] = a.display_name })

  list.forEach(order => {
    if (order.pay_asset_code && nameMap[order.pay_asset_code]) {
      order.pay_asset_name = nameMap[order.pay_asset_code]
    }
  })
}
```
