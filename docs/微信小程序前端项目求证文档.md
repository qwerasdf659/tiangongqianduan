## 二、需求 1：行情 4 接口对匿名开放

### 2.1 实库 / 实码核对结果

- 路由文件：`routes/v4/marketplace/price.js`，4 个接口**真实存在**，挂载于 `/api/v4/marketplace`（`app.js:689`）：
  - `GET /price/trend`、`GET /price/volume`、`GET /price/summary`、`GET /price/recent-trades`
  - **当前 4 个都用 `authenticateToken`**（强制登录）→ 匿名调用返回 401，与前端反馈一致 ✅
- 服务：`services/market/PriceDiscoveryService.js`，基于 `trade_orders` + `market_listings` 聚合，**零新建表**
- 实库数据量（2026-06-02 实时查询）：`trade_orders` **168 行**（completed **29**、cancelled 139）、`market_listings` **347 行** → 有真实成交可出图；completed 成交时间跨度 2026-01-28 ~ 2026-05-26
- `optionalAuth` 中间件**已存在**（`middleware/auth.js:809`）且 `system/status.js`（第33/96行）已在用 → 本需求**无需新写中间件**

### 2.2 这是谁的问题

- **后端数据库项目的问题**：鉴权在路由层强制，决定权在后端（前端 `needAuth` 开关绕不过）。**主改动在后端。**
- **小程序前端的问题**：前端文档里写的响应字段与后端真实返回**对不上**（见 2.4），需前端改字段适配后端。

### 2.3 后端改法（权威做法，长期零债务）

`routes/v4/marketplace/price.js` 把 4 个路由的 `authenticateToken` 换成 `optionalAuth`：

```js
// 改前
const { authenticateToken } = require('../../../middleware/auth')
router.get('/price/trend', authenticateToken, asyncHandler(...))

// 改后（匿名可访问；登录用户仍会被识别，行为不变）
const { optionalAuth } = require('../../../middleware/auth')
router.get('/price/trend', optionalAuth, asyncHandler(...))
```

四个路由全部同样替换。`optionalAuth` 对匿名请求 `next()` 放行、对带 Token 请求照常挂 `req.user`，**不破坏已登录体验**，符合验收第 2 条。

- 行情接口**不返回买卖双方昵称 / user_id / 订单号**（`getLatestTrades` 只返回 `trade_order_id`、金额、`completed_at`、资产信息）→ **匿名开放无脱敏顾虑**，建议第六节问题 1/2 直接全开。
- 限流：项目已有 `middleware/RateLimiterMiddleware.js` + `.env ENABLE_RATE_LIMIT=true`，如需按 IP 防刷在该 4 路由叠加即可（可选，非阻断）。
- 维护模式：`maintenance_mode=true` 时这些 `/api/*` 仍会 503（符合预期，无需特殊处理）。

## 三、需求 2：小程序版本闸门 `GET /api/v4/system/app-version`

- 现成可参考：`routes/v4/system/config.js` 里的公开只读接口（如 `/system/config/exchange-page`）已是"无鉴权 + 读 `getConfigValue` + 缺省返回默认值"范式，**版本闸门完全套用此范式**。

### 3.2 这是谁的问题

- **后端数据库项目**：新增 1 个公开只读接口（必须做）。
- **Web 管理后台前端**：是否要可视化配置版本闸门（可选增强，见 3.5）。
- **小程序前端**：已实现 `checkVersionGate()`，等接口即可（前端无需再改，除非字段微调）。

### 3.3 后端落地方案（权威做法）

**配置存储**：用既有"自由 JSON 配置"通道，存 1 个 key 到 `system_settings`，避免新增 4 个白名单项（降债务）：

```
setting_key = 'app_version_gate'
value_type  = 'json'
category    = 'feature'
setting_value(JSON) = {
  "latest_version": "5.3.0",
  "min_version": "5.2.0",
  "force_update": false,        // 默认安全值：不拦截
  "update_message": "检测到新版本，请更新后继续使用",
  "platform": "miniprogram"
}
```

读写直接用 `AdminSystemService.getConfigValue('app_version_gate')` / `upsertConfig('app_version_gate', {...}, { category:'feature' })`，无需动白名单（白名单只约束 category+key 的批量更新通道）。

**新增路由文件** `routes/v4/system/app-version.js`（套 `config.js` 范式，公开、无 `authenticateToken`）：

```js
'use strict'
const express = require('express')
const router = express.Router()
const { asyncHandler } = require('../../../middleware/validation')

const DEFAULTS = {
  latest_version: null,
  min_version: '0.0.0',
  force_update: false,           // 缺省不拦截，避免误伤
  update_message: '检测到新版本，请更新后继续使用',
  platform: 'miniprogram'
}

router.get('/app-version', asyncHandler(async (req, res) => {
  const svc = req.app.locals.services.getService('admin_system')
  const cfg = await svc.getConfigValue('app_version_gate')
  const data = { ...DEFAULTS, ...(cfg || {}) }
  return res.apiSuccess(data, '获取版本闸门配置成功', 'APP_VERSION_SUCCESS')
}))

module.exports = router
```

**挂载**（`routes/v4/system/index.js`，与 status 同级，公开）：

```js
const appVersionRoutes = require('./app-version')
router.use('/', appVersionRoutes)   // → GET /api/v4/system/app-version
```

- 响应即前端约定的 `{ success, code:'APP_VERSION_SUCCESS', data:{...} }`，字段名 `snake_case` 全部对齐前端，**前端无需改**。
- 公开范式天然不返回 401，满足验收"未登录可调"。
- 缓存：`getConfigValue` 已带 Redis 缓存（`SystemConfigService` 300s TTL），满足"高频冷启动加缓存"。
- ⚠️ 维护模式：`maintenance_mode=true` 时 `/api/v4/system/app-version` 会被维护中间件 503（它只放行 auth/console/health）。**已拍板（项5）：把该路径加入 `middleware/maintenanceMode.js` 的 `BYPASS_PREFIXES`**，确保维护期可读、能引导用户更新。

### 3.4 force_update 安全语义（与前端逻辑对齐）

前端 `checkVersionGate()`：接口报错/超时/`force_update!==true`/缺 `min_version` 一律放行，只有"明确更高 `min_version` + `force_update:true`"才拦。后端默认 `force_update:false` 与之吻合——**配置缺失绝不误拦**。

### 3.5 Web 管理后台前端：可选增强（非阻断）

- **MVP（推荐先做）**：不动管理后台，运营改库 / 用脚本写 `app_version_gate` 即可上线闸门。
- **增强（可后置）**：在管理后台加一个"版本闸门"配置页/卡片。
  ⚠️ **复用对象纠偏（实码核对结论）**：**不能**复用系统设置页 `system/composables/config.js`——该页保存走 `PUT /api/v4/console/settings/:category` → `AdminSystemService.updateSettings()`，而 `updateSettings` **强制校验白名单**（`config/system-settings-whitelist.js`，`AdminSystemService.js:707-712`：键不在白名单直接报错 `配置项 xxx 不在白名单内，禁止修改`）。`app_version_gate` 是自由 JSON key、不在白名单，**走这条通道会被拒**。
  ✅ **正确可复用范式**：照搬**已存在的"兑换页面配置"那一套**——后端 `routes/v4/console/system/exchange-page-config.js`（`GET/PUT`，内部用 `AdminSystemService.upsertConfig('exchange_page', body, { category:'feature' })` 直接 upsert、**绕开白名单**），前端 `admin/src/modules/system/composables/exchange-page-config.js`（`apiGet/apiPut` 一个独立 ENDPOINT + 表单）。新增一个 `app-version-config` 路由+composable 即可，**与现有 Vite+Alpine 技术栈完全一致、零新框架**。
  → 即：闸门管理入口若要做，应新增 `console/system/app-version-config`（仿 `exchange-page-config`），**不要**塞进通用系统设置页。

### 10.1 拍板项①②：行情数据要不要对未登录游客开放（含 90 天）

**各类公司怎么设计**

| 类型 | 代表 | 行情/价格数据对游客的策略 |
|------|------|--------------------------|
| 游戏饰品交易平台 | Steam 市场、网易 BUFF、悠悠有品 | **价格走势、历史成交、在售列表全部游客可看**，不登录就能浏览行情和参考价；只有"下单/绑定/提现"才要登录。把"看行情"当作拉新转化的核心钩子 |
| 电商大厂 | 美团/淘宝/京东 | 商品价、销量、评价**游客可见**；只有下单/收货地址/优惠券才登录。读写分离，读接口默认匿名 |
| 二手/小众平台 | 闲鱼、得物 | 商品和成交价**游客可见**，登录墙只设在交易动作 |
| 活动策划/积分商城类小公司 | 多数 SaaS 积分系统 | 两种都有：偷懒的全站登录墙；规范的把"行情/商品展示"做成公开只读接口 |

**区别本质**：是把登录墙设在**"看"**还是**"交易动作"**上。行业主流（尤其交易类）一致把墙设在交易动作，**只读行情对游客开放**，因为行情是公开市场信息、不含隐私，且是转化漏斗入口。

**最适合你的选择 → 全开（①是、②是，含 90 天）**
- 你的行情接口实测**不含买卖双方昵称/user_id/订单号**（`PriceDiscoveryService` 只返回金额+时间+资产），与 BUFF"历史成交公开可查"同构，**无脱敏顾虑**。
- 你已明确"降低注册转化门槛"，与行业把行情当拉新钩子的目的一致。
- 落到你的栈：4 个 `price/*` + `analytics/history` 把 `authenticateToken` 换 `optionalAuth`（已存在中间件，零新依赖、零新表）。这是**最低技术债务**的实现——不是加白名单层、不是复制一套匿名接口，而是复用同一套路由+服务，仅换鉴权件。

**各公司具体怎么划"登录墙"（细化到动作粒度）**

| 动作 | Steam 市场 | 网易 BUFF / 悠悠有品 | 美团/淘宝/京东 | 闲鱼/得物 | 你应采用 |
|------|-----------|--------------------|---------------|----------|---------|
| 看价格走势/历史成交 | 游客可看 | 游客可看（"历史成交公开可查"） | 游客可看 | 游客可看 | **游客可看** |
| 看在售列表/详情 | 游客可看 | 游客可看 | 游客可看 | 游客可看 | 游客可看（本次不改） |
| 下单/购买 | 登录 | 登录+绑定 Steam | 登录 | 登录 | 登录（`authenticateToken` 不动） |
| 挂单/出售 | 登录 | 登录+绑定 | 登录（开店） | 登录 | 登录（不动） |
| 提现/资产变动 | 登录+二次校验 | 登录+实名+二次校验 | 登录+支付密码 | 登录+实名 | 登录+敏感操作中间件（不动） |

**关键差异点**：Steam 自营市场手续费高达 15%，BUFF/悠悠靠 1.5%–2.5% 低费率 + **更强的行情工具（走势图、挂刀比例、建议价）** 抢用户——它们把"行情可视化"当作核心竞争力，**绝不锁在登录后**。你的行情面板定位与此完全一致：**行情是拉新钩子，必须对游客开放**。

> 反例（不要学）：部分积分商城类 SaaS 图省事做"全站登录墙"，未登录连首页商品都看不到——这类产品注册转化率显著偏低，是典型的"用工程便利牺牲业务转化"。你已明确要降低注册门槛，应明确避开。

### 10.2 拍板项③④⑤：版本闸门怎么做（限流 / 配置入口 / 维护期）

#### (③) 匿名行情接口要不要现在就加限流

**各类公司怎么设计**
- **大厂（美团/腾讯/阿里）**：公开只读接口**默认在网关层按 IP 限流**（令牌桶/滑动窗口 + Redis），超限返回 `429 Too Many Requests` + `Retry-After`，是标配。
- **中小项目**：业界共识（反爬实践原文）——"中小项目做好**接口鉴权 + 频率限制**两点就够了；复杂签名/行为风控只在数据价值高、爬虫烈度大时才投入"。
- **游戏交易平台**：行情接口限流较严（怕被刷价/爬库存），常按 IP + 设备维度。

**区别本质**：限流是**成本/风险的权衡**。不加→省事但可能被爬；加→多一层防护但要调阈值避免误伤高频真实用户。

**最适合你的选择 → 上线先开"宽松 IP 限流"，不做签名/行为风控**
- 你的行情是聚合统计、价值密度低，被爬损失小，没必要上签名/风控（过度设计=技术债）。
- 但"完全不限"在公开后有被刷风险。**折中**：直接复用你已有的 `middleware/RateLimiterMiddleware.js` + `.env ENABLE_RATE_LIMIT=true`，在 5 个匿名行情路由挂一个**宽松 IP 维度限流**（如 60–120 次/分钟/IP），超限走你统一响应返回限流错误。
- 落到你的栈：**零新依赖**（限流件+ioredis 都已在用），阈值进 `.env`（你项目"单一真相源 .env"的既有约定），运营可调、不改代码。这比"先不加、出事再补"的长期维护成本更低。

**具体参数建议（可直接抄进配置）**

| 维度 | 阈值建议 | 说明 |
|------|---------|------|
| 限流维度 | 按 IP（匿名无 user_id，IP 是唯一可用键） | 登录用户可叠加 user_id 维度，但行情无需 |
| 阈值 | **行情 4 接口合计 100 次/分钟/IP**（区间 60–120） | 一次展开面板并行 4 请求、切周期再请求，正常用户每分钟峰值约 20–40，100 留足余量 |
| 窗口算法 | 滑动窗口 / 令牌桶（你限流件已实现） | 避免固定窗口边界突刺 |
| 超限响应 | HTTP 429 或你统一包络 `success:false, code:'RATE_LIMITED'` + `Retry-After` | 前端 catch 后展示"稍后再试"，不影响已渲染数据 |
| 配置项 | `.env` 新增 `MARKET_PUBLIC_RATE_LIMIT_PER_MIN=100` | 运营可热调，符合单一真相源 |

> 落地形态：在 `price.js`/`analytics.js` 这 5 个路由前再挂一个 `RateLimiterMiddleware` 实例（IP 维度、读上面 env 阈值），放在 `optionalAuth` 之后。**不影响写接口的既有限流策略。**
> 升级路径（暂不做，留接口）：若上线后真遇到恶意爬库，再加"设备指纹 + 行为风控"分级拦截（低风险验证码、高风险封禁+申诉）——这是大厂/高价值数据平台的做法，你现在不需要，避免过度设计。

#### (④) 版本闸门配置入口：先脚本写库 还是 直接做管理后台可视化

**各类公司怎么设计**
- **大厂（腾讯小程序最佳实践原文）**：版本闸门是**后台可配 + 接口下发 min_version**，甚至用 CI 自动把"前第 N 个 tag 版本号"写进配置，运营/研发后台可改、热生效。
- **中小/活动公司**：常见"先在配置表里手填一条，后续再补管理界面"，避免首版就为低频操作做完整 UI。

**区别本质**：可视化入口是**运营便利性 vs 前端工时**的权衡。闸门是**低频操作**（发版才改），不是高频运营动作。

**最适合你的选择 → 先脚本/接口写库上线，管理后台卡片作为"可后置增强"**
- 你后端已有 `upsertConfig('app_version_gate', {...})` 通道，写一条配置即可上线闸门，**当天可用**。
- 管理后台用的是 Vite+Alpine，加配置页**可复用**已有的"兑换页面配置"范式（后端 `exchange-page-config.js` 的 `upsertConfig` + 前端 `exchange-page-config.js` composable），技术上无障碍；但它属于"锦上添花"，不阻断小程序闸门生效。**注意：不可走通用系统设置页 `config.js`，那条 `updateSettings` 通道有白名单校验会拒绝自由 key。**
- 长期看：闸门 1 年改不了几次，先脚本不构成技术债；等有空再加可视化卡片，是渐进演进而非返工。

**初始化脚本范例（一次性写库，放 `scripts/` 下用 `node` 跑）**

```js
// scripts/seed_app_version_gate.js
require('dotenv').config()
const AdminSystemService = require('../services/AdminSystemService')

;(async () => {
  await AdminSystemService.upsertConfig('app_version_gate', {
    latest_version: '5.2.0',
    min_version: '0.0.0',     // 初始不拦任何人
    force_update: false,       // 默认关闭闸门，绝不误伤
    update_message: '检测到新版本，请更新后继续使用',
    platform: 'miniprogram'
  }, { category: 'feature', description: '小程序版本闸门配置' })
  console.log('app_version_gate 初始化完成')
  process.exit(0)
})()
```

发新版要拦旧版时，只需把 `min_version` 抬到新版本号、`force_update` 置 `true`（运营改库或重跑脚本即可，无需后端发版）。

**大厂的进阶做法（你可借鉴的思路，非必做）**
- **腾讯小程序最佳实践**：用 CI 在每次发版时，自动把"前第 N 个 tag 的版本号"写进闸门配置，实现 min_version 自动滚动，避免人工漏配。你若接了 CI，可在发布流水线末尾调一次上面的 `upsertConfig` 即可复刻。
- **细粒度版本控制**（应用级/页面级/接口级）：腾讯对不同页面/接口设不同 min_version，未命中页面不弹强更，体验更平滑。**你当前只需应用级**（onLaunch 全局闸门）即可，页面级是后续可选演进，现在做属于过度设计。

#### (⑤) 维护模式期间，版本闸门接口要不要可读

**各类公司怎么设计**：维护页通常仍允许"版本检查/公告/客服"等极少数轻量公开接口，以便维护期引导用户更新或安抚；其余业务接口熔断。

**最适合你的选择 → 加入维护白名单（建议是）**
- 场景：你因"旧版有严重 Bug"开维护模式时，恰恰最需要把旧版用户挡到更新页。若闸门接口本身被 503，小程序拿不到 min_version 就只能放行，闸门失效。
- 落到你的栈：`middleware/maintenanceMode.js` 的 `BYPASS_PREFIXES` 加一行 `/api/v4/system/app-version` 即可。极低成本、强一致性收益。

### 10.3 拍板项⑥：字段契约——"前端适配后端、不做映射层"是否正确

**各类公司怎么设计**
- **大厂**：后端有统一 API 规范（字段命名、响应包络、错误码），**前端无条件适配后端契约**；绝不在后端为单个前端做特例字段。映射层只在"对接第三方/历史遗留系统且无法改源"时才用，是**不得已的债务**，不是常规设计。
- **中小项目**：常因前后端各写各的，事后加一堆 `a||b||c` 兼容字段映射 → 这是典型技术债来源。

**区别本质**：映射层=用永久的运行期复杂度，换一次性的改动成本。**项目未上线时做映射层是纯亏**——没有历史包袱却主动背债。

**最适合你的选择 → 维持"后端字段权威、前端改字段、零映射层"（⑥确认）**
- 你未上线、可一次性投入、不兼容旧接口——这正是**消灭映射层的最佳时机**，行业最佳实践完全支持。
- 本文档第二、四节已据此把 `time→time_bucket`、`limit→page_size`、`records→orders/data数组` 等全部改成"前端对齐后端"，不在后端加兼容字段。
- 长期：前端直接用后端原生字段，后续后端字段演进时只需版本化（/v4→/v5），不必维护映射表，债务最低。

### 10.4 拍板项一页速查（你只看这张表也能拍）

| # | 拍板项 | 行业主流做法 | 给你的建议 | 落到你后端的动作 | 工作量 |
|---|--------|------------|-----------|----------------|--------|
| ① | 行情 4 接口对游客开放 | 交易/电商/二手平台一致开放只读行情 | **开** | 4 路由 `authenticateToken→optionalAuth` | 极小（改 4 行） |
| ② | 90 天 `analytics/history` 一并开放 | 同上，历史价同属公开行情 | **开** | 1 路由换 `optionalAuth` | 极小（改 1 行） |
| ③ | 匿名行情是否加限流 | 大厂网关 IP 限流；中小项目鉴权+频控即可 | **加宽松 IP 限流**（100/分/IP） | 5 路由挂 `RateLimiterMiddleware`，阈值进 `.env` | 小 |
| ④ | 版本闸门配置入口 | 大厂后台可配+CI 自动滚动；中小先填表后补 UI | **先脚本写库**，管理后台页后置（仿 exchange-page-config 范式） | 跑一次 `upsertConfig` 脚本 | 小（脚本）/ 中（独立配置页） |
| ⑤ | 维护期闸门可读 | 维护页保留版本检查/公告等轻量公开接口 | **加入维护白名单** | `maintenanceMode.js` 加 1 行前缀 | 极小（改 1 行） |

## 十一、整体方案定性：你的项目最适合"哪一派"

把上面综合起来，你的项目最适合的整体设计取向是 **"交易类平台的读写分离 + 中小项目的克制"** 这一派：

- **学大厂/游戏交易平台的"读写分离"**：只读（行情/商品/版本/公告）走公开 `optionalAuth` + 网关式限流；写/交易动作走强鉴权 `authenticateToken` + RBAC。你后端 `auth.js` 已天然具备这套分层，**不需要新架构**。
- **学中小项目的"克制"**：不上签名风控、不做映射层、不为低频操作先做重 UI。够用即止，把投入花在刀刃（鉴权分层 + 字段对齐 + 配置热改）。
- **不学的**：不学"全站登录墙"（伤转化）、不学"前端兼容映射"（背债）、不学"为单前端定制后端字段"（破坏契约权威）。

**为什么这一派最适合你**：你是积分商城+C2C交易+小程序的组合，业务形态最接近游戏饰品/二手交易平台；同时团队体量与数据价值决定了不必上大厂的重型风控。三个需求落地后，后端**零新表、零新依赖、不改架构**，全部复用现有中间件与配置通道——这正是"长期维护成本低、技术债务低"的最优解。

### 11.1 五种"设计流派"全景对比（你问的各类公司差异）

| 流派 | 代表 | 鉴权取向 | 配置/版本管控 | 限流/风控 | 适合谁 | 你该取/舍 |
|------|------|---------|--------------|----------|--------|----------|
| 大厂电商 | 美团/淘宝/京东 | 读写彻底分离，读默认匿名 | 配置中心+灰度+CI 自动化 | 网关多维限流+完整风控 | 海量流量、强运营 | **取**读写分离思路；**舍**重型风控/配置中心（杀鸡用牛刀） |
| 游戏饰品交易 | Steam/BUFF/悠悠 | 看全开、交易动作强鉴权+绑定+实名 | 后台可配+接口下发版本 | 行情接口限流偏严（防爬价） | C2C 虚拟物品交易 | **最贴合你**：行情全开、交易锁权限 |
| 二手/小众平台 | 闲鱼/得物 | 浏览全开、交易登录+实名 | 后台配置 | 中等限流+人工审核 | C2C 二手 | **取**浏览全开 |
| 活动策划/积分 SaaS（规范派） | 规范的积分商城 | 展示公开、操作登录 | system_settings 类表+管理后台 | 基础频控 | 中小积分系统 | **你现状即此派**，继续保持 |
| 活动策划/积分 SaaS（偷懒派） | 部分外包项目 | 全站登录墙 | 散落硬编码/无热改 | 几乎无 | 交付了事 | **舍**：伤转化、背债 |

**三条主轴上的取舍**（这是各派真正的区别所在）：
1. **登录墙位置**：偷懒派设在"进门"，主流派设在"交易动作"。→ 你选**交易动作**。
2. **配置可变性**：大厂用配置中心+CI，偷懒派硬编码。→ 你用**已有的 `system_settings` 热改**，居中且零新基建。
3. **防刷强度**：大厂全套风控，中小项目只做鉴权+频控。→ 你选**鉴权+宽松频控**，需要时再分级升级。

### 11.2 长期维护成本视角：为什么这套是"债最低"

- **不新增数据表**：需求1/3 复用现有表，需求2 复用 `system_settings` 的 JSON 配置通道 → 没有新迁移、新模型、新索引要维护。
- **不新增依赖/框架**：限流件、Redis、optionalAuth、统一响应、ServiceManager 全部已存在 → 没有新的安全更新面、没有学习成本。
- **不引入映射层**：前端直连后端字段 → 没有"改一处要同步两处映射"的持续成本。
- **配置热改**：版本闸门、限流阈值都在 `.env`/`system_settings`，运营改值即可 → 没有"调参就发版"的运维成本。
- **演进留口子但不预先做**：限流可升级到设备指纹/行为风控、版本闸门可升级到页面级/ CI 自动滚动、管理后台可加可视化卡片——**全部是非破坏性增量**，需要时再加，现在做都是过度设计。

> 一句话：这套方案把"现在必须做的"压到最小（改几行中间件 + 新建 1 个公开路由 + 1 条配置），把"可能要做的"都留成非破坏性的后续增量，符合你"未上线、可投入、但要长期低债"的目标。
