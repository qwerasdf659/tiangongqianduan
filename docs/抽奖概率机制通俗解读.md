# 抽奖概率机制通俗解读

> 配套文档：`奖品概率管理-现状分析与设计方案.md`
>
> 用日常比喻解释本项目抽奖系统的概率原理，不涉及代码。

---

## 一、系统用的不是"摸球"，是"转盘"

文档里说"袋子里 100 个球"只是帮助理解百分比的比喻。真实的抽奖系统更像一个**转盘**：

```
        ┌─────────────┐
       /    红 30%      \
      /                  \
     │                    │
     │   绿              │
     │   50%    蓝 20%   │
      \                  /
       \                /
        └─────────────┘
```

每个顾客来转一次，指针停在哪就中哪个。转完之后**转盘不会变**——不会因为上一个人转到红色，红色区域就缩小。

系统的实际逻辑：每次抽奖生成一个 0~100 的随机数，看它落在哪个区间。

```
0 ───── 30 ───── 50 ──────────── 100
  红球30%   蓝球20%     绿球50%
```

- 随机数落在 0~30 → 红球
- 随机数落在 30~50 → 蓝球
- 随机数落在 50~100 → 绿球

每次抽奖都重新生成随机数，上一次抽到什么完全不影响下一次。

---

## 二、两种"摸球"模型对比

| | 不放回摸球（物理袋子） | 放回摸球 / 独立随机（本项目） |
|--|--|--|
| 抽走一个红球后 | 袋子变 99 球（29红/20蓝/50绿），红球概率变 29.3% | 概率不变，仍是 30% |
| 连续抽 10 个红球后 | 红球概率越来越低（20/90 = 22.2%） | 概率始终 30% |
| 本质 | 有限资源池，越抽越少 | 每次独立随机，互不影响 |
| 适用场景 | 福袋、盲盒（固定数量实物） | 线上抽奖、转盘、游戏掉落 |

**本项目属于右列**——每次抽奖独立计算概率，不存在"抽走一个就少一个"的情况。

---

## 三、什么时候概率会真的变

只有两种情况：

1. **运营手动修改**了权重 / 百分比配置
2. **库存耗尽**——奖品库存为 0 时，该奖品被踢出转盘，剩余奖品的区域自动变大

---

## 四、库存耗尽时发生了什么

### 4.1 处理方式

不管是权重模式还是百分比模式，库存耗尽的处理都一样：**先踢出去，剩下的按原有比例自动等比放大。**

假设红球 30%、蓝球 20%、绿球 50%，红球库存耗尽：

| | 权重模式（当前实现） | 百分比模式（未实现） |
|--|--|--|
| 存储值 | 红=3, 蓝=2, 绿=5 | 红=30%, 蓝=20%, 绿=50% |
| 红球踢出后参与计算的 | 蓝=2, 绿=5, 总权重=7 | 蓝=20%, 绿=50%, 总和=70% |
| 蓝球实际概率 | 2/7 = **28.6%** | 20/70 = **28.6%** |
| 绿球实际概率 | 5/7 = **71.4%** | 50/70 = **71.4%** |
| 概率怎么变大的 | 总权重变小了，每个的占比自然变大 | 总和不到100%，按比例归一化到100% |

结果完全一样。蓝球和绿球原本是 2:5 的关系，红球没了之后还是 2:5，只是各自的绝对概率都变大了。

### 4.2 转盘的变化

```
之前的转盘：                撕掉红色后的转盘：

     红 30%
   ┌────────┐              ┌────────────┐
   │        │              │  蓝 28.6%  │
   │ 蓝 20% │              │            │
   │        │              │            │
   │ 绿 50% │              │  绿 71.4%  │
   │        │              │            │
   └────────┘              └────────────┘
```

## 五、两张表的联系——转盘规则 vs 转盘改造

上面讲了两件事：

- **独立随机**（每次抽奖互不影响）
- **库存耗尽重算**（踢出奖品后概率变化）

它们不是两个互斥的方案，而是**同一个系统里的两个齿轮**：

| | 独立随机（日常规则） | 库存耗尽重算（特殊事件） |
|--|--|--|
| 触发条件 | 每次抽奖 | 某个奖品库存变成 0 |
| 发生频率 | 每抽一次都在用 | 可能整个活动只发生几次 |
| 影响范围 | 只影响"这一次和下一次之间" | 影响"之后所有人的所有次" |
| 概率变不变 | 不变 | 变（踢出后重算） |
| 比喻 | 转盘转起来的规则 | 转盘本身被改了 |

### 完整时间线

```
第 1 次抽奖             第 2 次抽奖           第 N 次（红球库存归零）    第 N+1 次抽奖
    │                       │                        │                      │
    ▼                       ▼                        ▼                      ▼
红30% 蓝20% 绿50%     红30% 蓝20% 绿50%      红30% 蓝20% 绿50%      蓝28.6% 绿71.4%
  （抽到红球）           （抽到绿球）           （抽到红球，最后一个）    （红球已被踢出）
    │                       │                        │                      │
    ▼                       ▼                        ▼                      ▼
  概率不变                概率不变               触发转盘改造              新转盘生效
  （日常规则）            （日常规则）           （特殊事件）              （日常规则继续）
```

假设活动有 100 个红球库存、无限蓝球、无限绿球：

| 阶段 | 发生了什么 | 用的哪个规则 | 概率 |
|------|-----------|-------------|------|
| 第 1~99 次抽到红球 | 每次抽完，转盘不变 | 日常规则（独立随机） | 红30% 蓝20% 绿50% |
| 第 100 次抽到红球 | 红球库存归零，从转盘上撕掉 | 特殊事件（重算） | 蓝28.6% 绿71.4% |
| 第 101 次以后 | 继续独立随机，只是转盘没红色了 | 日常规则（独立随机） | 蓝28.6% 绿71.4% |

**系统 99.9% 的时间都在用日常规则。只有在极少数"库存耗尽"的瞬间，才触发一次转盘改造，然后又回到日常规则继续跑。**

### 生活化比喻

- **日常规则**就像菜单上的菜——你点了鱼香肉丝，不影响下一桌也点鱼香肉丝，菜单不会因为你点了就变。
- **特殊事件**就像某道菜的食材用完了——厨房说"鱼香肉丝今天卖完了"，服务员把菜单上这道划掉，但其他菜不受影响。

## 六、四个概念之间的关系

### 6.1 两个独立的问题

权重/百分比 和 放回/不放回 是两个完全不同的问题，互相独立，可以自由组合：

| | 问的是什么 | 选项 |
|--|--|--|
| **第一个问题** | 概率数字**怎么存、怎么改**？ | 权重模式 / 百分比模式 |
| **第二个问题** | 每次抽奖**互相影响不影响**？ | 放回（独立随机） / 不放回（物理袋子） |

第一个问题管的是**后台运营怎么配置**，第二个问题管的是**抽奖那一瞬间怎么算**。

打个比方：

- 第一个问题 = 菜单上的价格写**人民币**还是写**美元**（标价方式）
- 第二个问题 = 这家餐厅是**自助餐**还是**限量供应**（供餐模式）

价格写人民币还是美元，和餐厅是不是自助餐，是两件完全无关的事。

### 6.2 四种组合都能存在

| | 放回（独立随机） | 不放回（物理袋子） |
|--|--|--|
| **权重模式** | 本项目、原神、王者荣耀 | 理论上可以，但很少见 |
| **百分比模式** | 小公司简单抽奖、商场转盘 | 泡泡玛特盲盒、福袋 |

- **权重 + 放回**（本项目）：运营填权重数字（如 3、2、5），系统自动算百分比（30%/20%/50%），每个顾客转同一个转盘，互不影响。
- **百分比 + 放回**（小公司抽奖）：运营直接填百分比（30%、20%、50%，必须自己凑到 100%），每个顾客转同一个转盘，互不影响。
- **百分比 + 不放回**（盲盒）：一箱 12 个，普通款 10 个（83.3%）、隐藏款 2 个（16.7%）。买走一个普通款后变成 9/11（81.8%）和 2/11（18.2%）——概率随着购买实时变化。
- **权重 + 不放回**（极少见）：理论上可以用权重来表示盲盒里每个款式的数量，但没人这么做，因为盲盒就那么几个，直接写数量比写权重更直观。

### 6.3 哪里一样，哪里不一样

| 比较对象 | 结果一样吗 | |
|--|--|--|
| 权重 vs 百分比 | **一样**——只是存储方式不同，算出来的概率完全相同 | 就像人民币和美元标价，换算完价格一样 |
| 放回 vs 不放回 | **不一样**——用户体验、成本控制、公平性全都不同 | 自助餐和限量供应，体验完全不同 |

### 6.4 放回 vs 不放回的概率变化差异

| | 放回 | 不放回 |
|--|--|--|
| 概率什么时候变 | **只在库存耗尽那一刻**变一次 | **每抽一次**都在变 |
| 变的频率 | 整个活动可能就变几次 | 抽多少次就变多少次 |
| 变的幅度 | 一次性跳变（30% → 0%） | 每次微调（30.0% → 29.9% → 29.8%...） |

同一个活动，两种模式跑出来的结果完全不同：

**放回模式**（红球库存 100 张，蓝球无限，绿球无限）：

| 第几次抽奖 | 红球概率 | 蓝球概率 | 绿球概率 | 发生了什么 |
|-----------|---------|---------|---------|-----------|
| 第 1 次 | 30% | 20% | 50% | 抽到红球，库存 99 |
| 第 2 次 | 30% | 20% | 50% | 抽到绿球 |
| 第 50 次 | 30% | 20% | 50% | 不管前面抽了啥，概率不变 |
| 第 200 次 | 30% | 20% | 50% | 概率不变 |
| 第 N 次（红球发完） | **0%** | **28.6%** | **71.4%** | 红球踢出，一次性跳变 |
| 第 N+1 次以后 | 0% | 28.6% | 71.4% | 新概率稳定不变 |

**不放回模式**（袋子里 1000 个球：红 300、蓝 200、绿 500）：

| 第几次抽奖 | 红球概率 | 蓝球概率 | 绿球概率 | 发生了什么 |
|-----------|---------|---------|---------|-----------|
| 第 1 次 | 300/1000 = 30.0% | 20.0% | 50.0% | 抽到红球，袋子剩 999 |
| 第 2 次 | 299/999 = 29.9% | 20.0% | 50.1% | 概率已经变了 |
| 第 50 次 | ~29.5% | ~20.0% | ~50.5% | 每次都在变 |
| 第 200 次 | 看前面抽了啥而定 | 每次都不一样 | 每次都不一样 | 完全不可预测 |
| 第 700 次 | 可能 15% | 可能 25% | 可能 60% | 概率剧烈波动 |
| 第 1000 次 | 必中最后一个球 | — | — | 100% 确定 |

### 6.5 怎么选：五个问题

| 问自己这个问题 | 选放回 | 选不放回 |
|--|--|--|
| 奖品有限还是无限？ | 虚拟奖品（积分、券）可以无限发 | 实物（一箱盲盒就 12 个） |
| 要不要对所有用户公平？ | 要——每个人概率一样 | 不需要——先买的和后买的概率本来就不同 |
| 运营需要精确控制成本吗？ | 需要——概率固定，大数定律保证发放量可预测 | 不需要——反正一箱就那么多，卖完拉倒 |
| 活动持续多久？ | 长期（几周到几个月） | 短期（一箱卖完就结束） |
| 有多少用户参与？ | 很多（成百上千次抽奖） | 很少（一箱 12 个，最多 12 个人） |

本项目五个问题全部指向**放回**。

一句话总结：**放回和不放回的选择标准就一条——你的奖品是不是物理上有限的实物，摸走一个就真的少一个？** 是的话用不放回，不是的话用放回。

## 七、百分比 vs 权重的存储区别

两者最终概率总和都是 100%，区别在于**谁来算**：

**百分比**：袋子里固定 100 个球，红球 30 个、蓝球 20 个、绿球 50 个。想让红球从 30% 变成 40%，必须同时减少蓝球或绿球，总数只能是 100。

**权重**：袋子里想放多少放多少，红球 3 个、蓝球 2 个、绿球 5 个，系统自动算出 30%/20%/50%。想让红球概率变高，直接加红球到 6 个，系统自动算出 46%/15%/39%，不用动蓝球和绿球。

| | 百分比 | 权重 |
|--|--------|------|
| 谁保证总和 = 100% | 你自己凑 | 系统自动算 |
| 加一个新奖品 | 要改所有奖品概率凑回 100% | 填一个权重数字就行 |
| 删一个奖品 | 要把剩余的改回 100% | 直接删，其他自动调整 |
| 改一个奖品概率 | 要同时改其他的让出空间 | 只改它一个，其他自动变 |
| 精确控制"必须 30%" | 直接填 30% | 要反推填多少权重 |

行业最佳实践：**后端存权重，管理后台显示百分比**。本项目数据库已经存的是权重（`win_weight`），只差管理后台把它换算成百分比展示。

## 八、行业方案全景对比

### 8.1 各类公司怎么设计

#### 互联网大厂（美团 / 腾讯 / 阿里）

模型：独立随机 + 放回。

| 维度 | 做法 |
|------|------|
| 概率存储 | 后端存权重，管理后台显示百分比 |
| 概率控制 | 多层：基础概率 → A/B 实验分流 → 实时风控调整 |
| 用户差异 | 通过实验平台做千人千面，不同用户看到不同概率 |
| 库存耗尽 | 踢出奖池，剩余奖品按比例自动放大 |
| 成本控制 | 活动总预算 + 每日预算 + 单品库存 + 实时 ROI 监控 |
| 保底 | 有，但很克制（优惠券类活动很少做保底） |
| 审批 | 概率变更需审批流 |

复杂度极高，有专门的实验平台、风控系统、审批链路、实时监控大盘。

#### 游戏公司（米哈游 / 网易 / 腾讯游戏）

模型：独立随机 + 放回 + 保底（pity）。

| 维度 | 做法 |
|------|------|
| 概率存储 | 权重（内部），百分比（公示给玩家） |
| 概率结构 | **分档位 + 档内权重**，和本项目一模一样 |
| 保底 | 硬保底：连续 N 次不出金，第 N+1 次必出。软保底：随失败次数递增概率 |
| 用户差异 | 分群权重矩阵（新用户 / 回归玩家 / 付费玩家） |
| 公示 | 只公示档位概率（如：5 星概率 0.6%），不公示单品概率 |
| 库存 | 虚拟物品通常无限库存，限定物品有库存上限 |

以原神为例：

```
档位设计：
  5星（0.6%）→ 角色池 50% / 武器池 50%（档内等权重）
  4星（5.1%）→ 角色 / 武器 / 其他
  3星（94.3%）→ 通用素材

保底机制：
  硬保底：连续 89 抽不出 5 星，第 90 抽必出
  软保底：第 74 抽开始概率逐渐提升（从 0.6% 到 ~33%）
  大保底：连续 2 个 5 星不是 UP 角色，第 3 个必是
```

#### 小公司 / 创业团队

模型：独立随机 + 放回（最简版）。

| 维度 | 做法 |
|------|------|
| 概率存储 | 直接存百分比 |
| 概率结构 | 扁平的，不分档位，所有奖品在一个池子里 |
| 用户差异 | 没有，所有人概率一样 |
| 保底 | 没有 |
| 编辑方式 | 管理员直接填百分比，前端校验总和 = 100% |
| 库存耗尽 | 简单踢出，很多不会自动重新归一化 |

优点：极简。缺点：改一个要动其他的，不支持分群和保底。

#### 活动策划公司（线下活动 / 商场促销）

模型：独立随机 + 时段控制。

| 维度 | 做法 |
|------|------|
| 概率存储 | 百分比 |
| 特色机制 | **时段概率**：上午高概率引流、下午正常、晚上低概率控成本 |
| 用户差异 | 不按用户分，按时间段分 |
| 编辑方式 | 可视化拖拽 / 时间轴调节 |
| 库存 | 严格按实物库存，每天限量 |

概率跟着时间变，不是跟着用户变。

#### 盲盒 / 福袋平台（泡泡玛特 / 名创优品）

模型：**不放回摸球**（有限资源池）——唯一用"不放回"的场景。

| 维度 | 做法 |
|------|------|
| 概率存储 | 不存概率，存每个款式的实际数量 |
| 概率计算 | 概率 = 剩余数量 / 剩余总数，实时变化 |
| 库存 | 是核心，一箱固定 12 个，卖完就没了 |
| 公示 | 公示每个款式的配比 |
| 特色 | 越买越能推算剩下的是什么 |

```
一箱 12 个盲盒（买走一个普通款 A 后）：
  普通款 A：2/12 = 16.7% → 1/11 = 9.1%（降了）
  隐藏款 F：1/12 = 8.3%  → 1/11 = 9.1%（升了）
```

用不放回是因为卖的是实物，一箱就那么多。

#### 虚拟物品交易 / 二手平台（Steam / BUFF）

模型：独立随机 + 放回（本质和游戏开箱一样）。

| 维度 | 做法 |
|------|------|
| 概率模型 | 独立随机，每次开箱独立 |
| 特色 | 开出的物品可交易，所以必须**强制公示概率** |
| 品质 | 概率嵌套——先选物品，再选品质 |
| 库存 | 无限（虚拟物品无限生成） |

本质就是分档位 + 档内权重，和本项目的 `tier_first` 一样。

### 8.2 六种方案一张表

| | 大厂 | 游戏公司 | 小公司 | 活动策划 | 盲盒/福袋 | 虚拟交易 |
|--|------|---------|--------|---------|----------|---------|
| 放回/不放回 | 放回 | 放回 | 放回 | 放回 | **不放回** | 放回 |
| 概率存储 | 权重 | 权重 | 百分比 | 百分比 | 数量 | 权重 |

## 九、本项目最匹配哪个

**结论：游戏公司方案（米哈游/网易模式），而且已经在用了。**

| 本项目已有 | 对应游戏公司方案 |
|------------|----------------|
| `tier_first`（分档位） | 原神的 5 星 / 4 星 / 3 星分层 |
| `win_weight`（档内权重） | 档内物品的出率权重 |
| `segment_key`（用户分群） | 新用户 / 回归用户差异化概率 |
| `pity`（保底系统） | 硬保底 / 软保底机制 |
| 三层库存保险 | 游戏的兜底保护机制 |
| 独立随机 + 放回 | 所有线上抽奖的标准做法 |

**不适合用盲盒模型**：本项目是餐厅抽奖活动，奖品以积分券、折扣券为主，没有"一箱固定 12 个"的物理限制。用不放回模型会导致先抽的用户和后抽的用户概率不同（不公平）、运营无法精确控制长期成本（概率一直在变）。

**不需要升级到大厂方案**：大厂的 A/B 实验、审批链路、实时风控是为千万级 DAU 设计的，本项目 3,030 次抽奖的规模用不上，过度设计反而增加维护成本。

**当前唯一需要做的**：把管理后台的 UI 补上，让运营能编辑 `win_weight` 和 `reward_tier`，并且显示换算后的百分比。底层引擎已经是行业最佳实践。
