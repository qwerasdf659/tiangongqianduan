> 文档创建时间：2026年02月24日  
> 最后更新时间：2026年02月25日  
> 适用项目：天工小程序 - 抽奖系统  
> 决策状态：已确认 - 完全学游戏公司模式（含前端代码审计 + 4项架构决策 + P2节奏决策）

## 一、心理学底层机制

所有抽奖/奖励系统的设计，无论哪个行业，都基于以下5个核心心理学原理。

### 1.1 斯金纳箱（变比率强化）

不确定什么时候给奖励，比固定给奖励更让人上瘾。老虎机、抽卡、翻牌全是这个原理。

- 用户不知道下一次能不能中，但知道"有可能中"
- 这种不确定性带来的多巴胺分泌，甚至超过真正获得奖励时的快感
- 应用：所有14种抽奖玩法的核心驱动力

### 1.2 锚定效应

先展示高价值奖品，用户心里就会以这个为参考。

- 奖品池预览中"神秘彩宝"、"精品首饰"放在最前面，用户心理锚点就是"我可能抽到这些好东西"
- 实际大概率抽到的是500积分券，但用户不会以此为心理预期
- 应用：奖品池预览排序策略 —— 大奖必须放最前面

### 1.3 损失厌恶

人对"失去"的痛苦是"得到"快乐的2倍。

- "抢光"标签就是利用这个 —— 你不抽，好东西就没了
- 限时活动、"已售罄"标签，都在制造"再不行动就来不及了"
- 应用：is_sold_out 遮罩

### 1.4 沉没成本

花了积分就不想走，"再来一次说不定就中了"。

- 连抽折扣（三连翻285积分 < 300积分原价）降低追加投入的决策门槛
- 保底计数器让用户觉得"已经抽了60次，放弃太亏，再抽28次就保底了"
- 应用：draw-buttons 连抽折扣体系 + 保底计数器

### 1.5 差一点效应（Near Miss Effect）

让用户感觉"差一点就中了"，比直接告诉他"没中"更让人想继续。

- 老虎机两个符号一样第三个差一格
- 翻牌时旁边的牌翻开是大奖（暗示"选另一张就中了"）
- 应用：抽奖动画设计、结果展示策略

## 二、各行业设计方案详细分析

### 2.1 腾讯（微信/QQ/王者荣耀/英雄联盟）

**策略定位：满屏展示 + 稀有度分级 + 概率公示**

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品展示 | 奖品池完整展示，传说皮肤放最大最亮，普通奖品缩小 |
| 稀有度 | 颜色分级：白 < 蓝 < 紫 < 金 < 红 |
| 概率公示 | 强制公示（国家法律要求），反而增加了信任感 |
| 保底机制 | 公开展示（"88次必得传说"），既合规又制造确定感 |
| 连抽 | 十连抽有折扣，且必出至少一个4星 |

**核心手法：让用户算账。**

用户会算："88次 x 10元 = 880元能拿到传说皮肤"，觉得"值"，然后开始抽。保底机制把"赌博"变成了"延时购买"，心理负担大幅降低。

**对天工项目的启示：**
- 保底计数器必须可见
- 概率展示增加信任感（即使概率低，公开比隐藏更好）
- 连抽必出机制减少挫败感

---

### 2.2 阿里巴巴（淘宝/支付宝/天猫）

**策略定位：实物图片 + 社会证明 + 限时紧迫**

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品展示 | 实物高清图片，不用图标/emoji |
| 社会证明 | 底部滚动"xxx刚刚抽中iPhone 15"，制造从众心理 |
| 紧迫感 | 限时活动 + 倒计时，"还剩2小时" |
| 成本感知 | 淘金币/积分是"免费"攒的，用户觉得"不用白不用" |

**核心手法：降低感知成本。**

积分是"日常签到攒的"，抽奖是"免费的"，所以"不抽白不抽"。把"花积分"重新定义为"用掉免费额度"。

**对天工项目的启示：**
- 积分来源要让用户觉得"免费得来的"
- 滚动中奖播报可以考虑加入（增加社会证明）

---

### 2.3 美团

**策略定位：实用优惠券 + 即时可用 + 嵌入消费流程**

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品类型 | "5元外卖红包"、"满30减3"等即时可用优惠 |
| 触发时机 | 下单后弹出抽奖，嵌入消费流程 |
| 奖品数量 | 少而精，不搞大奖池 |
| 用户感知 | "薅羊毛感" —— 觉得自己赚到了 |

**核心手法：即时满足。**

抽到就能用，奖励反馈链路极短。不需要攒、不需要等、不需要兑换。

**对天工项目的启示：**
- 如果有实用性奖品（优惠券、折扣），突出"立即可用"
- 但天工以虚拟奖品为主，这个模式参考价值有限

---

### 2.4 抖音/字节跳动

**策略定位：进度可视化 + 确定性奖励 + 社交裂变**

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品展示 | 极简 —— 就是"现金"二字 |
| 进度可视化 | "再看3个视频就能提现"，进度条 |
| 确定性 | 不是"可能得到"，而是"一定得到，只是多少的问题" |
| 裂变 | 邀请好友得更多，社交传播 |

**核心手法：确定性 + 社交裂变。**

用户行为路径：做任务 → 看进度 → 快到了 → 再做一点 → 提现 → 分享给朋友。

**对天工项目的启示：**
- 进度条/保底计数器的设计可以参考这个思路
- 连抽保底本质上也是"确定性奖励"

---

### 2.5 百度

**策略定位：事件驱动 + 简单粗暴 + 短期爆发**

| 设计要素 | 具体实现 |
|---------|---------|
| 活动形式 | 春节集卡 + 红包雨 |
| 奖品 | 现金，展示方式简单 |
| 时间窗口 | 极短，春节期间集中爆发 |
| 留存 | 活动结束后留存差 |

**核心手法：事件驱动，短期获客。**

靠春节等大事件集中爆发，不做长期留存。适合有大量预算做短期推广的场景。

**对天工项目的启示：**
- 参考价值较低 —— 天工需要长期留存，不是短期爆发
- 但节日限定活动可以借鉴这个思路

---

### 2.6 游戏公司（米哈游/网易/暴雪）

**策略定位：这是最成熟的抽奖设计体系，天工项目最应该学的标杆**

#### 米哈游（原神/崩坏：星穹铁道）

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品池展示 | UP池大图展示限定角色，常驻池小图，视觉层级分明 |
| 稀有度分级 | 5星(金) > 4星(紫) > 3星(蓝)，颜色、光效、动画全套差异 |
| 保底计数器 | 可见 —— "已抽62次，距离保底还有28次" |
| 抽卡动画 | 蓝色流星=3星，金色流星=5星，动画本身就是奖励体验 |
| 连抽保底 | 十连必出4星，降低"全是垃圾"的挫败感 |
| 详情查看 | 点击任意奖品可查看详细信息（立绘、技能、故事） |

#### 网易（阴阳师）

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品详情 | 每个式神有详细介绍页，让用户先爱上奖品再抽 |
| 碎片系统 | 没抽到也能攒碎片合成，减少"完全白抽"的感觉 |
| 概率UP | 限时UP池提高特定角色概率，制造紧迫感 |

#### 暴雪/CSGO开箱

| 设计要素 | 具体实现 |
|---------|---------|
| 开箱动画 | 滚动展示所有可能物品，最终停在你获得的那个 |
| 市场价格 | 每个物品有实时市场价，"这把刀值3000元" |
| 磨损度 | 同一物品不同品质，增加收藏深度 |

**游戏公司设计共性：**

1. 奖品池100%展示，且可以点击查看详情
2. 稀有度视觉差异极大（颜色、光效、大小、动画）
3. 保底机制透明公开，计数器可见
4. 连抽有折扣或额外保底
5. 抽奖动画本身就是内容和体验
6. 大奖视觉权重远高于普通奖品

**对天工项目的启示：**
- **这是天工项目的核心参考标杆**
- 奖品池预览必须加入稀有度视觉分级
- 保底计数器必须外显
- 奖品可点击查看详情
- 大奖放最前面，视觉权重最高

---

### 2.7 银行/投行/股票平台（招行/中信/东方财富）

**策略定位：信任感优先 + 实物展示 + 正规感**

| 设计要素 | 具体实现 |
|---------|---------|
| 奖品展示 | 像购物页面一样陈列，高清实物图 |
| 动画 | 不做花哨动画，强调正规、可信 |
| 中奖公示 | 真实用户信息（脱敏后），增强可信度 |
| 奖品类型 | 实物为主（数码产品、旅行券），体现价值感 |

**核心手法：信任背书。**

用户花的是真金白银的积分/投资收益，需要感觉"这不是骗我的"。

**对天工项目的启示：**
- 如果有实物奖品，用高清图片而不是emoji
- 中奖记录公示增加信任感
- 但整体风格太"正经"，不适合天工的娱乐属性

---

### 2.8 活动策划公司（线下活动/会展/商场）

**策略定位：实物C位 + 围观效应 + 即时体验**

| 设计要素 | 具体实现 |
|---------|---------|
| 大奖展示 | 汽车放展台中央，小奖品围一圈 |
| 感官刺激 | 中奖时放烟花/音效/主持人喊话 |
| 围观效应 | 别人中奖围观者也想抽，社会证明极强 |
| 参与门槛 | 极低 —— 消费满额就能抽 |

**核心手法：现场氛围。**

线下的核心优势是"围观"和"实物触感"，线上无法复制。

**对天工项目的启示：**
- 大奖视觉权重最大（对应线下的C位展示）
- 中奖动画要有"轰动感"（对应线下的烟花/音效）
- 中奖播报（对应围观效应）

---

### 2.9 游戏虚拟物品交易 / 小众二手平台

**策略定位：价格透明 + 可交易 + 资产属性**

| 设计要素 | 具体实现 |
|---------|---------|
| 价格标注 | 每个物品显示实时市场价格 |
| 可交易 | 抽到不想要的可以在市场卖掉 |
| 资产感 | 物品不是消耗品，是"资产" |
| 驱动力 | "这把刀值3000元" > "这把刀好看" |

**核心手法：金融化。**

奖品不只是奖品，是可交易的资产。用户抽奖动机从"想要"变成"能赚"。

**对天工项目的启示：**
- 天工已有兑换市场（exchange-market），具备交易属性
- 奖品如果可交易/可兑换，在奖品池预览中标注"可交易"会增加吸引力
- 但目前体量不足以支撑完整的交易经济

---

### 2.10 小公司常见做法

**常见问题：**

1. 预算有限，奖品少，奖品池看起来寒酸
2. 模仿大公司但做不到位，"四不像"
3. 奖品太假，用户不信
4. 没有保底机制，用户抽几次没中就流失

**正确做法：**

- 少而精 + 突出一个大奖 + 保底实用奖品
- 不需要14种玩法都做，做精2-3种核心玩法
- 保底机制是留存的关键 —— 没有保底，用户3次不中就走

## 四、天工项目最终方案

### 4.1 定位结论

天工项目当前系统特征：
- 14种抽奖玩法
- 积分经济体系
- 连抽折扣（三连/五连/十连）
- 稀有度光效系统（common/uncommon/rare/epic/legendary 五级）
- 保底机制
- 兑换商城 + 交易市场

**结论：天工项目本质是"游戏公司模式"，应全面对标米哈游/网易的设计体系。**

#### 优化项1：稀有度视觉分级

当前问题：奖品池预览中所有奖品同等大小、同等样式，没有利用已有的 rarity-effects 五级稀有度系统。

优化目标：
- legendary (传说) → 金色边框 + 光效动画 + 尺寸最大
- epic (史诗) → 紫色边框 + 微光效果
- rare (稀有) → 蓝色边框
- uncommon (优良) → 绿色边框
- common (普通) → 灰色边框 + 尺寸最小

涉及文件：所有14个抽奖子组件的奖品池预览区域

#### 优化项2：大奖前置排序

当前问题：奖品按后端返回顺序展示，高价值奖品可能排在后面用户看不到。

优化目标：
- 奖品按稀有度降序排列（legendary → epic → rare → uncommon → common）
- 同稀有度内，已抢光的保持原位置不动，用遮罩标注（已拍板）
- 用户打开页面第一眼就看到最好的奖品
- 利用锚定效应，让用户心理预期锚定在高价值奖品上

实现方式：前端排序，不依赖后端顺序

#### 优化项3：奖品可点击查看详情

当前问题：奖品池预览只展示图标和名称，无法查看详情。

优化目标：
- 点击任意奖品弹出详情弹窗
- 展示内容：奖品名称、描述、稀有度。**不展示概率数字、不展示库存数量**（已拍板：对标海底捞/瑞幸模式，惊喜感优先）。
- 学原神模式 —— 让用户先了解（爱上）奖品，再决定抽
- 增加用户在页面的停留时间

#### 优化项4：保底计数器外显

当前问题：有保底机制，但用户看不到距离保底还有多少次。

优化目标：
- 放置位置：连抽按钮旁边（已拍板：离操作最近，决策时第一时间看到）
- 展示格式："距离保底还有 X 次"
- 利用沉没成本心理 —— "已经抽了60次，再抽28次就保底了"
- 利用确定性心理 —— "最多再花X积分就一定能拿到"

## 七、四项架构决策（2026-02-25 确认）

> 决策约束条件：项目未上线、可一次性投入、无旧接口兼容需求、追求长期低维护成本和零技术债务。

### 7.1 决策1：奖品池预览稀有度样式方案

#### 各行业做法对比

| 行业 | 做法 | 是否统一系统 |
|------|------|------------|
| 游戏公司（米哈游/网易/暴雪） | 全站一套稀有度视觉体系，抽卡预览/背包/商店/交易/结果弹窗完全一致 | 统一 |
| 腾讯（王者荣耀/英雄联盟） | 白 < 蓝 < 紫 < 金 < 红，全UI表面统一 | 统一 |
| CSGO/Steam | 蓝/紫/粉/红/金，开箱预览/库存/市场完全一致 | 统一 |
| 小公司常见做法 | 为"性能优化"搞两套样式系统（完整版+精简版），维护成本翻倍 | 分裂 |

**核心发现**：所有成功的稀有度系统都使用一套统一视觉体系，从不为不同UI场景搞两套CSS。原因：玩家对稀有度颜色形成条件反射后，任何视觉不一致都会打破心理锚定。

#### 两种方案对比

| 维度 | 方案A：统一 `<rarity-effects>` 组件 | 方案B：轻量CSS类名 |
|------|------|------|
| 视觉效果 | 与九宫格游戏区完全一致（光效/旋转/星星全套） | 简化版（仅边框色+背景微光），与游戏区不一致 |
| 实现成本 | 11个子组件的预览项外套组件 | 新写一套"预览专用"SCSS |
| 维护成本 | 改颜色只改 rarity-effects.scss 一个文件 | 改颜色要同步两处（游戏区+预览区），遗漏风险高 |
| 性能 | scroll-view 里10来个组件实例，glass-easel 下无压力 | 最轻量 |
| 视觉一致性 | 100%一致 | 两套视觉，用户感知分裂 |
| 技术债务 | 零 | 永久存在两套并行系统 |

#### 决策结果：方案A — 统一用 `<rarity-effects>` 组件包裹

理由：
1. 项目未上线，没有旧代码兼容负担
2. 一套系统维护成本 = 方案B的一半
3. glass-easel 组件框架下，scroll-view 里十几个组件实例零性能问题
4. 九宫格 grid.wxml 已经用方案A运行稳定（8个实例）
5. 视觉一致性 = 用户稀有度条件反射更强 = 锚定效应更强

### 7.2 决策2：Grid/Wheel 是否增加独立奖品池预览区

#### 各行业做法对比

| 行业 | 游戏区 vs 预览区 | 是否分离 |
|------|----------------|---------|
| 米哈游（原神） | 抽卡动画区 + "详情"按钮打开完整奖品池列表 | 分离 |

| 线下活动策划 | 大转盘（互动）+ 旁边展板列出所有奖品和规则（信息） | 分离 |

**核心发现**：游戏区 = 交互体验（位置由玩法决定），预览区 = 信息展示（用户按需查看详情）。两个是不同的用户需求，不是"重复展示"。

#### 两种方案对比

| 维度 | 不加（保持12/14） | 加（14/14统一） |
|------|----------|----------|
| 代码一致性 | 12个有预览区 + 2个没有 = 到处需要 `if (mode !== 'grid')` 分支 | 14个统一模式，批量改动一把梭 |
| P2适配 | grid/wheel 只能点击游戏格子/扇区弹详情，动画中难以操作 | 独立预览区提供稳定的点击目标 |
| 维护成本 | 每次改预览区逻辑都要考虑2个特例 | 零特例 |
| 用户体验 | grid/wheel 用户无法便捷浏览所有奖品的详情 | 所有玩法体验完全统一 |
| 信息分层 | 游戏区被迫承担交互+信息双重职责 | 游戏区专注交互，预览区专注信息 |

#### 决策结果：加 — 14/14统一

理由：
1. 长期维护：14个组件统一模式 vs 12+2特例，前者维护成本远低
2. P2"点击查看详情"需要独立的可点击预览区（游戏区动画中不可靠）
3. 游戏区保持 sort_order 位置不变（九宫格顺时针、转盘扇区），预览区按稀有度降序
4. 不是"重复展示"——功能不同：游戏区是交互，预览区是信息查阅

### 7.3 决策3：概率数字的数据来源（已取消：不展示概率）

#### 各行业做法对比

| 行业 | 概率数据来源 | 是否前端估算 |
|------|-----------|------------|
| 中国游戏公司（全部） | 后端接口返回精确值，如"5星概率0.600%" | 绝不 |
| 腾讯/网易/米哈游 | 2017年文化部规定强制公示，后端提供 | 绝不 |
| 王者荣耀/原神 | 每个抽卡池的概率表由后端配置和返回 | 绝不 |
| 电商营销抽奖 | 同受市场监管总局管辖，后端提供 | 绝不 |
| 小公司常见错误 | 前端硬编码估算概率，与实际不符引发信任危机 | 做了但出问题 |

**核心发现**：
1. 中国法规（2017年文化部补充规定）要求网络游戏虚拟道具抽取概率必须公示
2. 后端一定有概率配置（rarity_code + 抽奖逻辑靠概率权重运行），只是未暴露给前端
3. 前端硬编码估算会出现"展示3%但实际0.6%"的信任危机
4. 后端概率可能随时调整（保底机制、概率UP池等），前端硬编码无法同步

#### 两种方案对比

| 维度 | 后端加 drop_rate 字段 | 前端根据 rarity_code 硬编码估算 |
|------|---------------------|-----|
| 数据准确性 | 100%与实际抽奖逻辑一致 | 估算值，可能与实际偏差巨大 |
| 后端改动量 | 序列化层加一个字段（极小） | 零 |
| 合规性 | 满足法规要求 | 展示与实际不符可能违规 |
| 维护成本 | 后端改概率，前端自动同步 | 后端改概率，前端需手动同步，遗漏风险极高 |
| 信任感 | 用户看到的是真实概率 | 用户如果发现估算不准，信任崩塌 |

#### 决策结果：~~后端在 prizes 接口增加 `drop_rate` 字段~~ → **已取消（2026-02-25）**

> 2026-02-25 修订：经行业调研（海底捞/瑞幸/麦当劳/支付宝集五福均不展示概率），项目用户为餐饮消费者而非游戏玩家，决定不展示任何概率数字。惊喜感优先于透明度。本决策及关联的决策6/7全部取消，B2 任务取消。

### 7.4 决策4：奖品详情弹窗组件归属位置

#### 各行业做法对比

| 行业 | 物品详情组件位置 | 复用场景 |
|------|---------------|---------|
| 游戏公司（全部） | 全局组件，统一"物品详情卡" | 抽卡 / 背包 / 商店 / 交易 / 锻造 |
| 电商平台 | 全局组件，统一"商品详情页" | 首页 / 搜索 / 购物车 / 订单 / 推荐 |
| 小公司常见错误 | 各模块各写一个详情弹窗 | 3处相同UI、3份代码、3倍维护 |

#### 天工项目需要物品详情的场景

| 场景 | 页面/组件 | 包位置 |
|------|----------|-------|
| 抽奖奖品池预览 | 14个子组件 | packageLottery |
| 抽奖结果展示 | result-modal | packageLottery |
| 兑换商城商品 | pages/exchange/ | 主包 |
| 道具仓库/背包 | packageTrade/trade/inventory/ | packageTrade |
| 交易市场商品 | packageTrade/trade/market/ | packageTrade |

#### 两种方案对比

| 维度 | 方案A：`packageLottery/shared/` | 方案B：`components/`（全局） |
|------|------|------|
| 复用范围 | 仅抽奖模块内 | 所有分包 + 主包均可引用 |
| 分包限制 | packageTrade / pages 引用跨分包组件有限制 | 主包组件对所有分包无限制 |
| 与现有架构一致性 | shared/ 已有3个抽奖专用组件 | components/ 已有4个全局组件（pagination / loading-spinner / empty-state / price-chart） |
| 5个场景覆盖 | 仅覆盖2个（抽奖相关） | 全部覆盖5个场景 |
| 长期维护 | 将来兑换/交易需要详情时还要再写一个 | 一次到位 |

#### 决策结果：方案B — 全局公共组件 `components/prize-detail-modal/`

理由：
1. 5个场景需要物品详情展示，全局组件一次覆盖
2. `components/` 目录已有 pagination / loading-spinner / empty-state / price-chart 等全局组件，架构一致
3. 主包组件对所有分包（packageLottery / packageTrade）无引用限制
4. 放 packageLottery/shared/ 里会导致 packageTrade 和 pages 引用跨分包组件，微信小程序分包引用有限制
5. 未上线项目，一步到位避免将来重构

### 10.4 ~~关键发现：`drop_rate` 的真实计算逻辑~~ — 已取消展示概率

**后端抽奖引擎的实际概率体系（`tier_first` 模式）**：

```
概率 = P(命中该档位) × P(在该档位内被选中)
     = (tier_weight / 1,000,000) × (win_weight / 该档位所有active奖品的win_weight之和)
```

**真实数据库数据示例（campaign_id=1, segment=default）**：

| 档位 | tier_weight | P(档位) |
|------|-----------|---------|
| high | 50,000 | 5.00% |
| mid | 150,000 | 15.00% |
| low | 800,000 | 80.00% |

| 奖品 | 档位 | win_weight | 同档位权重总和 | P(档位内) | 综合概率 |
|------|------|-----------|-------------|---------|---------|
| 八八折 | high | 10,000 | 1,000,000 | 1.00% | 0.050% |
| 100积分 | high | 90,000 | 1,000,000 | 9.00% | 0.450% |
| 2000积分券 | high | 90,000 | 1,000,000 | 9.00% | 0.450% |
| 500积分券 | high | 610,000 | 1,000,000 | 61.00% | 3.050% |
| 九八折券 | high | 200,000 | 1,000,000 | 20.00% | 1.000% |
| 甜品1份 | mid | 583,000 | 1,000,000 | 58.30% | 8.745% |
| 精品首饰 | mid | 250,000 | 1,000,000 | 25.00% | 3.750% |
| 生腌拼盘 | mid | 167,000 | 1,000,000 | 16.70% | 2.505% |
| 青菜1份 | low | 240,000 | 750,000* | 32.00% | 25.600% |
| 神秘彩蛋 | low | 130,000 | 750,000* | 17.33% | 13.867% |
| ... | low | ... | ... | ... | ... |

*注：low 档位 6 个 fallback 奖品权重合计约 760,000（示例计算用 750,000）

**复杂因素**：
1. 不同用户分层（default/new_user/vip_user）的档位概率不同
2. 预算不足时档位自动降级（B3→B2→B1→fallback）

4. 库存为0时奖品退出概率池

**结论**：前端文档简单地设想 `drop_rate` 是一个静态浮点字段，但后端实际概率是动态的、分层的、多因素影响的。

**原建议方案**：~~在 API 响应中返回基准概率~~ → **已取消：不展示概率**。后端概率体系仅在引擎内部使用。

### 14.3 B3：保底进度信息增强（修复 + 扩展）

**改动文件**：
1. `routes/v4/lottery/campaigns.js` 的 config 接口（L286-302）
2. `services/lottery/QueryService.js` 的 getCampaignConfig 方法（L178 处 `guarantee_rule` 硬编码为 `null` 需修复）

**当前问题**：`guarantee_info.exists` 永远为 `false`，用户看不到保底进度。原因：`QueryService.getCampaignConfig()` L178 将 `guarantee_rule` 硬编码为 `null`，而 config 路由 L295 用 `!!fullConfig.guarantee_rule` 判断。

**根因分析（基于真实代码和数据库双重验证）**：
- `guarantee_rule` 为 `null` 是正确的设计——V4.6 Pipeline 架构将保底逻辑迁移到了独立的 `PityCalculator` 和 `lottery_strategy_config` 表
- 但 config API 的 `guarantee_info` 构造仍依赖旧的 `guarantee_rule` 字段，导致永远返回 `false`
- **关键区分**：数据库中 `pity`（已启用，软保底 + 渐进概率提升 + 硬保底）和 `guarantee`（已禁用，旧的指定奖品硬保底）是两套独立配置。B3 必须从 `pity` 配置组读取，不是 `guarantee` 配置组

**改动内容**：
1. 从 `lottery_strategy_config` 表读取 `config_group='pity'` 的配置（`enabled`, `hard_guarantee_threshold`）
2. 从 `lottery_user_experience_state` 表（**注意：表名单数形式**）查询当前用户的 `empty_streak`
3. 计算 `remaining = hard_guarantee_threshold - empty_streak`
4. 在 config 接口 `guarantee_info` 中返回完整信息

**当前 guarantee_info 结构**（永远返回 exists=false）：
```json
{
  "exists": false,
  "description": "连续抽奖有惊喜哦~"
}
```

**增强后结构**：
```json
{
  "exists": true,
  "pity_enabled": true,
  "guarantee_threshold": 10,
  "current_pity": 6,
  "remaining": 4,
  "description": "距离保底还有 4 次"
}
```

**正确的数据源**（基于真实代码和数据库双重验证，已修正列名和表名）：

| 数据 | SQL 查询条件 | 说明 |
|------|------------|------|
| pity_enabled | `lottery_strategy_config` WHERE `config_group='pity'` AND `config_key='enabled'` | 当前值：`true`（注意：列名是 `config_group`，不是 `group_key`） |
| guarantee_threshold | `lottery_strategy_config` WHERE `config_group='pity'` AND `config_key='hard_guarantee_threshold'` | 当前值：`10` |
| current_pity (empty_streak) | `lottery_user_experience_state` WHERE `user_id=X` AND `lottery_campaign_id=Y` | 表名单数形式，唯一约束 `(user_id, lottery_campaign_id)` |

**注意事项**：
- **不要从 `config_group='guarantee'` 读取**：该配置组的 `enabled=false`（已禁用的旧系统）
- **不是 LotteryUserGlobalState**：`LotteryUserGlobalState` 存储全局运气债务（luck_debt），不是保底计数器
- **Pity 系统的核心计数器**：`LotteryUserExperienceState.empty_streak`（每次空奖 +1，非空奖重置为 0）
- **`config_value` 字段类型是 JSON**：读取后需要 `JSON.parse()` 或使用 `LotteryStrategyConfig.getParsedValue()` 方法

**可复用的后端已有能力**：
- `LotteryStrategyConfig.getConfigByGroup('pity')` — 按 `config_group` 批量读取配置（**方法参数名是 `config_group`**）
- `LotteryStrategyConfig.getConfigValue('pity', 'enabled', false)` — 读取单个配置值并提供默认值
- `LotteryUserExperienceState.findOrCreateState(user_id, lottery_campaign_id)` — 获取或创建体验状态记录
- `LotteryUserExperienceState.shouldTriggerPity(threshold)` — 判断是否应触发硬保底
- `PityCalculator._calculatePityProgress(empty_streak)` — 计算保底进度比例 `empty_streak / hard_pity_streak`

**可扩展方向**：
- 未来如需展示 Pity 概率提升曲线：`multiplier_table`（`{0:1, 1:1, 2:1.2, 3:1.5, 4:1.8, 5:2.2, 6:2.8, 7:3.5, 8:5, 9:10}`）数据已在 `config_group='pity'` 中
- 未来如需展示 AntiEmpty 保护状态：同样从 `LotteryUserExperienceState` 读取
- 抽奖引擎 `PityCalculator` 已返回 `pity_progress` 字段（0.0-1.0），可直接复用此计算逻辑
