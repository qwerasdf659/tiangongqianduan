## 二、前端依赖的后端API及字段（需后端确认/实现）

### 2.1 兑换商品列表 — 新增字段

**API：** `GET /api/v4/backpack/exchange/items`

前端读取的新增字段（需后端在响应中返回）：

| 字段 | 类型 | 说明 | 前端用途 |
|------|------|------|---------|
| `mint_instance` | BOOLEAN | 铸造开关 | 展示"兑换后将铸造专属物品实例"标识 |
| `item_template_id` | BIGINT/NULL | 关联物品模板 | 判断是否有关联模板 |

### 2.2 兑换成功响应 — 新增 minted_item 对象

**API：** `POST /api/v4/backpack/exchange`

当 `mint_instance=true` 时，前端期望响应中包含 `minted_item` 对象：

```json
{
  "success": true,
  "data": {
    "order": { "order_no": "EXC...", "status": "pending" },
    "minted_item": {
      "item_id": 8001,
      "tracking_code": "LT260320123456",
      "serial_number": 42,
      "edition_total": 100,
      "instance_attributes": {
        "quality_score": 87.42,
        "quality_grade": "精良",
        "pattern_id": 337,
        "颜色": "冰蓝"
      }
    }
  }
}
```

前端处理位置：`packageExchange/exchange-detail/exchange-detail.ts` 行682-720

### 2.3 用户背包 — 新增字段

**API：** `GET /api/v4/backpack`

背包物品数据需新增的字段（前端已在 inventory.ts 中处理）：

| 字段 | 类型 | 说明 | 前端用途 |
|------|------|------|---------|
| `instance_attributes` | JSON/NULL | 实例属性 | 展示品质等级标签（金/紫/蓝/白/灰） |
| `instance_attributes.quality_score` | DECIMAL(5,2) | 品质分 | 展示品质分数值 |
| `instance_attributes.quality_grade` | VARCHAR | 品质等级 | 展示等级文字（完美无瑕/精良/良好/普通/微瑕） |
| `instance_attributes.pattern_id` | INT | 纹理编号 | 展示纹理编号 |
| `serial_number` | INT/NULL | 限量编号 | 展示 #0042/0100 格式 |
| `edition_total` | INT/NULL | 限量总数 | 与 serial_number 配合展示 |
| `holds` | ARRAY | 锁定列表 | 提取 trade_cooldown 类型，计算冷却倒计时 |

`holds` 数组中 trade_cooldown 类型的格式：

```json
{
  "hold_type": "trade_cooldown",
  "status": "active",
  "reason": "新物品交易冷却期（7天）",
  "expires_at": "2026-03-27T18:30:00.000+08:00"
}
```

前端处理位置：`packageTrade/trade/inventory/inventory.ts` 行311-341

### 2.4 交易市场挂单列表 — 新增字段和筛选参数

```
formatEdition(serial_number=42, edition_total=100) → "#0042 / 0100"
```

- 补零到 edition_total 的位数
- serial_number 或 edition_total 为 null 时返回空字符串

实现位置：`utils/image-helper.ts` formatEdition 函数

## 五、前端冷却期计算规则（已实现）

1. 从 `holds` 数组中查找 `hold_type === 'trade_cooldown'` 且 `status === 'active'` 的记录
2. 读取 `expires_at`（ISO8601 格式），计算与当前时间的差值
3. 冷却期内：物品可见可用，但"上架"按钮禁用，展示倒计时文案（X天Y小时 / X小时Y分钟）
4. 冷却结束：展示"已结束"

实现位置：`utils/image-helper.ts` getTradeCooldown + formatCooldownRemaining

## 七、前端技术架构确认

| 维度 | 实现 |
|------|------|
| API路径前缀 | `/api/v4/` 统一配置于 `config/env.ts` |
| HTTP客户端 | `APIClient` 类，`utils/api/client.ts` |
| Token处理 | JWT双Token（access_token + refresh_token），401自动刷新重试 |
| Token有效性检查 | `validateJWTTokenIntegrity()` 请求前校验格式和长度 |
| 并发刷新控制 | `isRefreshing` + `refreshSubscribers` 队列 |
| 幂等键 | `Idempotency-Key` 请求头（非请求体） |
| 错误处理 | 400/401/403/404/409/429/500/503 全覆盖 |
| 字段命名 | API交互 100% snake_case，业务逻辑 100% camelCase |
| 展示字段 | 下划线前缀（`_qualityGrade`），与后端原始字段区分 |
| 工具函数导入 | 统一从 `utils/index.ts` 导入 |
| Mock数据 | 无，所有数据来自后端API |
| 状态管理 | MobX Store（user/points/trade/audit） |

## 八、前端数据流（纯文本流程图）

### 8.1 兑换流程（mint_instance=true）

```
用户 → 选择商品(exchange-detail.ts)
  → 选择SKU规格(spec_values)
  → 确认兑换
  → POST /api/v4/backpack/exchange { exchange_item_id, sku_id, quantity }
  → 后端: 扣材料 → 扣库存 → 建ExchangeRecord → mintItem() → 铸造实例
  → 响应: { order, minted_item: { serial_number, instance_attributes, ... } }
  → 前端: 展示铸造成功弹窗
    → 限量编号: #0042/0100
    → 品质等级: 精良(87.42分)
    → 纹理编号: #337
  → 跳转背包(inventory.ts)
```

### 8.2 背包展示

```
用户 → 打开背包(inventory.ts)
  → GET /api/v4/backpack
  → 后端返回: items[] (含 instance_attributes, serial_number, edition_total, holds)
  → 前端处理:
    → getQualityGradeStyle(quality_grade) → 颜色/CSS类名
    → formatEdition(serial_number, edition_total) → #0042/0100
    → getTradeCooldown(holds) → 冷却倒计时
    → can_sell = allowed_actions.includes('sell') && 无活跃冷却期
```

### 8.3 C2C市场筛选

```
用户 → 打开交易市场(market.ts / market-behavior.ts)
  → 选择品质等级筛选(完美无瑕/精良/良好/普通/微瑕)
  → GET /api/v4/market/listings?quality_grade=精良&sort=quality_score_desc
  → 后端: WHERE instance_attributes->quality_grade = '精良' ORDER BY quality_score DESC
  → 响应: products[].item_info.instance_attributes
  → 前端: 展示品质标签 + 限量编号 + 纹理编号
```
