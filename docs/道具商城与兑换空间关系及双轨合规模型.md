# 道具商城 / 幸运空间 / 臻选空间 三者关系与双轨合规模型

> **归属**：微信小程序前端项目 ←→ 后端数据库项目（积分抽奖系统）
> **核实日期**：2026-06-11 北京时间
> **核实方式**：Node.js（mysql2，读 `.env`）直连真实库 `restaurant_points_dev`（`dbconn.sealosbja.site:42569`，MySQL 8.0.30）+ 逐文件读后端代码，未引用任何历史报告
> **用途**：固化"道具商城 / 幸运空间 / 臻选空间"的层级关系与背后的合规逻辑，供产品、运营、法务对齐，避免反复追问"凭什么星石无价值"

---

## 一、一句话结论

三者**不是平级的三个分区**，而是"**两层两轨**"结构：

- **道具商城** 与 **商品兑换** 是**平级的两个顶层频道**（两条不同的合规轨）。
- **幸运空间** 与 **臻选空间** 是 **商品兑换** 这一个频道**内部**的两个子空间（同一条轨内按价值高低再分）。

> 第一刀按"**货币 / 合规轨**"切（道具商城 vs 商品兑换）；第二刀只在源晶轨内按"**价值高低**"切（幸运 vs 臻选）。道具商城不需要第二刀，因为它整条轨都是零价值。


### 三者关系的三句话

1. **道具商城 ⊥ 商品兑换**（正交、平级、两条轨）：区别是货币和合规逻辑——星石娱乐轨 vs 源晶兑换轨，物理分 Tab，合规叙事不同不能混。
2. **幸运空间 + 臻选空间 ⊂ 商品兑换**（从属、同轨）：同一条源晶轨内部按价值高低再分，区别只是 `value_tier`。
3. **分类用了两个不同维度**：先按"货币/合规轨"切，再只在源晶轨内按"价值高低"切。

## 四、为什么星石"无价值"、源晶"有价值"（核心合规逻辑）

> **常见误解**：星石和源晶都能靠抽奖获得，凭什么一个无价值一个有价值？
> **真相**：区分依据**不是"怎么获得"**，而是下面三个真实库证据决定的。

### 4.1 三个关键事实（真实库实测，2026-06-11）

**事实一：源晶带"预算价值锚"，星石不带（`material_asset_types.budget_value_points`）**

| asset_code | display_name | budget_value_points |
|---|---|---|
| star_stone（星石） | 星石 | **0**（不锚定任何商家成本） |
| red_core_shard | 红源晶碎片 | 1 |
| purple_core_gem | 紫源晶 | 1600 |

源晶系每一种都挂着 `budget_value_points`（商家真实投入的预算成本），**星石恒为 0**。源晶代表"商家真金白银的成本"，星石不代表任何成本。

**事实二：转换单向，星石是"终点蓄水池"（`asset_conversion_rules`）**

库里 9 条转换规则**全部是 `源晶 → 星石`，没有任何一条 `星石 → 源晶` 或 `星石 → 其它有价值资产`**：

- 紫源晶碎片 → 星石 (1:1)、蓝 → 星石 (2:1)、红碎片 → 星石 (10:1)、红源晶 → 星石 (2:1)……
- 星石**只进不出**，是整条链的最终汇聚点，无任何现金或价值出口。

**事实三：实物/券挂人民币参考价，prop 零价值（`item_templates.reference_price_points`）**

实测：`product`/`voucher` 共 30 条中 13 件挂着 `reference_price_points`（人民币锚）；道具商城的 prop 为"向下销毁"型、无人民币锚。

### 4.2 为什么"聚合顶端货币"必须被设计成无价值

这是最反直觉、也最精妙的一点：**恰恰因为星石是转换链的顶端（什么源晶都能换成它），它才必须被阉割成无价值。**

逻辑链：

1. **源晶 = 兑换货币（类比积分）**：带预算成本锚，被指定兑换有人民币锚的实物。这条路径合规上定性为"**积分兑换**"——确定性、明码标价、用户自选，不是赌。
2. **星石 = 聚合终点**：所有颜色源晶都能往上汇聚成星石。**假如星石能买有价值的东西**，则：
   - 所有源晶可通过"换成星石"间接获得购买有价值物品的能力；
   - 星石变成统一可流通的"**类代币**"；
   - 抽奖 → 各色源晶 → 全聚合成星石 → 买值钱东西 = **以小博大套现闭环** = 射幸/赌博定性。
3. **破局**：把"聚合顶端货币"焊死成**只能买零价值道具（买入即销毁）**。于是用户把源晶换成星石，是一个"**主动降级**"决定（放弃兑实物的价值，换成纯娱乐）——代码里的"**向下销毁**"（`SYSTEM_BURN`）语义即此。星石这条路没有任何现金出口，套现链被物理切断。

> 一句话：**源晶"低"但有用（能兑实物），星石"高"但被故意阉割成娱乐专用**。设计者把最危险的聚合货币主动"拔牙"，因为聚合货币才是套现风险的源头。

### 4.3 诚实说明：源晶轨不靠"无价值"，靠另一套软合规

"源晶也是抽奖来的、却能买值钱实物"这个张力**真实存在**。系统对"源晶兑实物"这条路的合规防御**不是靠无价值**，而是靠：

- 定性为**积分兑换**（确定性兑换，非抽奖直接掉落实物）；
- 高价值实物叠 `value_tier='high'` + `redeem_requirement` **会员/消费门槛**（即臻选空间），使其不是"纯以小博大"；
- 抽奖概率公示 + 保底等合规披露。

### 4.4 双轨合规模型总览

| | 星石轨（道具商城） | 源晶轨（商品兑换） |
|---|---|---|
| 货币 | 星石（`budget_value=0`、聚合终点、只进不出） | 源晶/水晶系（带预算锚、兑换货币） |
| 卖什么 | 零价值 prop（向下销毁） | 人民币锚定的实物/券 |
| 合规定性 | 娱乐闭环、**无现金出口**（硬隔离） | **积分兑换** + 消费门槛 + 概率公示（软合规） |
| 退款 | 禁退（`PROP_NO_REFUND`） | 可退、需履约 |
| 防御方式 | "无价值 + 无出口"机制焊死 | "兑换定性 + 门槛 + 披露"持续维护 |

> 两条轨走的是**两套不同的合规逻辑**，混在一个频道会让两套叙事互相污染。这是"道具商城与商品兑换必须物理分 Tab"的根本原因——不只是"两种货币"，而是"两种合规价值观"。
>
> **重要澄清**：星石轨的合规安全来自"**买入标的是零价值 prop、价值向下销毁**"，**不取决于付款货币是不是星石**。即便允许水晶系也买道具（向下销毁同样成立），合规依然安全（见第七节）。表中"星石"是该轨的**主货币定位**，非合规必要条件。

## 五、后端焊死这套合规的机制（实测代码）

这套"星石无价值、隔离套现"不是口头约定，是代码强制执行（fail-closed）：

| 守卫/常量 | 文件 | 作用 | 错误码 |
|---|---|---|---|
| `assertPriceAssetAllowed` | `services/shared/AssetProductGuard.js` | 星石永不得为有价值商品计价 | `STONE_BUY_VALUABLE_FORBIDDEN` |
| `assertPriceAssetAllowed` | 同上 | 实物/券仅限水晶系计价 | `VALUABLE_NEEDS_CRYSTAL` |
| `assertVirtualPropOnly` | 同上 | 道具商城只能上零价值 prop | `PROP_TYPE_FORBIDDEN` / `PROP_HAS_VALUE` |
| `assertBiddableTarget` | 同上 | 竞价标的仅允许 prop | `BID_TARGET_FORBIDDEN` |
| `assertPayoutNotCurrency` | 同上 | 产出禁为货币型资产（防借货币中转套现） | `PAYOUT_CURRENCY_FORBIDDEN` |
| prop 退款拦截 | `services/exchange/CoreService.js` | 虚拟道具单禁退 | `PROP_NO_REFUND` |
| 转换方向守卫 | `services/AssetConversionRuleService.js` | 禁止反向（星石→源晶） | `CONVERT_DIRECTION_FORBIDDEN` |
| 可竞价白名单 | `constants/ProductCurrencyWhitelist.js` | `BIDDABLE_ITEM_TYPES=[prop]`、`VALUABLE_ITEM_TYPES=[product,voucher]` | — |

> 三重防护：`assertPriceAssetAllowed`（星石不买有价值物）+ `is_biddable`（竞价币白名单，实测仅 `star_stone=1`）+ `assertBiddableTarget`（标的必须 prop）。

### 道具商城计价货币：两种都合法（运营选择，非合规风险）

> **结论**：道具商城收"只星石"还是"星石+水晶系"，**两种都不违法**。因为道具是零价值 prop，任何货币买它都是**价值向下、向 `SYSTEM_BURN` 销毁**，不创造任何套现/向外出口。合规红线只针对"价值向上/向外→现金"，与"向下买零价值道具"无关。

后端代码本就放行两者（`AssetProductGuard.assertPriceAssetAllowed` 注释明示"star_stone / 水晶系均可，两者买零价值道具都是向下销毁，放行"）。

| 选项 | 是否合法 | 好处 | 代价 |
|---|---|---|---|
| 只收星石 | ✅ 合法 | "星石=娱乐终点"叙事最干净 | 用户水晶余额只能去兑换商城花 |
| 星石 + 水晶系都收 | ✅ 合法 | 消化用户水晶余额、运营更灵活 | 水晶系跨两频道，"星石专属娱乐"标签略淡 |

> 这是**运营定位/心智偏好**选择，不是合规选择，由产品/运营决定即可。
> 边界说明：以上基于本系统自身合规设计逻辑 + 代码分析，"价值向下销毁不构成套现"这条逻辑成立；最终合规定性仍建议法务背书。

## 八、微信小程序「道具商城」Tab 前端对接说明（2026-06-11 实读后端代码核实）

> **结论**：小程序应在「商品兑换」旁新增平级顶层 Tab「道具商城」（星石轨）。后端 C 端接口大部分就绪，但**列表接口当前无法直接按 prop 过滤**，需后端补一个小改动或前端走详情兜底（见 8.3）。小程序代码不在本仓库，本节供小程序前端同事对接。

### 8.1 C 端现有接口清单（真实路由 `routes/v4/exchange/`）

| 用途 | 方法 + 路径 | 鉴权 | 关键入参 | 返回 |
|---|---|---|---|---|
| 商品列表 | `GET /api/v4/exchange/items` | optionalAuth | `space`(lucky/premium)、`keyword`、`category_id`、`status`、`page`、`page_size`(≤50)、`sort_by`、`sort_order` | `{ items, pagination, summary }` |
| 商品详情 | `GET /api/v4/exchange/items/:exchange_item_id` | optionalAuth | 路由参 `exchange_item_id` | `{ item }` |
| 下单兑换 | `POST /api/v4/exchange` | 登录 | Header `Idempotency-Key`(必填)；body `exchange_item_id`、`quantity`(≤10)、`sku_id`(多规格必填) | `{ order, remaining }` |
| 我的订单 | `GET /api/v4/exchange/orders` | 登录 | `status`、`source`、分页 | `{ orders, pagination }`（订单含 `is_prop`/`refundable`） |
| 订单详情 | `GET /api/v4/exchange/orders/:order_no` | 登录 | 路由参 `order_no` | `{ order }` |
| 空间统计 | `GET /api/v4/exchange/space-stats` | — | `space`(必填) | `{ space, total_items, ... }` |
| 星石装饰商城（模块D，可选） | `GET /api/v4/exchange/decorations` | optionalAuth | — | `{ decorations }` |

### 8.2 字段约定（一律用后端 snake_case 原名，禁止前端映射层）

- 列表项 `items[]` 下发字段集 = `EXCHANGE_MARKET_ATTRIBUTES.marketItemView`：`exchange_item_id`、`item_name`、`description`、`space`、`tags`、`is_new`、`is_hot`、`is_limited`、`is_recommended`、`sell_point`、`rarity_code`、`status`、`primary_media_id`、`category_id`、`sort_order`、`publish_at`、`created_at`，外加上提的 `value_tier`。
- **列表项不下发 `item_type`、`item_template_id`**（出于商业字段最小暴露）。
- 订单对象带后端派生字段（`services/exchange/QueryService.js`，BE-1）：
  - `is_prop`：是否虚拟道具（由 `exchange_item → item_template.item_type === 'prop'` 推导）。
  - `refundable`：prop 单恒 `false`；非 prop 单按退款规则（`status ∈ ['approved','shipped']`）。
  - 道具卡片/订单**不显示退款入口**时，读 `refundable=false`，**不要在前端用 item_type 散落判断**。

### 8.3 ⚠️ 真实差距：列表接口当前无法直接筛选道具（prop）

实读 `services/exchange/QueryService.js#getMarketItems`：`where` 仅支持 `status / space / keyword / category_id / min_cost / max_cost / stock_status`，**没有 `item_type` 条件**；列表视图也不返回 `item_type`。

因此小程序道具商城 Tab **不能**像 web 后台那样"拉全部商品再前端按 prop 过滤"（C 端拿不到 item_type）。三种落地选择：

| 方案 | 做法 | 评价 |
|---|---|---|
| A（推荐，后端小改） | 后端给 `GET /api/v4/exchange/items` 增加 `item_type` 白名单筛选（如 `item_type=prop`），列表视图加返回 `is_prop` 派生位 | 最干净，和"商品兑换用 space、道具商城用 item_type"的双轨契约一致 |
| B（运营约定，零改动） | 道具商品统一归到约定的 `category_id`（道具品类），小程序按 `category_id` 拉取 | 不改后端，但依赖运营纪律，category 与 prop 非强绑定，易错配 |
| C（模块D 独立路由） | 若道具按"星石虚拟装饰"形态运营，直接用 `GET /api/v4/exchange/decorations` 独立接口 | 与现有装饰商城共用，适合纯展示装饰类道具 |

> **本仓库内不擅自改 C 端列表接口**。方案 A 涉及后端数据库项目改动，**需产品确认后由后端同事实现**（符合"后端没实现的功能由后端决定是否开发"的协作规则）。小程序前端先按 A 的契约预留 `item_type=prop` 入参对接。

> **重要更正**：早期快照曾写"star_stone 渠道价 0 条"，那是道具商城页上线前的历史值。**道具商城的"一键上架"会自动完成"建模板→建商品→建 SKU→设星石价"四步**，运营每建一个道具，星石价同步配好，无需单独配价。
>
> 因此 web 后台侧"道具 + 星石定价"数据**已齐**。C 端小程序若仍看不到道具，原因**不是数据缺失**，而是 8.3 所述"C 端列表接口尚不支持按 `item_type=prop` 筛选"——需后端按方案 A 补筛选入口，与数据配齐与否无关。
