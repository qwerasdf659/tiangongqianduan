## 一、背景与问题复盘

在修复"店员商家功能按钮显示不全"问题过程中，暴露出两类本质不同的问题：

### 1.1 问题一：消费录入/我的提交按钮对店员不显示（执行层假设失误）

- **现象**：店员（role_level=20）登录金蛋页，「扫码核销」显示，但「消费录入」「我的提交」不显示。
- **根因**：`permissions` 判定按"扁平数组"形态写死 `permissions.includes('consumption:create')`，而后端实际下发的是"分组对象"形态 `{"consumption":["create","read","scan_user"]}`，`includes` 判定失败。
- **教训**：违反"先验证后假设"——对后端响应形态做了推测而未核实。正确做法是一开始就兼容两种等价形态。

### 1.2 问题二：商家工作台给店员显示了"审核管理"（历史口径不一致）

- **现象**：店员在「我的」页商家工作台看到了本不该有的「审核管理」入口。
- **根因**：「审核管理」卡片显隐用了 `isReviewer`（role_level≥20），而"审核"按业务应是管理员功能（role_level≥100）。同一业务概念在不同页面用了不同门槛。
- **教训**：缺少"权限的单一事实源"。每个页面各自硬编码 `role_level >= X`，没有统一抽象，导致口径漂移、改一个漏一个。

### 1.3 本质结论

不是"需求没规划清楚"（对接文档把需求写得很明确），而是**权限判定的实现没有统一抽象**——同一规则散落在多个页面、容易写歪。这是架构层面可以提前避免的，因此本次做一次纯前端重构收口。

## 三、方案设计：权限判定统一模块

新增 `utils/permission.ts`，把商家/管理类能力判定收口为语义化函数。

**核心原则**：只新增统一入口 + 让现有页面改调它，**不改变任何现有显示效果和门槛数值**（纯重构、行为等价）。

### 3.1 角色门槛常量（消灭魔法数字）

```typescript
const ROLE_LEVEL = {
  STAFF: 20,      // 商家店员（扫码核销/消费录入/我的提交/审核链待办）
  MANAGER: 40,    // 店长
  BUSINESS: 60,   // 业务经理（部分 console 接口仍 lv60）
  ADMIN: 100      // 平台管理员（console 审核/跨店特权）
} as const
```

> 这些数字是"对齐后端契约"而非"前端自定义业务规则"，集中常量维护，后端调整时只改此处。

### 3.2 permissions/me 两种形态兼容（从 lottery.ts 提取共用）

```typescript
// 兼容后端两种等价形态：
//   扁平数组 ["consumption:create"]
//   分组对象 { consumption: ["create"] }
function hasPermission(permissions, permissionKey) {
  if (Array.isArray(permissions)) return permissions.includes(permissionKey)
  if (permissions && typeof permissions === 'object') {
    const [domain, action] = permissionKey.split(':')
    return Array.isArray(permissions[domain]) && permissions[domain].includes(action)
  }
  return false
}
```

### 3.3 语义化能力判定（页面只调这些，不再写 role_level>=X）

| 函数 | 入参 | 含义 | 门槛 |
| --- | --- | --- | --- |
| `Permission.isMerchant(roleLevel)` | role_level | 商家店员及以上 | ≥20 |
| `Permission.isManager(roleLevel)` | role_level | 店长及以上 | ≥40 |
| `Permission.canScan(roleLevel)` | role_level | 扫码核销准入 | ≥20 |
| `Permission.canReviewChain(roleLevel)` | role_level | 审核链待办准入（店员可参与，区别于 isAdmin） | ≥20 |
| `Permission.isAdmin(userInfo)` | userInfo | 平台管理员（复用 determineUserRole） | ≥100 |
| `Permission.canSubmitConsumption(permissions)` | permissions | 消费录入准入 | `consumption:create` |
| `Permission.canViewMySubmissions(permissions)` | permissions | 我的提交准入 | `consumption:create` |

> 命名说明：审核链待办流程（lv20，店员可参与）采用 `canReviewChain`，与 lv100 的 `isAdmin`（console 审核）在语义上明确区分，避免再次出现"审核管理店员可见"的口径错配。

### 3.4 统一导出

`utils/index.ts` 新增 `Permission` 分类导出，遵守"统一从 utils/index.js 导入""index 只做分类导出"约定。`determineUserRole`/`checkAdmin` 保持不变，新模块内部复用，不重复造管理员判断轮子。

### 9.3 🔴 关键修正：§1.1/§3.2 根因判断有误（permissions 不是"分组对象"，是"多包了一层"）

前端 §1.1 推断"后端下发分组对象 `{consumption:[create]}`、导致 `includes` 失败"，§3.2 据此写了"两种形态兼容"的 `hasPermission`。**实测后端并非如此**：

- `UserRoleService.getUserPermissions()` 返回的顶层 `permissions` 字段是**扁平数组**：
  ```json
  ["points:read","lottery:read","lottery:participate","profile:read","profile:update",
   "consumption:create","consumption:read","consumption:scan_user"]
  ```
- **真正的坑在 `/me` 路由的响应组装**（`routes/v4/auth/permissions.js` L96-101）：
  ```js
  const permissions = await UserRoleService.getUserPermissions(user_id)
  //  permissions = { exists, user_id, role_level, permissions:[扁平数组], roles }
  const response_data = {
    user_id, roles: permissions.roles, role_level: permissions.role_level,
    permissions   // ← 整个对象塞进来，于是扁平数组其实在 permissions.permissions
  }
  ```
- 即前端拿到的真实结构是**双层嵌套**：

```text
res.data
 ├─ user_id
 ├─ role_level                  // 顶层有，正确
 ├─ roles: [{role_name, role_level, role_uuid}]   // 注意：roles 不含 permissions 字段
 └─ permissions:                // ← 这是个对象，不是数组！
      ├─ exists
      ├─ user_id
      ├─ role_level
      ├─ roles
      └─ permissions: ["consumption:create", ...]   // ← 扁平数组真身在这里
```

**结论**：前端之前 `res.data.permissions.includes('consumption:create')` 失败，是因为 `res.data.permissions` 是对象（应取 `res.data.permissions.permissions`），**不是因为后端返回分组对象**。§3.2 的"分组对象兼容分支"是基于错误推测的、用不上的补丁。

### 9.4 ⚠️ 需要你拍板的唯一一点：`/me` 响应的「双层嵌套」怎么处理

后端 `/me` 把 `getUserPermissions` 的完整对象整个塞进 `response_data.permissions`，造成 `permissions.permissions` 嵌套 + `role_level`/`roles` 字段在内外层重复。这是后端响应组装的设计瑕疵。按"以后端为权威、前端零映射、降低技术债、不兼容旧接口、项目未上线"的原则，三种方案：

| 方案 | 做法 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A（推荐）后端拍平响应** | 改 `/me` 的 `response_data.permissions` 直接等于扁平数组（`permissions.permissions`），消除嵌套与字段重复 | 响应干净、契约清晰、前端零映射直读 `res.data.permissions` 即数组；最符合"代码即文档" | 需改后端 1 处（未上线、不兼容旧前端，正可一次性改对） |
| B 前端适配嵌套 | 前端改读 `res.data.permissions.permissions` | 不动后端 | 嵌套结构长期留存=技术债，违背"以后端为权威修正到正确标准" |
| C 保留双形态兼容 | 前端 hasPermission 同时兼容数组/对象/嵌套 | 最"稳" | 给一个本不该存在的形态写兼容=纯技术债，明确不推荐 |

> 推荐 **方案 A**：后端把 `/me` 响应拍平（属后端数据库项目改动，约 1~2 行），前端 `hasPermission` 只需保留"扁平数组"判定、删掉 §3.2 的分组对象兼容分支。这才是"以后端为权威、前端零映射、降技术债"的正解。**这一点需你拍板：是否同意后端改 `/me` 响应为扁平结构。**

> ✅ **拍板结论（2026-06-24，已选方案 A 并由后端实施完成）**：后端已把 `GET /api/v4/permissions/me` 响应拍平。实测返回结构：
> ```json
> {
>   "user_id": 12796,
>   "role_level": 20,
>   "roles": [{ "role_uuid": "...", "role_name": "merchant_staff", "role_level": 20 }, ...],
>   "permissions": ["points:read","consumption:create","consumption:read","consumption:scan_user", ...]
> }
> ```
> 即 `res.data.permissions` 现在**直接就是扁平字符串数组**，不再有 `permissions.permissions` 嵌套、不再 role_level/roles 内外层重复。改动文件：`routes/v4/auth/permissions.js`（`/me` 路由响应组装），ESLint+Prettier 通过、权限相关 Jest 用例全通过、`/me` 实测验证拍平生效、`/health` healthy。**前端据此简化（见 §9.4.1）。**

### 9.4.1 前端需配合的简化（方案 A 落地后）

后端已拍平，前端按下述调整，**比原 §3.2 更简单**：

1. **`hasPermission` 删掉"分组对象兼容分支"**，只保留扁平数组判定：
   ```typescript
   // 后端 /me 的 permissions 已是扁平数组，零映射直读
   function hasPermission(permissions: string[], permissionKey: string): boolean {
     return Array.isArray(permissions) && permissions.includes(permissionKey)
   }
   ```
2. **取值路径**：`res.data.permissions`（数组）、`res.data.role_level`（数字）、`res.data.roles`（数组）——全部顶层直读，不再写 `res.data.permissions.permissions`。
3. §3.2 标题"两种形态兼容"随之作废——后端只有一种形态（扁平数组），前端不需要兼容分组对象。
4. 其余 §3.1 门槛常量、§3.3 语义化能力判定、§4 改造清单**全部不变**，照常实施。

#### 行业对标（权限响应结构设计，供拍板参考）

| 公司类型 | permissions 响应设计 | 特征 |
| --- | --- | --- |
| 大厂（美团/腾讯/阿里） | 统一鉴权中台下发**扁平字符串数组**（`["consumption:create"]`）或 bitmap，前端 SDK 直读，绝不嵌套/不重复字段 | 扁平 + 契约稳定 + 单一来源 |
| 中小公司/SaaS | 多为扁平数组或 `{resource:[actions]}` 分组对象二选一，**全栈统一一种** | 简单、不混用 |

## 十、权限显隐：「运营可调 vs 前端写死」决策与统一权限码落地（2026-06-24）

> 触发：用户提问"权限显示收口，是运营可调好还是小程序前端写死好"。本章给出行业对标、方案对比、最终拍板（方案②为主 + 选项A统一权限码），并记录后端已落地内容。

### 10.1 问题拆解：这其实是两层，不是单选题

| 层次 | 关注点 | 正确做法（本项目） |
| --- | --- | --- |
| **权限规则层**（谁能用某功能） | "店员能用哪些功能" | **运营后台可配（RBAC `roles.permissions`）**——已具备，改权限不发版 |
| **前端渲染层**（按钮怎么显隐） | "前端如何消费权限" | **前端按 `/me` 下发的 permissions 动态渲染**（数据驱动），不写死 role_level |

→ "运营可调"指的是权限规则层（已由 RBAC 满足）；"前端写死"是指渲染层硬编码 role_level（错误做法）。**正确终态 = 运营可配权限规则 + 前端动态读 permissions 渲染**，二者不矛盾。

### 10.2 行业对标（功能按钮"谁可见"怎么设计）

| 公司类型 | 设计 | 权限规则可配性 | 前端渲染 |
| --- | --- | --- | --- |
| 大厂（美团/腾讯/阿里） | 权限中台 RBAC，下发能力位/权限码扁平数组；部分连按钮菜单都后端动态下发 | 后台可配（改权限不发版） | 纯数据驱动，零硬编码 |
| 中小公司/SaaS | RBAC + `/me` 下发 permissions，前端 `hasPermission(key)` 显隐 | 后台可配 | 数据驱动 |
| 游戏公司 | 能力位 bitmask/段位，客户端按位判定；GM 权限独立配置 | 多为配置/代码定死，少实时调 | 按位渲染 |
| 活动策划公司 | 活动级布尔位（can_xxx），随活动配 | 活动配置可调（粗粒度） | 简单布尔 |
| 游戏虚拟物品交易/小众二手 | RBAC 扁平 permission 列表 + 后端精校，前端只显隐 | 后台可配 | 前端轻、后端权威 |

**分水岭**：①是否用显式 RBAC 权限码（而非前端写死 role_level）；②权限规则是后台可配还是代码写死。成熟平台（大厂/SaaS/二手）都用"权限码 + 后台可配 + 前端数据驱动"。

### 10.3 三种方案对比 + 最适合本项目的

| 方案 | 做法 | 评价 |
| --- | --- | --- |
| ① 前端写死 role_level | 每按钮 `if(role_level>=X)` | ❌ 技术债最高，规则散落、改门槛要发版、口径易漂移（本次 bug 根源），淘汰 |
| ② 前端按 permissions 动态渲染（**选定为主**） | 前端读 `/me` permissions，`hasPermission(key)` 显隐；权限规则在后端 `roles.permissions` 后台可配 | ✅ 复用现有 RBAC，零后端新增、降技术债；改"店员能用什么"后台改角色权限即可不发版；安全后端兜底。**最适合本项目** |
| ③ 后端下发按钮菜单清单 | 连"显示哪些按钮/文案/图标"都后端配置下发 | 🔸 大厂级，但对当前 3 个固定按钮属过度设计（YAGNI），投入大。未来按钮多到需频繁增删配菜单时再上 |

**结论：方案②为主**。理由：本项目已具备全部基础设施（RBAC `roles.permissions` 后台可配 + `/me` 下发 permissions + system_settings 可配门槛先例 + feature_flags），方案②几乎零后端新增、完全契合"基于现有技术栈、降技术债、未上线一次性投入"目标；运营调整"店员能用哪些功能"在后台角色权限页改即可、不发版，已比"小程序写死"灵活得多，也比方案③轻得多。

## 十一、数据范围分层隔离设计（四个商家功能"点进去看到的数据"按组织层级逐级收窄）

> 触发：用户要求"扫码核销/消费录入/我的提交/审核详情 点进去的数据，按层级过滤——高等级看低等级、只看自己所属范围、一级一级往下；目前店长+店员+管理员，以后会增加层级"。
> 区分：第十章解决的是「按钮可见性」（功能准入，RBAC 权限码）；本章解决的是「数据范围」（data scope，点进去看到的列表内容按组织层级隔离）。两者正交、互补。
> 责任：本章是**后端数据库项目主导的设计**（数据范围隔离必须后端强制，前端零参与过滤）；前端只负责展示后端已过滤好的数据。

### 11.1 核心设计原则：数据范围锚定「门店 + 组织层级树」，绝不用 role_level 数字判断

直觉误区：「role_level 高的看 role_level 低的」——**错误，禁止采用**。原因：
- `role_level` 只表达"职级高低"，不表达"谁管谁"。两个 role_level=40 的店长（A店、B店）平级，但谁都不该看对方的店——他们的隔离边界是**门店**，不是职级数字。
- 数据范围的正确锚点永远是**组织归属（哪个门店/哪个片区）**，职级只决定"在我的范围内能看多细"。

**正确模型 = 两个正交维度**：

| 维度 | 含义 | 数据来源 |
| --- | --- | --- |
| **横向（范围 scope）** | 我能看哪些门店的数据 | `store_staff`（我在职的门店）+ `user_hierarchy`（我作为上级管辖的下级门店，逐级展开） |
| **纵向（粒度 grain）** | 在我能看的门店里，能看多细 | 我在该店的角色：店员=仅自己经手；店长/区域=该范围全部 |

### 11.2 分层数据范围模型（现在三层可用，未来多层零改动向上兼容）

```
管理员 admin(≥100)
  范围：全部门店（跨店特权，无隔离）；粒度：全部
        ↓
区域负责人/业务经理（未来增加，靠 user_hierarchy 挂下级）
  范围：user_hierarchy 中我作为 superior 管辖的所有下级门店（可递归多级往下）；粒度：辖区全部
        ↓
店长 merchant_manager(40)
  范围：自己在职门店（store_staff）；粒度：本店全部
        ↓
店员 merchant_staff(20)
  范围：自己在职门店；粒度：仅自己经手/录入的
```

> 关键优势：现在只有三层也能跑（user_hierarchy 空着，店长/店员靠 store_staff 隔离即生效）；未来加区域负责人**只需在 Web 后台录 user_hierarchy 层级关系，后端代码零改动**。这是"以后会增加层级"诉求的最优解——一次设计、向上兼容。

### 11.3 现状盘点（实测后端各接口范围隔离程度）

| 功能 | 点进去查的数据 | 现有范围隔离 | 状态 |
| --- | --- | --- | --- |
| 审核详情/待办 | 待我审/已审的审核单 | `ApprovalChainService.getPendingStepsForUser` + `_getUserScopedStoreIds` 已做门店/区域隔离（含 user_hierarchy 下级） | ✅ 最完整，是标杆实现 |
| 消费录入/我的提交 | 消费记录 | `consumption/merchant/list` 已做"店员看自己录入(operator)/店长看全店"两级（is_manager 分流） | 🔸 缺"区域层"（未接 user_hierarchy 多店聚合） |
| 扫码核销 | 核销订单/记录列表 | 核销动作有 store 校验；核销「记录列表」的范围隔离需逐接口核查补齐 | 🔸 待核查补齐 |

> 即：**审核链这条链路已经是你要的"逐级隔离"标杆**，其它三个应向它对齐、复用同一套范围解析。

### 11.4 落地设计（基于现有技术栈，最低技术债）

**核心：把"我能看哪些门店"收口为统一能力，四个接口都调它（单一事实源，杜绝口径漂移）。**

1. **提升范围解析为统一服务**：现有 `ApprovalChainService._getUserScopedStoreIds(userId)` 是私有方法。建议提升为独立 `DataScopeService.getAccessibleStoreIds(userId)`（或放入既有合适的领域服务），返回"我可见的 store_id 集合 + 是否管理员全局标记"。四个模块统一复用，不各写各的。
2. **四个查询接口统一接入**：列表查询统一 `WHERE store_id IN (我的可见门店集合)`；店员粒度再叠加 `AND operator_id = 自己`（仅消费录入这类"经手人"语义）。

static async _getUserScopedStoreIds(userId, transaction) {

### 14.1 🔴 关键修正（前述章节均遗漏）：admin/super_admin 的 permissions 是通配符 `*:*`，前端纯 `includes(key)` 判定会对管理员失效

**实测**：`UserRoleService.getUserPermissions`（L922-932）把 `roles.permissions` 的 `{resource:[action]}` 展开为 `resource:action` 扁平串。但 `admin`/`super_admin` 的 `roles.permissions` 实测是 `{"*":["*"]}`，**展开后是字面量 `"*:*"`**，不是逐条业务权限码。

- 因此 admin 经 `/me` 拿到的 `permissions` 数组 = `["*:*"]`（不含 `consumption:scan_user` 等具体码）。
- **后端 `requireMerchantPermission`（L1224-1227）已正确处理三种形态**：
  ```js
  const hasPermission =
    req.user.permissions.includes('*:*') ||            // 超级通配
    req.user.permissions.includes(`${resource}:*`) ||  // 资源级通配（如 consumption:*）
    req.user.permissions.includes(capability)          // 精确码
  ```
  （且 `role_level >= 100` 在 L1218 直接 `next()` 放行，admin 后端不会被拦。）
- **但本文档 §9.4.1 / §10.7 给前端的 `hasPermission` 只写了精确匹配** `permissions.includes(permissionKey)`。→ 后果：**管理员（permissions=`["*:*"]`）在小程序端三个商家按钮（扫码核销/消费录入/我的提交）会全部判 false 而消失**——这正是本文档开篇要根治的"按钮显示不全"在管理员身上的重演。

**修正口径（以后端 `requireMerchantPermission` 为权威，前端零映射对齐后端三段判定）**：前端 `hasPermission` 必须与后端同构，支持通配符：

```typescript
// 与后端 middleware/auth.js requireMerchantPermission 完全同构（后端为权威）
function hasPermission(permissions: string[], permissionKey: string): boolean {
  if (!Array.isArray(permissions)) return false
  const [resource] = permissionKey.split(':')
  return (
    permissions.includes('*:*') ||            // 超级管理员通配
    permissions.includes(`${resource}:*`) ||  // 资源级通配
    permissions.includes(permissionKey)       // 精确权限码
  )
}
```

> 说明：这仍是"扁平数组单一形态"（§9 拍平结论不变），只是判定逻辑要识别通配符。这不是给历史脏数据打补丁，而是后端权限模型本就支持通配（`*:*` / `resource:*`），前端必须对齐，否则与后端口径漂移。§9.4.1 第 1 点、§10.7 的 `canScan` 等所有 `hasPermission` 调用据此统一。
>
> 补充：admin 仍可叠加 `Permission.isAdmin(role_level>=100)` 作为兜底显隐（与后端 L1218 一致），但**根因修正必须落在 `hasPermission` 支持通配**，否则任何"按权限码显隐"的按钮都会对管理员失效。

#### 拍板项 5：前端 `hasPermission` 是否支持通配符 `*:*` / `resource:*`（§14.1）

| 公司类型 | 权限判定如何处理"超级权限/通配" | 特征 |
| --- | --- | --- |
| 大厂（美团/腾讯/阿里） | 鉴权 SDK 前后端同构：下发 `*:*` 或 `resource:*` 通配位，前后端用**同一套判定函数**（含通配展开），super-admin 走通配命中 | 前后端同构、通配内建 |
| 中小公司/SaaS | `hasPermission(key)` 内置 `*` 通配判断；或后端对 admin 直接下发"全量展开的权限码" | 通配判断 或 全量展开 二选一 |
| 游戏公司 | bitmask 全 1 表示 GM 全通；按位与判定，天然支持"全权限" | 位运算全通 |
| 活动策划公司 | 简单 `is_admin` 布尔直接放行，不走权限码 | 布尔兜底 |
| 游戏虚拟物品/二手平台 | RBAC 扁平码 + 后端精校；admin 用通配 `*:*` + 前端同构判定 | 通配 + 前端轻判 |
| **本项目最适合** | ✅ **前端 `hasPermission` 支持 `*:*` / `resource:*` 通配，与后端 `requireMerchantPermission`（L1224-1227）逐字同构** | 你后端**本就是通配模型**（admin=`{"*":["*"]}`），后端三段判定已写好；前端只精确匹配会让管理员破按钮。让前端对齐后端同一套判定 = 零映射、单一口径、长期最稳。**比"后端给 admin 全量展开权限码"更优**：后端展开要随业务加码同步维护一份 admin 全集（新债），通配 `*:*` 一劳永逸 |

> 结论：**必须支持通配**（这是 bug 修复不是可选项）。同时建议保留 `Permission.isAdmin(role_level>=100)` 作为按钮显隐的兜底（与后端 L1218 admin 直接放行同构），双保险。不推荐"后端给 admin 展开全量权限码"——那是给每个新增能力位都要同步维护 admin 全集的持续技术债，与你"降债"目标相悖。
