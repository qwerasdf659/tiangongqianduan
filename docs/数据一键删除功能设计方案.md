## 一、行业数据清理方案对比

### 1. 大厂方案（美团 / 腾讯 / 阿里巴巴）

**核心理念：四层环境隔离 + 数据分级 + 审批工作流**

- **四套独立环境**：开发 → 测试 → 预发布 → 生产，数据库完全隔离，测试数据永远不进入生产
- **数据分类体系**：
  - L1 配置数据（system_settings、roles）—— 仅允许修改，不允许删除
  - L2 业务核心数据（用户、资产、交易）—— 需要审批 + 归档后才能删除
  - L3 运营数据（活动、奖品、广告）—— 需要审批
  - L4 日志/临时数据（操作日志、幂等请求、会话）—— 定时自动清理
- **审批工作流**：双人审批（发起人 ≠ 审批人），高危操作需要主管审批
- **归档优先**：数据先归档到冷存储（OSS/S3），再从热库删除。生产环境永远不会 TRUNCATE
- **回滚能力**：每次清理操作生成回滚快照，72 小时内可回滚

**优点**：安全性极高、可审计、可回滚  
**缺点**：运维成本极高、需要 4 套环境、审批流程慢、小团队养不起

---

### 2. 游戏公司方案（米哈游 / 网易 / 暴雪）

**核心理念：清档（Wipe）+ 测试服/正式服隔离**

- **清档概念**：测试阶段结束后，一次性清空全部玩家数据，保留配置，从零开始
- **GM（Game Master）工具**：管理后台有"一键清档"按钮，但只在测试服可用，生产服禁用
- **分层清档**：
  - 软清档：只清玩家数据（角色、物品、货币），保留活动配置
  - 硬清档：清一切，重建整个游戏世界
- **经济系统保护**：清档后会重新校准经济参数（和本项目的预算积分体系类似）
- **标准化流程**：公告 → 维护模式 → 清档 → 验证 → 开服
- **生产环境**：只做"过期道具回收""休眠账号资产冻结"，绝不做批量删除

**优点**：概念清晰、操作简单、适合有明确"测试期/公测/正式服"生命周期的项目  
**缺点**：清档后无法恢复、生产环境不适用、需要维护模式停机

---

### 3. 虚拟物品交易 / 小众二手平台方案（Steam / 得物 / 闲鱼）

**核心理念：金融级数据不可删 + 归档分离 + 对账驱动**

- **交易记录永不删除**：所有 transaction 级数据受监管合规要求，只归档不删除
- **复式记账一致性**：和本项目一样使用双流水（AssetTransaction + ItemLedger），删除任何一方都会破坏对账
- **冷热分离**：热数据（30 天内）在线库，温数据（30 天-1 年）读副本，冷数据（1 年+）归档到对象存储
- **下架而非删除**：商品/挂牌不是删除，而是状态变更（on_sale → withdrawn → archived）
- **对账驱动清理**：只有对账通过的历史数据才能归档，确保余额 = SUM(流水)

**优点**：数据完整性最高、合规友好、适合有资产交易的系统  
**缺点**：数据只增不减（库会越来越大）、需要冷热分离架构、实现复杂

---

### 4. 活动策划公司方案

**核心理念：活动生命周期管理 + 按活动隔离清理**

- **活动独立性**：每个活动有独立的数据域，活动结束后可以独立归档/清理
- **生命周期**：draft → active → paused → ended → archived → deleted
- **按活动清理**：不是清全库，而是清某个活动的所有关联数据（抽奖记录、奖品发放、参与者数据）
- **模板复用**：清理数据不清配置模板，下次活动可以复制模板

**优点**：颗粒度细、不影响其他活动  
**缺点**：跨活动数据（如用户资产总量）不好处理

---

### 5. 小公司"正规路线"方案

**核心理念：简单分层 + 脚本驱动 + 管理后台按钮**

- **两档清理**：保守（只清业务数据）和激进（清一切重建）
- **脚本封装**：把 SQL 脚本封装成 Node.js CLI 工具
- **管理后台入口**：在管理后台增加"数据管理"页面，提供可视化操作
- **预览模式**：先显示影响范围，确认后执行
- **基础审计**：记录谁在什么时间做了什么清理操作

**优点**：开发成本低、够用、灵活  
**缺点**：安全性依赖人的自觉、无回滚能力、生产环境高风险

## 三、本项目推荐方案：混合方案

### 方案选型依据

| 本项目特征 | 对应方案侧重 |
|-----------|-------------|
| 有复式记账（AssetTransaction + ItemLedger） | → 交易平台方案的"对账驱动"理念 |
| 有活动生命周期（draft/active/paused/ended） | → 活动策划方案的"按活动清理" |

| 愿意一次性投入、追求低长期维护成本 | → 不适合小公司的"凑合用"方案 |

**推荐方案：游戏公司清档思维 + 交易平台数据分级 + 大厂的审计能力**

## 八、后端 Service 设计（对齐 ServiceManager 注册体系）

### DataManagementService

文件位置：`services/DataManagementService.js`（放在 `services/` 根目录，与 `AuditLogService.js`、`IdempotencyService.js` 同级）

**ServiceManager 注册**（在 `services/index.js` 的 `initializeServices(models)` 中新增）：

```javascript
serviceManager.registerService('data_management', new DataManagementService(models))
```

**路由中获取**（对齐 orphan-frozen 模式）：

```javascript
const dataManagementService = req.app.locals.services.getService('data_management')
```

**核心职责**：

1. **数据量统计**（`getStats()`）
   - 对各表执行 `SELECT COUNT(*) FROM table`（使用 `sequelize.query()` 原生查询）
   - 按安全等级分组汇总
   - 计算数据库总大小

2. **影响预览**（`previewCleanup(options)`）
   - 根据选定的清理范围和时间条件
   - 计算各表将删除的行数
   - 分析 FK CASCADE 级联影响
   - 检测 RESTRICT 约束冲突
   - 生成 preview_token（5 分钟有效，存 Redis 或内存 Map）

3. **清理执行**（`executeCleanup(options)`）
   - 验证 preview_token 有效性
   - L3 数据：直接 `DELETE WHERE created_at < ? LIMIT 1000` 分批删除
   - L2 数据：先检查状态（不删进行中的），再分批删除
   - L1 数据：先运行对账校验，通过后才允许
   - L0 数据：硬编码拒绝
   - 按 FK 拓扑排序决定删除顺序（先子表后父表）
   - 写入审计日志：`AuditLogService.logOperation({ operation_type: 'data_cleanup', target_type: 'system', ... })`

4. **对账校验**（不新建，复用已有）
   - **已有实现**：`routes/v4/console/reconciliation.js` + `ItemLifecycleService`
   - 资产域：`ItemLifecycleService.reconcileAssets()` — SUM(delta_amount) GROUP BY asset_code 全部为 0
   - 物品域：`ItemLifecycleService.reconcileItems()` — SUM(delta) GROUP BY item_id 全部为 0
   - **纠正**：前版文档误写为 `BalanceService.getBalance()`，实际对账逻辑在 `ItemLifecycleService`（ServiceManager key: `asset_item_lifecycle`）
   - DataManagementService 在 L1 清理前调用 `ItemLifecycleService.reconcileAssets()` / `reconcileItems()` 做前置校验即可

**审计日志集成**（需扩展 `constants/AuditOperationTypes.js`）：

```
class DataCleanupJob {

}
```

- 每天 3:00 北京时间执行
- `UnifiedDistributedLock` 防止并发执行
- 读取 `SystemConfig.getValue('data_cleanup_policies')` 策略配置
- 按策略逐表清理，分批删除（每批 `batch_size` 行，间隔 100ms）
- 执行结果写入审计日志：`AuditLogService.logOperation({ operation_type: 'data_cleanup', ... })`
- 超过 `max_execution_time_seconds` 则中止当前表，下次继续
- 使用 `logger.info` 记录 `deleted_count` 和 `elapsed_ms`

## 九、FK 约束处理策略

### 数据库实际 FK 规则分布

| 规则 | 约束数 | 含义 |
|------|--------|------|
| CASCADE | 34 条 | 父表删除时子表自动删除 |
| RESTRICT / NO ACTION | 78 条 | 父表有引用时不允许删除 |
| SET NULL | 62 条 | 父表删除时子表外键置 NULL |

### 清理时的处理方式

**CASCADE 的表**（34 条约束）：只需删父表，子表自动清理。例如：

- 删除 `lottery_campaigns` → `lottery_alerts`, `lottery_daily_metrics`, `lottery_hourly_metrics` 等自动级联删除
- 删除 `users` → `authentication_sessions`, `admin_notifications`, `user_ad_tags` 等自动级联删除

**RESTRICT 的表**（78 条约束）：必须按依赖拓扑排序，先删子表再删父表。例如：

- 删除 `market_listings` 之前必须先删除 `trade_orders`（trade_orders.market_listing_id → market_listings RESTRICT）
- 删除 `items` 之前必须先删除 `item_ledger`, `item_holds`, `market_listings`, `redemption_orders`

**SET NULL 的表**（62 条约束）：删父表后子表外键变 NULL，需要评估是否需要一起清理。例如：

- 删除 `users` 后 `consumption_records.reviewed_by` 变 NULL（可接受）
- 删除 `lottery_prizes` 后 `lottery_draws.lottery_prize_id` 变 NULL（需要一起清理）

## 十、安全保障设计

| 安全机制 | 实现方式 |
|---------|---------|
| **权限控制** | 仅 admin（`role_level >= 100`）可访问，使用 `authenticateToken` + `requireRoleLevel(100)`（来自 `middleware/auth.js`） |
| **二次确认** | 手动清理需要输入"确认删除"文字 + 再次输入管理员验证码 |
| **预览必先** | execute 接口检查最近 5 分钟内是否有对应的 preview_token，没有则拒绝 |
| **维护模式** | 清档模式自动进入维护模式（`system_settings.maintenance_mode = true`），阻止用户端访问 |
| **速率限制** | 每小时最多执行 1 次手动清理（通过 `api_idempotency_requests` 控制） |
| **审计日志** | 所有清理操作记录到 `admin_operation_logs`，包含操作人、操作时间、各表删除行数 |
| **L0 硬拒绝** | Service 层白名单机制，L0 表永远不会出现在可清理列表中 |
| **生产环境识别** | `NODE_ENV === 'production'` 时禁用"上线前清档"模式 |
| **分批执行** | 所有删除操作分批执行（每批 1000 行），避免长事务锁表 |
| **超时中止** | 单次清理操作超过 300 秒自动中止，防止阻塞数据库 |

## 十七、需要拍板的决策项（含行业对比分析）

**决策前提**：项目未上线、愿意一次性投入、不兼容旧接口、追求长期维护成本低和技术债务少。

---

### 决策 1：清档模式的环境限制（高优先级）

**背景**：文档设计了"上线前清档"模式（模式一），仅在 `NODE_ENV !== 'production'` 时可用。当前 `.env` 中 `NODE_ENV=development`，上线后改为 `production` 即可自动禁用。

**各行业怎么做**：

| 行业 | 做法 | 原理 |
|------|------|------|
| **大厂（美团/腾讯/阿里）** | 环境变量 + 配置中心 + 审批工单 三重控制。清档功能只编译到测试环境镜像，生产镜像物理上不含清档代码 | 编译级隔离最彻底，但需要 CI/CD 分支构建 |
| **游戏公司（米哈游/网易）** | GM 工具按服务器类型判断：`server_type === 'test_server'`，硬编码在配置中。生产服的 GM 工具面板直接不显示清档按钮 | 服务器类型比环境变量更难篡改 |
| **交易平台（Steam/得物）** | 没有清档功能。测试环境用独立数据库实例，清档 = 直接 DROP DATABASE + 重建 | 环境完全隔离，不需要控制开关 |
| **活动策划公司** | 活动状态判断：只有 `status = 'draft'` 或 `status = 'ended'` 的活动数据才能清理 | 按业务状态控制，而不是环境变量 |
| **小公司** | 只有 `NODE_ENV` 判断，或者根本没有控制 | 最简单但最危险 |

**选项**：

| 选项 | 方案 | 开发成本 | 安全性 | 长期维护 |
|------|------|---------|--------|---------|
| A | 仅 `NODE_ENV !== 'production'` | 零 | 中（env 可以被误配） | 低 |
| B | 仅 `feature_flags` 表开关 | 0.5h | 中（数据库可被手动改） | 低 |
| C | NODE_ENV **且** feature_flag 双重 | 0.5h | 高（两道锁同时打开才行） | 低 |

**本项目适配分析**：

- 项目已有 `feature_flags` 表（7 条记录，全是抽奖策略开关），机制现成
- 项目已有 `NODE_ENV` 判断（`middleware/maintenanceMode.js` 等多处使用）
- 不需要大厂的编译级隔离（单体项目，不走 CI/CD 分支构建）
- 不需要游戏公司的服务器类型概念（没有多服架构）

**结论：选 C**。零额外学习成本，复用已有 `feature_flags` 表 + `NODE_ENV` 判断。上线后自动失效，日后测试环境需要清档时只需开 flag 即可。

### 决策 2：L1 金融数据清理是否需要归档（高优先级）

**背景**：L1 数据（`asset_transactions` 53,889 行 / `item_ledger` 18,068 行 / `account_asset_balances` 83 行）涉及复式记账。当前全部是测试数据。

**各行业怎么做**：

| 行业 | 上线前测试数据 | 上线后正式数据 | 原理 |
|------|-------------|-------------|------|
| **大厂** | 测试环境独立数据库，上线时不存在"清理"问题——因为生产库是空的 | 永不删除，只归档到 OSS/S3 冷存储 | 四套环境完全隔离 |
| **游戏公司** | 清档 = TRUNCATE，不归档（测试数据无价值） | 玩家资产数据永不删除；过期道具标记为 expired 不物理删 | 测试期数据和正式数据完全分开对待 |
| **交易平台** | 同大厂，测试库和生产库物理隔离 | 所有 transaction 级数据法定保留 5-10 年，只做冷热分离 | 监管合规要求 |
| **活动策划公司** | 活动结束后整体归档到备份库 | 按活动生命周期归档 | 按活动隔离 |
| **小公司** | 直接清空，不归档 | 通常也不归档（技术债） | 图省事 |

**选项**：

| 选项 | 方案 | 开发成本 | 数据安全 | 长期维护 |
|------|------|---------|---------|---------|
| A | 上线前清档 L1 直接 TRUNCATE | 零 | 测试数据丢失（无价值） | 零 |
| B | 上线前也归档到 `_archive` 后缀表 | 1-2天 | 留痕但无实际价值 | 需维护归档表 |
| C | 上线前直接清；上线后 L1 永久保留（不删只归档） | A 的成本 | 正式数据永久安全 | 未来需冷热分离 |

**本项目适配分析**：

- 项目使用复式记账（`AssetTransaction` + `ItemLedger`），和得物/闲鱼同类——上线后的金融数据确实不能删
- 但当前 317MB 数据库全是测试数据，归档测试数据等于归档垃圾
- 项目用的是 Sealos 云平台 MySQL，没有 OSS 冷存储——上线后的"归档"第一步只需要分库分表或 `_archive` 表，不需要对象存储
- 游戏公司方案最贴合：测试期无情 TRUNCATE，正式期严格保护

**结论：选 C**。和游戏公司 + 交易平台的混合做法一致。上线前清档直接 TRUNCATE（测试数据无保留价值），上线后 L1 数据永远不删除，只做 L2/L3 级别的日常清理。未来数据量大了再考虑冷热分离。

### 决策 3：清档时是否保留管理员账号（中优先级）

**背景**：清档会清空 `users` 表中的 77 行测试用户，但需要保留哪些管理员。`accounts` 表有 **7 个** `account_type='system'` 的系统账户（account_id: 1,2,3,4,12,15,239），这些是 `BalanceService` 运行的前提。

**各行业怎么做**：

| 行业 | 做法 | 原理 |
|------|------|------|
| **大厂** | 生产库初始化时通过 seed 脚本创建超管+系统账户，不存在"保留"问题 | 环境隔离，每个环境独立初始化 |
| **游戏公司** | 清档保留 GM 账号列表（硬编码或配置文件中的 GM ID 列表），其余全删。系统 NPC 账户通过 seed 重建 | GM 白名单是标准做法 |
| **交易平台** | 不做全库清档，不涉及此问题 | — |
| **活动策划公司** | 保留所有内部员工账号（按 `is_internal = true` 标记），清理外部用户 | 内部/外部标记 |
| **小公司** | 全删后 seed 重建，或者保留某些 user_id 硬编码 | 图省事 |

**选项**：

| 选项 | 方案 | 开发成本 | 灵活性 | 风险 |
|------|------|---------|--------|------|
| A | 保留所有 `role_level >= 100` 的管理员 | 低 | 低（未来可能有测试管理员需要清理） | 可能保留不需要的测试管理员 |
| B | 手机号白名单（UI 输入） | 中 | 高（精确控制保留谁） | 白名单输错会误删 |
| C | 全部清空 + seed 重建超管 | 低 | 低（只有一个超管） | 需要重新登录，关联数据丢失 |

**本项目适配分析**：

- 7 个系统账户（`accounts` 表中 `account_type = 'system'`，account_id: 1,2,3,4,12,15,239）无论哪个方案都必须保留，否则 `BalanceService` 直接报错
- `roles` 表 11 行是 L0 不可删，不受影响
- 真正需要决策的是：77 个 users 中保留谁
- 游戏公司的 GM 白名单最贴合——项目未上线、用户都是测试用户、只有少数管理员是真实需要保留的

**结论：选 C（全部清空 + seed 重建）**。理由：

1. 77 个用户全是测试数据，没有哪个"管理员"的历史操作记录有保留价值
2. 清档本身就是"从零开始"，保留旧管理员的 user_id 和关联数据反而是技术债
3. seed 脚本重建超管 = 游戏公司的"开服初始化"，干净彻底
4. 7 个系统账户由 `BalanceService` 初始化逻辑或 seed 脚本重建（它们属于 L0 不可删，清档逻辑中硬编码跳过 `account_type = 'system'` 的行）
5. 长期维护成本最低：不需要实现白名单 UI、不需要处理管理员关联数据的级联保留逻辑

**但如果你确实有 1-2 个管理员账号上线后会继续使用、且不想重新注册**，那就改选 B。

#### 清档后超管重建方案：各行业做法与本项目方案

**问题**：清档后 `users` 表是空的，没有管理员，如何创建第一个超管？

**各行业怎么做**：

| 行业 | 做法 | 超管信息存在哪 | 自动化程度 | 安全性 |
|------|------|-------------|-----------|--------|
| **大厂（美团/腾讯/阿里）** | CI/CD 部署流水线自动执行 seed 脚本，超管信息从配置中心（Apollo/Nacos）注入，不硬编码 | 配置中心（与代码分离） | 全自动 | 最高 |
| **游戏公司（米哈游/网易/暴雪）** | "开服初始化脚本"直接 INSERT GM 账号，是清档流程的最后一步，硬编码或读配置文件 | 初始化脚本 / 配置文件 | 全自动 | 高 |
| **交易平台（Steam/得物/闲鱼）** | 和大厂一样靠 CI/CD seed；交易平台不做全库清档，此问题基本不出现 | CI/CD | 全自动 | 最高 |
| **活动策划公司** | 每次新活动部署时 seed 脚本创建运营账号，和游戏公司类似 | seed 脚本 | 全自动 | 中 |
| **小公司** | 先用手机号注册普通用户，再手动连数据库 `UPDATE role_level = 100` | 人脑记忆 | 纯手动 | 最低 |

**全行业共识**：除了小公司以外，所有方案都是 **脚本自动创建，不手动改库**。

**本项目方案：游戏公司模式 + `.env` 配置**（最适合本项目技术栈）

本项目已有 `.env` 配置体系 + `npm run db:seed` 体系 + `UserService.registerUser()` 方法，直接组合使用：

**第 1 步：`.env` 新增超管手机号配置**

```
# 清档后自动重建的超管手机号
SUPER_ADMIN_MOBILE=13612227910
```

**第 2 步：清档流程最后一步自动执行超管重建**

`DataManagementService.executeCleanup()` 在清档模式（模式一）执行完所有 TRUNCATE 后，自动执行：

```
1. 读取 process.env.SUPER_ADMIN_MOBILE
2. 调用 UserService.registerUser(mobile, { transaction }) 创建用户（自动创建 users + accounts 记录）
3. 在 user_roles 表关联 role_id=2（admin, role_level=100）—— roles 表是 L0 不会被删，role_id=2 始终存在
4. 7 个 account_type='system' 的系统账户由 BalanceService 初始化逻辑或 seed 脚本重建
```

**第 3 步：清档完成，直接登录**

用 `SUPER_ADMIN_MOBILE` 配置的手机号 + 验证码（开发环境万能码 `123456`）登录管理后台，即为 `role_level=100` 的超管。

**不需要手动连数据库改任何东西。**

**为什么不用小公司的"先注册再手动提权"方式**：

1. 手动操作 = 不可重复、不可审计、容易出错
2. 清档可能执行多次（测试阶段反复清），每次都要手动改库不现实
3. 如果有多人协作，"超管手机号是什么"只存在某个人脑子里，不是代码化的
4. `.env` 方案把超管信息代码化了，换人部署也能正确重建

**已拍板**：`SUPER_ADMIN_MOBILE=13612227910`。

### 决策 4：对账校验不通过时的处理策略（中优先级）

**背景**：L1 数据清理前需要运行对账校验（`account_asset_balances.available_amount` vs `SUM(asset_transactions.delta_amount)`）。如果对账发现差异，如何处理？

**各行业怎么做**：

| 行业 | 对账不通过时 | 原理 |
|------|------------|------|
| **大厂** | 全部暂停 + 人工介入 + 生成工单。不允许任何自动处理 | 金融合规要求，差异必须人工确认 |
| **游戏公司** | 对于测试服清档：忽略差异直接清（反正是测试数据）。对于生产服：不存在清档场景 | 测试期无所谓，正式期根本不做 |
| **交易平台** | 差异 = 严重事故。立即锁定相关账户，启动差异溯源流程，找到差异原因后才解除 | 交易平台的对账差异可能意味着资金损失 |
| **活动策划公司** | 差异报告展示，运营决定是否继续。通常选择"先修复差异再清理" | 活动数据丢几条不影响大局 |
| **小公司** | 通常没有对账机制 | — |

**选项**：

| 选项 | 方案 | 安全性 | 灵活性 | 开发成本 |
|------|------|--------|--------|---------|
| A | 对账不通过 → 阻止全部清理 | 最高 | 最低（L2/L3 被无辜阻塞） | 最低 |
| B | 对账不通过 → 仅阻止 L1 清理，L2/L3 继续 | 高 | 高（各级别独立） | 中 |
| C | 对账不通过 → 展示差异 + 允许"强制清理" | 中（依赖人的判断） | 最高 | 最高 |

**本项目适配分析**：

- 当前全是测试数据，对账差异几乎必然存在（测试过程中可能有不完整的事务）
- 上线前清档场景：对账差异不重要（游戏公司思维——测试数据全清）
- 上线后运营场景：只做 L2/L3 清理（不碰 L1），对账机制是防御性的
- L2 数据（抽奖记录、客服记录、通知等）和 L1 记账完全无关，不应被 L1 差异阻塞

**结论：选 B**，但需要区分两个场景：

| 场景 | 对账不通过时的行为 |
|------|----------------|
| **上线前清档（模式一）** | 显示差异警告，但允许继续（因为测试数据全清，差异无意义） |
| **上线后日常清理（模式二/三）** | L1 清理被阻止 + 生成差异报告；L2/L3 清理不受影响 |

这是游戏公司（测试期宽松）+ 交易平台（正式期严格）的混合策略，最贴合项目"未上线但未来要上线"的阶段特征。

### 决策 5：自动清理定时任务的默认策略（低优先级）

**背景**：自动清理 Job 需要对 L3 级别表设定保留天数，超期自动删除。

**各行业怎么做**：

| 行业 | 幂等请求记录 | 操作日志 | 会话数据 | 广告日志 | 原理 |
|------|------------|---------|---------|---------|------|
| **大厂** | 3-7 天 | 180-365 天（审计合规） | 7-30 天 | 90-365 天 | 日志保留受合规驱动 |
| **游戏公司** | 1-3 天（请求量极大） | 90 天 | 7 天 | 不适用 | 游戏请求量大，清得更激进 |
| **交易平台** | 7-14 天 | 365 天（法定要求） | 30 天 | 90 天 | 交易记录法定保留 |
| **活动策划公司** | 7 天 | 90 天 | 7 天 | 30-90 天 | 活动结束后快速清理 |
| **小公司** | 不清理（直到磁盘满） | 不清理 | 不清理 | 不清理 | 没有清理机制 |

**本项目当前数据增长速度**（基于真实数据库推算）：

| 表 | 当前行数 | 日均增量（估算） | 不清理 1 年后 | 保留 N 天后稳态 |
|----|---------|---------------|-------------|--------------|
| `api_idempotency_requests` | 186,214 | ~500-800/天 | ~47 万行 | 7 天 ≈ 5,600 行 |
| `websocket_startup_logs` | 2,576 | ~10-20/天 | ~1 万行 | 7 天 ≈ 140 行 |
| `authentication_sessions` | 83 | ~3-5/天 | ~2,000 行 | 30 天 ≈ 150 行 |
| `admin_operation_logs` | 6,651 | ~20-30/天 | ~1.7 万行 | 90 天 ≈ 2,700 行 |
| 广告日志（6 表） | ~280 | ~5/天 | ~2,000 行 | 90 天 ≈ 450 行 |
| `reminder_history` | 745 | ~3/天 | ~1,800 行 | 90 天 ≈ 270 行 |

**选项**：

| 选项 | 方案 | 说明 |
|------|------|------|
| A | 直接采用文档默认值 | 幂等 7 天 / WS 日志 7 天 / 会话 30 天 / 操作日志 90 天 / 广告 90 天 |
| B | 全部改为更保守的值 | 幂等 14 天 / 操作日志 180 天 / 其余不变 |
| C | 全部改为更激进的值 | 幂等 3 天 / WS 日志 3 天 / 操作日志 60 天 |

**本项目适配分析**：

- `api_idempotency_requests` 是最大的表（186,214 行 / 208 MB，占数据库 65%），7 天保留已经足够——这个表的唯一用途是防止重复请求，7 天后的记录没有任何业务价值
- `admin_operation_logs` 是审计日志，90 天在行业中偏短——但项目未上线、没有合规要求，90 天够用。上线后如果有审计需求可以通过 UI 调到 180 天
- 所有保留天数都存在 `system_configs` 表中，管理员可在 UI 随时调整——所以默认值不需要纠结

**结论：选 A（直接采用默认值）**。

| 表 | 保留天数 | 理由 |
|----|---------|------|
| `api_idempotency_requests` | **7 天** | 行业标准 3-14 天，7 天平衡空间占用和安全余量 |
| `websocket_startup_logs` | **7 天** | 纯调试日志，无业务价值 |
| `authentication_sessions` | **30 天** | 登录会话，30 天 = 一个月内可追溯异常登录 |
| `admin_operation_logs` | **90 天** | 行业标准 90-365 天，先 90 天起步，上线后按需调高 |
| 广告日志（6 表） | **90 天** | 广告效果分析需要 30-90 天数据，90 天足够 |
| `reminder_history` | **90 天** | 提醒记录回溯，90 天合理 |

上线后通过管理后台 UI（Tab 2 自动清理策略）随时可调，无需改代码。

### 决策总结

| 决策 | 最终推荐 | 行业对标 | 技术实现依赖 |
|------|---------|---------|-------------|
| **决策 1**：环境限制 | **C：NODE_ENV + feature_flag 双重** | 游戏公司 GM 工具 + 大厂配置中心的简化版 | 复用已有 `feature_flags` 表 |
| **决策 2**：L1 归档 | **C：测试数据直接清，上线后永久保留** | 游戏公司清档 + 交易平台数据永不删 | 清档时 TRUNCATE；上线后 L1 不纳入可清理范围 |
| **决策 3**：保留管理员 | **C：全部清空 + seed 重建超管** | 游戏公司开服初始化 | 复用 `db:seed` 脚本 + L0 系统账户保护 |
| **决策 4**：对账不通过 | **B：仅阻止 L1，L2/L3 继续** | 游戏公司（测试宽松）+ 交易平台（正式严格） | 清档模式忽略差异，运营模式分级阻止 |
| **决策 5**：保留天数 | **A：直接采用默认值** | 行业中位数水平 | 存 `system_configs`，UI 可调 |
| **决策 6**：Legacy 遗留表 | **C：DROP TABLE + migration** | 大厂/游戏/交易平台通用做法 | 1 个 migration 文件，代码零引用已验证 |

**整体方案定位**：不是照搬任何一个行业方案，而是取每个行业最适合本项目的部分：

- 从**游戏公司**取：清档思维（测试数据无情清空、开服初始化、GM 工具的环境隔离）
- 从**交易平台**取：金融数据保护（复式记账数据永不删、对账驱动、数据分级）
- 从**大厂**取：审计能力（操作日志、二次确认、预览机制）
- 从**活动策划**取：不取（本项目不需要按活动隔离清理，一个 `lottery_campaign_id` 过滤即可）
- 从**小公司**取：不取（拒绝脚本散落、无审计、无预览的"凑合用"模式）

这个混合方案和本项目后端技术栈的匹配度：

| 需要的能力 | 后端已有 | 是否需要新建 |
|-----------|---------|------------|
| 环境变量判断 | `NODE_ENV` 多处使用 | 否 |
| Feature Flag | `feature_flags` 表 + 7 条记录 | 否（新增 1 条记录） |
| 对账校验 | `ItemLifecycleService.reconcileItems()` / `reconcileAssets()` + 已有 `/api/v4/console/reconciliation/*` 路由 | 否（直接复用） |
| 审计日志 | `AuditLogService.logOperation()` | 否（新增操作类型） |
| 维护模式 | `maintenanceMode.js` + `AdminSystemService` | 否 |
| 定时清理 | `node-cron` + `scheduled_tasks.js` + 21 个 Job | 否（新增 1 个 Job） |
| 配置存储 | `SystemConfig.getValue()` / `upsert()` | 否（新增 1 条配置） |
| 分批删除 | `cleanup_expired_idempotency.js` 的模式 | 否（参照实现） |
| **核心 Service** | **无** | **是（唯一需要新建的核心模块）** |

### 各行业怎么处理数据库迁移遗留表

#### 大厂方案（美团 / 腾讯 / 阿里巴巴）

**核心做法：DDL 变更审批 + 冷却期 DROP**

- **迁移完成后进入"冷却观察期"**（通常 30-90 天），期间旧表只读不写，监控是否有遗漏的慢查询或离线任务还在读旧表
- 冷却期结束后由 DBA 审批 `DROP TABLE`，**从不留着旧表不管**
- 大厂有专门的"Schema 变更工单系统"（如美团的 Cetus、阿里的 DMS），每次 DDL 变更都有审批链和回滚方案
- 如果旧表数据有分析价值，先 `mysqldump` 导出到数据仓库（Hive / ClickHouse），再 DROP
- **原则**：生产库只保留活跃表，dead table = tech debt，DBA 有 KPI 考核消除 dead table

**优点**：最干净，零技术债，DBA 长期维护成本为零
**缺点**：需要 DBA 角色、变更审批流程、冷却期等基础设施

---

#### 游戏公司方案（米哈游 / 网易 / 暴雪）

**核心做法：版本迭代时一刀切 DROP**

- 游戏每次大版本更新（如 2.0 → 3.0）时有"停机维护"窗口，在维护期间执行 schema 变更
- **旧版表直接 DROP，不保留**。测试服先验证一轮，确认无引用后，生产服维护时执行
- 游戏公司不像金融系统需要保留历史数据——旧版道具系统的数据已经迁移到新表，旧表只是躯壳
- 如果旧版 schema 和新版完全不兼容（字段重构），旧表数据在迁移脚本中已经转移完毕，DROP 是迁移脚本的最后一步
- **标准流程**：`CREATE new_table` → `INSERT INTO new_table SELECT ... FROM old_table`（数据迁移） → 验证 → `DROP old_table`，三步是原子性的
- 有些游戏公司会 `RENAME old_table TO old_table_backup_20260309`，保留 7 天后自动 DROP（cron job）

**优点**：概念简单，"迁移 = 搬家 + 拆旧房"
**缺点**：停机维护窗口，不适合 7×24 在线服务

---

#### 虚拟物品交易 / 小众二手平台方案（Steam / 得物 / 闲鱼）

**核心做法：影子表 + 双写验证 + 延迟 DROP**

- 交易平台对数据一致性要求极高，迁移通常采用"影子表 + 双写"（shadow table + double write）：旧表和新表同时写入，验证数据一致后切流到新表
- 切流完成后旧表进入"只读观察期"（7-14 天），确认无残留读请求
- 观察期结束后 `DROP TABLE`，**不保留**
- 如果是金融相关的旧表（交易流水），先归档到 `_archive` 库，再从在线库 DROP
- **核心约束**：在线库 `SHOW TABLES` 返回的每一张表都必须有对应的 Model 和 Service，否则视为"僵尸表"（zombie table），纳入技术债清理指标

**优点**：数据安全，验证充分
**缺点**：双写期间运维复杂，小团队做不起

---

#### 活动策划公司方案

**核心做法：活动结束 = 全部清理**

- 活动策划公司的系统生命周期短（一个活动几周到几个月），迭代快
- 旧版表通常是"上一个活动的数据结构"，新活动上线时直接 `DROP DATABASE` 重建
- 不存在"legacy 表"概念——每次活动都是独立的数据库或独立的表前缀
- **原则**：不值得维护旧结构，推倒重来最快

**优点**：零技术债（因为不积累）
**缺点**：不适合长期运营的产品

---

#### 小公司方案

**核心做法：不管它，留着吧**

- 小公司通常没有 DBA，开发人员做完迁移后，旧表就留在数据库里了
- 理由："万一以后要回滚呢""数据不大又不影响性能""改了怕出事"
- 结果：数据库里积累大量僵尸表，新人入职看到 `table_old`、`table_backup`、`table_v2`、`table_legacy` 一脸懵
- **典型技术债演变**：
  - 第 1 年：2 张 legacy 表，3 MB，无所谓
  - 第 3 年：15 张 legacy 表，200 MB，`SHOW TABLES` 输出半屏都是废表，新人不知道哪些是活的
  - 第 5 年：开始有人误写查询到 legacy 表，生产出 bug

**优点**：零短期成本
**缺点**：长期技术债指数级增长，是"温水煮青蛙"式的技术腐烂

| 维度 | 大厂方案 | 游戏公司方案 | 交易平台方案 | 活动策划方案 | 小公司方案 |
|------|---------|-------------|-------------|-------------|-----------|

### 本项目适配分析

| 本项目特征 | 对应分析 |
|-----------|---------|
| 项目未上线 | 无"冷却观察期"需求——根本没有线上流量在读旧表 |
| 全是测试数据 | 无归档价值——legacy 表里的数据是 V3→V4 迁移的测试残留 |
| 代码零引用 | `models/` + `services/` + `routes/` 中 **0 个文件**引用 legacy 表 |
| FK 入向零引用 | 没有任何其他表的外键指向 legacy 表 |
| FK 出向 1 条 | `item_instances_legacy.owner_user_id` → `users.user_id`（清档时 users 也会清，TRUNCATE 顺序：先 legacy 后 users 即可） |
| 愿意一次性投入 | 选最彻底的方案，不留债 |
| 追求低长期维护成本 | 小公司"不管"方案是最差选择 |
| Sequelize 迁移体系 | `sequelizemeta` 记录迁移历史，DROP TABLE 需要新 migration 文件 |

**选项（更新）**：

| 选项 | 方案 | 开发成本 | 长期维护成本 | 行业对标 |
|------|------|---------|-----------|---------|
| A | 清档时 TRUNCATE（清数据保表结构） | 零（DataManagementService 清理列表加两行） | **低**——空表占用 ~0.04 MB，但 `SHOW TABLES` 仍会显示 | 无行业对标，折中方案 |
| B | 保留不动 | 零 | **递增**——小公司典型技术债模式，3 MB 不大但僵尸表永存 | 小公司方案（最不推荐） |
| C | 清档后 DROP TABLE + migration | **0.5h**（1 个 migration 文件 + 从 baseline squash 中移除注释） | **零**——表彻底消失，数据库最干净 | 大厂 / 游戏公司 / 交易平台通用做法 |

**结论：改选 C（DROP TABLE）**。

理由（基于实际验证结果）：

1. **代码零引用已确认**——models / services / routes 中没有任何文件使用这两张表，不存在"万一有引用"的风险
2. **FK 零入向引用已确认**——没有任何表的外键指向 legacy 表，DROP 不会破坏任何约束
3. **项目未上线**——不需要冷却观察期，不需要双写验证，游戏公司的"一刀切 DROP"策略完全适用
4. **全行业共识**——除了小公司以外，大厂 / 游戏 / 交易平台都不会在生产库保留迁移后的旧表
5. **0.5h 的一次性投入 vs 永久的认知负担**——新人看到 108 张表里有 2 张 legacy 后缀的表，需要花时间确认"这是什么？能不能用？"。DROP 后只剩 106 张，每张都是活的
6. **migration 体系天然支持**——项目用 Sequelize CLI + `sequelizemeta`，新建一个 migration 文件 `DROP TABLE IF EXISTS item_instances_legacy` + `DROP TABLE IF EXISTS item_instance_events_legacy`，回滚时用 baseline squash 的 CREATE TABLE 语句即可

**执行步骤**：

```
1. 新建 migration：migrations/YYYYMMDDHHMMSS-drop-legacy-item-tables.js
   up(): DROP TABLE IF EXISTS item_instance_events_legacy; DROP TABLE IF EXISTS item_instances_legacy;
   down(): 从 baseline squash 复制 CREATE TABLE 语句（可回滚）
2. 在清档流程的 Phase 4 验证中运行此 migration
3. npm run db:migrate
4. 验证：SHOW TABLES LIKE '%legacy%' 返回空
```

**注意事项**：DROP 顺序必须先 `item_instance_events_legacy`（无 FK 出向），后 `item_instances_legacy`（FK 出向引用 users 表，但 DROP TABLE 会自动移除自身的 FK）。

### Web 前端直接使用后端字段名（不做映射）

本方案遵循"后端为准，前端适配"原则：

- 后端 API 响应字段名即最终字段名，前端 **直接使用**，不做 camelCase ↔ snake_case 映射
- 例如：`total_rows_to_delete`、`safety_level`、`cascade_effects`、`preview_token` 等字段，前端 Alpine.js 模板中直接 `x-text="item.total_rows_to_delete"` 使用
- 后端 `res.apiSuccess()` 返回的 `success` / `code` / `message` / `data` / `timestamp` / `version` / `request_id` 结构即标准契约，前端 `request()` 函数已适配此格式
