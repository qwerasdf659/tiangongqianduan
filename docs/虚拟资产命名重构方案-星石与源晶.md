## 一、行业方案对比总览

### 1.1 大厂方案

| 公司 | 资产体系 | 表结构策略 | code 与 display_name | 扩展方式 | 平台名称 |
|------|---------|-----------|---------------------|---------|---------|
| **美团** | 积分 / 红包 / 优惠券统一管理 | **单表 + asset_type_code** | 分离，code 不可变 | 配置驱动，加行不改表 | 万象平台 |
| **腾讯** | Q币 → Q点 → 游戏货币，多层级 | 联邦式单表（每应用一套，API 层统一） | app_id + currency_code 复合键 | Midas SDK 框架扩展 | 米大师 |
| **阿里** | 淘金币 / 天猫积分 / 88VIP / 蚂蚁积分 | **单表 + asset_code** | 分离，code 不可变，域名前缀 | 配置驱动，资产中台 | 资产中台 |
| **京东** | 京豆 / 京券 / E卡 / 红包 | **单表 + asset_type_code** | 分离，UPPER_SNAKE_CASE | 配置驱动，幂等事务 | 虚拟资产中台 |
| **拼多多** | 多多币 / 现金余额 / 实物权益 | 混合（可替代资产单表，不可替代资产独立表） | 规则引擎驱动 | 配置 + 规则引擎 | — |

**大厂共识：**
- code 是不可变契约，display_name 随时可改
- 单表 `(user_id, asset_type_code)` 是压倒性主流
- 新增资产类型 = 插入配置行，零 schema 变更

### 1.2 游戏公司方案

| 公司/游戏 | 货币层级 | code 设计 | 关键模式 |
|-----------|---------|----------|---------|
| **米哈游** (原神) | 创世结晶(付费) → 原石(混合) → 摩拉(软) | **整数 ID**（201/202/203），display_name 完全独立做 i18n | 付费→免费单向阀门 |
| **网易** (梦幻西游) | 仙玉(付费) → 金币(软) → 绑定金币(不可交易) | code 区分绑定/非绑定变体 | 绑定 vs 非绑定同名不同 code |
| **暴雪** (守望先锋) | Legacy Credits(旧) + Overwatch Coins(新) | 旧 code 保留不删，新 code 另建 | **改名≠改code**，旧货币降级保留 |
| **Epic** (堡垒之夜) | V-Bucks（跨平台分钱包） | 按平台拆分钱包账本 | 付费/免费分离结算 |
| **Roblox** | 曾双币(Tix+Robux)，2016年砍掉Tix | 合并为单一 Robux | 简化优于复杂 |
| **Steam** | 钱包 + 交易卡 + 宝石 | classid+instanceid(类型) / assetid(实例，交易后变) | 定义/实例分离 |

**游戏公司共识：**
- **三层标识**是黄金标准：数据库数字PK → 字符串code → display_name
- 重命名只改 display_name，**从不改 code**
- 货币废弃时：改名为"Legacy XXX"，缩小使用范围，另建新 code
- 软/硬货币严格分离，转换单向不可逆

|------|------|---------|---------|
| **Buff / C5Game** | 游戏物品交易 | 平台钱包(可替代) + 物品清单(不可替代) 严格分离 | 托管交易 + 双重记账 |

### 1.4 小公司 / 创业公司方案

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 单表 `user_balances(user_id, asset_type_code, balance)` | < 10 种资产，< 10万用户 | 简单，加类型无需改表 | 高并发下热表 |
| 多表（每种资产一张表） | 资产间业务逻辑差异极大 | 性能隔离 | 迁移多，代码重复 |
| **开源方案 medici** (Node.js) | 需要审计级别记账 | 真双重记账，层级账户 | MongoDB 依赖 |
| **开源方案 bavix/laravel-wallet** | PHP 生态 | 多态钱包，转账=提现+存入 | PHP 生态 |

**小公司共识：**
- 先用单表，遇到瓶颈再拆
- 乐观锁(version 字段)是标配
- 配置驱动优于硬编码

## 二、关键设计决策对比

### 2.1 asset_code 是否要改？

| 做法 | 谁这么干 | 理由 |
|------|---------|------|
| **只改 display_name，code 不动** | 美团、阿里、京东、米哈游、暴雪、Steam — **所有大厂和游戏公司** | code 是系统契约，改 code = 改接口，风险极高 |
| **code 也一起改** | 几乎没有公司这样做 | — |
| **废弃旧 code + 新建 code** | 暴雪(OW Credits→Legacy Credits + 新建 OW Coins)、Roblox(砍 Tix) | 仅限货币体系重构，不是"改名" |

> **行业结论：改 display_name 是唯一主流做法。但你的项目未上线，没有历史数据包袱。**

### 2.2 code 命名风格（大写 vs 小写）

#### 行业调研

| 阵营 | 代表 | DB 存储值示例 | 技术栈背景 |
|------|------|-------------|-----------|
| **UPPER_SNAKE_CASE** | Google Protobuf、PayPal、阿里Java开发手册、京东 | `GOLD_COIN`, `STAR_STONE` | 主要是 **Java 生态** |
| **lower_snake_case** | Stripe、GitHub API、Node.js/Sequelize 社区 | `gold_coin`, `star_stone` | 主要是 **Node.js/Ruby 生态** |
| **数字 ID** | 米哈游(201/202)、腾讯米大师、Steam | `201`, `202` | 游戏引擎 |
| **PascalCase** | Epic(Fortnite)、medici | `Currency:MtxPurchased` | 特殊场景 |

#### 关键发现：大写 vs 小写本质是技术栈之争

- **Java 世界**（阿里/京东/Google）：常量 = UPPER_SNAKE_CASE，这是 Alibaba Java 开发手册的强制规范，被国内大厂广泛采用
- **Node.js 世界**（Stripe/GitHub/Sequelize）：DB 值 = lower_snake_case，JS 代码中的常量键用 UPPER 但值用 lower
- **ISO 4217 货币码**：大写（`USD`, `CNY`），但 Stripe 故意用小写 `usd` 保持 JSON 风格一致性

#### 本项目现状（Node.js + Sequelize + MySQL）

扫描全部 50+ 个 Sequelize 模型，**所有 ENUM 值无一例外全是小写**：

```js
// 状态值
'active', 'inactive', 'pending', 'completed', 'cancelled', 'draft'
// 多词值
'pending_review', 'on_break', 'on_sale', 'auto_approve'
// 资产形态（form ENUM 同步改为 gem）
'shard', 'gem', 'currency', 'quota'
// 系统配置键
'basic/system_name', 'points/lottery_cost_points'
```

代码中的常量已经遵循 **键大写、值小写** 的 Node.js 惯例：

```js
const PRIORITY = { LOW: 'low', MEDIUM: 'medium', HIGH: 'high' }
const AGENT_STATUS = { ACTIVE: 'active', INACTIVE: 'inactive', ON_BREAK: 'on_break' }
```

唯独 `DIAMOND`、`POINTS`、`BUDGET_POINTS`、`DIAMOND_QUOTA` 这 4 个资产 code 是大写 — 这本身就是不一致。

#### 选型结论：lower_snake_case

| 维度 | 选大写 | 选小写 |
|------|-------|-------|
| 和项目现有 50+ 模型 ENUM 一致 | 要改 50+ 个模型 | **完全一致，零改动** |
| 和现有 12 个水晶 code 一致 | 要改 12 个 | **完全一致** |
| 和 Node.js/Stripe 生态一致 | 不一致 | **一致** |
| 和阿里/京东一致 | 一致 | 不一致（但他们是 Java） |
| 需要额外改的文件数 | 50+ 模型 + 全部服务 | **只改 4 个系统资产 code** |

**决定：全部统一为 lower_snake_case，跟着自己的技术栈走。**

### 2.3 code 中"源晶"的英文词选择

| 候选词 | code 示例 | 长度 | 评估 |
|--------|----------|------|------|
| `source` | `red_source_shard` | 17 | 易与 source code 混淆 |
| `origin` | `red_origin_shard` | 17 | 易与 git origin 混淆 |
| `core` | `red_core_shard` | 15 | **最短，无技术术语歧义** |
| `prime` | `red_prime_shard` | 16 | 游戏感强，但语义稍远 |

**决定：使用 `core`（核心/本源），最短且无歧义。**

### 2.4 完整形态用词：`gem` 替代 `crystal`

#### 行业调研

| 用词 | 谁在用 | 和电商撞车风险 |
|------|--------|--------------|
| **gem** | Steam（交易卡→宝石）、Diablo 全系列、Clash of Clans、WoW、Dota 2 — **游戏行业最广泛** | **低** — 电商商品 code 不会用 gem |
| crystal | Summoners War、王者荣耀 | **高** — 水晶饰品/水晶灯/施华洛世奇，电商大品类 |
| orb | Path of Exile、火焰纹章 | 低 — 但不够主流 |
| stone | Pokemon 进化石 | 中 — 且已用于 `star_stone` |

#### 关键发现

- 米哈游内部 code 叫 `Hcoin`/`Scoin`/`Mcoin`，玩家看到的"创世结晶"(crystal) 只是 display_name
- Diablo 的碎片→完整体层级：Chipped → Flawed → Normal → Flawless → **Perfect Gem**
- `shard → gem` 语义配对最自然：碎片(shard) → 完整宝石(gem)
- `gem` 3 个字母，比 `crystal` 短 4 个字符
- 电商商品 code 用具体矿物名（`amethyst`、`topaz`），不会用笼统的 `gem`

**决定：完整形态用 `gem` 替代 `crystal`。form ENUM 同步改为 `'shard'`, `'gem'`, `'currency'`, `'quota'`。**

### 2.4 表结构策略

| 策略 | 使用方 | 适用条件 |
|------|-------|---------|
| **单表 + code 区分** | 美团、阿里、京东 — **压倒性主流** | 资产类型 < 100，业务逻辑统一 |
| 联邦式 | 腾讯(每游戏一套) | 多应用多团队 |
| 混合 | 拼多多 | 可替代和不可替代资产差异大 |

## 四、已确认决策

### 前提条件

- 项目未上线，无历史数据包袱
- 愿意一次性投入成本
- 不需要兼容旧接口
- 目标：长期维护成本最低，技术债务最少

### 4.1 Q1: code + display_name 全改

**决定：code + display_name 全改，彻底改到位。使用 `core` 作为"源晶"的英文前缀。**

选择 `core` 而非 `source`/`origin` 的理由：
- `core` 最短（`red_core_shard` 15字符 vs `red_source_shard` 17字符）
- 语义清晰：core = 核心，和"源晶"的"本源"含义接近
- 不与技术术语撞车（`source` 易混淆 source code，`origin` 易混淆 git origin）

| 考量 | 分析 |
|------|------|
| 行业惯例 | 大厂不改 code，因为有海量历史数据。**但你没上线，这个约束不存在** |
| 长期维护 | 开发者看到 `DIAMOND` 会想到"钻石"而不是"星石"，心智负担永远在 |
| 技术债务 | 现在 code 和名字不一致 = 每个新人都要被教育一次"DIAMOND 其实是星石" |
| 成本 | 现在改：约 6-8 个文件，无数据迁移。上线后改：所有余额表、流水表、转换规则都要迁移 |

### 4.2 Q2: 前端图标改

**决定：改。**

- 图标文件名从 `diamond.png` 改为 `star-stone.png`，水晶类从 `red-crystal.png` 改为 `red-core-gem.png` — ✅ 已完成
- `_mapping.js` 跟着改 — ✅ 已完成
- 项目未上线，没有用户认知包袱
- 图标风格可以暂时复用，后续让设计师出新图

### 4.3 Q3: 历史数据不需要迁移

**决定：不需要。**

- 项目未上线，没有真实用户数据
- 种子数据 (seeder) 跟着改就行
- 如果有测试数据在 dev 库里，跑一个迁移脚本更新即可
- **重要原则**：流水记录应该存 asset_code 而不是 display_name（这是所有大厂的共识），这样改名永远不需要迁移历史数据

### 4.4 Q4: DIY 材料不动

**决定：不需要。**

- DIY 材料表 (`diy_materials`) 用的是真实矿物名（巴西黄水晶、粉水晶等），这些就是国标珠宝名称
- 虚拟资产改名后已经完全区分开了："紫源晶" vs "紫水晶"，不会再混淆
- 改真实矿物名反而不专业

### 4.5 顺带修复的技术债务

既然要动 asset_code，顺手解决现有的硬编码问题：

| 问题 | 修复方案 | 理由 |
|------|---------|------|

### 方案对比总结

| 维度 | 方案A: 只改 display_name | 方案B: code + display_name 全改 (已选) |
|------|------------------------|---------------------------------------|
| 改动量 | 1 个种子文件 | ~8 个文件 + 种子 + 迁移脚本 |

| 长期心智负担 | **永远存在** — 每个开发者都要记"DIAMOND=星石" | **零** — 代码自解释 |

### 12.1 BUDGET_POINTS 的 form 应该改成什么？

#### 行业做法

| 公司/方案 | 预算积分类资产的分类 | 理由 |
|----------|-------------------|------|
| 美团 | 和主积分同 form（currency） | 预算积分本质是货币的受限变体 |

### 13.1 PrizePoolService 中 DIAMOND 的特殊分治逻辑怎么处理？

当前代码逻辑：

```js
if (material_asset_code !== 'DIAMOND') → 强制用 material_amount × budget_value_points 计算 budget_cost
if (material_asset_code === 'DIAMOND') → 允许手动设置 budget_cost
```

这是一个业务逻辑，不只是命名问题。两个选项：

| 选项 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A** | 直接把 `'DIAMOND'` 改成常量引用 `AssetCode.STAR_STONE`，逻辑不变 | 最简单，改动量最小 | 逻辑仍绑定特定 code |
| **B** | 在 `material_asset_types` 表加字段（如 `is_primary_currency: true`），代码改为查表判断 | 彻底消除硬编码 | 要加字段 + 写迁移 |

#### 已确认：选 A — 改常量引用，不加字段

行业做法：所有公司在重构命名时，对这类"特定资产的业务分支"都是直接改引用，不会为了一个判断去加字段。加字段是"去硬编码"的优化，不是"改名"的必要步骤。这两件事应该分开做——本次重构只负责改名和常量化，后续如果要彻底去硬编码再加字段。

改法：`if (material_asset_code === 'DIAMOND')` → `if (material_asset_code === AssetCode.STAR_STONE)`

### 13.2 system_settings 中 key 含 DIAMOND 的配置怎么改？

当前有 `fee_rate_DIAMOND`、`daily_max_listings_DIAMOND` 等 key 是动态拼接的（`fee_rate_` + `asset_code`）。

| 选项 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A** | key 跟着改（`fee_rate_star_stone`） | 保持一致性 | 改动量稍大 |
| **B** | key 不改，只改 value 中的 asset_code | 减少改动量 | key 和 value 不一致 |

#### 已确认：选 A — key 跟着改

这不需要拍板，是技术事实决定的：这些 key 是代码中 `` `fee_rate_${asset_code}` `` 动态拼接读取的。`asset_code` 从 `'DIAMOND'` 改成 `'star_stone'` 后，代码拼出来的 key 自然就是 `fee_rate_star_stone`。如果数据库里的 key 不跟着改，代码就读不到配置了。

### 13.3 allowed_settlement_assets 和 allowed_listing_assets 的值要不要扩大？

当前值都是 `["DIAMOND","red_shard"]`。改名后变成 `["star_stone","red_core_shard"]`。

#### 已确认：不扩大，只改名

理由：扩大结算币种范围（比如允许 `orange_core_shard` 结算）是业务决策，不是重构范围。本次重构的原则是"改名不改逻辑"——保持现有业务行为完全不变，只替换 asset_code。如果以后要开放更多结算币种，改 `system_settings` 配置值即可，架构已经支持。

### 13.4 Admin 前端文件名含 DIAMOND 的要不要改？

| 当前文件名 | 改后文件名 | 影响 |
|-----------|-----------|------|
| `diamond-accounts.js` | `star-stone-accounts.js` | 对应 HTML 也要改 |
| `diamond-accounts.html` | `star-stone-accounts.html` | Vite MPA 入口配置要改 |
| `platform-diamond.js` | `platform-star-stone.js` | 对应 HTML 也要改 |
| `platform-diamond.html` | `platform-star-stone.html` | Vite MPA 入口配置要改 |

改文件名会影响 Vite 的 MPA 入口配置（`vite.config.js` 中的 `input` 映射）。

#### 选项分析

| 选项 | 文件名示例 | 优点 | 缺点 |
|------|-----------|------|------|
| **A: 产品名** | `star-stone-accounts.js` | 简单直观，一看就知道管的是星石 | 如果以后星石改名，文件名又得改 |
| **B: 抽象名** | `primary-currency-accounts.js` | 不管主货币叫什么，文件名永远不用改 | 不够直观，新人要想一下"这是哪个货币" |

#### 已确认：选 A — `star-stone-accounts.js`

理由：抽象文件名在这个场景下没有实际价值。文件名叫 `primary-currency-accounts.js`，但页面标题写的是"星石账户管理"，页面里所有文案都是"星石余额"、"星石发放"。如果以后改名叫"灵石"，文件名是不用改了，但页面里的文案、API 参数、`display_name` 全得改——文件名那点改动量根本不值一提。用产品名最直观，改动量也最小。

### 13.5 orphan-frozen.js 中的 gold_coin / silver_coin 是什么？

这两个 code 在 `material_asset_types` 表里不存在，在任何数据表里也没有数据。是更早期的资产名，只剩前端一个 `display_name` 映射表还在引用。

#### 已确认：清掉

行业做法：所有公司的做法都一样——清理死代码。没有任何公司会保留引用不存在实体的代码。

本次重构对这个映射表的改动：
1. 删掉 `gold_coin` 和 `silver_coin`（死代码）
2. 把 `diamond` 改成 `star_stone`
3. 加上新的 asset_code 映射

> **后续优化**（不在本次重构范围）：这个映射表本身应该改成从后端 API 获取 `display_name`，而不是前端硬编码。但那是另一个优化了。

### 13.6 迁移文件（migrations/）中的旧 code 要不要改？

| 选项 | 做法 | 理由 |
|------|------|------|
| **A** | 不改 | 迁移文件是历史记录，改了反而不安全 |
| **B** | 只改活跃迁移（非 archived 的），archived 的不动 | 折中方案 |

#### 已确认：选 A — 不改任何已执行的迁移文件，新建迁移文件做 code 变更

行业做法：

| 做法 | 代表 | 理由 |
|------|------|------|
| **绝不改已执行的迁移** | Rails 社区共识、Django 官方文档、Sequelize 官方建议、所有大厂 | 迁移文件是数据库的"git log"，改了等于篡改历史，可能导致 SequelizeMeta 校验失败 |
| squash 后重写 | 部分创业公司在上线前做一次 | 把所有迁移压缩成一个 baseline，然后从头开始 |

本项目实际情况：
- 已经有一个 `baseline-v4.1.0-squashed.js`，说明已经做过 squash
- `archived` 目录里的迁移是历史存档
- Sequelize 的 `SequelizeMeta` 表记录了哪些迁移已执行，改文件名或内容会导致状态不一致
- 迁移文件里的 `'DIAMOND'` 是历史事实——"当时确实叫 DIAMOND"

正确做法：新建一个迁移文件 `20260401-rename-asset-codes.js` 来做所有的 code 变更，这就是第十二章迁移脚本的 Sequelize 版本。
