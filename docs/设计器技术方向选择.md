我了解这类产品。「养个石头」「晶奥之境」这类 DIY 水晶手串小程序,核心是**「3D 珠子实时渲染 + 自动串珠算法 + 电商下单」**三块拼起来的。我按技术模块拆给你讲。

## 一、整体技术栈

微信小程序里做 3D,主流两条路:

- **WebGL / Three.js**:通过小程序的 `<canvas type="webgl">` + `wx.createSelectorQuery` 拿到 canvas node,再跑一套精简版 Three.js(社区有 `threejs-miniprogram` 官方适配包)。这是绝大多数水晶小程序用的方案。
- **原生 3D 引擎(如 Cocos / Layา)**:少数团队会用游戏引擎导出,但包体大、和电商业务耦合麻烦,用得少。

「养个石头」这类基本都是 **Three.js 小程序版**。

## 二、3D 珠子是怎么来的

珠子不是实时建模,而是**预制好的 3D 资源**

1. **建模**:美术用 Blender/C4D 把每种珠子(圆珠、算盘珠、隔片、貔貅/葫芦等配饰)做成 `.glb/.gltf` 模型,带 PBR 材质。
2. **材质关键**:水晶的通透感靠这几样堆出来——
   - 环境贴图(`envMap`)做反射,让珠子有"照进光"的感觉
   - 透明度 + 折射(`transmission` / 物理材质 `MeshPhysicalMaterial`)
   - 内部裂纹/包裹体用法线贴图或体积雾模拟
3. **优化**:珠子面数压到很低(几百面),纹理合并成图集,否则手机上串 20 颗珠子直接卡爆。

## 三、"点一下自动拼手串"的核心算法

这是最有意思的部分。用户选珠子,系统自动排布成一个圆环,本质是**几何布局问题**

```javascript
// 已知:手围周长 C(比如 16cm)、每颗珠子直径 d[i]
// 求:每颗珠子在圆环上的角度位置

// 1. 算总周长 = 所有珠子直径之和
const total = beads.reduce((sum, b) => sum + b.diameter, 0);

// 2. 圆环半径 R = 总周长 / (2π)
const R = total / (2 * Math.PI);

// 3. 依次沿圆周摆放,每颗占的弧度 = 自身直径对应的角度
let angle = 0;
beads.forEach(bead => {
  const arc = bead.diameter / R;      // 这颗珠子占的弧度
  const mid = angle + arc / 2;        // 摆在弧段中点
  bead.position = {
    x: R * Math.cos(mid),
    y: R * Math.sin(mid),
    z: 0
  };
  // 让珠子朝向圆心(法线对齐),配饰才不会歪
  bead.rotation = mid;
  angle += arc;
});
```

要点:

- **等周长分布**:不是等角度,而是按每颗珠子的实际直径分配弧长,大小珠混搭才不会挤或空。
- **朝向对齐**:貔貅、葫芦这种有方向的配饰,要按切线/法线旋转,让它"正对外"。
- **手围换算**:用户选 14/16/18cm 手围 → 反推需要几颗珠子,或固定珠数反推每颗尺寸。
- **实时增删**:用户拖进/删掉一颗,重新跑一遍布局 + 补间动画(Tween)让珠子平滑归位,体验才顺滑。

## 四、交互层

- **拖拽面板**:底部是珠子商品列表,点击/拖拽加入手串。
- **旋转查看**:监听 canvas 的 `touchmove`,转 3D 场景的相机或整个手串 group 的 rotation。
- **实时改价**:每加一颗珠子,累加单价 → 顶部实时显示总价和克重。

## 五、电商闭环(其实是主体)

3D 只是"皮",赚钱靠后面:

- 每颗 3D 珠子对应一个 **SKU**(材质/尺寸/等级/价格)
- 用户搭好的手串 = 一个 **自定义组合订单**,后端把珠子清单、手围、总价存下来
- 下单 → 工厂/手作师傅按清单**真人手工串制**发货
- 加上直播、盲盒、"能量寓意"文案、社交分享搭配图,做转化。

## 结论:不是 Three.js 方案,用的是微信官方的 `xr-frame`

这套 DIY 3D 手串代码**没有**采用你说的 `<canvas type="webgl">` + `wx.createSelectorQuery` + `threejs-miniprogram` 那条路线。它用的是微信小程序官方的 **XR-Frame(`xr-frame`)** 3D 框架。

证据很硬:

**1. 组件声明用 `renderer: "xr-frame"`**

```json
{
  "component": true,
  "renderer": "xr-frame",
  "usingComponents": {}
}
```

这是 XR-Frame 组件的标志。Three.js 方案这里会是普通组件,不会有这个字段。

**2. WXML 用的是 XR-Frame 标签,不是 canvas**

`xr-bead-scene.wxml` 里全是 `<xr-scene>`、`<xr-camera>`、`<xr-mesh>`、`<xr-light>`、`<xr-asset-material>` 这些声明式标签,珠子直接用 `geometry="sphere"` 摆出来:

```html
<xr-scene bind:ready="onSceneReady" bind:tick="onTick">
  <xr-camera camera-orbit-control="" .../>
  <xr-light type="directional" .../>
  <xr-mesh wx:for="{{beadNodes}}" geometry="sphere" material="{{item.matId}}" .../>
</xr-scene>
```

而 Three.js 方案会是命令式 JS:`new THREE.Scene()`、`new THREE.SphereGeometry()` 那一套。

**3. 全局搜索验证**
- 搜 `three` / `threejs-miniprogram`:**0 处引用**
- 搜 `<canvas type="webgl">`:**0 处**
- 搜 `xr-frame`:命中 `xr-bead-scene`、`xr-bead-viewer` 两个组件
- `package.json` 里**没有** three 依赖

## 两个方案的对比,以及这套代码的实际做法

| 维度 | 你猜的 Three.js 方案 | 实际用的 XR-Frame 方案 |
|---|---|---|
| 渲染载体 | `<canvas type="webgl">` | `<xr-scene>` 声明式标签 |
| 拿节点 | `wx.createSelectorQuery` | 框架托管,`bind:ready` 回调 |
| 3D 库 | `threejs-miniprogram` npm 包 | 微信内置 `xr-frame` 系统组件 |

**串珠算法**这套代码里也实现了,在 `xr-bead-scene.ts`:

## 核心差异:水晶质感能不能做出来

这是水晶手串这个品类最关键的点,珠子好不好看直接影响下单转化。

- **Three.js**:有 `MeshPhysicalMaterial`,支持 `transmission`(真透射)、`ior`(折射率)、`thickness`(厚度)、`clearcoat`(清漆层)。能做出**真正的光线穿过水晶、内部折射、边缘高光**那种高级感。「养个石头」那种让人一眼觉得"哇这珠子真水灵"的效果,基本要靠这套物理材质。
- **XR-Frame**:`effect="standard"` 是标准 PBR(金属度/粗糙度),**没有真正的透射折射**。你项目里现在就是靠 `baseColor` 的 alpha < 1 做半透明近似——能看,但和真水晶的通透感差一档,尤其白水晶、月光石这种高通透的料子差距明显。

**这一项 Three.js 明显胜。**

## 针对你这个项目的建议

你这是个**电商小程序**(有兑换、抽奖、支付),3D 是"展示皮",不是核心玩法。这种定位下:

**继续用 XR-Frame 是对的**,理由:
1. 你的 3D 只是让用户预览搭配、增强下单欲望,不是要做 3D 游戏。够用就行。
2. 包体积对小程序是硬约束(主包 2MB 限制),XR-Frame 不占包这点很值钱。
3. 声明式写法后期好维护,你团队接手成本低。
4. 稳定性、兼容性对电商很重要,不能因为某款手机渲染崩了就丢订单。

**唯一要补的短板就是质感。** 有两个折中办法,不用推翻重来:

- **办法 A(推荐,性价比最高)**:关键的高通透珠子(白水晶/月光石等)不靠实时渲染,而是美术**预渲染好的高质量贴图/序列帧**贴到球体上,或直接用 2.5D 图片。用户看的是"照片级"效果,比实时 alpha 近似好太多,还省性能。
- **办法 B**:XR-Frame 支持自定义 `effect`(自己写 shader)。可以写一个近似菲涅尔反射 + 假折射的着色器,把通透感做上去。成本较高,要懂 GLSL。
