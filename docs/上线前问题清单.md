# 天工小程序前端 — 上线前问题清单

> 审查日期：2026-04-30
> 项目版本：v5.2.0
> 基础库版本：3.15.0
> 审查范围：安全、废弃API、性能、代码质量、兼容性、合规

## P0 — 上线前必须修的

### 1. 环境切换是手动硬编码的，极易出事

**文件**：`config/env.ts` 第 286 行

**现状**：当前环境写死为 `'testing'`：

```typescript
let CURRENT_ENV: string = 'testing'
```

而且 production 和 testing 用的是同一个 URL（`omqktqrtntnn.sealosbja.site`），注释里写着"🚨 部署时更新为正式域名"。

**风险**：如果上线时忘了改这两处，用户直接连到测试服务器，测试验证码 `123456` 也能用。

**解决方案**：用 `wx.getAccountInfoSync().miniProgram.envVersion` 自动判断环境（`develop` / `trial` / `release`），不再手动切换。

**涉及文件**：
- `config/env.ts`（第 165 行 devVerificationCode、第 253 行 production baseUrl、第 286 行 CURRENT_ENV）

---

### 2. 两个废弃 API 在新版基础库上已经不能用

基础库是 `3.15.0`，但代码里还在调用已废弃/已删除的 API：

#### 2a. `wx.getUserProfile` — 已删除

**文件**：`utils/wechat.ts` 第 85 行

**现状**：`wx.getUserProfile` 从基础库 2.27.1 起已删除，调用直接报错。

**解决方案**：改成 `<button open-type="chooseAvatar">` + `<input type="nickname">` 组合方案。

#### 2b. `wx.chooseImage` — 已废弃

**文件**：`packageUser/feedback/feedback.ts` 第 173 行

**现状**：`wx.chooseImage` 从基础库 2.21.0 起标记为 deprecated。项目其他地方（dispute.ts、chat、ad-create.ts）已经用了 `wx.chooseMedia`，唯独这一处没改。

**解决方案**：替换为 `wx.chooseMedia`，保持全项目一致。

## P1 — 上线前应该修的

### 3. 没有强制更新机制

**文件**：`app.ts`

**现状**：没有调用 `wx.getUpdateManager()`。用户可能一直停留在旧版本，如果更新了 API 接口格式，旧版前端直接崩。

**解决方案**：在 `app.ts` 的 `onLaunch` 中添加：

```typescript
const updateManager = wx.getUpdateManager()
updateManager.onCheckForUpdate((res) => {
  if (res.hasUpdate) {
    updateManager.onUpdateReady(() => {
      wx.showModal({
        title: '更新提示',
        content: '新版本已准备好，是否重启应用？',
        success: (modalRes) => {
          if (modalRes.confirm) updateManager.applyUpdate()
        }
      })
    })
    updateManager.onUpdateFailed(() => {
      wx.showModal({
        title: '更新提示',
        content: '新版本下载失败，请删除小程序后重新搜索打开',
        showCancel: false
      })
    })
  }
})
```

**改动量**：约 20 行代码。

---

### 4. 没有全局 Promise 错误捕获

**文件**：`app.ts`

**现状**：有 `onError`，但没有 `onUnhandledRejection` 和 `onPageNotFound`。项目里大量 async/await，如果某个 Promise 没 catch，错误会被静默吞掉，日志里看不到。

**解决方案**：在 `App({})` 中添加：

```typescript
onUnhandledRejection(res) {
  console.error('[App] Unhandled Promise Rejection:', res.reason)
  // 上报到日志系统
},
onPageNotFound(res) {
  console.error('[App] Page Not Found:', res.path)
  wx.redirectTo({ url: '/pages/lottery/lottery' })
}
```

**改动量**：约 10 行代码。

### 5. 内存泄漏风险 — 定时器没清理

**涉及文件**：

| 文件 | 问题 |
|---|---|
| `packageTrade/trade/market/market.ts` 第 767 行 | `onUnload()` 是空函数，页面上的定时器和 WebSocket 订阅不会被清理 |
| `packageExchange/exchange-shelf/sub/bid-panel/bid-panel.ts` | 两个 `setInterval`（`_bidListTimer`、`_bidModalTimer`），需确认 `detached()` 里都清了 |
| `packageLottery/lottery-activity/sub/flashsale/flashsale.ts` | 两个 `setInterval`（`_broadcastTimer`、`_countdownTimer`），需确认 `detached()` 里都清了 |

**解决方案**：逐一检查每个使用 `setInterval` / `setTimeout` 的页面和组件，确保在 `onUnload()` 或 `detached()` 中调用 `clearInterval` / `clearTimeout`。

---

### 6. rich-text 的 XSS 风险

**文件**：`packageUser/chat/chat.wxml` 第 110 行

**现状**：

```html
<rich-text class="result-message-text" nodes="{{item.highlightContent}}"></rich-text>
```

如果 `highlightContent` 来自用户输入且没有做 HTML 转义，就是 XSS 漏洞。

**解决方案**：在渲染前对 `highlightContent` 做 HTML 实体转义（`<` → `&lt;`、`>` → `&gt;` 等），只保留允许的标签（如 `<em>`、`<span>`）。或者改用 `<text>` 组件 + 自定义高亮逻辑。

---

**重灾区（Top 10）**：

| `packageAdmin/audit-list/audit-list.ts` | 51 |

**解决方案**：分批替换，优先处理 `store/user.ts` 和 API 相关文件。使用 `typings/api.d.ts` 中已有的类型定义替代 `any`。

**改动量**：大，建议分阶段进行，不阻塞上线。

## P2 — 上线后尽快修的

### 8. 性能：setData 在定时器里每秒调用

**涉及模块**：拍卖倒计时、QR 码倒计时、竞价面板

**现状**：在 `setInterval` 里每秒调用 `setData`，每次都触发页面重渲染。

**解决方案**：
- 使用 path-based 更新：`this.setData({ 'countdownText': value })` 而不是传整个对象
- 或者用 WXS 在视图层直接计算倒计时，完全不经过逻辑层

---

### 9. 性能：同步 Storage 连续写入

**涉及文件**：

| 文件 | 问题 |
|---|---|
| `store/user.ts` 第 81-83 行 | 登录时连续 3 个 `wx.setStorageSync` |
| `utils/config-cache.ts` 第 423-425 行 | 连续 3 个 `wx.setStorageSync` |
| `app.ts` 第 877 行 | 每次 token 日志记录都做同步写入 |

**解决方案**：改成异步 `wx.setStorage`，或合并为一次写入（将多个值打包成一个对象存储）。

---

### 10. 没有网络状态监听

**文件**：`app.ts`

**现状**：`globalData` 里有 `network_status` 字段，但从来没有通过 `wx.onNetworkStatusChange` 自动更新。用户断网后没有任何提示。

**解决方案**：在 `onLaunch` 中添加：

```typescript
wx.onNetworkStatusChange((res) => {
  this.globalData.network_status = res.isConnected ? 'online' : 'offline'
  if (!res.isConnected) {
    wx.showToast({ title: '网络已断开', icon: 'none' })
  }
})
```

---

### 11. 没有内存警告监听

**文件**：`app.ts`

**现状**：项目有 14 种抽奖游戏（Canvas 动画）+ DIY 设计器 + 图片密集的商品列表，内存压力不小，但没有 `wx.onMemoryWarning` 监听。

**解决方案**：在 `onLaunch` 中添加：

```typescript
wx.onMemoryWarning((res) => {
  console.warn('[App] 内存警告，级别:', res.level)
  // 清理缓存、释放不可见页面资源
})
```

---

### 12. npm 依赖版本没锁定

**文件**：`package.json` 第 62 行

**现状**：

```json
"weapp.socket.io": "^3.0.0"
```

`^` 允许自动升级到 3.x.x 的任何版本。

**解决方案**：生产依赖锁定精确版本：

```json
"weapp.socket.io": "3.0.0"
```

同样检查其他依赖：
- `"mobx-miniprogram": "^6.12.3"` → 锁定为 `"6.12.3"`
- `"mobx-miniprogram-bindings": "^5.1.1"` → 锁定为 `"5.1.1"`

---

### 13. 隐私合规 — 缺少 Privacy API

**现状**：项目用了定位（`scope.userLocation`）、相册、相机，但没有实现微信要求的 Privacy API。

**背景**：2023 年起微信要求所有收集用户数据的小程序必须接入 Privacy API（`wx.getPrivacySetting`、`wx.requirePrivacyAuthorize`、`wx.onNeedPrivacyAuthorization`）。

**解决方案**：
1. 在微信后台配置隐私协议
2. 在 `app.json` 中添加 `"__usePrivacyCheck__": true`
3. 实现隐私弹窗组件，在首次使用敏感 API 前弹出

## P3 — 优化项

### 14. SVG 图片不支持

**涉及文件**：

```
images/icons/rank/crown-gold.svg
images/icons/rank/medal-bronze.svg
images/icons/rank/medal-silver.svg
images/empty/empty-network.svg
images/empty/empty-data.svg
images/empty/empty-order.svg
images/empty/empty-search.svg
```

**问题**：微信小程序 `<image>` 组件不支持 SVG 格式。

**解决方案**：转换为 PNG 或 WebP 格式，或使用 inline SVG 通过 `<rich-text>` 渲染。

### 17. 预加载过多

**文件**：`app.json` preloadRule

**现状**：
- `pages/user/user` 预加载 6 个分包（user-services, trade, admin, lottery, exchange, ad）
- `pages/exchange/exchange` 预加载 4 个分包

**问题**：弱网设备上可能拖慢页面加载。

**解决方案**：只预加载最可能跳转的 1-2 个分包，其余改为 `"network": "wifi"` 条件预加载。

**涉及文件**：

| 文件 | 行号 | 调用 |
|---|---|---|
| `config/env.ts` | 294, 371, 374 | `console.error`、`console.log` |
| `store/points.ts` | 118 | `console.error` |

**问题**：绕过了 `utils/logger.ts` 的日志级别过滤，会在 production 环境输出。
