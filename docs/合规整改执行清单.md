## 0. 整改后的目标架构

> **方案演进说明**：早期方案保留 C2C 用户间交易，需靠"焊死道具无价值 + 反对倒风控"换安全。**现已升级为更彻底的方案**：取消用户间交易，星石只能向官方单向购买道具。此举一举消除二级市场、场外套现、价格发现、对倒洗钱等全部 C2C 风险，并大概率免去 EDI 许可证。

```
【线一·消费积分兑换】对标京东京豆 —— 安全
  消费(真钱给商家) → 积分points(不可交易) → 抽奖
        ↓ 抽奖产出
  水晶/碎片/源晶(不可交易) → 兑换商城换实物 → 实物(不可再交易)
        ↓ 也可单向转换(价值高→低,等于销毁,安全)
  星石

【线二·官方道具商城(B2C 单向)】对标游戏点券商店 —— 安全
  抽奖产出星石 / 碎片转换得星石 → 星石只能向"官方道具商城"单向购买虚拟道具
        (签到/补签卡、置顶道具、对话框皮肤等,系统外零变现价值)
        ★ 单向:买了即消耗,不能卖回官方换星石(禁止官方回收)
        ★ 用户与用户之间无任何交易

允许的单向流: 碎片/水晶 → 星石(高价值→低价值,销毁方向)
严禁的反向流: 星石 → 碎片/水晶/实物/现金(无价值→有价值,套现方向)
```

**核心取舍一句话**：水晶系 = 京豆化（可换实物，不可在用户间流通）；星石 = 游戏点券（只能向官方单向买道具，不可炒、不可回笼、不可在用户间流通）。水晶可**单向**转星石（价值高→低，安全）。

**生死线**：
1. 道具商城必须**单向**——用户用星石买官方道具，买了就是消耗，**不能卖回换星石**（禁止官方回收）。
2. 资产转换必须**单向往无价值流**——允许 `碎片/水晶→星石`，**严禁 `星石→碎片/水晶/实物/现金`** 任何反向通道。

双向回购、或星石反向流回有价值侧，都会让星石获得"可回笼/保值"的货币属性，直接撞红线。

## 三、兑换商城改造（水晶换实物这条线）

> **重要修正（2026-06-02）**：优惠券不必一律剔除。京东/银行同样发优惠券且合法，关键不在"有没有券"，而在"券能不能被转手交易"。本节区分**折扣/抵扣券**（可保留，去交易化）与**储值礼品卡**（预付卡监管区，需单独处理）。

### 3.A 优惠券 ≠ 礼品卡，分开处理

| 类型 | 真实ID | 性质 | 处理 |
|---|---|---|---|
| **折扣/抵扣券** | 2（50元券）、3（9折券）、4（8折券）、6（单人套餐券）、8（家庭套餐券） | 依附于一笔具体消费的折扣，类似京东券 | **可保留**作兑换奖励，但必须做成京东那样：①不可交易 ②不可转让（记名）③单用途（仅合作商户抵扣）④不可兑现、不找零 |
| **储值礼品卡** | 12（100元）、13（200元）、14（500元） | 面值即现金，属储值凭证 | **移除或改造**，不作为虚拟资产兑换标的。储值卡受《单用途商业预付卡管理办法》约束，发卡需备案/资质，必须律师确认 |

> 京东/银行优惠券合法的四要件：**记名 + 不可转卖 + 单用途 + 不可兑现**。你的券之前问题在 `is_tradable=1`（能被挂牌交易），不是"存在优惠券"本身。

## 四-A、抽奖概率公示缺口（2026-06-04 新增诊断，连真实库 + 通读代码核实）

> **来源**：核实"抽奖管理干预"时连带发现，与 §4.1 同源。已连 `.env` 真实库 `restaurant_points_dev` + 通读抽奖管线代码确认，未引用历史报告。

### 1. 核心事实：你的中奖概率是「动态计算」的，不存在固定值

主活动（`lottery_campaign_id=1`，active，`budget_mode=user`）实测：

- 奖品仅三档实体：high 3 个 / mid 4 个 / low 5 个（各档总权重均 100 万），fallback 靠 `tier_fallback.prize_id=209` 兜底。
- 每次抽奖的档位概率由以下机制**实时计算**（均已在 `lottery_strategy_config` 启用）：预算分层 B0-B3、压力矩阵、Pity 软保底（连续不中第 9 次权重 ×10）、运气债务、防连空（连 5 次）、防连高（连 2 次）、首抽必中、**high 档 8% 硬上限**。
- 结论：同一用户因预算/手气/是否新人不同，high 档概率在 **0%~8%** 浮动，**没有一个能写进规则页的固定数字**。

### 2. 问题一：公示固定值 vs 动态概率，必然对不上（违 §4.1）

- 《用户服务协议》第 7 章公示为 **高 10 / 中 25 / 低 65**（三档、无空奖档、固定值）。
- 代码实际为 **四档（high/mid/low/fallback）+ high 封顶 8% + 全程动态**。
- 冲突：①公示"high=10%" 与 代码"high 封顶 8%" 矛盾；②公示无 fallback/空奖档；③公示是固定值，后台是 0~8% 浮动 → "后台=公示"根本无法成立。
- **已拍板方向：改公示文案（不改内核）**，且不能再换一组固定值，须改为「区间 + 上限 + 保底说明」。
  - 现阶段采用：**"中大奖（high 档）概率 0–8%，随账户状态与保底机制动态浮动，单次最高不超过 8%；设有连续不中保底"**。"最高 8%"是代码硬上限，可 100% 兑现。
  - 后续配合"用户等级制度"可升级为「按等级公示概率区间」（会员权益式）。

### 3. 问题二（更重要）：决定"能否中大奖"的预算账户，对用户完全隐藏

- 预算账户 `BUDGET_POINTS` **由消费按比例自动发放**（`services/consumption/CoreService.js`：`消费金额 × budgetRatio`，默认 0.22）。
- 它直接决定能抽哪些档（`BUDGET_TIER_AVAILABILITY`：B0/B1 仅 low+fallback，B2 解锁 mid，B3 解锁 high）——**预算不足时 high/mid 档概率为 0**。
- 但 `services/DataSanitizer.js:14` 将 `BUDGET_POINTS` 列入 `FORBIDDEN_FRONTEND_ASSET_CODES`，**绝对禁止暴露给前端**（余额、流水、asset_code 全屏蔽），设计意图是防抓包分析商业逻辑。
- **矛盾**：决定"谁能中大奖"的真实资格门槛，用户完全无感知。一个 B0 用户可能以为自己能中一等奖，实际概率恒为 0 且无从得知 → 隐瞒中奖真实资格条件，触 §4.1 + 消法知情权。
- **定性（非违法设计，是披露缺失）**：预算随消费可增长、可解锁大奖，本质是"消费越多越易中好奖"的**合法机制**，问题仅在"没告诉用户这个规则"。

### 4. 已拍板处置：公开"规则"但不公开"数字"

- **告知规则**：在抽奖规则页用人话写明 **"抽中高价值奖品需账户达到相应消费/积分门槛，消费越多可抽档位越高"**，如实披露"中奖与消费挂钩"。
- **不暴露数字**：仍**不**向前端展示 `BUDGET_POINTS` 余额/流水，`DataSanitizer` 屏蔽维持不变（兼顾防抓包）。
- 8% 上限、压力矩阵、Pity 等**纯内部算法**无须告知，仅以"设有保底、概率动态浮动"概括。
- **待办**：①修订《用户服务协议》第 7 章概率表为区间+上限+保底；②抽奖规则页补"中奖资格与消费挂钩"说明；③律师复核文案。

### 10.9 行业对照：四个决策的业界设计参考（论证已拍板项）

> 用于佐证 §10.8 决策。核心判据：**B2C 单向商城 与 C2C 撮合市场是两种不同的限界上下文**，业界从不混用。

**A. 决策 1（独立域 vs 原地改造）的业界分布**

| 阵营 | 典型做法 | 启示 |
|---|---|---|
| 大厂（腾讯/阿里/美团） | 王者荣耀仅 B2C 点券商城**无 C2C**；DNF「拍卖行(C2C)」与「点券商城(B2C)」两套独立系统；阿里把闲鱼(C2C)/淘宝(B2C) 拆成独立 App+后端 | B2C 与 C2C 是不同聚合根，**从不把一个改造成另一个** |
| 游戏公司 | 原神创世结晶买道具(买即消耗不可卖回) 与 Steam 市场/CSGO 交易行**永远两套代码** | 不变量不同，混在一起必出 bug |
| 小公司/活动策划 | 常**原地改造**省人力 | 留死字段、"表名 trade_orders 实存官方售卖"的债务 |
| 虚拟物品/小众二手（BUFF/IGXE/DD373） | 保留并强化 C2C 撮合 | 方向与你相反，不可参考 |

**→ 最适合你**：B2C 与 C2C 应是**不同限界上下文**（结论方向正确）。~~新建独立域~~ **⚠️ 修正（§10.13/§10.14-B）**：后端实测**已存在 `exchange` 自营域**（直购 + 竞价），故无需"新建 `/prop-shop`"，**直接复用 `exchange` 域**即满足"B2C 与 C2C 分离"——比新建更省、更低债。本表"B2C/C2C 不混用"的方法论成立，"用新域还是复用现有自营域"以**决策 11（复用 exchange）**为准。

**B. 决策 2（旧数据处置）的业界做法**

- 大厂上线前标准动作是**数据清档**；被砍掉的预发布功能，其**专属表 + 死代码一并删除**（YAGNI / 避免孤儿表）。
- "清空保表"看似稳妥，实为债务：孤儿表 + 死 Service 代码 + 新人困惑。你既然要"最低技术债"，应**彻底删除 C2C 专属表与代码**，仅保留 `exchange` 域复用的共享基建。
- 安全做法：**`exchange`/`exchange/bid` 等价能力验证通过后再删**，留一个回滚窗口（迁移脚本可写 `down` 以备恢复结构）。

**C. 决策 3（手续费）与决策 4（行情接口）的业界逻辑**

- **手续费**：B2C 点券商城（王者/原神）购买道具**不存在"撮合手续费"**这一概念——只有"花点券买、买完即消耗"。游戏经济学里"销毁(sink)"是稳定币值的标准手段。`destroy` = 合规最强（平台不靠交易牟利）+ 代码最简（无回流逻辑）。`recycle` 会制造"费用养奖池"的循环，更像"平台经营代币经济"，合规更弱、代码更繁 → 不选。
- **行情接口**：价格走势/成交量/价格发现是 **C2C 二级市场的内在特征**（Steam 市场、BUFF 都有 K 线）。B2C 官方定价不波动，点券商城**从不有行情图**。给虚拟资产展示"价格走势"正是监管最忌讳的**金融化/二级市场信号**——迁到自营商城等于主动把金融化特征引回来，**有害**。故随 C2C 一并删除，绝不迁移。

> **一句话总览**：你这套（消费返利 + 抽奖 + 自营兑换 + B2C 单向道具商城）最接近**腾讯/米哈游的"点券商城"模型**，而非任何 C2C 交易平台。四个决策全部朝"向点券商城靠拢、远离二级市场"方向收敛，与合规目标和你的低技术债诉求完全一致。

### 10.10 官方拍卖（B2O）决策论证：引擎复用 + 仅星石

> ⚠️ **本节"重构复用 C2C `auction_*`"的引擎选型已被 §10.14-A 推翻**：实测发现项目已有现成的 `exchange/bid` B2O 引擎，比改造 C2C 更干净。**最终选型见 §10.8 决策 9' / §10.14-A**。本节关于"官方拍卖比 C2C 更合规""仅星石竞价"的**合规论证仍然有效**，仅"用哪套引擎实现"一项被更新。
>
> 论证 §10.8 决策 10（仅星石竞价）。"引擎复用"结论已升级为：复用 `exchange/bid`（非改造 `auction_*`）。

**A. 合规性：官方拍卖比 C2C 拍卖更干净**

对照三条生死线：

| 红线 | C2C 拍卖 | 官方拍卖（B2O） |
|---|---|---|
| 单向/禁回收 | 用户↔用户，有二级市场 ❌ | 官方出货→竞价→最高者得→星石销毁，不可卖回 ✅ |
| 上架白名单（纯虚拟） | 啥都能挂 ❌ | 同直购：仅 `item_type='prop'`，拒实物/券/有 RMB 价值 ✅ |
| 资产单向往无价值流 | — | 中标星石被消耗(sink)，比直购销毁更彻底 ✅ |

三个定性澄清：
1. **落标者冻结星石 = 全额解冻退回**，是"退还没花出去的钱"（托管释放），**非官方回购**（回购指"拿道具换回星石"，这里没有）。
2. **非赌博/射幸**：中标者确定（出价最高者得），不是随机，射幸性**低于现有抽奖**。
3. **不构成二级市场**：虽有价格发现，但封闭在"官方→用户单向 + 道具不可转售 + 星石买完即焚"，用户间不流通即无二级市场。
> 建议律师确认项：拍卖会让星石产生"竞得价值感"，但不可提现/不可用户间交易/买完即焚 → 属 UX 体感，非法律意义的货币财产。

**B. 引擎选型 —— 已被 §10.14-A 取代（弃"改造 C2C `auction_*`"，改复用现存 `exchange/bid`）**

> ⚠️ 本节早期对比的是"改造 C2C `auction_*`(方案A) vs 全新表(方案B)"，曾选 A。但 §10.13-A 实测发现项目**已有现成的 `exchange/bid` B2O 引擎**（卖家=官方、中标款 sink、挂 `exchange_items`、存量 0），比改造 C2C 更干净。**最终引擎选型见 §10.8 决策 9' / §10.14-A：复用 `exchange/bid`，`services/auction/`+`auction_*` 随 C2C 删除。** 本节关于"成熟引擎加模式开关优于每玩法重写结算"的方法论仍成立，仅"用哪套引擎"已更新。

**C. 竞价货币：仅星石，碎片不参与（加碎片的 6 个问题）**

1. **碎片离实物/真钱最近**：`exchange_channel_prices` 117 条计价全是 `red_core_shard`，碎片背后挂着实物/RMB 参考价。给"最接近真钱的资产"做公开竞价 = 监管最忌讳的"有价值资产二级市场信号"。星石是纯点券，无此问题。
2. **与固定汇率套利**：碎片→星石是固定汇率，拍卖会冒出"市场隐含碎片价"，与固定价不一致即被套利，搞乱封闭经济。
3. **混币种无法同台竞价**：100 碎片 vs 5 星石谁高？没汇率没法比 → 要么拆成两场劈裂流动性，要么引入实时比价。
4. **逼出 `星石→碎片` 反向估值**：混拍需要碎片↔星石比价，直接顶到生死线（严禁星石→碎片反向）。
5. **代码复杂度翻倍**：`system_settings` 费率/限额本就按币种各一套，双币 = 冻结/退款/限额/对账全部 ×2，测试矩阵翻倍。
6. **与"碎片不流通"定性自相矛盾**：文档已定碎片"京豆化、不可交易"，给它开竞价是文档自相矛盾的把柄。

> 碎片已有两个健康出口（兑换实物 + 单向转星石），无需拍卖出口。**最干净格局：星石走商城+拍卖（点券消耗），碎片只走兑换实物（京豆），两线互不串。** 若想给碎片竞争性玩法，更安全的替代是**固定价限量秒杀/限购**（官方定价、先到先得，不产生价格发现），而非竞价。

### 10.11 价值流向统一规则 + 资产×商品 能力矩阵（代币风险总闸门）

> 这是判定"某资产能不能买某东西"的唯一权威依据，也是上架时校验 `price_asset_code` 的代码标准。**关 C2C 不等于代币风险归零——风险地基是"资产能否换到真钱"，由本表焊死。**

**A. 价值排序与流向铁律**

价值排序：`实物 ≈ 水晶/碎片系（有实物锚）` > `星石（纯点券）` > `虚拟道具（零价值）`

- ✅ **允许**：高价值 → 低价值 / 同级（销毁方向，资产被消耗）
- ❌ **严禁**：低价值 → 高价值（套现方向，资产凭空获得价值）

代币风险来源（与 C2C 无关，关 C2C 后仍需各自守住）：① 套现路径（星石买到有价值物）② 官方回购（道具卖回换资产）③ 反向转换（星石→水晶/碎片）④ 法币通道（已确认无 ✅）。**四条任一破裂，星石即获货币属性，代币风险全回。**

**B. 资产 × 商品 能力矩阵（上架/购买硬校验依据）**

| 资产 \ 可购买 | 实物（兑换商城） | 零价值虚拟道具（补签卡/置顶/皮肤等） | 拍卖竞价 | 反向换有价值资产 |
|---|---|---|---|---|
| **水晶系（碎片/源晶）** | ✅ 可换 | ✅ 可买（向下销毁） | ❌ 不参与（§10.10） | ❌ 禁 |
| **星石** | ❌ **绝对禁**（套现红线） | ✅ 可买 | ✅ 唯一竞价币 | ❌ 禁 |

要点：
- **补签卡/置顶卡/皮肤等零价值道具**：水晶、星石**都能买**（两条都是向下销毁，安全）。
- **实物礼品**：**只有水晶系能换，星石永远碰不到**（守命门）。
- **拍卖**：只用星石；水晶/碎片不参与。
- 三种币对**有价值侧**的反向通道一律禁止。

**C. 代码层硬校验（上架/定价时焊死，fail-closed）**

```js
// services/shared/AssetProductGuard.js（示意）—— 上架/改价统一调用
const STONE = 'star_stone'
const isCrystal = code => /_core_(shard|gem)$/.test(code)
// 商品是否"有价值"：实物/券，或有人民币参考价
function isValuable(tpl) {
  return ['product','voucher'].includes(tpl.item_type) || Number(tpl.reference_price_points) > 0
}
function assertPriceAssetAllowed(tpl, priceAssetCode) {
  // 红线①：星石永不得为"有价值商品"计价（杜绝星石→实物套现）
  if (priceAssetCode === STONE && isValuable(tpl))
    throw new BusinessError('星石不可用于购买有价值商品', 'STONE_BUY_VALUABLE_FORBIDDEN', 400)
  // 红线②：兑换商城实物仅限水晶系计价
  if (isValuable(tpl) && !isCrystal(priceAssetCode))
    throw new BusinessError('实物/券类仅限水晶系兑换', 'VALUABLE_NEEDS_CRYSTAL', 400)
  // 零价值虚拟道具：star_stone / 水晶系 均可（向下销毁，放行）
}
```

- 该守卫在 **`exchange` 直购上架、兑换商城上架、`exchange/bid` 竞价发起** 三处统一调用，从源头堵死"星石买实物"。
- 与 §10.4-C1（`assertVirtualPropOnly`）、C3（转换反向守卫）三件套构成代币风险的代码层防火墙。
- 固化进 §7.3 月度自查：断言"无 `price_asset_code=star_stone` 且商品 valuable 的上架记录"。

### 10.12 双币价值隔离设计原理（对照原神 / 王者）+ 法币边界

> 目的：解释"**为什么坚持两种币、为什么星石不能碰实物**"，防止后人以"合并成一种币更简单"为由拆掉这道隔离墙。核心手法 = **分币种 + 单向阀门做价值隔离**。

**A. 行业对照：硬通货 vs 软通货，故意不混**

| 项目 | 硬通货（高价值/碰钱侧） | 软通货（玩乐侧） | 隔离手法 |
|---|---|---|---|
| **原神** | 创世结晶（真钱充值）→ 单向变原石（抽卡） | 摩拉（遍地刷，只升级） | 三层货币、**层层单向不可逆**：钱→结晶→原石后永远回不到钱 |
| **王者** | 点券（真钱，买皮肤/限定） | 金币（打对局赚，买英雄） | 皮肤**只能点券买**，金币再多也够不到 → "玩出来的"够不到"花钱的" |
| **你的项目** | 水晶系（抽奖产出，换实物） | 星石（抽奖+水晶转，买零价值道具/拍卖） | 实物**只能水晶换**，星石永远够不到 → 同款"玩乐侧够不到价值侧" |

要点：原神/王者**完全有能力做成一种币**，刻意不做——因为"分币种"本身就是价值隔离手段：一旦软通货能买到硬通货才能换的东西，软通货立刻被赋予真钱价值，灰产即来套现。

**B. 你比原神/王者更保守（也更安全）**

- 原神/王者的硬通货是**真钱买的**（创世结晶/点券），其合规靠"真钱进来后单向锁死、出不去"。
- **你的硬通货水晶不是真钱买的，是抽奖产出的** → 你**连真钱入口都没有**，价值无从注入，比它们更干净。
- 因此你的安全公式不是"管好真钱通道"，而是更彻底的：**全系统零真钱 + 星石永远够不到实物**。这也是为什么"合并成一种币"绝不可行——一旦合并，那一种币既能换实物又能自由花/拍卖，立刻同时具备"真实价值 + 自由流通"，代币风险全回。

> **设计铁律（不可拆）**：水晶 / 星石两币分立是**价值隔离墙**，非冗余。任何"统一为一种币""让星石能换实物""给水晶开真钱购买"的提议，都是拆承重墙，一律拒绝。

**C. 法币边界规则（防止有人随手开内购口拆掉防火墙）**

系统**默认零真钱**：无充值、无提现、无任何法币入口（§7 已确认，零支付 SDK）。这是全部合规结论（星石非代币、EDI 大概率免）的地基。

- 卖虚拟道具收真钱本身**合法**（王者卖皮肤同理），但对本项目会**捅破法币防火墙**，连带需要支付资质 + 税务/发票 + 消费者保护 + 可能的预付卡/网络文化资质。
- **致命红线**：严禁"真钱 → 星石/水晶"（给代币标购买价 = 星石立刻变钱）、严禁"真钱 → 道具 → 桥接回星石/水晶/实物"（变相充值套现）。
- **若未来确需真钱内购**：该道具必须是**绝对死胡同**（买完即焚、不可退现、不可转让、与星石/水晶/抽奖经济**零交叉**），且做成**完全独立模块**，并**先经执业律师确认资质**。
- 建议：**保持现状（道具只用星石/水晶买，不碰真钱）** —— 防火墙完整、合规叙事最干净。

> 🔴 **2026-06-03 补充（广告位人民币定价）**：本节"卖虚拟道具收真钱会捅破防火墙"的逻辑**同样适用于广告位**。广告位若收人民币，等于平台开了一条真钱经营线。**只要满足"人民币↔星石零兑换、人民币不进 `account_asset_balances`"，星石防火墙不破**；反之"人民币→星石"则致命。完整评估见 §10.17。

> ⚠️ 本节涉及具体资质与监管口径，**最终以执业律师确认为准**；本文仅作架构层风险拆解。

#### 10.14-A 决策 9'：官方竞价引擎——"复用现存 B2O" vs "改造 C2C"

**业界怎么分（核心判据：B2C/B2O 单向售卖 与 C2C 撮合 是两个不同的限界上下文，从不混用）**

| 阵营 | 代表 | 竞价/拍卖怎么落地 | 对你的启示 |
|---|---|---|---|
| **大厂电商** | 阿里（闲鱼 C2C / 天猫 B2C 各自独立）、美团 | B2C 自营与 C2C 二手**物理隔离成两套系统**；自营"竞拍/秒杀"复用自营商品中心，绝不接 C2C 撮合栈 | 官方竞价要挂在**自营商品中心**（你的 `exchange_items`），不是 C2C 挂单表 |
| **游戏大厂** | 腾讯（王者）、米哈游（原神）、网易 | 同一套"活动售卖引擎"用 **mode 开关**切直购/限时/竞拍；DNF「拍卖行(C2C)」与「点券商城(B2C)」**永远两套代码** | "一套官方售卖引擎 + 模式开关"是主流；你的 `exchange`(直购)+`exchange/bid`(竞价) 正是这个形态 |
| **小公司/活动策划** | 中小 SaaS、营销活动外包 | 常**原地改造**省人力：把 C2C 拍卖表硬改成"官方专用"，留一堆死字段（`seller_user_id` 填个假官方 ID） | 这正是原决策 9 的"改造 C2C"路线——**最省一次性人力，但长期技术债最高**（孤儿字段、语义错位） |
| **游戏虚拟物品/小众二手** | BUFF、IGXE、DD373、Steam 市场 | 重 C2C 撮合 + K 线行情，**方向与你相反** | 反面教材，不可参考 |

**映射到你的现状**

- 你**已经有**一套天生为官方而生的竞价引擎（`exchange/bid`：卖家=官方 `created_by`、无 `seller_user_id`、中标款 `settleFromFrozen` 直接销毁、商品来自 `exchange_items`、存量 0 行）。这**就是大厂/游戏公司的标准形态**（自营商品中心 + 官方竞价模式）。
- "改造 C2C `auction_*`"是**小公司省钱路线**：把为用户间撮合写的 ④卖方入账 / `transferItem` 用户间转移 / `seller_user_id NOT NULL` 自拍校验逐条拆解，拆完还和已存在的 B2O 引擎功能重复。

**→ 结论：选「复用现存 `exchange/bid` B2O 引擎」（即你强烈建议的前者）。** 理由完全成立且有实测支撑：
1. **最贴合你的技术栈**：`exchange/bid` 已是 `/api/v4/{domain}` 扁平化原生分域、走 ServiceManager + TransactionManager + `res.apiSuccess`、字段全 snake_case，零架构摩擦。
2. **长期维护最低**：一套官方竞价逻辑、一处维护；`services/auction/`(C2C) 可随 C2C 一并删除，不留"改造后的混血引擎"。
3. **符合大厂/游戏公司主流**（自营商品中心 + 官方竞价），远离小公司"原地改造留死字段"的债务路线。

> 落地补丁仅 2 处（详见决策 13）：①竞价货币锁 `star_stone`；②`exchange_items` 上架接 §10.11-C 价值流向白名单。`services/auction/` 与 `auction_*` 表进入"随 C2C 删除"清单（决策 2 相应收敛）。

#### 10.14-B 决策 11：直购型"道具商城"——新建 `/prop-shop`+`prop_shop_orders` vs 复用 `exchange` 域

**业界怎么分（核心判据：一个"官方→用户单向售卖"的能力，用几个限界上下文承载）**

| 阵营 | 代表 | 直购/商城怎么落地 | 对你的启示 |
|---|---|---|---|
| **大厂电商** | 阿里、京东、美团 | 自营商品**一个商品中心 + 一个订单中心**，"兑换/积分商城/秒杀/拍卖"都是同一商品中心上的**售卖模式**，不为每种玩法建独立订单表 | 一个商品中心（`exchange_items`）+ 一个订单记录（`ExchangeRecord`）承载多种售卖模式 |
| **游戏大厂** | 王者、原神 | 点券商城 = 一套 SKU + 一套购买订单，道具/皮肤/礼包都是 SKU 的不同 `item_type`，**不会为"皮肤"和"补签卡"各建一张订单表** | 用 `item_type`/`category` 区分道具种类，订单表共用 |
| **小公司/活动策划** | 中小团队 | 两种：成熟的复用一张订单表；不成熟的"每上一个新功能建一套新表"→ 表爆炸、对账噩梦 | "每个玩法一套表"是典型反模式 |
| **DDD/微服务实践** | —— | 限界上下文按"业务不变量"切，不按"页面"切。"官方售卖"不变量相同（扣资产→发货→不可退）就该在一个上下文 | 直购与竞价不变量相同（官方出货、单向、买即消耗），应同域 |

**映射到你的现状**

- 实测：`exchange` 域**本身就是固定价直购引擎**——`POST /api/v4/exchange/` 按 `cost_asset_code+cost_amount` 扣减、写 `ExchangeRecord`（`order_no`/`pay_asset_code`/`pay_amount`/`status`）。§10.4-B 设想的 `/purchase`（买即消耗、不可退）**语义已经存在**，差别只是当前计价资产是 `red_core_shard`。
- 新建 `/prop-shop` + `prop_shop_orders` = 在已有"官方售卖"上下文之外**再造一个职责重叠的上下文**，正是大厂和 DDD 都避免的"按页面切上下文"。`prop_shop_orders` 与 `ExchangeRecord` 字段会高度重复，将来对账要并两张表。

**→ 结论：选「选项 A·复用 `exchange` 域承载直购，不新建 `/prop-shop`、不建 `prop_shop_orders`」。** 与大厂/游戏公司"一个商品中心 + 一个订单中心 + 多售卖模式"完全一致，技术债最低。落地方式：
- 道具用 `item_type='prop'`（`varchar(50)` 无需改表）登记进 `exchange_items`，计价资产按 §10.11 矩阵：**零价值虚拟道具**允许 `star_stone` 或水晶系；**实物/券**仅水晶系。
- 售卖模式用一个字段区分：`exchange_items` 现有 `min_cost_amount/max_cost_amount` 已能表达"固定价/区间价"；竞价由 `bid_products` 关联同一 `exchange_item_id` 表达。即**直购走 `exchange`、竞价走 `exchange/bid`，两者共享 `exchange_items` 商品中心与 `ExchangeRecord` 订单记录**。
- 产品上若要把"道具商城"与"实物兑换"分开呈现：**用 `category_id`/`space`/`item_type` 在同一域内分类即可**（前端分页签），无需后端分域。

> 这同时让 §10.8 原决策 1（新建 `prop-shop` 独立域）与决策 3（新建 `prop_shop_orders`）**双双收敛为"复用 `exchange` 域 + `ExchangeRecord`"**——原决策当时不知道 `exchange` 已是直购引擎，现据实测修正。**实现以 §10.15 为准（在 `exchange` 域加 `item_type='prop'` 与价值流向白名单，不新建 prop-shop）。**

#### 10.14-C 决策 13：竞价货币白名单——硬编码 `star_stone` vs 新增 `is_biddable` 列

> 背景（实测）：`BidService._getAllowedBidAssets()` 取 `material_asset_types.is_tradable=1 且 ∉{points,budget_points}`，5 分钟缓存。**执行 §10.15 Step 1 把碎片/源晶/星石全设 `is_tradable=0` 后，这个白名单瞬间变空，竞价全瘫**。所以必须改判定来源——这不是可选项，是去交易化的连带必改。问题只是改成什么。

**业界怎么分（核心判据：用"语义复用旧字段"还是"为新语义建专用字段"）**

| 做法 | 业界场景 | 优点 | 缺点 |
|---|---|---|---|
| **硬编码常量**（`IN ('star_stone')`） | 小公司/早期 MVP；规则极稳定且唯一时 | 代码最简、零迁移、改一行 | 将来想"再放开一种竞价币"要改代码发版；规则散在代码里 |
| **专用布尔列**（`is_biddable`） | 大厂/成熟游戏：能力位（capability flag）独立建模 | 运营可在后台开关、规则进 DB 可审计、与 `is_tradable` 解耦语义清晰 | 加列要迁移 + 后台加开关 + model 改动，工作量略大 |
| **能力位集中表**（如 `asset_capabilities`） | 超大厂多能力（可交易/可竞价/可兑换/可转换各一位） | 扩展性最强 | 对你属过度设计（YAGNI） |

**映射到你的现状**

- 你的资产**语义已经分裂**：去交易化后 `is_tradable=0` 表达的是"不可在用户间流通"，而"能否作为官方竞价计价币"是**另一个正交的能力**。继续让竞价白名单读 `is_tradable` 就是**语义耦合**——这正是当前会"瘫痪"的根因（一个字段背了两个含义）。
- 你的诉求是"长期维护成本低、降低技术债"。**硬编码** `IN ('star_stone')` 短期最省，但把"竞价币规则"焊进代码，将来任何调整都要改代码发版，且与"配置进 DB / 运营可控"的现有体系（`system_settings`/`feature_flags`）不一致。
- 你的后端**已大量使用 DB 驱动配置**（`material_asset_types` 本就是资产能力的权威表，已有 `is_tradable`/`form` 等能力位）。**新增 `is_biddable` 列与现有建模风格完全一致**，是"基于现有技术栈"的自然扩展。

**→ 结论：选「新增 `material_asset_types.is_biddable` 布尔列」，仅 `star_stone` 置 1。** 这是大厂"能力位独立建模"的标准做法，且贴合你"配置进 DB、低技术债、不焊死规则"的目标。落地：
1. 迁移加列 `is_biddable TINYINT(1) NOT NULL DEFAULT 0`，`UPDATE ... SET is_biddable=1 WHERE asset_code='star_stone'`。
2. `_getAllowedBidAssets()` 查询条件从 `is_tradable=true` 改为 `is_biddable=true`（黑名单 points/budget_points 可保留作双保险）。
3. 这样去交易化（`is_tradable=0`）与"星石可竞价"（`is_biddable=1`）**互不干扰**，决策 10（仅星石竞价）由数据表达、运营可审计。

> 若你更看重"今天就上、绝不加列"，硬编码 `IN ('star_stone')` 也能合规且能跑，只是把灵活性换成了即时省事——与你"肯一次性投入、要长期低债"的取向相反。**故首选 `is_biddable` 列。**

#### 10.16-C 🔴 前几轮遗漏的 5 条关键发现（含 1 条需你拍板）

**F1（需你拍板）⚠️ 复用 `exchange` 域做道具直购，会"继承一套可退款的订单生命周期"，与"买入即消耗不可退"单向铁律冲突**

实测 `exchange` 域（决策 11 要复用它承载道具直购）的订单生命周期**自带退款能力**，这是前几轮把它当"干净直购引擎"时漏掉的：
- `CoreService.createExchange` 建单后 `status='pending'`（`CoreService.js:398`），**不是 `completed`**——因为 exchange 现服务于"水晶换实物→等发货"，pending=待审核/待发货。
- 存在**三条退款通道**：`cancelOrder`（用户取消 pending 单，退还 `pay_amount`，`CoreService.js:1022/1052`）、`rejectOrder`（管理员驳回退款，`:1130/1160`）、退款方法（`:1244/1277`），全部 `BalanceService.changeBalance(+pay_amount)` 原币退回。
- `ExchangeRecord.status` ENUM 实测含 **`refunded`/`cancelled`/`rejected`**（实库列定义），即天然支持退款态。

> **冲突点**：§10.4-B/§2.2 的单向铁律是"道具买入即消耗、`status` 仅 `created/completed/failed`、**无 refund**、不提供任何 sell-back/recycle"。但若道具直接复用 `exchange` 域，就**继承了 pending + cancel/reject/refund 的整套退款语义**——用户买道具后可能走 `cancelOrder` 把星石退回来，等于变相"官方回收/可回笼"，**直接撞星石货币属性红线**。
>
> **这是决策 11（复用 exchange）落地时必须先解决的、前几轮未暴露的问题。** 给你三个选项拍板（见 §10.16-D）。

### 10.17 🔴🔴 重大合规变更评估：广告位"后续支持人民币定价"（2026-06-03 你新提出，最高优先级）

#### 10.17-B 为什么"人民币定价广告位"是红线级别变更（核心矛盾）

§10.12-C 已立铁律：**系统默认零真钱，这是全部合规结论（星石非代币、EDI 大概率豁免、无金融牌照要求）的地基。** 广告位人民币定价会从两个层面冲击它，取决于你指的是哪种：

**情形一（致命）：商户用人民币购买星石 / 人民币直接充值广告账户 → 桥接到统一星石账本**
- 一旦"人民币 → 星石"通道打开，而星石又能在系统里**买道具、参与竞价、（历史上）参与交易**，则**星石立刻获得"可用真钱购买"的货币属性** → 直接推翻 §0/§10.11/§10.12 的全部结论，星石重新变"代币"，套现链条（人民币→星石→道具/竞价/…）成立。
- 更糟：商户和消费者**共用同一套 `account_asset_balances`**。若商户能人民币充星石，等于给"所有用户的星石"开了一个真钱入口（哪怕只对商户开），价值注入无法隔离。**这是 §10.12-C 明令禁止的"真钱→星石"红线。**

**情形二（可控但需隔离）：人民币只用于广告结算、与星石账本完全不交叉**
- 即广告位标人民币价、商户用人民币付广告费（走真实支付通道/对公转账/线下合同），**这笔钱只进平台对公收入，绝不转成星石、绝不进 `account_asset_balances`、绝不与用户虚拟资产经济有任何兑换关系**。
- 这种"卖广告位收人民币"**本身合法**（和报纸卖版面、App 卖开屏广告同理），但它会**捅破"全系统零真钱"这条最干净的合规叙事**，连带触发一串**新的资质/合规义务**（见 §10.17-C）。

> **判定关键**：人民币能不能、会不会变成星石（或反向）。**能 = 情形一 = 撞红线，禁止；完全隔离 = 情形二 = 合法但要补资质、且必须做成独立模块。**

#### 10.17-C 情形二（合法路线）会触发的新合规义务

一旦平台开始**对商户收人民币**（哪怕只为广告位），就从"零真钱的纯虚拟系统"变成"有真实经营收入的平台"，连带需要：
- **支付资质**：接入微信/支付宝商户号或对公收款，需企业实名、可能需 ICP 经营性备案。
- **税务/发票**：广告费是应税经营收入，需开增值税发票、计税申报。
- **《广告法》合规**：广告主资质审核（营业执照、行业资质）、广告内容审核、不得虚假宣传，平台作为广告发布者有审查义务。
- **互联网广告管理办法**：需可识别"广告"标识、可一键关闭、留存广告档案。
- **经营范围**：营业执照经营范围需包含"广告发布/广告代理"。
- **合同与对账**：商户广告投放协议、人民币计费的对账/退款/争议处理流程（不能复用现在的星石 `refundStarStone`）。

> 这些**与星石/虚拟资产的合规是两条独立的线**。好消息：因为广告人民币与星石隔离，**不影响 §0~§10.16 关于星石/道具/兑换的任何结论**；坏消息：你要额外承担"一个收人民币的广告业务"的全套资质，§2 两句务实提醒里"EDI 大概率豁免、无特殊牌照"的轻量结论，**对广告这块不再适用**。

#### 10.17-G "买服务送积分"模型 + 两条广告通道并存 + 自推算汇率分析（2026-06-03 追加）

**G1 买真实服务/商品 + 附赠积分 = 合法（京豆模型，系统现已在跑）**

实测 `consumption_records`（`models/ConsumptionRecord.js`）：用户消费 → 商家录入 → 审核通过 → 按 **1 元=1 分** 发 `points`（`is_tradable=0`，只进抽奖）。P 图/咨询等服务**结构完全相同**：用户花真钱买**真实服务**，平台**附赠**积分抽奖。生死线：
- ✅ 合法："买 P 图服务 **赠** 500 积分"——人民币买的是服务，积分是赠品。
- ❌ 不合法："充 500 元 = 500 积分"——那是直接拿钱卖积分/代币。
- 服务必须真实提供、定价对得起服务；积分继续不可交易、只进抽奖；遵守 §4 有奖销售合规（概率公示、留存≥2 年、奖池无实物）。

**G2 两条广告通道可并存，均不破红线**
- **通道①（线下人民币 → 运营后台发时长）**：广告主线下/对公付**人民币给广告业务**（经营范围含广告，合法收），运营在后台**手动发放广告时长道具**。人民币只进对公收入，**不转星石、不进 `account_asset_balances`**。= 路线 A。
- **通道②（站内用赚来的星石兑换广告时长）**：用户用抽奖/转换得来的星石兑换广告时长，是星石的又一**消耗出口**，与"星石买道具/竞价"同样安全。
- 两条并存的前提红线：① 人民币永不变星石；② 时长道具纯消耗、**不可退星石/不可转让/不可卖回**；③ 见 G3。

**G3 线下人民币标价 + 站内只标星石（含同款广告位双标价）→ 合法，"汇率可推算"仅次要观感、非违法（2026-06-04 校准）**

> 📌 **本条校准**：早期把"两边同款同价 = 事实上挂汇率"列为需刻意规避的"灰区"，**措辞偏重**。据京豆等成熟先例校准如下——**同款广告位线上星石/线下人民币并存，合法，无需为此把两边硬做成不同产品**。

- **定性依据**：法律判一个虚拟资产是否代币/类货币，看三条——①能否用钱买到它 ②能否变现/提现/转让流通 ③发行方是否官方设定人民币面值。**你三条都不踩**。"汇率能否被算出"**不在判定要件内**；用户脑内推算的比率没有任何出口（不能卖、不能赎、不能用钱买星石），算得出 ≠ 变得现。
- **行业先例（同款双标价合法）**：**京东京豆**——同一台手机可纯人民币买、也可京豆抵扣，京东**自己明示** 1 豆≈0.01 元；航空里程同理。它们汇率可算、同款双价，**照样合法**，因为积分不可提现、不可转让流通。**你比京豆更保守**：京豆能换"可转卖的实物"，你的星石只能换"用完即焚、不可转卖的广告曝光"。
- **结论**：**同一个固定位 / 同一个信息流曝光包，线上星石、线下人民币，可以并存**。"两边同款同价"顶多是懒政监管多看两眼的**叙事观感**，实质不构成违法。
- **真正的命门（守住这三条即稳，与汇率无关）**：① 永不开"人民币→星石"；② 星石付的曝光**自用、用即消耗、不可退星石/不可转让/不可卖回**；③ 站内**只标星石、不主动写"折合 XX 元"**（不官方挂面值）。

> **一句话（校准版）**：同款广告位线上星石、线下人民币**合法**（京豆即先例，且你因"曝光不可转卖"更干净）。命门不是"汇率能不能算"，而是**「星石买不到 + 曝光不可变现转让 + 站内不标人民币面值」**——守住即稳，**无需为汇率把两边硬拆成不同产品**。

- 理由：① 少一个中间物 → **彻底绕开《单用途商业预付卡》监管联想**（无"卡/储值/余额"形态）；② 少一层资产、对账简单；③ 就是系统现状（4 种计费模式全星石），**零新建**。

**四条红线（决定合法性，必须全焊死）**
1. 星石**买不到**：永无"人民币 → 星石"通道（现状满足，零支付 SDK）。
2. 星石付的广告曝光**自用、用即消耗**：不可退回星石、不可退现、不可转让第三方、不可卖回平台。
3. **单用途**：星石付的就是这次广告曝光，不串作通用余额。
4. **不反向**：广告曝光不可变回星石/水晶/人民币。

**合法退款 vs 禁止回购（代码层须分清，勿误删 `refundStarStone`）**
- ✅ 保留：广告主**冻结预算但未投放**就取消/被拒 → 解冻退回星石（退"没花出去的钱"，托管释放，非回购）。
- ❌ 禁止：**已生效/已消耗的广告曝光** → 退回星石（=官方回购，给星石装"可回笼"货币属性，叠加广告的 RMB 锚即接通套现链）。
- 界限：**钱没花出去可退；曝光已买到手不可退回星石。**
- **实现要点（关键，防做错）**："未使用可退"必须实现成**解冻冻结的星石**（escrow 释放），**不要铸"权益包/卡"再回收**。即：出价/预约 → 星石**冻结**（不销毁、不铸新资产）→ 未投放取消则**解冻**、投放则 `settle` 正式扣除。**全程无独立"权益包"资产 = 根本不存在"回购"动作**，"未使用退款"天然等于解冻、合法。一旦先把星石铸成"权益包"再换回，哪怕没用过，那个动作本身就是回购（=红线）。

**配套（与币种无关，照常做）**
- 站内**只标星石、绝不写"折合 XX 元"**（硬红线）。
- 线下收人民币广告费：开票、广告主资质审核、广告内容审核、广告可识别标识+可关闭+档案留存（《广告法》《互联网广告管理办法》，§10.17-C）；经营范围已含广告 ✅。
- 广告主与消费者共用同一星石账本：在"人民币不进系统"前提下**无需隔离**（路线 A）。
- 把"星石付广告曝光"登记进 §7.3 月度自查，作为星石合法消耗出口之一。

> **一句话总览**：线下两种广告位均 RMB 定价（合同/对公/运营发放），线上两种均**直接星石支付**（不做卡）。守住"星石买不到 + 曝光自用不可退星石不可转让不反向"，星石仍是点券、非代币。**人民币与星石永不兑换——这条不可让渡。**

### 10.19 价值流动总闸 + 以物易物玩法 + 竞品对照（2026-06-04 讨论沉淀，本轮新增）

> 本节把"X 能不能换回 Y""C2C/以物易物能不能做""竞品为什么能做"等反复纠结的问题，收敛成**单一权威判据 + 一表查清**，挂在 §10.11 价值流向闸门之下。与上文冲突处，以本节判据为准。

#### 10.19-A 核心判据（一切"能不能换/能不能转"的唯一标尺）

**判据一·价值变现 = 入口 × 出口**（套现风险只在两头都通时成立）
- **入口墙**：永不开"真钱 → 星石/水晶"（系统零真钱，§7/§10.12）。
- **出口墙**：用户间不可转移（无 C2C）+ 任何资产换出的东西不可回流为货币。
- **任一墙在，套现链即断。** 本项目两道墙都在 → 比"有真钱入口"的竞品低一个量级的风险。

**判据二·看对手方是谁**
- **用户 ↔ 官方** = 正常经营兑换（以旧换新、兑换、直购）→ **不算监管定性的"交易/二级市场"**。
- **用户 ↔ 用户** = 财产在用户间转移 + 可私下收钱 → **C2C = 二级市场/场外套现红线**。

**判据三·看换回来的是不是"可再花费的货币型资产"**
- 换回 **实物 / 不可交易虚拟道具**（终点消耗品）→ 安全。
- 换回 **星石 / 积分 / 水晶 / 碎片 / 源晶 / 配额**（任何 `material_asset_types` 资产）→ **"可回笼"红线**（§2.3 回收的真正判据，不是"官方收不收旧物"，而是"有没有货币回流给用户"）。

#### 10.19-B 价值单向流动总表（一表查清"X 能不能换 Y"）

| 链条 | 判定 | 理由 |
|---|---|---|
| 水晶 → 实物（实物 `is_tradable=0`，只能用/收藏） | ✅ 安全 | 终点消耗，价值向下 |
| 旧实物 → 新实物（以物易物，产出不可交易） | ✅ 安全 | 终点⇄终点，用户↔官方，无货币回流 |
| 旧实物 → 不可交易虚拟道具 | ✅ 安全 | 向下，无货币回流 |
| 碎片/水晶 → 星石（现有 6 条规则） | ✅ 安全 | 高价值→低价值，单向销毁 |
| 星石/水晶 → 零价值道具 | ✅ 安全 | 向下销毁 |
| 水晶 → 实物 → **换回水晶** | ❌ 禁 | 水晶变"可存可取"=货币（§3.2 焊死） |
| 实物/道具 → **星石/积分** | ❌ 禁 | §2.3 回收（货币回流） |
| 任何 → **水晶/碎片/星石/配额** 作为产出 | ❌ 禁 | 产出货币型资产=可回笼 |
| 星石 → 实物/券（有价值物） | ❌ 禁 | §10.11 套现红线 |
| 用户A 物品 → 用户B（封装/分享/挂卖/拍卖） | ❌ 禁 | C2C，用户间财产转移 |

> **代码守卫**：任何兑换/以物易物/道具购买/竞价的**产出**，硬断言 `NOT IN material_asset_types`（星石/积分/水晶/碎片/源晶/配额全禁作产出）且 `is_tradable=0`；**禁止借中转绕过**（不能"实物折算成水晶再花"，中间出现货币型资产即违规）。

#### 10.19-C 「以物易物」玩法（✅ 已决定要做，B2C 官方合成，非 C2C）

> 起因：竞品（东西潮玩）有"以物易物"功能。经判据核验，**其 B2C 形态可借鉴且已决定纳入本项目**（但其"封装/分享 CD-key"是 C2C 转移，**不可碰**）。执行步骤见 §10.15 阶段六。

**思路**：`用户旧实物 ──放入──▶ 官方(核销/销毁旧物) ──发放──▶ 官方库存新实物 ──▶ 用户`。全程**用户↔官方**，旧物销毁、价值向下闭环，**不涉及用户间财产转移、无货币回流**，故安全。本质 = "水晶→实物"同源的 B2C 兑换，复用 `exchange` 域即可，无需新建领域。

**4 条落地守卫（必须全焊死）**：
1. 只能 **用户↔官方**，永远没有用户↔用户（旧物进官方池销毁，新物从官方池出，两个用户永不直接接触）。
2. 旧物必须**真销毁/核销**（`items` 状态置 consumed/burned），不可复活、不可退回。
3. 产出**只能是不可交易消耗品（实物/虚拟道具，`is_tradable=0`）**，**禁止任何货币型资产**（星石/积分/水晶/碎片/配额），**不可借货币中转**。
4. 方向**等价或向下**（不可"破烂换高价值"形成变相价值放大/射幸）。

**前置条件**：抽奖入口本身需合规（§4/§四-A）；实物履约走常规义务（发货/消费者保护/有奖销售折算上限）。**正式上线前以执业律师确认为准。** 执行步骤见 **§10.15 阶段六（Step 14~18，P1）**。

#### 10.19-D C2C 为何不做 + 竞品对照 + "不可提现≠可做 C2C"

**为什么 C2C 不做（即使走实物）**：
- C2C 合法化要把你刻意砍掉的全补回来：EDI 经营许可、反洗钱(KYC)、消费者保护、可能的支付资质，且只要标的能间接变现就撞**代币/二级市场**红线（国内近乎死路）。
- **C2C 是你附加玩法、非主营**，为它扛全套资质 + 代币风险，投入产出严重倒挂。官方竞价（§10.10）已能满足"竞价玩法"体验，无需开 C2C。

**"不可提现" ≠ 可做 C2C（两道墙缺一不可）**：
- "不可提现"只挡**站内**直接变现，挡不住 C2C 的**场外套现**：用户B 私下微信转钱给用户A → A 在站内把资产转给 B。钱走系统外、资产走系统内，平台成场外撮合台。
- 也挡不住 C2C **赋予资产市价**（可流通+有价=类代币）。**安全 = 不可提现（挡站内）+ 用户间不可转移（挡场外），两道墙都要在。**

**竞品对照（东西潮玩，为何"它能做"不等于"你能抄"）**：
- 它交易的是**真钱买盲盒抽来的实物**（二手转售范畴）；你的实物是**免费抽奖/水晶兑换**来的（零真钱成本）→ 你一旦 C2C，等于接通"免费抽奖→实物→卖钱"的套现桥（正是 §3.2 焊死的）。
- 且竞品**入口有真钱（赌资）+ 出口能套现**，本身处盲盒强监管灰区、带风险跑，**不是合规样板**。你真正的护城河是**入口零真钱**——这是它没有、你有的，务必死守（永不开"真钱→星石/水晶"）。
- **可借鉴**：它的「以物易物」（B2C 官方合成，见 §10.19-C）；**绝不可碰**：它的「封装/分享 CD-key」（C2C 转移）。

> ⚠️ 本节涉及代币/盲盒/赌博/套现的监管定性，且含对竞品法律结构的推测，**最终一律以执业律师为准**；本文仅作架构层风险拆解。
