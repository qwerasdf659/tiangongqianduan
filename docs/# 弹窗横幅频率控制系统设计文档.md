# 弹窗横幅频率控制系统设计文档

## 解决什么问题

当前抽奖页面（`pages/lottery/lottery`）的弹窗横幅（popup-banner）存在以下问题：

1. **每次进入页面都会弹出横幅**：用户关闭后，切换页面再回来、或页面销毁重建后，横幅会再次弹出
2. **没有频率控制机制**：`loadPopupBanners()` 只过滤了 `is_active + 时间范围`，没有任何"已看过就不再弹"的逻辑
3. **频率规则写死在前端**：如果未来要改弹出策略（比如从"每天一次"改成"每3天一次"），必须修改前端代码并发版
4. **所有 banner 一视同仁**：系统公告、活动推广、日常促销没有分级，无法针对不同类型设置不同的弹出策略

本文档设计一套**服务端驱动 + 客户端执行**的弹窗频率控制系统，让运营通过后台配置即可灵活控制每个 banner 的弹出策略，前端无需发版。

## 行业调研摘要

| 公司类型 | 代表 | 频率控制方式 | 状态存储 | 特点 |
|---|---|---|---|---|
| 大厂 | 美团/淘宝/微信 | 服务端 `frequency_rule` + 客户端兜底 | Redis + localStorage | 优先级队列、用户分群、A/B测试、全链路埋点 |
| 小公司 | 创业团队 | 客户端 `localStorage` 日期判断 | 纯客户端 | 简单够用，无埋点无分群 |
| 游戏公司 | 米哈游/网易 | 服务端已读状态 + 分类分级 | 服务端为主 | 公告强制弹、活动弹一次、维护不可关闭 |
| 活动平台 | 活动行/Eventbrite | 生命周期驱动 | 混合 | 跟着活动阶段走，支持"稍后提醒" |
| 交易平台 | 闲鱼/Steam | 事件触发 + 合规强制 | 混合 | "今日不再显示"、交易安全提醒不可跳过 |

**结论**：对于本项目（微信小程序积分抽奖系统），采用**游戏公司的分类分级思路** + **大厂的服务端规则配置**，由客户端执行频率判断，是性价比最高的方案。

## 系统设计

### 一、Banner 类型分级

在 `popup_banners` 表新增 `banner_type` 字段（避免与 Sequelize 保留字 `type` 冲突），不同类型对应不同的默认行为：

| banner_type | 名称 | 使用场景 | 默认行为 |
|---|---|---|---|
| `notice` | 系统公告 | 系统维护、规则变更、紧急通知 | 强制弹出，必须点击"我知道了"才能关闭，关闭后永不再弹同一条 |
| `event` | 活动推广 | 新活动上线、限时抽奖、节日活动 | 每个活动期间弹一次，关闭后不再弹 |
| `promo` | 日常促销 | 积分翻倍、新品推荐、优惠信息 | 按 `frequency_rule` 控制弹出频率 |

> **设计说明**：字段命名为 `banner_type` 而非 `type`，因为 Sequelize 默认将 `type` 用于 STI（Single Table Inheritance），使用 `type` 会导致模型层冲突。现有 `SystemAnnouncement` 模型的 `type` 字段能正常工作是因为它没有启用 STI，但为避免潜在风险和歧义，新字段统一命名为 `banner_type`。

### 二、频率规则（frequency_rule）

后端接口 `GET /api/v4/system/popup-banners` 返回的每个 banner 新增以下字段：

```json
{
  "popup_banner_id": 15,
  "title": "春节积分翻倍活动",
  "banner_type": "promo",
  "image_url": "https://objectstorageapi.bja.sealos.run/br0za7uc-tiangong/popup-banners/xxx.jpg",
  "display_mode": "square",
  "image_width": 750,
  "image_height": 750,
  "link_url": "/pages/lottery/lottery",
  "link_type": "page",
  "frequency_rule": "once_per_n_days",
  "frequency_value": 3,
  "force_show": false,
  "priority": 100,
  "start_time": "2026-02-01T00:00:00+08:00",
  "end_time": "2026-02-28T23:59:59+08:00"
}
```

#### 支持的 frequency_rule 枚举值

| frequency_rule | frequency_value | 含义 | 适用场景 |
|---|---|---|---|
| `always` | - | 每次进入页面都弹 | 紧急维护公告、强制阅读的通知 |
| `once` | - | 整个活动周期只弹一次 | 活动上线通知、一次性公告 |
| `once_per_session` | - | 每次小程序冷启动弹一次 | 每日登录提醒、欢迎弹窗 |
| `once_per_day` | - | 每天最多弹一次 | 日常促销、每日推荐 |
| `once_per_n_days` | N（天数） | 每 N 天弹一次 | 周期性活动提醒（如每3天、每周） |
| `n_times_total` | N（总次数） | 累计最多弹 N 次 | 新功能引导、限次推广 |

#### 新增字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| `banner_type` | VARCHAR(20) | 是 | `'promo'` | banner 类型：`notice` / `event` / `promo` |
| `frequency_rule` | VARCHAR(30) | 是 | `'once_per_day'` | 频率规则枚举值 |
| `frequency_value` | INT | 否 | `1` | 频率参数（配合 `once_per_n_days`、`n_times_total`） |
| `force_show` | BOOLEAN | 否 | `false` | 是否强制弹出（不可通过点击遮罩关闭，必须点按钮） |
| `priority` | INT | 否 | `0` | 弹出优先级（越大越优先），与 `display_order`（排序权重，越小越靠前）职责不同 |

> **`priority` vs `display_order` 区分说明**：现有 `display_order` 控制的是 banner 在管理后台列表和前端轮播中的"排列顺序"（数字小的排前面）；新增 `priority` 控制的是多个 banner 竞争弹出时的"弹出优先级"（数字大的先弹）。两者职责不同，需要共存。

### 三、客户端存储结构

使用 `wx.setStorageSync` 存储用户对每个 banner 的交互记录：

```typescript
// Storage Key
const STORAGE_KEY = 'popup_banner_records'

// 存储结构
interface BannerRecord {
  /** 最后一次展示的时间戳（毫秒） */
  lastSeen: number
  /** 累计展示次数 */
  seenCount: number
  /** 是否被用户主动关闭过 */
  dismissed: boolean
}

// 完整存储示例
{
  "popup_banner_records": {
    "15": { "lastSeen": 1740000000000, "seenCount": 3, "dismissed": true },
    "16": { "lastSeen": 1739900000000, "seenCount": 1, "dismissed": true }
  }
}
```

> **注意**：key 为 `popup_banner_id`（整数转字符串），与后端主键一致。

### 四、客户端频率判断逻辑

```
loadPopupBanners()
  │
  ├─ 1. 调用 GET /api/v4/system/popup-banners?status=active&position=home
  │     （后端已过滤 is_active + 时间范围，现在额外返回频率字段）
  │
  ├─ 2. 读取本地存储 wx.getStorageSync('popup_banner_records')
  │
  ├─ 3. 对每个 banner 执行 shouldShowBanner(banner, record) 判断
  │     │
  │     ├─ frequency_rule === 'always'
  │     │   └─ → 始终返回 true
  │     │
  │     ├─ frequency_rule === 'once'
  │     │   └─ → record.seenCount === 0 时返回 true
  │     │
  │     ├─ frequency_rule === 'once_per_session'
  │     │   └─ → 检查 App.globalData.sessionSeenPopups 是否包含该 popup_banner_id
  │     │
  │     ├─ frequency_rule === 'once_per_day'
  │     │   └─ → record.lastSeen 不在今天则返回 true
  │     │
  │     ├─ frequency_rule === 'once_per_n_days'
  │     │   └─ → (今天 - record.lastSeen) >= N天 则返回 true
  │     │
  │     └─ frequency_rule === 'n_times_total'
  │         └─ → record.seenCount < N 则返回 true
  │
  ├─ 4. 过滤后的 banner 按 priority 降序排序
  │
  ├─ 5. 展示第一个（最高优先级的 banner）
  │
  └─ 6. 展示后更新本地存储：seenCount++, lastSeen = Date.now()
```

### 五、关闭行为处理

```
用户关闭 banner
  │
  ├─ banner_type === 'notice' && force_show === true
  │   └─ 必须点击"我知道了"按钮才能关闭
  │       └─ 写入 record: { dismissed: true, seenCount: 1 }
  │           └─ 永不再弹（因为 frequency_rule 通常是 'once'）
  │
  ├─ banner_type === 'event'
  │   └─ 点击关闭按钮或遮罩层
  │       └─ 写入 record: { dismissed: true, seenCount++ }
  │           └─ 按 frequency_rule 决定是否再弹
  │
  └─ banner_type === 'promo'
      └─ 点击关闭按钮或遮罩层
          └─ 写入 record: { dismissed: true, seenCount++, lastSeen: now }
              └─ 按 frequency_rule 决定下次何时再弹
```

## 存储清理策略

本地存储不能无限增长，需要定期清理过期记录：

- 每次读取 `popup_banner_records` 时，删除 `lastSeen` 超过 90 天的记录
- 或者在小程序 `onLaunch` 时执行一次清理
- 预估存储量：每个 banner 记录约 100 字节，100 个 banner 约 10KB，远低于微信小程序 10MB 的存储上限
