## 八、后端答复（2026-06-21，直连真实库 `restaurant_points_dev` 核查后给出）

> 核查方式：直读后端路由 `routes/v4/shop/redemption/**`（全部 6 个端点）+ Node.js 直连真实库核对 `redemption_orders` 表结构与 1662 条真实数据分布。结论以真实代码 / 真实数据为准，不引用文档旧结论。

### 8.1 先纠正文档旧表述（§8.4.2 / §8.5 / §12.5 B 的"接口已就绪"不准确）

前端实测正确：**后端当前没有任何"按门店聚合核销数量"的查询接口。** `shop/redemption/*` 实际只有 6 个端点：

| 端点 | 维度 | 能否做"本店核销概况" |
| --- | --- | --- |
| `GET /shop/redemption/me` | 按本人 `redeemer_user_id` | ❌ 按兑换人本人，非门店 |
| `GET /shop/redemption/orders/:order_id` | 单条详情 | ❌ |
| `GET /shop/redemption/items/:item_id/order` | 按物品 | ❌ |
| `POST /shop/redemption/orders`、`/fulfill`、`/scan` | 创建 / 执行核销 | ❌ 非查询 |

§8.4.2 / §12.5 B 写的"`shop/redemption/*` 按 store 聚合、接口已就绪"是**当初未落到真实代码的乐观结论，属错误表述**，以本章为准修正。

### 8.2 更关键：当前数据模型下"本店待核销数"这个口径**不成立**（业务模型问题，非缺接口）

直连真实库核对 `redemption_orders` 表（1662 条）：

- 表**没有 `store_id`（兑换码归属门店）字段**，只有两个门店相关字段：
  - `fulfilled_store_id` — **核销发生的门店**（在哪个店把码核掉的）
  - `fulfilled_by_staff_id` — 执行核销的店员
- 二者都是**核销动作发生时才写入**的。真实数据印证：

| status | fulfilled_store_id 为空 | 有值 | 合计 |
| --- | --- | --- | --- |
| pending（待核销） | **69** | 0 | 69 |
| fulfilled（已核销） | 5 | 0 | 5 |
| cancelled | 1 | 0 | 1 |
| expired | 1587 | 0 | 1587 |

**两个硬事实：**
1. **待核销单（pending）天然没有"所属门店"**：兑换码是用户在 C 端兑换生成的，核销前没去任何店，`fulfilled_store_id` 必然为 NULL（69/69 全空）。所以"**本店**待核销数"在当前模型里**无法计算**——待核销的码不属于任何店，它属于用户，可拿去任意门店核销。
2. 连已核销单的 `fulfilled_store_id` 当前也全是 NULL（5/5）——即核销流程虽有这两个字段，但**尚未确认是否真正回填**（需另查 `fulfill`/`scan` 写入逻辑），所以"本店已核销数"目前也聚合不出有效数据。

> 结论：这不是"补个聚合接口"就能解决的，而是**业务模型与口径问题**——"本店核销概况"依赖"兑换单归属哪个门店"，而当前兑换码模型是"用户持码、任意门店可核销"，待核销阶段根本没有门店归属。

### 8.3 需要你（业务 / 产品）先拍板的口径问题

请先明确"本店核销概况"到底要统计什么，再决定后端怎么做：

**问题 A：「本店待核销数」按什么归店？**（当前模型下待核销码无门店归属）
- 选项 A1：取消"待核销"这一项，**只统计"本店已核销数"**（按 `fulfilled_store_id` 聚合，语义清晰：本店核掉了多少单）——推荐，无歧义、不改模型。
- 选项 A2：把"待核销"重新定义为"**本店可核销的兑换单**"（如该兑换商品限定本店核销）——但当前兑换码模型是通用核销码、不绑门店，需先确认业务上兑换码是否要绑定门店，属较大业务改造。
- 选项 A3：暂不做"本店核销概况"这一项，小程序看板就保留已对接的"核心数字 + 超时预警"两项（§8.4.2 三项缩为两项）。

**问题 B：「本店已核销数」的 `fulfilled_store_id` / `fulfilled_by_staff_id` 是否已在核销流程真实回填？**
- ✅ 已查实（`RedemptionService.js` L323-383、L766、`RedemptionAdminService.js` L116）：核销执行（fulfill/scan）**已有写入逻辑**——优先取传入 `options.store_id`，否则按核销操作人的 `store_staff`（active）绑定自动匹配 `fulfilled_store_id` + `fulfilled_by_staff_id`，并做"物品商家与核销门店商家一致性校验"。
- ⚠️ 但真实库 5 条 fulfilled 的 `fulfilled_store_id` 全为 NULL 的根因：这些是**历史/测试核销，当时操作人无 active `store_staff` 绑定**（正是 §8.6 第 5 项那 5 个店长当前都 inactive 导致）→ 走到 L345"门店字段为空"分支。**激活真实店长（§8.6-5）后，新核销会正常回填门店**，按门店聚合即有效。历史 NULL 数据可选迁移回填或忽略。

### 8.4 后端可提供的接口（待你拍板 8.3 后落地，不自行臆造）

- 若选 **A1（只做已核销）**：后端新增 `GET /api/v4/shop/consumption/merchant/stats` 内并入核销字段，或新增 `GET /api/v4/shop/redemption/store-stats`，返回形如：
  ```
  data: { redemption: { fulfilled_count: number } }   // 本店已核销数（按 fulfilled_store_id + 登录身份门店隔离）
  ```
  并按登录身份做门店/区域隔离、不下发兑换用户 PII（与 §12.5 C 一致）。
- 前提：先确认 8.3-B 的 `fulfilled_store_id` 已真实回填。

> 在你就 8.3 拍板前，后端**不擅自新增接口、不臆造字段、不回填数据**（避免做出与真实业务口径不符的统计，违反"先验证后假设、不造假数据"原则）。

### 8.5 一句话回前端

**「本店核销概况」当前无法对接，且不是缺接口那么简单**：`redemption_orders` 表没有 `store_id`，只有"核销发生门店 `fulfilled_store_id`"，而**待核销单还没核销、门店归属必然为空（真实数据 69/69 全 NULL）**，故"本店待核销数"在现模型下算不出来；已核销单的 `fulfilled_store_id` 当前也全为空。这是业务模型/口径问题，需先由业务拍板（见 8.3：建议只做"本店已核销数"或暂缓本项），后端再按确认口径补接口。文档 §8.4.2 / §12.5 B 的"接口已就绪"表述有误，已在本章修正。

## 九、业务决策：新增「门店专属兑换券」业务线（2026-06-21 拍板定稿，精简版）

> 拍板结论（2026-06-21）：不走 A1/A3 的"绕过"，而是**正式新增第二条业务线**，让"本店核销概况"从根上成立。
> - 业务线 1：**通用兑换物**（现状，任意门店可核，不动）
> - 业务线 2：**门店专属兑换券**（新增，券限定门店，只能在限定门店核销）
> 节奏：**先定稿本章方案、你最终确认后再写代码**（design_first），本章为编码依据，未开始编码。
> 设计基准：参照卡券核销行业标准（微信卡券/美团券/星巴克）+ 最大化复用本项目现有结构（`RedemptionService`、`fulfilled_store_id`、商家一致性校验、`expires_at` 到期机制），**最低字段数、最低技术债**。

### 9.1 已拍板的业务规则（最终选择）

| 决策点 | 最终选择 | 行业依据 |
| --- | --- | --- |
| **专属粒度** | 支持"指定门店集合" + "商家全门店"两种 | 微信卡券 location_id_list / 星巴克 BY_BRAND |
| **用户选门店时机** | **核销时落地**（券在允许范围内，到哪个店核就算哪个店；**不做兑换时预选**） | 微信/美团/星巴克卡券标准：券通用持有、使用时落地 |
| **merchant_all 商家** | **取商品自身商家**（不另存字段，复用现有"物品商家 vs 核销门店商家"一致性校验） | 99% 卡券场景，语义自洽 |
| **到期未核销** | **与通用券完全一致**（沿用现有 `expires_at` / `status='expired'`，无特殊处理） | 卡券行业标准：到期作废、积分不退 |
| **谁来建/配专属券** | **平台运营在 Web 后台建商品时统一配**（商家不自助配） | 运营集中管控 |
| **用户可兑范围** | **所有用户都能兑**（兑了之后限门店核销，兑换页不按门店过滤） | 通用商城，门店约束只在核销端 |
| **指定门店来源** | **从现有 active 门店列表勾选**（复用现有门店列表，编码时校验门店存在且 active） | 复用现有 5 门店 |
| **本店核销概况谁能看** | **店长(manager)看本店**；**店员(staff)需店长批准后才能看**（见 9.8 授权机制） | 门店内分级查看 |
| **落地节奏** | design_first | 本章定稿 → 你确认 → 再编码 |

> 相比初版砍掉：`scoped_merchant_id` 字段（复用现有商家校验）、`require_user_pick` 字段 + 兑换预选逻辑（改为核销时落地）、`redemption_orders.scoped_store_ids`/`scoped_merchant_id`（用允许集合一个 JSON 表达即可）。能力不减，字段从 7 个降到 3 个。

### 9.2 核销范围模型（一个枚举三态，覆盖两种粒度）

用 `applicable_scope` 一个字段统一表达核销范围，`all` 为默认 → 现有商品/核销零影响：

| `applicable_scope` | 含义 | 核销校验口径 |
| --- | --- | --- |
| `all` | 通用（业务线1，默认） | 任意门店可核，不校验门店 |
| `specified_stores` | 限指定门店集合 | 核销门店 ∈ 允许集合 |
| `merchant_all` | 限商家全门店 | 核销门店所属商家 == 商品所属商家（复用现有一致性校验） |

> 命名定稿：采用卡券行业通用词 `applicable_scope`（与微信卡券/美团同义）。**最终字段名 = `applicable_scope`，全链路统一使用，不再有备选。**

### 9.3 字段改动（精简版，走 migration，snake_case）

**① `exchange_items`（兑换商品定义）加 2 个字段**
- `applicable_scope` ENUM('all','specified_stores','merchant_all') NOT NULL DEFAULT 'all' —— 核销范围类型（区分两条业务线的总开关；默认 all = 现有商品全是通用券，零影响）
- `scoped_store_ids` JSON NULL —— `specified_stores` 时的允许门店集合（如 `[7,8,9]`）；其它类型为 NULL
- `merchant_all` 不需要额外字段：直接取商品自身商家归属

**② `redemption_orders`（核销单）加 1 个字段**
- `scoped_store_id_list` JSON NULL —— 核销码**生成时**从商品固化的允许门店集合（通用券 NULL；`merchant_all` 存该商家全部门店 id 或留标记 + 核销时按商家校验）
- 已有 `fulfilled_store_id`（核销落地门店）**复用不变**

> 设计理由：生成核销码时把"允许门店范围"固化进 order（不回查商品，避免商品改配置影响存量券），与项目"实例化固化"既有模式一致（审核链 step 固化 approve_mode 同理）。核销门店 `fulfilled_store_id` 仍按现有逻辑在核销时写入。

### 9.4 流程改动（顺着现有 RedemptionService，不另起炉灶）

1. **兑换生成核销码**（`RedemptionService` 创建 order 处）：按商品 `applicable_scope` 固化允许范围到 order（`specified_stores` 写 `scoped_store_id_list`；`all`/`merchant_all` 按规则处理）。**不要求用户选门店**。
2. **核销校验**（`RedemptionService` fulfill/scan，在现有"商家一致性校验"旁加一道门店校验）：
   - `all`：不校验门店（现状不变）。
   - `specified_stores`：当前核销门店 ∈ `scoped_store_id_list`，否则拒"此券限指定门店核销"。
   - `merchant_all`：复用现有"物品商家 == 核销门店商家"校验（已存在，L351），无需新增。
3. **本店核销概况**这下成立（核销时门店落地，范围在生成时固化）：
   - 本店待核销 = `status='pending' AND 本店 ∈ scoped_store_id_list`（或 merchant 匹配）→ "能来本店核的待核销券"
   - 本店已核销 = `status='fulfilled' AND fulfilled_store_id=本店`

### 9.5 接口与前端

- **小程序兑换页**：**无需改动**（不做兑换时选门店）。

### 9.6 兼容与影响面

### 10.2 二次核查推翻 §9 的关键事实（§9.3 必须改）

直连真实库逐条核对（数据为 2026-06-21 实测）：

**事实 1：核销单挂的是「物品实例 `items`」，不是「兑换商品 SPU `exchange_items`」。**
`redemption_orders.item_id` 外键 → `items.item_id`（见 `models/RedemptionOrder.js` L71-79、`RedemptionService.createOrder(item_id)` 锁的是 `Item.findByPk`）。`exchange_items` 主键是 `exchange_item_id`，二者是不同的表。
→ §9.3「在 `exchange_items` 加 `applicable_scope` 然后核销时读它」的链路**断了**：核销单根本不直接认识 SPU。

**事实 2：核销单来源五花八门，绝大多数不来自兑换商城。**
真实 1662 条核销单按 `items.source` 分布：`legacy`=1321、`test`=319、`lottery`=19、`exchange`=**仅 3**。按 `items.item_type`：`voucher`=1640、`product`=21、`prop`=1。
→ "门店专属券"若只在 `exchange_items`（兑换商城 SPU）上配置，只能覆盖极少数 exchange 来源的券；抽奖/历史发放的 voucher 实例拿不到任何门店范围。**范围必须固化在核销单自身**，不能依赖回查 SPU。

**事实 3：商家一致性校验在真实数据里从不生效（全是平台自营）。**
`items.merchant_id` 全部为 NULL（7572/7572），`stores.merchant_id` 也全为 NULL（5/5），`merchants` 表仅 1 条。`RedemptionService` L351-371 的"物品商家==门店商家"校验，因两边都是 NULL 实际从未触发。
→ §9.2 的 `merchant_all`（商家全门店）口径**当前无数据支撑、无法验证**，本期落地它属于"为不存在的场景写代码"。建议本期只做 `all` + `specified_stores` 两态，`merchant_all` 留枚举位但不实现校验分支（见 10.4 拍板点）。

**事实 4：`fulfilled_store_id` / `fulfilled_by_staff_id` 回填逻辑已存在，但当前 5 个 store_staff 全是 inactive。**
真实库 `store_staff` 共 5 条，全部 `role_in_store='manager'` 且 `status='inactive'`，全部属于同一个 `user_id=31`。`RedemptionService` L327-349 的自动匹配只认 `status='active'`，所以现存 fulfilled 单门店字段才全是 NULL（与 §8.3-B 一致）。
→ 不是代码缺陷，是没有 active 员工。**激活任意一个 manager 后核销即会回填门店**，"本店已核销数"立即可聚合。

**事实 5：`exchange_items` 已有 `fulfillment_type ENUM('physical','virtual','voucher')`，但真实库 0 条 voucher。**
64 条 SPU 中 61 physical、3 virtual、**0 voucher**。
→ 门店专属券是"voucher 履约 + 门店范围约束"的组合，本期需要运营真正建出 voucher 商品，否则前端看板永远是 0。这是配置/运营动作，非代码。

### 10.4 流程改动（顺现有 RedemptionService / CoreService，修正 §9.4 的取数链）

**难点（必须正视）：`RedemptionService.createOrder(item_id, ...)` 只拿到 `items` 实例，怎么知道这张券限哪些门店？**
因为 `redemption_orders.item_id → items.item_id`，而 `items` 表没有 `exchange_item_id` 也没有门店字段。只有 `exchange_records`（兑换记录）同时持有 `exchange_item_id` + 产出的 `item_id`（见 `models` 实测 comment）。所以固化范围有两条可行路径：

- **路径 A（推荐，改动最小且语义最准）**：在 `createOrder` 的调用方把范围算好传进来。
  - 由谁调 `createOrder`？兑换链 `CoreService` 铸造出 `item` 后、`exchange_records` 已知 `exchange_item_id`，此处最清楚这张券的 `applicable_scope`/`scoped_store_ids`。
  - 改 `createOrder(item_id, { transaction, creator_user_id, scoped_store_id_list })`：新增可选入参，直接写进 `redemption_orders.scoped_store_id_list`。调用方（兑换流程）从 `exchange_items` 读 `scoped_store_ids` 传入；其它来源（lottery/legacy/admin 手工）不传 = NULL = 通用券。
  - 优点：核销单一次固化、与商品后续改配置解耦、对 lottery/legacy 零影响。
- **路径 B（不推荐）**：核销单存 `exchange_item_id`，核销时回查 SPU。违背"实例化固化"既有模式，且 lottery/legacy 券无 SPU 可查，回查会 NULL。否决。

**核销校验**（`RedemptionService.fulfillOrder` / `adminFulfillOrderById`，在现有门店匹配 L323-349 之后加一道）：
- `scoped_store_id_list` 为 NULL → 通用券（`all`），不校验门店（现状不变）。
- `scoped_store_id_list` 非空 → 当前 `fulfilledStoreId` 必须 ∈ 该集合，否则 `throw new BusinessError('此券限指定门店核销', 'REDEMPTION_STORE_NOT_ALLOWED', 400)`。`specified_stores` 走这一条。
- `merchant_all`（已拍板 M1，靠商家归属）：`scoped_store_id_list` 传 NULL，核销时复用现有"物品 `merchant_id` == 核销门店 `merchant_id`"校验（L351-371），M1 已让 `items.merchant_id=6` 有值，校验即生效。

**本店核销概况（取数口径，全部落在 `redemption_orders` 单表）：**
- 本店已核销数 = `COUNT(*) WHERE status='fulfilled' AND fulfilled_store_id = :store_id`（语义最稳，立即可用，依赖事实 4 激活 active 员工）。
- 本店待核销数 = `COUNT(*) WHERE status='pending' AND JSON_CONTAINS(scoped_store_id_list, CAST(:store_id AS JSON))`（仅门店专属券计入；通用券 NULL 不计入"本店待核销"，符合 §9.5）。
