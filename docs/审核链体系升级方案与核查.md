# 审核链体系升级方案与核查（消费/商家积分审核）

> 文档演进说明：本文档最初仅核查“店长/店员被审核接口门槛挡死”这一 bug，后经多轮需求确认，已扩展为**审核链体系的整体升级方案**——涵盖权限门槛、门店/区域数据隔离、会签、越级留痕、超时升级、双端数据看板、基础管理（门店/员工/层级）修复与 Web 增强。故 2026-06-20 改名为《审核链体系升级方案与核查》。

> 核查方：后端数据库项目
> 核查方式：直读实际代码（`routes/v4/console/config/approval-chain.js`、`middleware/auth.js`、`services/ApprovalChainService.js`、`models/ApprovalChainNode.js`、`routes/v4/shop/consumption/**`）+ Node.js 直连真实数据库（`restaurant_points_dev @ dbconn.sealosbja.site`）核对审核链节点、角色等级、store_staff 表与消费记录门店关联，未引用任何历史报告/备份
> 核查时间：2026-06-20（北京时间）
> 一句话结论：审核链存在**两个互相关联的真实问题**——
> 1. **门槛挡死**：配置层允许把店长（merchant_manager, lv40）、店员（merchant_staff, lv20）配为审核人，但唯一审核接口门槛 `requireRoleLevel(60)` 会在业务校验前 403 挡住 lv60 以下用户 →“配得了、审不了”。
> 2. **门店越权（更严重）**：审核权限校验只看“有没有这个角色”，**完全不校验门店**。一旦让店长用角色池方式审核，5 个门店的店长会**互相能审对方门店的发积分单**（跨门店越权操作资金）。
> 故：若要让店长/店员参与审核，**降门槛与门店隔离必须同时做**，只降门槛会把“审不了”变成“跨店越权”，制造更大漏洞。

---

## 一、问题本质（一句话）

**配置层与执行层的权限门槛不一致**：
- 配置层（运营建审核链时）：审核角色下拉列出全部角色，**可选店长（lv40）、店员（lv20）**
- 执行层（实际点审核时）：审核接口要求 **role_level ≥ 60**，店长/店员被中间件挡在业务校验之外

## 四、影响与触发条件

- **触发条件**：运营在审核链配置页给任一节点选了店长（lv40）或店员（lv20）作为审核角色。
- **后果**：被指派的店长/店员登录后，调用审核接口（或查"我的待审"）会收到 403 `INSUFFICIENT_LEVEL`，**无法审核、甚至看不到待办**。该审核节点成为"无人可审"的死结，整条审核链卡住（除非配了超时自动推进，或由 admin 越级顶替）。
- **当前风险等级**：中（暂未配置 lv60 以下节点，但配置入口开放，随时可能踩中）。

## 五、与"小程序店员/店长审核功能"的关系

- 后端**唯一审核入口在 console 域**（`/api/v4/console/approval-chain/steps/*`），门槛 lv60。
- 消费记录的**提交**在 shop 域（`/api/v4/shop/consumption/submit`，商户端/小程序），但其代码注释明确写："审核功能位于 /api/v4/console/consumption（仅限 admin）"——即**提交在小程序侧、审核在管理端侧**。
- 因此：**若微信小程序前端确实开放了"店长/店员审核"页面，它最终调用的仍是这个 lv60 门槛的 console 接口**，对 lv40/lv20 的店长店员会调用失败。小程序前端不在本仓库，其页面是否存在需小程序端确认，但**后端铁证是：lv60 以下无法通过审核接口**。

## 五-补、门店越权隐患（更严重，本轮新增排查）

> 你的提问场景：有 5 个门店，lv40 店长是不是 5 个店的单都能审？**核查结论：是，会跨门店越权。**

### 5补.1 问题本质

审核权限校验 `ApprovalChainService._verifyOperatorPermission` 的“角色池”分支**只判断“操作人有没有这个角色”，完全不校验“这笔业务属不属于他管的门店”**：

```1253:1262:services/ApprovalChainService.js
if (step.assignee_role_id) {
  const hasRole = userRoles.some(ur => ur.role_id === step.assignee_role_id)
  if (!hasRole) {
    throw new BusinessError(`您不具备当前步骤要求的审核角色...`)
  }
  return  // ← 只要有该角色就放行，不看门店
}
```

**后果**：5 个门店各自的店长都挂 `merchant_manager` 角色 → 任一店长都能审全部 5 个店的消费单。A 店店长能批 B/C/D/E 店的发积分单 = 跨门店操作他人门店资金。当事人回避（`exclude_parties`）也挡不住——A 店店长不是 B 店那笔消费的当事人，回避放行、角色校验又不看门店，畅通无阻。

### 5补.2 `submitter_manager` 分配方式是“定义了但未实现”的半成品

`models/ApprovalChainNode.js` 的 `assignee_type` 枚举有第三个值 **`submitter_manager`**，注释写明“利用 store_staff 表自动查找提交人所在门店的 manager”——**它本就是为门店隔离设计的**（谁的店谁审）。但实际代码里：

- **创建步骤**（`createChainInstance`，166-167 行）：只处理 `'user'` 和 `'role'`，`submitter_manager` 既不写 `assignee_user_id` 也不写 `assignee_role_id` → 会创建出两者皆 null 的步骤。
- **权限校验**（`_verifyOperatorPermission`）：只处理 `assignee_user_id`、`assignee_role_id` 两种，`submitter_manager` 落到末尾 `抛错“无法确定审核人分配方式”`。
- **通知解析**（`_resolveAssigneeUserIds`，1335 行）：只处理 user/role，`submitter_manager` 返回空数组（连“通知谁审”都做不到）。

即：选了 `submitter_manager` 不仅不隔离门店，反而直接报错不可用。**这是个未完整实现的设计，正是本应解决门店隔离的那块缺失。**

### 5补.3 门店隔离的数据基础已具备（直连数据库核对）

- `store_staff` 表完整：含 `user_id` + `store_id` + `role_in_store enum('staff','manager')` + `status` → **完全支持“按门店反查店长/店员”**。
- 真实门店总数 = **5**（与提问场景一致）。
- `consumption_records` 的 `store_id` **100% 有值**（52 条全部非空）→ 每笔消费都能定位到门店，门店隔离有数据可依。
- `merchant_points`（商家积分申请）以申请人 user_id 为主体，无门店维度，门店隔离主要针对 `consumption`（消费审核）。

### 5补.4 结论

**门店隔离当前完全缺失。** 若按方案 A 降门槛让店长参与审核，必须同时补门店隔离，否则“审不了”会变成“跨店越权”——这是比原问题更严重的资金安全漏洞。

## 六、需要你拍板的根本业务问题

> ✅ **已拍板（2026-06-20）：**
> 1. **店长（lv40）/店员（lv20）参与消费审核 → 采用方案 A**（支持“店员初审→店长复审→管理员终审”等多级链）。
> 2. **门店隔离 → 采用路线一**（补全 `submitter_manager` 分配方式，按门店自动派人，行业标准做法）。
> 3. **配套两项都做**：① ops(lv30) 只读角色排除审核；② 高权限越级代审留痕。
> 4. 审核接口门槛 `requireRoleLevel(60)` → **降到 lv20**（精准放行，非 lv>0 一刀切）。
> 下方方案 A / 路线一 / 6.x 即最终落地依据；方案 B 仅作存档对比，不采用。

**业务上到底要不要让店长（lv40）、店员（lv20）参与消费/商家积分审核？**

### 方案 A：要让店长/店员参与审核（推荐，但必须同时做门店隔离，按下文 6.1 + 6.3 落地）
- 调整审核接口门槛：把 `requireRoleLevel(60)` **降到店员等级 lv20**（而非 lv0/“已登录即可进”一刀切，原因见 6.2），**把关交给 Service 层 `_verifyOperatorPermission` 的精确角色校验**（它本就按节点配置的角色精确判断谁能审）。
- **必备前提（门店隔离，见 6.3）**：必须同时补门店范围校验，否则降门槛后 5 个店长会互审对方门店的单（跨店越权，见五-补）。
- 影响范围：后端审核接口（approve/reject/batch/my-pending）门槛调整 + 门店隔离逻辑 + 回归测试；前端无需改（小程序若有审核页，调通后即可用）。
- 优点：配置层与执行层一致，"配了谁、谁就能审"；支持"店员初审→店长复审→管理员终审"等多级链；门店隔离后“谁的店谁审”，无跨店越权。

### 方案 B：不让店长/店员参与，审核仅限 business_manager（lv60）及以上
- 保持接口门槛不动，**改配置层**：审核链配置页"审核角色"下拉**过滤掉 lv60 以下角色**，从源头禁止运营配店长/店员，消除矛盾。
- 影响范围：仅前端下拉过滤 + 可选后端 createTemplate/updateTemplate 增加"节点角色 ≥lv60"校验。
- 优点：审核权收口到中层及以上，权限更集中。

> 两个方案都能消除矛盾，区别在于**业务上谁该有审核权**。这是业务规则决策，需你拍板。

### 6.1 推荐落地：门槛降到 lv20（店员），不要用“lv>0 一刀切”

| 方案 | 谁能进审核路由 | 评价 |
| --- | --- | --- |
| 现状 lv60 | business_manager 及以上 | 店长/店员被挡死（即本文 bug） |
| **lv20（推荐）** | 店员（20）及以上 | 覆盖所有“可能当审核人”的角色，挡掉无关角色，保留纵深防御 |
| lv>0 一刀切 | 除普通用户外全部（含 ops/抽奖角色等） | 过度放开：攻击面大、语义乱（见 6.2） |

- 降到 **lv20** 后：店员(20)、店长(40)、业务经理(60)、区域(80)、admin(100) 均可进路由；仍挡住 lv0 普通用户、lv10 抽奖角色(campaign_2)、lv-1 系统任务这些**绝不可能审核**的角色。
- 进路由后仍由 Service 层 `_verifyOperatorPermission` 按节点配置的角色精校，决定“能不能审这个节点”——保留“粗过滤 + 精校”两层纵深防御。
- `my-pending`、`steps/:id/approve`、`steps/:id/reject`、`steps/batch` **四个接口门槛同步降到 lv20**，保持一致。

### 6.2 为什么不采用“lv>0（除 lv0 外都能进）一刀切”

“lv>0 都能进”不会直接造成越权（Service 层角色精校仍兜底），但放得过宽，带来 4 个问题：

1. **丢纵深防御**：等于废掉粗过滤层，把关全压在 Service 一层；该层将来若被改出 bug，无第二道防线。审核=发积分（发钱），单层防御风险偏高。
2. **攻击面扩大**：lv10 抽奖角色、lv30 ops 只读等**与审核无关**的角色也能打到审核写接口，多了被探测/刷接口的入口。其中 **ops(lv30) 是只读角色，却能进审核写接口路由，语义自相矛盾**。
3. **`my-pending` 语义混乱**：一堆无审核职责的角色都能查“我的待审核”，查出来多为空，产品语义乱。
4. **放大死锁面**：店员/店长常常**正是业务当事人**，叠加 `exclude_parties` 当事人回避会被拒，更容易出现“配了某低权限角色审、却因当事人回避无人可审”的死结。

### 6.3 门店隔离落地方向（方案 A 的必备前提，针对五-补的越权隐患）

> ✅ 已拍板：**采用路线一（补全 submitter_manager）**。行业依据：连锁多门店场景（大厂/连锁 SaaS 如美团管家、客如云）的标准做法是“审批人按门店动态指派 + 门店级数据隔离”，路线一正是这一标准的实现，且贴合本项目既有设计（`submitter_manager` 本就为此而设，只是没做完）。路线二仅作对比存档，不采用。

- **路线一（推荐）：补全 `submitter_manager` 分配方式**
  - 在 `createChainInstance` / `_verifyOperatorPermission` / `_resolveAssigneeUserIds` 三处实现 `submitter_manager`：审核时通过 `store_staff` 反查“该消费单所属门店的 manager / staff”，只有属于该门店的店长/店员才能审。
  - 优点：天然“谁的店谁审”，运营配置选“提交人店长”即可，无需手动指定；彻底消除跨店越权；正好补上这个半成品设计。
  - 工作量：中（补 3 处逻辑 + 建议审核步骤冗余存 store_id 便于校验与查询）。
- **路线二：角色池 + 门店范围校验**
  - 保留“角色池”分配方式，但在 `_verifyOperatorPermission` 角色校验通过后**追加门店校验**：操作人须在 `store_staff` 中与该消费单 `store_id` 同店且在职。
  - 优点：改动集中（主要在校验函数加门店判断）。缺点：仍需运营理解“角色池+门店”的隐含语义。

> 两条路都依赖 `consumption_records.store_id`（已具备）与 `store_staff`（结构完整）。`merchant_points` 无门店维度、不受影响。**最终采用路线一**（见本节顶部已拍板说明）。

## 七、附带建议（不阻塞，供参考）

- 无论选 A 或 B，**配置层与执行层的权限标准必须统一**（要么都允许低等级、要么都禁止），避免"配得了、审不了"。
- 若选 A（推荐 lv20 精准放行），建议同时落地两项配套：

  - **高权限越级代审留痕**：admin 顶替低等级节点时打标记、记录原应审角色、可选理由必填，保证发积分这类敏感审核可追溯。

| **区域负责人区域隔离** | ✅ 按**组织层级 `user_hierarchy`** 判定 | 区域负责人只能审/看其在 `user_hierarchy` 中管辖门店（`store_id`）的单，**不按地理行政区**（省市区）判定 |
| **会签（同节点 N 人通过）** | ✅ 本轮就做 | 节点可配“N 人会签”，凑够 N 个 approve 才推进；任一 reject 整体拒绝 |
| 越级代审留痕 | ✅ 做 | admin 越级顶替低等级节点时打标记、记录原应审角色、理由必填 |
| ops(lv30) 只读角色 | ✅ 排除审核 | 即便降门槛，ops 不可成为审核人 |
| **web 管理端统计看板** | ✅ 完整看板 | 按门店/区域的待审/已审/通过率/超时等 |
| 审核接口门槛 | lv60 → **lv20** | 精准放行，非 lv>0 一刀切 |

### 8.2 区域隔离数据口径（已核查 + 已定）

系统存在两套“区域”口径，**已定采用组织层级**：
- ✅ **采用：组织层级 `user_hierarchy`**（字段：`user_id` + `superior_user_id` + `role_id` + `store_id` + `is_active`）——后台手动指派“区域负责人管哪些门店”，按此做隔离。贴合运营实际管辖关系，可灵活调整。
- ❌ 不采用：地理行政区划（`stores.province_code/city_code/district_code`）——那是地址级联用的，不等于运营管辖范围。

> 含义：区域负责人能审/看的门店 = `user_hierarchy` 中其管辖（`store_id`）且 `is_active=1` 的门店集合。

### 8.3 会签（多人审批）落地要点

- **节点模型 `approval_chain_nodes` 新增字段**：如 `approve_mode`（single 单签 / countersign 会签）+ `required_approvals`（会签需通过人数）。
- **步骤执行模型 `approval_chain_steps` 改造**：当前为“一步一人一结果”，会签需支持“一个节点记录多人审核结果”（方案：会签节点拆多条 step，或新增会签子记录表），凑够 `required_approvals` 个 approve 才推进；任一 reject 即整体拒绝。
- **`processStep` 流转改造**：approve 后不立即推进，先判定“是否已满足会签人数”。
- ⚠️ **风险提示**：会签改动 `approval_chain_steps` 核心数据模型 + 推进逻辑，影响现有 in_progress 实例，是本次升级**风险最高**的部分，需充分回归测试 + 迁移兼容现有数据。

### 8.6 落地前仍需补确认的细节（2026-06-20 已拍板，附行业对比）

> 以下 4 项原为待定，现结合大厂/SaaS/连锁门店行业做法已拍板，理由见各条。

1. 会签节点的"原应审角色/指定人"如何与 N 人配合：**采用"角色池满足 N 人即可"**（不指定 N 个具体人），与门店隔离叠加后即"本店该角色凑够 N 个 approve"。
2. **会签 + 门店隔离凑不齐 N 人的兜底 → 采用"上级补位"**。
   - 行业对比：钉钉/飞书"审批人为空自动转上级"；BPM 引擎用完成条件表达式；连锁门店 SaaS 取"在职人数与 N 的较小值"。
   - 选"上级补位"理由：本店在职审核人不足 N 时，缺额由上级（区域负责人/管理员，经 `user_hierarchy.superior_user_id`）补足——复用已拍板的"超时转上级"同一机制，零新概念；比"降单签"更安全（发钱审批不削弱管控）；不卡死。
3. **admin 越级在会签节点语义 → "一次越级直接满足整个会签节点"**。
   - 行业对比：钉钉/飞书超管"强制通过"一键结束节点并留痕；BPM 的 admin-skip 是节点级。
   - 理由：越级是异常兜底通道，一刀一节点最简、不把越级名额混进会签进度状态；已拍板"越级必留痕+理由必填"保证可追溯；避免异常通道叠正常流程增技术债。
4. **小程序"审核总数"口径 → "范围内全部消费记录数（待审+已审结）"**。
   - 行业对比：美团商家/抖音来客等 C 端工作台"我的数据"普遍 = 待办+已办，给一线完整工作量感知；通过率类分析才用"已审结"做分母。
   - 理由：`merchant/stats` 现成即返回此口径，**后端零改动**；符合店员/店长直觉；通过率等分析按归属放 Web 管理端。
5. **5 个 inactive 店长 → 一并激活为 active（上线前数据准备）**。
   - 说明：真实库 `store_staff` 现有 5 个 manager 均 `status='inactive'`，门店隔离按"在职"过滤审核人，不激活会"配了店长审却无人可审"。
   - 落地要求：先确认这 5 条对应真实在岗店长（排除测试/离职数据），通过第九章修复后的员工管理接口激活（不手改生产库），顺带验证员工管理流程。

### 8.6.1 超时接管语义改造（2026-06-20 新增拍板）

> 背景：店员/店长有“超时未审则上级接管代审”的诉求。经核查，**现状与诉求不符**，需改造。

- **现状（已核查）**：`ApprovalChainTimeoutService._escalateStep` 的 `escalate` 动作 = “标记当前步 timeout → `advanceToNextStep` 推进到**下一个审核链节点**”，即**跳过当前步**，不是“转给上级审同一步”。
- **✅ 已拍板改造**：超时行为改为 **“升级给上级代审”**——店员超时 → 转交其**上级（店长）**审同一笔；店长超时 → 转交其上级（区域负责人/管理员）。
- **数据基础**：`user_hierarchy` 表含 `superior_user_id`（上级）+ `store_id`，可据此找“当前审核人的上级”。
- **要点**：超时升级须仍受门店/层级隔离约束（升级给的是“管辖该单的上级”，不是任意上级）；升级动作要留痕（谁因超时被升级、转给了谁）。
- ⚠️ 这是对超时服务核心逻辑的改造，需兼容现有 in_progress 实例与定时任务（每 30 分钟扫描）。

### 8.6.2 “只看/只审自己范围”隔离强化（与门店/区域隔离同批）

> 经核查，`getPendingStepsForUser`（“我的待办”查询）**当前只按“分配给我的角色/用户”过滤，无任何门店/层级隔离** → 角色池模式下同角色的人会互相看到、互相能审对方门店的待办（串信息/串审核）。

- **✅ 须改造**：`my-pending` 查询 + `_verifyOperatorPermission` 校验都叠加门店/层级范围过滤：
  - 店长/店员：只看/审本门店（`store_staff` + `consumption_records.store_id`）。
  - 区域负责人：只看/审其 `user_hierarchy` 管辖门店集合内的单。
  - 杜绝“串信息、串审核、帮别人审”。

### 8.6.3 看板新增“超时未处理”维度（2026-06-20 新增拍板）

> 现状 `shop/consumption/merchant/stats` 与审核统计只按 `status`（pending/approved/rejected）分组，**无“超时”独立维度**。

- **✅ 已拍板**：统计看板（8.4）增加 **“超时未处理”** 维度（基于 `approval_chain_steps.timeout_at` 已过期且仍 pending，或 status=timeout 的步骤聚合），按门店/区域展示。

### 11.1 数据库迁移（统一走 `npm run migration:create`，禁止手改生产；新增列全 snake_case）

**① `approval_chain_nodes`（节点定义，加会签配置）**
- `approve_mode` ENUM('single','countersign') NOT NULL DEFAULT 'single' —— 单签/会签
- `required_approvals` INT NOT NULL DEFAULT 1 —— 会签需通过人数（single 恒为 1）

**② `approval_chain_steps`（步骤执行，加门店冗余/会签计数/越级留痕）**
- `store_id` INT NULL —— 冗余该步所属门店（来源 `consumption_records.store_id`），门店隔离校验与统计免回查
- `approve_mode` ENUM('single','countersign') NOT NULL DEFAULT 'single'，`required_approvals` INT NOT NULL DEFAULT 1（实例化时从节点固化）
- `approved_count` INT NOT NULL DEFAULT 0 —— 会签已通过人数
- `is_escalated` TINYINT(1) NOT NULL DEFAULT 0 —— 越级/超时升级代审标记
- `original_assignee_role_id` INT NULL —— 越级时原应审角色（留痕）
- `escalated_from_user_id` INT NULL —— 由谁超时/越级转交而来

**③ 新增 `approval_chain_step_actions`（会签子记录表，"一个 step 多人审批"）**
- `step_action_id` BIGINT PK AI；`step_id` BIGINT NOT NULL（FK→`approval_chain_steps.step_id`，DB 层外键，ON DELETE RESTRICT）
- `actioned_by` INT NOT NULL；`action` ENUM('approve','reject') NOT NULL；`action_reason` TEXT NULL；`is_escalated` TINYINT(1) DEFAULT 0；`actioned_at` DATETIME NOT NULL
- 唯一约束 `uk_step_actor`(step_id, actioned_by) —— 防同一人对同一步重复投票（DB 层幂等）；索引 idx(step_id)
- 设计理由：会签用**子表累加**而非 step 塞 JSON，便于审计 + 并发安全（DB 唯一约束兜底），符合本项目「幂等键 + DB 强约束」规范。

> 存量兼容：迁移给现有 in_progress 实例的 step 填 `approve_mode='single'`/`required_approvals=1`/`approved_count=0`，老逻辑等价不变。

## 十三、9.3 Web 管理端增强落地 + "批量配置审核人"行业方案对比与决策（2026-06-20）

> 第九章 9.3 的 4 项增强本轮全部落地。其中 ③「批量配置审核人」涉及"审批人怎么定"的架构选型，下面先给行业对比与决策依据，再列实现。

### 13.1 行业做法对比（按"审批人怎么确定"分四种范式）

| 范式 | 代表 | 审批人怎么定 | 多门店扩展性 | 换人成本 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| **范式1 规则引擎 + 动态解析人** | 美团/阿里/钉钉/飞书审批、Activiti/Flowable BPM | 不写死人，写"解析规则"（发起人上级 / 门店店长 / 角色），运行时按组织架构动态算 | 一条链覆盖全部门店 | 零（改组织数据即可） | 多门店 / 分级 / 资金审批 |
| 范式2 每组织单元一条流程 | 部分连锁 SaaS（客如云、美团管家早期）、小公司后台 | 每个门店建一条链、逐店挂人 | 线性增长（配置地狱） | 逐条改（高） | 门店少、强差异化 |
| 范式3 纯角色池 | 游戏发行 / 活动策划运营后台 | 有"审核员"角色即可审，先到先得 | 天然全覆盖 | 改角色成员 | 中心化小团队、无门店隔离 |
| 范式4 弱审批 / 后置风控 | 游戏虚拟物品交易、二手 C2C 平台 | 先放行，靠风控 + 事后抽审 + 申诉仲裁 | N/A | N/A | 高频低风险交易 |

### 13.2 本项目选型结论：范式1（规则引擎 + 动态解析人）

理由（基于真实代码，非套话）：

1. 业务是"发积分=发钱"+ 5 门店资金隔离，必须"谁的店谁审"+ 分级 + 区域隔离 → 范式3/4 直接出局；范式2 制造配置地狱、增技术债，违背"长期维护成本低"。
2. **本项目模型天生就是范式1**：`approval_chain_nodes.assignee_type` 的 `submitter_manager`（提交人门店店长）+ `user_hierarchy.superior_user_id`（上级）+ `store_staff`（门店任职）正是"动态解析人"的三块拼图，本次升级已全部接通（submitter_manager 按门店派人、超时转上级、角色池叠加门店/区域隔离）。

### 13.3 因此「③ 批量配置审核人」的正确实现：不碰匹配模型

- **诉求A（5 个门店不想逐店配审核人）**：已被 `submitter_manager` 彻底解决——一条链配一个"提交人门店店长"节点，N 个门店自动按各自店长隔离审核，换店长只在"员工管理"把新店长设 active，流程零改动。**这是 ③ 价值的主体，无需任何新机制。**
- **诉求B（给某类单统一指定/换审核人）**：用"按金额条件匹配的链 + 节点 assignee_type=user/role"已支持，改人只改那条链的那个节点。
- **本轮额外补充的便利工具**：跨多条审核链**批量设置某节点的审核人**（终审/指定步号），一次配置覆盖多条链。这是"批量配置审核人"字面、且**零破坏匹配模型**的实现（只改目标节点的 `assignee_*` 单字段，不重建节点、不新建门店级链表），不引入范式2 的配置地狱。

> 明确不采用：为每个门店建链 + 批量指派工具（范式2），那是给自己造痛点再打补丁，纯增技术债。
