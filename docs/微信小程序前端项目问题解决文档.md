# 前端请求治理方案 · 限流 429「请求过于频繁」问题与客户端治理设计

> 提出方：微信小程序前端
> 问题来源：真机 vConsole 实测——冷启动首页时，`/api/v4/lottery/campaigns/active`、配置缓存、版本闸门检查等多个接口在同一秒集体返回 `HTTP 429 + code: RATE_LIMIT_EXCEEDED`
> 命名规范：全链路 `snake_case`，前端业务逻辑 `camelCase`，以后端数据库为准
> 时间：全链路北京时间（`+08:00`）
> 文档性质：**方案设计 + 责任划分**（待拍板后实施，本文档不含已落地代码）

---

## 一、问题现象（真机 vConsole 实测）

冷启动微信小程序首页（用户**未登录**，`isAuthenticated:false`），同一秒（`01:19:00`）多个接口集体失败：

```
[ERROR][api] 频率限制(429)
[ERROR][api] API请求失败 Error: 请求过于频繁，请稍后再试
响应: { success:false, code:"RATE_LIMIT_EXCEEDED", message:"请求过于频繁，请稍后再试" }

受影响接口（同一秒并发）:
  GET /api/v4/lottery/campaigns/active   （活动列表，needAuth:false）
  配置缓存接口（config-cache）
  版本闸门检查接口
```

> 注：日志里的 `no such file or directory wxfile://usr/miniprogramLog/log1` 是微信开发者工具自身日志组件的告警，与本次 429 无关，不是前端代码缺陷。

## 二、根因数据流图（纯文本）

```
微信小程序冷启动首页（一次正常开屏，onShow 并行发起多个请求）
        │  GET /lottery/campaigns/active   （公开只读，needAuth:false）
        │  GET 配置缓存接口
        │  GET 版本闸门检查接口
        │   ⚠️ 用户未登录：storeHasToken:false / isAuthenticated:false
        ▼
后端限流中间件（按 IP / 设备聚合计数）
        │  判定"单位时间请求数超阈值" → 全部拦截
        ▼
后端统一返回 429 { success:false, code:"RATE_LIMIT_EXCEEDED" }
        ▼
前端 utils/api/client.ts:469 收到 429 → 如实抛错 + Toast "请求过于频繁"
        ▼
结果：campaigns/active、config-cache、版本闸门 同一秒集体 429 失败
```

**关键判断（已核对前端实际代码）：**
1. 前端 `APIClient` 对 429 **直接抛错、不重试、不轮询**（`client.ts:469`），**没有重试风暴**，不是把后端打到限流的肇因。
2. 触发限流的是**一次正常冷启动**的少量并行请求（版本检查 + 配置 + 活动列表），且用户**未登录**、请求的是 `needAuth:false` 公开接口。
3. 多个**不同**接口在同一秒**全部**一起 429 → 后端限流器按 **IP/设备聚合计数**，是后端策略，前端无法干预。

→ **治本在后端限流配置（阈值/窗口/计数维度）；前端只能做"治标 + 体验兜底 + 不自伤"。**

## 五、行业设计对比（按公司类型）

| 公司类型 | 代表 | 客户端请求治理做法 | 本质 |
|---------|------|-------------------|------|
| 大厂电商 | 美团/阿里/京东 | 统一网络层（Mtop/Shark）：请求去重 + 并发控制 + 指数退避+抖动重试 + 熔断降级 + 缓存兜底，业务层无感 | 重型统一网络中台 |
| 大厂内容/社交 | 腾讯/字节 | 客户端 SDK 内置 backoff+jitter，配 `Retry-After`；首屏用 **BFF 聚合接口**（一个请求拿全首屏）从源头减并发 | BFF 聚合 + SDK 退避 |
| 小公司/创业 | 多数 | 基本不治理：429 弹 toast，顶多防连点 | 几乎裸奔 |
| 游戏公司 | 腾讯游戏/米哈游/网易 | 请求队列 + 优先级 + 客户端令牌桶自限流 + 退避重试 + 弱网降级；开服洪峰用**随机错峰延迟**打散 | 客户端自限流 + 错峰 |
| 活动策划/运营平台 | 中小活动平台 | 整点洪峰用**随机抖动错峰** + 简单退避 | 错峰为主 |
| 游戏虚拟物品/小众二手 | BUFF/悠悠有品/交易猫/闲鱼 | 列表行情高频：请求节流 + 短 TTL 缓存 + 失败退避；交易类"宁可慢不可重复提交"（幂等+防连点） | 节流+短缓存+幂等 |

**核心规律（公司越成熟越遵守）：**
1. 请求治理**收口到统一网络层**，不散落到页面（本项目已有 `APIClient`，天然满足）。
2. 退避**必带抖动（jitter）+ 次数上限**——无抖动的重试会让一批客户端"同步重试"形成二次洪峰。
3. 首屏并发要么**聚合（BFF）**要么**错峰**，不让 N 个请求挤在同一毫秒。

## 六、✅ 已拍板的治理方案（2026-06-12 确认）

> 项目未上线、可一次性投入、不兼容旧接口、追求长期低维护 / 低技术债。
> 选型思想：把请求治理**一次性收口到已有的 `APIClient` 单例**，全项目自动生效、新增页面零额外成本。

| 项 | ✅ 选定方案 | 归属 | 一句话理由 |
|----|-----------|------|-----------|
| 1. 失败重试 | **C：指数退避 + 抖动 + 上限**（`delay = base * 2^n + random()`，最多 2~3 次，**仅幂等 GET**） | 微信小程序前端 | 大厂/游戏标准；抖动避免二次洪峰；上限避免自伤；写操作绝不重试（防重复下单） |
| 2. 首屏削峰 | **D：后端 BFF 首屏聚合接口**（一次返回首屏全部数据） | **后端数据库项目** | 从源头减并发，最治本的削峰；但这是后端的活，需后端新增聚合接口 |
| 3. onShow 刷新 | **B：时间窗节流**（距上次刷新 < N 秒则跳过；业务数据仍后端为准，只控刷新频率） | 微信小程序前端 | 削掉频繁切 Tab 的重复请求，不碰业务数据真实性 |
| 4. 请求去重 | **B：in-flight 去重**（相同请求在途时，后来者复用同一个 Promise，不重复发） | 微信小程序前端 | 冷启动多组件同时要同一份配置时只发一次，几十行代码、收益高 |

## 七、前端实施清单（B/C/B 三项，待动手）

> 全部收口到 `utils/api/client.ts` 的 `APIClient`（治理项 1、4）+ 页面/缓存层（治理项 3）。

### 7.1 治理项1：指数退避 + 抖动 + 上限（`APIClient`，仅幂等 GET）

- 触发条件：响应 `statusCode === 429`（可选扩展到网络层 `fail`），**且** `method === 'GET'`（幂等）。
- 退避公式：`delay = baseMs * 2^attempt + random(0, jitterMs)`（如 base=500ms，jitter=300ms）。
- 上限：最多重试 **2 次**（共 3 次请求）；超出仍失败则抛原错误、走现有 Toast。
- 优先读后端 `Retry-After` 头或响应 `data.retry_after_seconds`（若后端提供），有则以其为准。
- 写操作（POST/PUT/DELETE）**绝不自动重试**（与项目"幂等键"规则一致，避免重复兑换/重复提交消费）。

### 7.2 治理项4：in-flight 请求去重（`APIClient`）

- 以 `method + url + 序列化 data` 为 key，维护"在途请求 Map"。
- 相同 key 的请求在途时，后来者直接复用同一个 Promise，不发第二次 wx.request。
- 请求 settle（成功/失败）后从 Map 删除 key。
- 仅对幂等 GET 去重；写操作不去重（每次都是独立业务意图）。

### 7.3 治理项3：onShow 时间窗节流（页面层）

- 在 `pages/lottery/lottery.ts` 的 `onShow` 对 `_loadCampaigns` / `loadHomeAnnouncements` / `_refreshBadgeCounts` 加"距上次刷新 < N 秒（建议 10~15s）则跳过"。
- 节流窗口用 UI 常量管理（前端可自主决定的延迟参数，非业务数据）。
- 仅控刷新频率；下拉刷新（`onPullDownRefresh`）作为用户显式意图，**不受节流限制**，保证用户能手动拿最新数据。

### 7.4 实施后质量门禁

- `npm run lint`（ESLint）+ `tsc --noEmit`（类型检查）+ `npm run format:check`（Prettier）全通过。
- 真机 vConsole 复验：冷启动不再出现"同一秒多接口集体 429"；偶发 429 时前端退避后自动恢复、无风暴。

## 八、后端需配合的事项（治本，非前端范围）

> 以下为后端数据库项目的工作，前端无法代劳，整理给后端开发人员。

### 8.1 限流策略排查/调整（必做，治本）

1. **阈值是否过低**：未登录用户冷启动的少量并行公开请求（版本检查 + 配置 + 活动列表）即触发 429，阈值不应这么严。
2. **计数维度**：是否把同一 IP/设备下所有接口合并计数？公开只读接口（如 `campaigns/active`）的限流应比写接口宽松。
3. **开发/测试环境**：devbox 单实例 + 测试期反复刷新是否导致计数累积？是否需要对开发 IP 放宽或加白名单。
4. **429 响应补 `Retry-After`**：在 429 响应头加 `Retry-After`（秒）或 `data.retry_after_seconds`，让前端退避更精准（配合 7.1）。

### 8.2 治理项2（D：BFF 首屏聚合接口）（建议，长期削峰）

- 后端新增一个"首屏聚合接口"（如 `GET /api/v4/home/bootstrap`），一次性返回首屏所需的活动列表 + 位置配置 + 公告等，前端冷启动从"多个并发请求"变为"一个请求"，从源头消除瞬时并发峰值。
- 这是大厂（腾讯/字节）首屏标准做法，但属后端工作；前端在该接口就绪后改为单次调用即可。

## 十、重要边界声明

1. **前端治理是"治标"**：即便 B/C/B 三项全做，只要后端限流阈值过低，正常冷启动**仍可能 429**。真正修复必须后端调整限流（第 8.1 节）。前端治理的价值是：偶发限流时体验不崩 + 前端绝不自己制造洪峰。
2. **不碰业务数据真实性**：节流/去重只控"刷新频率与重复发送"，活动、积分等业务数据仍以后端实时返回为准，不用旧数据糊弄用户。
3. **退避只对幂等 GET**：写操作（兑换、提交消费）绝不自动重试，避免重复下单，与项目"幂等键（Idempotency-Key）"规则一致。
4. **BFF 聚合接口（治理项2/D）是后端的活**，前端无法代劳，仅作为削峰的长期建议提给后端。

### 核实证据

**#9 JWT 明文敏感信息 — 后端问题（前端无责）**
- `utils/auth-helper.ts:369` 读取 `payload.mobile`（完整手机号），`:370` 读取 `payload.role_level`。
- 关键判断：JWT payload 是 **Base64 编码（非加密）**，任何拿到 Token 的人可直接解码。前端读取是正常用法，**敏感信息是后端签发时就放进去的** → 治本在后端：手机号脱敏（`136****7930`）、`role_level` 移出 payload 改为接口实时查询。
- **前端无需改代码**（前端没把数据写进 Token，也无法改 Token 内容）。

### 一句话总览
- **纯后端问题**：#9（JWT 脱敏）。前端不用改。
- **本文件上半部分（429 限流治理）**：429 治本在后端限流配置；前端做退避/节流/去重属"治标 + 不自伤"。详见上半部分第三、八节。
