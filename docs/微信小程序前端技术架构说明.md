## 1. 技术路线总览

### 1.1 一句话总结

| | 描述 |
|---|---|
| 🔵 **原（v2.0）** | **WXML + WXSS + JavaScript + JSDoc注释** + globalData手动状态管理 + 无npm构建 + Skyline未启用 |
| 🟢 **升级后（v3.0）** | **WXML + Sass + TypeScript + MobX响应式状态** + npm编译插件构建 + Skyline全面启用 |

> 本质上是：**从"手动挡"升级到"自动挡"**——类型安全从人眼看注释变成编译器强制检查，状态管理从手动搬运变成自动同步，样式从硬编码变成变量驱动。

### 1.2 核心技术栈对比

| 层级 | 🔵 原技术路线（v2.0） | 🟢 升级后技术路线（v3.0） | 变化 |
|------|-------------|-----------------|------|
| **语言** | JavaScript（ES6+） | **TypeScript（strict）** | 🔄 升级 |
| **类型系统** | JSDoc 注释模拟（`@param {string}`） | **TS 原生类型 + interface** | 🔄 升级 |
| **模板** | WXML | WXML（不变） | ➡️ 不变 |
| **样式** | WXSS（原生CSS） | **Sass（.scss）→ 编译为WXSS** | 🔄 升级 |
| **状态管理** | `app.globalData` 手动传递 | **MobX（响应式自动同步）** | 🔄 升级 |
| **渲染引擎** | glass-easel + Skyline（配了但**未启用**） | glass-easel + **Skyline（全面启用）** | 🔄 启用 |
| **npm构建** | ❌ 未使用（`useCompilerPlugins: false`） | ✅ **启用编译插件 + npm构建** | 🔄 启用 |
| **模块系统** | CommonJS（`require`/`module.exports`） | CommonJS（不变，TS编译后仍是CJS） | ➡️ 不变 |
| **组件化** | `Component({})`原生 | `Component({})`原生 + **TS类型** | ➡️ 增强 |
| **页面** | `Page({})`原生 | `Page({})`原生 + **TS类型 + MobX绑定** | ➡️ 增强 |
| **基础库** | ≥ 3.0.0 | ≥ 3.0.0（不变） | ➡️ 不变 |
| **组件框架** | glass-easel | glass-easel（不变） | ➡️ 不变 |

## 4. 升级核心对比详解

### 4.1 状态管理对比

**🔵 原：app.globalData（手动搬运模式）**

```
┌──────────┐
│  app.js  │  globalData: { points_balance, userInfo, ... }
└────┬─────┘
     │  每个页面手动读取 ↓
     │
├── lottery.js:  const pts = app.globalData.points_balance; this.setData({...})
├── exchange.js: const pts = app.globalData.points_balance; this.setData({...})
├── user.js:     const pts = app.globalData.points_balance; this.setData({...})
└── ...每个页面都要重复写一遍

问题：页面A改了积分 → 页面B不知道 → 要手动刷新
```

**🟢 升级后：MobX Store（响应式自动同步）**

```
┌─────────────────┐
│  store/index.ts  │  pointsBalance, userInfo（observable）
└────────┬────────┘
         │  自动同步（数据变 → UI自动变）↓
         │
├── lottery.ts:  storeBindings → this.data.pointsBalance 自动更新
├── exchange.ts: storeBindings → this.data.pointsBalance 自动更新
├── user.ts:     storeBindings → this.data.pointsBalance 自动更新
└── ...所有页面自动同步，零手动代码

优势：store改了积分 → 所有绑定页面自动更新 → 无需手动刷新
```

### 4.4 JSDoc 注释迁移策略

TypeScript 迁移后，注释遵循"**保留业务信息，删除类型声明**"原则：

| 注释类型 | 示例 | TS后 | 原因 |
|---------|------|------|------|
| 功能描述 | `加载抽奖页面数据` | ✅ **保留** | TS表达不了"干什么" |
| 业务说明 | `开发环境支持万能验证码123456` | ✅ **保留** | TS表达不了业务规则 |
| API路径 | `后端路由: POST /api/v4/auth/login` | ✅ **保留** | 前后端对接的关键桥梁 |
| 注意事项 | `⚠️ 生产环境禁止Mock数据` | ✅ **保留** | TS表达不了警告 |
| `@example` | `checkAuth({ redirect: false })` | ✅ **保留** | 帮助快速理解用法 |
| `@version` | `@since 2026-02-06` | ✅ **保留** | 追踪变更历史 |
| `@param {string}` | 类型声明 | ❌ **删掉** | `mobile: string` 已替代 |
| `@returns {Promise<Object>}` | 返回类型 | ❌ **删掉** | `: Promise<LoginResponse>` 已替代 |
| `@async` | 异步标记 | ❌ **删掉** | `async` 关键字已表达 |
| `@throws {Error}` | 异常类型 | ❌ **删掉** | TS的错误处理更清晰 |
| `@param {string} mobile - 手机号` | 类型+说明 | ⚠️ **类型删，说明保留** | 变成行内注释 |

**改造前后示例：**

```javascript

/**
 * @param {string} mobile - 手机号（11位）
 * @param {string} code - 验证码（6位）

 * @returns {Promise<Object>} 登录结果
 */

```

```typescript

  /** 登录失败后是否自动跳转到登录页 */

}

  success: boolean
  data?: { access_token: string; refresh_token: string }
  message?: string
}
```

### 4.5 升级兼容性保证：用户可见层零变化

## 9. API架构

### 9.1 基础配置

| 维度 | 方案 |
|------|------|
| 接口版本 | V4.0（`/api/v4/`） |
| 基础地址（测试） | `https://omqktqrtntnn.sealosbja.site/api/v4` |
| 认证方式 | JWT（access_token + refresh_token 双Token） |
| 请求格式 | JSON（`Content-Type: application/json`） |
| 认证头 | `Authorization: Bearer {access_token}` |
| 超时时间 | 15000ms（默认） |

### 9.2 APIClient类特性

| 特性 | 说明 |
|------|------|
| 自动loading | 请求时自动 `showLoading`，完成后自动 `hideLoading` |
| 自动错误提示 | 请求失败自动 `showToast` 显示错误 |
| Token自动刷新 | 401 TOKEN_EXPIRED 时自动调用 refresh 接口 |
| Token无效处理 | 清除认证数据 + 跳转登录页 |
| JWT完整性校验 | 每次请求前验证Token格式和完整性 |
| 防并发刷新 | Token刷新时其他请求排队等待 |
| 幂等键 | 消费提交等操作自动生成 Idempotency-Key |
| HTTP错误码全覆盖 | 401/403/404/409/429/500/503 分别处理 |

### 9.3 命名规范

| 场景 | 命名方式 | 示例 |
|------|---------|------|
| 前端业务逻辑（变量、函数） | camelCase | `pointsBalance`、`loadLotteryData()` |
| API交互字段（请求参数、响应字段） | snake_case | `user_id`、`access_token`、`cost_per_draw` |
| 类名/构造函数 | PascalCase | `APIClient`、`FormValidator` |

### 9.5 工具函数统一导入

所有页面和组件统一从 `utils/index` 导入，不直接引用内部模块：

```typescript
// ✅ 正确：统一从index导入
const { Utils, API, Wechat, Constants } = require('../../utils/index')

// ❌ 禁止：直接引用内部模块（会导致引用混乱）
const api = require('../../utils/api')
```

`utils/index` 导出6个分类模块：

| 模块 | 包含内容 |
|------|---------|
| `Utils` | 日期格式化、JWT处理、防抖节流、认证助手 |
| `Validation` | 手机号/验证码/积分/昵称验证 |
| `API` | 全部API方法（63个） |
| `Wechat` | showToast/showLoading/navigateTo等 |
| `ErrorHandler` | showError/handleJWTExpired |
| `Constants` | TIME/DELAY/LOTTERY/UI_SIZE等常量 |
