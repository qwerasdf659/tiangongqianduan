# 客服工作台重构设计 — 订单-工单关联模式

> 版本：v1.3.0  
> 日期：2026-05-27  
> 状态：决策已确认，可开始实施  

---

## 1. 业务背景与目标

### 1.1 当前问题

当前客服工作台（`customer-service.html`）采用「会话列表 → 聊天窗口」的单一模式：

- 客服搜索用户手机号后，A区展示该用户的所有历史会话（按优先级分组）
- 用户投诉通常是因为某个订单出了问题，但客服需要在多个会话中翻找才能定位到具体订单
- 工单管理（`cs-issues.html`）是独立页面，与实时聊天割裂，无法快速关联

### 1.2 设计目标

1. **订单驱动**：客服搜索手机号后，优先看到用户近期订单，快速定位问题源头
2. **工单关联订单**：每个工单明确关联到具体订单（交易/兑换/消费），实现「订单 → 工单」的追踪链路
3. **多Tab工作区**：B区支持同时打开多个聊天/订单详情页签，减少页面切换
4. **信息聚合**：C区右侧面板增加订单维度的上下文信息

### 1.3 改动归属划分

| 归属 | 改动内容 |
|------|---------|
| 后端数据库项目 | 数据库迁移、模型修改、Service 扩展、新增路由、TradeDisputeService 适配多态字段 |
| web端管理后台前端项目 | customer-service.html 三区域改造、composable 扩展、API 层新增端点 |
| 微信小程序前端项目 | 无改动（本次重构仅涉及管理后台；小程序代码在独立仓库，不在本项目中） |

```

    │
    ▼

    │

    ├─ 选中订单时 →「订单工单」Tab 显示该订单的关联工单
```

## 4. 后端改动（对齐实际技术框架）

### 4.0 技术框架对齐说明（已通过真实数据库和代码验证）

后端项目实际技术栈：
- **ORM**：Sequelize 6.35 + MySQL2
- **服务层模式**：静态类 + `static async method(models, params, options={})` 
- **服务注册**：`ServiceManager._services.set('snake_case_key', ServiceClass)`
- **路由获取服务**：`req.app.locals.services.getService('key')`
- **路由获取模型**：`req.app.locals.models`
- **事务管理**：写操作用 `TransactionManager.execute(async transaction => { ... })`
- **响应格式**：`res.apiSuccess(data, message)` / `res.apiError(msg, code, details, status)`
- **迁移规范**：`YYYYMMDDHHMMSS-description.js` + 事务包裹 + `up/down`
- **分页模式**：`findAndCountAll` → `{ rows, count, page, page_size }`

已注册的可复用服务（ServiceManager 中的 key）：
- `trade_order_query` → TradeOrderQueryService（有 `getUserOrders(params)` 方法，params 为 `{user_id, role, status, page, page_size}`，返回 `{orders, total, page, page_size}`）
- `redemption_order` → RedemptionService（有 `getOrderDetail(order_id)` 方法，**无** `getUserOrders` 列表方法）
- `consumption_query` → ConsumptionQueryService（有 `getUserConsumptionRecords(userId, options)` 方法，返回 `{records, pagination: {total, page, page_size, total_pages}, stats}`）
- `cs_issue` → CustomerServiceIssueService（有 `list(models, params)` 和 `create(models, params, options)` 方法）
- `trade_dispute` → TradeDisputeService（实例化服务，有 `createDispute(params, options)` 方法，**直接使用 `trade_order_id` 字段**）

#### 字段设计说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `order_type` | ENUM('trade','redemption','consumption') | 订单类型标识，NULL 表示工单不关联订单 |
| `order_id` | VARCHAR(64) | 订单ID（统一用字符串存储，兼容 bigint 和 UUID） |

为什么用多态关联而不是三个独立外键：
- 三种订单主键类型不同（bigint / char），无法用统一外键约束
- 多态关联更灵活，未来新增订单类型只需扩展 ENUM
- 查询模式固定：总是 `WHERE order_type = ? AND order_id = ?`

### 4.4 新增路由：`routes/v4/console/customer-service/user-orders.js`

```javascript
/**
 * 客服工作台 - 用户订单聚合查询
 *
 * 路径前缀：/api/v4/console/customer-service/user-orders
 * 职责：聚合查询用户在三种订单表中的近期订单
 *
 * 数据来源（通过 ServiceManager 获取已有服务）：
 * - trade_order_query → TradeOrderQueryService.getUserOrders()
 * - consumption_query → ConsumptionQueryService.getUserConsumptionRecords()
 * - redemption_order → RedemptionService.getOrderDetail()（需扩展列表查询）
 */

const express = require('express')
const router = express.Router()
const { authenticateToken, requireRoleLevel } = require('../../../../middleware/auth')
const { asyncHandler } = require('../../../../middleware/validation')

router.use(authenticateToken, requireRoleLevel(1))

/**
 * GET /:user_id - 获取用户近期所有类型订单（聚合）
 *
 * 聚合逻辑：
 * 1. 并行查询三种订单表
 * 2. 统一格式化为 { order_type, order_id, order_no, summary, status, created_at, issue_count }
 * 3. 按 created_at 倒序混合排序
 * 4. 应用层分页（三表数据量有限，不需要 SQL 级 UNION 分页）
 */
router.get('/:user_id', asyncHandler(async (req, res) => {
  const { user_id } = req.params
  const { page = 1, page_size = 20 } = req.query
  const models = req.app.locals.models

  // 通过 ServiceManager 获取已有服务（复用，不新建）
  const TradeOrderQueryService = req.app.locals.services.getService('trade_order_query')
  const ConsumptionQueryService = req.app.locals.services.getService('consumption_query')

  // 并行查询三种订单
  const [tradeResult, consumptionResult, redemptionOrders] = await Promise.all([
    TradeOrderQueryService.getUserOrders({
      user_id: parseInt(user_id), role: 'all', page: 1, page_size: 50
    }),
    ConsumptionQueryService.getUserConsumptionRecords(parseInt(user_id), {
      page: 1, page_size: 50
    }),
    models.RedemptionOrder.findAll({
      where: { redeemer_user_id: parseInt(user_id) },
      attributes: ['redemption_order_id', 'order_no', 'status', 'created_at', 'item_id'],
      include: [{ model: models.Item, as: 'item', attributes: ['item_id', 'display_name'] }],
      order: [['created_at', 'DESC']],
      limit: 50
    })
  ])

  // 统一格式化（注意各服务返回格式不同）
  const allOrders = [
    ...formatTradeOrders(tradeResult.orders || []),
    ...formatConsumptionRecords(consumptionResult.records || []),
    ...formatRedemptionOrders(redemptionOrders)
  ]

  // 按时间倒序排序
  allOrders.sort((a, b) => new Date(b.created_at) - new Date(a.created_at))

  // 查询每个订单的关联工单数
  await attachIssueCounts(models, allOrders)

  // 应用层分页
  const start = (parseInt(page) - 1) * parseInt(page_size)
  const paged = allOrders.slice(start, start + parseInt(page_size))

  return res.apiSuccess({
    orders: paged,
    total: allOrders.length,
    page: parseInt(page),
    page_size: parseInt(page_size)
  }, '获取用户订单列表成功')
}))

module.exports = router
```

辅助函数（同文件内定义）：

```javascript
function formatTradeOrders(orders) {
  return orders.map(o => ({
    order_type: 'trade',
    order_id: String(o.trade_order_id),
    order_no: o.order_no,
    summary: `${o.asset_code} ×${o.gross_amount}`,
    amount: String(o.gross_amount),
    status: o.status,
    created_at: o.created_at
  }))
}

function formatConsumptionRecords(records) {
  return records.map(r => ({
    order_type: 'consumption',
    order_id: String(r.consumption_record_id),
    order_no: r.order_no,
    summary: `门店消费 ¥${r.consumption_amount}`,
    amount: String(r.consumption_amount),
    status: r.status,
    created_at: r.created_at
  }))
}

function formatRedemptionOrders(orders) {
  return orders.map(o => ({
    order_type: 'redemption',
    order_id: String(o.redemption_order_id),
    order_no: o.order_no,
    summary: `兑换 ${o.item?.display_name || '物品'}`,
    amount: '1',
    status: o.status,
    created_at: o.created_at
  }))
}

async function attachIssueCounts(models, orders) {
  if (orders.length === 0) return
  const { Op } = require('sequelize')
  const conditions = orders.map(o => ({ order_type: o.order_type, order_id: o.order_id }))
  const issues = await models.CustomerServiceIssue.findAll({
    where: { [Op.or]: conditions },
    attributes: ['order_type', 'order_id'],
    raw: true
  })
  const countMap = {}
  issues.forEach(i => {
    const key = `${i.order_type}:${i.order_id}`
    countMap[key] = (countMap[key] || 0) + 1
  })
  orders.forEach(o => {
    o.issue_count = countMap[`${o.order_type}:${o.order_id}`] || 0
  })
}
```

```javascript
export function useCsWorkTabsState() {
  return {
    work_tabs: [],           // Tab 列表 [{id, type, label, data}]
    active_work_tab: null,   // 当前激活的 Tab ID
    MAX_TABS: 5
  }
}

export function useCsWorkTabsMethods() {
  return {
    openWorkTab(config) {
      const tabId = `${config.type}_${config.id}`
      const existing = this.work_tabs.find(t => t.id === tabId)
      if (existing) {
        this.active_work_tab = tabId
        return
      }
      if (this.work_tabs.length >= this.MAX_TABS) {
        Alpine.store('notification').show('最多同时打开5个标签页，请关闭不需要的标签', 'warning')
        return
      }
      this.work_tabs.push({
        id: tabId,
        type: config.type,    // 'chat' | 'order'
        label: config.label || (config.type === 'chat' ? '聊天' : `订单#${config.id}`),
        data: config.data
      })
      this.active_work_tab = tabId
    },

    closeWorkTab(tabId) {
      const idx = this.work_tabs.findIndex(t => t.id === tabId)
      if (idx === -1) return
      this.work_tabs.splice(idx, 1)
      if (this.active_work_tab === tabId) {
        this.active_work_tab = this.work_tabs.length > 0 ? this.work_tabs[this.work_tabs.length - 1].id : null
      }
    },

    getActiveTab() {
      return this.work_tabs.find(t => t.id === this.active_work_tab) || null
    }
  }
}
```

## 8. 技术约束与注意事项

1. **多态关联不加外键约束**：由于三种订单表主键类型不同，`order_id` 字段不在数据库层面加外键，由应用层保证数据一致性
2. **订单聚合查询性能**：三表独立查询后在应用层合并排序（不用 SQL UNION），各表限制 50 条，总量可控
3. **Tab 内存管理**：B区最多 5 个 Tab，超出时提示关闭
4. **WebSocket 不变**：会话Tab 的实时消息推送逻辑保持不变
5. **独立工单页面保留**：`cs-issues.html` 独立页面保留，作为工单全量管理入口
6. **前端命名规范**：新增端点常量遵循 `{资源类型}_{操作}` 命名，如 `CS_USER_ORDERS`
7. **后端服务复用**：不新建 Service 类，复用已有的 `trade_order_query`、`consumption_query` 服务
8. **事务边界**：工单创建是写操作，必须通过 `TransactionManager.execute()` 包裹
9. **trade_order_id 字段**：本次迁移直接删除（决策1），同时清理模型定义和所有引用代码
10. **TradeDisputeService 必须同步修改**：该服务的 `createDispute()`、`resolveDispute()`、`getDisputeDetail()` 都直接操作 `trade_order_id`，是本次迁移的**强依赖项**
11. **disputes.js 路由统一字段**（决策A2）：POST `/disputes` 改为接收 `order_type` + `order_id`，前端直接修改字段名，不做映射
12. **C区 Tab 数量**：从现有 9 个增加到 11 个，需注意移动端/小屏幕下的横向滚动体验
13. **迁移命令**：项目使用 `npm run db:migrate`（不是 `migration:up`）

### 决策2：RedemptionService 列表查询 → 扩展 Service

**结论：给 RedemptionService 新增 `static async getUserOrders(userId, options)` 方法**

| 维度 | 分析 |
|------|------|
| 项目架构规范 | 「路由禁止直接访问 models，读操作收口到 Service/QueryService」 |
| 大公司做法 | 美团/阿里所有数据访问通过 Service 层，路由层是薄路由 |
| 长期维护 | 如果兑换订单查询逻辑变了（加软删除过滤等），只改 Service 一处 |

具体做法：在 `services/RedemptionService.js` 中新增：

```javascript
/**
 * 获取用户的兑换订单列表（客服工作台用）
 * @param {number} userId - 用户ID
 * @param {Object} [options={}] - 查询选项
 * @param {number} [options.page_size=50] - 每页数量
 * @returns {Promise<Object>} { rows, count }
 */
static async getUserOrders(userId, options = {}) {
  const { page_size = 50 } = options
  return await RedemptionOrder.findAndCountAll({
    where: { redeemer_user_id: parseInt(userId) },
    attributes: ['redemption_order_id', 'order_no', 'status', 'created_at', 'item_id'],
    include: [{ model: models.Item, as: 'item', attributes: ['item_id', 'display_name'] }],
    order: [['created_at', 'DESC']],
    limit: Math.min(page_size, 50)
  })
}
```

### 决策3：订单聚合查询限制 → 各表50条

**结论：保持各表50条限制，应用层合并排序**

| 维度 | 分析 |
|------|------|
| 美团客服 | 默认展示最近30天/50条，超出需手动搜索 |
| 腾讯客服 | 默认最近7天，点「更多」才加载 |
| 游戏GM工具 | 默认展示最近100条操作记录 |
| 性能 | 3个并行查询各50条 → 应用层排序150条 → 响应 < 200ms |
| 本项目 | 客服场景是「快速定位近期问题」，50条覆盖近期订单足够 |

如果后续发现不够，可以加 `date_range` 参数，但现在不过度设计（YAGNI）。

### 决策4：创建工单交互 → 弹窗模态框

**结论：弹出模态框填写工单信息（不离开当前 Tab）**

| 维度 | 分析 |

| 阿里/淘宝客服 | 弹窗表单 |
| Zendesk/Freshdesk | 弹窗或侧边栏 |
| 游戏GM工具 | 弹窗快速创建 |

只有 Jira 这种「工单本身就是核心产品」的系统才用独立页面，我们的工单是辅助功能。
