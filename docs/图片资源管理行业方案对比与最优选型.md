## 一、你的项目是什么

一句话：餐厅积分抽奖系统 — 用户消费获得积分/材料资产，通过抽奖获得虚拟物品（优惠券、水晶等），物品可在市场交易、兑换实物商品、到店核销。

核心业务域：
- 抽奖系统（活动、奖品、档位规则、决策审计）
- 兑换市场（B2C 商品兑换，材料资产支付）
- 交易市场（C2C 物品挂牌、竞价、钻石结算）
- 物品系统（三表模型：items + item_ledger + item_holds）
- 资产系统（统一账户、双录记账、冻结模型）
- 核销系统（SHA-256 核销码、门店扫码）
- 广告投放（弹窗/轮播/横幅，素材管理）
- 管理后台（Alpine.js + Tailwind，80+ 路由）

技术栈：Node.js 20 + Express + Sequelize + MySQL + Redis + Sealos S3 + Sharp

图片涉及的业务场景：
- 奖品图片、商品主图、物品模板图标
- 广告素材（弹窗 banner、轮播图）
- 分类图标、材料资产图标
- 商家 logo、用户头像
- 活动 banner/背景图
- 系统占位图

## 二、行业方案全景对比

### 2.1 大厂方案（美团/阿里/腾讯）

#### 美团：独立图片中心 + CDN + 实时裁剪

```
客户端 → CDN（边缘节点缓存）→ 图片处理集群（Nginx + 实时裁剪）→ 分布式对象存储
                                    ↑
                              URL 参数驱动裁剪
                              例：/image.jpg?w=300&h=300&q=80
```

核心设计：
- 独立的"图片中心"微服务，与业务完全解耦
- 原图只存一份，所有尺寸通过 URL 参数实时生成（?w=300&h=300&q=80&f=webp）
- CDN 边缘缓存裁剪结果，命中率 95%+
- 数据库只存一个 image_id，不存任何 URL 或 object key
- 图片元数据独立存储（宽高、格式、EXIF、审核状态）
- 全链路 WebP/AVIF 自适应（根据 Accept 头自动选格式）

适用规模：日均千万级图片请求，数百 TB 存储

#### 阿里（淘宝/天猫）：OSS + 图片处理服务 + CDN

```
客户端 → CDN → OSS（阿里云对象存储）
                  ↑
            OSS 图片处理参数
            例：?x-oss-process=image/resize,w_300/quality,q_80/format,webp
```

核心设计：
- 不自建图片处理集群，直接用 OSS 内置的图片处理能力
- URL 后缀参数驱动：`?x-oss-process=image/resize,w_300/quality,q_80`
- 图片审核（鉴黄、暴恐）集成在上传链路
- 商品图片有独立的"图片空间"管理后台
- 支持图片版本（同一商品图可以有历史版本）
- 水印、裁剪、旋转全部通过 URL 参数

适用规模：百亿级图片，PB 级存储

#### 腾讯（微信/QQ）：COS + 数据万象 + CDN

```
客户端 → CDN → COS（腾讯云对象存储）+ 数据万象（CI）
                                          ↑
                                    处理模板 + URL 参数
```

核心设计：
- COS 存储 + 数据万象（CI）做图片处理
- 预定义处理模板（样式），URL 后加样式名即可
- 例：`/image.jpg!style_thumbnail` 或 `/image.jpg?imageMogr2/thumbnail/300x300`
- 微信生态内图片有专门的 media_id 体系
- 小程序端通过 wx.cloud 直传 COS，不经过业务服务器

适用规模：十亿级日活，EB 级存储

### 2.2 小公司/创业团队方案

#### 方案 A：业务表直存 URL（最简单）

```
业务表
├── product_name: "商品A"
├── image_url: "https://cdn.xxx.com/products/abc.jpg"
├── thumbnail_url: "https://cdn.xxx.com/products/abc_thumb.jpg"
└── banner_url: "https://cdn.xxx.com/banners/xyz.jpg"
```

特点：
- 零额外开发成本
- 图片 URL 散落在各业务表
- 换域名/换存储要全库 UPDATE
- 无法统一管理、无法检测孤立文件
- 删除业务记录后图片成垃圾

典型用户：早期 MVP、个人项目、外包项目

#### 方案 B：统一图片表 + 外键关联（中间方案）

```
image_resources 表（统一管理）
├── image_id
├── file_path（object key）
├── thumbnail_paths（JSON）
├── business_type
├── context_id
└── status

业务表通过 image_resource_id 外键关联
```

特点：
- 图片有统一的生命周期管理
- 但 context_id 是"万能外键"，指向不同表，无法建真正的外键约束
- business_type 是 ENUM，新增业务类型要改 schema
- 删除业务记录时需要手动清理图片

典型用户：中小型 SaaS、内部管理系统

#### 方案 C：独立附件/媒体服务（推荐方案）

```
media_files 表（纯存储层，不关心业务语义）
├── media_id（UUID）
├── object_key
├── file_size / mime_type / width / height
├── thumbnails（JSON）
├── uploaded_by / uploaded_at
└── status

media_attachments 表（关联层，多态关联）
├── attachment_id
├── media_id → media_files
├── attachable_type（'lottery_prize' / 'exchange_item' / 'ad_creative' / ...）
├── attachable_id（业务记录 ID）
├── role（'primary' / 'gallery' / 'icon' / 'banner' / 'background'）
├── sort_order
└── created_at
```

特点：
- 存储层和关联层完全分离
- 一张图可以被多个业务记录引用（去重）
- 新增业务类型不需要改 schema，只需要新的 attachable_type 字符串
- 删除业务记录时，通过 attachments 表解除关联，引用计数归零才物理删除
- 可以轻松实现"图片库"功能（选择已有图片而非重新上传）

典型用户：Shopify、WordPress、Strapi 等成熟 CMS

### 2.3 游戏公司方案

#### 端游/手游（米哈游、网易游戏等）

```
asset_bundles 表
├── bundle_id
├── asset_type（texture / icon / spine / effect）
├── asset_key
├── variants（JSON：{ "1x": "key", "2x": "key", "4x": "key" }）
├── tags（JSON：["character", "weapon", "rare"]）
└── version

asset_references 表
├── reference_id
├── bundle_id → asset_bundles
├── entity_type（'character' / 'weapon' / 'skill'）
├── entity_id
├── slot（'portrait' / 'icon' / 'full_art' / 'thumbnail'）
└── priority
```

核心设计：
- 资源按"包"管理（Asset Bundle），支持热更新
- 同一资源有多个分辨率变体（1x/2x/4x），客户端按设备选择
- 资源有版本号，支持灰度发布和回滚
- 通过 tags 做资源分类和搜索
- 引用关系通过 slot 区分用途（头像/图标/全身像/缩略图）
- CDN + 增量更新，客户端本地缓存

与你项目的相关性：
- 你的物品模板（item_templates）有 rarity_code，类似游戏的稀有度系统
- 你的材料资产（红水晶/橙水晶等）有图标，类似游戏的道具图标
- 你的抽奖系统类似游戏的抽卡系统

#### 虚拟物品交易平台（Steam 市场、网易 BUFF）

```
item_images 表
├── image_id
├── object_key
├── image_type（'inspect' / 'icon' / 'screenshot' / 'sticker'）
├── quality_variants（JSON：{ "low": "key", "medium": "key", "high": "key" }）
└── hash（用于去重）

item_image_bindings 表
├── item_id → items
├── image_id → item_images
├── display_context（'listing' / 'detail' / 'trade_offer'）
└── sort_order
```

核心设计：
- 图片按内容哈希去重（同一张图只存一份）
- 不同展示场景用不同尺寸（列表页用小图，详情页用大图）
- 交易场景需要图片不可篡改（哈希校验）
- 支持用户上传截图（UGC），需要审核

### 2.4 活动策划公司方案（活动行、互动吧等）

```
活动素材库
├── media_library 表（全局素材库）
│   ├── media_id
│   ├── object_key
│   ├── folder_id（虚拟文件夹）
│   ├── tags（JSON）
│   ├── usage_count（引用计数）
│   └── uploaded_by
│
├── activity_media 表（活动-素材关联）
│   ├── activity_id
│   ├── media_id
│   ├── usage_type（'poster' / 'banner' / 'prize_image' / 'sponsor_logo'）
│   └── sort_order
```

核心设计：
- 全局素材库，跨活动复用（同一张 logo 用在多个活动）
- 虚拟文件夹组织（按客户/按活动/按类型）
- 引用计数，只有无引用的素材才能删除
- 活动结束后素材归档而非删除
- 支持素材标签搜索

### 2.5 小众二手平台方案（闲鱼、转转、Mercari）

```
user_uploads 表
├── upload_id（UUID）
├── user_id
├── object_key
├── content_hash（SHA-256，用于去重和盗图检测）
├── moderation_status（pending / approved / rejected）
├── exif_data（JSON，含 GPS 需脱敏）
└── variants（JSON：{ "original": "key", "watermarked": "key", "blurred": "key" }）

listing_images 表
├── listing_id → listings
├── upload_id → user_uploads
├── position（1-9，最多9张）
├── is_cover（是否封面图）
└── created_at
```

核心设计：
- UGC 为主，需要内容审核（鉴黄、违禁品检测）
- 图片去重（content_hash），防止盗图
- EXIF 脱敏（移除 GPS 等隐私信息）
- 水印变体（交易完成前加水印保护）
- 每个商品最多 N 张图，有封面图概念
- 图片与商品是多对多关系（同一张图可能出现在多个商品）

## 3.5 各方案优缺点详细分析（针对本项目）

### 方案 A：大厂方案（URL 参数实时裁剪）

代表：美团图片中心、阿里 OSS 图片处理、腾讯数据万象

优点：
- 原图只存一份，任意尺寸通过 URL 参数生成，存储成本最低
- CDN 边缘缓存，性能极好
- 新增尺寸需求不需要重新生成文件

缺点：
- Sealos S3 不支持 URL 参数裁剪，要换存储或自建图片处理集群
- 本项目图片量级（54 个文件 / 2.57MB）完全用不上这套
- 开发和运维成本极高，需要专门团队

适合度：不适合。杀鸡用牛刀。

---

### 方案 B：业务表直存 URL（最简单）

代表：早期 MVP、外包项目

优点：
- 零开发成本，加个字段就完事

缺点：
- 换域名/换存储要全库 UPDATE
- 无法统一管理，无法检测孤立文件
- 删除业务记录后图片成垃圾，永远清不掉
- 本项目的 category_defs.icon_url、material_asset_types.icon_url、ad_creatives.image_url 当前就是这个状态，已经在承受它的问题

适合度：不适合。应该往上走而不是留在这。

---

### 方案 C：统一图片表 + 万能外键（本项目当前方案）

代表：本项目的 image_resources 表

优点：
- 图片有统一的生命周期管理
- 已经有完整的上传/缩略图/垃圾回收逻辑

缺点：
- `business_type` 是 ENUM，每新增一个场景要改 schema + 改 FOLDER_MAP + 改代码
- `context_id` 是万能外键，指向不同表，无法建真正的外键约束
- 无法反查"这张图被谁引用了"（不知道 context_id 指向哪个表）
- 不支持图片复用（一张图只能绑一个 context_id）
- 不支持去重（同一张图上传两次就是两条记录）
- 当前覆盖率只有 3.7%（54 个文件只有 2 个走了这套）

适合度：骨架不错但有结构性缺陷，修补不如重建。

---

### 方案 D：独立媒体服务（media_files + media_attachments）— 推荐

代表：Shopify、WordPress、Strapi、Laravel Media Library

优点：
- `attachable_type` 是 VARCHAR，新增业务类型传个字符串就行，零 schema 改动
- `media_id` 有真正的外键约束（不是万能外键）
- 一张图可以被多个业务记录引用（去重 + 复用）
- 能反查"这张图被谁引用"（查 media_attachments 表）
- 引用计数归零才物理删除，不会误删
- `content_hash` 去重，同一张图不会存两份
- `role` 字段区分用途（primary/icon/banner/background），比 category 更清晰
- 存储层（SealosStorageService）和 URL 生成（ImageUrlHelper）完全复用，改动集中在中间层

缺点：
- 查询时需要 JOIN 一次（media_attachments + media_files），比直接读字段多一次关联
- 一次性迁移成本（但本项目只有 54 个文件，迁移量极小）

适合度：最适合本项目。

---

### 方案 E：游戏 Asset Bundle 方案

代表：米哈游、网易游戏

优点：
- 版本管理、灰度发布、增量更新
- 多分辨率变体（1x/2x/4x）

缺点：
- 本项目不需要客户端本地缓存和增量更新
- 小程序端不需要多分辨率变体
- 过度设计

适合度：不适合。

---

### 最终推荐：方案 D

核心理由：

1. 项目未上线、存量数据极少（54 个文件），迁移成本几乎为零
2. 有 8+ 种图片场景且还在增长，ENUM 方案每次都要改 schema，方案 D 永远不需要
3. SealosStorageService、ImageUrlHelper、Sharp、multer、图片代理路由全部原样复用，只替换中间的 ImageService → MediaService 和 image_resources → media_files + media_attachments
4. 一次投入，长期受益，技术债务降到最低

- **整体图片治理率（有引用的文件 / 总文件）：42/54 = 77.8%**

### 本质问题

image_resources 骨架完整但接入率极低。54个文件中仅2个原图走了完整的 ImageService 流程，其余28个有业务引用的文件全部绕过它直存 object key。ENUM 类型的 `business_type` 只覆盖 lottery/exchange/trade/uploads 四种场景，无法覆盖 categories、materials、popup-banners、carousel 等实际已有的 6 个存储文件夹。

⚠️ **2026-03-16 真实数据库验证发现的结构性问题**：`category_defs` 表的主键是 `category_code VARCHAR(50)`（如 `'food'`、`'electronics'`、`'collectible'`），不是整数自增 ID。这意味着如果 `media_attachments.attachable_id` 定义为 `BIGINT UNSIGNED`，将无法存储 `category_defs` 的字符串主键。111 张表中仅此 1 张表使用非整数主键，但它恰好是需要图片管理的 9 张表之一。此问题影响 `media_attachments` 表的 `attachable_id` 字段类型设计，详见决策 10。

## 五、最适合你项目的方案：独立媒体服务（方案 C 改良版）

### 5.1 为什么选这个

| 考量因素 | 分析 |
|---------|------|
| 项目未上线 | 可以一次性重构，不需要兼容旧数据 |
| 愿意投入成本 | 可以做彻底的方案，不需要折中 |
| 长期维护成本 | 方案 C 新增业务类型零改动，方案 B 每次要改 ENUM |
| 技术债务 | 方案 C 消除"万能外键"和"ENUM 硬编码"两大债务源 |
| 业务复杂度 | 你有 8+ 种图片使用场景，方案 B 的 ENUM 已经不够用 |
| 图片复用需求 | 同一张分类图标可能被多个地方引用，方案 C 天然支持 |
| 你的技术栈 | Node.js + Sequelize + Sealos S3，方案 C 完全兼容 |

### 5.2 为什么不选其他方案

大厂方案（URL 参数实时裁剪）：
- 需要自建图片处理集群或用阿里云 OSS 图片处理
- Sealos S3 不支持 URL 参数裁剪
- 你的图片量级（54 个文件 / 2.57MB）完全不需要
- 预生成缩略图对你的规模更合适

游戏方案（Asset Bundle + 版本管理）：
- 你不需要客户端本地缓存和增量更新
- 你不需要多分辨率变体（小程序端屏幕尺寸有限）
- 过度设计

小公司直存（方案 A）：
- 你已经比这个阶段更高级了
- 回退到直存是倒退

保持现状（方案 B 半成品）：
- 每新增一个业务场景就要改 ENUM、改 FOLDER_MAP、改引用检查
- context_id 万能外键无法建约束，数据一致性靠代码保证
- 长期维护成本持续累积

## 六、具体方案设计（基于你的技术栈）

### 6.1 数据库设计

#### 表 1：media_files（纯存储层）

```sql
CREATE TABLE media_files (
  media_id        BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  object_key      VARCHAR(500) NOT NULL COMMENT '原图 Sealos object key',
  thumbnail_keys  JSON         DEFAULT NULL COMMENT '缩略图 keys: {small, medium, large}',
  original_name   VARCHAR(255) DEFAULT NULL COMMENT '原始文件名',
  file_size       INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '文件大小(bytes)',
  mime_type       VARCHAR(100) NOT NULL COMMENT 'MIME 类型',
  width           INT UNSIGNED DEFAULT NULL COMMENT '图片宽度(px)',
  height          INT UNSIGNED DEFAULT NULL COMMENT '图片高度(px)',
  content_hash    CHAR(64)     DEFAULT NULL COMMENT 'SHA-256 内容哈希(去重)',
  folder          VARCHAR(100) NOT NULL COMMENT '存储文件夹(products/materials/...)',
  uploaded_by     INT UNSIGNED DEFAULT NULL COMMENT '上传用户 ID',
  status          ENUM('active','archived','deleted') NOT NULL DEFAULT 'active',
  created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE KEY uk_object_key (object_key),
  UNIQUE KEY uk_content_hash (content_hash),
  KEY idx_folder_status (folder, status),
  KEY idx_uploaded_by (uploaded_by, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='媒体文件表(纯存储层)';
```

关键设计决策：
- media_id 用 BIGINT，不用 UUID（Sequelize 自增主键，与你现有风格一致）
- content_hash 做唯一索引，上传时先查哈希，重复文件直接复用
- folder 是普通 VARCHAR 而非 ENUM，新增文件夹不需要改 schema
- 不存任何业务语义（没有 business_type、context_id）
- status 保留 archived 状态，支持软删除/回收站

#### 表 2：media_attachments（关联层）

```sql
CREATE TABLE media_attachments (
  attachment_id   BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  media_id        BIGINT UNSIGNED NOT NULL COMMENT '关联的媒体文件',
  attachable_type VARCHAR(50)  NOT NULL COMMENT '业务实体类型(lottery_prize/exchange_item/...)',
  attachable_id   BIGINT UNSIGNED NOT NULL COMMENT '业务实体 ID（决策 10：配合 category_defs 加整数 PK 后统一为 BIGINT）',
  role            VARCHAR(30)  NOT NULL DEFAULT 'primary' COMMENT '用途(primary/icon/banner/background/gallery)',
  sort_order      INT          NOT NULL DEFAULT 0,
  created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,

  KEY idx_attachable (attachable_type, attachable_id, role),
  KEY idx_media (media_id),
  CONSTRAINT fk_attachment_media FOREIGN KEY (media_id) REFERENCES media_files(media_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='媒体关联表(多态关联)';
```

关键设计决策：
- attachable_type 是 VARCHAR 而非 ENUM，新增业务类型零改动
- **attachable_id 是 BIGINT UNSIGNED** — 决策 10 结论为方案 B（给 category_defs 加整数 PK），全系统统一整数 PK 后 BIGINT 可覆盖所有业务表
- role 区分同一业务记录的不同图片用途（主图/图标/banner/背景/画廊）
- media_id 上建了真正的外键约束（CASCADE 删除）
- 复合索引 (attachable_type, attachable_id, role) 覆盖最常见的查询

#### 对比：新旧方案

| 维度 | 旧方案 (image_resources) | 新方案 (media_files + media_attachments) |
|------|------------------------|----------------------------------------|
| 新增业务类型 | 改 ENUM + 改 FOLDER_MAP + 改代码 | 传一个字符串，零改动 |
| 图片复用 | 不支持（context_id 只能指向一个） | 天然支持（多条 attachment 指向同一个 media） |
| 外键约束 | 无法建（context_id 指向不同表） | 真正的外键（media_id → media_files） |
| 删除安全 | 手动检查引用 | 查 attachments 表引用计数 |
| 查询"某业务的图片" | WHERE business_type=? AND context_id=? | WHERE attachable_type=? AND attachable_id=? |
| 查询"某图片被谁引用" | 无法查（不知道 context_id 指向哪个表） | SELECT * FROM media_attachments WHERE media_id=? |
| 去重 | 不支持 | content_hash 唯一索引 |

## 八、方案 D 进一步优化（D+ 增强版）

基础方案 D 已经解决了核心架构问题，以下 6 个优化点可以在实施时一并落地，进一步降低长期维护成本。

### 8.1 高频查询 JOIN 优化 — 冗余缓存字段

问题：每次查业务数据都要 JOIN media_attachments + media_files，小程序端加载商品列表、奖品列表等高频场景有性能代价。

优化：在需要高频读取主图的业务表上保留一个冗余的 `primary_media_id` 字段作为快照缓存，但数据真相仍在 media_attachments。

```
exchange_items 表：
  primary_media_id  BIGINT  (冗余缓存，由 MediaService 自动维护)

查询策略：
  小程序端（高频读）→ 直接 JOIN media_files ON primary_media_id，一次 JOIN
  管理后台（低频，需要多图/多角色）→ 走 media_attachments

维护策略：
  MediaService.attach() 时 → 如果 role='primary'，自动更新业务表的 primary_media_id
  MediaService.detach() 时 → 自动置空 primary_media_id
  MediaService.replaceMedia() 时 → 原子更新
```

适用的表：exchange_items、lottery_prizes、item_templates、ad_creatives（这些是小程序端高频查询的）

不需要的表：category_defs、material_asset_types、merchants（管理后台低频查询，直接 JOIN 即可）

### 8.2 content_hash 去重策略优化 — 建议复用而非强制复用

问题：如果 content_hash 做唯一索引强制去重，运营上传同一张图给"奖品 A"和"商品 B"后，想分别替换成不同的图时，会发现它们共享同一个 media_file，替换一个会影响另一个。

优化：去重改为"建议复用"模式。

```
上传流程：
  1. 计算 content_hash（SHA-256）
  2. 查询是否已有相同 hash 的 media_file
  3. 如果命中：
     → API 返回 { duplicate: true, existing_media_id: 123, ... }
     → 前端提示："检测到相同图片已存在，是否复用？"
     → 用户选择复用 → 直接创建 attachment 指向已有 media，不重复上传
     → 用户选择新建 → 正常上传为新的 media_file
  4. 如果未命中：正常上传

数据库改动：
  content_hash  CHAR(64)  — 改为普通索引（非唯一索引）
  用于查重建议，不强制约束
```

好处：
- 节省存储（大部分场景运营会选择复用）
- 不会因为强制共享导致替换时的副作用
- 管理后台可以展示"重复文件"列表，运营自行决定是否合并

### 8.3 轻量标签系统 — 解决图片量增长后的检索问题

问题：当前 media_files 只有 `folder` 字段做分类。图片量增长后，运营在管理后台找图会很痛苦。

优化：media_files 表新增 tags JSON 字段。

```sql
-- media_files 表新增
tags  JSON  DEFAULT NULL  COMMENT '标签: ["奖品", "活动A", "2026春季"]'
```

查询方式：
```sql
-- 按标签筛选
SELECT * FROM media_files WHERE JSON_CONTAINS(tags, '"奖品"')

-- 按标签 + 文件夹组合筛选
SELECT * FROM media_files WHERE folder = 'prizes' AND JSON_CONTAINS(tags, '"活动A"')
```

前端交互：
- 上传时可选填标签（逗号分隔或标签选择器）
- 媒体库浏览器支持按标签筛选
- 支持批量打标签

不需要单独建标签表，JSON 字段 + MySQL 的 JSON_CONTAINS 对你的数据量级完全够用。

### 8.4 回收站机制 — 防误删

问题：当前设计引用计数归零后直接物理删除 Sealos 文件，一旦误删不可恢复。

优化：软删除 + 延迟物理删除。

```
删除流程（改进后）：
  1. detach 解除关联
  2. 引用计数归零 → media_files.status 改为 'trashed'（而非直接删除）
  3. Sealos 文件暂不删除
  4. 管理后台"回收站"页面可以看到 trashed 状态的文件
     → 支持一键恢复（status 改回 'active' + 重新 attach）
     → 支持永久删除
  5. 定时任务：trashed 超过 7 天 → 物理删除 Sealos 文件 + 数据库记录

media_files.status 枚举扩展：
  'active'   → 正常使用
  'trashed'  → 回收站（7天内可恢复）
  'archived' → 归档（保留但不在常规列表显示）
```

新增字段：
```sql
trashed_at  DATETIME  DEFAULT NULL  COMMENT '移入回收站时间'
```

### 8.5 media_attachments 增加 meta 字段 — 解决关联级元数据

问题：ad_creatives 当前有 image_width、image_height、display_mode 字段。方案 D 把图片字段删了，但 display_mode 是运营手动选的展示配置，不等于图片的实际宽高比。这类"关联级元数据"没地方存。

优化：media_attachments 表增加 meta JSON 字段。

```sql
-- media_attachments 表新增
meta  JSON  DEFAULT NULL  COMMENT '关联元数据'
```

各业务场景的 meta 用法：

```javascript
// 商品图片
{ alt_text: '商品正面图', crop_rect: { x: 0, y: 0, w: 800, h: 600 } }

// 活动 banner
{ position: 'top', click_action: 'navigate', target_url: '/lottery' }

// 通用
{ alt_text: '无障碍描述文本' }

// 注：ad_creatives 的 display_mode/link_url/link_type 留在 ad_creatives 表（决策 7），不放 meta
```

分层原则：
- media_files 存图片本身的属性（实际宽高、大小、格式、哈希）
- media_attachments.meta 存"这张图在这个业务场景下的展示配置"

这样 ad_creatives 表可以删除 image_url、image_width、image_height（图片物理属性收归 media_files），但 display_mode / link_url / link_type 留在 ad_creatives 表不动（决策 7：投放逻辑和素材管理分离）。meta 字段用于其他场景的关联级元数据（如 alt_text、crop_rect 等）。

### 8.6 批量操作支持

问题：管理后台有批量工具模块。运营批量创建奖品或批量上传商品时，需要一次性上传多张图并绑定。

优化：MediaService 增加批量方法。

```
MediaService 新增方法：

├── batchUpload(files[])
│   ├── 并行上传多个文件（Promise.all）
│   ├── 批量计算 content_hash 查重
│   ├── 批量创建 media_files 记录（bulkCreate）
│   └── 返回 media_id 数组
│
├── batchAttach(attachments[])
│   ├── 输入：[{ media_id, attachable_type, attachable_id, role, sort_order }]
│   ├── 批量创建 media_attachments 记录（bulkCreate）
│   └── 批量更新业务表的 primary_media_id 缓存
│
└── batchDetach(attachableType, attachableIds[])
    ├── 批量删除 media_attachments
    ├── 批量检查引用计数
    └── 引用计数归零的批量移入回收站
```

API 端点：
```
POST /api/v4/console/media/batch-upload     批量上传（multipart，最多 10 个文件）
POST /api/v4/console/media/batch-attach     批量绑定
POST /api/v4/console/media/batch-detach     批量解绑
```

## 十三、需要用户拍板的决策点（含行业方案对比）

### 决策 1：media_files.media_id 主键类型

#### 各行业怎么做的

| 行业/公司 | 方案 | 具体做法 | 为什么 |
|----------|------|---------|--------|
| **美团** | 自增 BIGINT | 内部图片中心用数据库自增 ID，对外暴露时做 ID 混淆（hash-id 或 snowflake） | 内部系统性能优先，对外安全做一层转换 |
| **阿里 OSS** | UUID/随机字符串 | object key 本身就是唯一标识，不依赖数据库自增 | 分布式对象存储天然无中心化自增 |
| **腾讯 COS** | 混合方案 | 内部用自增 BIGINT 做索引主键，对外暴露 media_id 时用 Snowflake ID（64位分布式ID） | 兼顾索引性能和分布式安全 |
| **小公司/SaaS** | BIGINT 自增 | 绝大多数用数据库自增主键，Sequelize/Rails/Django 默认行为 | 开发成本最低，单库场景够用 |
| **游戏公司（米哈游等）** | 自定义 ID 生成器 | 资源 ID 用自定义规则（如 `asset_10001`），内部存储仍是自增 | 需要可读的资源编号 |
| **虚拟物品交易平台（Steam/BUFF）** | BIGINT 自增 | 图片表用自增主键，物品 ID 才用分布式 ID | 图片只是附属资源，不需要分布式 |
| **活动策划平台（活动行等）** | BIGINT 自增 | 单库单服务，直接自增 | 业务体量不需要分布式 |
| **二手平台（闲鱼/Mercari）** | UUID | 用户上传的 UGC 图片用 UUID，避免 ID 枚举攻击 | UGC 场景对安全性要求高 |

#### 核心区别

用 UUID 的场景有两个硬条件：**多数据库分片**或**ID 直接暴露给外部用户且需要防枚举**。不满足这两个条件的都用 BIGINT 自增。

#### 对你项目的分析

- 你的后端是**单库 MySQL**（`restaurant_points_dev`），不存在分库分片
- 你现有的 **111 个 Sequelize 模型全部用 BIGINT 自增主键**（user_id、lottery_prize_id、exchange_item_id...）
- media_id 只在管理后台和 API 内部使用，不直接暴露给终端用户（小程序端通过 object_key 访问图片，不需要 media_id）
- Sequelize 6 对 BIGINT 自增是零配置，UUID 需要手动配置 `defaultValue: DataTypes.UUIDV4`

**结论：BIGINT AUTO_INCREMENT。** 与你 111 个模型完全统一，零额外配置，查询性能最好。未来如果需要对外暴露，加一层 hashid 转换即可。

### 决策 2：content_hash 去重策略

#### 各行业怎么做的

| 行业/公司 | 方案 | 具体做法 | 为什么 |
|----------|------|---------|--------|
| **美团** | 强制去重 | 上传时计算 MD5/SHA-256，命中则直接返回已有 image_id，不重复存储 | 日均千万图片，存储成本敏感，1% 重复就是 TB 级浪费 |
| **阿里（淘宝）** | 强制去重 | OSS 层面做内容寻址（Content-Addressable Storage），相同内容自动去重 | PB 级存储，去重是刚需 |
| **腾讯** | 强制去重 | COS 支持对象级去重，微信图片系统强制哈希去重 | EB 级存储，不去重会破产 |
| **小公司/SaaS** | 不去重 | 大多数直接存，不做哈希计算 | 图片量小（几百到几千），去重收益<开发成本 |
| **游戏公司** | 强制去重 | Asset Bundle 天然按内容哈希管理，同一资源只打包一次 | 客户端下载量敏感，重复资源增加包体大小 |
| **虚拟物品交易（Steam/BUFF）** | 强制去重 | 同一皮肤的检视截图用 content_hash 去重，相同截图不重复存储 | 用户上传量大，且大量相同物品 |
| **活动策划平台** | 建议复用 | 上传时检测重复，提示"素材库中已有相同图片，是否使用？"，不强制 | 运营需要灵活控制，同一 logo 可能想存两份独立管理 |
| **二手平台（闲鱼/Mercari）** | 强制去重 | content_hash 唯一索引，防止盗图（同一张图出现在多个商品中会被检测到） | 安全需求：去重同时用于盗图检测 |

#### 核心区别

**强制去重**省存储但有副作用：多个业务共享一个 media_file 记录，替换其中一个业务的图片时需要特殊处理（先 detach 旧的，再创建新的），否则会影响其他业务。

**建议复用**多一点存储成本但无副作用：每个上传都是独立的 media_file 记录，替换/删除不影响任何其他业务。

#### 对你项目的分析

- 你的 bucket 当前只有 **54 个文件 / 2.58MB**，存储成本约等于零
- 你的运营场景是管理员在后台上传奖品图、商品图、广告素材，**不是 UGC 大量上传**
- 即使上线后增长到 1000 张图片 / 100MB，Sealos 的存储成本依然可忽略
- 你不需要盗图检测
- 你需要的是运营在管理后台上传时，如果发现已有相同图片，可以选择复用节省操作（而不是强制共享带来的替换副作用）

**结论：普通索引 + 建议复用。** content_hash 用普通索引（非唯一），上传时查重，命中则 API 返回 `{ duplicate: true, existing_media_id: 123 }`，前端提示运营选择复用或新建。你的数据量级完全不需要强制去重来省存储。

### 决策 3：高频业务表是否保留冗余 primary_media_id

#### 各行业怎么做的

| 行业/公司 | 方案 | 具体做法 | 为什么 |
|----------|------|---------|--------|
| **美团** | 保留冗余 | 商品表保留 `cover_image_id`（冗余缓存），指向图片中心。列表页直接 JOIN 图片表，不走关联表 | 商品列表是最高频查询，多一次 JOIN 不可接受 |
| **阿里（淘宝）** | 保留冗余 | 商品表有 `main_image_url` 冗余字段（直接存 URL），详情页才查图片空间 | 商品列表 QPS 极高，连 JOIN 都不要，直接读冗余字段 |
| **腾讯** | 保留冗余 | 朋友圈/公众号文章表有 `thumb_media_id` 冗余字段 | 信息流滚动加载，性能敏感 |
| **小公司/SaaS** | 看情况 | 表少量级小的不冗余（Strapi/WordPress），有列表性能需求的冗余（Shopify） | 取决于列表查询的频率和数据量 |
| **游戏公司** | 保留冗余 | 角色表有 `icon_asset_id` 冗余字段，客户端列表加载只查一次 | 客户端性能极端敏感 |
| **虚拟物品交易** | 保留冗余 | 物品列表展示用 `cover_image_id`，详情页才加载完整图片列表 | 市场列表是核心页面 |
| **活动策划平台** | 不冗余 | 活动列表查询量不大，直接 JOIN activity_media 关联表 | 管理后台查询频率低 |
| **二手平台** | 保留冗余 | 商品表有 `cover_upload_id`，列表页直接 JOIN user_uploads 一次 | 瀑布流加载性能需求 |

#### 核心区别

冗余不冗余取决于一个问题：**你的列表查询是否高频且对响应时间敏感？**

- 冗余方案：列表查询 `SELECT ... FROM exchange_items JOIN media_files ON primary_media_id`（一次 JOIN）
- 不冗余方案：列表查询 `SELECT ... FROM exchange_items JOIN media_attachments ON (type+id+role) JOIN media_files ON media_id`（两次 JOIN）

两次 JOIN 在小数据量下差距不大（毫秒级），但当列表查询变成小程序端的热点接口（每次打开页面都请求）时，单次 JOIN 有明显优势。

#### 对你项目的分析

- **小程序端**的商品列表（exchange_items）、奖品列表（lottery_prizes）是高频查询：用户每次进入兑换页面/抽奖页面都会请求
- **管理后台**的查询是低频的（只有管理员使用），不需要优化
- 你的 Sequelize 已有的查询模式是 `include: [{ model: X, as: 'y' }]`，冗余 FK 可以直接用 `include`，代码最简洁
- 冗余字段由 MediaService 在 attach/detach/replace 时自动维护，业务层无感
- 需要冗余的表只有 4 个（exchange_items/lottery_prizes/item_templates/ad_creatives），其余低频表（category_defs/material_asset_types/merchants）不需要

**结论：保留 primary_media_id 冗余缓存。** 4 个高频业务表加一个冗余 FK，由 MediaService 自动维护。低频表不加，直接走 media_attachments JOIN。这是美团/淘宝/Steam 等所有有列表性能需求的系统的标准做法。

- 7 天足够覆盖"发现误删 → 联系管理员恢复"的场景

**结论：7 天回收站。** 开发成本极低（2 小时），防误删收益明确，存储成本可忽略。活动策划平台和二手平台的 7 天标准适合你的体量。

### 决策 7：ad_creatives 的 display_mode / link_url / link_type 放哪里

#### 真实数据库数据（ad_creatives 表 3 条记录）

| ad_creative_id | display_mode | link_url | link_type | image_url (object key) | image_width | image_height |
|----------------|-------------|----------|-----------|----------------------|-------------|-------------|
| 5 | tall | (空) | none | popup-banners/...797.jpg | 819 | 1092 |
| 6 | tall | (空) | none | popup-banners/...361.jpg | 648 | 592 |
| 7 | wide | (空) | none | carousel/...546.png | 345 | 569 |

这 6 个字段分两类：
- **图片物理属性**：image_url、image_width、image_height — 描述"这张图片本身长什么样"
- **广告业务属性**：display_mode、link_url、link_type — 描述"这个广告怎么展示、点了跳哪里"

#### 各行业怎么做的

| 行业/公司 | 图片物理属性放哪 | 业务展示属性放哪 | 为什么 |
|----------|----------------|----------------|--------|
| **美团** | 图片中心（独立服务） | 广告表/商品表 | 图片中心只管"图是什么"（宽高、格式、大小），不管"图怎么用"。展示模式、跳转链接是广告投放逻辑，跟图片无关 |
| **阿里（淘宝直通车）** | OSS 元数据 | 广告计划表（ad_plan） | 同一张图可以在不同广告计划中用不同的展示模式和跳转链接。如果 display_mode 跟图片绑定，就没法复用 |
| **腾讯（广点通）** | COS / 数据万象 | 创意表（creative） | 创意表存投放参数（展示样式、落地页、按钮文案），素材表只存文件元数据 |
| **小公司/SaaS** | 业务表或 media 表 | 业务表 | 多数小公司不做分离，图片字段和业务字段混在一张表。但做了 media 拆分的（如 Strapi），业务属性一定留在业务表 |
| **游戏公司** | 资源包（Asset Bundle） | 配置表（活动配置/banner 配置） | 游戏内的 banner 展示方式（全屏/半屏/滑动）和点击行为（跳转商城/跳转活动）在运营配置表里，不在资源表里 |
| **活动策划平台（活动行）** | 素材库（media_library） | 活动表（activity） | 同一张海报可能在不同活动中用不同的跳转链接。跳转链接是活动属性，不是图片属性 |
| **虚拟物品交易（Steam）** | 图片表（item_images） | 挂单表（listing） | 物品截图是通用的，但展示优先级、是否置顶是交易业务逻辑 |
| **二手平台（闲鱼/Mercari）** | 上传表（user_uploads） | 商品表（listing） | 同一张图可能出现在多次上架中。商品描述、展示位置是商品属性 |

#### 核心区别

一个检验标准：**同一张图片能不能在不同业务场景下用不同的 display_mode？**

如果 display_mode 存在 media_attachments.meta 里，那同一张图被两个广告引用时，两个广告必须共享相同的 display_mode — 这不合理。运营完全可能把同一张图在弹窗广告里用 `tall` 模式、在轮播广告里用 `wide` 模式。

如果 display_mode 留在 ad_creatives 表，每个广告创意独立控制自己的展示模式 — 图片只管"我是一张 819x1092 的 jpg"，广告说"我要用 tall 模式展示这张图"。

#### 对你项目的分析

- display_mode 的可选值是 `tall` / `wide`，这是**运营在创建广告时手动选择的展示模式**，跟图片内容无关
- link_url 是广告点击后的跳转目标，100% 是广告业务逻辑
- link_type 是跳转类型（none/miniprogram/webview），100% 是广告业务逻辑
- 你的 ad_creatives 表本身就有 `title`、`content_type`、`text_content`、`review_status` 等广告业务字段，display_mode / link_url / link_type 跟这些同属一类
- image_width / image_height 是图片的物理像素尺寸，这个确实应该跟图片走（存在 media_files.width / media_files.height）

**结论：display_mode / link_url / link_type 留在 ad_creatives 表不动。** 只把 image_url / image_width / image_height 删掉，换成 primary_media_id 指向 media_files。宽高信息从 media_files 获取。这是所有大厂广告系统的标准做法 — 投放逻辑和素材管理分离。

之前文档中步骤 2.1 来源 B 里写的 `meta={display_mode, link_url, link_type}` 是错误的，应该修正为：这三个字段留在 ad_creatives 表，不进 meta。

#### 真实数据库数据分类

```
有真实图片数据（30 个 Sealos 文件被引用）：
  exchange_items     2 条，primary_image_id 指向 image_resources

  merchants           1 条，logo_url NULL

特殊场景：
  users              82 条，仅 1 条有 avatar（见决策 9 单独讨论）
```

#### 各行业怎么做的

| 行业/公司 | 分期策略 | 具体做法 | 为什么 |
|----------|---------|---------|--------|
| **美团** | 按业务线分期 | 外卖先接入图片中心 → 酒旅 → 到店 → 其他。每条业务线 2-4 周 | 线上系统不能停，按业务线切流量最安全 |
| **阿里** | 按风险分级 | 低风险业务先接入 → 核心电商后接入。每批 1-2 个月 | 交易核心链路风险高，放最后 |
| **腾讯** | 按团队分期 | 每个事业部独立排期，平台先行建好中间件 | 大公司组织架构决定了不可能一次全做 |
| **小公司（已上线）** | 分期 | 新功能用新方案，旧功能逐步迁移，典型周期 1-3 个月 | 没有足够人力一次全改，且线上系统需要保证稳定 |
| **小公司（未上线）** | 一期全做 | 趁没上线一次改完，不留技术债 | 没有线上用户，不存在分批切流量的需求。改一半留一半反而增加认知成本 |
| **游戏公司** | 版本切换 | 新版本一次性切换所有资源管理方案，旧资源包在打包时自动转换 | 游戏更新是全量切换，不存在部分表用新方案部分用旧方案 |
| **活动策划平台** | 一次性 | 所有活动类型同时切换到新素材库 | 内部系统，用户就是自己团队，不需要灰度 |
| **虚拟物品交易** | 一期全做 | 物品图片、用户截图、商品封面同一批迁移 | 交易系统的图片关联是紧耦合的，分批改会导致部分查询走新表部分走旧表 |
| **二手平台（闲鱼）** | 按模块分期 | 商品图片先迁 → 头像 → 社区内容图。每批 2-4 周 | 用户量大，需要灰度 |

#### 核心区别

分期的唯一合理原因是：**有线上用户和线上流量，需要灰度切换降低风险**。

不分期的合理原因是：**项目未上线，分期会导致代码里同时存在新旧两种图片管理方式，增加认知成本和出错概率**。

如果你分两期：
- 代码里 exchange_items 用 `primary_media_id` + `media_files`，lottery_prizes 还用 `image_resource_id` + `image_resources`
- 前端开发要记住"这个页面走新 API，那个页面走旧 API"
- 测试要覆盖两种路径
- 等做第二期时还要再改一次路由/前端/测试

如果你一期全做：
- 全 NULL 的表（lottery_prizes/item_templates/lottery_campaigns/merchants）改造量极小：只是删一个旧列、加一个 primary_media_id 列、改一下 Sequelize include 查询。无数据迁移，无 Sealos 操作
- 有数据的表（exchange_items/ad_creatives/category_defs/material_asset_types）也只有 30 条记录需要迁移
- 全部改完后，整个项目只有一种图片管理方式，零认知负担

#### 对你项目的分析

- **项目未上线**，零线上风险
- 有真实数据的只有 30 条记录（2+3+9+16），迁移脚本 30 分钟写完
- 全 NULL 的 4 个表改造是纯 schema 操作（删旧列 + 加新列 + 改查询），每个表约 30 分钟
- 分两期的额外管理成本（代码里维护两种模式、前端记住哪些走新哪些走旧）远大于一期多改 4 个空列的成本
- 你明确说了"愿意一次性投入大量成本"

**结论：一期全部改完（不含 users.avatar_url，见决策 9）。** 8 个表一次迁移，全项目统一到一种图片管理方式。这是所有未上线项目和内部系统的标准做法。

### 决策 9：users.avatar_url 是否纳入 media 体系

#### 真实数据库数据

82 个用户，仅 1 条有 avatar：
```
user_id: 11494, nickname: "测试商户-张三", avatar_url: "test-seeds/avatars/merchant-01.png"
```

这是测试种子数据。上线后真实用户的头像来源是**微信登录返回的头像 URL**（`https://thirdwx.qlogo.cn/mmopen/...`），不是用户上传到 Sealos 的文件。

#### 各行业怎么做的

| 行业/公司 | 用户头像放哪 | 具体做法 | 为什么 |
|----------|------------|---------|--------|
| **美团** | 用户表直存 URL | `users.avatar_url` 存微信/手机号登录后获取的头像 URL。不走图片中心 | 头像来源是第三方（微信/QQ），不是用户上传到美团服务器的。图片中心只管美团自己存储的图片 |
| **阿里（淘宝）** | 用户表直存 URL | `users.avatar` 存支付宝/淘宝头像 URL。自定义头像上传后存到 OSS 再写回 avatar 字段 | 默认头像是第三方 URL，自定义上传是小众场景 |
| **腾讯（微信）** | 用户表直存 URL | 微信用户表存头像 URL（`headimgurl`），不走 COS 图片管理体系 | 微信头像本身就是腾讯 CDN 上的资源，不需要再存一份 |
| **小公司/SaaS** | 用户表直存 URL | 绝大多数项目 `users.avatar_url` 存第三方社交登录返回的 URL | 简单直接，头像不需要缩略图、标签、引用计数等 media 功能 |
| **游戏公司** | 混合方案 | 默认头像是预设的资源 ID（走 Asset Bundle），自定义头像存 URL | 游戏有预设头像框/角色头像，不是用户上传的 |
| **虚拟物品交易（Steam）** | 用户表直存 URL | Steam 头像存 URL 指向 CDN。交易物品的截图才走图片管理 | 头像是个人资料的一部分，不是交易资源 |
| **活动策划平台** | 不管头像 | 用微信/钉钉登录后直接用第三方头像，不存本地 | 内部系统不需要头像管理功能 |
| **二手平台（闲鱼）** | 用户表直存 URL | 支付宝头像 URL。用户上传的商品图才走图片管理体系 | 头像和商品图是两个完全不同的管理域 |

#### 核心区别

头像要不要走 media 体系取决于两个问题：

**问题 1：头像的来源是什么？**
- 如果来源是**第三方登录**（微信/QQ/支付宝），头像是一个外部 URL（`https://thirdwx.qlogo.cn/...`），不是你 Sealos 上的文件。media_files 表的设计假设是"存储在 Sealos 上的文件的 object_key"，外部 URL 硬塞进去会破坏这个假设
- 如果来源是**用户上传到你服务器**，那走 media 体系是合理的

**问题 2：头像需要 media 体系的哪些功能？**
- 缩略图？不需要。微信头像本身就是小图（132x132 或 0/46/64/96/132 多尺寸）
- 引用计数？不需要。头像只有一个用户引用
- 标签/搜索？不需要。没人在管理后台搜索用户头像
- 回收站？不需要。换头像就是覆盖，不需要恢复旧头像
- 统一管理？不需要。管理员不需要"管理用户头像"这个功能

如果把头像纳入 media 体系：
1. 微信登录返回头像 URL 后，需要先**下载**到服务器，再**上传**到 Sealos，再创建 media_files 记录，再创建 media_attachments 关联 — 多 3 步操作
2. 微信头像 URL 会过期或变化，需要定期刷新并重新下载
3. 用户换头像时需要 detach 旧图 + attach 新图 + 引用计数检查 + 可能触发回收站 — 一个简单的 UPDATE 操作变成了 4 步事务
4. 82 个用户 × 每人 1 张头像 = 82 条 media_attachments 记录，全是无意义的 1:1 映射

#### 对你项目的分析

- 你的登录方式是**微信小程序登录**（.env 有 WX_APPID / WX_SECRET），头像来源是微信 `wx.getUserProfile()` 返回的 `avatarUrl`，是 `https://thirdwx.qlogo.cn/...` 格式的外部 URL
- 你的 media_files 表设计是存 Sealos 的 `object_key`（如 `products/xxx.jpg`），不是存外部 URL
- 用户不会在管理后台上传头像，头像是登录时自动获取的
- 头像不需要 media 体系的任何高级功能（缩略图/引用计数/标签/回收站）
- 把外部 URL 塞进 media_files 会破坏表的设计语义，增加无意义的复杂度

**结论：users.avatar_url 不纳入 media 体系，保持现状。** avatar_url 字段继续存微信头像 URL 或测试种子数据的 Sealos key。这是美团/淘宝/微信/Steam/闲鱼所有主流平台的一致做法 — 第三方社交登录的头像直存 URL，只有用户主动上传到你服务器的图片才走图片管理体系。

如果未来需要"自定义头像上传"功能，到时候再加一个入口：用户上传 → MediaService.upload() → 拿到 object_key → 写回 users.avatar_url。avatar_url 字段可以兼容存 Sealos key 或外部 URL（ImageUrlHelper 已有 URL 格式检测能力）。

**111 张表中仅 `category_defs` 使用非整数主键**（`category_code` 如 `'food'`、`'electronics'`、`'collectible'`），但它恰好是需要图片管理的 9 张表之一（9 条 icon_url 全部有数据）。

#### category_defs.category_code 的实际影响范围（真实数据库 FK 查询）

3 张表通过外键引用 `category_defs.category_code`：

```
exchange_items.category                  → category_defs.category_code  (2 条记录)
item_templates.category_code             → category_defs.category_code  (13 条记录)
market_listings.offer_item_category_code → category_defs.category_code
```

如果选方案 B（给 category_defs 加整数 PK），需要改动 4 张表 + 4 个模型 + 相关路由。数据量极小（9+2+13=24 条需要迁移），成本是 30 分钟的迁移脚本。

#### 各行业怎么做的

| 行业/公司 | 多态关联的 ID 字段类型 | 具体做法 | 为什么 |
|----------|---------------------|---------|--------|
| **美团** | BIGINT | 美团内部图片中心用 BIGINT 做关联 ID。但美团的 DBA 规范**强制所有表使用自增整数主键**，不允许 VARCHAR PK 出现。所以美团不存在这个问题 — 从源头消灭了 VARCHAR PK | 大厂 DBA 团队有统一建表规范，所有表 PK 必须是 `id BIGINT AUTO_INCREMENT`，业务码作为 UNIQUE 索引而非 PK |
| **阿里（淘宝/天猫）** | BIGINT | 阿里的商品类目表有 `category_id BIGINT` 和 `category_code VARCHAR` 两个字段。PK 是 category_id，category_code 是业务唯一键。图片关联用 category_id | 阿里数据中台强制"双键设计"：每张表必须有自增数字 PK + 业务唯一键。数字 PK 用于 JOIN 和关联，业务键用于人类识别和 API 传参 |
| **腾讯（微信/QQ）** | BIGINT | 微信的 `content_type` + `content_id` 多态关联，content_id 是 BIGINT。腾讯的表设计规范同样强制数字 PK | 微信生态的表设计全部走整数 PK，VARCHAR 字段只做业务标识 |
| **小公司/SaaS（Shopify/Strapi/WordPress）** | BIGINT/INTEGER | **Shopify**（Ruby on Rails）：用 BIGINT，Rails 默认所有表有 `id` 整数 PK。**Strapi**（Node.js）：用 INTEGER。**WordPress**：用 BIGINT（`wp_postmeta.post_id`） | 主流 ORM（Rails ActiveRecord、Sequelize、Django ORM）默认生成整数自增 PK，SaaS 产品天然都是 BIGINT 关联 |
| **Django 项目（Instagram 早期、Pinterest）** | **CharField（字符串）** | Django 的 `GenericForeignKey` 框架用 `object_id` + `content_type`。Django 官方文档**推荐 object_id 用 CharField**，原因是："你可能需要关联到使用非整数主键的模型" | Django 是**唯一一个官方推荐字符串多态 ID** 的主流框架，因为 Django 社区有大量使用 UUID 或自然键做 PK 的项目 |
| **Laravel 项目（Spatie Media Library）** | BIGINT（默认） | Laravel 的 `morphTo()` 默认用 `model_id BIGINT UNSIGNED`。如果某张表有 VARCHAR PK，需要手动改为 `string` 类型。Spatie 文档说："如果你的模型使用 UUID 主键，需要自定义迁移" | Laravel 默认假设所有表用整数 PK，非整数 PK 被视为特殊情况需要额外处理 |
| **游戏公司（米哈游/网易游戏）** | INTEGER | 游戏的 asset_references 表用 `entity_id INTEGER`。游戏数据库**绝对不允许 VARCHAR PK**。所有"代码"（角色代码、武器代码）都只是 UNIQUE 索引，PK 永远是整数 | 游戏 QPS 极高，整数索引 B+ 树比较效率比字符串高。热更新配置表需要高频 JOIN，整数 PK 是硬性要求 |
| **虚拟物品交易（Steam/BUFF）** | INTEGER | item_image_bindings 用 `item_id INTEGER`。物品分类表有 `category_id INT` + `category_internal_name VARCHAR`，PK 是 category_id | 交易平台和游戏公司一脉相承，整数 PK 是标配 |
| **活动策划平台（活动行/互动吧）** | INTEGER/BIGINT | activity_media 用 `activity_id INTEGER`。活动类型表有 `type_id INT` + `type_code VARCHAR`，PK 是 type_id | 活动平台数据量虽小但也遵循整数 PK 惯例 |
| **二手平台（闲鱼/Mercari）** | BIGINT | listing_images 用 `listing_id BIGINT`。分类表用 `category_id BIGINT` + `category_slug VARCHAR`，PK 是 category_id | 二手平台的分类体系和电商一样，双键设计 |

#### 核心区别

**两个流派**：

**流派 1：强制整数 PK，改异常表（美团/阿里/腾讯/游戏/交易/活动/二手平台 — 绝大多数）**
- 从源头规定所有表必须有整数自增 PK
- VARCHAR 业务码只能做 UNIQUE 索引，不能做 PK
- 多态关联的 ID 字段用 BIGINT
- 遇到 VARCHAR PK 的表 → **改那张表**，加整数 PK，不改多态关联表
- 理念："全局表设计规范比单个功能需求更重要"

**流派 2：兼容任意 PK 类型（仅 Django）**
- 多态 ID 字段用 CharField/VARCHAR
- 可以关联整数 PK 也可以关联字符串 PK
- 不要求被关联表改动
- 理念："多态关联应该适配被关联表"

**流派 1 占压倒性多数**。在 Node.js + Sequelize 生态中，没有主流库使用 VARCHAR 做多态 ID。

#### 对你项目的深度分析

**category_defs 用 VARCHAR PK 本身就是技术债务**：

1. 你的 111 张表中 **110 张**用整数自增 PK，仅 `category_defs` 用 VARCHAR PK — 这不是"设计风格"，这是**一致性缺陷**
2. category_code 做 PK 带来的 3 个 FK 约束全部是 VARCHAR FK JOIN，比整数 JOIN 慢（虽然你的数据量级不敏感）
3. 如果未来需要改 category_code 的值（如把 `'food'` 改成 `'food_drink'`），因为是 PK 且有 3 个 FK 引用，需要 CASCADE UPDATE 3 张表 — 这就是为什么大厂不允许 VARCHAR PK
4. 你的 Sequelize 全局配置 `underscored: true` + `freezeTableName: true` 假设所有表有 `xxx_id` 格式整数 PK，category_defs 的 VARCHAR PK 打破了这个假设

**改 category_defs 的真实成本（基于数据库 FK 查询）**：

```
category_defs       → 加 category_def_id INT AUTO_INCREMENT 做新 PK，category_code 降级为 UNIQUE KEY
                      只有 9 条记录，回填 9 个自增 ID
exchange_items      → category VARCHAR(50) FK → 改为 category_def_id INT FK
                      只有 2 条记录
item_templates      → category_code VARCHAR(50) FK → 改为 category_def_id INT FK
                      只有 13 条记录
market_listings     → offer_item_category_code VARCHAR(50) FK → 改为 offer_category_def_id INT FK
                      通过 FK 关联
对应 Sequelize 模型  → 4 个模型的关联定义更新
```

**总计 24 条数据迁移，30 分钟的迁移脚本，手动验证每一条。**

#### 三个方案的完整对比

| 维度 | 方案 A：attachable_id 改 VARCHAR(100) | 方案 B：category_defs 加整数 PK | 方案 C：category_defs 不走 media |
|------|--------------------------------------|-------------------------------|--------------------------------|
| 改动范围 | 只改 media_attachments（新表） | 改 category_defs + 3 张引用表 + 4 个模型 | 零改动 |
| 迁移成本 | 零（新表直接建） | 30 分钟迁移脚本（9+2+13 条数据） | 零 |
| 技术债务 | **新增一个债务**：111 张表都是整数 PK + JOIN，唯独 media_attachments 用 VARCHAR JOIN | **消除一个债务**：category_defs 终于跟其他 110 张表统一为整数 PK | **保留一个例外**：category_defs 不在统一 media 体系中 |
| 长期维护 | 每次查 media_attachments 时 attachable_id 的类型与业务表 PK 不匹配（VARCHAR vs INT），Sequelize 查询需要类型转换 | 全系统 PK 统一整数，所有 JOIN 都是整数比较，零心智负担 | 需要记住"category_defs 是例外，icon_url 不走 media" |
| 行业对标 | Django（少数派） | 美团/阿里/腾讯/游戏/交易/活动/二手平台（绝大多数） | 无行业对标，是妥协方案 |
| 是否符合你的技术栈 | ⚠️ 不太符合 — Sequelize 全局配置是 `underscored: true` + 整数 PK，VARCHAR 多态 ID 是非标准用法 | ✅ 完全符合 — 消除唯一的 VARCHAR PK 异常，全系统统一 | ⚠️ 不符合 — 破坏"全部统一到 media 体系"的设计目标 |
| 未来扩展 | 如果未来新增其他 VARCHAR PK 的表，天然兼容 | 所有表都是整数 PK（本来就应该是），不需要兼容 | 每遇到一个非整数 PK 的表就多一个例外 |
| 代码复杂度 | MediaService 内部需要处理 attachable_id 类型（有时是数字字符串 `'234'`，有时是纯字符串 `'food'`） | MediaService 内部 attachable_id 始终是整数，零歧义 | MediaService 逻辑不变，但 category_defs 的图片逻辑单独维护 |

#### 结论

**方案 B：给 category_defs 加整数自增 PK，attachable_id 保持 BIGINT。**

核心理由：

1. **你说了"项目还没上线、愿意一次性投入大量成本、不要兼容旧接口、从长期维护成本低和降低技术债务的角度出发"** — 方案 B 是唯一能**消除**技术债务的方案。方案 A 和 C 都是**绕过**或**新增**技术债务

2. **美团/阿里/腾讯/游戏公司/交易平台/活动平台/二手平台**全部使用整数 PK。你的 110 张表已经遵循了这个规范，只有 category_defs 是例外。趁未上线修正，代价极小（30 分钟迁移脚本，改 24 条数据）

3. **核心思维**：改 category_defs 是"修正一张表的历史遗留问题"，改 media_attachments 是"为了迁就一张异常表而让新系统妥协"。应该让异常的那个变正常，而不是让正常的那个去适配异常

4. 改完之后全系统 111 张表统一整数自增 PK，media_attachments.attachable_id 用 BIGINT，所有 JOIN 都是整数比较。**零例外、零特殊处理、零心智负担**

5. 与你的后端技术栈完全一致 — Sequelize 全局 `underscored: true` + 整数 PK 风格不被打破
