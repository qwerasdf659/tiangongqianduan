## 〇、拍板决议（2026-07-10 终稿，执行以本节为准）

架构结论：现有等级体系（累计积分单调递增、实时派生、不衰减、阈值表驱动）即"腾讯型成长值 + 游戏VIP阈值曲线"组合，为长期维护成本最低方案，**架构不动，只定数字**。不采用电商大厂滑动窗口衰减方案（需数仓/快照/降级链路，技术债过重）。

3. **【运营】配置试点配方（拍板④）**：管理后台"以物易物配方"页先写 1 条（毛巾礼盒×2 → 新毛巾礼盒），产出价值 ≤ 投入价值，验证后再扩充。

### 1.4 产出快递到家改造方案（拍板⑩，新增开发项）

**现状与目标的差距**：换物订单创建时直接 `status='completed'`、不收地址，产出物只能背包+到店核销；而快递链（地址快照 `address_snapshot`、运单号 `shipping_no`、`ShippingTrack` 轨迹、快递100/快递鸟双通道、7 天自动确认收货）全部挂在 `ExchangeRecord` 的 pending→approved→shipped→received 生命周期上，换物订单从未进入。

**改造原则：复用不新建**——换物与普通实物兑换共用同一张订单表和同一条发货链，改动点收口在 BarterService 与其路由：

1. `POST /api/v4/exchange/barter` 增加 `address_id` 参数（实物产出必填，校验地址归属本人，复用 `UserAddressService.buildSnapshot()` 写入 `address_snapshot`）；
2. `BarterService.executeBarter` 订单状态改：产出为实物（模板 `item_type='product'`）→ `status='pending'` 走发货链；产出为券/道具 → 维持 `completed` 即时到账；
3. 实物产出**不再 mint 进背包**（对齐普通实物邮寄兑换的处理：`mint_instance` 与发货链二选一），防止"背包里能核销 + 快递又寄一件"双重履约；
4. 发货/轨迹/确认收货全部复用现有管理端发货操作与 webhook，零新增接口；运费按拍板⑯-(b) 平台包邮（延续现状，无运费字段无新逻辑）。

**投入边界（拍板⑩前半）无需改动**：`_loadAndValidateOldItems` 已强制 `status='available'`——已核销实物（used）、核销订单锁定中（held）的物品天然不可投入，与"电子态才可换"的业务规则完全一致。

### 2.4 发放线九档阶梯改造方案（拍板②，新增开发项）

**改造点收口在消费审核链路（`services/consumption/CoreService.js`）与积分累加口径（`services/asset/BalanceService.js`）**：

1. **等级倍数配置落位**：`user_growth_levels` 表加 `earn_multiplier DECIMAL(4,2)` 列（一次迁移），九档写入 §0-② 阶梯值——倍数与等级定义同表，管理后台调档零散配置不分家；
2. **可用积分——发放组合器（拍板⑮-(e) 升级）**：审核通过发分改为"基础笔 + N 个加成笔"（同事务）——基础笔 `round(消费金额 × points_award_ratio)`（business_type='consumption_reward'，计入 `history_total_points`）+ 等级加成笔 `round(基础 × (earn_multiplier - 1))`（business_type='level_bonus_reward'，**不计** `history_total_points`）+ 未来活动加成笔（business_type='activity_bonus_reward'，同样不计）。各加成独立按基础分计算（加法叠加），总倍数封顶 3.0，加成规则做成列表——新增活动加成即插规则，不碰主逻辑；
3. **防复利实现**：`BalanceService.changeBalance` 的 `history_total_points` 累加处（该字段唯一写收口）增加 business_type 排除名单（`level_bonus_reward`）——等级永远由"基础分≈真实消费"驱动，杜绝"高等级→加成→更快升级"复利；
4. **预算积分同步（含拍板⑮-(c) 硬封顶）**：预算分配处 `round(消费金额 × budget_allocation_ratio × 加法叠加总倍数)`，且注入率硬封顶 16%（`min(计算值, 消费金额 × 0.16)`，运营手动发放除外）——预算变厚 → 预算分层升档 → 现有 BxPx 闸门自然放行更高奖品档位；
5. **等级倍数提交时锁定（拍板⑬-(a) 修订）**：小票**提交时点**用 `UserGrowthLevelService.resolveUserLevel` 派生等级、锁定倍数并随 `points_to_award` 一并存入消费记录——与 `points_award_ratio` 现有"提交时锁定，保证用户承诺一致"原则完全对齐，审核快慢不影响到账金额；
6. **升级即时通知（拍板⑬-(d)）**：审核发分后对比发分前后等级（两次 `resolveLevelKey`），跨档则经 `NotificationService` 发站内通知（如"恭喜升级金卡，您的积分回馈已提升至 1.1 倍"）——等级实时派生无升级事件，此处是唯一可靠的触达点。

**应急回滚预案（拍板⑬-(b)）**：`earn_multiplier` 九档全部改回 1.00 → 加成笔金额为 0 不再产生、预算注入回到 8% 基准，系统行为**完全回到现状**。成本失控或发现缺陷时回滚只需改九个数字（管理后台等级管理页操作），无需回滚代码、无需 feature flag。

**用户可见性口径（2026-07-10 确认）**：
- **预算积分对用户完全不可见（现有机制，零改动）**：`BUDGET_POINTS` 在 `DataSanitizer` 全局前端资产黑名单（`FORBIDDEN_FRONTEND_ASSET_CODES`）中——余额接口拒查、资产列表剔除、流水记录整条过滤；加成后的预算笔同样被拦，用户只感知结果（更容易抽中好奖）不见机制；
- **加成笔主动展示（权益感知点）**：`level_bonus_reward` 为独立流水，C 端积分明细可显示"会员等级加成 +X"（京东PLUS"多返京豆"同款展示），用户每次消费直观看到等级价值。

**明确不做**：`level_probability` 概率倍数不配置（保持默认 1.0）；`TierPickStage` 的 B 线代码保留不动（未来若需活动级概率活动可复用）。

### 2.5 兑换商城等级区间门槛（拍板⑪，新增开发项）

**现有一半**：`exchange_redeem_requirement.min_growth_level_key`（及以上）已实现——按商品/SKU 配置（SKU 精确优先回退商品级）、启停 + 生效时间窗、`CoreService._assertRedeemRequirement` **扣费前**校验并给出明确提示。

**扩展1——等级上限（及以下）**：
1. 表加列 `max_growth_level_key VARCHAR(32) NULL`（一次迁移，NULL=不限）；
2. 校验：`resolveLevelKey` 派生用户当前等级，按 `sort_order` 比较 `min ≤ 用户等级 ≤ max`；超上限报 `REDEEM_GROWTH_LEVEL_EXCEEDED`（文案如"银卡及以下会员专享"）；
3. 组合语义：min 单配=及以上（拍板③划线）、max 单配=及以下（新人专享）、min=max=仅某等级（v9 专属纪念品）、双 NULL=不限；
4. 管理端写入校验：min/max 必须为启用中 level_key，min.sort_order ≤ max.sort_order（防倒挂）。

**扩展2——C 端"锁住但可见"**：商品列表/详情（`QueryService`）为有门槛商品附加脱敏摘要 `level_requirement: { min_level_name, max_level_name, satisfied }`——`satisfied` 后端按当前用户实时计算，只下发等级展示名（不下发倍数/权重，守脱敏红线）；前端渲染专享角标 + 置灰按钮。橱窗效应：低等级用户看得见高等级专享商品，比运营文案更能拉动消费升级。

**工作量**：一列迁移 + 校验分支约十行 + 管理端字段校验 + `QueryService` 摘要字段 + 契约测试；全部收口 exchange 域现有文件，无新表无新链路。

## 八、复核期两项拍板（⑰⑱ 均已落定，2026-07-10）

**⑰ 积分是否过期——已拍板（2026-07-10 终稿）：积分永不过期。**

行业方案谱系（拍板依据）：
- **电商大厂（阿里淘金币年底清零 / 京东京豆按年滚动过期 / 美团短效期券点）**：过期是"促复购杠杆 + 负债出清（breakage）"，代价是 FIFO 计龄批次表、过期扣减定时链路、到期提醒触达、清零客诉承接四套系统——高频海量业态才摊得平这套成本；
- **航司（里程 3 年滚动，高卡免过期）**：里程是真金负债，靠过期控财务敞口，同样养着整条计龄+提醒链路；
- **游戏公司（主流双轨制）**：充值硬通货**永不过期**（口碑+合规红线），限时紧迫感全部交给赛季代币/活动货币到期清零——主货币零客诉、活动货币天然出清；
- **小公司/单商户餐饮系统**：绝大多数积分永不过期，因为过期系统的维护成本远超单商户的负债出清收益。

选定理由（贴本项目实际）：中高端餐饮、宴席大客户、等级终身（拍板⑦）——大客户攒的积分被清零是最伤的客诉；且**本后端已内建游戏双轨制全部基建**：`points` 永续 + `event_points` 活动货币按活动分桶、活动结束自动清零（`EventBudgetService` + `jobs/daily-event-budget-expiry.js`），"限时刺激"零开发即有承载。选过期路线反而要新建计龄/扣减/提醒三件套（纯增技术债）。
落地动作（已执行/已修订）：`system_settings.points_expire_days` 365→0（直连执行并回读验证）；B7 开发项永久出列；P1 看板改为积分负债余额监控（无 breakage 项）；沉睡召回改由 event_points 运营活动承担（§4.4）。

**⑱ 风控阈值存放方式——已拍板（2026-07-10 终稿）：迁入 system_settings 配置化。**

行业方案谱系（拍板依据）：
- **大厂（美团/蚂蚁风控中台）**：独立规则引擎 + 配置中心热更新 + 模型分灰度——单商户体量纯属过度设计；
- **中型公司（选定）**：阈值进配置中心，代码只读配置，改阈值即改即生效不发版；
- **小公司**：硬编码常量，改阈值=发版，与"长期维护成本最低"目标相悖。

选定理由：本项目已有现成"配置中心"——`system_settings` + 白名单校验机制（`points_award_ratio`/`budget_allocation_ratio` 均在其中），迁入即符合"运营参数不进代码"的既有技术路线；桌均口径必随菜单/业态调整，配置化一次到位。
落地口径（并入 §7.3 第 8 步）：`config/system-settings-whitelist.js` 补 `risk/anomaly_large_amount_threshold`（默认 5000）与 `risk/anomaly_new_user_large_threshold`(默认 3000) 两个 key；`AnomalyService` 检测时经 `AdminSystemService.getSettingValue` 读配置替代 ANOMALY_RULES 内写死的 threshold/amount_threshold（规则权重/天数窗口等结构性参数仍留常量）；管理后台系统设置页自然接管（现有 system-settings 界面通用编辑，零新增 UI）。

> 所有接口走后端统一响应壳 `{ success, code, message, data, timestamp, version, request_id }`，字段一律 snake_case，**直接消费原字段，不建任何驼峰映射/适配层**。时间字段为 UTC ISO8601 带 Z，展示层转北京时间。

### M1 换物入口页

- **查配方**：`GET /api/v4/exchange/barter/recipes` → `data.recipes[]`（仅启用项），字段：`recipe_code / name / required_item_template_id / required_quantity / output_exchange_item_id / per_user_limit / total_limit`。
- **换物订单不可逆**：barter 订单（source='barter'）不可取消/退款（订单能力位 `refundable=false`，强行调用返回 `BARTER_ORDER_IRREVERSIBLE`），"我的订单"列表对 barter 单隐藏取消/退款按钮；商品名读 `item_snapshot.item_name`（与普通兑换单同字段）。
- **执行换物**：`POST /api/v4/exchange/barter`
  - Header 必带 `Idempotency-Key`（唯一幂等键）；
  - body：`{ recipe_code, old_item_ids: number[], address_id? }`；
  - **实物产出必传 `address_id`**（缺失返回 `BARTER_ADDRESS_REQUIRED`）；券/道具产出无需地址；
  - 返回 `{ order_no, order_status, consumed_item_ids, minted_item }`：`order_status='pending'`=实物走快递发货链、`'completed'`=券/道具即时到账（`minted_item` 为背包物品，实物时为 null）；
  - 错误码原样处理：`BARTER_RECIPE_NOT_FOUND / BARTER_ITEM_NOT_AVAILABLE / BARTER_DIRECTION_UPWARD_FORBIDDEN / BARTER_PER_USER_LIMIT_EXCEEDED / BARTER_TOTAL_LIMIT_EXCEEDED / BARTER_ADDRESS_REQUIRED / BARTER_OUTPUT_TEMPLATE_MISSING`（末项为运营配置缺陷兜底：产出商品未挂模板，正常不会出现，出现时提示用户稍后再试即可）。
- **首条真实配方已上线（2026-07-11）**：`towel_set_x2_to_new_towel_set`（毛巾礼盒×2 → 新毛巾礼盒，每人限 1 次、总量 100，实物快递履约）——小程序可直接联调。

### M2 会员尊享页

- `GET /api/v4/user/growth-level` → 直接渲染后端脱敏视图：
  - `current_level_key / current_level_name`（如 v9 / 荣耀殿堂）；
  - `history_total_points`（累计积分）；
  - `thresholds_confirmed`（阈值是否定稿，false 时 `levels[].min_history_points` 为 null，仅显示等级名）；
  - `levels[]`（等级阶梯，含门槛数字）；
  - **`next_level: { level_key, level_name, points_needed }`**（顶档为 null）→ 渲染"再消费 X 元升{下一级名}"（1 元≈1 积分）。
- **禁止**在小程序侧自算等级或要求下发倍数（发放倍数对外公示文案由运营在会员权益页配置，不是接口字段）。

### M3 积分明细展示加成笔

- 流水按后端 `business_type` 原值区分：`consumption_reward`（基础消费奖励）/ `level_bonus_reward`（会员等级加成）/ `activity_bonus_reward`（限时活动加成，2026-07-11 起生效——运营配置活动加成率后消费审核会多出此笔）；
- `level_bonus_reward` 笔显示"会员等级加成 +X"（京东PLUS 多返京豆同款）、`activity_bonus_reward` 笔显示"活动加成 +X"；不得要求后端改字段名。
- 注意：预算积分 `budget_points` 对 C 端完全不可见（DataSanitizer 全局黑名单），小程序不会也不应查到。

### M4 兑换商城门槛标注

- 商品列表 `GET /api/v4/exchange/items` 与详情 `GET /api/v4/exchange/items/:exchange_item_id` 新增字段 `level_requirement`（无门槛为 null）：
  - `{ min_level_name, max_level_name, satisfied }`（仅等级展示名 + 是否满足，**不下发门槛积分/倍数**）；
  - `satisfied=false` 时渲染专享角标 + 置灰兑换按钮（橱窗效应，未登录恒 false）；
- 兑换被拦时处理错误码：`REDEEM_GROWTH_LEVEL_INSUFFICIENT`（等级不足）/ `REDEEM_GROWTH_LEVEL_EXCEEDED`（超上限，如"银卡及以下专享"）。

### M5 升级通知

- 复用现有 `user_notifications` 拉取 + WebSocket `new_notification` 通道，零新协议；
- 升级通知 `type='growth_level_upgrade'`，`data` 含 `to_level_name / to_earn_multiplier`，文案后端已生成（"恭喜升级金卡，您的积分回馈已提升至 1.1 倍"）。
