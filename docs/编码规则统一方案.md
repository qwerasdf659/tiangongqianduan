### 25.2 问题诊断：不统一之处

> #### 25.2.0 与当前实现对账（2026-03-23）
>
> 本节（25.2～25.3 等）保留为 **迁移/统一方案落地前的问题档案**，便于理解演进背景。下列事项已在 **§25.1**、迁移 **`20260322194500-unify-encoding-order-nos-and-seq-columns.js`**、**`OrderNoGenerator` / `IdempotencyHelper`** 及管理端/API 联调中 **收口**，除非另注「仍有效」否则 **不再代表当前缺陷**：
>
> | 原小节 | 结论（2026-03-23） |
> |--------|-------------------|
> | 问题 1 `order_no` 前缀 | **已统一**：新写入为 16 位 `OrderNoGenerator`（EM/BD/TO/RD/LT/CS 等），见 §25.1.1；**存量**兑换非 16 位由 **`20260324143000`** 回填 |
> | 问题 2 随机源 | **部分已统一**：面向用户的 `order_no` 经 `OrderNoGenerator` 使用 `crypto`；`lottery_draw_id` 等内部 ID 与 `order_no(LT)` 分离，抽号路径见仓库 |
> | 问题 3 幂等键风格 | **仍多样**：属刻意保留（Stripe 式派生后缀等），非本次必须再改 |
> | 问题 4 `business_id` | **新写入已四段式**：`exchange_*`、`trade_*` 等见 §25.1.3；历史行仍为旧格式 |
> | 问题 5 模型层 `generateOrderNo` | **已清理完毕**（2026-03-23 验证：`ExchangeRecord.js` 中无 `generateOrderNo` 方法） |
>
> 阅读时请先对照 **§25.1**，再以本表判断条目是否仍具操作性。

#### 问题 1：order_no 前缀不一致（档案：落地前）

| 场景 | 前缀（档案期） | 生成方式（档案期） | **当前实现（2026-03-24）** |
|------|--------------|------------------|---------------------------|
| 兑换订单 | `EM` | 曾用服务层自写函数 | **`OrderNoGenerator.generate('EM', …)`**，`CoreService` 占位创建后回写 |
| 兑换订单（模型层弃用） | `EXC` | `ExchangeRecord.generateOrderNo()` | 死代码待清理；**不参与**新单 |
| 竞价订单 | 曾 `BID` | `BidService` 内联 | **`OrderNoGenerator.generate('BD', …)`**，前缀 **BD** |

**根因（档案）：** 模型层与服务层曾各写一套；竞价曾用 `BID` 与兑换 `EM` 不一致。**现已** 收口到 §25.1 + `OrderNoGenerator`；库内旧长号由 **`20260324143000`** 处理。

#### 问题 2：随机部分生成方式不统一（档案 + 当前）

| 位置 | 随机方式（档案期） | **当前（以仓库为准）** |
|------|------------------|------------------------|
| 兑换/竞价 `order_no` | 曾 `Math.random` | ✅ **`OrderNoGenerator`**：`crypto.randomBytes` |
| `IdempotencyHelper` | `crypto.randomBytes()` | ✅ 密码学安全 |
| `SettleStage._generateDrawId()` | 曾 `Math.random` | ✅ **`crypto.randomBytes`** |
| `RedemptionCodeGenerator` | `crypto.randomInt()` | ✅ 密码学安全 |
| `TrackingCodeGenerator` | 无随机（基于 item_id） | ✅ 确定性 |

#### 问题 3：幂等键前缀风格不统一

- 有些用下划线连接：`exchange_1774201431167_eaa37bcae175335e`
- 有些用冒号分隔层级：`quota_grant_batch:31:29569938:f7b9b6b3d42e22f7`
- 有些用冒号做派生后缀：`exchange_debit_xxx:counterpart`
- trade_orders 的幂等键前缀在不同调用方各不相同：`first_order_`、`order_`、`withdraw_test_order_`、`market_purchase_`

#### 问题 4：business_id 命名和分隔符不统一

- `exchange_31_p7_s7_1774201431234`（混合 `_` 和 `p`/`s` 前缀）
- `trade_order_32_11114_1774135815938`（纯 `_` 分隔）
- `lottery_draw_31_lottery_tx_...`（嵌套了 session_id，过长）
- `ad_campaigns` 用 UUID（与其他模块完全不同风格）

#### 问题 5：模型层 vs 服务层双重定义

**档案：** 模型层与服务层曾各有一套。**当前：** 新单仅经 **`OrderNoGenerator` + `CoreService`/`BidService` 回写**；模型层若仍残留 `generateOrderNo` 属待清理死代码。

### 25.4 统一方案（基于现有技术栈）

#### 25.4.1 要不要统一？

**结论：部分统一 + 规范化，不做全面重构。**

理由：
- 项目还未上线，可以一次性修正，但改动范围要可控
- `tracking_code`（`TrackingCodeGenerator`）设计成熟，参考了美团/Steam，**不改**
- `redemption_code`（`RedemptionCodeGenerator`）设计成熟，密码学安全，**不改**
- 幂等键的 `:` 派生后缀（`:counterpart`、`:in`/`:out`、`:reward_1`）是成熟的业界模式，**不改**
- 需要统一的：order_no 前缀规范、随机数生成方式、business_id 格式规范、清理死代码

#### 25.4.2 需要你拍板的决策点

| # | 决策项 | 选项 A | 选项 B | 建议 | 影响范围 |
|---|--------|--------|--------|------|---------|

### 25.9 行业方案对比与选型分析（6 项决策的行业依据）

#### 25.9.1 大厂方案对比

##### 美团（Leaf 分布式 ID 系统）

美团开源的 Leaf 系统是业界标杆，支持两种模式：

**号段模式（Leaf-Segment）：**
- 从 DB 批量预加载号段（如 1~1000），内存中分配，双 Buffer 异步预加载
- DB 表结构：`biz_tag`（业务标签）、`max_id`（当前最大）、`step`（步长）
- 优势：DB 短暂不可用时内存号段仍可用，10万+ QPS
- 适用：订单系统、分库分表场景

**雪花算法模式（Leaf-Snowflake）：**
- 64 位 = 41 位时间戳 + 10 位机器 ID + 12 位序列号
- 理论 QPS：409 万/秒，通过 ZK 分配机器 ID
- 适用：秒杀、实时日志

**与本项目的关系：** Leaf 是为美团外卖/金融等百万 QPS 场景设计的。本项目是**单体 Node.js + 单 MySQL**，峰值并发远低于需要号段/雪花的阈值。**引入 Leaf 属于过度工程**。

##### 阿里巴巴/淘宝

淘宝订单号采用**基因法分段编码**：

`[时间基因6位] + [业务类型基因2位] + [随机序列6位] + [分片路由基因6位]`

核心设计：同一用户的订单后 6 位固定（用户 ID 哈希），用于**分库分表路由**——同一用户的订单被路由到相同分片，避免跨库查询。

**与本项目的关系：** 淘宝的基因法是为**几十亿订单 + 几千个分库分表**设计的。本项目是单库单表，**不需要分片路由基因**。

##### 腾讯游戏

腾讯游戏虚拟物品交易的 ID 设计：
- 类 Snowflake 64 位整数：大区号 + 虚拟机器号 + 功能号 + 校验序号 + 序列号 + 时间戳
- 幂等键：**预置令牌模式**——客户端先申请 UUID token，交易请求携带该 token
- 跨服交易：分布式锁 + 消息队列异步同步

**与本项目的关系：** 腾讯的方案是为多区多服、百万同时在线设计的。本项目单服架构，**不需要大区号/机器号**。但腾讯的"预置令牌"思路与项目的 `IdempotencyHelper` 请求幂等键方案一致。

#### 25.9.2 C2C 交易平台 / 小众二手平台

##### Steam 物品系统

Steam 的三层标识体系：
- `assetid`：物品实例唯一 ID（等价于项目的 `item_id`）
- `classid`：物品类型 ID（等价于项目的 `item_template_id`）
- `instanceid`：变体标识（等价于项目的 `sku_id`）

交易 ID 由平台自增分配，不嵌入业务信息。追踪码是**人类可读的短码**。

**与本项目的关系：** 项目的 `TrackingCodeGenerator`（`LT260322039810`）正是参考了 Steam 的人类可读追踪码设计。**已经对齐了。**

##### 闲鱼/转转/得物

这些平台的订单号都是**纯数字长串**（18~22 位），服务端生成，前端纯展示。不嵌入业务语义。幂等性由后端网关层控制。

**与本项目的关系：** 这些平台是大流量场景，用数字 ID 是为了分库分表效率。本项目的带业务前缀方案（`EM...`/`BID...`）在可读性上反而更优。

#### 25.9.3 小型创业公司 / 活动营销平台

业界共识：**不到万级 QPS，不需要雪花算法或独立 ID 服务。**

典型方案：
- `时间戳 + 随机数`（最简单，大多数创业公司的选择）
- `前缀 + 时间戳 + 随机数`（加了业务区分，可读性好）
- `UUID v4`（纯随机，不可读但绝对唯一）

**与本项目的关系：** 本项目的方案（`前缀 + 时间戳 + 随机数`）正好处于**创业公司最佳实践的甜蜜点**——比纯 UUID 可读，比雪花算法简单，碰撞概率足够低。

#### 25.9.4 幂等键行业标准

##### Stripe（全球支付标杆）
- 幂等键由**客户端生成 UUID v4**，通过 `Idempotency-Key` HTTP Header 传入
- 服务端缓存 24 小时，重复请求返回缓存响应
- 键长度限制 255 字符

##### Shopify
- 同样使用 `Idempotency-Key` Header
- 仅对 POST 请求生效（GET/DELETE 天然幂等）
- 建议用 UUID v4

##### PayPal
- 使用 `PayPal-Request-Id` Header（自定义名称，但本质相同）

##### 本项目的幂等键对比

| 维度 | Stripe/Shopify | 本项目 |
|------|---------------|-------|
| 入口幂等键 | 客户端生成 UUID | 服务端生成 `req_{ts}_{hex}_{seq}` |
| 派生幂等键 | 不支持（单层） | 支持 `{入口key}:{事务类型}`（多层） |
| 分隔符 | 无（纯 UUID） | 混用 `_` 和 `:` |
| 存储 | Redis/DB，24h 过期 | DB `api_idempotency_requests` 表 |

**关键发现：** 大厂的幂等键都是**单层 UUID**，不包含业务语义。本项目的设计**更先进**——入口幂等 + 派生幂等是分布式事务的业界标准方案（Stripe 不需要是因为它每个请求只产生一条记录）。本项目一次抽奖产生多条流水（扣费 + 发奖 + 审计），**必须有派生层**。

`:` 作为派生分隔符（`req_xxx:reward_1`）是 Redis 键命名的业界惯例，在本项目中用于幂等键派生是合理的。

### 25.10 基于行业分析的决策定性（替代 25.4.2 的"待拍板"）

根据上述行业分析，6 项决策已全部拍板：

#### D1：order_no 前缀规范 → ✅ **已拍板 B（各业务保留前缀，格式统一）**

**行业依据：**
- 阿里的"业务类型基因"就是在订单号中嵌入业务码
- Steam 的物品 ID 不嵌入业务信息（但它们有独立的 `assetid`/`classid` 体系）
- 创业公司最佳实践：`前缀 + 时间戳 + 随机数` 是最实用的方案

**具体方案：** 保留 `EM`（兑换）、`BID`（竞价）各自的前缀，但统一随机部分为 6 位 crypto hex 大写。不需要引入雪花算法。

**为什么不用雪花算法：** 本项目是单体 Node.js + 单 MySQL，没有分库分表需求。雪花算法的机器 ID 分配、时钟回拨处理反而增加复杂度。`时间戳 + crypto随机数`在本项目并发量下碰撞概率为零（13位毫秒时间戳 + 6位hex = 2^24 种组合/毫秒）。

#### D2：随机数统一用 `crypto.randomBytes()` → ✅ **已拍板 B（统一）**

**行业依据：**
- Stripe/PayPal 推荐 UUID v4（底层是 `crypto` 级随机）
- 腾讯游戏明确要求 ID 生成使用密码学安全随机
- `Math.random()` 在 V8 引擎中使用 xorshift128+ 算法，**理论上可预测**

**风险量化：** `Math.random()` 的碰撞风险在本项目量级下极低，但改为 `crypto.randomBytes()` 的成本也极低（改 3 行代码）。**收益/成本比极高。**

#### D3：删除死代码 → ✅ **已拍板 B（直接删除）**

**行业依据：** 无需行业对比，这是基本工程规范。死代码是维护负担。

#### D4：business_id 统一为 4 段式 → ✅ **已拍板 B（统一）**

**行业依据：**
- 阿里的订单号分段设计（时间+业务+随机+路由）证明分段式有利于检索
- 腾讯游戏的 ID 也是分段式（大区+机器+功能+序列）
- 本项目当前的 `lottery_draw_31_lottery_tx_1774196338159_aede83_413_draw_mn1yow8b_31_v41atd` 长达 70+ 字符，嵌套了完整的 session_id，违反了"business_id 应该简洁可检索"的原则

**具体方案：** 4 段式 `{domain}_{subject_id}_{object_id}_{timestamp}`，最长约 40 字符，满足 VARCHAR(150) 字段限制，适合 `LIKE 'exchange_%'` 模糊查询。

#### D5：trade_orders 幂等键收口 → ✅ **已拍板 B（收口到 IdempotencyHelper）**

**行业依据：**
- Stripe 的幂等键由客户端统一生成，服务端不生成也不修改
- 本项目是服务端生成幂等键，但散落在多个调用方（`first_order_`、`order_`、`market_purchase_` 等前缀各不相同）
- **单一职责原则：** 幂等键生成应该有一个权威入口

#### D6：ad_campaigns 的 business_id → ✅ **已拍板 A（保持 UUID）**

**行业依据：**
- 广告系统在行业中通常是独立模块，有自己的 ID 体系
- UUID 在广告系统中是标准做法（Google Ads、Facebook Ads 的内部 campaign ID 都是类 UUID）
- 广告模块数据库暂无数据，改不改都没有迁移成本，但保持 UUID 可以与第三方广告平台对接时更兼容

| 业务 | 表名 | 面向用户的订单号 | **D7 前**状态（档案） |
|------|------|---------------|---------|
| 兑换 | `exchange_records` | `order_no`（曾见旧长号如 `EM177420…`） | ✅ 有字段；**现规范 16 位 EM**，存量见 `20260324143000` |
| 竞价 | `exchange_records`（同表） | `order_no`（曾用 `BID…`） | ✅ 有字段；**现规范 16 位 BD** |
| C2C 交易 | `trade_orders` | **当时无** `order_no` | ❌ → ✅ **已补 TO** |
| 核销 | `redemption_orders` | **当时无**（PK 是 UUID） | ❌ → ✅ **已补 RD** |
| 抽奖 | `lottery_draws` | **当时无**（PK 是内部字符串） | ❌ → ✅ **已补 LT** |

**真实场景痛点（D7 前；迁移后已缓解）：**
- 客服场景：用户说"我交易出了问题"，客服问"订单号多少？"——trade_orders 没有人类可读的订单号
- 核销场景：门店说"这个核销单号是什么？"——redemption_orders 只有 UUID，不可读
- 抽奖场景：用户投诉"我抽奖没到账"——lottery_draws 的 ID 是 `draw_mn1yow8b_31_v41atd`，不适合给用户看

#### 25.13.2 统一订单号方案

**给所有"订单/记录类"表统一加 `order_no` 字段，格式统一为：`{2位业务码}{YYMMDD}{6位序列}{2位随机}`**

| 业务 | 前缀 | 示例 | 说明 |
|------|------|------|------|
| 兑换 | `EM` | `EM260323039810A3` | 保持现有前缀 |
| 竞价 | `BD` | `BD260323000001K7` | 保持现有前缀 |
| C2C 交易 | `TO` | `TO260323007087F2` | **新增** |
| 核销 | `RD` | `RD260323000001P8` | **新增** |
| 抽奖 | `LT` | `LT260323039677X4` | **新增**（复用 tracking_code 的 LT 前缀） |

**格式拆解：**
```
EM 260323 039810 A3
│  │      │      └─ 2位 crypto 随机（防猜测）
│  │      └──────── 6位序列号（基于记录 ID 补零）
│  └─────────────── YYMMDD 日期（北京时间）
└────────────────── 2位业务码（人眼可识别业务类型）
```

**总长度：12 位**，人类可读、可报给客服、可打印在小票上。

#### 25.13.3 这个方案与现有体系的关系

```
          ┌──────────────── order_no（统一订单号，面向用户/客服）
          │
用户抽奖 ──→ lottery_draws     → LT260323039677X4
          │                         ↓
          │                    items.tracking_code = LT260323039677（物品追踪码）
          │
用户兑换 ──→ exchange_records  → EM260323039810A3
          │                         ↓
          │                    items.tracking_code = EX260323039810（物品追踪码）
          │
用户交易 ──→ trade_orders      → TO260323007087F2
          │                         ↓
          │                    items.tracking_code 不变（物品转移，追踪码跟着走）
          │
用户核销 ──→ redemption_orders → RD260323000001P8
                                    ↓
                               items.tracking_code 不变（核销不改变物品编码）
```

**两条线各司其职：**
- `order_no`：**订单/操作** 的编号，每次操作产生一个新号，面向用户和客服
- `tracking_code`：**物品** 的编号，终身不变，面向用户和客服

**不冲突，互补。** 就像快递场景：包裹有运单号（tracking），每次下单有订单号（order_no），两个都要。

#### 25.13.4 为什么不能只用一个编码？

| 方案 | 问题 |
|------|------|
| 只用 `tracking_code` | 一个物品可能经历多次交易，每次交易需要独立编号；核销码不是物品码 |
| 只用 `order_no` | 物品在多次交易中需要一个不变的身份标识，order_no 每次操作都不同 |
| 只用 UUID | 不可读，客服无法使用 |
| 只用自增 ID | 无业务含义，无法从编号判断业务类型和时间 |

**真实业务场景：**

物品 39810（收藏宝石）的完整链路可能是：
```
1. 用户A兑换  → 兑换订单 EM260323039810A3  → 铸造物品 tracking_code=EX260323039810
2. 7天后上架  → 市场挂牌（无需订单号）
3. 用户B购买  → 交易订单 TO260325007100K2  → 物品 tracking_code 不变，owner 从A变B
4. 用户B核销  → 核销订单 RD260330000050P8  → 物品 tracking_code 不变，状态变 used
```

客服可以通过任何一个号查到完整链路。

|----|------|------|
| `trade_orders` | **新增** `order_no VARCHAR(20) UNIQUE` | 交易订单号 |
| `redemption_orders` | **新增** `order_no VARCHAR(20) UNIQUE` | 核销订单号 |
| `lottery_draws` | **新增** `order_no VARCHAR(20) UNIQUE` | 抽奖订单号 |

### 25.15 为什么不能只用一种编码？（行业论证 + 本项目真实数据证明）

#### 25.15.1 先做一个思想实验

假设我们强行只用一种编码 `TG{YYMMDD}{6位序列}{2位随机}`，所有业务共用。

用你数据库里的真实场景来推演——物品 39810（收藏宝石）的生命周期：

```
Step 1: 用户A兑换宝石
  → 生成编码 TG260323039810A3
  → 问题：这个编码同时代表"兑换这个操作"和"宝石这个东西"吗？

Step 2: 7天后用户A把宝石上架市场，用户B购买
  → 需要生成一个新编码吗？
  → 如果生成新编码 TG260330007100K2，那宝石的编码从 TG...A3 变成了 TG...K2？
     物品换了一个编码？之前的编码作废了？客服用旧编码查不到了？
  → 如果不生成新编码，那这次"交易操作"没有编号？
     用户B投诉"交易有问题"，客服问"编号多少"，答不上来？

Step 3: 用户B到店核销
  → 又需要一个编码。但店员需要扫的码必须是短的、随机的、不能被猜到的。
  → TG260330007100K2 这种格式包含日期和序列号，顾客可以猜到其他人的编码。
  → 安全问题。
```

**结论：一种编码无法同时解决"物品身份"、"操作编号"、"安全凭证"三个不同性质的问题。**

这不是你项目的特殊问题，是所有业务系统的共性。

#### 25.15.2 全行业无一例外：至少 2~3 种编码

| 公司/平台 | 物品/商品标识 | 订单/操作标识 | 安全凭证 | 总数 |
|----------|------------|------------|---------|------|
| **淘宝** | 商品 ID（itemId） | 订单号（20位数字） | 退货码、取件码 | 3种 |
| **美团** | 商品 ID | 订单号（Leaf生成） | 核销码（短数字） | 3种 |
| **京东** | SKU ID | 订单号 | 取件码、退货码 | 3种 |
| **Steam** | assetid（物品） | trade offer ID（交易） | trade token（安全） | 3种 |
| **腾讯游戏** | 道具 ID | 交易流水号 | 兑换码/礼品码 | 3种 |
| **闲鱼** | 商品 ID | 订单号 | 取件码 | 3种 |
| **得物** | 商品编号 | 订单号 | 取件码/鉴定码 | 3种 |
| **Stripe** | product ID | payment intent `pi_xxx` | — | 2种 |
| **小公司活动平台** | 奖品 ID | 抽奖记录号 | 兑奖码 | 3种 |

**没有任何一家公司只用一种编码。原因是它们解决的是三个本质不同的问题：**

| 问题 | 性质 | 要求 |
|------|------|------|
| 这个**东西**是谁？ | 身份标识（终身不变） | 稳定、可追踪、人类可读 |
| 这次**操作**是什么？ | 事件标识（每次新建） | 唯一、可排序、含业务类型 |
| 这个**凭证**合法吗？ | 安全令牌（一次性使用） | 随机、不可猜测、短小可输入 |

#### 25.15.3 为什么不能合并？逐一论证

**尝试 1：把 order_no 和 tracking_code 合并**

你的数据库里，`items` 表有 **7194** 个物品，`trade_orders` 有 **154** 笔交易，`redemption_orders` 有 **1360** 个核销。

一个物品可以经历：创建 → 交易1 → 交易2 → 核销，即 **1 个物品对应 N 次操作**。

如果合并成一个编码：
- 物品创建时给编码 `X001`
- 交易时需要新编码（因为是新操作），给 `X002`
- 那物品的编码到底是 `X001` 还是 `X002`？
- 如果物品编码跟着变，之前的记录就断链了
- 如果物品编码不变，交易操作就没有独立编号

**这是 1:N 关系，逻辑上不可能用一个编码表示。**

**尝试 2：把 redemption_code 和 order_no 合并**

核销码的核心要求是**不可猜测**——门店扫码验证，如果编码包含日期+序列号，顾客可以伪造。

`EM260323039810A3` 这种格式，看到一个就能猜出 `EM260323039811A4` 可能存在。

而 `A3K7-NP2M-X9QR`（Base32 随机）完全无规律，无法从一个码推断另一个码。

**安全凭证和业务编号的设计目标相反：业务编号要可读可排序，安全凭证要不可预测。**

**尝试 3：全部用 UUID**

`d30ecaae-aa44-499b-9018-a608786dff3e` — 36 个字符。

- 客服电话里念给用户听？不现实
- 用户截图投诉时报编号？记不住
- 打印在小票上？太长
- 无法判断业务类型和时间

UUID 适合内部系统间传递，不适合面向用户。

#### 25.15.4 最适合本项目的方案

基于你的项目实际情况（单体 Node.js + 单 MySQL + 未上线 + 愿意一次性投入）：

**不需要学大厂**（雪花算法、Leaf、基因分段 — 那是为百万 QPS + 分库分表设计的）

**不需要学小公司省事做法**（全用 UUID 或纯自增 ID — 可读性差、客服不友好）

**最适合的是：3 种编码 + 统一格式规范 + 统一生成入口，这就是当前方案。**

| 维度 | 大厂做法 | 小公司做法 | **本项目方案** |
|------|---------|-----------|-------------|
| 编码种类 | 3~5 种（更多细分） | 1~2 种（省事） | **3 种（刚好够用）** |
| 格式统一性 | 统一长度/分段 | 不统一 | **统一为 16 位定长** |
| 生成方式 | 独立 ID 服务 | 散落各处 | **3 个工具类收口** |
| 可读性 | 纯数字（不可读） | UUID（不可读） | **前缀+日期+序列（可读）** |
| 安全性 | crypto 级 | Math.random | **crypto 级** |
| 复杂度 | 高（ZK/Redis 依赖） | 低（但有隐患） | **低（零外部依赖）** |

#### 25.15.5 最终结论

**3 种编码是数学上的最小必要数量，不是过度设计。**

减少到 2 种 → 要么物品没有稳定身份，要么操作没有独立编号
减少到 1 种 → 1:N 关系无法表达，安全凭证无法保证

增加到 4 种以上 → 过度设计，本项目不需要

**3 种编码、统一格式、统一生成入口——这就是方案的全部。**

#### 25.18.4 新发现的三个实施风险

##### 风险 1（P0）：OrderNoGenerator 对非整数主键表的序列号来源

**影响范围：** `lottery_draws`（PK: STRING）和 `redemption_orders`（PK: UUID）

**问题描述：** 文档设计 `OrderNoGenerator.generate(bizCode, recordId, createdAt)` 依赖 `recordId`（整数）作为 6 位序列号。但 `lottery_draws` 的主键是字符串 `draw_mn1yow8b_31_v41atd`，`redemption_orders` 的主键是 UUID `d30ecaae-aa44-499b-...`，都不是 auto_increment 整数，无法直接补零。

**对比其他 5 张表：** `exchange_records`、`trade_orders`、`consumption_records`、`asset_transactions`、`ad_billing_records` 的主键均为 BIGINT AUTO_INCREMENT，可以直接用。

**决策 D11：**

| # | 选项 | 方案 | 优点 | 缺点 |
|---|------|------|------|------|
| **D11** | **A** | 给 `lottery_draws` 加 `draw_seq INT AUTO_INCREMENT UNIQUE`，给 `redemption_orders` 加 `redemption_seq INT AUTO_INCREMENT UNIQUE`，用这个辅助整数列作为序列号来源 | 与其他 5 张表完全一致；序列号有明确含义（第 N 条记录）；6 位可容纳 999,999 条记录 | 需 1 个 migration 加 2 个列；已有数据会自动填充 1~1399 和 1~1360 |
| | B | 序列号部分改用 `timestamp_ms % 1000000` 补零 + 2 位随机 | 不需要改表结构 | 格式含义与其他表不一致（其他表是 recordId，这两张是时间戳截断）；极小概率重复 |
| | C | 使用关联的 `item_id` 作为序列号 | 不需要改表 | `redemption_orders` 同一 `item_id` 可能有多条记录（过期重建），不唯一 |

✅ **D11 已拍板 A：** 给 `lottery_draws` 和 `redemption_orders` 各加一个 `AUTO_INCREMENT` 辅助整数列，全部 7 张表的 `order_no` 格式和生成方式完全统一。已有 1399 + 1360 条数据会自动获得连续序号。

##### 风险 2（P1）：Web 管理后台前端客户端幂等键生成

**影响范围：** `admin/src/modules/asset/composables/adjustment.js` 等前端文件

**问题描述：** Web 管理后台在资产调整等操作中，前端使用 `Date.now()` 生成客户端侧幂等键（如 `admin_adjustment_{Date.now()}`），然后传给后端。当前后端 `IdempotencyHelper` 并不强制校验幂等键格式，所以不会冲突。

**但需注意：** 如果未来后端对幂等键格式做更严格的校验或解析（如根据前缀判断业务类型），前端生成的幂等键格式需要同步。

**建议：** 本次编码规则统一方案中的幂等键改动（D4、D5）仅影响**后端服务层自动生成**的幂等键，不影响前端传入的。前端当前方式可以保持不变，但建议在后端 `IdempotencyHelper` 中增加一个 `generateAdminOperationKey(operatorId, operationType)` 函数，供前端调用的 API 在后端侧统一生成，逐步收口。

**结论：不阻塞本方案执行。Web 前端当前无需改动。**

##### 风险 3（P2）：IdempotencyHelper 中 Math.random 残留

**影响范围：** `utils/IdempotencyHelper.js` 的 `generateLotterySessionId()` 和 `generateRequestIdempotencyKey()` 中 3 位序列号

**问题描述：** 这两个函数的主体随机部分已使用 `crypto.randomBytes()`，但 3 位 sequence（`Math.floor(Math.random() * 1000)`）仍用 `Math.random()`。Phase 1 的 crypto 迁移计划只覆盖了 `CoreService`、`SettleStage`、`BidService`，未覆盖这两处。

**风险评估：** 3 位 sequence 仅用于同一毫秒内区分请求顺序，`Math.random()` 在此场景下碰撞概率极低（同毫秒内 1/1000），不是安全性问题。

**建议：** 在 Phase 1 中顺带改掉，成本极低（改 2 行），但不阻塞。优先级低于 D11。

#### 25.29.3 决策记录（已拍板）

##### 决策 D12：抽奖概率引擎 `Math.random()` → `crypto` 是否纳入本方案？ ✅ 已拍板

**结论：A — 不纳入本方案。**

概率抽样（PrizePickStage / TierPickStage / LoadDecisionSourceStage / StrategySimulationService / AttributeRuleEngine 共 16 处）与编码规则统一是不同维度。`Math.random()` 用于概率分布采样不需要密码学安全性，`crypto` 在高频抽奖场景慢 3~5 倍。`StrategySimulationService` 中 6 处是模拟计算而非真实抽奖。单独立项评估性能影响。

##### 决策 D13：内部辅助 ID 的 `Math.random()` 是否替换？ ✅ 已拍板

**结论：A — 替换为 `crypto.randomBytes`。**

10 处改动（services 5 处 + models 5 处），成本极低，消除所有非概率场景的 `Math.random()` 残留，与 `OrderNoGenerator` / `IdempotencyHelper` 的 `crypto` 路线一致。执行见 Phase 7-2。

##### 决策 D14：Web 前端幂等键 `Date.now()` 收口方式？ ✅ 已拍板

**结论：A — 后端 API 层自动生成幂等键。**

在资产调整 API 路由层（`routes/v4/console/assets/`），如果客户端未传 `idempotency_key`，由服务端调用 `IdempotencyHelper.generateStandaloneIdempotencyKey()` 生成。前端删除 3 处 `idempotency_key` 字段的手动拼接。符合"前端不做编码生成"原则。执行见 Phase 7-8。

##### 决策 D15：trade_orders 历史幂等键前缀是否回填统一？ ✅ 已拍板

**结论：回填。** 154 条存量是测试数据，一次性回填统一前缀。文档更新明确 Stripe 客户端幂等键模式为最终方案（`generateTradeIdempotencyKey()` 供测试/脚本使用，生产流量使用客户端传入）。执行见 Phase 7-9。

##### Phase 7-9：trade_orders 幂等键存量回填（D15 已拍板回填）

**层级：** 后端项目（一次性数据迁移脚本）
**背景：** 154 条 `trade_orders` 的 `idempotency_key` 有 10+ 种前缀（`order_`/`stress_`/`withdraw_`/`first_`/`shared_`/`single_` 等），均为测试数据。
**最终方案：** Stripe 客户端幂等键模式为生产最终方案；`generateTradeIdempotencyKey()` 供测试/脚本使用。存量一次性回填为统一格式。

```
方式：Sequelize 迁移脚本（或一次性 Node.js 脚本）
逻辑：
  1. 查询所有 trade_orders 记录
  2. 对每条记录生成统一格式幂等键：
     trade_{order_no}_{原 idempotency_key 的 hash 后 8 位}
     （保留原 key 的 hash 确保回填后仍然唯一）
  3. 批量 UPDATE trade_orders SET idempotency_key = 新值 WHERE trade_order_id = ?

可复用：
  - IdempotencyHelper.generateTradeIdempotencyKey() 可作为参考格式
  - crypto.createHash('sha256') 对原 key 取 hash

注意事项：
  - idempotency_key 列有 UNIQUE 约束，回填时需确保新值不冲突
  - 建议先在事务中执行，验证无冲突后 COMMIT
  - 回填完成后在 SequelizeMeta 记录迁移（如用迁移文件方式）

文档更新：
  - Phase 3 步骤 3-3 描述更新为：
    "TradeOrderService 采用 Stripe 客户端幂等键模式（客户端通过 Idempotency-Key
     HTTP Header 传入），服务端 generateTradeIdempotencyKey() 供测试/脚本使用。
     存量 154 条测试数据已通过 Phase 7-9 回填统一前缀。"
```
