# 微信小程序前端图片位运营可配置化 — 后端对接需求

> 提出方：微信小程序前端项目
> 接收方：后端数据库项目
> 基准：以后端为唯一权威，前端零映射直读后端字段（snake_case）
> 核查方式：全量检索微信小程序前端实际代码（`pages/**`、`packageXxx/**`、`utils/api/system.ts`、`typings/api.d.ts`、`app.json`），排除组件库 `miniprogram_npm`
> 日期：2026-06-21（北京时间）

---

## 一、需求背景

抽奖页（`pages/lottery`）顶部那张沉浸式大背景图，目前是**前端写死的本地图片**，运营无法在 Web 管理端替换；每次换图都要改前端代码、替换小程序包内图片、重新发版上线。

期望：让运营能像配置轮播图一样，在 Web 管理端**随时替换/调整顶部 Banner**，无需前端发版。经全量排查，**顶部 Banner 并非抽奖页独有**——5 个主 Tab 页都有同一形态、且都未接后端，应统一设计为运营可配。

本文档只聚焦以下图片位：

- **A 类（需新接后端）**：各页面头部 Banner 图。
- **B 类（已接后端，现状说明）**：抽奖页轮播图、弹窗横幅、公告条（如带图）、广告素材图。

---

## 二、A 类 — 各页面头部 Banner（需运营可配，本需求重点）

前端 5 个主 Tab 页（对应 tabBar：回馈/发现/商城/DIY/我的）顶部都有「沉浸式顶部横幅」，统一变量名 `topBannerUrl`，但**该变量在所有页面的 `.ts` 中从未被赋值**（无任何接口下发），现状：

| Tab 文案 | 页面 | 当前图源（前端实测） | 现状 |
| --- | --- | --- | --- |
| 回馈 | `pages/lottery`（抽奖页） | `topBannerUrl \|\| '/images/抽奖页面图-banner.jpg'` | 写死本地图，运营改不了 |
| 我的 | `pages/user` | `topBannerUrl \|\| '/images/我的页面-banner.jpg'` | 写死本地图，运营改不了 |
| 发现 | `pages/diy`（DIY 选择页） | `topBannerUrl \|\| ''`，无图时显示占位框「顶部横幅图片位置」 | 未接后端，长期空着/占位 |
| DIY | `pages/camera` | `topBannerUrl \|\| ''`，同上占位 | 未接后端，长期空着/占位 |
| 商城 | `pages/exchange` | （无顶部 Banner 区域） | 如需统一，需新增 |

### 代码证据（以抽奖页为例）

```10:14:pages/lottery/lottery.wxml
      <image
        class="top-banner-image"
        src="{{topBannerUrl || '/images/抽奖页面图-banner.jpg'}}"
        mode="aspectFill"
      />
```

- 图源为 `topBannerUrl || '/images/抽奖页面图-banner.jpg'`。
- 经全文检索 `pages/lottery/lottery.ts`，**`topBannerUrl` 变量从未被定义或赋值**，也无任何接口为其下发值。
- 结果：顶部 Banner 永远回退到 `||` 后的**前端包内本地图**，运营改不了。`pages/user` 同理（写死 `我的页面-banner.jpg`）；`pages/diy`/`pages/camera` 已预留 `topBannerUrl` 占位但从未接后端。

> 五个页面应统一为同一套「顶部 Banner」运营配置，按 `position` 区分页面。

## 四、需求方案（首选：复用现有投放系统，零新架构）

> 原则：不为顶部 Banner 另起一套接口/表，直接复用 `GET /api/v4/system/ad-delivery` 投放系统，新增一个槽位类型即可。前端改动最小、长期一致。

后端已有成熟的 `ad-delivery` 投放系统（现支持 popup / carousel / announcement / feed）：

```52:53:utils/api/system.ts
 * @param params.slot_type - 广告位类型（必填）: popup / carousel / announcement / feed
 */
async function getAdDelivery(params: { slot_type: string; position?: string }) {
```

### 4.1 后端需提供的能力

**在现有 `ad-delivery` 投放系统中新增槽位类型 `slot_type = top_banner`**：

| 项 | 要求 |
| --- | --- |
| 接口 | 复用现有 `GET /api/v4/system/ad-delivery?slot_type=top_banner&position={页面}` |
| position 维度 | 区分 5 个 Tab 页：`lottery` / `user` / `diy` / `camera` /（如需）`exchange` |
| 管理端配置 | 运营可在 Web 管理端为各 position 配置/替换图片、生效时间、跳转（与轮播图同一套配置流程） |
| 响应结构 | 复用现有 `{ success, data: { items: AdDeliveryItem[], slot_type, position, total } }`，**无需为本需求新增任何字段** |
| 数量约定 | 顶部 Banner 通常单张：按 priority 降序取**生效中的第 1 条**；前端只用 `items[0]` |
| 无配置时 | `items` 返回空数组（`[]`）。前端据此回退本地兜底图，不报错、不留白 |

### 4.2 前端需要的字段（均为现有 `AdDeliveryItem` 已有字段，后端无需新增）

| 字段 | 来源（typings/api.d.ts 现有定义） | 前端用途 |
| --- | --- | --- |
| `primary_media.public_url` | `primary_media: MediaObject \| null` | 顶部 Banner 图片地址（后端 JOIN media_files 生成完整 URL） |
| `ad_campaign_id` | `ad_campaign_id: number` | 曝光/点击日志上报用（复用现有 `reportAdImpression`/`reportAdClick`） |
| `ad_slot_id` | `ad_slot_id: number \| null` | commercial 类型日志上报依赖（运营内容可为 null） |
| `campaign_category` | `campaign_category: string` | `commercial` 时前端按《广告法》标"广告"标识 |
| `link_url` / `link_type` | `link_url: string\|null` / `link_type: string` | 点击 Banner 跳转（none/page/miniprogram/webview，复用现有跳转逻辑） |

> 以上字段在现有 `AdDeliveryItem`（typings/api.d.ts 第 1363-1430 行）中**全部已存在**，本需求不要求后端新增任何字段，仅需新增 `top_banner` 槽位类型的可配置与下发。

### 4.3 前端将做的改造（后端能力就绪后实施）

1. 各 Tab 页 `loadTopBanner()`：调 `GET /api/v4/system/ad-delivery?slot_type=top_banner&position={本页}`，取 `items[0].primary_media.public_url` 赋给 `topBannerUrl`。
2. 接口空数组 / 无图 / 图片加载失败时，回退本地兜底图（保证永不留白）。
3. 若 `campaign_category === 'commercial'`，按现有轮播图同款逻辑加"广告"标识与曝光/点击日志上报。

> 前端不依赖任何"已移除/敏感"字段，不自行计算，图片地址完全以后端 `public_url` 为准。

## 五、需要后端确认/拍板的点

1. **是否采用本方案**（复用 `ad-delivery` 新增 `slot_type=top_banner`，用 `position` 区分页面）。如后端有其它既定方案（如单独的"页面装修"配置接口），请告知接口路径与字段，前端照此对接。
2. **覆盖范围**：顶部 Banner 是否 5 个 Tab 页全做，还是先做 `lottery`/`user` 两个已写死的？（`diy`/`camera` 已预留占位，`exchange` 当前无该区域。）
3. **单张还是多张**：顶部 Banner 是否允许多张轮播？
   - 若**单张**：后端取 priority 最高的生效项，前端用 `items[0]`。
   - 若**多张轮播**：前端可复用 carousel 的 swiper 逻辑，但需后端明确该槽位也下发 `slide_interval_ms`。
   - 前端建议：顶部大图保持**单张**，视觉更稳定；轮播交给已有 carousel 区。
4. **跳转是否需要**：顶部 Banner 是否允许点击跳转（`link_url`/`link_type`），还是纯展示不可点。
5. **尺寸/比例规范**：顶部 Banner 在前端是 `aspectFill` 沉浸式铺满顶部，请运营出图按统一比例（建议后端在管理端给运营标注推荐尺寸，避免变形）。

> 核查方：后端数据库项目
> 核查方式：Node.js 直连真实库 `restaurant_points_dev`（`dbconn.sealosbja.site`，非备份）+ 直读当前实际代码（`routes/v4/system/ad-delivery.js`、`models/AdSlot.js`、`routes/v4/console/ad/ad-slots.js`、`services/AdBiddingService.js`、`services/AdCampaignAdminService.js`、`admin/src/modules/content/pages/ad-management.js` 等），未引用任何历史报告/备份。

### 8.1 真实环境核查证据（直连数据库 + 实际代码）

| 核查点 | 真实结果 | 说明 |
| --- | --- | --- |
| `ad_slots` 真实槽位 | 共 **7** 个：`home_popup/lottery_popup/profile_popup/home_carousel/home_announcement/market_list_feed/exchange_list_feed` | **没有任何 `top_banner` 槽位**；轮播只有 `home_carousel`，**无 `lottery_carousel`** |
| `slot_type` 校验点（3 处且不一致） | ① `models/AdSlot.js` = `['popup','carousel','announcement','feed']`；② `routes/v4/system/ad-delivery.js` = 同上；③ `routes/v4/console/ad/ad-slots.js`（运营建槽位）= **仅 `['popup','carousel']`** | 运营建槽位接口比模型更严，**目前连 announcement/feed 都建不了，更别说 top_banner** |
| 管理端槽位类型下拉 | `admin/ad-management.html` 第 1016-1019 行 = 弹窗/轮播图/系统公告（**无 feed、无 top_banner**） | 前端下拉也要加项 |

### 8.2 两处必须以后端为准的校正（前端适配后端，不做映射）

> 按"以后端数据库为唯一权威、前端改字段适配后端"原则，前端文档以下两点需改：

1. **响应字段名校正**：后端 `ad-delivery` 返回的是**扁平字段 `image_url`**，不是 `primary_media.public_url`。前端 `loadTopBanner()` 应直接取 `items[0].image_url`（已是后端 JOIN `media_files` 拼好的完整代理 URL），**不要期望 `primary_media` 嵌套对象**。同理日志上报用 `items[0].ad_campaign_id`（`ad_slot_id` 后端当前不在 delivery 响应里下发，若日志必需见 8.4 拍板项）。
2. **"已后端化"程度校正**：B 类（轮播/弹窗/公告）**接口通但内容全空**（`ad_campaigns`/`ad_creatives` 0 行）。即"运营可配"是能力就绪、**实际尚未配置任何内容**。这不是 bug，但前端"已运营可配"表述应理解为"接口就绪、需运营录入内容后才有图"。

### 8.4 后端执行步骤（小而清晰，复用现有投放系统，零新表/零新接口）

> 核心思路：top_banner 只是 ad 投放系统的**第 5 种 slot_type**，全程复用现有 `ad_slots`/`ad_campaigns`/`ad_creatives`/`media_files` 四表与 `selectWinners` 竞价/选择逻辑，**不新建表、不新建接口、不新增响应字段**。

**步骤 1 — 统一 slot_type 白名单（消除三处不一致，单一真相源）**
- `models/AdSlot.js`：`VALID_SLOT_TYPES` 改为 `['popup','carousel','announcement','feed','top_banner']`，并更新 `slot_type` 字段 comment。
- `routes/v4/system/ad-delivery.js`：`validSlotTypes` 同步加 `top_banner`（建议直接 `require` 模型导出的 `VALID_SLOT_TYPES`，避免再硬编码第 4 处）。
- `routes/v4/console/ad/ad-slots.js`：`VALID_SLOT_TYPES` 从 `['popup','carousel']` 对齐为模型同一份（含 announcement/feed/top_banner）。
- 收益：顺手修掉"运营连 announcement/feed 槽位都建不了"的既有技术债，三处校验收敛为一处（模型导出）。

**步骤 2 — 建 top_banner 槽位种子数据（走 migration，禁止手改生产）**
- 为覆盖范围内每个页面建一行 `ad_slots`：`slot_key={position}_top_banner`、`slot_type='top_banner'`、`slot_category='display'`、`position∈{lottery,profile,diy,camera}`、`is_active=1`、定价字段给运营内容默认值（参考现有 popup 槽位取值）。
- ⚠️ position 命名对齐后端既有口径：前端文档写 `user`，但后端现有 profile 槽位用 **`profile`**（`profile_popup`）。**以后端为准用 `profile`**，前端 `pages/user` 调接口时传 `position=profile`（前端适配后端，不为前端改后端命名）。
- `diy`/`camera` 两个 position 后端目前无任何槽位，按需建（见 8.5 拍板覆盖范围）。

**步骤 3 — 选择逻辑零改动（复用 selectWinners）**
- `AdBiddingService.selectWinners('{position}_top_banner', userId)` 自动按 `slot_key` 匹配新槽位、按 priority 选生效中的投放，返回扁平 `items`。顶部 Banner 单张：前端取 `items[0]`；后端无需为"单张"特殊改造（priority 降序天然第 1 条即最高优先级）。

**步骤 4 — 响应结构零改动**
- 沿用现有扁平响应 `{ items:[{image_url, link_url, link_type, campaign_category, ad_campaign_id, ...}], slot_type, position, total }`。**不新增字段、不改字段名**。前端按 `image_url` 直读（见 8.2 校正 1）。

**步骤 5 — 运营配置闭环（管理端，已具备能力，仅需 B5 前端放开下拉）**

### 12.1 拍板项1 — 管理端 position 是否一次性补全 5 个（diy/camera/exchange）

**问题本质**：运营位的"位置维度"是预先全建，还是按需开。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂（美团/腾讯/阿里） | "资源位"按需开通：上游有位才在管理台出现该 position，不预建空位（空位增加配置噪音与下发开销，且影响数据看板口径） | 按需开位、位即数据 |
| 小公司/SaaS | 配置表里把所有可能位置写死成枚举，一次列全，省得以后改代码 | 全量预置、图省事 |
| 游戏公司 | 位置走资源热更配置（JSON 下发），新增位改配置不改代码，倾向"用到才加" | 配置即位、按需 |
| 活动策划公司 | 按活动模板的"楼层/坑位"动态生成，活动要哪几个坑就开哪几个 | 强按需、随活动上下线 |
| 虚拟物品交易/小众二手 | 首页就几个固定推荐位，数量少，往往一次建全 | 位少、全量预置 |

**方案区别**：全量预置=管理端一次列全 5 个 position，运营随时可建（哪怕前端还没消费）；按需开位=前端真正要用某 position 时才在管理端加该选项，避免"建了没人用的空位"。

**对你项目的建议**：✅ **一次性补全 5 个 position**（diy/camera/exchange 一起加进 `POSITION_MAP` + 下拉）。
- 理由：你已拍板1 "5 个 Tab 全做"，且后端 B4 已要为 5 个 position 建 `{position}_top_banner` 槽位种子。若管理端 position 下拉不同步补全，会出现"后端建了 `exchange_top_banner` 槽位、运营在管理端却选不到 exchange"的断层。
- 这与大厂"按需开位"看似冲突，但你的 position 是**有限固定的 5 个 Tab**（不是大厂那种成百上千动态资源位），全量预置的配置噪音可忽略，反而省掉"日后再改 `POSITION_MAP` 发版"的技术债——符合你"一次性投入、降低技术债"。
- 成本：管理端 `POSITION_MAP` + 下拉各加 3 个常量项，几行代码，零风险。

### 12.2 拍板项2 — `is_carousel` 新增列默认值

**问题本质**：给存量 7 个槽位加"是否轮播"开关时，默认值取什么、会不会影响现状。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂 | 加列必带"向后兼容默认值"，新列默认 = 还原旧行为，存量数据零感知 | 默认即现状、灰度无感 |
| 小公司/SaaS | 加列时常忘设默认或设了与现状不符的值，导致存量行为突变（典型事故） | 易踩坑 |
| 游戏公司 | 配置版本化，新字段给保守默认 + 灰度开关 | 保守默认 |
| 活动策划公司 | 新坑位属性默认"关"，要用再显式开 | 默认关闭 |
| 虚拟物品交易/小众二手 | 字段少，默认值多取"最常见形态" | 默认常见值 |

**方案区别**：`DEFAULT 0`（单张）= 存量 7 个槽位迁移后都是单张，与现状完全等价；`DEFAULT 1`（轮播）= 会把现有 `home_carousel` 等位的行为语义改变，且单张位被误标轮播。

**对你项目的建议**：✅ **`DEFAULT 0`（单张）**。

- 这是行业标准、也是技术债最小的选择，确认采用。

### 12.3 拍板项3 — 轮播间隔 `slide_interval_ms` 的来源（多条投放间隔不一致时以谁为准）

**问题本质**：`slide_interval_ms` 在 `ad_campaigns`（每条投放各配），一个轮播槽位下多条投放间隔可能不同，前端轮播该用哪个值。

| 公司类型 | 典型做法 | 特征 |
| --- | --- | --- |
| 大厂（美团/腾讯/阿里） | 轮播节奏是**槽位/楼层级**展示属性，由位统一控制，不让单条素材各报间隔（避免一个 swiper 内忽快忽慢） | 间隔属于"位"、全局统一 |
| 小公司/SaaS | 间隔配在素材上，前端取第一条或硬编码，常出现节奏不一致 | 配在素材、易乱 |
| 游戏公司 | 轮播参数在"轮播容器配置"里，素材只管图，参数归容器 | 容器级参数 |
| 活动策划公司 | 楼层配置带轮播节奏，素材挂楼层下，节奏归楼层 | 楼层级参数 |
| 虚拟物品交易/小众二手 | 多直接前端写死 3~5 秒，不做后台可配 | 前端写死 |

**方案区别（你项目的现实约束：`slide_interval_ms` 已在 campaign，不在 slot）**：
- **方案甲（零改动，推荐起步）**：前端轮播取 `items[0].slide_interval_ms`（priority 最高项）为准。后端零改、契约已满足。缺点：语义上轮播节奏挂在"某一条投放"上，运营要理解"间隔以最高优先级那条为准"。
- **方案乙（更贴大厂，小改）**：在 `ad_slots` 增加槽位级 `slide_interval_ms`（轮播节奏归位），响应顶层下发该值；campaign 的同名字段保留兼容或弃用。最干净、最符合"间隔属于位"的行业共识，但要再加一个槽位字段 + 管理端把"轮播间隔"从投放表单挪到槽位表单。

**对你项目的建议**：✅ **先用方案甲（取 `items[0]`，零改动）落地；若运营反馈节奏混乱，再升级方案乙**。
- 理由：你已在 11.4 决定 `is_carousel` 加在槽位（位级展示形态），按大厂"轮播节奏属于位"的逻辑，`slide_interval_ms` **理应也归位（方案乙）**才最一致。但方案乙要再加一个槽位字段 + 管理端表单迁移，属本轮之外的额外成本。
- 权衡"长期维护成本低"：方案甲零改动、契约已通，先上；它与方案乙的差异仅在"多条投放且间隔填得不一致"这一边缘场景，而你 top_banner 默认单张（拍板3），该场景实际很少触发。
- 若你倾向"一次做到位、不留边缘隐患"（符合你"愿意一次性投入"），则直接选**方案乙**：把 `slide_interval_ms` 提升为槽位级，与 `is_carousel` 同表同维度，语义最干净、技术债最小。这是唯一需要你二选一的点（见 12.4）。

### 13.2 轮播时长运营能设多少（含一处真实现状校正）

- 数据库 `ad_campaigns.slide_interval_ms`：`INTEGER`、`allowNull=true`、`DEFAULT 3000`、校验 `min=500`（"轮播间隔不能小于500毫秒"），无上限校验。

**建议运营可设范围（结合现有 `min=500` 校验 + 行业 banner 轮播节奏）**：

- `slot_type=top_banner`（固定）
- `position=` 取本页：`lottery`(回馈/抽奖页) / `profile`(我的页，注意：前端 `pages/user` 传 `profile`) / `diy`(发现页) / `camera`(DIY页) / `exchange`(商城页)

**响应结构（真实下发，扁平）**：

```json
{
  "success": true,
  "data": {
    "items": [
      {
        "ad_slot_id": 16,
        "ad_campaign_id": 344,
        "campaign_category": "operational",
        "image_url": "https://<PUBLIC_BASE_URL>/api/v4/images/<object_key>",
        "link_url": null,
        "link_type": "none",
        "priority": 102
      }
    ],
    "slot_type": "top_banner",
    "position": "lottery",
    "is_carousel": false,
    "slide_interval_ms": 3000,
    "total": 1
  }
}
```

**前端取数规则（按真实验证行为）**：
1. `is_carousel === false`（单张）：后端只返回 1 条，前端取 `items[0].image_url` 赋给 `topBannerUrl`。
2. `is_carousel === true`（轮播）：后端返回多条(priority 降序)，前端用 swiper 渲染 `items[].image_url`，轮播间隔用顶层 `slide_interval_ms`(毫秒)。
3. `items` 为空数组：回退本地兜底图，不报错、不留白。
4. 点击跳转：`link_type !== 'none'` 时按 `link_url`/`link_type`(none/page/miniprogram/webview) 跳转；为 none 则纯展示。
5. 曝光/点击上报：用 `items[i].ad_slot_id` + `ad_campaign_id` 调现有 `ad-events` 上报接口。

> `image_url` 已是后端 JOIN `media_files` 拼好的完整代理 URL，前端直接用，不要自行拼接。
