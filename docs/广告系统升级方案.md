### 1.6 Web 管理后台前端现状与命名歧义

#### 内容管理中心三个 tab 对应的后端系统

Web 管理后台的「内容管理中心」页面有 3 个 tab，分别对接 3 张独立的后端表：

| 图片资源 | 图片资源 | `image_resources` | ImageResource（无歧义） |

#### 命名歧义说明

Web 前端将 `popup_banners`（弹窗横幅）标记为"轮播图管理"，与后端命名不一致：

| 对比项 | 后端命名（权威） | Web 前端命名（有歧义） |
|---|---|---|
| 数据库表 | `popup_banners` | — |
| 主键 | `popup_banner_id` | 前端 JS 中也用 `popup_banner_id` |
| Model | `PopupBanner` | — |
| Service | `PopupBannerService` | — |
| API 路径 | `/api/v4/console/popup-banners` | 调用正确，但 UI 标签写"轮播图" |
| 代码注释 | 全部写"弹窗 Banner" | JS 日志写"加载轮播图列表" |
| **tab 标签** | — | **"轮播图管理"（应改为"弹窗管理"）** |

"轮播图"通常指页面内自动滑动切换的横幅组件（swiper/carousel），而 `popup_banners` 是弹窗浮层（modal），两者交互形态完全不同。

#### `popup_banners` 与 `system_announcements` 是两个独立系统

| 对比维度 | popup_banners（弹窗横幅） | system_announcements（系统公告） |
|---|---|---|
| 核心内容 | 图片（image_url 必填） | 纯文字（title + content TEXT） |
| UI 形式 | 弹窗浮层 modal，覆盖页面 | 文字条/通知列表 |
| 数据量 | 2 条记录 | 50+ 条记录（含测试数据） |
| 类型字段 | 无（本次新增 `banner_type`） | `type` ENUM: system/activity/maintenance/notice |
| 优先级 | 无（本次新增 `priority` INT） | `priority` ENUM: high/medium/low |
| 目标控制 | `position`（home/profile） | `target_groups` JSON |
| 阅读追踪 | 无（客户端本地存储） | `view_count`（服务端计数） |
| 频率控制 | **本次新增**（客户端执行） | 无（依赖 is_active + expires_at） |
| 图片支持 | 有（完整的上传/存储/CDN 转换） | 无 |

> 本文档的频率控制方案仅针对 `popup_banners`，不涉及 `system_announcements`。两者是完全独立的系统，不存在功能交叉。

## 二、需要新增的 5 个字段（以后端数据库为准）

### 2.1 数据库迁移

在 `popup_banners` 表新增 5 列：

| 字段名 | 类型 | NOT NULL | 默认值 | 说明 |
|---|---|---|---|---|
| `banner_type` | VARCHAR(20) | YES | `'promo'` | 类型分级：`notice`(系统公告) / `event`(活动推广) / `promo`(日常促销)。轮播图独立为 `carousel_items` 表（拍板决策1）。远期 `user_ad` 通过 `ad_campaigns` 表管理 |
| `frequency_rule` | VARCHAR(30) | YES | `'once_per_day'` | 频率规则枚举（见 2.2） |
| `frequency_value` | INT | NO | `1` | 频率参数（配合 `once_per_n_days` / `n_times_total` 使用） |
| `force_show` | TINYINT(1) | NO | `0` | 是否强制弹出（1=不可点遮罩关闭） |
| `priority` | INT | NO | `0` | 弹出优先级（数字越大越优先，区别于 display_order 的管理排序） |

新增 2 个索引：

| 索引名 | 字段 | 用途 |
|---|---|---|
| `idx_popup_banners_type_active` | (banner_type, is_active) | 按类型筛选活跃弹窗 |
| `idx_popup_banners_priority` | (priority) | 优先级排序 |

> **命名说明**: 用 `banner_type` 而非 `type`，避免 Sequelize STI 保留字冲突。

### 2.2 `frequency_rule` 枚举值定义

| frequency_rule | frequency_value 用法 | 含义 |
|---|---|---|
| `always` | 忽略 | 每次进入页面都弹 |
| `once` | 忽略 | 整个活动周期只弹一次 |
| `once_per_session` | 忽略 | 每次小程序冷启动弹一次 |
| `once_per_day` | 忽略 | 每天最多弹一次 |
| `once_per_n_days` | N（天数） | 每 N 天弹一次 |
| `n_times_total` | N（总次数） | 累计最多弹 N 次 |

### 2.3 `priority` 与 `display_order` 的职责区分

| 字段 | 职责 | 排序方向 | 使用场景 |
|---|---|---|---|
| `priority` | 弹出竞争优先级 | DESC（大的先弹） | 多个活跃弹窗竞争时决定谁先弹 |
| `display_order` | 管理后台列表排序 | ASC（小的排前） | 管理员在后台看到的排列顺序 |

## 三、可复用的现有基础设施

以下是项目中已成熟运行的组件，本次直接复用，无需重建：

### 3.1 迁移机制

- **工具**: `sequelize-cli`，迁移文件放 `migrations/` 目录
- **命名规范**: `YYYYMMDDHHMMSS-描述.js`（如 `20260218170000-add-frequency-fields-to-popup-banners.js`）
- **复用方式**: `queryInterface.addColumn()` + `queryInterface.addIndex()` + `queryInterface.bulkInsert()` 字典数据

### 3.2 字典体系

- **存储**: `system_dictionaries` 表，字段 dict_type / dict_code / dict_name / dict_color / sort_order
- **注册**: `utils/displayNameHelper.js` 的 `DICT_TYPES` 常量
- **调用**: `attachDisplayNames(data, [{ field, dictType }])` 自动附加 `_display` 和 `_color` 后缀
- **复用方式**: 新增 `banner_type` 和 `banner_frequency` 两组字典，注册 `BANNER_TYPE` 和 `BANNER_FREQUENCY` 到 `DICT_TYPES`

### 3.3 Service 层模式

- **基类**: 静态方法 class，无需实例化
- **日志**: `logger.info/error()` 结构化日志
- **图片 URL**: `_transformBannerImageUrl()` 统一将对象 key 转完整 CDN URL
- **中文名附加**: `attachDisplayNames()` 自动为枚举字段附加中文显示名
- **复用方式**: 在现有 `PopupBannerService` 的方法中追加字段即可

### 3.4 路由层模式

- **认证**: `adminAuthMiddleware` 管理员鉴权中间件
- **异步**: `asyncHandler(fn)` 包装器
- **服务获取**: `req.app.locals.services.getService('popup_banner')`
- **响应格式**: `res.apiSuccess(data, message)` / `res.apiError(message, code, detail, status)`
- **复用方式**: 在现有路由文件中追加参数校验和筛选逻辑

### 3.5 前端适配约定

- 前端**直接使用后端字段名**（snake_case），不做映射
- 前端通过 `_display` 后缀字段获取中文名（如 `banner_type_display`）

## 十、注意事项

1. **项目未上线**: 无存量数据兼容压力，一次性做到位
2. **现有 2 条记录**: 迁移自动填入默认值（`banner_type='promo'`, `frequency_rule='once_per_day'`），不影响现有功能
3. **频率判断在客户端执行**: 后端只负责存储和下发规则配置，不做频率计算（减少后端状态管理复杂度）
4. **前端直接使用后端字段名**: snake_case，不做 camelCase 映射，不做字段重命名
5. **字典中文名自动附加**: 前端通过 `_display` / `_color` 后缀获取，无需额外接口调用
6. **命名统一原则**: 以后端 `popup_banners` / "弹窗横幅" 命名为权威，Web 管理后台前端原有的"轮播图"命名有歧义（轮播图指 swiper/carousel，弹窗指 modal popup，交互形态完全不同），必须改为"弹窗"
7. **本方案不涉及 `system_announcements`**: 系统公告是独立系统（纯文字、服务端 view_count、ENUM 优先级），与弹窗横幅（图片、客户端频率控制、INT 优先级）无功能交叉

## 十一、行业设计方案对比与选型

### 11.1 弹窗/轮播图/公告的本质区别

| | 弹窗 Popup | 轮播图 Carousel | 公告 Announcement |
|---|---|---|---|
| 交互形态 | 浮层覆盖页面，必须关闭才能继续 | 嵌在页面内，自动滑动，不阻断操作 | 文字条/列表，用户主动查看 |
| 用户感受 | 打断式，有侵入感 | 浏览式，自然融入页面 | 信息式，被动接收 |
| 典型位置 | 屏幕正中央 | 页面顶部/中部固定区域 | 页面顶部滚动条或通知中心 |
| 关闭方式 | 点关闭按钮/遮罩 | 不需要关闭，一直在 | 点已读/忽略 |
| 频率敏感度 | 极高（弹太多用户会卸载） | 低（一直展示不打扰） | 中等 |

这三者在大多数公司是不同的东西，但很多公司会把它们放在同一个后台系统里统一管理。

### 11.2 四种行业设计方案

| 方案 | 设计思路 | 适合场景 |
|---|---|---|
| **A. 大厂统一运营位** | 1 张表 + `slot_type` 区分所有内容类型（弹窗/轮播/横幅/悬浮按钮），后端不关心渲染方式，前端根据 slot_type 选组件 | 日活百万+，运营团队 10 人+，投放位 20+ |
| **B. 游戏队列系统** | 弹窗有独立的队列调度器（一次弹一个，关了再弹下一个），轮播图/通知各自独立系统 | 同屏弹窗 5+，需要严格排队，需要服务端展示日志做转化分析 |
| **C. 简单分离** | popup_banners + swiper_banners + announcements 各一张表各管各的 | 小团队，功能明确 |
| **D. 弹窗轮播合并** | 一张 popup_banners 表通过 `banner_type` 区分弹窗和轮播图，前端根据类型选择渲染组件 | 内容量 < 20 条，运营 1-3 人 |

### 11.3 方案 A / B / D 核心区别

| 对比维度 | A. 统一运营位 | B. 游戏队列 | D. 弹窗轮播合并 |
|---|---|---|---|
| 表数量 | 1 张（所有类型） | 3+ 张（弹窗/轮播/通知各一套） | 1 张 popup_banners |
| 弹窗排队 | 无专门队列，共用优先级池 | 有独立队列调度器，一次弹一个 | 有 priority 排序，无队列 |
| 展示日志 | 不记录或仅记"是否展示过" | 服务端日志表记录展示时长、关闭方式 | 客户端本地存储 |
| 渲染控制 | 后端 `render_config` JSON 存渲染参数 | 各系统各自硬编码 | 前端根据 banner_type 硬编码 |
| 扩展新类型 | 加枚举值 + render_config 模板 | 新建表 + 接口 + 管理页 | 加 banner_type 枚举值 |
| 开发量 | 中（需要 render_config 抽象层） | 高（每种类型全套） | 低（加字段即可） |

### 11.4 本项目选型：方案 B + 虚拟货币广告平台

**所有广告位均采用方案 B**（队列系统 + 服务端展示日志 + 逐个追踪），同时建设**完整的虚拟货币广告平台**（竞价+定向+反作弊+DMP），用钻石（DIAMOND）代替真实货币。

**核心决策**：

| 决策项 | 结论 |
|---|---|
| 弹窗展示方式 | 方案 B 队列系统：一个接一个弹，关了再弹下一个，最多 5 个排队 |
| 轮播图展示方式 | 方案 B：按 priority 排序决定滑动顺序，每张滑过都记录服务端展示日志（展示时长、是否点击），逐个追踪 |
| 弹窗与轮播图 | 区分开：弹窗是浮层 modal（必须关闭），轮播图是页面内嵌 swiper（自动滑动）。渲染方式不同，但都走方案 B 的服务端追踪和优先级排序 |
| 所有图片类内容 | 全部接入广告管理系统统一管理（弹窗/轮播图/广告位图片） |
| 自营内容 | 不再有"自营"概念，运营配置的 notice/event/promo 也纳入广告系统，运营只调优先级排序 |
| 计费方式 | 两种并存：广告主可选"固定价格包天"（N 钻石=M 天）或"竞价排名"（出价高的优先展示） |
| 竞价引擎 | 需要：竞价模式下，多个广告主出不同价格竞争同一广告位 |
| 人群定向（DMP） | 需要：基于用户行为标签做精准定向投放 |
| CPC/CPM 计费 | 不需要：不按点击/曝光扣费，按天扣费 |
| 反作弊 | 需要：防止刷曝光量虚假抬高展示数据 |
| 归因追踪 | 需要：追踪广告点击后是否产生了转化行为 |
| 广告主后台 | 需要：用户在小程序内自助创建广告、查看报表 |
| 素材审核 | 需要：管理员人工审核 |
| 架构 | 单体 Node.js，不拆微服务 |

**广告位类型**（统一管理，全部方案 B）：

| 广告位类型 | 渲染方式 | 方案 B 体现 | 服务端日志 |
|---|---|---|---|
| 弹窗广告位 | 弹窗 modal（浮层覆盖） | 队列排序：一个接一个弹，关了再弹下一个，最多 5 个 | 每个弹窗记录：展示时长、关闭方式、是否点击跳转 |
| 轮播图广告位 | 页面内嵌 swiper（自动滑动） | 优先级排序：priority 高的排在前几张，先被用户看到 | 每张轮播记录：曝光时长（停留在该张的时间）、是否点击跳转 |

**运营调配的内容类型**（纳入广告系统统一管理，运营只设优先级）：

| 内容类型 | 广告位 | 优先级范围 | 说明 |
|---|---|---|---|
| `notice`（系统公告） | 弹窗 | 900-999（最高） | force_show=true，用户必须点"我知道了" |
| `event`（活动推广） | 弹窗/轮播 | 500-899 | 活动期间展示 |
| `promo`（日常促销） | 弹窗/轮播 | 100-499 | 日常促销信息 |
| `user_ad`（广告主投放） | 弹窗/轮播 | 1-99（最低） | 广告主花钻石投放 |

**弹窗队列行为**（方案 B）：

```
用户打开首页 → 后端返回弹窗队列（最多 5 个，按 priority DESC 排序）
  → 弹出第 1 个（notice: 系统维护公告, priority=999）
  → 用户关闭 → 服务端记录：展示 3.2 秒，点了"我知道了"
  → 弹出第 2 个（event: 春节活动, priority=500）
  → 用户关闭 → 服务端记录：展示 1.1 秒，点了遮罩关闭
  → 弹出第 3 个（user_ad: 竞价胜出的广告, priority=50）
  → 用户关闭 → 服务端记录 + 触发计费
  → 终于看到首页
```

**轮播图行为**（方案 B）：

```
首页加载 → 后端返回轮播图列表（按 priority DESC 排序）
  → swiper 组件按顺序自动滑动
  → 第 1 张（event: 春节活动海报, priority=600）展示 3 秒
     → 服务端记录：曝光 3 秒，未点击
  → 自动滑到第 2 张（promo: 新品推荐, priority=300）展示 3 秒
     → 服务端记录：曝光 3 秒，未点击
  → 自动滑到第 3 张（user_ad: 竞价广告, priority=50）展示 3 秒
     → 服务端记录：曝光 3 秒 + 触发计费
  → 用户手动滑回第 1 张 → 服务端记录：二次曝光
  → 用户点击第 1 张 → 服务端记录：点击事件 + 跳转
```

## 十二、虚拟货币广告平台（完整方案）

### 12.1 业务目标与定位

建设完整的广告投放平台，所有计价/预算均使用虚拟钻石（DIAMOND）代替真实货币，为虚拟道具增加核心消耗场景。保持单体架构，不拆微服务。

| 维度 | 规格 |
|---|---|
| 货币 | 虚拟钻石（DIAMOND），不涉及真实货币 |
| 广告主规模 | 数万级（用户自助注册投放） |
| 用户规模 | 百万级 |
| 在投广告量 | 数千条 |
| 决策延迟 | 3 秒内 |
| QPS | 数万级 |
| 架构 | 单体 Node.js（现有架构不变） |
| 计费模式 | 固定价格包天 + 竞价排名（广告主二选一） |

### 12.2 现有虚拟经济体系（数据库实查）

| 虚拟资产 | asset_code | 持有用户 | 总流通量 | 可交易 | 当前消耗途径 |
|---|---|---|---|---|---|
| 钻石 | `DIAMOND` | 18 人 | 727,132 | 可交易 | 抽奖、C2C 市场买入、高级解锁 |

现有 6 种消耗途径，广告平台 = **第 7 种核心消耗场景**。

### 12.3 两种计费模式

广告主创建广告时二选一：

| 模式 | 广告主操作 | 扣费方式 | 展示排序 | 适合 |
|---|---|---|---|---|
| **固定价格包天** | 选展示天数，支付 N 钻石买 M 天 | 创建时一次性冻结钻石，审核通过后扣除 | 按运营设定的 priority 排序 | 确定想展示、不想竞争的广告主 |
| **竞价排名** | 设日出价（如 10 钻石/天），设总预算 | 每天从账户扣日出价钻石，预算耗尽停投 | 出价高的优先展示（同一广告位多个竞价广告竞争） | 想花最少钱获得最多展示的广告主 |

**竞价排名流程**（同一个弹窗/轮播广告位有多个竞价广告竞争时）：

```
弹窗广告位：3 个竞价广告竞争
  广告主甲 出价 10 钻石/天，定向"抽奖活跃用户" → 对用户 A 有效
  广告主乙 出价 15 钻石/天，定向"全部用户"      → 对用户 A 有效
  广告主丙 出价 20 钻石/天，定向"新用户"         → 对用户 A 无效（不是新用户）
  → 用户 A 看到广告主乙的广告（有效出价最高）

轮播图广告位：按出价排序决定 swiper 中的展示顺序
  出价高的排在前几张，出价低的排在后面
```

### 12.9 全广告位方案 B 展示流程

```
用户打开小程序首页
  │
  ▼
后端（3 秒内完成）
  │
  │ === 弹窗广告位 ===
  │
  │ 第 1 步：获取运营弹窗
  │   查询广告系统中 banner_type IN (notice/event/promo)，广告位=弹窗，is_active=true，时间范围内
  │   运营弹窗按 priority DESC 排序
  │
  │ 第 2 步：竞价获取广告弹窗
  │   查询用户标签（user_ad_tags）
  │   从 ad_campaigns 中召回定向匹配 + 广告位=弹窗 + 预算未耗尽的候选
  │   固定包天广告：直接入选
  │   竞价广告：按出价高低排序，选出胜出者
  │
  │ 第 3 步：合并弹窗队列
  │   运营弹窗（priority 900-999）排在队列前面
  │   广告弹窗（priority 1-99）排在队列后面
  │   总队列最多 5 个
  │
  │ === 轮播图广告位 ===
  │
  │ 第 4 步：获取运营轮播
  │   查询广告系统中 banner_type IN (event/promo)，广告位=轮播，is_active=true，时间范围内
  │   按 priority DESC 排序
  │
  │ 第 5 步：竞价获取广告轮播
  │   从 ad_campaigns 中召回定向匹配 + 广告位=轮播 + 预算未耗尽的候选
  │   固定包天广告：直接入选
  │   竞价广告：按出价高低排序
  │
  │ 第 6 步：合并轮播图列表
  │   运营轮播按 priority 排在前面
  │   广告轮播按出价/priority 排在后面
  │   priority 高的排第 1 张（用户最先看到）
  ▼
小程序前端
  │
  │ 弹窗队列（方案 B）：
  │   弹第 1 个 → 用户关闭 → 上报展示日志（popup_show_logs：展示时长、关闭方式）
  │   弹第 2 个 → 用户关闭 → 上报展示日志
  │   ...最多弹 5 个
  │   广告弹窗关闭 → 额外上报（ad_impression_logs）
  │
  │ 轮播图（方案 B）：
  │   swiper 按后端返回的优先级顺序自动滑动
  │   每张滑过 → 上报展示日志（carousel_show_logs：曝光时长、是否手动滑入）
  │   广告轮播曝光 → 额外上报（ad_impression_logs）
  │   用户手动滑回某张 → 记录二次曝光
  │
  │ 点击行为（弹窗和轮播图统一）：
  │   用户点击任意内容 → 上报点击事件（ad_click_logs）
  │   后端检查反作弊规则 → 有效点击记入归因追踪
```

#### 14.4.2 核心 Service

**`services/AdBiddingService.js`**（新建）：

核心方法 `selectWinners(slotKey, userId)`，3 秒内完成：
1. 查 `ad_slots` 获取广告位配置（`max_display_count`, `min_bid_diamond`）
2. 查 `ad_campaigns` 召回候选：`ad_slot_id` 匹配 + `status='active'` + 日期范围内 + `budget_spent < budget_total`
3. 分类处理：
   - `fixed_daily`：直接入选（已付费）
   - `bidding`：按 `daily_bid_diamond DESC` 排序
4. Phase 5 启用后：增加 `targeting_rules` 与 `user_ad_tags` 匹配过滤

7. 返回胜出的广告列表

#### 14.4.3 定时任务：竞价日扣费

**`AdBillingService` 新增方法**：`processDailyBidding()`

- 执行时机：`node-cron` 凌晨 1 点
- 逻辑：
  1. 查所有 `billing_mode='bidding'` + `status='active'` 的 campaign
  2. 对每个 campaign 扣 `daily_bid_diamond` 钻石
  3. `budget_spent += daily_bid_diamond`
  4. 若 `budget_spent >= budget_total`，自动 `status='completed'`（预算耗尽）
  5. 扣费失败（余额不足）则 `status='paused'`
  6. 每笔扣费写入 `ad_billing_records`（`billing_type='daily_deduct'`）

3. 将广告 creative 数据映射为与弹窗相同的响应结构
4. 合并队列：运营弹窗在前 + 广告弹窗在后（按 priority 统一排序）

#### 14.5.4 targeting_rules JSON 格式

存储在 `ad_campaigns.targeting_rules` 字段中：

```json
{
  "match_all": [
    { "tag_key": "lottery_active_7d", "operator": "eq", "value": "true" },
    { "tag_key": "diamond_balance", "operator": "gte", "value": "500" }
  ],
  "match_any": [
    { "tag_key": "new_user", "operator": "eq", "value": "true" },
    { "tag_key": "market_trader", "operator": "eq", "value": "true" }
  ]
}
```

- `match_all`：所有条件都满足才投放（AND）
- `match_any`：任一条件满足即投放（OR）
- 两者同时存在时：`match_all AND match_any`
- 为空/null 时：投放给所有用户

## 十六、注意事项补充

1. **所有日志表用 BIGINT 主键**：`popup_show_logs` / `carousel_show_logs` / `ad_impression_logs` / `ad_click_logs` / `ad_billing_records` / `ad_antifraud_logs` / `ad_attribution_logs` / `ad_report_daily_snapshots` 均用 BIGINT，预留日志量增长空间
2. **幂等键覆盖所有扣费操作**：`ad_campaigns.business_id` 和 `ad_billing_records.business_id` 均设 UNIQUE 约束，复用 `IdempotencyService`
3. **广告素材复用现有图片上传能力**：`ad_creatives.image_url` 存对象 key，CDN 转换复用 `_transformBannerImageUrl()` 逻辑
4. **carousel_items 不需要频率控制**：轮播图是页面内嵌组件（一直展示），不像弹窗有打断性，因此不需要 `frequency_rule` 等字段
5. **弹窗队列最大数量可热更新**：存 `system_configs` 表，Service 每次读取，无需重启服务
6. **所有 ad_ 系列表的 FK 都设 `RESTRICT`**：防止误删关联数据
7. **targeting_rules 使用 JSON 字段而非关联表**：60 个用户、数千广告量级下，JSON 存储+应用层匹配足够，无需建独立的定向规则关联表

#### 弹窗频率控制（客户端实现）

后端只下发规则配置，**频率判断逻辑在客户端执行**，使用 `wx.getStorageSync` / `wx.setStorageSync` 存储展示记录。

**本地存储结构**（建议 key：`popup_records`）：

```json
{
  "15": {
    "last_seen": 1740000000000,
    "seen_count": 3,
    "dismissed": false,
    "session_shown": false
  },
  "16": {
    "last_seen": 1739900000000,
    "seen_count": 1,
    "dismissed": true,
    "session_shown": true
  }
}
```

**6 种频率规则判断逻辑**：

| `frequency_rule` | `frequency_value` | 跳过条件（满足则不展示） |
|---|---|---|
| `always` | 忽略 | 永不跳过，每次都展示 |
| `once` | 忽略 | `seen_count >= 1` |
| `once_per_session` | 忽略 | `session_shown === true`（小程序冷启动时重置为 false） |
| `once_per_day` | 忽略 | `Date.now() - last_seen < 86400000`（24小时） |
| `once_per_n_days` | N（天数） | `Date.now() - last_seen < frequency_value × 86400000` |
| `n_times_total` | N（总次数） | `seen_count >= frequency_value` |

**弹窗展示伪代码**：

```
1. GET /api/v4/system/popup-banners?status=active&position=home
2. 读取本地存储 popup_records
3. 遍历 banners 数组：
     对每个 banner 检查频率规则 → 通过的进入展示队列
4. 弹出队列第 1 个弹窗
5. 用户关闭：
     a. 更新本地存储（last_seen = Date.now(), seen_count++, session_shown = true）
     b. 上报展示事件（API 3）
     c. 弹出队列下一个弹窗
6. 所有弹窗展示完毕 → 用户看到正常页面
```

#### 弹窗 UI 行为（按 `banner_type` 区分）

| `banner_type` | 标题区域 | 遮罩点击 | 关闭按钮 | 额外按钮 |
|---|---|---|---|---|
| `notice` | 展示 `title` 字段 | `force_show=true` → 不可关闭 | 无 | "我知道了"（关闭弹窗） |
| `event` | 可选展示 `title` | 可关闭 | 有 | "查看详情"（跳转 `link_url`） |
| `promo` | 不展示标题 | 可关闭 | 有 | 无（点击图片跳转） |

| `image_width` | int/null | 原图宽度 px |
| `image_height` | int/null | 原图高度 px |
| `link_url` | string/null | 跳转目标 |
| `link_type` | string | `none`/`page`/`miniprogram`/`webview` |

**使用方式**：直接绑定到 `<swiper>` 组件

```html

</swiper>
```

### A.6 广告主功能对接（用户自助创建广告）

> 所有接口需要 JWT Token 鉴权

#### API 7：我的广告活动列表

```
GET /api/v4/user/ad-campaigns?status=&page=1&limit=20
```

| Query 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|

```
POST /api/v4/user/ad-campaigns

{
  "campaign_name": "春节促销广告",
  "ad_slot_id": 11,
  "billing_mode": "fixed_daily",
  "fixed_days": 7,
  "start_date": "2026-03-01",
  "end_date": "2026-03-07",
  "priority": 50
}
```

| Body 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `campaign_name` | string | **是** | 活动名称 |
| `ad_slot_id` | int | **是** | 广告位 ID（从广告位列表获取） |
| `billing_mode` | string | **是** | `fixed_daily`（固定包天）/ `bidding`（竞价排名） |
| `fixed_days` | int | 条件 | 固定包天天数（`billing_mode=fixed_daily` 时必填） |
| `daily_bid_diamond` | int | 条件 | 日出价钻石数（`billing_mode=bidding` 时必填，≥50） |

| `end_date` | string | 否 | 投放结束日期（YYYY-MM-DD） |

#### API 9：活动详情

```
GET /api/v4/user/ad-campaigns/:id
```

返回活动完整信息，含素材列表和计费记录。

#### API 10：更新活动（仅 `draft` 状态可编辑）

```
PUT /api/v4/user/ad-campaigns/:id

{ ...同创建字段... }
```

#### API 11：提交审核

```
POST /api/v4/user/ad-campaigns/:id/submit
```

- 固定包天模式：提交时**冻结**对应钻石（`daily_price × fixed_days`），余额不足则失败
- 竞价模式：提交时**冻结**首日出价钻石
- 状态变为 `pending_review`

#### API 12：取消活动

```
POST /api/v4/user/ad-campaigns/:id/cancel
```

- 冻结的钻石**自动退回**用户账户
- 状态变为 `cancelled`

#### API 13：活动数据报表

```
GET /api/v4/user/ad-campaigns/:id/report
```

返回曝光数、点击数、消耗钻石等统计数据。

### A.10 `popup_banners` 与 `carousel_items` 对比

前端需要理解两者是**完全独立**的系统，渲染方式不同：

| 对比维度 | 弹窗（popup_banners） | 轮播图（carousel_items） |
|---|---|---|
| 渲染方式 | **浮层 modal**（覆盖页面，必须关闭） | **页面内嵌 swiper**（自动滑动，不阻断） |
| 频率控制 | 有（客户端执行 6 种规则） | 无（一直展示） |
| 队列机制 | 有（最多 5 个排队，一个接一个） | 无（全部在 swiper 中同时展示） |
| 排序方式 | `priority DESC`（数字大的先弹） | `display_order ASC`（数字小的排前面） |
| `force_show` | 有（不可点遮罩关闭） | 无 |
| 获取 API | `GET /api/v4/system/popup-banners` | `GET /api/v4/system/carousel-items` |
| 上报 API | `POST .../popup-banners/show-log` | `POST .../carousel-items/show-log` |

### A.11 注意事项

1. **`image_url` 已是完整 CDN URL**：后端自动将对象存储 key 转换为完整 URL，前端直接使用，无需拼接
2. **排序由后端完成**：前端按返回数组的顺序展示即可，不需要自行排序
3. **频率控制在客户端执行**：后端不记录展示状态，只下发规则，客户端负责判断和本地存储
4. **`once_per_session` 的 session 定义**：小程序冷启动（`onLaunch`）时将所有记录的 `session_shown` 重置为 `false`
5. **展示事件上报是异步的**：上报失败不影响弹窗正常展示和关闭，catch 错误即可
6. **竞价广告由后端混排**：当广告系统上线后（Phase 4），`user_ad` 类型广告会和运营内容混排在同一个弹窗队列/轮播列表中，前端不需要区分来源，按数组顺序展示即可
