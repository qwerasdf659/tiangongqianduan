### 1.3 架构决策：渲染器选择

> **决策日期**: 2026-02-11
> **决策结论**: 选择方案A — 全局 WebView，清理 Skyline 残留配置

**问题背景**：`app.json` 中原有 `rendererOptions.skyline` 配置残留（已清理），项目实际运行在 WebView 上。此决策明确渲染器策略为全局 WebView，不采用 Skyline。

**Skyline 是什么**：微信自研的原生渲染引擎，绕过浏览器 WebView，用原生组件直接绘制 UI。卖点是长列表滚动更流畅（worklet 动画独立线程）、原生瀑布流组件 `masonry-layout`、共享元素过渡动画、理论上更快的首屏渲染。

**备选方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A：全局 WebView | 清理 `rendererOptions.skyline` 配置，保留 `glass-easel` | CSS 兼容性完整、Canvas 可在开发者工具调试、零迁移成本 | 无法使用 Skyline 原生瀑布流和 worklet 动画 |
| B：全局 Skyline | 添加 `"renderer": "skyline"` 全面启用 | 长列表性能理论更优、原生瀑布流组件 | CSS 限制多、Canvas 无法开发者工具调试、3D transform 未文档化 |
| C：混合模式 | 部分页面 Skyline、部分 WebView | 按需使用各自优势 | 两套渲染器维护成本翻倍、页面切换有额外开销 |

**选A的理由**：

1. **14种抽奖玩法全部与 Skyline 不兼容**：
   - Canvas 类玩法（转盘、刮刮卡、老虎机、弹珠机、砸金蛋等）：Skyline 下 Canvas 2D 能运行但无法在开发者工具调试，只能连真机。14种玩法的开发调试量很大，全程真机调试严重拖慢开发效率
   - CSS 动画类玩法（盲盒开箱、红包雨、扭蛋机）：需要 `rotateY`、`preserve-3d`、`backface-visibility` 等 CSS 3D transform，Skyline 对此没有明确文档支持，属于灰色地带，随时可能在某个基础库版本出问题
   - 视觉效果类（霓虹灯光效、发光按钮、卡片层叠阴影）：Skyline 只支持单个 `box-shadow`，多重阴影效果会被削弱
2. **商品瀑布流不需要 Skyline**：exchange 和 market 页面的瀑布流已用 JS 计算绝对定位实现，运行在 WebView 上无性能问题。要用 Skyline 的原生 `masonry-layout` 反而需要重写布局逻辑
3. **数据量有分页控制**：exchange 分页加载、market 无限加载均有分页，不会出现需要 Skyline 优化的超长列表场景
4. **行业实践**：美团/阿里/游戏公司/活动策划公司 95%+ 使用 WebView，只有腾讯自家应用激进采用 Skyline。Skyline 社区踩坑资料少，出问题排查困难
5. `glass-easel` 组件框架与渲染器无关，保留它不影响 WebView 运行，且是微信官方推荐的组件框架

**实施要点**：

- ~~删除 `app.json` 中的 `rendererOptions.skyline` 配置块~~ ✅ 已完成
- 保留 `componentFramework: "glass-easel"`（这是组件框架，不是渲染器）
- 不需要任何页面级别的渲染器声明（WebView 是默认值）

### 1.4 架构决策：14个子组件的加载策略

> **决策日期**: 2026-02-11
> **决策结论**: 选择方案A — 分包隔离（`packageLottery` 独立子包）

**问题背景**：`lottery-activity` 万能组件在 `lottery-activity.json` 的 `usingComponents` 中注册了14个子组件（grid/wheel/card/egg/scratch/blindbox/gashapon/luckybag/redpacket/slotmachine/whackmole/pinball/cardcollect/flashsale）。小程序 `usingComponents` 声明的组件无论是否使用都会加载，用户只看九宫格也会加载全部14个子组件的代码。

**备选方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A：分包隔离 | 把 `lottery-activity` 整体放进 `packageLottery` 独立子包，利用分包 + 分包预下载隔离体积 | 简单可靠、全行业验证、兼容所有基础库版本 | 整个抽奖模块变成子包 |
| B：动态组件 | 用 `<component is="{{name}}" />` 按需加载子组件 | 更精细的按需加载 | 需基础库 2.23.4+、调试困难、行业采用率 <5% |

**选A的理由**：

1. 全行业（美团/腾讯/阿里/游戏公司/活动策划公司）95%+ 的项目使用分包方案，踩坑资料多，出问题好排查
2. 分包预下载配合后，用户从首页进抽奖模块体感几乎无延迟
3. 不依赖高版本基础库，兼容性最好
4. 用户进了抽奖页大概率会用到多个组件，方案B的包内按需加载收益很小
5. 活动模块天然适合独立分包，与主包业务边界清晰

**实施要点**：

- `lottery-activity` 组件及全部子组件（sub/ + shared/ + themes/ + mock/）整体放入 `packageLottery/` 子包
- `app.json` 中配置 `subpackages` 和 `preloadRule`（从首页/抽奖入口预下载）
- 主包 `pages/lottery/lottery.json` 通过分包引用组件路径

### 1.5 架构决策：多实例状态隔离

> **决策日期**: 2026-02-11
> **决策结论**: 选择方案A — 组件内部 data 管理运行时状态，lotteryStore 降级为缓存/预加载层

**问题背景**：现有 `store/lottery.ts` 是单例 MobX Store，同一页面放两个 `lottery-activity` 组件会共享状态（prizes、isDrawing 互相打架）。

**备选方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A：组件内部 data | 万能组件用自身 `data` 管理运行时状态，lotteryStore 降级为缓存/预加载 | 天然隔离、组件销毁自动清理、行业标准做法 | 需要把 store 逻辑搬到组件，改动较大 |
| B：Store 改 Map | lotteryStore 改成 `Map<campaignCode, state>` 按 key 隔离 | 组件层改动小 | key 管理复杂、需手动清理防内存泄漏、同 key 跨页面仍会冲突 |

**选A的理由**：

1. 全行业（美团/腾讯/阿里/游戏公司/活动策划公司）90%+ 采用组件内部 data 管理运行时状态，Store 只用于跨页面共享数据（用户信息、积分余额）
2. 天然隔离，不可能状态串台——游戏行业对此极其敏感，两个抽卡池状态串了是致命 bug
3. 组件销毁时 data 自动清理，不用操心内存泄漏；方案B需要手动 `store.delete(key)`
4. 调试体验好，DevTools 直接看组件 data 即可，不用展开 Map 找 key
5. 现有 `lotteryStore` 里真正需要跨页面共享的只有用户积分余额（已在 `pointsStore` 里），抽奖运行时状态无跨页面需求

**不删除 lotteryStore 的理由**：

项目未上线，理论上可以直接删除 `store/lottery.ts`。但选择保留降级而非删除：

1. `lotteryStore` 缓存活动配置列表可避免多个组件实例重复请求同一活动配置，降级后仍有实际价值
2. 保留 store 文件为未来可能的跨页面共享需求（如"上次抽奖结果"通知栏）留出扩展点，成本仅为维护一个轻量文件
3. 与 `userStore` / `pointsStore` / `exchangeStore` / `tradeStore` 保持一致的 5 store 架构，认知模型统一

**实施要点**：

- `lottery-activity.ts` 的 `data` 管理全部运行时状态：prizes、isDrawing、highlightIndex、drawResult、displayMode、displayConfig 等
- `lotteryStore` 降级职责：缓存活动配置列表、预加载奖品数据（可选优化）。不删除，保留为轻量缓存层
- `pointsStore` / `userStore` 保持不变，组件通过 MobX observer 读取积分余额和用户信息
- 子组件（sub/grid、sub/wheel 等）通过 properties 接收父组件传入的数据，不直接访问任何 store

### 1.6 架构决策：Mock数据打包策略

> **决策日期**: 2026-02-11
> **决策结论**: 选择方案A — 运行时 `USE_MOCK` 开关，mock 文件随代码打包，上线前改为 `false`

**问题背景**：文档方案是运行时 `if (USE_MOCK)` 判断数据来源，mock 文件会进入生产包。是否需要构建时物理剔除？

**备选方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A：运行时开关 | `USE_MOCK = false` 即可，mock 文件留在包里 | 零成本、已实现、不引入构建复杂度 | mock 数据约 6KB 留在生产包 |
| B：构建时剔除 | 引入 gulp/自定义脚本在打包时物理删除 mock 目录 | 包体积完全干净 | 需要搭建构建管线，增加维护成本和出错风险 |

**选A的理由**：

1. 项目是原生微信小程序，没有现成构建管线，为 6KB 数据引入 gulp/构建脚本投入产出比太低
2. mock 数据总计约 6KB（压缩后），占 `packageLottery` 分包 2MB 上限的 0.3%，完全可忽略
3. mock 文件在 `packageLottery` 分包内，不占主包体积
4. `USE_MOCK = false` 加进上线 checklist，比维护构建脚本更可靠
5. 未来项目规模膨胀需要构建管线时，再升级到方案B也很容易

**实施要点**：

- `mock-switch.ts` 加醒目注释提醒上线前修改：

  ```typescript
  // ⚠️ 上线前必须改为 false ⚠️
  export const USE_MOCK = true
  ```

- 上线 checklist 中增加一项：确认 `USE_MOCK = false`

### 1.7 架构决策：特殊玩法API模型

> **决策日期**: 2026-02-11
> **决策结论**: 选择方案A — 万能组件统一处理标准流程，3个特殊玩法子组件自己调额外API

**问题背景**：14种玩法中，11种使用标准 `performLottery(code, count)` 即可，但集卡/秒杀/打地鼠需要额外API：

| 特殊玩法 | 额外需要的API | 与标准流程的差异 |
|---------|-------------|----------------|
| 集卡 | `getCollectionProgress()` 查进度、`redeemCollection()` 兑换大奖 | 抽卡用标准API，但有额外的进度查询和兑换流程 |
| 秒杀 | `getRealTimeStock()` 实时库存、`rushBuy()` 抢购 | 完全不用 `performLottery`，是先到先得模型 |
| 打地鼠 | `submitGameScore(score)` 提交得分换奖品 | 完全不用 `performLottery`，是游戏计分模型 |

**备选方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A：子组件自己调特殊API | 万能组件只管标准流程，3个特殊子组件自己 import API 调用 | 简单直接、万能组件保持简洁、调试容易 | 3个子组件不纯粹，会直接调API |
| B：父组件提供扩展钩子 | 万能组件设计 onBeforeDraw/onCustomAction 等钩子协议 | 所有API调用统一经过父组件 | 钩子设计复杂、3种特殊交互差异大难以统一覆盖、过度设计 |

**选A的理由**：

1. 14种玩法只有3种（21.4%）需要额外API，为此设计通用钩子系统是过度设计
2. 这3种玩法的额外API差异极大（查进度/实时库存/提交得分），无法用一套钩子优雅覆盖
3. 与决策1.5（组件内部data管理状态）天然配合——子组件既然已经自治管理状态，自己调API也顺理成章
4. 全行业（美团/腾讯/游戏公司/活动策划公司）90%+采用此模式：父组件管通用流程，特殊交互子组件自治

**实施要点**：

万能组件 `lottery-activity.ts` 统一处理（所有玩法通用）：

- `loadConfig(campaignCode)` — 加载活动配置
- `loadPrizes(campaignCode)` — 加载奖品列表
- `performDraw(campaignCode, count)` — 11种标准玩法的抽奖调用
- 通过 properties 把配置和奖品传给子组件

特殊子组件自己额外处理：

- `sub/cardcollect/cardcollect.ts` — 自己调 `getCollectionProgress()` 和 `redeemCollection()`
- `sub/flashsale/flashsale.ts` — 自己调 `getRealTimeStock()`（轮询）和 `rushBuy()`（替代 performDraw）
- `sub/whackmole/whackmole.ts` — 自己调 `submitGameScore(score)`（替代 performDraw）

### 1.8 架构决策：模板路由方案

> **决策日期**: 2026-02-11
> **决策结论**: 接受 wx:elif 条件渲染方案，不加 JS 映射表，wxml 末尾用 `wx:else` 兜底默认九宫格

**问题背景**：`lottery-activity.wxml` 中 14 个 `wx:elif` 根据 `display.mode` 条件渲染子组件，每新增一种玩法需要改 wxml 加一行。是否需要用动态组件替代？是否需要额外的 JS 映射表？

**结论**：elif 不是硬编码问题。运营动态调整的能力由后端 `display.mode` 字段保证——运营在后台改 mode 值，前端自动渲染对应组件。elif 只是"前端支持哪些 mode"的声明。新增全新玩法无论用什么方案都需要前端写新组件代码并发版，elif 不是瓶颈。

**不加映射表的理由**：

- wxml 的 elif 链本身就是 mode 的完整声明，映射表是重复信息
- wxml 末尾 `wx:else` 兜底默认九宫格，已能处理后端返回不认识的 mode，效果与映射表校验一致
- 不加映射表少维护一处，新增玩法改动集中在 wxml + json 两处即可

**新增玩法 checklist**：① 开发子组件 ② wxml 加一行 elif ③ json 注册组件

**不选动态组件的理由**：

- 决策 1.3 已选分包隔离方案，与动态组件互斥
- 动态组件需基础库 2.23.4+，分包方案兼容所有版本
- 行业采用率 <5%，elif 方案 85%+

## 7. 明确不做的事项

以下事项经《UI设计方案》行业调研后明确排除：

| 不做的事 | 原因 |
|---------|------|
| 动态网格数量（3×3/4×4/5×5自适应） | 全行业无公司这样做，格子数量固定是行业共识 |
| 前端根据奖品数量重新布局 | 奖品超过8个时由后端选8个展示，其余在奖池中 |
| 动态计算顺时针路径 | 固定8格路径即可，无需动态计算 |
| 骨骼动画 / Live2D | 微信小程序不支持 WebGL |
| 视频过场CG | video组件可用但包体积爆炸，不推荐 |
| 全屏烟花（Canvas粒子版） | 中低端机卡顿，改用简化CSS版本 |
| 12格/15格九宫格 | 格子小文字挤、动画节奏差、无行业先例 |

**奖品超过8个的处理方式**（行业标准做法）：

```
后端奖池（例如12个奖品）
├── 展示层（8个）→ sort_order 1-8 → 放在九宫格上
│   └── 运营选择最能吸引用户的8个奖品
│
└── 隐藏层（4个）→ 不在格子上显示
    └── 抽中时在结果弹窗展示
```

万能组件 `lottery-activity.ts` 中根据开关选择数据来源：

  return API.getLotteryConfig(campaignCode)

// 加载奖品列表

  if (USE_MOCK) {

// 执行抽奖
async performDraw(campaignCode: string, count: number) {

### 10.4 mock-draw-result.ts — 抽奖结果模拟数据

模拟中奖/未中奖结果，按概率随机返回：

```typescript
/**
 * 模拟抽奖结果
 * 按奖品概率加权随机，模拟真实后端行为
 */
function draw(campaignCode: string, count: number) {
  const MockPrizes = require('./mock-prizes')
  const { data: prizes } = MockPrizes.getPrizes(campaignCode)

  const results = []
  for (let i = 0; i < count; i++) {
    const prize = weightedRandom(prizes)
    results.push({
      is_winner: true,
      prize_id: prize.id,
      name: prize.name,
      icon: prize.icon,
      rarity: prize.rarity,
      sort_order: prize.sort_order,
      type: 'prize'
    })
  }

  // 单抽返回单个对象，连抽返回数组
  return {
    success: true,
    data: count === 1 ? results[0] : { items: results, count: count }
  }
}

/** 按概率加权随机选择奖品 */
function weightedRandom(prizes: any[]) {
  const totalWeight = prizes.reduce((sum, p) => sum + p.probability, 0)
  let random = Math.random() * totalWeight
  for (const prize of prizes) {
    random -= prize.probability
    if (random <= 0) return prize
  }
  return prizes[prizes.length - 1]
}

module.exports = { draw }
```
