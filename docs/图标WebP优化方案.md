# 天工小程序 - 图标资源 WebP 优化方案

> 编写日期：2026-02-22

> 状态：已决策，待执行  
> 决策人：项目负责人  
> 涉及目录：`images/icons/materials/`、`images/icons/categories/`  
> 不涉及：TabBar 图标（微信要求 PNG）、`images/ui/grain-texture.png`（体积小，不处理）

---

## 1. 问题背景

微信开发者工具代码依赖分析显示，主包体积 **1.59MB**，其中 `images/` 目录占 **954KB（60%）**。

主要原因是 `images/icons/` 下的 24 个 256×256 PNG 图标文件体积过大：

| 目录 | 文件数 | 总体积 | 用途 |
|------|--------|--------|------|
| `images/icons/materials/` | 15 个 | 554.8 KB | 材料资产图标（钻石、积分、碎片×6、水晶×6） |
| `images/icons/categories/` | 9 个 | 362.9 KB | 商品分类图标（数码、食品、礼品卡等 9 类） |
| **合计** | **24 个** | **917.8 KB** | |

单个图标平均 38KB，最大 45.2KB（collectible.png），最小 27.7KB（yellow-shard.png）。

## 2. 行业方案调研

### 2.1 大厂方案

| 公司 | 图标策略 | 特点 |
|------|---------|------|
| 美团 | CDN + 按设备 DPR 返回 @1x/@2x/@3x | 需要图片处理服务，运维成本高 |
| 阿里/淘宝 | OSS 图片处理，URL 参数动态裁剪 | 依赖阿里云 OSS，有使用成本 |
| 腾讯/微信 | 腾讯云 COS + 数据万象，自动 WebP 转换 | 生态内方案，成本中等 |

### 2.2 游戏公司方案

| 公司 | 图标策略 | 特点 |
|------|---------|------|
| 米哈游 | TexturePacker 雪碧图 + JSON 坐标 | H5 活动页常用，维护成本高 |
| 网易游戏 | CSS Sprite 按场景分组 | 增删图标需重新生成，不灵活 |
| 腾讯游戏 | 英雄/道具图标雪碧图 | 适合大量同尺寸图标场景 |

### 2.3 虚拟物品交易/二手平台

| 平台 | 图标策略 | 特点 |
|------|---------|------|
| Steam/CSGO 饰品 | 全部走 CDN，按需加载缩略图 | 图标数量巨大，必须 CDN |
| 网易 BUFF | CDN + 列表小图/详情大图双版本 | 需后端图片处理 |
| 转转/闲鱼 | CDN + OSS 图片处理自动裁剪 | 依赖云服务 |

### 2.4 小公司/独立开发者

| 策略 | 特点 |
|------|------|
| 本地 PNG 直接打包 | 零成本，但包体积大（当前状态） |
| TinyPNG 压缩 | 简单易用，但对渐变图标效果有限 |
| 转 WebP 格式 | 压缩率高，微信小程序原生支持 |

### 2.5 不适合本项目的方案

| 方案 | 排除原因 |
|------|---------|
| Iconfont 字体图标 | 只支持单色，水晶/碎片图标是多色渐变 |
| SVG 矢量图 | 复杂渐变光影效果 SVG 体积可能反而更大，微信小程序 SVG 支持有限 |
| CSS Sprite 雪碧图 | 图标在不同页面显示尺寸差异大（36rpx vs 260rpx），难以适配 |
| CDN 远程加载 | 图标数量固定仅 24 个且不频繁变动，上 CDN 增加不必要的复杂度和网络依赖 |

## 4. 决策结论

**采用 WebP quality 90 方案。**

### 4.1 决策理由

1. **高质量优先**：quality 90 视觉上与原图无差异，在水晶渐变光影等细节上零损失
2. **压缩率依然可观**：918KB → 366KB，减少 60%，主包从 1.59MB 降至约 **1.04MB**
3. **分辨率不变**：保持 256×256，在所有显示尺寸（36rpx ~ 260rpx）下均清晰
4. **兼容性已成熟**：微信小程序基础库 2.9.0+（2020 年起）原生支持 WebP，`<image>` 组件可直接加载
5. **改动范围最小**：仅替换图片文件 + 修改 `utils/image-helper.ts` 中的路径后缀
6. **不引入外部依赖**：无需 CDN 服务、无需字体文件、无需构建工具

### 4.2 选择 quality 90 而非 80 的理由

| 对比项 | quality 80 | quality 90 |
|--------|-----------|-----------|
| 总体积 | 238 KB | 366 KB |
| 减少比例 | 74% | 60% |
| 视觉差异 | 几乎无法分辨（仔细对比可察觉极微弱差异） | 与原图无差异 |
| 体积差 | — | 比 q80 多 128KB |

多出的 128KB 换来的是零视觉损失的保障，对于暗黑奢华风的游戏素材类图标，画质优先是正确的选择。

### 4.3 排除分包方案的理由

虽然分包可以将主包图标降为 0KB，但存在以下问题：
- materials 和 categories 图标被 `packageTrade` 和 `packageExchange` 两个分包共用，移到某一个分包后另一个分包无法跨包引用
- 需要复制两份图标到两个分包，反而增加总体积
- WebP 压缩后主包图标仅 366KB，已从 918KB 大幅下降，主包总体积可控

### 4.4 不处理的图片资源

| 文件 | 体积 | 不处理原因 |
|------|------|-----------|
| `images/icons/*.png`（TabBar 图标 ×8） | 14.8 KB | 微信 tabBar 的 `iconPath` 仅支持 PNG 格式，不支持 WebP |
| `images/ui/grain-texture.png` | 20.6 KB | 体积极小，优化收益不足 1%，不值得改动 |

## 附录 A：为什么 PNG 无损压缩对这批图标无效

测试结果显示 PNG compressionLevel 9 仅减少 0.1%（918KB → 917KB）。

原因：这批图标在生成时已经使用了较优的 PNG 压缩参数（DEFLATE 算法），再次无损压缩无法进一步减小。PNG 的无损压缩只能优化编码方式，不能减少实际像素数据量。

## 附录 B：为什么 TinyPNG（PNG palette）不适合这批图标

TinyPNG 的核心算法是 pngquant，原理是将真彩色（1600 万色）降至 128-256 色调色板。

这批图标特点：
- 暗黑奢华风格，大量平滑渐变（水晶光泽、碎片光影）
- 颜色过渡细腻（红→暗红、蓝→深蓝等连续色谱）

降色后的问题：
- 渐变区域出现可见的色阶断层（color banding）
- 在 240-260rpx 大卡片上肉眼可辨
- 在 36rpx 小图标上不明显

实测 PNG palette 压缩率为 64%（918KB → 328KB），虽然体积减少可观，但视觉质量下降不可接受。

## 附录 C：WebP 为什么能同时兼顾体积和质量

WebP 使用 VP8 编码器（基于视频压缩技术），与 PNG palette 的关键区别：

| 特性 | PNG palette | WebP 有损 |
|------|------------|-----------|
| 压缩原理 | 减少颜色数量 | 频率域变换 + 预测编码 |
| 渐变处理 | 色阶断层 | 平滑保持 |
| 适合场景 | 色块分明的简单图标 | 渐变、光影、照片类 |
| 本项目实测 | 64% 压缩率，质量下降 | 60% 压缩率（q90），质量无损 |

WebP 的频率域变换能智能识别人眼不敏感的高频细节并压缩，同时保留人眼敏感的低频渐变信息，因此在保持视觉质量的同时实现更高的压缩率。
