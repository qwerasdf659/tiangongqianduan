# 前端需求求证：抽奖系统 API 响应数据结构确认

> 提交方：前端开发
> 日期：2026-02-12
> 优先级：高（阻塞九宫格/转盘奖品显示）
> 关联接口：3 个
> 后端求证完成日期：2026-02-11
> 求证方式：基于后端实际代码 + 真实数据库数据（非文档推测）

---

## 一、背景

前端在对接抽奖系统时，发现九宫格和转盘组件**无法正确显示奖品名称**。
经排查，根本原因是前端对以下接口的 `data` 字段结构存在不确定性，需要后端明确确认。

---

## 二、后端确认结果总览

```
接口1：方案 A 变体 — data 直接返回数组，但外层有 apiSuccess 信封包装
接口2a：Header 名 Idempotency-Key 正确，HTTP header 不区分大小写，无格式要求，有效期 7 天，超时重试应复用同一 key
接口2b：单抽和连抽统一结构 — data 内含 prizes 数组（始终为数组，即使单抽也是长度为1的数组）
接口3：placement 字段后端尚未实现（数据库无此列，代码无此逻辑），属于待开发功能
奖品字段：icon 为 emoji 字符串（由 prize_type 自动映射），sort_order 从 1 开始，奖品数量不固定（非固定8个）
```

## 三、接口逐项确认

### 接口 1：获取奖品列表

| 项目 | 内容 |
|------|------|
| 路由 | `GET /api/v4/lottery/campaigns/:campaign_code/prizes` |
| 用途 | 加载九宫格/转盘等玩法的奖品展示数据 |

**后端确认：方案 A 变体**

所有 API 响应经过 `res.apiSuccess()` 统一封装，实际返回结构为：

```json
{
  "success": true,
  "code": "PRIZES_SUCCESS",
  "message": "奖品列表获取成功",
  "data": [
    {
      "id": 1,
      "name": "八八折",
      "type": "coupon",
      "icon": "🎫",
      "rarity": "uncommon",
      "available": true,
      "display_points": 150,
      "display_value": "中价值",
      "status": "active",
      "sort_order": 1
    }
  ],
  "timestamp": "2026-02-11 20:30:00",
  "version": "v4.0",
  "request_id": "xxx"
}
```

**关键事实（源自代码 `DataSanitizer.sanitizePrizes()`）：**

- `data` 直接是数组，不是 `{ prizes: [...] }` 也不是 `{ items: [...] }`
- 前端应使用 `response.data` 直接作为奖品数组
- 字段映射关系（数据库字段 → API 输出字段）：

| 数据库字段 | API 输出字段 | 转换逻辑 |
|-----------|-------------|---------|
| `lottery_prize_id` | `id` | 直接映射（防止暴露数据库结构） |
| `prize_name` | `name` | 直接映射 |
| `prize_type` | `type` | 直接映射 |
| （无对应列） | `icon` | 由 `prize_type` 自动映射生成 emoji |
| （无对应列） | `rarity` | 由 `prize_type` 自动映射生成稀有度 |
| `stock_quantity` | `available` | `stock_quantity > 0` 转为布尔值 |
| `prize_value` | `display_points` | DECIMAL 转 number |
| （无对应列） | `display_value` | 由 `prize_value` 数值区间映射文本 |
| `status` | `status` | 直接映射 |
| `sort_order` | `sort_order` | 直接映射 |

**前端不会收到的字段（已脱敏移除）：**
`win_probability`、`stock_quantity`、`prize_value`、`cost_points`、`max_daily_wins`、`daily_win_count`、`angle`、`color`、`prize_value_points`

**前端需要注意的差异点：**

前端原始 `Prize` 接口定义中的以下字段与后端实际不一致：

| 前端期望字段 | 后端实际情况 | 说明 |
|------------|------------|------|
| `tier` (SSR/SR/R/N) | 不返回 | 后端使用 `reward_tier`（high/mid/low），但该字段仅在抽奖结果中返回，奖品列表不返回 |
| `probability` | 不返回 | 已脱敏，用 `rarity` 替代（common/uncommon/rare/epic/legendary） |
| `is_winner` | 不返回 | V4.0 已废弃此概念，100% 中奖，用 `reward_tier` 区分档位 |
| `description` | 不返回 | 数据库有 `prize_description` 字段但脱敏时未输出 |
| `icon` (图片URL) | 返回 emoji | 由 `prize_type` 映射，非图片 URL |

### 接口 2：执行抽奖

| 项目 | 内容 |
|------|------|
| 路由 | `POST /api/v4/lottery/draw` |
| 请求体 | `{ "campaign_code": "xxx", "draw_count": 1 }` |

#### 问题 2a：Idempotency-Key 确认

**后端确认（源自 `IdempotencyService.js` 第 78-93 行）：**

| 确认项 | 后端实际实现 | 代码依据 |
|--------|-------------|---------|
| Header 名称 | `Idempotency-Key` | 代码读取 `req.headers['idempotency-key']`，HTTP header 规范不区分大小写 |
| 值格式 | 无强制格式要求 | 类型为 `VARCHAR(100)`，最大长度 100 字符，任意字符串均可 |
| 有效期 | 7 天 | 代码常量 `TTL_DAYS = 7`，创建时 `expires_at = now + 7天` |
| Processing 超时 | 60 秒 | 代码常量 `PROCESSING_TIMEOUT_SECONDS = 60`，超时自动转 `failed` 允许重试 |
| 重试场景 | **必须复用同一 key** | 复用 key 时返回缓存结果 + `is_duplicate: true`；换新 key 会导致重复扣费 |

**幂等状态机：**

```
新 key → processing → completed（成功）→ 后续相同 key 返回缓存 + is_duplicate: true
                   ↘ failed（失败/超时）→ 允许相同 key 重试
相同 key + 不同参数 → 409 IDEMPOTENCY_KEY_CONFLICT
相同 key + processing 中 → 409 REQUEST_PROCESSING（建议 1 秒后重试）
```

**真实数据库幂等记录示例（`api_idempotency_requests` 表）：**

```
idempotency_key: "rate_limit_0_1770380743627_0245c090"
api_path: "/api/v4/lottery/draw"
http_method: "POST"
status: "completed"
expires_at: "2026-02-13T12:25:43.000Z"
```

## 四、奖品字段确认

### icon 字段

**后端确认：返回 emoji 字符串，非图片 URL**

由 `DataSanitizer.getPrizeIcon(prize_type)` 自动映射，规则如下：

| prize_type | icon | rarity |
|-----------|------|--------|
| `points` | 🪙 | common |
| `physical` | 🎁 | epic |
| `voucher` | 🎫 | uncommon |
| `virtual` | 💎 | rare |
| `special` | ⭐ | legendary |
| `coupon` | 🎁（默认值） | common（默认值） |
| 其他/空值 | 🎁（默认值） | common（默认值） |

注意：数据库中 `prize_type` 的 ENUM 值为 `points/physical/virtual/coupon/service`，但 `getPrizeIcon` 的映射表中使用的 key 是 `points/physical/voucher/virtual/special`。这意味着：
- `coupon` 类型的奖品（如"八八折"、"九八折券"）会命中默认值，icon 为 🎁 而非 🎫
- `service` 类型同样命中默认值

如果前端需要奖品图片，应使用 `image_resource_id` 关联的图片资源，但当前数据库中所有奖品的 `image_resource_id` 均为 `null`。

### sort_order 字段

**后端确认：从 1 开始**

- 数据库真实数据：BASIC_LOTTERY 活动的 9 个正式奖品 sort_order 为 1-9
- 同一活动内 sort_order 唯一（有唯一索引 `idx_unique_campaign_sort_order`）
- 模型有 `beforeCreate` / `beforeUpdate` 钩子防止重复

### 奖品数量

**后端确认：不固定为 8 个**

真实数据库数据：
- BASIC_LOTTERY 活动：9 个正式奖品（sort_order 1-9）+ 6 个 fallback 空奖（sort_order 209-709）= 共 15 个
- 其他活动：7-8 个不等

前端九宫格如果固定 8 格，需要自行处理：
- 奖品不足 8 个时：前端补位（如显示"谢谢参与"占位格）
- 奖品超过 8 个时：前端截取前 8 个（按 sort_order 排序），或过滤掉 `is_fallback` 的空奖

注意：普通用户 API 不返回 `is_fallback` 字段（已脱敏），但可以通过 `type === 'virtual'` 且 `display_points === 0` 来识别空奖。

## 五、前端兼容措施更新建议

基于后端确认结果，前端应做如下调整：

| 文件 | 当前兼容逻辑 | 建议调整 |
|------|------------|---------|
| `lottery-activity.ts:144` | `Array.isArray(data) ? data : (data?.prizes \|\| data?.items \|\| [])` | 奖品列表：直接用 `response.data`（是数组）；抽奖结果：用 `response.data.prizes` |
| `grid.wxml` | `prizes[N].name \|\| ''` | 保留兜底，但字段名 `name` 已确认正确 |
| `wheel.wxml` | `item.name \|\| '奖品'` | 保留兜底，字段名正确 |
| `api.ts:performLottery` | 自动生成 `Idempotency-Key` header | 正确，但注意：网络超时重试时**必须复用同一 key**，不能重新生成 |
| Prize 类型定义 | 包含 `tier`/`probability`/`is_winner` | 移除这三个字段，改用 `rarity`/`reward_tier`/`available`/`display_points`/`display_value` |
