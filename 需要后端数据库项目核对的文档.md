## 二、核心问题核查结果：今日积分趋势

### 问题原始描述

微信小程序"我的"页面中 **今日获得** 和 **今日消费** 显示为 0。

### 后端实际状态：接口已完整实现，功能正常

**接口路径**：`GET /api/v4/lottery/points`
**源码位置**：`routes/v4/lottery/user-points.js:38-73`
**服务调用链**：

```
router.get('/points')
  → Promise.all([
      UserService.getUserWithPoints(user_id),
      AssetQueryService.getTodaySummary({ user_id, asset_code: 'POINTS' })
    ])
```

后端使用 `Promise.all` 并行查询积分余额 + 今日汇总（`user-points.js:46-52`），性能已优化。

**后端 getTodaySummary 实际 SQL**（`services/asset/QueryService.js`）：

```sql
SELECT
  COALESCE(SUM(CASE WHEN delta_amount > 0 THEN delta_amount ELSE 0 END), 0) AS today_earned,
  COALESCE(SUM(CASE WHEN delta_amount < 0 THEN ABS(delta_amount) ELSE 0 END), 0) AS today_consumed,
  COUNT(*) AS transaction_count
FROM asset_transactions
WHERE account_id = :account_id
  AND asset_code = :asset_code
  AND created_at >= :today_start
  AND created_at < :today_end
```

- 时间范围由 `BeijingTimeHelper.todayStart()` / `todayEnd()` 生成，基于 `Asia/Shanghai` 时区
- 统计范围覆盖 **所有 business_type**（不限于抽奖）

**数据库真实数据验证**（2026-02-21 Node.js 直连查询）：

| 指标 | 值 |
|------|-----|
| asset_transactions 总记录数 | 28,012 |
| 今日 POINTS 交易笔数（全用户） | 279 |
| 今日 POINTS 全用户获得总额 | 3,128,762 |
| 今日 POINTS 全用户消费总额 | 24,960 |

**结论：后端接口和数据库均正常，today_summary 字段存在且有数据。**

### "显示为0"的可能根因排查方向

| 排查点 | 说明 |
|--------|------|
| 1. 用户当天无交易 | 如果用户当天确实没有任何积分变动，返回 0 是正确行为 |
| 2. 用户无 accounts 记录 | 新用户首次查询时 `getTodaySummary` 会因无 account 直接返回全 0 |
| 3. 前端请求是否成功 | 检查网络请求日志，确认 HTTP 200 且 `success: true` |
| 4. 前端解析路径是否正确 | 确认读取的是 `data.today_summary.today_earned`，不是 `data.today_earned` |

## 七、决策记录

### 决策 D-1：积分今日汇总接口路径 ✅ 已拍板 → 新增资产域通用接口

**决定**：新增 `GET /api/v4/assets/today-summary?asset_code=POINTS`，原 `/lottery/points` 中的 `today_summary` 拼接逻辑删除。

#### 行业对比分析

| 公司类型 | 典型代表 | 钱包/资产 vs 业务活动 的 API 关系 | 设计原则 |
|----------|---------|-------------------------------|----------|
| 大厂 | 美团/支付宝/腾讯 | **完全分离**。支付宝"余额"和"花呗"是独立服务域，美团"钱包"和"外卖红包"是不同微服务。钱包不会出现在外卖下单接口的响应里 | 领域驱动设计（DDD），一个域不侵入另一个域 |
| 游戏公司 | 米哈游/网易/暴雪 | **完全分离**。原神的"原石余额查询"和"祈愿抽卡"是独立接口。抽卡接口只返回抽到什么，不附带钱包余额 | 抽卡和钱包是两个独立子系统 |
| 活动策划 | 有赞/微盟/小鹅通 | **完全分离**。积分余额走账户系统 API，营销活动（抽奖/转盘）走活动系统 API | SaaS 平台天然按领域分离 |
| 游戏交易平台 | Steam/5173/交易猫 | **完全分离**。Steam 钱包余额、市场挂牌、交易记录各自独立 API | 资产域和交易域分离 |
| 初创小公司 | 常见反模式 | 为了"少请求一次"，把余额塞进抽奖接口 → 后期需要在其他页面也展示余额时被迫重构 | 短期省事，长期负债 |

#### 为什么你的项目必须拆分

1. **后端已有完整资产域**：`/api/v4/assets/` 下已有 balance、balances、transactions、conversion-rules 四个路由文件，`today-summary` 天然属于这个域
2. **服务层已是通用的**：`QueryService.getTodaySummary({ user_id, asset_code })` 支持任意资产代码，不限于 POINTS
3. **项目有 15 种资产类型**（material_asset_types 表），DIAMOND/水晶碎片等今后都可能需要"今日汇总"，如果绑定在 lottery 域下则每次新增都要改 lottery 路由
4. **项目未上线**，拆分零成本，不拆分长期负债

**后端执行步骤**：
1. `routes/v4/assets/` 下新增 `today-summary.js`，调用 `AssetQueryService.getTodaySummary`
2. `routes/v4/assets/index.js` 注册：`router.use('/', require('./today-summary'))`
3. `routes/v4/lottery/user-points.js:46-61` 删除 `today_summary` 并行查询和拼接，只保留 `UserService.getUserWithPoints(user_id)`

**小程序前端执行步骤**：
1. 拆分 `getUserStatistics()` 为两个独立 API 调用
2. 积分余额：`GET /api/v4/lottery/points`
3. 今日汇总：`GET /api/v4/assets/today-summary?asset_code=POINTS`

**Web 管理后台**：无需改动

**可复用性**：`getTodaySummary` 支持任意 asset_code，未来查 DIAMOND/red_shard 等资产的今日汇总零代码改动，前端只需换 query param。
